## 1. Summary

지금 사용자가 `works-<handle>` 채널의 **과제 스레드** 안에서 메시지를 보내면 **Chief가 못 읽는다** — 디스코드 리스너가 `command-<handle>` 채널만 처리하고 스레드는 입구에서 버리기 때문(`discord-adapter.ts:339-348`). 이는 **권한 문제가 아니라 코드 경계**다(초대 권한엔 `SendMessagesInThreads`, 인텐트엔 `MessageContent` 가 **이미 포함**). 본 patch는 리스너를 **works 스레드로 확장**해 스레드 메시지를 Chief 로 라우팅하고 **그 스레드 안에서 응답**하게 한다. **세션은 기존 단일 `(user,org)` Chief 세션을 공유**(per-과제 격리 없음 — 후속). Chief 가 어느 과제 스레드인지 알도록 스레드↔워크플로 매핑(`discord-thread.txt`)을 **컨텍스트로 주입**한다.

## 2. Background

- **JTBD:** "works 스레드에서 진행되는 과제에 대해, command 채널로 돌아가지 않고 **그 스레드 안에서 바로** Chief에게 후속 질문·지시를 하고 답을 받고 싶다."

### 2.1 현재 상태 (코드 근거)

| 항목 | 코드 | 함의 |
|---|---|---|

| **권한·인텐트** | 초대 URL 권한에 `SendMessagesInThreads`·`CreatePublicThreads`·`ManageThreads`·`ReadMessageHistory`(`discord-invite-url.ts`), 인텐트 `GuildMessages`+`MessageContent`(`discord-adapter.ts:233-239`) | **이미 충족** — 권한 추가 불필요, 순수 코드 |
| **스레드 소유** | works 스레드는 봇이 직접 생성(`discord-task-card` `startThread`) | 봇은 이미 스레드 멤버 |

### 2.2 결함

works 스레드는 "과제 진행 표면"으로 만들어졌지만 **읽기 전용**(봇이 내레이션·카드를 post 할 뿐, 사용자 입력은 수신 안 함). 사용자는 과제 맥락이 스레드에 있는데도 **command 채널로 나가서** 말해야 하고, Chief 는 "어느 과제 얘기인지" 텍스트로 다시 설명받아야 한다.

## 3. Objective

v1.x KR "대화로 운영"의 **표면 확장** 조각: 과제가 사는 곳(스레드)에서 바로 대화. 측정 KR: ①스레드 메시지→Chief 응답이 **그 스레드 안**에 도달(MTTA·정확위치) ②command 채널 회귀 0(기존 동작 무변경).

## 4. Target Segments

- **주 segment:** works 스레드에서 과제를 추적하며 후속 지시를 스레드에서 하고 싶은 운영자.
- **non-target:** per-과제 *세션 격리*("그 작업 컨텍스트만") 기대 사용자 — 본 patch 는 단일 세션 공유라 다른 과제 맥락이 섞일 수 있음(후속 S-4).

## 5. Value Proposition

> works 스레드에서 과제를 보는 운영자가(누가), 후속 대화를 위해 command 채널로 나갔다 오는 마찰과 "어느 과제인지" 재설명 부담 때문에(왜), **리스너의 스레드 확장 + 스레드↔과제 컨텍스트 주입 + 스레드 내 응답**으로(어떻게), **"과제가 있는 자리에서 그대로 대화"** 를 얻는다(가치).

## 6. Solution

### 6.1 동작

1. **리스너 확장** — `messageCreate` 에서 채널이 **스레드**이고 그 **부모가 `works-<handle>`** 이면 수신(현재처럼 버리지 않음). 부모 채널명으로 봇/org/handle 해소(스레드명이 아니라 `channel.parent.name` 파싱).
2. **소유자 게이트 유지** — command 채널과 동일하게 *이 봇 사용자 소유* + owner-only 가드 적용(타 사용자 스레드·타 봇 스레드 무시). 봇 자신의 메시지는 기존 `message.author.bot` 필터로 제외.
3. **과제 컨텍스트 주입** — 스레드 id 로 `discord-thread.txt` 역참조해 워크플로 id 를 찾고, Chief 입력에 `[thread-context] 이 스레드는 과제 <wf-id> 입니다` 앰비언트 주입(세션은 공유하되 Chief 가 맥락을 알게). 매핑 없으면 일반 대화로 처리.
4. **스레드 내 응답** — `DiscordMessageContext` 를 **그 스레드로 스코프**해 `reply()` 가 스레드에 post(command 채널 아님). `[Chief]` 프리픽스·세션 시작 마커(v1.4.0) 그대로.
5. **세션** — 기존 `(user,org)` Chief 세션 **그대로 공유**(handleUserMessage). 신규 세션 키 없음.

### 6.2 ≥2 approaches (gstack rule)

| | **A — 단일 세션 + 리스너 확장 (본 patch, 권고)** | B — per-과제 자식 세션 (§11 하이브리드, 후속 S-4/S-5) |
|---|---|---|
| 세션 | 기존 단일 `(user,org)` 공유 | 신규 `(user,org,task)` 자식 세션 |
| 격리 | ❌ 과제 맥락 섞임(컨텍스트 주입으로 완화) | ✅ "그 작업에 대해서만" |
| 작업량 | **소** — 리스너 분기 + 스레드 컨텍스트 + 스레드 응답 | **대** — 세션 키 확장·hydrate/flush·환류 |
| 리스크 | 낮음(세션 모델 무변경) | 중(세션 모델·spawn hot-path) |
| patch 적합성 | ✅ | ⚠️ minor |

- **권고: A.** "스레드에서 대화" 라는 핵심 가치를 **세션 모델 무변경**으로 즉시 제공. B(격리)는 A 의 리스너·라우팅 토대 위에 자연히 얹힘.
- **반증 조건:** 단일 세션에서 **여러 과제 스레드를 오가면 맥락 오염**이 실사용에 거슬림 → B 로 승격. 컨텍스트 주입(§6.1-3)이 그 오염을 충분히 줄이는지 dogfood 로 판정.

### 6.3 Hard gate

- works 과제 스레드에 보낸 사용자 메시지 → Chief 가 읽고 **그 스레드 안에** 1회 응답(command 채널 아님).
- **보안 패리티:** 타 사용자/타 봇/비-works 스레드는 무시(command 채널 가드와 동형). 회귀 박제.
- **command 채널 동작 무변경**(기존 평탄 응답·카드 라우팅 그대로).

## 8. Metrics & Verification

- **Success threshold:** ①과제 스레드 메시지→**그 스레드 내** Chief 응답(dogfood 100%) ②보안 가드(타인/비-works 무시) 회귀 통과 ③command 채널 회귀 0.
- **수용 기준 (R8 — Given-When-Then):**
  ```
  Given  works-<handle> 의 과제 스레드(discord-thread.txt 매핑 존재)에서
  When   소유자가 그 스레드에 메시지를 보낸다
  Then   Chief 가 그 메시지를 읽고 같은 스레드 안에 응답하며,
         입력엔 [thread-context] 로 해당 과제 wf-id 가 주입돼 있다

  Given  타 사용자의 스레드 / 비-works 스레드 / 봇 자신의 메시지
  When   해당 스레드에 메시지가 올라온다
  Then   Chief 는 처리하지 않는다(command 채널 보안 가드와 동일)

  Given  command-<handle> 채널
  When   기존처럼 메시지를 보낸다
  Then   동작은 v1.4.0 과 동일(평탄 응답 / 카드 라우팅 무변경)
  ```

> **R6 분기 — LLM/에이전트 surface 변경?** 부분 Yes — Chief 입력에 `[thread-context]` 앰비언트가 추가되나 *허용 답변 범위·툴·프롬프트 구조*는 무변경(앰비언트 텍스트 주입은 기존 `[ambient] focus` 와 동형). AI 부록 경량 적용.
