## 1. 배경 / 문제

SoloSquad 에는 사용자·Chief 가 다루는 5개의 **1급 자산(first-class artifact)** 이 있다:

| 자산 | 정의 | 런타임 효과 |
|---|---|---|
| **skill** | `SKILL.md` (frontmatter + 본문) | 트리거 시 에이전트에 주입되는 절차적 역량 |
| **agent** | `agents/{main,specialists}/<name>/SKILL.md` + `agent-profile.yaml` | 과제를 수행하는 **행위자(actor)** — main(오케스트레이션)·specialist(실무) |
| **workflow** | `workflow.yaml` + `_status.yaml` | 다단계 specialist 오케스트레이션 |
| **goal** | `goal.md` (metrics/pipeline/budget) | 자율 엔진의 keep/discard 사이클 실행 |
| **cron** | `crons/<id>.md` + cron | 정기 cron 실행 |

> **레이어링 (이번 개정의 핵심 관점).** 5개 자산은 평면이 아니라 2층이다:
> **역량 층** = skill(무엇을 할 수 있나) + agent(누가 그 역량을 들고 행동하나),
> **호출 층** = agent 를 *언제·어떻게* 깨우는가. 호출은 3개 자산이 전부가 아니라 **4가지** —
> ⓪ **직접 대화 디스패치**(사용자 → Chief → spawn, `handleUserMessage`→DECOMPOSE→DISPATCH;
> 기본·휘발·완전 동적, 즉 "그냥 말하기") + 3개의 *지속·구조화* 자산 ① workflow(정적 DAG)
> ② goal(자율 루프) ③ cron(시간 트리거, 발화 시 ⓪/①/② 중 하나를 깨움).
> **workflow/goal/cron 은 직접 디스패치(⓪)를 *박제·자동화*한 형태**일 뿐 — 별도 종족이
> 아니다. 그래서 agent 가 호출보다 근본적이다: 어떤 호출이든 결국 actor 를 깨우고,
> `<team>/<agent>` 참조가 가리킬 진실원(source of truth)이 필요 → **agent-manager 가
> 직접 디스패치 + 나머지 3개 자산의 참조 무결성 토대.**

각 자산마다 **생성 경로는 있으나 생애주기 관리가 제각각·불완전**하다. v1.3.1 §7 의 결론
("`skill-author` 는 생성만 함의해 역할이 좁다")은 **5개 모두에 동일하게 적용**된다. 공통 진단:

- **생성은 있고 검증/개선은 빈약** — skill 은 author 가 있지만 validate 갭(아래 §3), workflow 는
  `workflow.yaml` validator 가 **아예 없음**, goal 은 parse-time 검증만, cron 은 **사용자 생성
  경로 자체가 없음**(`CRONS[]` 하드코딩), agent 는 번들 30여 개가 **불변**이고 frontmatter
  그래프(collaborators·skills_used·used_by)의 **참조 무결성·순환 검증이 없다.**
- **개선 루프 단편적** — `skill-refinement`·`workflow-refinement` skill 은 있으나 agent/goal/cron
  은 없고, 어느 것도 측정→bounded-edit→재검증 루프(§9)를 갖추지 않음.
- **lifecycle 표면 불균형** — goal 은 풍부한 CLI(new/list/show/run/status/stop/verify/queue),
  workflow 는 list/show/focus, skill·agent·cron 은 거의 없음.

목표: 5개 자산을 **일관된 "매니저" 추상(create/review/validate/refine + lifecycle ops)** 으로
정렬해, 사용자가 "만들고 → 검증받고 → 돌려보고 → 개선" 하는 경험을 동일하게 갖게 한다.

### 1.1 As-Is → To-Be 한눈에

| 자산 | As-Is (지금) | To-Be (v1.3.2) |
|---|---|---|
| **skill** | 생성 성숙하나 **이름/중복/길이 미검증** → 충돌 skill 도 통과 | `validate` 강화(naming·dir-match·conflict) + `list/show` |
| **agent** | 번들 30여 개 **불변·생성 불가**, frontmatter 그래프 무검증, 위임 순환 가능 | 생성 경로 + `validate`(**참조 무결성·위임 순환 Kahn**) + 그래프 시각화 |
| **workflow** | **validator 부재**(순환 미탐지), `exit_criteria` 자유서술 → LLM 자가판정 | `validateWorkflow`(Kahn) + `exit_criteria` 측정가능 + stage `mode`/guardrail |
| **goal** | 엔진 성숙하나 metric **수기 편집**, source/agent **semantic 무검증** | authoring 보조 + **semantic validate** + 동일출력 loop-detection |
| **cron** | `CRONS[]` **하드코딩**, 사용자 생성 경로 없음 | **동적 레지스트리** + `validate`(next-N 미리보기) + CRUD CLI |

**공통:** 그래프 코어(agent·workflow 순환탐지 공유) + 가드레일 코어 + 검증 코어(§9). **엔진·Chief·
실행 방식은 불변** — 위에 검증·관리 레이어만 얹는다.

**핵심 이동:** "돌려보고 터지던 것 → **저장 전에 걸러지는 것**". 우선순위는 **검증 4종(P0)** 먼저,
생성보조·refine 은 그 위에(P1/P2).

> **진입점 — 에셋 채택(§10).** 5개 매니저는 "직접 만든" 자산뿐 아니라 **사용자가 등록한 repo
> (`init`/`repo add`)에서 발견·마이그레이션한** 자산도 관리한다. 채택 파이프라인이 외부
> skill·agent·workflow·cron 을 워크스페이스 taxonomy(team/tier)로 끌어오고, **채택 전 각
> 매니저의 validate 를 통과시킨다**(validate-then-adopt) → 매니저 검증이 채택에서 회수됨.

## 2. 매니저 패턴 (공통 추상)

5개 매니저가 공유하는 동사(verb) 집합. 모든 매니저가 5개를 다 갖출 필요는 없으나, **인터페이스는
동일한 이름**으로 노출한다(학습 비용 최소화).

| 동사 | 의미 | 공통 토대 |
|---|---|---|
| **create** | 의도 → 초안 자산 1-shot/대화형 생성 | 도메인별 author/maker |
| **review** | 생성·수정된 자산을 체크리스트로 LLM 검토(품질·구조·안티-시코펀시) | `skill-pr-review/` 패턴 |
| **validate** | 정적 스키마/규칙 검증(공식 표준 + SoloSquad 확장) | 도메인별 parser |
| **refine** | 측정 → 반성 → **bounded edit** → held-out 재검증 게이팅 | SkillOpt 식 루프(§9) |
| **lifecycle** | list / status / edit / enable·disable / archive | goal CLI 가 레퍼런스 |

> **설계 원칙:** "공유 폴더/싱글톤 매니저"가 아니라 **도메인별 모듈 + 공통 인터페이스**. validate/
> refine 의 *엔진*(bounded-edit 루프, corpus 게이팅, **그래프 검증**)은 §9 의 공유 인프라로
> 추출해 5곳이 재사용. 특히 **참조 무결성·순환 탐지(Kahn)** 는 agent(collaborator 그래프)와
> workflow(depends_on 그래프)가 **같은 코어**를 쓴다.

## 4. skill-manager (v1.3.1 §7 흡수·확장)

`src/bot/skill-author.ts` → `src/bot/skill-manager.ts` 개명, 단일 "저자"에서 전 생애주기로 확장.

- **create** — 기존 `runAuthorLoop`(대화형 5-state) 유지. 표준 description 공식("`<무엇을>` —
  `<구조>`. `<출처>` 기반. 사용 시점: A,B,C. 제외: X"), 3인칭·트리거 첫 문장.
- **review** — `skill-pr-review/` 모듈(frontmatter-diff·keyword-conflict·domain-overlap)을 LLM
  검토 표면으로 승격. 본문 <500줄·참조 1단계·non-goal 유무 소프트 게이트 추가.
- **validate(P0, 가장 싸고 효과 큼)** — `validateSkill` 강화. **미검사 보강:** `name` ≤64자·
  `^[a-z0-9]+(-[a-z0-9]+)*$`·**부모 디렉터리명 일치**·예약어, `description` ≤1024자·3인칭(1인칭
  린트)·"Use when" 트리거절. cross-SKILL name 충돌 + domain 용어 충돌 탐지.
- **refine** — §9 공유 루프 적용. `skill-author`(공식 "Claude A") + `validator-corpus`/`test:corpus`
  (eval 골격)을 재활용해 rollout→reflect→bounded-edit→held-out 검증.
- **lifecycle** — `solosquad skill list/show/validate` 신설(현재 없음).

> P0 = validate 강화(단독 선출고 가능, corpus 호환↑). create/review/refine = P1.

## 5. agent-manager (신설 — actor 레이어, 나머지 4개의 참조 토대)

`src/bot/agents-builder.ts` · `src/engine/agents-md-loader.ts` · `src/util/agent-profile.ts` 를
매니저로 정렬. **자산 = `agents/{main,specialists}/<name>/SKILL.md`(actor 정의, skill 의 SKILL.md 와
구분 — 이건 *역할*이지 *절차*가 아님) + `agent-profile.yaml`(org 단위 thin modifier).**

**두 tier 의 통제권 차이(§6 control-locus 와 직결):**
- **main** (`agents/main/` — chief·pm·engineer·designer·marketer, `tier: leader`) = **오케스트레이션
  actor.** 다음 경로를 *동적으로* 결정(엣지 결정자 = LLM) → **agent 진영 → 가드레일 필수.**
- **specialist** (`agents/specialists/` — architect·backend-engineer 등 20여 개, `tier: member`) =
  **bounded worker.** workflow 노드(`<team>/<agent>`)·goal pipeline stage 에 바인딩되는 실행자.
- **customize 층** — `agent-profile.yaml`: tone/priorities/budget 의 org 수정자. 3-tier 상속
  (workspace→user→org), **budget narrower-only invariant**(자식 cap ≤ 부모, 이미 `agent-profile.ts`
  에 구현·테스트됨) = 자율 오케스트레이션의 1차 가드레일.

**현재 상태:** 번들 30여 개 actor 는 **불변**. `agent-profile.yaml` 로 *수정*은 가능하나
**새 actor 추가 경로 없음**(cron 과 동형 갭). 더 큰 문제 — frontmatter 가 풍부한 **그래프**
(`collaborators`·`skills_used`·`used_by`·`team`·`tier`)인데 **참조 무결성·순환 검증이 전무**하다.
오타난 collaborator, 사라진 skill 참조, 오케스트레이터끼리 서로 위임하는 순환이 런타임까지 안 잡힌다.

- **create** — 신설. 새 specialist/main scaffold(frontmatter + body + `CUSTOMIZATION_GUIDE.md`).
  description 공식·3인칭 트리거는 **skill-manager 와 공유**(§4). tier/team/category·dev_capability
  선택 가이드 + collaborator/skills_used 후보 제안. **번들 30여 개는 불변 유지** — 사용자 actor 는
  org 레이어(`<org>/agents/`)에 둬 upstream 재조정(v1.5.0)과 충돌 회피.
- **review** — 신설. 역할 명확성, **기존 actor 와 책임 중복 탐지**(skill domain-overlap 과 동형 —
  두 specialist 가 같은 일을 하나?), collaborator 그래프 sanity, `dev_capability` 적정성
  (design-only actor 가 코드 쓰기 권한을 요구하지 않나).
- **validate(P0 — 구현 완료, `src/bot/agent-validate.ts` + 공유 `src/util/graph.ts` §9.2).**
  frontmatter 는 그래프다:
  - `collaborators`·`used_by` 가 **실존 actor 참조**(`<team>/<agent>`·`main/<name>`·bare 해소,
    `*` 와일드카드 허용) — 참조 무결성. **(라이브 번들 25개 검증 시 가장 가치 큰 검사 — `*` 미해소
    오인·오타를 잡음.)**
  - `skills_used` 가 **실존 skill 참조**(skill-manager 레지스트리와 교차).
  - **순환·도달성** — workflow `depends_on` 과 **동일한 Kahn O(V+E)** 공유.
    > **결정 1 — 경험적 정정.** 당초 "collaborator 순환 = 무한위임 error" 로 잡았으나, 라이브
    > 번들 실행 시 **`collaborators`/`used_by` 가 위임 트리가 아니라 peer 협업 메시**임이
    > 드러남(architect↔backend-engineer 상호 참조 등 순환 46개). SoloSquad 런타임은 Chief 가
    > specialist 를 spawn(agents-as-tools)하고 specialist 끼리 재귀 spawn 하지 않으므로 이 순환은
    > 무한위임이 아님 → **collaborator 순환은 error 아닌 *요약 경고*(`AGENT_PEER_CYCLES`),
    > self-ref 는 경고.** 무한위임 가드는 collaborators 가 아니라 **런타임 spawn depth** 에 건다.
    > error-grade Kahn 순환검사는 **workflow `depends_on`(진짜 acyclic DAG)** 의 몫.
  - `tier`(leader/member) ↔ bucket(main/specialists) 정합, `name` kebab-case·**dir-match**.
  - 도달성: leader 에서 닿지 않는 **orphan actor** 경고. 협업 depth > cap(5) 경고.
  - (P1) `agent-profile.yaml` **budget narrower-only invariant 사전 표면화**(현재 로드 시 warning 만).
- **refine** — §9 bounded-edit 루프를 actor 의 SKILL.md body(역할 프롬프트) / `agent-profile.yaml`
  tone 에 적용. protected-section 으로 frontmatter 그래프(collaborators 등) 보호 — refine 은 산문만 손댐.
- **lifecycle** — `solosquad agent list/show/validate/enable/disable` 신설 + **org 위임 그래프
  시각화**("누가 누구에게 위임 가능한가" → mermaid/DOT). workflow 의 `<team>/<agent>` 노드 바인딩이
  실존 actor 를 가리키도록 보장하는 **단일 진실원** 역할.

**control-locus 책임 분담:** agent-manager 가 *actor 와 그 위임 위상*을 소유하고, workflow-manager
(§6)는 *그 actor 들을 엮는 정적 경로*를 소유한다. workflow 의 노드 실존성 검증은 agent-manager
레지스트리에 위임 → 중복 제거.

비범위(v1.4+): 런타임 동적 actor 생성(spawn-time), actor 버전 히스토리, per-agent 사용자 정의
메모리 스코프, agent 단위 termination policy.

## 6. workflow-manager

`workflow-maker`/`workflow-refinement` skill + `workflow-resolver`/`workspace-meta` 을 매니저로 정렬.

> **핵심 개선 — "다음 엣지를 누가 결정하나" 를 스키마의 1급 시민으로.** workflow 는 *개발자가
> 경로를 코드로 명시*하는 진영, agent 는 *LLM 이 동적으로 결정*하는 진영이다(Anthropic "Building
> Effective Agents"). SoloSquad 는 이미 하이브리드(Chief=오케스트레이터, 각 stage actor=노드 내부
> 자율)인데 **경계가 스키마에 안 그려져** 있어 어디까지 예측가능하고 어디부터 자율인지 코드로
> 드러나지 않는다. 아래 review/validate 가 이 경계를 명문화한다.

- **create** — 기존 `workflow-maker`(matcher 0.5 임계 → 템플릿 커스터마이즈) 유지.
- **review** — 신설. stage 그래프 sanity(actor ref 는 **§5 agent-manager 레지스트리로 실존성 확인**,
  고아 stage, `handoff_to` 도달성) + **control-locus 명시 검토**(각 stage 가 `mode: fixed` 인지
  `mode: agentic` 인지, agentic 인데 가드레일 없는 stage 경고).
- **validate(P0) — 최대 갭.** `validateWorkflow(workflow.yaml)` 신설:
  - schema_version, stage `id` 유일성, **순환 의존(`depends_on`) 탐지 — §9 그래프 코어(Kahn)**.
  - actor ref(`<team>/<agent>`·`_main/`·`_skill/`) 실존성 → **agent-manager 에 위임**(중복 제거).
  - **`exit_criteria` 를 측정가능 조건으로(measure + operator + threshold).** 자유서술이면 게이트
    통과 여부를 LLM 이 자가판정 → 가장 위험한 모호 엣지. SonarQube quality-gate 구조로 회수해
    "다음으로 넘어가도 되는가"라는 엣지 결정을 **규칙(개발자)** 쪽에 둔다.
  - `hard_gate` 문법, `target_repo` 존재성.
- **스키마 보강 — stage `mode` + agentic 가드레일(P1).** stage 에 `mode: fixed | agentic` 명시:
  - `fixed` = 도구 고정·반복 없음(Prompt Chaining 진영, 예측가능).
  - `agentic` = LLM 이 도구 선택·반복(Orchestrator-Worker 진영) → **`guardrails`(max_iterations·
    budget_usd·loop_detection) 필수**. validate 가 "agentic 인데 guardrails 없음 → reject".
    (loop_detection 은 goal 문서의 동일출력 반복 갭과 인프라 공유.)
- **refine** — 기존 `workflow-refinement`(bottleneck·handoff fidelity 분석)에 §9 루프 결합 —
  분석에서 그치지 말고 stage 재배치/prompt 보정을 bounded-edit 로 제안·게이팅.
- **lifecycle** — 기존 list/show/focus 유지 + `validate` 추가. **`_events.jsonl` 에 정적 전이
  (`static_advance`)와 Chief 동적 재계획(`dynamic_replan`)을 별도 이벤트 클래스로 기록** → 사후에
  "개발자가 의도한 경로 vs LLM 이 즉흥 선택한 경로"를 구분·감사·재구성(Event Sourcing 정합).

비범위(v1.4+): 병렬 stage(async merge), 워크플로 중첩/파라미터화, artifact manifest, Chief 동적
엣지를 그래프에 reify(현재는 별도 레이어 유지 + 로깅으로 재구성).

## 7. goal-manager

`src/cli/goal.ts` + `src/engine/*` 위에 authoring·refine 보조를 얹는다(엔진 루프는 안정, 손대지 않음).

- **create** — 신설 authoring 보조(`goal assist` 또는 skill). 현재는 템플릿 주석 읽고 수기 편집.
  metric/pipeline 섹션 채움 가이드 + 후보 metric 제안.
- **review** — 신설. metric 정의 품질(provenance 명시·direction·threshold 합리성·**composite
  guardrail** 여부 = 단일 metric Goodhart 게이밍 회피) LLM 검토.
- **validate(P1)** — parse-time 위에 **semantic 검증** 추가: `source` 파일 존재성, formula 평가
  가능성, pipeline 의 `<team>/<agent>` 실존 검사(**agent-manager §5 레지스트리에 위임**). (현재
  parser 는 문자열만 수용.)
- **refine** — 신설. `goal verify`(결정성 replay) 위에 "metric 비결정/oscillation 진단 → threshold·
  pipeline bounded-edit 제안" 루프 + **동일 pipeline 출력 반복 loop-detection**(discard-streak 와
  별개, workflow agentic 가드레일과 인프라 공유). **단, 실행 중 goal 의 mid-run edit 은 다음 사이클
  재파싱으로 한정.**
- **lifecycle** — 이미 성숙(new/list/show/run/status/stop/verify/queue/active/next). `goal estimate`
  (예상 비용) + `goal measure <metric>`(단일 metric 디버그) 추가.

비범위: goal 별 사용자 정의 termination policy(stop-hook 은 현재 SKILL 전용), goal 버전 히스토리,
진화/유전 탐색(local-optima 탈출 — 비용·복잡도↑).

## 8. cron-manager (가장 미성숙 — 사용자 생성 경로 자체가 없음)

현재 `CRONS[]` 하드코딩 + cron 시간만 `workspace.yaml.briefings` 로 설정 가능. 사용자가 커스텀
cron 을 만들려면 코드 수정·재배포 필요.

- **create(P0) — 핵심.** **동적 CRONS 레지스트리**로 전환. `crons/` 디렉터리(또는
  `.solosquad/crons.yaml` manifest)에서 cron 정의(id·cron·kind·channel·memoryTargets·
  overlap·jitter·catchup)를 로드 → 하드코딩 배열 제거. `loadCronPrompt`(이미 `getCronsDir`
  배선, v1.3.1 §9)는 그대로 재사용. node-cron v4 의 `noOverlap`·`maxRandomDelay`(지터) 활용.
- **review** — 신설. cron 식 유효성·중복 시간·프롬프트 기본 구조 검토.
- **validate(P0)** — cron 표현식 파싱(필드 수 = node-cron 5/6 초-우선 **문서화**), kind enum,
  channel 존재, 프롬프트 파일 존재성. 저장 전 **next-N 실행 미리보기 + cronstrue 영어 readback** +
  최소 간격 가드. timezone 은 **IANA 이름**(offset 금지)·워크스페이스 tz 상속.
- **refine** — 신설(경량). cron-logs 기반 "실행 실패/저효용 cron 탐지" + 제안.
- **lifecycle** — `solosquad cron create/list/edit/enable/disable/delete` 신설(**pause ≠ delete**).
  현재는 `cron run`(수동 실행)·enable 플래그만. catch-up 기본 off + cron 프롬프트 idempotent
  전제 + **조용한 미스 경보(dead-man's-switch)**.
- **orphan 3개 처리** — `trace-rotate` 는 `system-housekeeping` 에 흡수됨(삭제 후보). `bot-health-check`·
  `leading-indicator` 는 관측 레이어(세션 추적·budget 질의) 의존 → v1.4.0 까지 추적 유지 또는
  동적 레지스트리 위에서 opt-in 배선.

> **구현 현황 보정 (v1.3.3 기준 — 위 §8 은 설계 목표이며 일부는 미배선):**
> - ✅ **완료:** 동적 CRONS 레지스트리, create/validate/lifecycle(new·edit·enable·disable·delete),
>   `noOverlap`(중첩 방지), dead-man's-switch(조용한 미스 경보), chokidar 무중단 반영, catch-up 부재(=기본 off).
> - ❌ **미구현(이월):**
>   - **지터(`maxRandomDelay`)** — 미배선. 빌트인 4종 + 유저 brief 가 정시 동시 발화(thundering herd).
>   - **저장 전 미리보기** — 실제는 **next-1**(`nextRun`) + **자체 `describeSchedule`** 수준. 레퍼런스가 권하는
>     **next-N 미리보기 · cronstrue 영어 readback 아님** (cronstrue 미도입).
>   - **IANA timezone 검증** — 부재. tz 문자열을 node-cron 에 그대로 전달(오타 시 런타임 throw/무음).
>   - **다운타임 catch-up · DST 위험창(00–03시) 경고 · 5분 미만 최소 간격 가드** — 없음(every-minute 만 경고).
>
> 기능 구현은 **보류**. cron 목표·매니저 역할 재정의 후 별도 기획에서 다룬다.

## 10. 에셋 채택 (asset adoption — 외부 repo → 워크스페이스 마이그레이션)

**배경.** 사용자가 repo 를 등록(`init` 또는 `repo add`)하면 그 repo 의 기존 AI 에셋
(skill·agent·workflow·cron·prompt)을 **발견 → SoloSquad 워크스페이스의 역량·에이전트 팀으로
마이그레이션**한다. **prior art:** `solosquad analyze repo`(v0.5 §6, `src/analyze/`
scanner·classifier·ledger·applier)가 이미 **skill 한정** "scan → diff → classify → merge →
report" 파이프라인을 보유(ledger 로 재실행 멱등). 본 절은 이를 **5개 자산 + 자동 트리거 +
taxonomy 매핑**으로 일반화한다(새 엔진이 아니라 확장).

**레퍼런스 수렴 5단계**(Claude plugin/marketplace · Backstage 카탈로그 · Terraform import ·
kubectl SSA · Nx Crystal · AGENTS.md 조사 — `docs/ideation/` 후속 또는 trend-record 로 분리 가능):

- **10.1 Discover** — `scanRepoSkills` → **멀티에셋 스캐너**. 신호 = 디렉터리(`.claude/skills`→skill,
  `.claude/agents`·CrewAI·AGENTS.md→agent, `workflow.yaml`→workflow, cron·`.github/workflows`·
  `crons/`→cron) + frontmatter + 파일명 규약(`.instructions.md`·`.mdc`). 모노레포 중첩
  스캔(Claude 중첩 `.claude/` 선례).
- **10.2 Identify / Dedup** — 기존 ledger(내용 diff·멱등) 확장: **SHA content hash + locationKey**
  (출처 repo+경로 고정, Backstage 모델). 재실행 = 멱등(2번째 0 LLM).
- **10.3 Map → taxonomy** — `classifier.ts` 재사용. 외부 agent → SoloSquad **team/tier 매핑**:
  **heuristic-first(디렉터리/frontmatter 규칙) + LLM fallback + 미지→default 팀 + review 플래그**
  (ETL default-fallback 패턴). skill→skill, cron→cron, workflow→workflow.yaml.
- **10.4 Conflict 해소** — **namespacing 우선**: 외부 에셋은 출처 repo 로 prefix → **번들 30여 개
  actor 를 덮어쓰지 않음**(§5 불변 원칙 정합, Claude `apps/web:deploy` 선례). 다음 순위 =
  closest-wins override > locationKey 일치 시 덮어쓰기 > SHA dedup.
- **10.5 Adopt UX (결정: dry-run → confirm)** — `init`/`repo add` 시 **dry-run 리포트**(무엇이
  들어오고 무엇이 충돌하는지 + 컨텍스트 비용; Terraform `plan`·Claude "Will install" 패턴) →
  **사용자 confirm** → **org 레이어**(`<org>/agents/`·skills·workflows·crons)에 기록(v1.5.0 정합).
  선택적 채택 + 멱등 재싱크 + orphan 처리(`--prune-orphans`, Backstage orphan 마킹).
- **10.6 Validate-then-adopt 게이트** — 채택 *전* §4–8 validator 통과 강제(agent 참조 무결성·위임
  순환 §5, workflow 순환 §6, skill naming §4). 통과 못 하면 채택 보류 + 리포트에 사유. **검증
  매니저와 채택이 한 흐름** — 5개 매니저 작업이 채택 파이프라인에서 회수됨.

> **구현 상태(2026-06-19 갱신).** ✅ 10.1 Discover(`src/analyze/asset-scanner.ts`) · ✅ 10.5 dry-run
> (`adoption-report.ts` + `solosquad adopt <repo>`) · ✅ 10.6 validate-then-adopt(P0 validator
> 회수) · ✅ 10.3 Map heuristic(`agent-map.ts`) **+ ✅ LLM fallback**(`mapAgentTeam` —
> heuristic 이 `default` 로 떨어진 모호 actor 만 escalate, injectable `AgentTeamCaller`,
> dry-run 은 호출 0; CLI `solosquad adopt <repo> --classify`).
>
> **`--apply`(실제 쓰기) — v1.5 불필요, additive 로 안전(정정 2026-06-18).** 초기 우려(override
> replace 로 번들이 가려짐)는 **틀렸다**: `init` 이 번들을 `.solosquad/{agents,skills,crons}`
> 로 통째 복사하므로(merge-don't-clobber, `init.ts` §v1.1 bundle block), init 된 워크스페이스의
> `.solosquad/agents` 엔 25개가 이미 실재한다 → 채택분을 거기 **추가**하는 건 순수 additive, 손실
> 없음. **채택 = 새 id 파일 추가**(충돌 시 namespace)이므로 v1.5.0 의 *동일 파일 3-way edit-merge*
> 문제와 무관하다. 유일 가드: **init 된 워크스페이스(.solosquad 존재)에만 쓰기**, 번들(설치 패키지)
> 엔 쓰지 않음. → ✅ `solosquad adopt <repo> --apply`(skill·agent·cron·**workflow** additive,
> namespace 충돌해소, 멱등 no-clobber) + ✅ repo-add **및 `init`** 직후 채택 가능 에셋 안내(§10.5 hook,

> 라이브러리 확장, org 스코프 아님). **§10 잔여 없음.**
>
> **번들 스코프 결정성(footgun 정정 2026-06-19).** 채택 roster·`agent/workflow validate` 가
> `loadAgentSpecs()`(=cwd 에서 위로 walk 하는 `getAgentsDir()`)에 의존 → 패키지 체크아웃이 *상위
> 워크스페이스 안*에 있으면(개발 머신, repo 를 워크스페이스에 클론한 사용자) 그 상위 워크스페이스의
> (낡았을 수 있는) agents 를 검증하던 버그. **정정:** 번들 스코프는 `getBundledAgentsDir()`/
> `getBundledSkillsDir()`(= `getBundleRoot()` 기준, **cwd 독립**)로 결정적으로 해석. `buildAdoptionReport`
> 는 `AdoptionScopeOpts` 주입도 허용(테스트 hermetic).
>
> **`--apply` 쓰기 타깃 일관화(2026-06-19).** `--apply`/init·add-repo 오퍼의 가드는 `getWorkspaceRoot()/.solosquad`
> 를 보는데, 쓰기 타깃을 `getAgentsDir()`/`getSkillsDir()`/`getCronsDir()` 로 잡으면 각 resolver 가
> 해당 override 부재 시 **번들로 폴백**해 타깃이 갈라질 수 있었다(예: `.solosquad/skills` 없는 워크스페이스 →
> 채택 skill/workflow 가 설치 패키지로). **정정:** 쓰기 타깃을 가드와 동일하게 `<ws>/.solosquad/{agents,skills,
> crons}` 로 **고정** → 항상 워크스페이스에만 쓰고 번들 오염 불가.

**CLI:** `solosquad adopt <repo>`(명시 실행) + `init`/`repo add` 의 자동 dry-run hook.

**v1.5.0 경계:** 본 절 = **import(최초 편입)**. 이후 upstream 개선 전파·재조정은 v1.5.0
(`customization-upstream-reconciliation`)이 담당 — **import vs propagation 분리.**

**비범위(v1.4+):** 비-Claude 포맷 심층 변환(LangGraph/AutoGen 코드 정의 → `SKILL.md`), 자동 충돌
머지(현재는 namespacing 격리만), 양방향 sync.
