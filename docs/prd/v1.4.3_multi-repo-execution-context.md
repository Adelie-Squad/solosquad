## 1. Summary

`resolveExecutionContext(org, targetSpec?) → { cwd, addDirs, repoManifest, scope }` 단일 리졸버를 신설하고, 현재 각자 다른 방식으로 실행 컨텍스트를 구하는 **네 경로(chief·goal·cron·workflow)** 가 전부 이걸 호출하게 배선한다. 부수적으로 단일 repo 지정(`target_repo: string|null`)을 **복수 지정(`target_repos: string[]`, 생략=전체 repo)** 으로 확장한다. **새 아키텍처가 아니라 배선** — 세션 모델·engine·spawn 동작은 불변, hot-path 신설 없음(비파괴·저위험).

## 2. Background — 실행 컨텍스트가 경로마다 제각각 (코드 근거)

### 2.1 현재 상태 (2026-07-03, 코드 실측)

실행 컨텍스트(어느 repo 에서·어떤 디렉토리 접근권으로 도는가)를 구하는 로직이 **세 곳에 분산**돼 있고, 경로마다 획득 능력이 다르다:

| 경로 | repo cwd 획득 | `--add-dir`(다중 repo 읽기) | repo 매니페스트(로스터) |
|---|---|---|---|
| **chief 대화** | 고정 org-root + 프롬프트 내 `@slug` | `collectRegisteredRepoPaths`(`chief-runner.ts:148`) → 전 repo | 단일-repo Layer 만(spawn-assembler) |
| **cron** | `resolveOrgCwd`(`workflow-resolver.ts:117`) — S-1 로 외부경로 해석됨 | **없음** | **없음** |
| **goal** | org-root 맨(`target_repo` 단일, 미해소) | **없음** | **없음** |
| **workflow stage** | stage `target_repo`(단일, `program-parser.ts`) | **없음** | **없음** |

**핵심 함수(현재):**
- `resolveOrgCwd(orgDir) → { cwd, reason, workflowId?, repoSlug? }` — `src/bot/workflow-resolver.ts:117`. **단일 cwd 만** 반환(addDirs·manifest 없음). cron(`src/cron/index.ts:80`)만 사용.
- `collectRegisteredRepoPaths(orgCwd) → string[]` — `src/bot/chief-runner.ts:148`. `--add-dir` 목록. **Chief 전용.**
- `resolveRepoCwd(orgSlug, repoSlug, workspace)` — `src/util/paths.ts:334`. 외부경로(`repositories/<slug>.yaml` 의 `path:`) 해석. **재사용 대상**(위 둘의 공통 하위 블록).

### 2.2 문제 (공백)

- **능력 편재:** "멀티 repo·멀티 자산 지향" 인데 그 능력이 **chief 한 경로에만** 온전하다. goal·cron·workflow 는 repo 도, 다중 repo 읽기(`--add-dir`)도, org 로스터도 못 받는다.
- **중복·표류:** `resolveOrgCwd`(cwd) 와 `collectRegisteredRepoPaths`(add-dir)가 **같은 `repositories/*.yaml` 을 서로 다르게 읽는다** → 한쪽 수정이 다른 쪽에 안 전파. S-1 이 cron cwd 만 고쳤을 때 goal/workflow 는 여전히 blind 였던 게 그 증거.
- **단일 repo 가정:** `target_repo: string|null` 은 "한 repo 아니면 전체" 만 표현. "이 두 repo 만" 을 못 적는다(멀티 repo 작업의 자연스러운 단위).
- **S-4 차단:** per-repo 워커 세션(v1.4.4)은 "워커가 어느 컨텍스트에서 스폰되나" 를 이 리졸버에서 받아야 한다. 지금은 그 단일 진입점이 없다.

## 3. 핵심 개념 — `resolveExecutionContext`

```
resolveExecutionContext(org: OrgRef, targetSpec?: string[] | null) → ExecutionContext

interface ExecutionContext {
  cwd: string;            // 지정 1개→그 repo, 그 외→org-root
  addDirs: string[];      // scope 내 전 repo 절대경로 (--add-dir 용)
  repoManifest: string;   // org 로스터 1블록 (slug·경로·role) — 프롬프트 주입용
  scope: {
    repos: string[];      // 해소된 repo slug 목록
    reason: "explicit" | "all-repos" | "workflow-stage" | "legacy-root";
  };
}
```

- **입력 규칙 — "비우면 전체, 적으면 그것들"(§10.3):** `targetSpec` 생략/빈 배열 → org 전 repo. 값 있으면 그 repo 들로 좁힘(1개면 `cwd`=그 repo).
- **하위 블록 재사용:** repo slug → 절대경로 변환은 기존 `resolveRepoCwd`(S-1) 경유. cron 의 org-root 추락·goal 의 add-dir 누락이 **동시에** 사라진다.
- **매니페스트 주입(§10.2):** `repositories/*.yaml` → "이 org 엔 a·b·c 가 있고 각 경로·role" 한 블록. 현재 spawn-assembler 의 단일-repo Layer 를 "org 전체 로스터" 로 일반화. 코드는 scope 로 좁히되 **매니페스트(구조 맵)는 전체** 주입(GitHub AGENTS.md·Google repo manifest 동형).

## 4. 접근 옵션 (≥2)

### A. 신규 통합 리졸버 + 4경로 배선 (권고)
`src/bot/execution-context.ts` 신설로 `resolveExecutionContext` 를 두고, `resolveOrgCwd`·`collectRegisteredRepoPaths` 를 그 내부 호출로 **재구현**(기존 export 는 얇은 어댑터로 유지 → 호출부 점진 이관). 4경로가 새 리졸버를 호출.
- **장점:** 단일 진입점, S-4 가 바로 얹힘, 중복 소멸. 매니페스트/scope 를 한 곳에서 관리.
- **단점:** 신규 모듈 + 4경로 배선으로 diff 가 넓음. 어댑터 유지 기간 동안 2중 경로 잠시 공존.

### B. `resolveOrgCwd` 를 확장해 그대로 사용
기존 `resolveOrgCwd` 의 반환 타입에 `addDirs`·`repoManifest` 를 얹고 goal/workflow 도 이걸 부르게.
- **장점:** 신규 모듈 없음, diff 작음.
- **단점:** 이름(`OrgCwd`)이 "cwd" 에 묶여 매니페스트/scope 의미가 안 맞음. `collectRegisteredRepoPaths`(chief add-dir)와 여전히 2원화 — 중복 근본 해소 실패. 확장할수록 함수 책임이 뒤엉킴.

**선택 = A.** S-6 의 목적이 "3분산 → 1통합" 이라 B 는 중복(chief add-dir)을 남겨 목적 미달. A 로 하되 기존 두 함수는 **호출 위임 어댑터**로 남겨 회귀 위험을 낮춘다(빅뱅 교체 아님).

## 5. 스코프 (S-6)

**In:**
1. `resolveExecutionContext` 신설(`src/bot/execution-context.ts`) — cwd·addDirs·repoManifest·scope.
2. `resolveOrgCwd`·`collectRegisteredRepoPaths` 를 새 리졸버 위임 어댑터로 재구현(외부 시그니처 보존).
3. **4경로 배선:** cron(`src/cron/index.ts`) · chief(`src/bot/chief-runner.ts`) · goal(engine goal 스폰 경로) · workflow(stage 스폰 경로)가 컨텍스트를 새 리졸버에서 획득.
4. **`target_repo`(단일) → `target_repos`(배열, 생략=전체) 확장** — `goal-parser.ts`·`program-parser.ts`·`WorkflowStage` 인터페이스. cron 정의(`crons/<id>.yaml`)에 `repos: [a,b]` 선택 필드.
5. 매니페스트 블록 주입 — spawn-assembler 의 단일-repo Layer 를 org 로스터로 일반화.

**Out(비범위, §9):** S-7(cron 이 skill·agent·workflow 활용) · S-4(per-repo 워커 세션·works 자식 세션·goal 중간개입) · S-5(하이브리드 역할분리) · 세션 회전/GC(S-2b/S-3b).

## 7. `target_repo` → `target_repos` 마이그레이션 · 하위호환

- **파서 하위호환:** frontmatter 에 `target_repo`(구, string|null)만 있으면 `target_repos=[그것]`(null→[]=전체)로 승격 파싱. `target_repos`(신)가 있으면 우선. **양쪽 다 있으면** 검증 오류(모호).
- **데이터 마이그레이션 `1.4.2-to-1.4.3`:** 기존 goal `.md`·workflow `_status.yaml` 의 `target_repo` 키를 `target_repos` 배열로 정규화(no-op 성격 — 파서가 이미 하위호환이라 강제 재작성은 옵션). 세션 리셋 없음(spawn 동작 불변 → `archiveOrgChiefSessions` 호출 안 함).
- **immutable 규칙 준수:** 발행분 마이그레이션 미편집(fix-forward), 신규 `1.4.2-to-1.4.3` 만 추가.

## 10. 검증 (Given-When-Then AC — R8)

- `npx tsc --noEmit` · `npm run build` · `npm test` · `solosquad validate` green.
- **AC-1 (파리티):** *Given* 외부경로 repo 2개 등록 org, *When* chief·goal·cron·workflow 각 경로가 컨텍스트를 해소, *Then* 넷 다 동일한 `cwd`·`addDirs`(전 repo 절대경로)·`repoManifest` 를 받는다(경로별 편차 0).
- **AC-2 ("비우면 전체"):** *Given* `target_repos` 미지정, *When* 해소, *Then* `scope.reason="all-repos"` + `addDirs`=org 전 repo.
- **AC-3 (좁힘):** *Given* `target_repos: [b]`, *When* 해소, *Then* `cwd`=b 의 실경로, `scope.repos=["b"]`.
- **AC-4 (외부경로·S-1 회귀):** *Given* `repositories/b.yaml` 의 외부 `path:`, *When* 해소, *Then* org-root 추락 없이 실경로 반환(S-1 무회귀).
- **AC-5 (하위호환):** *Given* 구 `target_repo: "a"` 만 있는 goal, *When* 파싱, *Then* `target_repos=["a"]` 승격 + 경고 없음. `target_repo`+`target_repos` 동시 → 검증 오류.
