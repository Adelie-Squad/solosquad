# v1.3.9 — 1.3.2→1.3.3 마이그레이션 충돌 핫픽스 (hotfix on v1.3.8)

> **상태:** 출시 준비 (2026-06-25). v1.3.8(`docs` 스킬·문서 관리 체계, tag push·npm 미게시) 직후
> dogfood 업그레이드에서 노출된 **기존 마이그레이션 결함** 핫픽스. 본 PRD 는 `prd` 스킬 **핫픽스
> 양식**(증상→근본원인→영향→해결책→복구절차→회고)을 따른다(선례 `v1.2.3-bundle-files-hotfix.md`).
>
> **로드맵 위치:** v1.3.8 (docs 관리, npm 미게시) → **v1.3.9 (본 hotfix — 마이그레이션 결함 + 버전
> 3자리 모델 정정)** → 다음. v1.3.8 은 npm 으로 안 나가고 1.3.9 가 이 배치의 첫 게시분.

## 1. 발견 경위

dogfood 사용자가 실 워크스페이스 `C:\Dev\bv-ai-native-po` 에서 `solosquad migrate --apply`
(1.2.9 → 1.3.8) 실행. 플랜은 1.3.7→1.3.8 시드 포함 전 단계 정상 표시. 적용 중 **[2/7] = 1.3.2 →
1.3.3 verify 실패**:

```
[2/7] Applying 1.3.2 → 1.3.3 ...
✗ Migration failed: Verify failed at 1.3.3: legacy cron dirs still present:
    .solosquad/routines → .solosquad/crons
  Run: solosquad migrate --rollback
```

워크스페이스는 partial 상태(1.3.2 적용 후 1.3.3 apply 가 schedules→crons 는 끝냈으나 routines 잔존)
로 고립. step 1(1.2.9→1.3.2)은 적용·검증 완료.

## 2. 근본 원인

`src/migrations/scripts/1.3.2-to-1.3.3.ts` 의 `moveDir` 가 **한 단계 깊이만** 병합. v1.3.3 은
`.solosquad/{schedules,routines}` 두 레거시 디렉토리를 **같은 `crons/`** 로 접는다:

1. `moveDir(schedules, crons)` — crons 부재 → 통째 rename. crons 가 schedules 내용으로 채워짐.
2. `moveDir(routines, crons)` — crons 존재 → 항목별 병합. 그런데 routines 의 항목이 crons 에 **동명
   으로 이미 있으면**(파일 또는 하위 디렉토리) 그 항목을 **routines 에 그대로 남김**. routines 가
   비지 않아 삭제 안 됨 → verify 의 "legacy dir 잔존" 검사 실패.

이 워크스페이스는 schedules·routines 에 **동명 항목**이 있어 드러났다(둘 다 가진 워크스페이스는
v1.0.x→v1.1 override 이력이 겹친 경우 발생).

## 3. 영향

- `.solosquad/schedules` 와 `.solosquad/routines` 를 **둘 다** 가지며 **동명 항목**이 있는 워크스페이스
  = **결정적 verify 실패** → 1.2.9~1.3.2 어디서 올라오든 1.3.3 에서 chain 고립.
- 한쪽만 있거나 동명 충돌이 없으면 정상(그래서 기존 테스트·대부분 워크스페이스는 통과 — sleeping bug).
- 부수: 1.3.2 이후 ~1.3.8 **나머지 마이그레이션 재검토 결과** 1.3.2→1.3.3 만 블로킹. 1.3.4→1.3.5 도
  데이터 이동이 있으나 verify 가 충돌 관용적(dest 존재면 OK)이라 크래시 없음. 1.3.3/1.3.5/1.3.6→ 는
  no-op 범프. 1.3.7→1.3.8 은 멱등 시드.

## 4. 해결책

### 4.1 즉시 (코드 수정)

`moveDir` 를 **재귀 병합 + 충돌 처리**로:
- 동명 하위 디렉토리 → **재귀 병합**(orphan 방지).
- 리프(파일) 충돌 → **더 새 override(`dst`=schedules)가 이김**, 밀려난 `src`(routines) 중복은 드롭.
  마이그레이션 프레임워크가 **전체 백업**을 먼저 뜨므로 드롭분은 백업에 보존.
- 결과: `src` 가 항상 비워져 제거됨 → verify 통과(동명 공존에도).

손댈: `src/migrations/scripts/1.3.2-to-1.3.3.ts`(`moveDir` + 헤더 주석).

### 4.2 버전 3자리 모델 정정 (부가 — 문서/프로세스)

v1.3.8 작업 중 핫픽스를 `1.3.8.1`(4자리)로 표기하고 "핫픽스=부모 PRD §Hotfix 섹션"으로 규정한 것은
**오류**(메모리 `feedback_npm_version_format` = 항상 3자리 위반). 정정: **버전 항상 3자리; 핫픽스 =
다음 patch(1.3.8→1.3.9) + 자체 핫픽스-양식 PRD**(PRD↔버전 1:1 유지, 선례 v1.2.3). 반영:
`prd` 스킬(R1 3자리 규칙 + 핫픽스 PRD 양식 신설) · `docs` 스킬 §3 · v1.3.8 PRD §3.4/OQ#1 ·
release docs(architecture·roadmap·CHANGELOG·README)의 `1.3.8.1`·"부모 PRD §Hotfix" 제거.

### 4.3 사용자 복구 절차

```
solosquad migrate --rollback      # 백업에서 깨끗한 1.2.9 복원
solosquad migrate --apply         # 수정된 CLI 로 1.2.9 → 1.3.9 재실행 (이제 통과)
```
> 동명 충돌 시 더 새 `schedules` 정의 채택, 옛 `routines` 중복은 백업에 보존. 무엇이 충돌했는지
> 보고 싶으면 재실행 전 백업의 `.solosquad/routines/` 확인.

## 6. 회고 — 왜 못 잡았나

- `test/migration-v1.3.3-cron.test.ts` 가 **schedules 단독 + routine-logs** 만 픽스처로 썼다 —
  schedules·routines **공존 + 동명 충돌** 케이스가 없어 한-단계-병합의 결함이 잠복(sleeping bug).
- 후속 가드 후보: 데이터-이동 마이그레이션은 (a) 다중 소스→단일 타깃 (b) 동명 충돌 (c) 하위 디렉토리
  충돌 픽스처를 **표준 테스트 매트릭스**로. `moveDir` 류 헬퍼를 공용화해 1.3.4→1.3.5 의 orphan-on-
  subdir-collision(현재 비블로킹)도 같은 견고성으로.
