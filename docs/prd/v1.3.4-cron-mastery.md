## 3. 개선 필요 항목 (v1.3.4 스코프)

### 3.1 §A — 지터 (thundering herd 방지)

**문제:** 빌트인 4종 + 모든 opt-in 유저 brief 가 "정시"(예: 09:00:00)에 동시 발화한다.
유저가 늘수록 같은 초에 N개의 모델 호출이 몰려 부하·rate-limit·로그 폭주를 유발.

**설계:**
- `CronDef` 에 `maxRandomDelay?: string`(예: `"90s"`, `"5m"`) 추가 — 발화 시점에 `0..max`
  사이 랜덤 지연 후 실행. 미설정 시 0(현행 동작 유지).
- 빌트인 brief 에는 **기본 지터** 적용(예: 유저 brief 는 0–120s). `system-housekeeping`·
  `pm-compaction` 은 결정적 유지(지터 0).
- 구현 위치: `runCronForProduct` 진입 직전 지연. one-shot(`at`)은 지터 미적용(정밀 발화가
  목적).
- `cron-validate`: `maxRandomDelay` 가 cron 간격의 절반을 넘으면 `CRON_JITTER_TOO_LARGE`(warn).
- 결정성 주의: 스크립트/테스트에서 `Math.random()` 회피 규칙이 있으므로(harness), 지터
  계산은 런타임 데몬 경로에 한정하고 단위 테스트는 주입식 RNG 로 검증.

### 3.2 §B — 저장 전 미리보기 (next-N + 사람 말 readback)

**현재:** `cron show`/`new` 가 next-1(`nextRun`) + 자체 휴리스틱 `describeSchedule()` 만 제공.

**설계:**
- **next-N 미리보기:** `nextRuns(expr, tz, n=5)` 신설 — 향후 N회 발화 시각을 유저 tz 라벨과
  함께 출력. `cron new`/`edit` 저장 직전 + `cron show` 에 표시.
  ```
  $ solosquad cron new daily-standup --cron "0 9 * * 1-5"
  스케줄: 평일 매일 오전 9시 (Asia/Seoul)
  다음 5회:
    1. 2026-06-22 (월) 09:00 KST  — in 18h
    2. 2026-06-23 (화) 09:00 KST
    ...
  저장할까요? [Y/n]
  ```
- **사람 말 readback:** `cronstrue`(i18n, 한국어 로캘 지원) 도입을 검토. 단 의존성 추가
  비용 대비, 우선 `describeSchedule()` 의 한국어 커버리지를 넓히는 자체 구현을 1순위로
  하고 cronstrue 는 fallback/영어 옵션으로 둔다. (결정: §6 오픈 이슈)

### 3.3 §C — IANA timezone 검증 + 선택 UI (사용자 요청 핵심)

> **사용자 질문:** "오타 못 내게 보기를 주고 사용자가 선택하게 가능?" → **가능. 일부는 이미 구현됨.**

**현황:** `cli/init.ts` 에 이미
- `TIMEZONE_PRESETS`(5종 + "Other — type IANA string") inquirer 리스트 선택,
- `isValidIanaTimezone()`(`new Intl.DateTimeFormat({timeZone})` throw 검사)
가 **존재**하나 **`init.ts` 로컬·export 안 됨**. cron CLI·유저 개인화·대화형 표면에는
미적용 → tz 오타가 node-cron 까지 흘러가 런타임 throw 또는 무음.

또한 런타임 검증: `Intl.supportedValuesOf('timeZone')` 가 Node 25 에서 **418개 IANA 존**
전체 목록을 반환(확인 완료) → 자동완성·"did you mean" 퍼지 제안의 데이터 소스로 사용 가능.

**설계 — 3개 표면에 일관 적용:**

1. **공용 util 추출:** `src/util/timezone.ts` 신설 →
   `TIMEZONE_PRESETS`, `isValidIanaTimezone()`, 신규 `suggestTimezone(input)`(레벤슈타인/
   대소문자·언더스코어 정규화로 418개 중 최근접 제안), `allTimezones()`. `init.ts` 는 이걸
   재사용(중복 제거).

2. **CLI 표면 (오타 불가):**
   - `cron new`/`edit` 에서 tz 를 받을 때(또는 유저 개인화 설정 CLI) **inquirer 리스트
     선택** — init 과 동일 UX. "Other" 선택 시에만 자유 입력 + `isValidIanaTimezone()`
     검증 + 실패 시 `suggestTimezone()` 제안.
   - 비대화형(`--timezone` 플래그/CI)에서는 즉시 검증, invalid 면 제안과 함께 종료.

3. **대화형 표면 (Chief, Discord/Slack):** 네이티브 드롭다운은 없지만 동일 효과를:
   - 유저가 "내 타임존 뉴욕으로" 류 발화 시 Chief 가 **번호 매긴 단답 보기** 제시
     ("1) America/New_York  2) America/Los_Angeles … 번호로 답해줘") 후 선택 매핑.
   - 자유 입력 tz 는 `isValidIanaTimezone()` 통과 시에만 `users/<handle>.yaml.timezone`
     에 기록. 실패 시 `suggestTimezone()` 으로 "혹시 `Asia/Seoul`?" 되묻기.
   - 이 로직은 chief SKILL 프롬프트 + 검증 헬퍼(CLI 노출)로 배선.

4. **저장 시점 검증(라스트 가드):** `cron-validate` 에 `CRON_TZ_INVALID`(error) 추가 —
   `CronDef.timezone`(신규 옵션 필드) 또는 user yaml tz 가 비IANA면 등록 거부 + 제안.
   `user-crons.resolveUserCrons` 도 invalid tz 유저는 skip + 경고 로그(현재는 그대로
   node-cron 에 전달).

> **답변 요약:** 네 — CLI 에선 `inquirer` 리스트로 "보기 선택"이 init 에 이미 있고 cron
> 표면으로 확장만 하면 됩니다. 대화형(메신저)에선 드롭다운이 없으니 **번호 보기 +
> IANA 검증 + 퍼지 제안**으로 같은 "오타 불가" 경험을 만듭니다.

### 3.4 §D — 운영 안전장치 3종

- **다운타임 catch-up (opt-in):** 데몬 재기동 시, `cron-runs.jsonl` 의 마지막 성공 이후
  놓친 발화가 있고 `CronDef.catchUp: true` 면 **1회만** 만회 실행(중복 폭발 방지 위해
  N회 누락도 1회로 합침). 기본 off(현행 유지).
- **DST 위험창 경고:** `cron-validate` 에서 발화 시각이 DST 전환 위험대(통상 로컬
  00:00–03:00)에 걸리면 `CRON_DST_WINDOW`(warn) — "이 시각은 서머타임 전환 시 건너뛰거나
  중복될 수 있음". 결정성 중요한 잡은 02시 회피 안내.
- **최소 간격 가드 확장:** 현재 `* * * * *`(매분)만 `CRON_TOO_FREQUENT`. 이를 **5분 미만**
  전체로 확장(`*/2`, `*/3`, `*/4` 등) — estimatePeriodMinutes < 5 → warn. (하드 차단 아님,
  의도적 고빈도 허용하되 경고.)

### 3.5 검증 에러코드 변화 요약

| 코드 | 신규/기존 | 심각도 | 트리거 |
|---|---|---|---|
| `CRON_TZ_INVALID` | 신규 | error | timezone 이 비IANA |
| `CRON_JITTER_TOO_LARGE` | 신규 | warn | `maxRandomDelay` > 간격/2 |
| `CRON_DST_WINDOW` | 신규 | warn | 발화 시각이 00–03시 DST 위험창 |
| `CRON_TOO_FREQUENT` | 확장 | warn | 5분 미만 간격(기존: 매분만) |

### 3.6 §E — 대화형 운영 + 신뢰성 (사용자 요청)

> v1.3.3 의 현황을 코드로 확인한 결과: ① 등록 전 **확인 없음**, ② 실패 시 **메신저 알림
> 없음**(`index.ts:124-133` 은 jsonl 기록 + `console.error` 후 `return`), ③ 대화형은
> **등록(create)만** 지원(나머지 CRUD 는 CLI 전용), ④ 메신저에서 **실행/실패 조회 수단
> 없음**. v1.3.4 는 이 격차를 닫는다. (결정 2026-06-21)

**§E1 — 등록/수정/삭제 전 확인 (대화형 + CLI 모두).**
- **대화형:** Chief 가 cron 을 만들기/바꾸기/지우기 전에 **요약 + §B next-N 미리보기**를
  제시하고 "등록할까요? (예/아니오)" 되묻는다. 사용자 확정 후에만 파일을 쓴다. chief
  SKILL 의 신규 "Cron 운영" 섹션에 명문화(현재는 분류 즉시 등록).
- **CLI:** `cron new`/`edit`/`delete` 에 확인 프롬프트 추가, `-y/--yes` 로 생략(현 `delete`
  의 `--yes` 패턴을 new/edit 로 확장). CI/스크립트는 `--yes` 필수 → 비대화형 결정성 유지.

**§E2 — 실패 시 메신저 알림 (해당 cron 채널/스레드).**
- `runCronForProduct` 의 error 경로(`status:"error"`)에서 **그 cron 의 채널/스레드**
  (개인화면 `works-<handle>`, background 면 자기 `threadName`)로 실패 메시지를 게시한다.
  형식: `⚠️ [<name>] 실행 실패 — <error 요약> (<시각>)`. 게시 실패해도 jsonl 기록은 유지.
- **노이즈 가드:** 연속 실패 시 매번 도배하지 않도록, 직전 run 도 error 면 N회마다 1회만
  게시(또는 "연속 K회 실패" 1줄로 합침). dead-man's-switch(누락 경보)와 역할 분리 —
  E2 는 "방금 실패", switch 는 "한참 안 돎".

**§E3 — 대화형 전체 CRUD.**
- chief SKILL 에 list/show/edit/enable/disable/delete/runs 자연어 인텐트를 정의. 예:
  "내 cron 목록", "daily-standup 잠깐 멈춰", "회고 cron 시간 10시로", "그거 지워".
- 봇은 슬래시를 실행하지 않는 기존 모델을 따라(§ /create 와 동일), Chief 가 prompt-land
  에서 의미를 갖고 **결정적 동작은 `solosquad cron *` 헬퍼로 위임**(파일 직접 조작이 아니라
  검증을 거치도록). 파괴적 동작(disable/delete)은 §E1 확인 게이트 통과.
- triage: 이런 발화가 새 cron 등록이 아니라 **기존 cron 관리**임을 구분(오분류 방지).

**§E4 — 메신저에서 실행/실패 조회.**
- "cron 상태/이력 보여줘" → Chief 가 `cron runs` 요약(최근 성공/조용/실패 · 마지막 실행 ·
  overdue 여부)을 본문 텍스트로 응답. CLI `cron runs`/`show` 의 데이터를 대화형으로 노출.

**§E5 — `pm-compaction` → `chief-compaction` 개명 (id까지 + 마이그레이션).**
- 표시명은 이미 "Chief Compaction"이나 내부 식별자가 `pm-*` 로 남아 있다. 직전 `63900c0`
  이 의도적으로 보존한 on-disk 계약을 이번엔 **마이그레이션과 함께 전환**한다:
  - cron **id** `pm-compaction → chief-compaction` (`CRONS`, `resolveCrons` case)
  - 스레드명 `system-pm-compaction → system-chief-compaction`
  - `workspace.yaml` 키 `pm.compaction_time → chief.compaction_time`(읽기 시 legacy
    fallback 유지)
  - 런로그 id 연속성: `cron-runs.jsonl` 의 과거 `pm-compaction` 레코드는 그대로 두되,
    `lastSuccessfulRun` 등 조회가 신구 id 를 함께 인식하도록(또는 마이그레이션이 rename).
  - 마이그레이션 스크립트(예: `1.3.3-to-1.3.4.ts`) — yaml 키 이전 + 스레드 참조. 멱등.
- 이벤트 네임스페이스 `pm.*` 는 별도 판단(영향 범위 큼) — 본 개명에서는 cron 도메인에
  한정하고 이벤트 rename 은 비목표로 둔다(§4).

---

## 4. 비목표 (Non-goals)

- cron 표현식 빌더 GUI / 웹 대시보드 — CLI·대화형으로 충분.
- 분 단위 미만(초 단위) 정밀 스케줄 — node-cron 6필드로 충분, 권장 안 함.
- 다중 타임존 한 cron 동시 발화 — 유저별 개인 brief 로 이미 커버.
- catch-up 의 N회 정확 재현 — 의도적으로 "1회 합침"만(폭주 방지 우선).
- **`pm.*` 이벤트 네임스페이스 rename** — 영향 범위가 cron 을 넘어서므로 §E5 개명은
  cron 도메인(id·스레드·yaml 키)에 한정. 이벤트는 별도 작업.

## 5. 구현 순서 (제안)

1. **§C 공용 util 추출**(`src/util/timezone.ts`) + init 리팩터 — 최저위험, 즉시 가치.
2. **§C 검증 라스트 가드**(`CRON_TZ_INVALID`) + `user-crons` skip — 데이터 안전.
3. **§B next-N 미리보기**(`nextRuns`) — `cron new/edit/show` 표면 (§E1 확인의 전제).
4. **§E2 실패 메신저 알림** — `runCronForProduct` error 경로, 노이즈 가드. 단위 테스트.
5. **§E5 개명** `pm-compaction → chief-compaction` + 마이그레이션 `1.3.3-to-1.3.4.ts`.
6. **§E1 확인 게이트** (CLI `--yes` 확장 + chief SKILL 확인) + **§E3 대화형 CRUD** +
   **§E4 조회** — chief SKILL/triage + CLI 위임.
7. **§C CLI 리스트 선택** + **대화형 번호 보기**(chief SKILL).
8. **§A 지터** + **§D 안전장치 3종** — 데몬 경로, 주입식 RNG 테스트.
9. **§B cronstrue** 여부 결정(§6) 후 readback 마감.

각 단계 단위 테스트 동반. 목표 테스트 수: 911 → ~945.
