## 1. 배경 / 문제

SoloSquad 에는 사용자·Chief 가 다루는 4개의 **1급 자산(first-class artifact)** 이 있다:

| 자산 | 정의 | 런타임 효과 |
|---|---|---|
| **skill** | `SKILL.md` (frontmatter + 본문) | 트리거 시 에이전트에 주입되는 절차적 역량 |
| **workflow** | `workflow.yaml` + `_status.yaml` | 다단계 specialist 오케스트레이션 |
| **goal** | `goal.md` (metrics/pipeline/budget) | 자율 엔진의 keep/discard 사이클 실행 |
| **schedule** | `schedules/<id>.md` + cron | 정기 routine 실행 |

각 자산마다 **생성 경로는 있으나 생애주기 관리가 제각각·불완전**하다. v1.3.1 §7 의 결론
("`skill-author` 는 생성만 함의해 역할이 좁다")은 **4개 모두에 동일하게 적용**된다. 공통 진단:

- **생성은 있고 검증/개선은 빈약** — skill 은 author 가 있지만 validate 갭(아래 §3), workflow 는
  `workflow.yaml` validator 가 **아예 없음**, goal 은 parse-time 검증만, schedule 은 **사용자 생성
  경로 자체가 없음**(`ROUTINES[]` 하드코딩).
- **개선 루프 단편적** — `skill-refinement`·`workflow-refinement` skill 은 있으나 goal/schedule
  은 없고, 어느 것도 측정→bounded-edit→재검증 루프(§8)를 갖추지 않음.
- **lifecycle 표면 불균형** — goal 은 풍부한 CLI(new/list/show/run/status/stop/verify/queue),
  workflow 는 list/show/focus, skill·schedule 은 거의 없음.

목표: 4개 자산을 **일관된 "매니저" 추상(create/review/validate/refine + lifecycle ops)** 으로
정렬해, 사용자가 "만들고 → 검증받고 → 돌려보고 → 개선" 하는 경험을 동일하게 갖게 한다.

## 2. 매니저 패턴 (공통 추상)

4개 매니저가 공유하는 동사(verb) 집합. 모든 매니저가 4개를 다 갖출 필요는 없으나, **인터페이스는
동일한 이름**으로 노출한다(학습 비용 최소화).

| 동사 | 의미 | 공통 토대 |
|---|---|---|
| **create** | 의도 → 초안 자산 1-shot/대화형 생성 | 도메인별 author/maker |
| **review** | 생성·수정된 자산을 체크리스트로 LLM 검토(품질·구조·안티-시코펀시) | `skill-pr-review/` 패턴 |
| **validate** | 정적 스키마/규칙 검증(공식 표준 + SoloSquad 확장) | 도메인별 parser |
| **refine** | 측정 → 반성 → **bounded edit** → held-out 재검증 게이팅 | SkillOpt 식 루프(§8) |
| **lifecycle** | list / status / edit / enable·disable / archive | goal CLI 가 레퍼런스 |

> **설계 원칙:** "공유 폴더/싱글톤 매니저"가 아니라 **도메인별 모듈 + 공통 인터페이스**. validate/
> refine 의 *엔진*(bounded-edit 루프, corpus 게이팅)은 §8 의 공유 인프라로 추출해 4곳이 재사용.

## 3. 도메인별 현황 요약 (감사 결과 2026-06-18)

| 도메인 | create | review | validate | refine | lifecycle | 최대 갭 |
|---|---|---|---|---|---|---|
| **skill** | ✅ `skill-author.ts`(대화형 5-state) | ✅ `skill-pr-review/`(CI) | 🔶 `validateSkill`(naming/길이/dir-match 미검사) | 🔶 `skill-refinement` skill + corpus(round-trip만) | ❌ 거의 없음 | naming·conflict 검증, refine 루프 |
| **workflow** | ✅ `workflow-maker` skill + matcher | 🔶 `workflow-refinement` skill | ❌ `workflow.yaml` validator 부재 | 🔶 refinement skill(분석만) | ✅ list/show/focus | YAML validator, 순환의존·exit_criteria 검사 |
| **goal** | ✅ `goal new` + spec-gate | ❌ | ✅ parse-time(`goal-parser`) | ❌ | ✅ 풍부(new/run/status/verify/queue) | authoring 보조, metric provenance 검증, mid-run edit |
| **schedule** | ❌ 하드코딩(`ROUTINES[]`) | ❌ | ❌ | ❌ | 🔶 enable/disable(yaml) | 동적 레지스트리, 사용자 생성/관리 CLI, orphan 3개 배선 |

핵심 관찰: **skill 은 create 가 가장 성숙·validate 가 갭**, **workflow 는 validate 부재**,
**goal 은 lifecycle 성숙·authoring/refine 부재**, **schedule 은 거의 전부 부재**.

> **근거 문서(전수 조사).** 4개 도메인 각각 "좋은 X 란 무엇인가"의 크로스벤더 마스터 문서가
> 있다 — §4~§7 의 전략은 이들에서 도출:
> - skill — [`docs/ideation/260617-skill-md-authoring-best-practices.md`]
> - workflow — [`docs/ideation/260618-workflow-authoring-best-practices.md`]
> - goal — [`docs/ideation/260618-goal-authoring-best-practices.md`]

## 4. skill-manager (v1.3.1 §7 흡수·확장)

`src/bot/skill-author.ts` → `src/bot/skill-manager.ts` 개명, 단일 "저자"에서 전 생애주기로 확장.

- **create** — 기존 `runAuthorLoop`(대화형 5-state) 유지. 표준 description 공식("`<무엇을>` —
  `<구조>`. `<출처>` 기반. 사용 시점: A,B,C. 제외: X"), 3인칭·트리거 첫 문장.
- **review** — `skill-pr-review/` 모듈(frontmatter-diff·keyword-conflict·domain-overlap)을 LLM
  검토 표면으로 승격. 본문 <500줄·참조 1단계·non-goal 유무 소프트 게이트 추가.
- **validate(P0, 가장 싸고 효과 큼)** — `validateSkill` 강화. **미검사 보강:** `name` ≤64자·
  `^[a-z0-9]+(-[a-z0-9]+)*$`·**부모 디렉터리명 일치**·예약어, `description` ≤1024자·3인칭(1인칭
  린트)·"Use when" 트리거절. cross-SKILL name 충돌 + domain 용어 충돌 탐지.
- **refine** — §8 공유 루프 적용. `skill-author`(공식 "Claude A") + `validator-corpus`/`test:corpus`
  (eval 골격)을 재활용해 rollout→reflect→bounded-edit→held-out 검증.
- **lifecycle** — `solosquad skill list/show/validate` 신설(현재 없음).

> P0 = validate 강화(단독 선출고 가능, corpus 호환↑). create/review/refine = P1.

## 5. workflow-manager

`workflow-maker`/`workflow-refinement` skill + `workflow-resolver`/`workspace-meta` 을 매니저로 정렬.

- **create** — 기존 `workflow-maker`(matcher 0.5 임계 → 템플릿 커스터마이즈) 유지.
- **review** — 신설. stage 그래프 sanity(에이전트 ref 형식 `<team>/<agent>`·`_main/`·`_skill/`,
  고아 stage, handoff_to 도달성).
- **validate(P0) — 최대 갭.** `validateWorkflow(workflow.yaml)` 신설: schema_version, stage id
  유일성, **순환 의존(depends_on) 탐지**, `exit_criteria`/`hard_gate` 문법, target_repo 존재성.
  현재 템플릿은 사전검증되나 사용자 커스텀/생성 인스턴스는 무검증.
- **refine** — 기존 `workflow-refinement`(bottleneck·handoff fidelity 분석)에 §8 루프 결합 —
  분석에서 그치지 말고 stage 재배치/prompt 보정을 bounded-edit 로 제안·게이팅.
- **lifecycle** — 기존 list/show/focus 유지 + `validate` 추가.

비범위(v1.4+): 병렬 stage(async merge), 워크플로 중첩/파라미터화, artifact manifest.

## 6. goal-manager

`src/cli/goal.ts` + `src/engine/*` 위에 authoring·refine 보조를 얹는다(엔진 루프는 안정, 손대지 않음).

- **create** — 신설 authoring 보조(`goal assist` 또는 skill). 현재는 템플릿 주석 읽고 수기 편집.
  metric/pipeline 섹션 채움 가이드 + 후보 metric 제안.
- **review** — 신설. metric 정의 품질(provenance 명시·direction·threshold 합리성) LLM 검토.
- **validate(P1)** — parse-time 위에 **semantic 검증** 추가: `source` 파일 존재성, formula 평가
  가능성, pipeline 의 `<team>/<agent>` 실존 검사. (현재 parser 는 문자열만 수용.)
- **refine** — 신설. `goal verify`(결정성 replay) 위에 "metric 비결정/oscillation 진단 → threshold·
  pipeline bounded-edit 제안" 루프. **단, 실행 중 goal 의 mid-run edit 은 다음 사이클 재파싱으로 한정.**
- **lifecycle** — 이미 성숙(new/list/show/run/status/stop/verify/queue/active/next). `goal estimate`
  (예상 비용) + `goal measure <metric>`(단일 metric 디버그) 추가.

비범위: goal 별 사용자 정의 termination policy(stop-hook 은 현재 SKILL 전용), goal 버전 히스토리.

## 7. schedule-manager (가장 미성숙 — 사용자 생성 경로 자체가 없음)

현재 `ROUTINES[]` 하드코딩 + cron 시간만 `workspace.yaml.briefings` 로 설정 가능. 사용자가 커스텀
schedule 을 만들려면 코드 수정·재배포 필요.

- **create(P0) — 핵심.** **동적 ROUTINES 레지스트리**로 전환. `schedules/` 디렉터리(또는
  `.solosquad/schedules.yaml` manifest)에서 routine 정의(id·cron·kind·channel·memoryTargets)를
  로드 → 하드코딩 배열 제거. `loadRoutinePrompt`(이미 `getSchedulesDir` 배선, v1.3.1 §9)는 그대로 재사용.
- **review** — 신설. cron 식 유효성·중복 시간·프롬프트 기본 구조 검토.
- **validate(P0)** — cron 표현식 파싱, kind enum, channel 존재, 프롬프트 파일 존재성.
- **refine** — 신설(경량). routine-logs 기반 "실행 실패/저효용 routine 탐지" + 제안.
- **lifecycle** — `solosquad schedule create/list/edit/enable/disable/delete` 신설. 현재는
  `run-routine`(수동 실행)·enable 플래그만.
- **orphan 3개 처리** — `trace-rotate` 는 `system-housekeeping` 에 흡수됨(삭제 후보). `bot-health-check`·
  `leading-indicator` 는 관측 레이어(세션 추적·budget 질의) 의존 → v1.4.0 까지 추적 유지 또는
  동적 레지스트리 위에서 opt-in 배선.

## 8. 공통 인프라 (cross-cutting)

매니저 4곳이 중복 구현하지 않도록 **validate/refine 엔진을 공유 모듈로 추출**.

- **8.1 검증 코어** — 도메인 무관 "rule → finding(error|warning)" 러너. 각 매니저는 규칙 집합만 제공.
  `skill-pr-review/` 의 finding/comment-formatter 패턴을 일반화.
- **8.2 refine 루프 (SkillOpt 식)** — `corpus rollout → 성공/실패 반성 → bounded edit(스텝당 4–8) →
  held-out 검증 시에만 채택 + 거부 buffer`. protected-section 으로 내구 지침 보호. skill 의
  `validator-corpus`/`test:corpus` 골격을 도메인 일반화(workflow/goal/schedule corpus 추가).
- **8.3 naming/예약어** — 4개 자산의 id 규칙 통일(kebab-case, 예약어, 충돌 탐지)을 한 모듈로.
- **8.4 lifecycle CLI 패턴** — goal CLI 의 list/show/status 패턴을 4개에 동형으로.

## 9. 범위 / 단계 / 비범위

**P0 (가장 싸고 단독 선출고 가능):**
- skill `validate` 강화(§4) — naming/dir-match/description 규칙.
- workflow `validate` 신설(§5) — `validateWorkflow`, 순환의존·exit_criteria.
- schedule 동적 레지스트리 + `validate`(§7) — `ROUTINES[]` 하드코딩 제거.

**P1:**
- 각 매니저 `review` LLM 표면 + `create` 보조(goal/schedule).
- 공통 검증 코어(§8.1) 추출.

**P2 (refine 루프):**
- §8.2 SkillOpt 식 refine 을 skill 먼저 → workflow/goal/schedule 로 확장.

**Out (v1.4+):**
- workflow 병렬 stage·중첩, goal 사용자 정의 termination policy, schedule orphan(health-check·
  leading-indicator) 배선(관측 레이어 선행 필요), stateful skill(v0.6 actor).

**개명/하위호환:** `skill-author.ts → skill-manager.ts` 시 import 처 + 호출부 grep 후 일괄,
공개 CLI `author` 서브커맨드는 alias 유지. workflow/goal/schedule 은 기존 CLI 표면 보존하며 verb 추가.

**검증 동반(구현 PR 시):** 각 매니저 verb 에 단위 테스트 + corpus 케이스. `npm run validate-skills`
(현 `agent validate --all && test:corpus`)에 강화된 validate + (선택) workflow/goal/schedule validate 합류.
