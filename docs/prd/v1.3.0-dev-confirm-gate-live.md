# v1.3.0 — dev-confirm 게이트 라이브화 (git push 승인 게이트)

> **상태:** 설계 초안 (2026-06-04 발의, 2026-06-08 알림 범위 제거로 개정). 구현
> 미착수. 본 문서는 v0.8.2 에서 "정의만" 깔아둔 dev-confirm 게이트를 **실제 spawn
> 실행 경로에 연결**해, 에이전트의 `git push` 가 실행 직전에 **사용자 승인**을 받게
> 한다. spawn hot-path 를 건드리는 위험 작업이라 별도 minor(1.3.0)로 분리한다.
>
> **범위 개정(2026-06-08):** 초안의 "git-feed 알림 실작동"은 **제거**됐다. SoloSquad
> 는 push 알림 기능에 관여하지 않는다 — 알림이 필요하면 사용자가 **GitHub→Discord
> 네이티브 webhook** 을 직접 연결한다. `git-<handle>` 채널/`notifyGitPush` 는
> v1.2.10 에서 롤백됐다([`v1.2.10-chief-rename-and-git-channel-removal.md`](./v1.2.10-chief-rename-and-git-channel-removal.md) §Part C). 본 버전의 본체는 **승인 게이트
> only** 이며, 여기에 **브랜치 가드레일 + 감사↔커밋 매핑 + 커밋 workflow-id 스탬프**
> (구 `v1.3.x-git-guardrails` 문서 흡수)를 더한다.

**로드맵 위치:** v1.2.10 (리네임 + git 채널 롤백) → **v1.3.0 (승인 게이트 라이브화 +
가드레일/감사)** → v1.4.0 (세션 오케스트레이션).

**의존:**
- v0.8.2 §5.2 — `dev-confirm.ts` 게이트 인프라(`createDevConfirm`/`detectSensitiveCommand`/`DevConfirmController`/audit JSONL) — **정의만 존재, 미연결**.
- v1.2.9 §E — spawn 시 `--settings` 로 PreToolUse Bash **deny** hook 주입(dev-OFF). v1.3.0 은 이를 **승인(approve) 흐름**으로 승격.
- v1.2.10 §C — `git-<handle>`/`notifyGitPush` 롤백. `dev-confirm.ts` `onApproved` 는 generic 훅으로 잔존(감사/스탬프용 재사용).

**분류:** minor (1.3.0). spawn 실행 경로에 실시간 개입 메커니즘을 신설하므로 patch 가 아님.

## 1. 배경 — 왜 dormant 인가

dev-confirm 게이트는 v0.8.2 에서 정의됐지만 **spawn 경로에 연결되지 않아** 한 번도
발사되지 않았다. 코드 조사(2026-06-04) 결론:

| 항목 | 상태 | 근거 |
|---|---|---|
| `createDevConfirm()` production 호출 | ❌ 없음 | `src/bot/` 전역에서 정의 외 호출 0 (테스트만) |
| `detectSensitiveCommand()` 실호출(Bash 가로채기) | ❌ 없음 | 정의 + 테스트만 |
| `Map<confirmId, controller>` + 사용자 "y" → resolve 배선 | ❌ 없음 | bot/index.ts·chief-runner.ts 어디에도 없음 |
| `<org>/memory/dev-confirmations.jsonl` 실제 기록 | ❌ 없음 | `createDevConfirm` 미호출 → audit 미발생 |

→ **dev-confirm 게이트는 DORMANT.** "git push 가 게이트를 통과하는 순간"이라는
전제가 현재 코드엔 존재하지 않는다. v1.2.9 §E 가 dev-OFF 에서 `git push` 를 **deny**
하는 hook 은 깔았으나, 그것은 *차단*이지 *승인 흐름*이 아니다.

## 2. 핵심 제약 — specialist 는 봇 프로세스 밖에서 돈다

게이트를 라이브화하는 데 가장 큰 구조적 제약:

1. **specialist 는 별도 (sub)프로세스.** Chief 가 Claude Code 네이티브 `Task` 도구로
   delegate 하면 specialist 는 **그 `claude --print` 서브프로세스 안의 sub-agent** 로
   실행된다. 봇 프로세스는 그 안의 개별 `Bash` tool_use 를 직접 가로채거나 일시정지시킬
   수 없다 — `task_started`/`task_notification` 시스템 이벤트만 stream 으로 관찰한다.
2. **현재 spawn 플래그엔 permission/hook 훅이 없다.** `claude-process.ts` `buildArgs()`
   는 `--print --session-id/--resume --output-format stream-json …` 만 넘긴다.
   `--permission-mode` / `--permission-prompt-tool` 는 없음. (`--settings` 는 v1.2.9
   §E 가 deny hook 주입용으로 이미 사용 중.)
3. **기존 도구 제어는 정적 allow/deny.** `spawn-assembler.ts` 의 `applyDevPermissions`
   (현재 미호출)가 allow/deny 리스트를 만들지만, 이는 *사전 차단*이지 *런타임 승인*이
   아니다.

따라서 "push 를 실시간으로 멈추고 사용자에게 물어보는" 흐름은 **새 메커니즘**이 필요하다.

## 3. 설계 옵션

| 옵션 | 메커니즘 | 장점 | 단점 | 채택 |
|---|---|---|---|---|
| **A. PreToolUse hook + 파일 IPC** | spawn 시 `--settings`(또는 cwd `.claude/settings.json`)로 Claude Code `PreToolUse` Bash hook 주입. hook 스크립트가 `git push` 감지 → `<org>/memory/pending-confirms/<id>.json` 작성 → 봇이 watch → 사용자 "y" → 봇이 resolve 파일 작성 → hook 이 polling/대기 후 exit code 로 allow(0)/deny(2). | Claude Code 공식 hook 사용. specialist 프로세스 안에서 정확히 Bash 직전 가로챔. 봇은 파일 watch 만. v1.2.9 §E deny hook 과 동일 주입 경로 재사용. | hook↔봇 파일 IPC + polling 레이턴시. hook 스크립트 크로스플랫폼(node shebang) 필요. | ✅ **권고** |

## 4. 권고 설계 — A안 (PreToolUse hook + 파일 IPC)

### 4.1 흐름

```
specialist(claude sub-process)가 git push 시도
  └─ Claude Code PreToolUse(Bash) hook 발화
       └─ hook 스크립트: detectSensitiveCommand(cmd) ?
            ├─ null → exit 0 (통과)
            └─ "git push" → ⓐ 보호 브랜치 직 push? → 즉시 exit 2 (차단, §4.4)
                 ⓑ 아니면 write <org>/memory/pending-confirms/<id>.json
                      { id, cmd, repoSlug, branch, ts, user }
                    그리고 <id>.decision 파일을 polling(짧은 interval, timeout 30m)
   ─────────────────────────────────────────────────────────────
봇 프로세스(파일 watcher)
  └─ pending-confirms/<id>.json 등장 감지
       └─ createDevConfirm({...}) 생성, Map<id, controller> 보관
       └─ command-<handle> 채널에 "확인 #<id>: git push ... ? (y/n)" 전송
사용자가 command 채널에 "y"
  └─ bot/index.ts 메시지 핸들러: 대기 중 confirm 있으면 controller.resolve("y")
       └─ finish("y"): audit 기록(+ commit hash 매핑, §4.3) → onApproved(req)
       └─ 봇이 <id>.decision 파일에 "y" 기록
   ─────────────────────────────────────────────────────────────
hook 스크립트
  └─ <id>.decision = "y" 읽음 → exit 0 (push 진행)
     <id>.decision = "n"/timeout → exit 2 (push 차단)
```

> **알림은 없다.** 승인되면 push 가 진행될 뿐, SoloSquad 는 별도 알림을 쏘지 않는다.
> push 결과 피드가 필요한 사용자는 GitHub repo settings 에 **Discord webhook URL**
> 을 직접 등록한다(네이티브 GitHub→Discord). §4.5 의 커밋 스탬프 덕에 그 네이티브
> 피드에 "어느 에이전트 작업인지"가 자연히 노출된다.

### 4.2 손댈 파일 (승인 게이트)

| # | 파일 | 작업 |
|---|---|---|
| 4.2.a | `src/bot/claude-process.ts` `buildArgs` | spawn `--settings` 에 `PreToolUse` Bash hook 등록(v1.2.9 §E deny hook 과 병합). hook command = 번들된 `dist/bot/hooks/dev-confirm-hook.js` 절대경로. |
| 4.2.b | `src/bot/hooks/dev-confirm-hook.ts` (신규) | hook 진입점. stdin `tool_input.command` 에 `detectSensitiveCommand` → 보호 브랜치 차단(§4.4) / pending 파일 write + decision polling → exit 0/2. **순수 파일 IPC, 네트워크 0.** |
| 4.2.c | `src/bot/dev-confirm-bridge.ts` (신규) | 봇 측: `pending-confirms/` watcher. 파일 등장 → `createDevConfirm` → `Map` 보관 → command 채널 질의 → decision 파일 write. |
| 4.2.d | `src/bot/index.ts` 메시지 핸들러 | command-<handle> 유입 메시지가 대기 confirm 의 y/n 이면 Chief 턴으로 안 보내고 `controller.resolve` 로 라우팅. |
| 4.2.e | `src/bot/spawn-assembler.ts` | `dev_capability` / `requirePushConfirmation` 를 SKILL frontmatter 에서 읽어 hook 활성 여부 결정(`applyDevPermissions` 라이브화). |

### 4.3 감사 ↔ 커밋 해시 매핑 (구 v1.3.x §2.3 흡수)

- dev-confirm 승인 레코드(`dev-confirmations.jsonl`)에 push 범위 커밋 해시를 함께
  기록: hook 이 `git log --oneline <remote>/<branch>..HEAD`(best-effort)로 수집한
  목록을 pending 파일에 실어 보내고, bridge 가 audit 레코드에 "승인 시각 ↔ 커밋 해시
  ↔ workflow-id"로 1:1 기록.
- 이는 알림이 아니라 **로컬 감사 추적** — 외부 전송 0.

### 4.4 브랜치 가드레일 (구 v1.3.x §2.1 흡수)

- hook 단계에서 push target branch 파싱 → 보호 브랜치(`main`/`master`/`develop`,
  설정화) **직 push 는 승인 흐름 이전에 자동 차단**(exit 2 + 가이드). feature 브랜치
  (`feat/*` 등)만 승인 흐름 진입.
- 설정: `chief.git.protected_branches`(기본 `["main","master","develop"]`) /
  `chief.git.require_feature_branch`(기본 true).
- 자가학습/장기작업 중 에이전트가 `main` 으로 직접 push 하는 사고를 *구조적으로* 차단.
  로컬 hook 레벨이라 호스트 무관(사설/온프렘 포함).

### 4.5 커밋 workflow-id 스탬프 (구 v1.3.x §2.2 흡수, 우선순위 낮음)

- 에이전트 spawn 시 커밋 committer/trailer 에 식별값(예: trailer `SoloSquad-Workflow: <wf-id>`)
  을 주입 → 사용자 GitHub→Discord 네이티브 webhook 의 커밋 메시지에 귀속이 자연 노출.
- GPG/SSH 서명은 단일사용자 로컬엔 과함 — 옵션(`chief.git.sign_commits`, 기본 off).

## 5. 리스크

1. **모든 push 차단 위험.** hook exit code 오류 / decision 파일 race → push 영구 블록
   가능 → fail-open vs fail-closed 정책 결정 필요(기본: timeout=차단, 단 hook 자체
   오류는 fail-open + 경고 로그).
2. **다중 동시 push.** id = hash(cmd+ts+pid). command 채널 질의는 건별.
3. **봇 재시작 중 대기 confirm 유실.** pending-confirms/ 파일은 디스크에 남지만
   in-memory controller 는 소실 → 재시작 시 pending 스캔 복구.
4. **single-bot-process 불변식 의존.** in-memory Map 전제 → 다중 봇 프로세스에선 파일
   락 필요(현재 v0.7 lifecycle precheck 가 단일 프로세스 보장).
5. **Claude Code hook 스키마 의존.** `PreToolUse` 입출력 포맷(stdin JSON, exit code
   의미)은 버전에 묶임 → 버전 가드 + doctor 점검.
6. **브랜치 파싱 정확도.** force/부분 push, 비표준 refspec → 파싱 실패 시 fail-safe =
   승인 흐름으로 폴백(차단 아님) + 경고 로그.

## 6. Open Questions

1. **repo 식별** — push 가 어느 repo 인지: spawn `target_repo` 우선, 없으면 hook 의
   cwd 캡처. (audit `repoSlug` 입력.)
2. **다중 push** — 건별 메시지 vs 묶음. MVP: 건별.
3. **승인 의미론** — 게이트는 "승인" 시점만 관측(실제 push 성공은 hook exit 후
   specialist 내부). 사후 성공 확인이 필요하면 사용자 네이티브 webhook 이 담당.
4. **fail-open vs fail-closed** — 리스크 1. 사용자/보안 트레이드오프 결정 필요.
5. **커밋 스탬프 author 분리** — 사용자 본인 커밋과 섞이지 않게 author/trailer 정책
   (사용자 선호 확인).

## 7. 비범위

- **push 알림 자체 구현** — 영구 비범위. 사용자 GitHub→Discord 네이티브 webhook 위임.
- `git-<handle>` 채널 / `notifyGitPush` — v1.2.10 §C 에서 롤백 완료.
- commit 단독(수동 로컬 commit) 포착 — 사용자 webhook 영역.
- repo 레벨 GitHub branch protection Rule 자동 설정 — 사용자 책임 영역.
- `gh pr merge`/`gh pr close` 의 별도 기록 — 현재는 works 카드 유지.

## 검증 (구현 시 수용 기준)

- `git push origin feat/x` → command 채널에 "확인 #id" 질의 → "y" → push 진행 / "n"·
  timeout → 차단.
- `git push origin main`(보호 브랜치) → 승인 질의 없이 자동 차단 + 가이드(§4.4).
- 승인 시 `dev-confirmations.jsonl` 에 커밋 해시 + workflow-id 매핑 기록(§4.3).
- 알림 미발생 — SoloSquad 는 어떤 채널에도 push 알림을 쏘지 않음.
- `npx tsc --noEmit` · `npm test` green + hook 크로스플랫폼(node shebang) doctor 점검.
