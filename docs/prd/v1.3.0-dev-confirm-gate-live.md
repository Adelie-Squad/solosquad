# Part A — dev-confirm 승인 게이트 라이브화 (git push)

## A.1 배경 — 왜 dormant 인가

dev-confirm 게이트는 v0.8.2 에서 정의됐지만 **spawn 경로에 연결되지 않아** 한 번도
발사되지 않았다. 코드 조사(2026-06-04) 결론:

| 항목 | 상태 | 근거 |
|---|---|---|
| `createDevConfirm()` production 호출 | ❌ 없음 | `src/bot/` 전역에서 정의 외 호출 0 (테스트만) |
| `detectSensitiveCommand()` 실호출(Bash 가로채기) | ❌ 없음 | 정의 + 테스트만 |
| `Map<confirmId, controller>` + 사용자 승인 → resolve 배선 | ❌ 없음 | bot/index.ts·chief-runner.ts 어디에도 없음 |
| `<org>/memory/dev-confirmations.jsonl` 실제 기록 | ❌ 없음 | `createDevConfirm` 미호출 → audit 미발생 |

→ **dev-confirm 게이트는 DORMANT.** v1.2.9 §E 가 dev-OFF 에서 `git push` 를 **deny**
하는 hook 은 깔았으나, 그것은 *차단*이지 *승인 흐름*이 아니다.

## A.2 핵심 제약 — specialist 는 봇 프로세스 밖에서 돈다

1. **specialist 는 별도 (sub)프로세스.** Chief 가 네이티브 `Task` 로 delegate 하면
   specialist 는 `claude --print` 서브프로세스 안의 sub-agent 로 실행된다. 봇은 그 안의
   개별 `Bash` tool_use 를 직접 가로채거나 일시정지시킬 수 없다 — 시스템 이벤트만
   stream 으로 관찰한다.
2. **현재 spawn 플래그엔 permission/hook 훅이 없다.** `claude-process.ts` `buildArgs()`
   는 `--print --session-id/--resume --output-format stream-json …` 만 넘긴다.
   (`--settings` 는 v1.2.9 §E deny hook 주입용으로 이미 사용 중.)
3. **기존 도구 제어는 정적 allow/deny.** `spawn-assembler.ts` `applyDevPermissions`(미호출)
   는 *사전 차단*이지 *런타임 승인*이 아니다.

따라서 "push 를 실시간으로 멈추고 사용자에게 물어보는" 흐름은 **새 메커니즘**이 필요하다.

## A.3 설계 옵션

| 옵션 | 메커니즘 | 채택 |
|---|---|---|
| **A. PreToolUse hook + 파일 IPC** | spawn 시 `--settings` 로 `PreToolUse` Bash hook 주입. hook 이 `git push` 감지 → `pending-confirms/<id>.json` 작성 → 봇이 watch → 사용자 승인 → 봇이 `<id>.decision` 작성 → hook 이 polling 후 exit 0(allow)/2(deny). | ✅ **권고** |
| B. permission-prompt-tool (MCP) | `--permission-prompt-tool <mcp-tool>` + 로컬 MCP 서버. 구조적 정석이나 lifecycle/헤드리스 인증 복잡. | △ 후보 |
| C. git wrapper (PATH 주입) | 이식성 낮음, repo 침습. | ✕ |
| D. allowlist/denylist 만 | "승인"이 아니라 "금지"(v1.2.9 §E 현행). 1-click 승인 불가. | ✕ |

## A.4 권고 설계 — A안 (PreToolUse hook + 파일 IPC)

### A.4.1 흐름

```
specialist(claude sub-process)가 git push 시도
  └─ Claude Code PreToolUse(Bash) hook 발화
       └─ hook: detectSensitiveCommand(cmd) ?
            ├─ null → exit 0 (통과)
            └─ "git push" → ⓐ 보호 브랜치 직 push? → 즉시 exit 2 (차단, §A.4.4)
                 ⓑ 아니면 pending-confirms/<id>.json write { id, cmd, repoSlug, branch, ts, user, commits }
                    그리고 <id>.decision 파일 polling (timeout 30m)
   ─────────────────────────────────────────────────────────────
봇 프로세스(파일 watcher = dev-confirm-bridge.ts)
  └─ pending-confirms/<id>.json 감지
       └─ createDevConfirm({...}) → Map<id, controller>
       └─ command-<handle> 채널에 ★승인 카드(버튼) 게시  ← Part B (§B.2), 텍스트 y/n 아님
사용자가 [✅ 승인] 버튼 클릭 (또는 [❌ 거절])
  └─ interactionCreate 핸들러: customId 파싱 → controller.resolve("y"/"n")  ← Part B (§B.4)
       └─ finish: audit 기록(+커밋 해시 매핑 §A.4.3) → onApproved(req)
       └─ 봇이 <id>.decision 에 "y"/"n" write
   ─────────────────────────────────────────────────────────────
hook: <id>.decision = "y" → exit 0 (push 진행) / "n"·timeout → exit 2 (차단)
```

> **알림은 없다.** 승인되면 push 가 진행될 뿐, SoloSquad 는 별도 알림을 쏘지 않는다.
> push 피드가 필요하면 사용자가 GitHub→Discord 네이티브 webhook 을 직접 등록한다.
> §A.4.5 커밋 스탬프 덕에 그 네이티브 피드에 "어느 에이전트 작업인지"가 자연 노출된다.

### A.4.3 감사 ↔ 커밋 해시 매핑

- 승인 레코드(`dev-confirmations.jsonl`)에 push 범위 커밋 해시 동반 기록: hook 이
  `git log --oneline <remote>/<branch>..HEAD`(best-effort)로 수집 → pending 파일에 실어
  보내고, bridge 가 "승인 시각 ↔ 커밋 해시 ↔ workflow-id" 1:1 기록. **외부 전송 0.**

### A.4.4 브랜치 가드레일

- hook 단계에서 push target branch 파싱 → 보호 브랜치(`main`/`master`/`develop`) **직
  push 는 승인 흐름 이전에 자동 차단**(exit 2 + 가이드). feature 브랜치만 승인 진입.
- 설정: `chief.git.protected_branches`(기본 `["main","master","develop"]`) /
  `chief.git.require_feature_branch`(기본 true). 로컬 hook 레벨이라 호스트 무관.

### A.4.5 커밋 workflow-id 스탬프 (우선순위 낮음)

- 커밋 trailer 에 `SoloSquad-Workflow: <wf-id>` 주입 → 네이티브 webhook 의 커밋 메시지에
  귀속 노출. GPG/SSH 서명은 옵션(`chief.git.sign_commits`, 기본 off).

# Part B — 인터랙션 UX (버튼/메뉴 + 오발복구)

> **조사 근거:** [`260610-messenger-streaming-and-artifacts.md`](../ideation/260610-messenger-streaming-and-artifacts.md) §2.4·§5(입력 UX).
> Discord 메시지 컴포넌트는 우리 레포에서 **이미 검증된 패턴**(`discord-onboarding.ts:121`)이라
> 추상적 신설이 아니라 일반화다.

## B.1 목표 — 텍스트 `y/n` 의 퇴출

현재 모든 사용자 선택은 **자유 텍스트 파싱**(승인 "y", TRIAGE 응답 등)에 의존한다.
이는 (1) 오타·모호성, (2) 모바일 타이핑 부담, (3) "선택지 매핑 불명확" 문제를 낳는다.
Part B 는 **승인·선택을 Discord 컴포넌트(버튼·셀렉트 메뉴)로** 받는다.

## B.2 컴포넌트 매핑

| 용도 | 컴포넌트 | 비고 |
|---|---|---|
| **승인 게이트(Part A)** | 버튼 `✅ 승인` / `❌ 거절` / (옵션)`💬 의견` | 온보딩 버튼 패턴 복제 |
| **Chief 보기 선택 (≤5)** | 버튼 한 줄 (라벨+이모지) | AskUser 류 질문 |
| **Chief 보기 선택 (6+)** | `StringSelectMenu` 드롭다운 (최대 25) | 라벨/설명 포함 |
| **가벼운 투표/방향** | 이모지 리액션 토글 | un-react 로 복구(§B.3) |

- 컴포넌트 한계: 버튼 한 Action Row 5개·메시지 5행(25개), Select 25개, **클릭 후 3초 내
  ack 필수**(`deferUpdate()` 로 장시간 작업 우회).

## B.3 오발복구 — 4종 (잘못 눌렀을 때)

Discord 엔 native undo 가 없다. **클릭(의도)과 실행을 분리**해서 푼다.

| 패턴 | 적용 | 메커니즘 |
|---|---|---|
| **① 2단계 확인** | 되돌릴 수 없는 선택(거절·배포·삭제) | ephemeral "정말? [확정][취소]" — 본인만 보임 |
| **② 되돌리기 유예창** | 일반 보기 선택 | 클릭 직후 `update()` → "✅ 처리됨 · ↩ 되돌리기(10s)", 실행을 N초 지연 |
| **③ 클릭 후 비활성화** | 모든 확정 | `setDisabled(true)` 로 중복·연타 차단, "처리됨" 시각화 |
| **④ 리액션 토글** | 가벼운 투표 | un-react(`messageReactionRemove`)로 자연 취소 — add 즉시 실행 금지, 토글로 관측 |

## 리스크 (전체)

1. **모든 push 차단 위험(Part A).** hook exit 오류/decision race → fail-open(hook 자체
   오류) vs fail-closed(timeout=차단) 정책. 기본: timeout=차단, hook 오류=fail-open+경고.
2. **인터랙션 3초 룰(Part B).** 장시간 처리 시 `deferUpdate` 누락하면 "상호작용 실패"
   표시 → 승인 핸들러는 즉시 ack 후 비동기 처리.

   stuck 버그 정리 로직 필수.
4. **embed vs plain 충돌(C.4).** 스트리밍 embed 와 v1.2.9 §D plain 의 경로 분리 누락 시
   포맷 회귀.
5. **봇 재시작 중 대기 confirm/카드 유실.** pending-confirms/ 디스크 잔존, in-memory
   Map/카드 핸들 소실 → 재시작 시 스캔 복구.
6. **Claude Code hook 스키마 의존.** `PreToolUse` 입출력 포맷 버전 결합 → 버전 가드 +
   doctor 점검.

## 검증 (구현 시 수용 기준)

**Part A — 게이트**
- `git push origin feat/x` → command 채널에 ★승인 카드(버튼) → [✅ 승인] → push 진행 /
  [❌ 거절]·timeout → 차단.
- `git push origin main`(보호 브랜치) → 승인 질의 없이 자동 차단 + 가이드(§A.4.4).
- 승인 시 `dev-confirmations.jsonl` 에 커밋 해시 + workflow-id 매핑(§A.4.3).
- 알림 미발생 — 어떤 채널에도 push 알림 안 쏨.

**Part B — 인터랙션**
- 승인/선택이 **버튼/메뉴**로 게시되고, 클릭이 `interactionCreate` 로 라우팅(텍스트 y/n 불필요).
- 되돌릴 수 없는 선택은 2단계 확인(ephemeral) → [취소] 시 무동작.
- 확정 후 버튼 비활성화(중복 클릭 무해). 리액션 투표는 un-react 로 취소 반영.

**Part C — 스트리밍**
- 작업 시작 직후 진행 카드 즉시 게시(타이핑 점 아님), 단계가 **진행 중 실시간**으로 갱신.
- 진행 카드 [🛑 중단] → 돌던 턴이 취소됨.
- 산출물(P1) → `workspace/<org>/artifacts/` 저장 + 채팅엔 카드/링크(긴 텍스트 본문 아님).
- (P3 채택 시) 최종 답 edit throttle ≥1s, 429 미발생, 완료 시 ⚪ 제거/색 전환.

**공통**
- `npx tsc --noEmit` · `npm test` green + hook 크로스플랫폼(node shebang) doctor 점검.
- 메신저 미구현 어댑터(Slack 일부)는 **텍스트 폴백**으로 회귀 없이 동작.
