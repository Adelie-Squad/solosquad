# v1.3.0 — dev-confirm 게이트 라이브화 (git push 승인 + git-feed 실작동)

> **상태:** 설계 초안 (2026-06-04 발의). 구현 미착수. 본 문서는 v1.2.9 Part B
> 에서 "1안"으로 깔아둔 `git-<handle>` 채널 + push 알림 코드가 **실제로 발사**
> 되게 하려면 무엇이 필요한지를 설계한다. 핵심은 **dev-confirm 게이트를 spawn
> 실행 경로에 연결(라이브화)** 하는 것 — 이는 spawn hot-path 를 건드리는 위험
> 작업이라 별도 minor(1.3.0)로 분리한다.

**로드맵 위치:** v1.2.9 (git 채널 인프라 + 알림 코드 / 게이트 dormant) → **v1.3.0 (게이트 라이브화 + 알림 실작동)** → v1.3.x (commit 폴링 / Slack git 채널)

**의존:**
- v1.2.9 Part B — `src/bot/git-event-notify.ts`(`PushEvent`/`notifyGitPush`), `createDevConfirm` 의 `onApproved` hook, `git-<handle>` 채널/설정/마이그레이션 (모두 라이브)
- v0.8.2 §5.2 — `dev-confirm.ts` 게이트 인프라(`createDevConfirm`/`detectSensitiveCommand`/`DevConfirmController`/audit JSONL) — **정의만 존재, 미연결**

**분류:** minor (1.3.0). spawn 실행 경로에 실시간 개입 메커니즘을 신설하므로 patch 가 아님.

## 1. 배경 — 왜 dormant 인가

v1.2.9 Part B 는 `git-<handle>` 채널과 push 알림 emit 코드(`git-event-notify.ts`)를
완비하고, `createDevConfirm` 에 승인 시 호출되는 `onApproved` hook 까지 심었다.
그러나 알림은 발사되지 않는다 — **그 hook 을 트리거할 dev-confirm 게이트 자체가
spawn 경로에 연결되어 있지 않기 때문**이다.

코드 조사(2026-06-04) 결론:

| 항목 | 상태 | 근거 |
|---|---|---|
| `createDevConfirm()` production 호출 | ❌ 없음 | `src/bot/` 전역에서 정의 외 호출 0 (테스트만) |
| `detectSensitiveCommand()` 실호출(Bash 가로채기) | ❌ 없음 | 정의 + 테스트만 |
| `Map<confirmId, controller>` + 사용자 "y" → resolve 배선 | ❌ 없음 | bot/index.ts·chief-runner.ts 어디에도 없음 |
| `<org>/memory/dev-confirmations.jsonl` 실제 기록 | ❌ 없음 | `createDevConfirm` 미호출 → audit 미발생 |

→ **dev-confirm 게이트는 DORMANT.** "git push 가 게이트를 통과하는 순간"이라는
Part B 의 전제가 현재 코드엔 존재하지 않는다.

## 2. 핵심 제약 — specialist 는 봇 프로세스 밖에서 돈다

게이트를 라이브화하는 데 가장 큰 구조적 제약:

1. **specialist 는 별도 (sub)프로세스.** Chief 가 Claude Code 네이티브 `Task`
   도구로 delegate 하면 specialist 는 **그 `claude --print` 서브프로세스 안의
   sub-agent** 로 실행된다. 봇 프로세스는 그 안의 개별 `Bash` tool_use 를 직접
   가로채거나 일시정지시킬 수 없다 — `task_started`/`task_notification` 시스템
   이벤트만 stream 으로 관찰한다.
2. **현재 spawn 플래그엔 permission/hook 훅이 없다.** `claude-process.ts`
   `buildArgs()` 는 `--print --session-id/--resume --output-format stream-json
   --input-format stream-json --verbose --exclude-dynamic-system-prompt-sections
   [--include-partial-messages] [--max-budget-usd] [--append-system-prompt]
   [--allowed-tools] [--disallowed-tools] [--add-dir]` 만 넘긴다.
   `--permission-mode` / `--permission-prompt-tool` / `--settings`(hook 주입) 는
   **없음**.
3. **기존 도구 제어는 정적 allow/deny.** `spawn-assembler.ts` 의
   `applyDevPermissions`(현재 미호출) 가 `allowedTools`/`disallowedTools`/
   `bashAllowlist`/`bashDenylist` 를 만들지만, 이는 *사전 차단*이지 *런타임 승인*이
   아니다.

따라서 "push 를 실시간으로 멈추고 사용자에게 물어보는" 흐름은 **새 메커니즘**이
필요하다.

## 4. 권고 설계 — A안 (PreToolUse hook + 파일 IPC)

### 4.1 흐름

```
specialist(claude sub-process)가 git push 시도
  └─ Claude Code PreToolUse(Bash) hook 발화
       └─ hook 스크립트: detectSensitiveCommand(cmd) ?
            ├─ null → exit 0 (통과)
            └─ "git push" → write <org>/memory/pending-confirms/<id>.json
                 { id, cmd, repoSlug, ts, user }
                 그리고 <id>.decision 파일을 polling(짧은 interval, timeout 30m)
   ─────────────────────────────────────────────────────────────
봇 프로세스(파일 watcher)
  └─ pending-confirms/<id>.json 등장 감지
       └─ createDevConfirm({...}) 생성, Map<id, controller> 보관
       └─ command-<handle> 채널에 "확인 #<id>: git push ... ? (y/n)" 전송
사용자가 command 채널에 "y"
  └─ bot/index.ts 메시지 핸들러: 대기 중 confirm 있으면 controller.resolve("y")
       └─ finish("y"): audit 기록 + onApproved(req) → notifyGitPush(...) (git-feed!)
       └─ 봇이 <id>.decision 파일에 "y" 기록
   ─────────────────────────────────────────────────────────────
hook 스크립트
  └─ <id>.decision = "y" 읽음 → exit 0 (push 진행)
     <id>.decision = "n"/timeout → exit 2 (push 차단)
```

## 5. 리스크

1. **모든 push 차단 위험.** hook exit code 오류 / decision 파일 race → push 가
   영구 블록될 수 있음. → fail-open vs fail-closed 정책 결정 필요(기본: timeout =
   차단, 단 hook 자체 오류는 fail-open + 경고 로그).
2. **다중 동시 push.** 한 spawn 내 연속 push, 또는 병렬 spawn → `Map<id>` 키
   충돌 없게 id = hash(cmd+ts+pid). command 채널 질의는 건별.
3. **봇 재시작 중 대기 confirm 유실.** pending-confirms/ 파일은 디스크에 남지만
   in-memory controller 는 소실 → 재시작 시 pending 스캔 복구 필요.
4. **single-bot-process 불변식 의존.** dev-confirm 설계가 in-memory Map 전제 →
   다중 봇 프로세스 워크스페이스에선 파일 락 필요(현재 v0.7 lifecycle precheck 가
   단일 프로세스 보장).
5. **Claude Code hook 스키마 의존.** `PreToolUse` hook 입출력 포맷(stdin JSON,
   exit code 의미)은 Claude Code 버전에 묶임 → 버전 가드 + doctor 점검 추가.

## 6. Open Questions (v1.2.9 B.5 에서 이관)

1. **repo 식별** — push 가 어느 repo 인지: spawn `target_repo` 우선, 없으면 hook
   의 cwd 캡처. (`PushEvent.repoSlug` 입력.)
2. **다중 push** — 건별 메시지 vs 묶음. MVP: 건별.
3. **push 범위 커밋 요약** — `git log <remote>/<branch>..HEAD` 는 승인 시점 로컬
   기준 → force/부분 push 시 부정확 가능. best-effort + footnote.
4. **승인 의미론** — 게이트는 "승인" 시점만 관측(실제 push 성공은 hook exit 후
   specialist 내부). 알림 카피는 "✅ Push 승인"(곧 push 됨) 유지 — 사후 성공 확인은
   commit 폴링(v1.3.x).
5. **fail-open vs fail-closed** — 위 리스크 1. 사용자/보안 트레이드오프 결정 필요.
