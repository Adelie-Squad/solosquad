## 1. 배경 / 문제

v1.3.2 는 5개 1급 자산(skill·agent·workflow·goal·**schedule**)을 공통 "매니저" 추상으로
정렬했다. 그런데 정기 실행 자산이 코드 전반에서 **두 개의 용어로 쪼개져** 있었다:

- **routine** — `ROUTINES[]` 하드코딩 빌트인 잡(morning/evening brief, pm-compaction,
  system-housekeeping). `scheduler/routines.ts`, `runRoutineForProduct`, `loadRoutinePrompt`,
  `memory/routine-logs/`.
- **schedule** — v1.3.2 가 추가한 사용자 작성 잡(`schedules/<id>.yaml`). `scheduler/schedule-def.ts`,
  `ScheduleDef`, `getSchedulesDir`, CLI `solosquad schedules`, 별도 데몬 명령 `solosquad schedule`,
  수동 실행 `solosquad run-routine`.

동일 개념(node-cron 으로 도는 정기 잡)을 가리키는데 자산 종류에 따라 이름이 달라 학습 비용·검색
비용이 컸다. **통일 명사 = `cron`.**

### 2.2 CLI 통합 — 단일 `cron` 그룹

기존 3갈래(`schedule` 데몬 · `schedules` 관리 · `run-routine` 수동)를 하나로:

```
solosquad cron start              # 데몬 (구 `solosquad schedule`)
solosquad cron run [cron-id]      # 수동 실행 (구 `solosquad run-routine`)
solosquad cron list               # 빌트인 + 사용자 cron 일람
solosquad cron new <id>           # crons/<id>.yaml + <id>.md scaffold
solosquad cron show <id>          # 단건 + validation 상태
solosquad cron validate           # 사용자 cron 정의 검증
```

### 2.3 보존(의도적 비개명)

- **`scheduler` 서브시스템명·`startScheduler`** — cron 을 *돌리는* 데몬은 별개 개념. `src/scheduler/`
  디렉토리·`[Scheduler]` 로그 프리픽스 유지.
- **node-cron API**(`cron.schedule()`·`cron.validate()`) — 외부 라이브러리 계약.
- **아카이브 이벤트 enum `routine_log`** — `archive.sqlite` 에 저장된 데이터 계약(언더스코어 형).
  검색 표면(`solosquad memory search --kind routine_log`)도 그대로.
- **과거 migration 스크립트의 경로 문자열**(`.solosquad/routines` 생성 등) — 동결된 이력. `detect.ts`
  의 v0.1.x 레거시 마커 `["agents","routines","core"]` 도 *당시* 레이아웃이므로 유지.

### 2.4 마이그레이션 (1.3.2 → 1.3.3)

## 3. Part C — cron lifecycle CRUD + 라이브 반영 (✅ 구현 완료)

> **레퍼런스:** OpenClaw(`openclaw cron create/edit/enable·disable/run/list/show/remove`,
> 스케줄=cron식·`every 1h`·one-shot, pause≠delete, `show`=next-run/route 미리보기) +
> Hermes(`/cron add` 자연어·`pause`/`resume`/`run`/`remove`/`edit`, **id 또는 name**
> 대소문자무시·모호시 거부, isolated 세션 실행, `[SILENT]` 억제). 두 패턴을 SoloSquad
> 파일기반(yaml+md) 모델에 매핑.

기존엔 `cron new/list/show/validate`만 있어 **수정/시작·중지/삭제 경로가 부재**하고, 데몬이
startup 에만 정의를 읽어 변경이 재시작 전까지 무효였다. Part C 가 전 생애주기 + 무중단 반영을 채운다.

### 3.1 lifecycle CLI (신설)

```
solosquad cron new <id> [--cron <sched>] [--kind] [--channel]   # 친화적 스케줄 + next-run 미리보기
solosquad cron edit <ref> [--cron|--name|--kind|--channel]      # 필드 패치 → 자동 재검증
solosquad cron enable <ref> / disable <ref>                      # pause ≠ delete (정의·프롬프트 보존)
solosquad cron delete <ref> [--hard] [-y]                        # 기본 아카이브(crons/_archived/), --hard 영구삭제
```
- **참조 = id 또는 name**(대소문자무시, 모호하면 거부 — Hermes). 빌트인 4종은 코드 소유 → edit/disable/delete 거부.
- **친화적 스케줄**(`cron-schedule.ts`): cron식 passthrough + `@daily/@hourly/@weekly/@monthly` + `every <n>m|h|d`. one-shot(ISO/`20m`)은 별도 실행경로 필요 → 거부(비범위). 입력 시 사람이 읽는 readback(`describeSchedule`) + **next-run 미리보기**(`nextRun`, node-cron 권위값).
- **검증 회수:** new/edit 후 `validateCronDef` 자동 실행 — 저장과 동시에 게이트.

### 3.2 백킹 스토어 (`cron-def.ts`)

`writeCronDef`/`readCronDef`/`patchCronDef`/`setCronEnabled`/`deleteCronFiles`/`resolveCronRef` —
순수 파일 CRUD(dir 주입 가능 → hermetic 테스트). 쓰기 타깃은 **`getCronsWriteDir()`(=`<ws>/.solosquad/crons`)
로 고정** — getCronsDir(읽기 resolver, override 부재 시 번들 폴백)와 달리 번들 오염 불가(v1.3.2 §10 write-target 핀 정합).

### 3.3 라이브 반영 (무중단 hot-reload)

- `scheduler/index.ts` `reconcileUserCrons()` — 현 user task 전부 destroy 후 valid+enabled 정의만
  재등록(node-cron v4 `getTasks`/task `.destroy()`, `{ name, noOverlap, timezone }`). 멱등.
- chokidar 가 `.solosquad/crons` 감시 → 변경 시 디바운스 reconcile. `cron new/edit/enable/disable/delete`
  가 **데몬 재시작 없이** 반영. 빌트인 cron 은 무영향.

### 3.4 cron 도메인 마무리 (✅ 추가 — OpenClaw/Hermes 패턴 채택)

- **`cron run` user-cron 지원** — id 또는 name 으로 사용자 cron 도 수동 실행(빌트인 한정 해소).
  명시 ref 는 disabled 도 실행(활성화 전 테스트), `--all` 은 빌트인+enabled user cron.
- **`[SILENT]`/빈 출력 억제**(`isSilentResult`) — 출력이 비었거나 `[SILENT]` 로 시작하면 로그·메모리는
  남기되 채널 게시는 생략(조용한 폴러). Hermes `[SILENT]`/OpenClaw `NO_REPLY` 정합.
- **run 히스토리**(`cron-runlog.ts`) — 매 실행을 `<org>/memory/cron-runs.jsonl` 에 outcome 기록
  (ok|silent|error+ms). `cron runs [ref] [-n]` 조회 + `cron show` last-run 표면(OpenClaw `cron runs`).
- **dead-man's-switch** — `estimatePeriodMinutes`/`isOverdue` 가 cadence 추정 → 마지막 성공 실행이
  2× cadence 초과면 overdue. 일일 `system-housekeeping` 이 overdue cron 1줄 경보 게시(정상 시 조용).
- **one-shot `--at`** — `cron new <id> --at <ISO|"20m">` → `at` 필드 1회 실행 후 자동 삭제
  (delete-after-run, setTimeout 기반, `getNextRun` 대신 절대시각). `cron show`/`list` 가 one-shot 표시.

### 3.5 검증

✅ build/tsc green · ✅ `npm test` 897(신규 `cron-schedule`/`cron-lifecycle`/`cron-runlog` + one-shot
validate) · ✅ validate-bundled · 임시 워크스페이스 스모크(recurring + one-shot 전 lifecycle).

**비범위(후속):** sub-분 one-shot 정밀도(6-field), 결과 webhook 라우팅(OpenClaw `delivery.webhook`),

## 4. Part B — cron-manager 사용자 표면 (기획)

v1.x plan §4·§5·§4.3 흡수. v1.3.2 가 *저장 전 검증 게이트*를 깔았으니, v1.3.3 은 그 위에 **사용자가
직접 만드는 경로**를 얹는다("검증은 되는데 사용자가 못 만든다" 갭 해소).

### 3.1 cron 개인화 (구 §5 루틴 개인화)

- 디폴트 빌트인 cron(morning/evening brief + pm-compaction)을 *사용자별*로: 각자 timezone, 각자
  채널(`works-<handle>`). alice(KST 08:00) ≠ bob(PST 08:00) 비침범.
- **변경:** `CronConfig`/`CronDef` 에 `userHandle?`·`timezone?` 추가. `resolveCrons` 를 user yaml
  (`<org>/.solosquad/users/<handle>.yaml` 의 `crons:` 블록 + tz) 스캔으로 확장 → user × cron = N개
  node-cron entry 동적 등록(`cron.schedule(expr, fn, { timezone })`).
- **유지:** `system-housekeeping` 은 workspace-scoped(시스템 유지보수, 사용자 무관).
- broadcast cross-user feed(v0.8.0 §3.6 v2) 정합 — 개인 brief 는 각자 채널, workspace 요약만 broadcast.
- **손댈 파일:** `scheduler/crons.ts`·`scheduler/index.ts`(resolveCrons)·`cron-def.ts`·`util/config.ts`.

### 3.2 `/create` 트리거 (구 §4 save-as-skill → **create**)

- 메신저에서 `/create <name>` 슬래시 또는 자연어("지금 한 거 SKILL로 저장해") → 직전 N 메시지를
  컨텍스트로 SKILL 초안 생성 → SANDBOX → APPLIED.
- **엔진은 준비됨:** v1.3.2 가 `skill-author.ts → skill-manager.ts` 개명, 대화형 author 루프 보유.
  **남은 건 트리거 표면.**
- **변경:** `bot/slash-commands.ts` 화이트리스트에 `/create` 추가(현재 `/think /plan /build /review
  /ship /help /cancel /grant /revoke`) → `skill-manager` create 진입. Chief 자연어 의도 파싱.
- APPLIED 타깃 = org 레이어(`<org>/agents/<name>/SKILL.md`) — 번들 불변. 저장 직후 v1.3.2
  `validateSkill` 강화분(naming/conflict)이 게이트.
- **이름:** "save-as-skill" 대신 **`create`** — 매니저 공통 동사(§2.2 v1.3.2 PRD: create/review/
  validate/refine)와 일치. skill 뿐 아니라 cron/agent scaffold 도 같은 동사 표면.

### 3.3 freq miner alert (구 §4.3)

- 30일 빈도 마이너(`scheduler/freq-keyword-miner.ts`) 결과를 morning brief 에 *제안 한 줄*만 inline.
  **자동 등록 절대 금지** — 사용자 `/create` 명시 승인 시에만 APPLIED. 30일 거절 cooldown 유지.

> **Part B 우선순위:** §3.1 cron 개인화 = P0(멀티유저 실사용·저위험), §3.2 `/create` = P0(엔진
