# v0.8.4 — CLI Surface Reduction

> v1.0 정식 출시 전 **CLI 명령·플래그 표면적**을 정리하는 마지막 패치. `docs/policy/schema-stability.md` §4에 명시된 "Removing a flag is major" 약속이 v1.0에서 발효되므로, **이 슬롯이 비파괴적으로 플래그를 줄일 수 있는 마지막 기회**다. 여섯 축: (a) **`solosquad uninstall` 플래그 8→5** — 3-state mode를 `--mode <full|keep|archive-only>`로 단일화, `--scrub-content` 폐기, `--also-purge-backups`는 `solosquad backup purge`로 분리, (b) **`add repo --inspect` 별칭 제거** — `--dry-run`만 정식, (c) **`solosquad import --merge`/`--replace` → `--mode <merge|replace>`** — uninstall과 동일한 mode 패턴, (d) **`agent validate --corpus` 내부 이동** — 사용자 노출에서 빼고 dev-only test script로, (e) **`solosquad backup` subgroup 신설** — `migrate --list-backups`/`--delete-backup`/`uninstall --also-purge-backups` 3건을 `backup list|delete|purge`로 집결, (f) **`solosquad init` 워크스페이스 경로 prompt 정합** — 기존 walk-up 자동 감지가 사용자 의도와 어긋날 수 있는 케이스 대응(CWD 기본 + 명시적 확인). 부수: **v1.0 surface freeze 체크리스트** 박제 — v1.0 진입 시 어떤 표면이 SemVer 약속에 포함되는지 본 plan에서 최종 확정.

**로드맵:** v0.8.3 Onboarding UX → **v0.8.4 CLI Surface Reduction** → v0.9 안정화 → v1.0 정식 출시

**의존:** v0.7 (`uninstall` 플래그 도입), v0.8.1 (`import`·`archive` 표면 + api-stability §4 약속), v0.8.3 (`logs`·`migrate` 흐름도 — 본 패치에서 surface freeze 대상)

**기준 버전:** v0.8.4. v0.8.x 시리즈 *완료* 패치. v0.9 진입 전 마지막 표면 정리.

**분류:** CLI Refactor + Deprecation. 새 기능 추가 0. 표면적 축소 + deprecation alias 도입만.

**v0.8.4 부활 사유 (2026-05-15 박제 amendment):** 같은 날 directive에서 `v0.8.4 plan 폐기, v0.8 단일 plan으로 흡수`라 박제했으나, 그 폐기는 **메신저 UX polish** 한정. uninstall 플래그 매트릭스 정리·CLI 표면 reduction은 v0.8 메신저 영역과 결이 다르고, **api-stability §4 "Removing a flag is major"** 규칙 때문에 v1.0 진입 전 별도 슬롯이 필요. 따라서 v0.8.4를 *CLI Surface Reduction* 스코프로 재정의해서 부활. v0.8 메신저 polish는 여전히 `v0.8-multiuser-messenger.md` §3.8~§3.13으로 흡수, 본 plan과 무관.

| 문제 | 원인 |
|---|---|

## 2. 원칙

| 원칙 | 구현 |
|---|---|
| **mode 패턴 일관성** | N-state 의도를 boolean N-1개로 표현하지 않음. `--mode <a\|b\|c>` 단일 플래그로 통합 (uninstall, import 모두 적용) |
| **deprecation은 1 minor** | v0.8.4에서 alias 도입 + warning, v1.0에서 alias 제거. api-stability §4 약속에 정합 |
| **책임 분리** | 백업 관리는 backup 명령, 마이그레이션은 migrate, uninstall은 uninstall. 한 명령이 인접 영역 플래그 흡수 금지 |
| **dev-only는 environment variable 또는 internal script** | 사용자 노출 플래그(`--corpus`)는 줄이고, 회귀 테스트는 `npm run test:corpus` 같은 dev 채널로 |
| **v1.0 freeze 체크리스트 박제** | 본 plan §10에 v1.0 시점 surface 전체를 명시. publish 게이트 |

## 3. `solosquad uninstall` 플래그 매트릭스 정리 (8 → 5)

### 3.1 변경 후 플래그 (5개)

```
--mode <full|keep|archive-only>   (default: full)
--dry-run
-y, --yes
--force
--archive-path <path>
```

### 3.2 사용자 의도 매핑

| 의도 | 명령 |
|---|---|
| 깨끗하게 다 지우기 (기본) | `solosquad uninstall` |
| 재설치 위해 workflows/memory/knowledge 남기기 | `solosquad uninstall --mode keep` |
| 새 머신 마이그레이션 (아카이브만, cleanup 스킵) | `solosquad uninstall --mode archive-only` |
| CI/스크립트 자동화 | `solosquad uninstall --yes` |
| 뭐가 지워질지 먼저 보기 | `solosquad uninstall --dry-run` |

### 3.3 제거 / 이동 사유

| 플래그 | 처리 | 사유 |
|---|---|---|
| `--archive-only` | `--mode archive-only`로 흡수, deprecated alias로 1 minor 유지 | mode 일원화 |
| `--keep-workspace` | `--mode keep`로 흡수, deprecated alias로 1 minor 유지 | mode 일원화 |
| `--also-purge-backups` | **제거** → `solosquad backup purge`로 분리 (§7). v0.8.4에서 호출 시 deprecation warning + `solosquad backup purge` 안내 후 hand-off | uninstall의 책임 아님 |
| `--scrub-content` | **즉시 제거**. 사용 시나리오 불명확 + best-effort regex 신뢰도 낮음. v0.8.x에서만 존재했으므로 SemVer 안전 (v1.0 약속 발효 전) | speculative 플래그 정리 |

### 3.4 모호 케이스 처리

- `--mode` × `--dry-run` 조합: 모두 valid (dry-run이 우선, mode는 "어떤 모드를 dry-run할지" 결정)
- `--mode keep` 선택 시 프롬프트: **"이 모드에선 봇 토큰이 디스크에 남습니다. REVOKE는 별도 확인 필요"** 명시 (현재 confirm 메시지 보강)
- deprecated alias와 `--mode` 동시 지정 시: deprecation warning + alias 우선, `--mode` 무시 (1 minor 후 alias 제거되면 에러)

### 3.5 코드 변경 위치

- `src/cli/index.ts:455-469` — flag 정의 갱신
- `src/cli/uninstall.ts:34-43` — `UninstallOpts` interface 갱신 (`mode?: "full" | "keep" | "archive-only"`)
- `src/cli/uninstall.ts:55-63` — mode 디코딩 로직 (3-state → 1 enum)
- `src/cli/uninstall.ts:92-109` — confirm 프롬프트 메시지 mode-aware 재작성
- 신규: deprecation warning 헬퍼 (`src/util/deprecation.ts` 신설 가능)

## 5. `solosquad import` mode 패턴 정합

### 5.1 변경

```
이전: --merge (default) + --replace
변경: --mode <merge|replace>  (default: merge)
```

### 5.2 사유

- `--merge`와 `--replace`는 mutually exclusive boolean = uninstall과 동일한 매트릭스 문제
- 둘 다 켜면? — 현재 코드(`src/cli/import.ts`)는 `--replace` 우선이지만 사용자 명시 안 됨
- uninstall과 같은 패턴으로 통일하면 사용자 학습 비용↓

### 5.3 deprecation 정책

- v0.8.4: `--merge`·`--replace` deprecated alias로 유지, warning
- v1.0: alias 제거, `--mode`만 남김

### 5.4 코드 변경 위치

- `src/cli/index.ts:517-518` — flag 갱신
- `src/cli/import.ts` — opts 디코딩 로직 갱신

## 7. `solosquad backup` subgroup 신설

### 7.1 신설 명령

```
solosquad backup list                     # 백업 목록 (~/.solosquad-backups/)
solosquad backup delete <id>              # 특정 백업 삭제
solosquad backup purge [--keep-recent N]  # 디렉터리 전체 정리 (옵션: 최근 N개 유지)
```

### 7.2 흡수 대상

| 기존 위치 | 신규 위치 |
|---|---|
| `solosquad migrate --list-backups` | `solosquad backup list` |
| `solosquad migrate --delete-backup <id>` | `solosquad backup delete <id>` |
| `solosquad uninstall --also-purge-backups` | `solosquad backup purge` |

### 7.3 사유

- 백업은 cross-cutting 자원 — migrate / uninstall 양쪽이 모두 다룸. 한 명령에 묶이는 게 자연스러움
- migrate 명령이 너무 다중 책임: 적용·롤백·백업 조회·백업 삭제 4축. backup 책임 분리 시 migrate는 적용·롤백 2축만
- uninstall도 마찬가지 — backup purge는 uninstall과 무관한 housekeeping

### 7.4 deprecation 정책

- v0.8.4: 기존 플래그들 호출 시 deprecation warning + 신규 명령 안내. 동작은 유지(forward-compat)
- v1.0: 기존 플래그 제거

### 7.5 코드 변경 위치

- 신규: `src/cli/backup.ts` (대부분 `src/cli/migrate.ts`에서 분리 + `src/lifecycle/cleanup.ts`의 backup purge 로직 호출)
- `src/cli/index.ts` — `backupGroup` 신설 + 기존 플래그 deprecated 처리
- `src/cli/migrate.ts` — `listBackupsCommand`·`deleteBackupCommand` export 유지 (backup.ts에서 import)

## 8. `solosquad init` 워크스페이스 경로 prompt

### 8.1 현재 동작

- `initCommand()` → `getWorkspaceRoot()` → `findWorkspaceRoot(cwd) ?? cwd`
- `findWorkspaceRoot`는 CWD에서 **부모 디렉터리로 walk-up**하면서 `.solosquad/` 마커를 찾음 (`src/migrations/detect.ts:findWorkspaceRoot`)
- 결과: 사용자가 `~/projects/new-product/`에서 `solosquad init`을 실행해도, 부모 어딘가(예: `~/`)에 기존 `.solosquad/`가 있으면 **그 부모를 워크스페이스로 잡음** — 신규 init 의도와 어긋남

### 8.2 문제 시나리오

```
~/.solosquad/              # alice가 옛날에 만든 워크스페이스
~/projects/new-saas/       # alice의 새 프로젝트, 여기서 init 실행
$ solosquad init
→ 워크스페이스가 ~/에 생성됨 (예상: ~/projects/new-saas/)
```

`init`은 신규 워크스페이스 생성 의도가 강한 명령인데, 자동 walk-up이 사용자에게 *어디에 만들지* 묻지 않고 결정해버림.

### 8.3 변경

`initCommand()` 진입 직후, 다음 분기 추가:

1. CWD 자체에 `.solosquad/`가 이미 있으면: 기존 워크스페이스 재초기화로 간주, 안내 후 계속 (현재 동작 유지)
2. 부모 디렉터리에 `.solosquad/`가 발견되면: **선택지 프롬프트**
   - "기존 워크스페이스 사용: `<found-path>`"
   - "현재 경로에 새로 생성: `<cwd>` (default)"
   - "다른 경로 직접 입력"
3. 어디서도 발견 안 되면: **현재 경로 확인 프롬프트** ("Initialize workspace at: `<cwd>`")

### 8.4 사유

- `init`은 *생성* 명령, *탐색* 명령이 아님 — 자동 walk-up이 의도와 어긋날 위험
- 다른 CLI(`bot`, `schedule`, `status`) 등은 *기존 워크스페이스 탐색*이 자연스러우므로 walk-up 유지
- 즉, **`init`만 walk-up 동작을 비활성화하고 명시 확인**으로 분리

### 8.5 코드 변경 위치

- `src/cli/init.ts:399` — `getWorkspaceRoot()` 호출을 새 헬퍼로 교체
- `src/cli/init.ts` — 새 함수 `resolveInitWorkspace()`: 위 3-way 분기 prompt
- `src/util/paths.ts` — (선택) `findWorkspaceRoot`을 init에서만 명시 호출하도록 의존 정리, 또는 init.ts 내부 helper로 유지
- 코드 변경 영역은 좁음 (단일 함수 분기 + 1 prompt 추가)

### 8.6 backward-compat

- `--yes`(없음, init은 wizard 명령) 대신 환경변수 `SOLOSQUAD_INIT_PATH=<path>`로 CI/스크립트에서 비대화형 지정 가능 (옵션, v0.8.4 1차에는 미포함 — v0.9에서 필요 시 추가)
- 기존 사용자에게는 *추가 확인 step 1개*만 보임 — destructive 변경 아님

## 9. (deferred) `solosquad migrate` 컨벤션 역전

### 8.1 현황

- `solosquad migrate --dry-run` (default) + `--apply` — 다른 CLI는 *기본 dry-run* 아님
- 예: `uninstall`은 기본이 destructive, `--dry-run`이 opt-in. `add repo`도 마찬가지
- migrate만 *기본 안전, --apply가 opt-in* — 컨벤션 불일치

### 8.2 결정: **v0.8.4에서는 변경 안 함**

사유:
- migrate는 *되돌리기 어려운 작업* — 사용자가 무심코 `solosquad migrate` 쳤다가 schema 변환되면 복구 시간이 큼. 기본 dry-run은 안전 기본값으로 합리적
- 일관성 < 안전성. 컨벤션을 위해 안전 default를 깨는 건 ROI 음수
- `docs/policy/schema-stability.md` §4가 발효되면 이 컨벤션도 freeze — v1.0에서도 그대로 유지

### 8.3 후속 작업

- `docs/policy/schema-stability.md`에 *"migrate is intentionally dry-run-by-default — safety convention exception"* 절 추가
- master-guide / CHANGELOG에서도 명시

## 10. Deprecation 정책 정리

### 10.1 v0.8.4 deprecated 목록

| 플래그 / 명령 | 대체 | 동작 (v0.8.4) | v1.0 |
|---|---|---|---|
| `uninstall --archive-only` | `uninstall --mode archive-only` | warning + 동작 유지 | 제거 |
| `uninstall --keep-workspace` | `uninstall --mode keep` | warning + 동작 유지 | 제거 |
| `uninstall --also-purge-backups` | `backup purge` | warning + 동작 유지 (내부에서 backup purge 호출) | 제거 |
| `uninstall --scrub-content` | (대체 없음, speculative 폐기) | **즉시 제거** (v0.8.x 한정 surface, v1.0 약속 발효 전) | 이미 제거 |
| `add repo --inspect` | `add repo --dry-run` | warning + 동작 유지 | 제거 |
| `import --merge` (default 명시 시) | `import --mode merge` | warning + 동작 유지 | 제거 |
| `import --replace` | `import --mode replace` | warning + 동작 유지 | 제거 |
| `agent validate --corpus` | `npm run test:corpus` | **즉시 제거** (dev-only, 사용자 surface 아님) | 이미 제거 |
| `migrate --list-backups` | `backup list` | warning + 동작 유지 | 제거 |
| `migrate --delete-backup <id>` | `backup delete <id>` | warning + 동작 유지 | 제거 |

### 10.2 Deprecation warning 형식

```
[deprecated] --archive-only will be removed in v1.0. Use --mode archive-only instead.
```

- `stderr`로 출력 (스크립트 stdout 파이프라인 깨지지 않게)
- 환경변수 `SOLOSQUAD_NO_DEPRECATION_WARN=1`로 silenceable (CI 노이즈 억제용)

### 10.3 헬퍼 위치

- `src/util/deprecation.ts` 신설 — `warnDeprecated(oldFlag, newFlag, removalVersion)`
- 모든 CLI command action에서 deprecated alias 디코딩 직후 호출

## 11. v1.0 Surface Freeze 체크리스트

> **본 절은 v1.0 진입 시 SemVer 약속에 포함되는 CLI surface의 최종 박제다. v0.8.4 publish 시점에 이 체크리스트가 `solosquad --help` 출력과 1:1 일치해야 한다.**

### 11.1 명령 그룹 (v1.0 약속 대상)

|---|---|---|
| Top-level | `init`, `bot`, `schedule`, `status`, `update`, `doctor`, `run-routine`, `sync`, `migrate`, `uninstall`, `logs`, `import` | 단일 명령 12개 |
| `readiness` | `check` | 1 subcommand |

| `memory` | `search`, `stats` | 2 subcommands |

| `messenger` | `broadcast-handover` | 1 subcommand (확장 슬롯) |
| `archive` | `verify`, `info`, `list` | 3 subcommands |
| **`backup`** (신규 v0.8.4) | `list`, `delete`, `purge` | 3 subcommands |

총: 12 top-level + 30 subcommands across 11 groups = **42 사용자 호출 가능 명령** (구현 후 `src/cli/index.ts`의 `.command()` 호출 53건에는 그룹 등록 11건도 포함되어 합산 53. 그룹 자체는 호출 불가하므로 surface 약속 대상은 42)

### 11.2 v1.0 freeze 후 정책

- 명령 추가: minor bump
- 명령 제거 또는 rename: major bump (v2.0 이상)
- 플래그 추가: minor bump (default 동작 변화 없을 시)
- 플래그 제거: major bump
- 플래그 default 변경: major bump (예외 — security fix는 minor + CHANGELOG 명시)
- subcommand rename: major bump

### 11.3 사용자 surface 측정 (v0.8.3 → v0.8.4)

| 지표 | v0.8.3 | v0.8.4 | Δ |
|---|---|---|---|
| Top-level 명령 | 12 | 12 | 0 |
| Subcommand 합계 | 27 | 30 | +3 (`backup` 그룹 3개 신설) |
| 사용자 노출 플래그 합계 | TBD (구현 시 측정) | TBD | **순 감소 목표** |
| Deprecated alias 수 | 0 | 10 | +10 (1 minor 후 제거) |

→ 명령 수는 +3이지만 *분산되어 있던 백업 플래그 3건*을 흡수하므로 *논리적 surface*는 줄어듦. 사용자 노출 플래그 수치는 구현 시 측정 + CHANGELOG에 박제.
