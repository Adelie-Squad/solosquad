# Part A — Discord Application ID 자동 감지 + Invite URL 부활

## A.1 발견 경위

dogfood (2026-06-01) 에서 다음 증상:

1. `solosquad init` Discord 경로 완주 — 토큰 입력, handle 확인까지 정상
2. **앱(Application) ID 를 묻는 단계가 없음** — 사용자는 Developer Portal 에서 Application ID 를 복사해 둔 채 입력할 곳을 못 찾음
3. init 마지막에 **"Discord Invite URL" 블록이 아예 출력 안 됨** — 브라우저도 안 열림
4. 이후 `solosquad discord invite-url` 도 `✗ No bot_application_id found` 로 실패

→ v1.2.6 PRD §3.1 line 76/120-122 가 약속한 *"application_client_id prompt → invite URL 자동 합성 → 브라우저 open → click 1회"* 흐름의 **앞단 두 단계가 통째로 부재**.

## A.2 근본 원인 — 존재하지 않는 API 필드 1개

`src/cli/init.ts` 의 `fetchBotIdentity()` (discord 분기):

```ts
const res = await fetch("https://discord.com/api/v10/users/@me", { ... });
const body = (await res.json()) as { id?; username?; application_id? };
return { handle, botUserId: body.id, appId: body.application_id };  // ← 항상 undefined
```

**Discord 의 `GET /users/@me` 는 봇의 *User* 객체를 반환하며, 이 객체에는 `application_id` 필드가 없다.** (봇 User 객체 = `id` + `username` + avatar 등. application id 는 별도 리소스.) 따라서:

| 단계 | 결과 |
|---|---|
| `body.application_id` | 언제나 `undefined` |
| `BotIdentity.appId` | 언제나 `undefined` |
| init Step 4 invite URL 블록 (`if (... && identityChoice?.bot.appId)`) | **항상 skip** → URL 출력 / 브라우저 open 안 됨 |
| `user.yaml.bot_application_id` | `undefined` 저장 |
| 이후 `solosquad discord invite-url` | `bot_application_id` 없음 → 실패 |

게다가 **prompt fallback 도 없었다** — 자동 감지가 조용히 실패하면 사용자에게 물어볼 단계 자체가 코드에 없어, "앱 ID 를 안 물어본다" 는 증상으로 직결.

같은 결함이 `src/cli/doctor-discord.ts` Hop 2 (`liveAppId = me.application_id ?? null`) 에도 존재 → doctor 의 invite-url 힌트 + `bot_application_id missing` 경고도 항상 죽어있었음.

## A.3 수정

### 핵심 사실 2가지

- **정식 엔드포인트**: `GET /oauth2/applications/@me` (Bot 토큰 auth) 가 Application 객체를 반환하며 `.id` 가 곧 application(client) id.
- **불변식**: Discord 봇은 봇 User id == Application id (동일 snowflake). 따라서 위 호출이 실패해도 봇 user id 로 안전하게 폴백 가능.

### A.3.1 — init.ts: 자동 감지 + 명시적 prompt

| # | 작업 |
|---|---|
| A.3.1.a | `fetchDiscordApplication(token)` 신규 — `GET /oauth2/applications/@me` 로 `{ id, ownerUserId? }` 조회, 실패 시 `null`. |
| A.3.1.b | `fetchBotIdentity()` discord 분기 — 존재하지 않는 `body.application_id` 의존 제거. `appId = (await fetchDiscordApplication(token))?.id ?? body.id` (봇 user id 폴백). |
| A.3.1.c | `promptHandleSelection()` 에 **Discord Application(Client) ID 확인 prompt** 추가 (PRD §3.1). 자동 감지값을 기본값으로 Enter 수락. 감지 실패 시 *Developer Portal → General Information → Application ID* 붙여넣기 안내. `/^\d{17,20}$/` 검증, 빈 입력 skip 허용. discord-only. |
| A.3.1.d | **owner User ID 자동 prefill** — `application.owner.id` 를 owner-only 게이트 prompt 의 default 로 사용 (team app 은 `owner` 가 합성 team user 라 skip). |

### A.3.2 — doctor-discord.ts: Hop 2 app id 채움

| # | 작업 |
|---|---|
| A.3.2.a | `fetchApplicationId(token)` helper 신규 (동일 엔드포인트). |
| A.3.2.b | Hop 2 `liveAppId = (await fetchApplicationId(token)) ?? me.id`. `UsersMeBody.application_id` 필드 제거 (죽은 필드). → Hop 3 `bot_application_id missing` 경고 + Hop 4 invite URL 힌트 실작동. |

# Part B — Git 이벤트 전용 채널 (`git-<handle>`)

> **상태: "1안" 구현 완료** (2026-06-04). 채널 배선(B.3.1)·설정 토글(B.3.3)·마이그레이션(B.3.4)은 **라이브**. push 알림(B.3.2)은 포매터/emit 모듈(`src/bot/git-event-notify.ts`)과 `createDevConfirm` 의 `onApproved` hook 까지 **코드 완비** 됐으나, **dev-confirm 게이트가 dormant**(정의만 존재, spawn 경로에 미연결 — `createDevConfirm`/`detectSensitiveCommand` 가 production 에서 호출되지 않음)라 실제 알림은 아직 발사되지 않는다(inert). 게이트 **라이브화 + 알림 실작동**은 `docs/prd/v1.3.0-dev-confirm-gate-live.md` 에서 설계 — 그때 `onApproved` 에 `notifyGitPush` 를 연결하면 알림이 자동 작동한다.

## B.1 동기

유저 repo 에서 발생하는 commit/push 는 현재 **어디로도 알림이 가지 않는다.** 에이전트가 작업을 마치고 push 하면, 그 사실은 PM 세션의 일반 대화에 섞이거나 (dev-confirm 확인 메시지로만) 보이고, "코드가 origin 에 올라갔다"는 신호가 별도로 모이지 않는다. 솔로 파운더 입장에서 *"오늘 내 봇이 무엇을 push 했나"* 를 한 채널에서 시간순으로 보고 싶다.

→ **`git-<handle>` 전용 채널** 신설. command(입력) / works(산출물 카드·스레드) 와 분리된 **VCS 이벤트 피드**.

## B.2 범위 결정 — push-only, dev-confirm 후킹 (MVP)

이벤트 소스로 세 후보를 검토(폴링 / git hooks / dev-confirm 후킹) 후 **dev-confirm 후킹**을 채택:

| 후보 | 장점 | 단점 | 채택 |
|---|---|---|---|
| dev-confirm 후킹 | 가장 적은 코드. push 가 *이미* 게이트를 통과하므로 cmd 문자열·repo·user 가 손에 있음. 폴링/훅 인프라 0. | push 만 잡음. 수동 로컬 commit 미포착. | ✅ |
| repo 폴링 | commit 까지 포착. | 스케줄러 폴링 신설, last-seen 상태 관리, 다중 repo 부하. | ✕ (v1.3+) |
| git hooks 설치 | 가장 정확. | 유저 repo 에 침습, 크로스플랫폼 훅 셸 이슈. | ✕ |

**MVP 정의:** 에이전트 주도 `git push` 가 dev-confirm 게이트에서 `y` 로 승인되는 순간, 그 push 를 `git-<handle>` 채널에 1건의 메시지로 기록한다. (commit 단독은 v1.3 폴링에서 다룬다 — 본 채널 description 에 "현재 push 이벤트 기준" 명시.)

## B.5 Open Questions

1. **repo 식별** — dev-confirm `cmd` 만으로는 어느 repo 의 push 인지 모호(`cmd` 에 cwd 없음). spawn 의 `target_repo` 를 dev-confirm request 에 실어 보낼지, 아니면 `git push` 실행 cwd 를 cmd 래퍼에서 캡처할지. → MVP 는 `target_repo` 우선, 없으면 org 기본 repo.
2. **다중 push** — 한 spawn 에서 연속 push 시 N건 메시지 vs 1건 묶음. MVP: 건별.
3. **push 범위 커밋 요약** — `git log <remote>/<branch>..HEAD` 는 승인 시점 로컬 기준. ahead 커밋이 정확히 push 대상이 아닐 수 있음(force/부분 push). MVP: best-effort, 부정확 가능성 footnote.
4. **Slack 대응** — 본 plan 은 discord-only. Slack 은 채널 명명·생성 모델이 달라 별도(`git-<handle>` Slack 채널) 후속.

> **이관:** OQ 1~3 은 push 알림이 *실제 발사*되는 시점(= dev-confirm 게이트 라이브화)에야 의미를 갖는다. 따라서 게이트 라이브화 설계와 함께 **`docs/prd/v1.3.0-dev-confirm-gate-live.md`** 로 이관해 다룬다. v1.2.9 의 emit 모듈(`git-event-notify.ts`)은 이들을 입력 파라미터(`PushEvent`: repoSlug / remote / branch / commits)로 받도록 일반화해 두었으므로, 라이브화 시 값만 채우면 된다.

## C-1 동기

두 갈래의 dogfood 피드백:

1. **"Chief 가 지금 자기가 CLI 에서 얘기하는지 Discord 에서 얘기하는지 아느냐"** — 코드 조사 결과 **모름**. 메시지가 어댑터→`bot/index.ts`→`ChiefRunner` 로 흐르는 동안 출처(platform/source)가 단계마다 버려졌다. `MessageContext` 에 surface 필드가 없고, `ChiefCall` 에도, system prompt 주입에도 없었음.
2. **messenger 응답 품질** — Chief 응답이 통째로 \`\`\` 코드블록으로 감싸져 Discord/Slack 의 native markdown(링크·굵게·inline code)이 죽고 로그 덤프처럼 읽힘. 말 끝마다 `-{이름}` 서명이 붙고, 후속 질문이 산발적으로 쪼개져 옴.

추가로, **터미널에서 Chief 와 직접 대화할 방법 자체가 없었다** — Chief 도달 경로가 messenger 어댑터 단 하나(`bot/index.ts:126` 가 유일한 production 호출처). `solosquad bot` 은 봇을 *켜는* 명령이지 대화 명령이 아님.

## C-2 surface 배선 (어댑터 → Chief system prompt)

surface 신호를 한 줄로 끝까지 흘려보낸다. 5개 지점:

| # | 파일 | 변경 |
|---|---|---|
| C-2.a | `src/messenger/base.ts` | `ChiefSource = "discord" \| "slack" \| "cli"` 타입 + `MessageContext.source` 필드 신설. |
| C-2.b | `discord-adapter.ts` / `slack-adapter.ts` / `discord-chat-slash.ts` | 각 `MessageContext` 구현이 자기 `source`(discord/slack) 를 채움. |
| C-2.c | `src/bot/index.ts` | `chiefRunner.handleUserMessage({ … source: ctx.source })` 로 전달. |
| C-2.d | `src/bot/chief-runner.ts` `ChiefCall` | `source?: ChiefSource` 필드(optional — 미지정 시 messenger 로 fallback, back-compat). |
| C-2.e | `chief-runner.ts` `resolveSourceHint()` | source → system prompt `[surface]` 힌트 1줄 주입(`appendSystemPrompt` 에 `chiefIdentity + sourceHint + …`). messenger 면 no-code-block + inline 질문 + 간결 규칙, cli 면 경량 힌트. cache-friendly(같은 source → 같은 문자열). |

## C-3 터미널 `chat` 명령

- `src/cli/index.ts` 에 `chat` 커맨드 등록.

# Part D — 진행 중 작업 취소 (`/cancel`)

> **상태: 구현 완료** (2026-06-04).

## D-1 동기

dogfood: *"디스코드에서 chief 에게 작업을 보냈는데 잘못 보냈을 때 취소가 안 된다."*
코드 확인 결과 **취소 수단이 전무**했다 — 진행 중 turn 은 `SessionMutex` 가 직렬화
하므로 두 번째 메시지를 보내면 *취소가 아니라 큐에 쌓여 대기*했다. spawn 된
`claude` 자식 프로세스를 죽이는 `stream.abort()` 핸들은 존재했으나 **아무 데도
연결돼 있지 않았다**(타임아웃/봇 종료 시에만 kill).

## D-2 설계 — `/cancel` 슬래시 (메신저 + 터미널 동일)

| # | 파일 | 작업 |
|---|---|---|
| D-2.a | `src/bot/slash-commands.ts` | `/cancel` 인식 → `SlashHandlingResult.cancel = true`(shortCircuit, PM 미전달). `/help` 텍스트에 `/cancel` 추가. |
| D-2.b | `src/bot/chief-runner.ts` | `inflight: Map<\`${orgSlug}:${userId}\`, {stream, cancelled}>`. `invokeWithSessionRecovery` 가 stream 을 try/finally 로 등록·해제. `cancelTurn(orgSlug, userId)` → `entry.cancelled=true` + `stream.abort()`(mutex 무관, 큐 안 탐). `ChiefReply.aborted`/`InternalTurnResult.aborted` 전파. |
| D-2.c | `src/bot/index.ts` | slash 선처리에서 `cancel` 이면 `chiefRunner.cancelTurn(...)` 호출 + 확인 메시지(세션 mutex *이전*). `reply.aborted` 면 부분 reply·task card 억제(단, 부분 spawn 이 파일을 건드렸을 수 있어 snapshot 은 유지). |
| D-2.d | `src/cli/chat.ts` | non-blocking REPL — `busy` 플래그로 일반 입력은 한 turn 씩, 작업 중에도 `/cancel` 만 통과시켜 `cancelTurn` 호출. `reply.aborted` 면 부분 출력 억제. |

**핵심 불변식:** `/cancel` 은 **세션 mutex 를 거치지 않는다.** 거치면 자신이 취소하려는
turn 뒤에서 대기하게 되므로, `bot/index.ts` 의 slash 선처리 단계(mutex 진입 전)에서
가로채 `cancelTurn` 을 직접 호출한다.

## D-3 동작

- 디스코드: command 채널에 `/cancel` → 🛑 확인 메시지, 진행 중 spawn 즉시 중단.
- 터미널 `solosquad chat`: 작업 중 `you>` 에 `/cancel` 입력 가능(REPL 이 입력을 막지
  않음). 진행 중 turn 없으면 "취소할 작업이 없습니다".
- 취소된 turn 의 (부분) 응답은 표시하지 않는다 — `aborted` 플래그로 억제.

## D-4 비범위

- spawn 된 specialist *내부*의 개별 도구 호출 단위 취소(현재는 turn 전체 abort).
- 취소 시점까지의 부분 산출물 롤백 — snapshot 은 남기되 자동 revert 는 하지 않음
  (`solosquad rollback` 수동 사용).
