# Part A — Discord Application ID 자동 감지 + Invite URL 부활

## A.1 발견 경위

dogfood (2026-06-01) 에서 다음 증상:

1. `solosquad init` Discord 경로 완주 — 토큰 입력, handle 확인까지 정상
2. **앱(Application) ID 를 묻는 단계가 없음** — 사용자는 Developer Portal 에서 Application ID 를 복사해 둔 채 입력할 곳을 못 찾음
3. init 마지막에 **"Discord Invite URL" 블록이 아예 출력 안 됨** — 브라우저도 안 열림
4. 이후 `solosquad discord invite-url` 도 `✗ No bot_application_id found` 로 실패

→ v1.2.6 PRD §3.1 line 76/120-122 가 약속한 *"application_client_id prompt → invite URL 자동 합성 → 브라우저 open → click 1회"* 흐름의 **앞단 두 단계가 통째로 부재**.

## A.2 근본 원인 — 존재하지 않는 API 필드 1개

`src/cli/init.ts` 의 `fetchBotIdentity()` (discord 분기):

```ts
const res = await fetch("https://discord.com/api/v10/users/@me", { ... });
const body = (await res.json()) as { id?; username?; application_id? };
return { handle, botUserId: body.id, appId: body.application_id };  // ← 항상 undefined
```

**Discord 의 `GET /users/@me` 는 봇의 *User* 객체를 반환하며, 이 객체에는 `application_id` 필드가 없다.** (봇 User 객체 = `id` + `username` + avatar 등. application id 는 별도 리소스.) 따라서:

| 단계 | 결과 |
|---|---|
| `body.application_id` | 언제나 `undefined` |
| `BotIdentity.appId` | 언제나 `undefined` |
| init Step 4 invite URL 블록 (`if (... && identityChoice?.bot.appId)`) | **항상 skip** → URL 출력 / 브라우저 open 안 됨 |
| `user.yaml.bot_application_id` | `undefined` 저장 |
| 이후 `solosquad discord invite-url` | `bot_application_id` 없음 → 실패 |

게다가 **prompt fallback 도 없었다** — 자동 감지가 조용히 실패하면 사용자에게 물어볼 단계 자체가 코드에 없어, "앱 ID 를 안 물어본다" 는 증상으로 직결.

같은 결함이 `src/cli/doctor-discord.ts` Hop 2 (`liveAppId = me.application_id ?? null`) 에도 존재 → doctor 의 invite-url 힌트 + `bot_application_id missing` 경고도 항상 죽어있었음.

## A.3 수정

### 핵심 사실 2가지

- **정식 엔드포인트**: `GET /oauth2/applications/@me` (Bot 토큰 auth) 가 Application 객체를 반환하며 `.id` 가 곧 application(client) id.
- **불변식**: Discord 봇은 봇 User id == Application id (동일 snowflake). 따라서 위 호출이 실패해도 봇 user id 로 안전하게 폴백 가능.

### A.3.1 — init.ts: 자동 감지 + 명시적 prompt

| # | 작업 |
|---|---|
| A.3.1.a | `fetchDiscordApplication(token)` 신규 — `GET /oauth2/applications/@me` 로 `{ id, ownerUserId? }` 조회, 실패 시 `null`. |
| A.3.1.b | `fetchBotIdentity()` discord 분기 — 존재하지 않는 `body.application_id` 의존 제거. `appId = (await fetchDiscordApplication(token))?.id ?? body.id` (봇 user id 폴백). |
| A.3.1.c | `promptHandleSelection()` 에 **Discord Application(Client) ID 확인 prompt** 추가 (PRD §3.1). 자동 감지값을 기본값으로 Enter 수락. 감지 실패 시 *Developer Portal → General Information → Application ID* 붙여넣기 안내. `/^\d{17,20}$/` 검증, 빈 입력 skip 허용. discord-only. |
| A.3.1.d | **owner User ID 자동 prefill** — `application.owner.id` 를 owner-only 게이트 prompt 의 default 로 사용 (team app 은 `owner` 가 합성 team user 라 skip). |

### A.3.2 — doctor-discord.ts: Hop 2 app id 채움

# Part B — Git 이벤트 전용 채널 (`git-<handle>`)

## B.1 동기

유저 repo 에서 발생하는 commit/push 는 현재 **어디로도 알림이 가지 않는다.** 에이전트가 작업을 마치고 push 하면, 그 사실은 PM 세션의 일반 대화에 섞이거나 (dev-confirm 확인 메시지로만) 보이고, "코드가 origin 에 올라갔다"는 신호가 별도로 모이지 않는다. 솔로 파운더 입장에서 *"오늘 내 봇이 무엇을 push 했나"* 를 한 채널에서 시간순으로 보고 싶다.

→ **`git-<handle>` 전용 채널** 신설. command(입력) / works(산출물 카드·스레드) 와 분리된 **VCS 이벤트 피드**.

## B.2 범위 결정 — push-only, dev-confirm 후킹 (MVP)

이벤트 소스로 세 후보를 검토(폴링 / git hooks / dev-confirm 후킹) 후 **dev-confirm 후킹**을 채택:

| 후보 | 장점 | 단점 | 채택 |
|---|---|---|---|
| dev-confirm 후킹 | 가장 적은 코드. push 가 *이미* 게이트를 통과하므로 cmd 문자열·repo·user 가 손에 있음. 폴링/훅 인프라 0. | push 만 잡음. 수동 로컬 commit 미포착. | ✅ |
| repo 폴링 | commit 까지 포착. | 스케줄러 폴링 신설, last-seen 상태 관리, 다중 repo 부하. | ✕ (v1.3+) |
| git hooks 설치 | 가장 정확. | 유저 repo 에 침습, 크로스플랫폼 훅 셸 이슈. | ✕ |

**MVP 정의:** 에이전트 주도 `git push` 가 dev-confirm 게이트에서 `y` 로 승인되는 순간, 그 push 를 `git-<handle>` 채널에 1건의 메시지로 기록한다. (commit 단독은 v1.3 폴링에서 다룬다 — 본 채널 description 에 "현재 push 이벤트 기준" 명시.)

### B.3.2 이벤트 소스 — dev-confirm 후킹

dev-confirm 게이트(`src/bot/dev-confirm.ts`)는 `git push` / `gh pr merge` / `gh pr close` 를 가로채 PM→유저 확인을 받고 `<org>/memory/dev-confirmations.jsonl` 에 audit 한다. 여기에 **push 승인 sink** 를 건다:

| # | 작업 |
|---|---|
| B.3.2.a | 결정이 `y` 이고 `cmd` 가 `git push` 로 시작하면, push 알림 emit. (`gh pr merge`/`close` 는 본 채널 범위 밖 — works 카드 유지.) |
| B.3.2.b | `cmd` 에서 remote/branch 파싱 (`git push <remote> <branch>`); 미지정 시 `(default)`. repo 식별은 spawn 의 `target_repo` 또는 cwd 에서 해석 (B.5 open question). |
| B.3.2.c | 메시지 포매터 신규 — `<repo-slug>` · `<branch>` → `<remote>` · push 한 user(handle) · ISO 시각. 가능하면 push 범위 커밋의 `git log --oneline <remote>/<branch>..HEAD` 요약 N건 첨부. |
| B.3.2.d | `adapter.sendToChannel(config, \`git-${handle}\`, formatted)` 호출. 봇 프로세스 컨텍스트에서 호출 (dev-confirm controller 가 이미 봇-프로세스 state). |

> **후킹 위치 정밀화:** dev-confirm 은 *승인* 시점만 안다(실제 push 성공은 spawn 안에서 발생, 게이트는 비관찰). 따라서 MVP 알림은 **"승인된 push"** 의미론 — "곧 push 됨"이지 "push 성공 확인"이 아님. 메시지 카피에 이를 반영("✅ Push 승인: …"). 사후 성공 확인은 v1.3 폴링 범위.

## B.5 Open Questions

1. **repo 식별** — dev-confirm `cmd` 만으로는 어느 repo 의 push 인지 모호(`cmd` 에 cwd 없음). spawn 의 `target_repo` 를 dev-confirm request 에 실어 보낼지, 아니면 `git push` 실행 cwd 를 cmd 래퍼에서 캡처할지. → MVP 는 `target_repo` 우선, 없으면 org 기본 repo.
2. **다중 push** — 한 spawn 에서 연속 push 시 N건 메시지 vs 1건 묶음. MVP: 건별.
3. **push 범위 커밋 요약** — `git log <remote>/<branch>..HEAD` 는 승인 시점 로컬 기준. ahead 커밋이 정확히 push 대상이 아닐 수 있음(force/부분 push). MVP: best-effort, 부정확 가능성 footnote.
4. **Slack 대응** — 본 plan 은 discord-only. Slack 은 채널 명명·생성 모델이 달라 별도(`git-<handle>` Slack 채널) 후속.
