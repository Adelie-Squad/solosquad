## 1. 발견된 drift 직접 수정 — `dev_capability` 4-level vs boolean

### 1.1 증상

`manual/master-guide_{ko,en}.html` §3.11이 SKILL frontmatter의 `dev_capability` 필드를 **4단계 enum**으로 설명:

> "4단계: `read`(읽기만) / `propose`(파일 변경 제안만 출력) / `patch`(실제 변경 후 dev_confirm 요청) / `pr`(PR 생성까지)"

그러나 실제 코드(`src/bot/skill-parser.ts:124`):
```typescript
dev_capability?: boolean;     // true 또는 false. 4-level enum 아님
```

권한 분리는 별도 sub-tree로:
```typescript
dev_permissions?: SkillDevPermissions {
  bash?: { allowed?: string[]; denied?: string[] };
  network?: boolean;
  push_targets?: { requires_confirmation?: boolean };
  merge?: { auto?: boolean };   // 영구 false 박제
};
```

### 1.2 원인

v0.8.2 plan 초기 design intent는 4-level enum이었음. 구현 시점에 *단순화*되어 boolean + dev_permissions sub-tree 분리로 굳어짐. 그러나 master-guide §3.11은 plan의 design intent를 채택했다가 *코드 reality 반영 안 됨*. v0.8.6 작업 중 grep으로 식별.

### 1.3 수정 — 직접 edit (KO + EN)

§3.11 본문 다음 내용으로 교체:

> SKILL.md frontmatter의 `dev_capability: true` 필드로 에이전트가 코드 수정 dev action(파일 변경·git commit·push)을 수행할 수 있음을 선언. **boolean** — 켜기/끄기. 세부 권한은 별도 `dev_permissions` sub-tree로 분리:
>
> - `bash.allowed` / `bash.denied` — 호출 가능·차단 bash 명령 리스트. workspace-level denylist가 항상 위에 merge
> - `network` — 외부 HTTP 호출 (curl/wget 등) 허용 여부. MCP server는 별도
> - `push_targets.requires_confirmation` — `git push` 시 사용자 확인 게이트
> - `merge.auto` — *영구 false 박제* (v0.8.2 §2 / §3.1)
>
> 모든 sensitive 명령 (`git push`, `gh pr merge`, `gh pr close`)은 `SENSITIVE_BASH_PREFIXES`에 박혀 있고 30분 timeout dev-confirmation 게이트를 통과해야 실행됩니다. 자동 머지는 *영구 거부*. 결과는 `<org>/memory/dev-confirmations.jsonl`에 append-only 기록.

**스코프 한정**: §3.11 한 문단만. 다른 곳에 4-level 표현이 잔존하면 별도 grep으로 1회 sweep (publish 직전 *수동* 체크).

## 2. 회귀 catcher — `migrate-default-target.test.ts`

### 2.1 의도

v0.8.6의 `CLI_VERSION_TARGET = "0.4.0"` 회귀가 *1년 잠복*한 핵심 원인 = *해당 default를 검증하는 테스트가 0건*. 같은 클래스 회귀 재발 방지:
- `migrate.ts`가 `--to` 없이 호출됐을 때 target이 **현재 CLI 버전과 동등**해야 함
- 미래에 누군가 `CLI_VERSION_TARGET = "0.X.Y"`로 하드코딩 회귀 도입하면 *이 테스트가 fail*해서 즉시 발견

### 2.2 테스트 — 약 10줄

```typescript
// test/migrate-default-target.test.ts
import { test } from "node:test";
import assert from "node:assert/strict";
import { SOLOSQUAD_VERSION } from "../src/util/version.js";

test("migrate.ts default target equals current CLI version (no stale constant)", async () => {
  // Read migrate.ts source and verify CLI_VERSION_TARGET resolves to SOLOSQUAD_VERSION.
  // We can't directly check the const value without importing the CLI module
  // (which has side effects), so we verify by source inspection.
  const fs = await import("node:fs");
  const path = await import("node:path");
  const { fileURLToPath } = await import("node:url");
  const __dirname = path.dirname(fileURLToPath(import.meta.url));
  const migrateTs = fs.readFileSync(
    path.resolve(__dirname, "..", "src", "cli", "migrate.ts"),
    "utf-8",
  );

  // Must NOT contain hardcoded version string default.
  assert.ok(
    !/CLI_VERSION_TARGET\s*=\s*"\d+\.\d+\.\d+"/.test(migrateTs),
    "migrate.ts must not hardcode CLI_VERSION_TARGET to a version literal — use SOLOSQUAD_VERSION import",
  );

  // Must import SOLOSQUAD_VERSION from version.ts.
  assert.ok(
    /import\s*\{\s*SOLOSQUAD_VERSION\s*\}\s*from\s*["']\.\.\/util\/version\.js["']/.test(migrateTs),
    "migrate.ts must import SOLOSQUAD_VERSION from ../util/version.js",
  );

  // SOLOSQUAD_VERSION must be a non-empty semver string.
  assert.match(SOLOSQUAD_VERSION, /^\d+\.\d+\.\d+/, "SOLOSQUAD_VERSION must be semver");
});
```

### 2.3 적용 범위

- 본 테스트는 **`migrate.ts` 한정**. 같은 패턴이 다른 CLI 파일에 또 생기면 *그때 그 파일을 추가*. v0.8.7에서 *지금 잡힌 회귀의 직접 catcher*만 추가, 일반화된 lint 인프라는 §0 결정에 따라 만들지 않음
- 회귀 발견 시 즉시 fail → CI에서 차단
