## 0. 문제 정의

### 0.1 현 구조는 "보존"만 한다

코드 기준 사실(2026-06-16 확인):

- **tier-1 우선 해석.** `src/util/paths.ts` 의 `getAgentsDir`/`getSkillsDir`/
  `getRoutinesDir`/`getTeamsDir` 등은 **워크스페이스 `.solosquad/<x>/` → legacy →
  번들** 순으로 해석한다. 워크스페이스에 사용자 파일이 있으면 **그게 이기고 번들
  기본값은 무시**된다.
- **migrate 는 기존 파일을 건드리지 않는다.** 마이그레이션 스크립트는 새 기본값을
  seed 할 때 `if (fs.existsSync(dest)) continue;` 가드를 쓴다(예: `1.1.0-to-1.2.6.ts`).
  파괴 동작 전 `~/.solosquad-backups/` 에 전체 백업.
- **결과:** 사용자가 `code-review/SKILL.md` 를 커스텀하면, SoloSquad 가 같은 파일을
  개선해도 **그 개선이 영원히 도달하지 못한다.** 보존엔 완벽하나 전파 경로가 0.

### 0.2 자동 전파 도구가 없다

`--force`/`--reseed`/diff/3-way merge 어느 것도 코드에 없다(확인). 사용자가 개선을
받으려면 **수동으로** 번들 기본값(`$(npm root -g)/solosquad/...`)을 찾아 자기 파일과
diff 하고 손으로 옮겨야 한다. cookiecutter/yeoman 이 겪은 바로 그 한계 — copier/cruft
가 등장해 고친 그 문제다.

### 0.3 레퍼런스 합의 (trend-record 요약)

| 패턴 | 대표 | 보존 | 전파 |
|---|---|---|---|
| 전용 custom 존 | oh-my-zsh `custom/`, Claude Code `.claude/` | 자동 | 수동 |
| **버전드 패키지 + shadow-override** | **Claude Code 플러그인** | 자동 | **자동** |
| **3-way template merge** | **copier/cruft** (`_commit` 박제) | 자동 | **자동** |
| 생성형(콘텐츠 미배포) | Hermes (trajectory→skill) | — | 소거 |
| health-gated rollback | **OpenClaw** `update`→doctor→health→auto-rollback | 자동 | (config 한정) |
| clean clone-own | gstack (`git clone`+`git pull`) | 수동 | 수동(충돌) |

생태계는 "복사→편집"에서 **"버전드 패키지 + shadow-override"** 로 수렴. Tier-1
레퍼런스 phuryn/pm-skills 조차 자체 메커니즘을 버리고 Claude Code 플러그인에 위임.

## 1. 설계 원칙 (불변 제약)

1. **보존 회귀 0.** 현 tier-1 우선/`fs.existsSync` 가드/백업이 주는 "네 수정은 절대
   안 날아간다" 보장을 **약화하지 않는다.** 전파는 *추가* 축이지 보존을 거래하지 않음.
2. **편집이 아니라 오버레이.** 사용자가 "파일을 통째 복사해 편집"하게 두지 말고
   "override 를 얹게" 유도 — 충돌 면적을 구조적으로 줄인다.
3. **불투명 강요 금지.** SoloSquad 의 가치는 ship 하는 콘텐츠(프롬프트/워크플로)
   자체라, React 식 "콘텐츠 미배포"(Hermes 생성형 포함)는 본 버전 범위 밖(장기 비전).
4. **런타임 정렬 우선.** 런타임이 Claude Code 이므로 reconciliation 을 자체
   재발명하기 전에 **Claude Code 플러그인 precedence/마켓**에 정렬 가능한지 먼저 본다.

## 2. M1 — User-global overlay tier 실배선 (저위험·즉효)

### 2.1 문제

문서·Docker 마운트(`~/.solosquad:/root/.solosquad`)는 `~/.solosquad/` 를 "v0.5
3-tier user-global SKILL override"로 광고하지만, **현 `paths.ts` 해석 경로엔 home
tier 가 없다.** `~/.solosquad/` 에서 실제 읽히는 건 `agent-profile-defaults.yaml`
(`src/util/agent-profile.ts:209`) 하나뿐. 즉 광고된 override 존이 inert.

### 2.2 변경

`getAgentsDir`/`getMainAgentsDir`/`getSpecialistsDir`/`getSkillsDir`/`getRoutinesDir`/
`getTeamsDir` 의 해석 순서를 **3-tier 로 확정**:

```
1. 워크스페이스 org-local  (<ws>/.solosquad/<x>/)   ← 최우선 (기존)
2. user-global overlay     (~/.solosquad/<x>/)       ← 신규 배선
3. 번들 기본값             (<bundle>/<x>/)            ← 폴백 (기존)
```

oh-my-zsh `custom/` 동형 "절대 안 덮는 존" 을 완성. 사용자는 머신 전역 노하우를
`~/.solosquad/skills/` 에 두면 **모든 워크스페이스에서** 살아남고, 엔진 업그레이드가
번들을 갱신해도 무손상.

### 2.3 리스크

- 테스트가 home-tier 를 가정하지 않아 회귀 가능 → `paths.test.ts` 에 3-tier 우선순위
  케이스 추가. `HOME` 을 가짜 디렉토리로 격리(기존 `agent-profile.ts` 테스트 패턴 재사용).
- 성능: 핫패스에서 `fs.existsSync` 1회 추가 — 무시 가능.

## 3. M2 — Seed manifest + `solosquad update` 3-way reconcile (copier 식)

### 3.1 문제

M1/오버레이는 *보존*을 완성하지만 *전파*는 여전히 수동. copy-edit 모델의 유일한
자동 전파 해법은 copier 의 3-way merge: **갈라진 지점을 기억**했다가 old→new diff 를
사용자 편집본에 패치.

### 3.2 변경

- **seed manifest 박제.** `init`/`migrate` 가 워크스페이스에 파일을 seed 할 때,
  `<ws>/.solosquad/.seed-manifest.yaml` 에 `{path, engine_version, sha256}` 기록
  (copier `_commit` 등가물). "이 파일은 엔진 vX 의 기본값에서 출발했다"를 박제.
- **`solosquad update --reconcile`(신규).** 엔진 업그레이드 후:
  1. manifest 의 `engine_version` 기준 **옛 번들 기본값 재현**(번들은 npm 캐시/패키지
     tarball 에서 해당 버전 조회, 불가 시 manifest 의 sha 로 "사용자가 안 건드림" 판정).
  2. 사용자 현재 파일 vs 옛 기본값 = **사용자 변경분(diff) 추출**.
  3. 새 번들 기본값에 사용자 diff **재적용** → 충돌 시 git 스타일 마커(`<<<<<<<`)
     또는 `.rej` 로 표면화(copier `--conflict inline|rej` 동형).
  4. 사용자가 안 건드린 파일(sha 일치)은 **조용히 새 기본값으로 교체**(전파 자동).
- **escape hatch.** `solosquad update --recopy`(copier `recopy` 등가) — smart merge
  를 건너뛰고 새 기본값으로 재생성하되 사용자 파일은 `.bak` 백업.

### 3.3 충돌 지점

- **산문 diff 품질.** SKILL.md 는 코드가 아닌 산문이라 줄단위 3-way 가 거칠 수 있음
  → 단락/섹션 경계 인지 머지 또는 "충돌 시 사용자 우선 + 새 기본값을 인접 표시" 폴백.
- **번들 옛 버전 재현.** npm 이 옛 버전을 항상 캐시하지 않음 → sha 기반 "건드림/안
  건드림" 2-way 판정으로 graceful degrade(건드린 파일은 수동 안내, 안 건드린 건 자동 교체).

## 4. M3 — Health-gated auto-rollback (OpenClaw 식)

### 4.1 문제

현 `migrate` 는 백업은 뜨지만 적용 후 실패해도 **자동 복원하지 않는다**(수동
`--rollback` 만). OpenClaw `update` 는 doctor→health check→**실패 시 자동 rollback**.

### 4.2 변경

`migrate --apply` 파이프라인을 **백업 → 적용 → `doctor` 헬스체크 → 실패 시 직전
백업 자동 restore + 봇 재기동** 으로 확장. v1.2.8 의 graceful-drain(SIGTERM →
graceful exit → Docker respawn)과 결합해 무인 갱신의 안전 경계를 닫는다.

### 4.3 비고

reconciliation 과 직교한 별개 이득(데이터 보존 축)이라 M1~M2 와 독립 배포 가능 —
가장 저비용/즉효라 M1 과 함께 선출시 후보.

## 5. M4 — Claude Code 플러그인 정렬 (실험·장기)

### 5.1 가설

SoloSquad 기본 스킬/에이전트를 **Claude Code 플러그인으로 패키징**(`plugin.json`
version + 마켓플레이스)하면, 사용자는 `.claude/` 에서 **동명 shadow-override**(project
> user > plugin precedence)로 커스텀하고 플러그인 버전 bump 으로 개선을 받는다 —
reconciliation 을 런타임에 위임, 자체 재발명 제거.

### 5.2 미해결

- SoloSquad 의 3-tier(workspace/org/bundle) 의미론과 Claude Code 의 plugin
  precedence 가 **이중 precedence 충돌**을 일으키지 않는지.
- 워크스페이스 `.solosquad/` 모델과 플러그인 모델의 병존·마이그레이션 경로.
- → 본 버전에선 **조사/PoC 만**, 본격 채택은 후속 메이저.
