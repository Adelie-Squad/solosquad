## 1. 배경 / 문제

SoloSquad 는 이미 깊은 기획 인프라(problem-definition 6-Phase, PM + product 6 스페셜리스트,
hypothesis-design·opportunity-tree·prd-writer…)를 갖췄다. 그러나:

1. **"기획 단계"가 하나의 파이프라인으로 뭉뚱그려져** 있었다 — 실제로는 *각 단계가 독립 워크플로*
   여야 하고, 상위 워크플로가 이들을 **서브워크플로로 합성**해야 한다(현 `discovery-cycle` 은
   problem-definition 을 *skill 호출 중복*으로 흡수 — 타입 혼동).
2. **메인 워크플로가 정의돼 있지 않다** — 사용자가 "무엇을 기획하느냐"(신규 vs 개선)에 따른
   기본 진입점이 없다.
3. **PRD 가 양식(template)으로 누적되지 않는다** — 단계 결과가 구조화 양식에 기입되지 않음.
4. **시장 조사 리포트 산출물이 없다** — 별도 리포트를 내는 워크플로가 부재.
5. **workflow 가 cron 동형 정합(org 종속·`works-<handle>` 보고·자산 CRUD 확인)을 안 따른다.**

v1.3.5 는 이 다섯을 닫는다. (goal 은 보류 — §4.)

## 2. 핵심 개념 (ideation 확정 — 본 PRD 의 토대)

- **명사는 3종뿐:** **Agent**(행위자·동적), **Workflow**(공정·결정적), **Skill**(방법). `main/sub` 는
  *타입이 아니라 호출 위치*다(워크플로 안에서 불리면 sub-workflow, 단독 실행하면 main).
- **프레임워크 = skill, 스페셜리스트 agent 가 이를 활용해 워크플로 수행.** (예: SCQA·MECE =
  `problem-definition` skill, XYZ = `hypothesis-design`, OST = `opportunity-tree`.)
- **`_workflow/<id>` 참조형 신설** — stage 가 다른 워크플로를 서브워크플로로 호출(Argo
  "Workflow of Workflows" / GH reusable-workflow 패턴). 기존 stage `agent:` 참조 3형(`<team>/
  specialist` · `_main/<bot>` · `_skill/<x>`)에 4번째 추가.
- **자산 기반:** 모든 워크플로는 **SoloSquad 에 정의된 skill·agent 로 구성**된다(임의 코드 아님).

## 3. 스코프

### 3.1 메인 워크플로 2종 (현재 버전 기본 제공)

| 메인 워크플로 | 구성(서브워크플로 순서) | 비고 |
|---|---|---|
| **신규 구축 (new-build)** | (**아이디어 구체화** \| **요구사항 분석**) → **시장 조사** → **가설 수립** | 둘 중 하나로 시작(맥락 판단) |
| **개선 (improvement)** | **KPI 확인** → **데이터 분석** → **가설 수립** | 기존 제품 지표 출발 |

- **맥락 적응:** 각 메인은 사용자에게 **추가 정보를 요청**하거나 맥락에 따라 **적절한 과정만**
  수행한다(불필요 단계 건너뜀). 신규 구축의 시작점(아이디어 vs 요구사항)은 입력의 구체성으로 판단.
- **가설 수립**은 두 메인이 **공유하는 서브워크플로**(재사용 — main/sub 는 위치).

### 3.2 서브워크플로 6종 (각각 skill + 스페셜리스트 agent 기반)

> 모두 **재사용 우선**. ✅=기존 자산 재사용, 🆕=신규 제안(승인 시 생성, §3.7).

| 서브워크플로 | 스페셜리스트 agent | 사용 skill (프레임워크) | 산출 | 사용자 입력 |
|---|---|---|---|---|
| **아이디어 구체화** | product/idea-scoper | ✅ discovery-synthesis · opportunity-tree · jobs-stories(Job Story=JTBD) · lean-canvas | PRD §아이디어 | **고객 발견 = 필수**(§3.6) |
| **요구사항 분석** | product/feature-planner (PM 초안) | ✅ problem-definition(SCQA·MECE) · jobs-stories(User Story·INVEST) · prioritization | PRD §요구사항 | 제약·우선순위 확인 |
| **시장 조사** | design/researcher + product/business-strategist | 🆕 **market-research** · ✅ lean-canvas · discovery-synthesis | PRD §시장 **+ 리포트**(§3.4) | 타깃 시장·경쟁사 힌트 |
| **KPI 확인** | product/data-analyst | ✅ okr-writer(KR/KPI) · 🆕 *(필요 시)* kpi-frame | PRD §KPI | 목표 지표 확인 |
| **데이터 분석** | product/data-analyst | ✅ experiment-design · data-analyst 추론(Confidence Score) | PRD §데이터 | 데이터 출처 |
| **가설 수립** | product/pmf-planner | ✅ hypothesis-design(XYZ·If-Then-Because) · opportunity-tree(Six Forcing Q) | PRD §가설 | — |

**확정 신규 skill: `market-research`** (현재 리포트 산출 skill 0). 나머지는 기존 재사용 — 갭이
드러나면 §3.7 절차로 제안.

### 3.3 `_workflow/<id>` 참조형 신설 (서브워크플로 합성)

- workflow stage 의 `agent:` 가 `_workflow/<id>` 를 받으면 그 워크플로를 **서브워크플로로 실행**
  하고 결과(+ open_questions[])를 부모에 반환. 인터페이스 = stage inputs/outputs + 드롭박스
  (`memory/decisions.jsonl` + `_handoff.md`, 라이브 동기화 아님).
- **깊이 2단계 권장**(부모 main → sub-workflow → skill/agent). 자식은 **단독 실행도 가능**(위치만 바뀜).
- 손댈 곳: `workflow-resolver`(stage agent 파싱에 `_workflow/` 추가), 실행기(서브워크플로 dispatch),
  cycle 가드(중첩 깊이·순환 검출 — `agent validate --graph` 동형).

### 3.4 PRD 2 양식 + 단계별 기입 + 별도 산출물

- **메인별 PRD 양식 2종**: `new-build` PRD(아이디어/요구사항 · 시장 · 가설 섹션) / `improvement`
  PRD(KPI · 데이터 · 가설 섹션). **각 서브워크플로 완료 시 해당 섹션을 양식에 기입**(누적).
- **별도 산출물 워크플로** = **PRD 기입 + 산출물 생성**. 현재 버전 기준 **시장 조사** 만 해당 →
  **리포트** 생성. 위치: **`<org>/docs/reports/market-research-<slug>-<date>.md`**. PRD §시장은
  요약 + 리포트 evidence_ref(경로) 참조.
- **`docs/` 디렉토리는 PM 에이전트가 관리**(생성·정리·인덱싱). PRD 본체는 `<org>/docs/prd/`
  (또는 워크플로 인스턴스), 리포트는 `<org>/docs/reports/`.
- `prd-writer` skill 을 2-양식 인지하도록 확장(메인 종류로 분기).

### 3.5 요구사항 확정 프로세스 (리뷰 게이트)

요구사항 분석 서브워크플로의 산출은 **PM 초안 → 담당 스페셜리스트 리뷰 → PRD 최종 확정**:
1. PM(또는 feature-planner)이 요구사항 **초안** 작성.
2. 그 작업을 **담당하게 될 스페셜리스트 agent 들**(engineering/design/… 해당자)이 리뷰 —
   실행 가능성·예외/사이드이펙트·누락을 지적(open_questions[] 로).
3. 리뷰 반영 후 **PRD §요구사항에 최종 확정** 기입.

### 3.6 고객 발견 = 사용자 입력 요구

- 워크플로는 **사용자의 기획 역량이 발휘되도록 돕는 도구**다 — 자동 생성기가 아니다.
- **고객 발견**(아이디어 구체화의 A 렌즈)은 **사용자 입력을 요구**한다. 고객 정보·인터뷰 등
  필요한 정보가 있으면 `open_questions[]`/직접 질의로 **요청**하고, 없으면 추측으로 채우지 않는다
  (1인 초기엔 desk research·시장 신호로 보완 + "추후 인터뷰 계획" 산출).

### 3.7 자산(skill·agent) CRUD — 재사용 우선

- **재사용 우선 → 없으면 제안 → 사용자 승인 시 생성/수정.** 번들 불변, **워크스페이스 override
  레이어**에 생성, `asset validate` 게이트 통과. (v1.3.4 §G cron-manager 와 동형.)
- 파괴적 동작(수정·삭제)은 적용 전 확인.

### 3.8 cron 동형 정합 (org·채널·멀티 repo)

- 메인/서브 워크플로는 **org 종속**, 결과·진행을 **`works-<handle>`** 에 보고(v1.3.4 §F2 채널 모델).
- 멀티 repo 컨텍스트는 v1.4.0 §10 공통 리졸버를 **소비**(매니페스트 주입 · 지정 문법 "비우면
  전체, 적으면 그것들"). 기획은 주로 *읽기*라 비용이 개발보다 작다.

## 4. 비목표 (Non-goals)

- **goal 기능** — 본 버전 **보류**(변경 없음). 기획 goal 표준화는 후속(v1.4.x, 멀티 repo 실행 이후).
- **개발 요구사항(spec-driven) 서브워크플로** — 2개 메인에 미포함. 요구사항 분석의 *리뷰 게이트*
  (§3.5)가 예외/사이드이펙트를 일부 흡수. 전용 spec-clarify 워크플로는 후속.
- **멀티 repo 실행 배선 자체** — v1.4.0 §10 소관(본 버전은 소비만).
- **기존 4 워크플로 전면 폐기** — discovery-cycle/pmf-validation/autoplan-pm 은 신규 2-메인의
  프리셋/서브셋으로 재정의(중복 제거), 즉시 삭제 아님.

## 5. 손댈 것 / 신규 자산

| 종류 | 항목 | 작업 |
|---|---|---|
| **workflow(main)** | `new-build` · `improvement` | 신규 — 서브워크플로를 `_workflow/` 로 합성 |
| **workflow(sub)** | idea-refinement · requirements-analysis · market-research · kpi-check · data-analysis · hypothesis | 신규(기존 problem-definition/discovery-cycle 자산 재사용해 구성) |
| **skill** | `market-research` | 🆕 — 멀티에이전트 병렬 리서치 + 인용/groundedness 검증 → 리포트 |
| **skill** | `prd-writer` | 확장 — 2 양식(new-build/improvement) 분기 + 섹션 누적 기입 |

## 6. 구현 단계 (게이팅)

1. **`_workflow/` 참조 + 합성 실행** (workflow-resolver/runner + 깊이/순환 가드 + 테스트).
2. **서브워크플로 6종** 정의(기존 skill/agent 재사용 배선) + **가설 수립 공유**.
3. **메인 2종**(new-build/improvement) + 맥락 적응(시작점 판단·단계 skip·추가 정보 요청).
4. **`market-research` skill + 리포트 산출**(`<org>/docs/reports/`) + PM docs 관리.
5. **`prd-writer` 2 양식** + 단계별 섹션 누적 + 시장조사 evidence_ref.
6. **요구사항 리뷰 게이트**(§3.5) + **고객 발견 입력 요구**(§3.6).
7. **cron 동형 정합**(works-<handle> 보고 · 자산 CRUD 확인) + 기존 4 워크플로 프리셋 재정의.

## 7. 오픈 이슈

- **PRD/리포트 정확 경로** — PRD `<org>/docs/prd/` vs 워크플로 인스턴스(`workflows/<id>/`).
  리포트 `<org>/docs/reports/` 확정. docs/ PM 관리 규약(인덱스 파일?).
- **problem-definition 분할 여부** — "각 프레임워크=skill" 원칙상 SCQA/5-Whys/MECE/TDCC/XYZ 를
  서브워크플로별로 쪼갤지 vs 현 6-Phase 단일 skill 재사용. 권장: 단일 유지 + 서브워크플로가 필요
  Phase 만 호출(분할은 갭 발생 시).
- **개선 워크플로의 KPI/데이터 skill** — okr-writer·experiment-design 재사용으로 충분한지, kpi-frame/
  data-analysis 신규가 필요한지(§3.7 절차로 판정).
- **메인 선택 UX** — 사용자가 신규/개선을 명시 vs Chief 가 입력으로 추론(+사후 라벨링).

## 8. 수용 기준 (Acceptance)

- [ ] stage `agent:` 가 `_workflow/<id>` 를 받아 서브워크플로를 실행하고 결과를 부모에 반환한다(깊이/순환 가드 포함).
- [ ] 메인 워크플로 2종(new-build/improvement)이 정의돼, 각자 명시된 서브워크플로 순서로 합성된다.
- [ ] 신규 구축은 (아이디어 구체화|요구사항 분석)→시장 조사→가설, 개선은 KPI→데이터→가설로 진행하며 맥락에 따라 추가 정보를 요청한다.
- [ ] 각 서브워크플로가 기존(또는 승인된 신규) skill·스페셜리스트 agent 로 구성된다.
- [ ] 시장 조사가 `<org>/docs/reports/` 에 리포트를 산출하고 PRD §시장이 이를 참조한다.
- [ ] PRD 가 메인별 2 양식을 가지며, 각 단계 완료 시 해당 섹션이 누적 기입된다.
- [ ] 요구사항이 PM 초안 → 담당 스페셜리스트 리뷰 → PRD 최종 확정 흐름을 따른다.
- [ ] 고객 발견 단계가 사용자 입력을 요구하고, 없으면 추측 대신 open_questions[]로 남긴다.
- [ ] 자산 CRUD 가 재사용 우선 → 없으면 제안 → 승인 시 생성/수정(파괴적 동작 확인) 규칙을 따른다.
- [ ] 워크플로가 org 종속이며 `works-<handle>` 로 보고한다. goal 은 변경 없음.
- [ ] 전 테스트 green, `asset validate`(workflow 포함) 통과.
