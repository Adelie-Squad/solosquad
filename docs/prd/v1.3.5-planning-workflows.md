## 1. 배경 / 문제

SoloSquad 는 이미 깊은 기획 인프라(problem-definition 6-Phase, PM + product 6 스페셜리스트,
hypothesis-design·opportunity-tree·prd-writer…)를 갖췄다. 그러나:

1. **"기획 단계"가 하나의 파이프라인으로 뭉뚱그려져** 있었다 — 실제로는 *각 단계가 독립 워크플로*
   여야 하고, 상위 워크플로가 이들을 **서브워크플로로 합성**해야 한다(현 `discovery-cycle` 은
   problem-definition 을 *skill 호출 중복*으로 흡수 — 타입 혼동).
2. **메인 워크플로가 정의돼 있지 않다** — 사용자가 "무엇을 기획하느냐"(신규 vs 개선)에 따른
   기본 진입점이 없다.
3. **PRD 가 양식(template)으로 누적되지 않는다** — 단계 결과가 구조화 양식에 기입되지 않음.
4. **시장 조사 리포트 산출물이 없다** — 별도 리포트를 내는 워크플로가 부재.
5. **workflow 가 cron 동형 정합(org 종속·`works-<handle>` 보고·자산 CRUD 확인)을 안 따른다.**

v1.3.5 는 이 다섯을 닫는다. (goal 은 보류 — §4.)

## 2. 핵심 개념 (ideation 확정 — 본 PRD 의 토대)

- **명사는 3종뿐:** **Agent**(행위자·동적), **Workflow**(공정·결정적), **Skill**(방법). `main/sub` 는
  *타입이 아니라 호출 위치*다(워크플로 안에서 불리면 sub-workflow, 단독 실행하면 main).
- **프레임워크 = (개별) skill, 스페셜리스트 agent 가 이를 활용해 워크플로 수행.** 각 프레임워크는
  **독립 skill** 로 분할한다 — `scqa` · `five-whys` · `mece` · `tdcc` · `xyz-hypothesis` (기존 6-Phase
  단일 `problem-definition` skill 을 쪼갬). **`problem-definition` 은 워크플로우급 이름** → 이 프레임
  워크 skill 들을 chain 하는 **(서브)워크플로**로 남긴다(skill 아님). OST = `opportunity-tree` skill.
- **`_workflow/<id>` 참조형 신설** — stage 가 다른 워크플로를 서브워크플로로 호출(Argo
  "Workflow of Workflows" / GH reusable-workflow 패턴). 기존 stage `agent:` 참조 3형(`<team>/
  specialist` · `_main/<bot>` · `_skill/<x>`)에 4번째 추가.
- **자산 기반:** 모든 워크플로는 **SoloSquad 에 정의된 skill·agent 로 구성**된다(임의 코드 아님).

## 3. 스코프

### 3.1 메인 워크플로 2종 (현재 버전 기본 제공)

| 메인 워크플로 | 구성(서브워크플로 순서) | 비고 |
|---|---|---|
| **신규 구축 (new-build)** | (**아이디어 구체화** \| **요구사항 분석**) → **시장 조사** → **가설 수립** | 둘 중 하나로 시작(맥락 판단) |
| **개선 (improvement)** | **KPI 확인** → **데이터 분석** → **가설 수립** | 기존 제품 지표 출발 |

- **메인 선택:** Chief 가 **입력 맥락으로 추론**한다 — "이 아이디어 기획해줘"→신규 구축,
  "전환율 떨어졌어, 개선하자"→개선. **애매하면 사용자에게 되묻는다.** (TRIAGE + 사후 라벨링
  규약과 정합 — 프레임워크/메인을 선처방하지 않고 의도가 모이면 명명.)
- **맥락 적응:** 각 메인은 사용자에게 **추가 정보를 요청**하거나 맥락에 따라 **적절한 과정만**
  수행한다(불필요 단계 건너뜀). 신규 구축의 시작점(아이디어 vs 요구사항)은 입력의 구체성으로 판단.
- **가설 수립**은 두 메인이 **공유하는 서브워크플로**(재사용 — main/sub 는 위치).

### 3.2 서브워크플로 6종 (각각 skill + 스페셜리스트 agent 기반)

> **재사용 우선**(§3.7). ✅=기존 자산 재사용, ✂️=기존 단일 skill 을 프레임워크별로 분할,
> 🆕=신규 제안(승인 시 생성).

| 서브워크플로 | 스페셜리스트 agent | 사용 skill (프레임워크) | 산출 | 사용자 입력 |
|---|---|---|---|---|
| **아이디어 구체화** | product/idea-scoper | ✅ discovery-synthesis · opportunity-tree · jobs-stories(Job Story=JTBD) · lean-canvas | PRD §아이디어 | **고객 발견 = 필수**(§3.6) |
| **요구사항 분석** | product/feature-planner (PM 초안) | ✂️ `scqa` · `mece` · ✅ jobs-stories(User Story·INVEST) · prioritization | PRD §요구사항 | 제약·우선순위 확인 |
| **시장 조사** | design/researcher + product/business-strategist | 🆕 **market-research** · ✅ lean-canvas · discovery-synthesis | PRD §시장 **+ 리포트**(§3.4) | 타깃 시장·경쟁사 힌트 |
| **KPI 확인** | product/data-analyst | ✅ okr-writer(KR/KPI) (재사용 — §3.7) | PRD §KPI | 목표 지표 확인 |
| **데이터 분석** | product/data-analyst | ✅ experiment-design · data-analyst 추론(Confidence Score) (재사용) | PRD §데이터 | 데이터 출처 |
| **가설 수립** | product/pmf-planner | ✂️ `five-whys` · `tdcc` · `xyz-hypothesis` · ✅ opportunity-tree(Six Forcing Q) · hypothesis-design(If-Then-Because) | PRD §가설 | — |

> **프레임워크 skill 분할(✂️):** 기존 `problem-definition` 단일 skill(SCQA→5-Whys→MECE→TDCC→XYZ
> 6-Phase)을 **개별 skill 5종**(`scqa`·`five-whys`·`mece`·`tdcc`·`xyz-hypothesis`)으로 쪼갠다. 그러면
> 요구사항 분석은 B(구조분해: scqa·mece), 가설 수립은 C+D(원인추적: five-whys·tdcc + 가설: xyz)를
> *필요한 것만* 조립한다. `problem-definition` 이름은 이 skill 들을 chain 하는 **(서브)워크플로**로
> 보존(워크플로우급 이름).

**확정 신규 skill: `market-research`** (현재 리포트 산출 skill 0). 나머지는 기존 재사용 — 갭이
드러나면 §3.7 절차로 제안.

### 3.3 `_workflow/<id>` 참조형 신설 (서브워크플로 합성)

- workflow stage 의 `agent:` 가 `_workflow/<id>` 를 받으면 그 워크플로를 **서브워크플로로 실행**
  하고 결과(+ open_questions[])를 부모에 반환. 인터페이스 = stage inputs/outputs + 드롭박스
  (`memory/decisions.jsonl` + `_handoff.md`, 라이브 동기화 아님).
- **깊이 2단계 권장**(부모 main → sub-workflow → skill/agent). 자식은 **단독 실행도 가능**(위치만 바뀜).
- 손댈 곳: `workflow-resolver`(stage agent 파싱에 `_workflow/` 추가), 실행기(서브워크플로 dispatch),
  cycle 가드(중첩 깊이·순환 검출 — `agent validate --graph` 동형).

### 3.4 PRD 2 양식 + 단계별 기입 + 별도 산출물

- **메인별 PRD 양식 2종**: `new-build` PRD(아이디어/요구사항 · 시장 · 가설 섹션) / `improvement`
  PRD(KPI · 데이터 · 가설 섹션). **각 서브워크플로 완료 시 해당 섹션을 양식에 기입**(누적).
- **별도 산출물 워크플로** = **PRD 기입 + 산출물 생성**. 현재 버전 기준 **시장 조사** 만 해당 →
  **리포트** 생성. 위치: **`<org>/docs/reports/market-research-<slug>-<date>.md`**. PRD §시장은
  요약 + 리포트 evidence_ref(경로) 참조.
- **`docs/` 디렉토리는 PM 에이전트가 관리**(생성·정리·인덱싱). **PRD 본체 = `<org>/docs/prd/`**,
  **리포트 = `<org>/docs/reports/`** (확정 — 사용자 관점에서 사업 산출물이 한곳에 모임).
- `prd-writer` skill 을 2-양식 인지하도록 확장(메인 종류로 분기) + 요구사항 3대 유형 렌더(§3.4.1).

### 3.4.1 요구사항 분류 체계 (PRD 요구사항 섹션의 구조)

기획 워크플로가 산출하는 PRD 의 **요구사항(requirement)** 은 **3대 유형**으로 나뉜다. 한 PRD 는
필요에 따라 **섹션을 나눠 복수의 요구사항**을 담을 수 있다(유형 혼재 가능 — 예: 웹사이트 개발 +
런칭 아티클 + 성과 리포트가 한 PRD 에).

| 유형 | 산출물 | 예시 | 주 실행 |
|---|---|---|---|
| **개발 요구사항** | 코드 | 웹사이트, 제품(앱/기능) | engineering 스페셜리스트 |
| **콘텐츠 제작 요구사항** | 글·미디어 | 광고 소재 · 아티클(긴 글) · 포스트(짧은 글) | marketing/content 스페셜리스트 |
| **리포트 작성 요구사항** | 문서 | 보고서(docs 형태) · PPT(슬라이드) | product/research 스페셜리스트 |

**디자인 = 독립 4번째 유형이 아니라 각 요구사항에 내포(embedded).** 각 요구사항이 디자인 요소를
포함하는지 여부에 따라 그 요구사항 *안에* 디자인 요구사항이 딸린다. 디자인 요소 예: 글 내부 도표 ·
인포그래픽 · UI · 콘텐츠 이미지 · 영상/시나리오 · 데이터 시각화 · PPT 레이아웃 등. (디자인 요소가
있으면 design 스페셜리스트가 해당 요구사항의 실행에 합류.)

**요구사항 1건의 구성:** ① **핵심 내용**(무엇을·왜 — 목표·맥락·수용 기준) + ② **작업 체크리스트**
(실행 단위 to-do). 디자인 요소가 있으면 체크리스트에 디자인 작업 항목이 포함된다. 이 구조가
**다운스트림 실행**(스페셜리스트 dispatch · v1.4.0 멀티 repo)의 입력 단위가 된다.

### 3.5 요구사항 확정 프로세스 (리뷰 게이트)

요구사항 분석 서브워크플로의 산출은 **PM 초안 → 담당 스페셜리스트 리뷰 → PRD 최종 확정**:
1. PM(또는 feature-planner)이 요구사항 **초안** 작성.
2. 그 작업을 **담당하게 될 스페셜리스트 agent 들**(engineering/design/… 해당자)이 리뷰 —
   실행 가능성·예외/사이드이펙트·누락을 지적(open_questions[] 로).
3. 리뷰 반영 후 **PRD §요구사항에 최종 확정** 기입.

### 3.6 고객 발견 = 사용자 입력 요구

- 워크플로는 **사용자의 기획 역량이 발휘되도록 돕는 도구**다 — 자동 생성기가 아니다.
- **고객 발견**(아이디어 구체화의 A 렌즈)은 **사용자 입력을 요구**한다. 고객 정보·인터뷰 등
  필요한 정보가 있으면 `open_questions[]`/직접 질의로 **요청**하고, 없으면 추측으로 채우지 않는다
  (1인 초기엔 desk research·시장 신호로 보완 + "추후 인터뷰 계획" 산출).

### 3.7 자산(skill·agent) CRUD — 재사용 우선

- **재사용 우선 → 없으면 제안 → 사용자 승인 시 생성/수정.** 번들 불변, **워크스페이스 override
  레이어**에 생성, `asset validate` 게이트 통과. (v1.3.4 §G cron-manager 와 동형.)
- 파괴적 동작(수정·삭제)은 적용 전 확인.

### 3.8 cron 동형 정합 (org·채널·멀티 repo)

- 메인/서브 워크플로는 **org 종속**, 결과·진행을 **`works-<handle>`** 에 보고(v1.3.4 §F2 채널 모델).
- 멀티 repo 컨텍스트는 v1.4.0 §10 공통 리졸버를 **소비**(매니페스트 주입 · 지정 문법 "비우면
  전체, 적으면 그것들"). 기획은 주로 *읽기*라 비용이 개발보다 작다.

### 3.9 자산 매니저 일관성 (Workstream B — A 와 독립·병렬 가능)

> 5개 1급 자산(skill·agent·workflow·goal·cron)의 CRUD 를 **4축(네이밍·매니저 skill·저장 스코프·CLI)**
> 으로 일관 정렬한다. v1.3.2(도메인 생애주기 매니저)의 정렬을 **네이밍·표면 통일로 마감**. skill·agent 의
> *세부 고도화*(refine 루프·frontmatter 그래프 심화·런타임 생성)는 **v1.3.6 으로 분리**(§4).
> **공통 원칙:** create(보조)·review·refine 의 정식 경로 = **Chief 대화**(비결정적); validate·lifecycle = **CLI**(결정적).

**B-D1 개명 — `workflow-maker` → `workflow-manager`.** 전 repo 유일 `-maker` 잔재. `skills/workflow-maker/`
→ `skills/workflow-manager/`, `name:`·triggers·`used_by`·88개 참조 일괄 갱신. v1.3.2 §6 이 이미
"workflow-manager"로 호명 중 → 구현만 따라오면 됨. 잔여 `-maker` 참조 0 이 수용 기준.

**B-D2 매니저 = skill 통일 (`{asset}-manager` 5종 + 공통 `asset-review`).** 명사 3종(agent=행위자 ·
workflow=공정 · skill=방법)에서 **매니저는 "방법" → skill 이 맞다. 행위자는 이미 Chief/PM agent →
`*-manager` agent 신설 금지**(actor 중복). create-보조·review·refine 을 Chief 가 대화로 부르는
**`{asset}-manager` skill 5종**(workflow-manager[개명]·skill-manager·agent-manager·goal-manager·cron-manager)
+ 공통 `asset-review` skill 로 표면 정렬. 기존 코드 모듈(`src/bot/skill-manager.ts` 등)은 skill 이
호출하는 **결정적 백엔드**(validate/scaffold/lifecycle)로 역할 분리·유지.

**B-D3 저장 스코프 = org 종속 통일 (workflow·goal·cron 모두 `<org>/...`).** 현재 workflow=`<org>/workflows/`·
goal=`<org>/goals/` ✅ 인데 **cron 만 `.solosquad/crons`(워크스페이스 전역)** ⚠️. cron 사용자 def 를
**`<org>/crons/`** 로 이전(`getCronsWriteDir(orgSlug)`), 번들 빌트인 cron 은 불변. **migration
`1.3.4-to-1.3.5`**: 기존 `.solosquad/crons/*` → org 별 재배치(단일 org 면 그곳, 다중이면 기본 org +
경고). v1.3.3 per-user 개인화(tz·채널)와 정합(org ⊃ user).

**B-D4 CLI 일관성 = 5종 동일 verb 바닥 ("있으면 다, 없으면 다").** **결정적 공통 바닥(5종 전부):**
`list · show · validate · new(무-LLM scaffold) · <lifecycle: edit/enable/disable/archive|delete>`.
**비결정적(create-보조·review·refine)은 5종 전부 Chief 대화**(CLI 없음). 도메인 고유 verb(goal
`run/stop/queue/verify` · cron `start/run` · workflow `focus`)는 공통 바닥 위 **확장으로 허용**. 갭
메우기: `workflow new`·`skill new`·`agent new` scaffold 추가(goal/cron 은 기존 `new` 보유),
`asset <verb> <kind>` 파사드가 list/show/validate 를 균일 제공(v1.3.2 기반).

  (B-D3/D4 는 goal 의 저장 스코프·CLI 바닥만 *정렬 확인* — 이미 충족. 기능 변경 아님.)

  심화 검증·agent 런타임 동적 생성·버전 히스토리는 **v1.3.6 으로 분리**. 본 버전(B)은 매니저의

- **기존 4 워크플로 전면 폐기** — discovery-cycle/pmf-validation/autoplan-pm 은 신규 2-메인의
