# Part A — PM → Chief 리네임 (구현 완료)

v1.1 이 org-level supervisor 를 PM→Chief 로 리브랜딩했으나 출력 카피·내부 식별자·CLI

## A.1 "pm" 은 세 가지 의미 — 블라인드 치환 금지

| 층위 | 의미 | 처리 |
|---|---|---|
| **(A) 세션 드라이버 / 사용자 표면** | org-level supervisor = Chief 그 자체 | ✅ rename |
| **(B) 영속 데이터 계약** | 디스크/와이어 식별자(이벤트 kind, `workspace.yaml` `pm:` 키, `pm-compaction` 루틴 id, `system-pm-compaction` 스레드, `memory/pm-skills/`) | ⚠️ read-compat 후 rename 가능한 것만 / 나머지 keep+문서화 |
| **(C) 유령 "PM 에이전트"** | chief-runner 주석의 `agents/main/pm/` — **디스크에 없음** | 🧹 stale 주석 정정 |

- **타입**: `PmConfig`→`ChiefConfig`(컴파일 타임만, yaml 키 `pm` 유지).

# Part C — git-<handle> VCS 채널 제거 (롤백)

## C.0 결정 — SoloSquad 는 알림 기능에 관여하지 않는다

v1.2.9 Part B 는 `git-<handle>` 채널 + push 알림 코드(`git-event-notify.ts`)를
깔았다(게이트 dormant 라 실제 발사는 안 됨). 검토(2026-06-08) 결과 **이 자체 알림
파이프라인을 폐기**한다:

- **필요한 것은 "push 승인"뿐.** 안전의 핵심은 dev-confirm **승인 게이트**(→ v1.3.0)
  이고, 승인 UX 는 `command-<handle>` 채널에서 끝난다. 전용 git 채널은 승인과 **분리
  가능한 별도 관측 레이어**라 안전 기능엔 불필요하다.
- **알림은 플랫폼 네이티브가 우월.** "완료된 push 알림"은 사용자가 **GitHub→Discord
  네이티브 webhook**(repo settings 에 Discord webhook URL 등록)으로 제로코드·전건·
  실제 도착 기준으로 받을 수 있다. SoloSquad 가 로컬 `git log` 로 재구현하는 것은
  플랫폼이 공짜로 주는 것의 재발명이다.
- **귀속(attribution)도 위임 가능.** "이 push 가 에이전트 작업인가"는 v1.3.0 이
  **커밋 trailer 에 workflow-id 를 스탬프**하면 네이티브 webhook 의 커밋 메시지에
  자연히 노출된다 — 자체 알림 파이프 불필요.

→ **SoloSquad 역할 수축:** ⑴ push **승인 게이트**(v1.3.0) + ⑵ (선택) 커밋
workflow-id 스탬프. **알림 채널 운영은 사용자/GitHub/Discord 에 위임.**

## C.1 제거 범위 (코드만 — 기존 데이터 방치)

# Part D — 루트 repo Docker 스택 제거 + 사용자 Docker 자산 정리 (구현 완료)

## D.0 동기

`deploy/docker/`(Dockerfile + docker-compose.yml + README) 는 **이 repo 의 루트를
Docker 워크스페이스로 띄우는** 메인테이너 dogfood 스택이다. compose 가 워크스페이스를
`${SOLOSQUAD_WORKSPACE:-../..}` — 즉 **repo root 디폴트**로 잡는다. 이 구성은 혼란만
주고 실익이 없어 제거한다. 단, **사용자의 Docker 사용 경험은 1급 기능으로 무손상
유지**하고, repo 자체-호스팅 경로만 걷어낸다.

## D.1 제거 근거 (코드 기준 사실)

1. **repo 소스를 실행하지 않는다.** `deploy/docker/Dockerfile` 은
   `RUN npm install -g @anthropic-ai/claude-code solosquad` 로 **npm published 버전**을
   설치한다. "루트 repo 를 도커로 띄워도" 컨테이너가 도는 코드는 로컬 `src/` 가 아니라
   npm 최신 릴리스 → "repo 를 도커로 돌린다"는 멘탈 모델 자체가 허상.
2. **루트 repo 는 워크스페이스가 아니다.** repo root 에 `.env`·`<org>/` 없음.
   `.solosquad/` 엔 `repo.yaml` 뿐. `.gitignore` 가 `.solosquad/` 를
   `# SoloSquad workspace / path-reference mirror (dogfood case — local-only)` 로 명시.
3. **사용자 경로·표준과 다르다.** 일반 사용자는 `npm i -g solosquad` 로 엔진을 받고
   자기 워크스페이스를 볼륨 마운트한다. "엔진=레지스트리 설치 / 데이터=볼륨 마운트" 가
   self-hosted 도구의 정석(Postgres·Nginx·Gitea 동형). repo-root-as-workspace 만 이
   모델을 깨고 "소스 repo = 런타임 워크스페이스" 로 혼동시킨다.

## D.2 원칙 (불변 제약)

- **사용자 Docker 경험은 1급 기능 — 기능 퇴행 0.** `solosquad init` 후 워크스페이스
  루트에서 `docker compose up -d --build` 로 bot+scheduler 가 24/7 도는 흐름은 **그대로
  보장**한다. 이번 작업은 *오직* repo 자체를 로컬 Docker 로 띄우는 dogfood 경로만 제거.
- **삭제 대상은 repo-self-hosting 한정.** `deploy/docker/` 디렉토리 전체 제거.

## D.3 재설계 — 사용자 Docker 자산의 단일 위치

현재 사용자용 Docker 자산은 `assets/Dockerfile` · `assets/docker-compose.yml` 로
**`assets/` 루트에 평평하게** 놓여 있다(다른 자산은 `assets/agents/`·`assets/routines/`
처럼 디렉토리로 묶임). Docker 만 두 파일이 루트에 흩어져 일관성이 깨진다. 재설계:

- **단일 홈:** 사용자용 Docker 자산을 **`assets/docker/`** 하위로 묶는다
  (`assets/docker/Dockerfile`, `assets/docker/docker-compose.yml`,
  선택 `assets/docker/README.md`). `deploy/docker/` 의 정식 후계 = 여기 하나뿐.
- **배치 위치(init 복사 대상)는 불변:** `solosquad init` 은 이 둘을 **워크스페이스
  루트**로 복사한다(현행과 동일 목적지). 소스 위치(assets/docker/)만 정리하고, 사용자에게
  깔리는 위치(워크스페이스 루트)는 그대로 — `build: .` / `.:/workspace` 컨텍스트가
  사용자 워크스페이스 루트와 일치하는 구조 유지.
- **기능 병합(상위집합화):** `deploy/docker/docker-compose.yml` 에만 있던 3가지를
  `assets/docker/docker-compose.yml` 로 이관해 사용자 스택이 기능 퇴행 없는 상위집합이
  되게 한다:
  1. `stop_grace_period: 130s` (v1.2.8 graceful drain 대비)
  2. `${HOME}/.solosquad:/root/.solosquad` 마운트 (v0.5 3-tier user-global override)
  3. `${HOME}/.solosquad-backups:/root/.solosquad-backups` 마운트 (마이그레이션 rollback)
  - 단, `SOLOSQUAD_WORKSPACE=../..`(repo-root 디폴트) override 는 **이식하지 않는다** —
    그게 제거 대상인 repo-self-hosting 가정이므로. 사용자 compose 는 워크스페이스
    루트에 깔리니 `build: .` 로 충분.
