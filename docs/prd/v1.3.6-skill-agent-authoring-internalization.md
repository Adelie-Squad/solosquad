## 1. 배경 / 문제

SoloSquad 는 5종 자산(skill·agent·workflow·goal·cron)의 생애주기 매니저(v1.3.2)와 CRUD 정렬(v1.3.5)을
갖췄다. 그러나 **"잘 만드는 법" 자체가 시스템 안에 없다:**

1. **작성 표준이 코드/문서에 흩어져 있다.** 좋은 SKILL.md 의 규칙(description 공식·정량 한도·번들
   구조·eval)이 ideation 문서에만 있고, 매니저 skill 은 *얇은 CRUD 오케스트레이터*라 표준을 들고 있지 않다.
2. **검증기가 공식 규칙을 미검사.** `validateSkill` 은 64자·1024자·예약어·kebab·디렉터리일치·3인칭·vague
   를 검사하지 않는다(`anthropics/skills` corpus 라운드트립 호환·마켓 배포 사전조건 미달).
3. **decorative frontmatter 필드의 부채.** `pm_conventions`(~28 skill)·`category` 등은 skill-parser 의
   `known` 집합에 없어 **`extra` 백으로 떨어진다 = 파싱되나 검증·구동 안 됨**. "적혀 있으니 작동한다"는
   착각만 남기고 드리프트 위험을 키운다(§3.4).
4. **"좋아졌다"를 측정할 수 없다.** 개선은 감으로 한다 — description 트리거 eval·output A/B 가 없다.
5. **자가개선 구조 부재.** asset 은 write-once 다. loop engineering 도입 전, *스스로 다듬는* 최소 골격이 없다.

v1.3.6 은 이 다섯을 닫는다. **핵심 통찰: asset 을 "한 번 쓰는 파일"이 아니라 "eval 로 다듬고 스스로
학습하는 자산"으로 본다**(SkillOpt: "skill = frozen agent 의 external state, trained").

## 2. 핵심 개념

- **매니저 = 작성 권위(authoring authority).** `skill-manager`/`agent-manager` 가 단순 CRUD 가 아니라
  **"좋은 작성이 무엇인가"의 단일 기준**을 보유한다. 다른 skill/agent 를 만들거나 고칠 때마다 이 표준을
  적용한다 → addyosmani `using-agent-skills`(meta-skill 라우터) / Anthropic `skill-creator` 의 SoloSquad 판.
- **표준은 점진공개로 담는다.** 매니저 SKILL.md 본문은 lean(요지만), 상세는 `references/` 번들로 — *스킬이
  설파하는 원칙을 스스로 실천*.
- **load-bearing vs decorative.** 모든 커스텀 필드는 동작을 구동하거나 검증기가 강제해야 한다. 둘 다
  아니면 추가하지 않는다(§3.4).
- **자가개선 = 2층.** ①**행동층**(역할/절차 프롬프트를 eval-게이트로 텍스트 학습 — SkillOpt: rollout→
  reflect→bounded patch-edit `Lt=4`→held-out gate + rejected-edit buffer + protected slow-update;
  **자동 채점기 필수**) + ②**경험층**(세션 간 메모리 누적 — 고정 taxonomy·session-start 주입·human-approved
  write·full-block replace; **검증기 불필요, 지금 가능**). 둘은 SkillOpt 의 fast/slow 2-timescale 과 동형.
- **순서 의존:** **채점기(eval) 없이는 행동층 자가개선 없음.** 그래서 §3.3(eval) 이 §3.5(자가개선)의 전제.

## 3. 스코프

### 3.1 매니저를 작성 권위로 격상 ✅ (skill 측 착수 — 본 PRD 작성 중 1차 구현)
`skill-manager` 를 작성 표준 보유체로 재작성하고, 표준을 `references/` 3종으로 점진공개:
- **`skills/skill-manager/SKILL.md`** — lean 본문 + 3대 요지(description=디스커버리 · 본문=절차적 규율 ·
  decorative 필드 금지) + CRUD 흐름에 표준 적용 단계 삽입(description 먼저·eval 설계). 인접 skill 분담
  명문화(정적=validate, 리뷰=asset-review, 후보식별=skill-refinement, agent=agent-manager).
- **`references/authoring-guide.md`** — description 공식·본문 명명패턴(Gotchas·Verification 게이트·
  anti-rationalization·Red Flags)·정량 한도·**번들 구조(scripts/references/assets) 규칙**·점진공개.
- **`references/frontmatter-fields.md`** — 필드별 load-bearing/decorative **감사** + 작성 체크.
- **`references/eval-recipe.md`** — description 트리거 eval + output A/B + 자가개선 루프 연결.

> **agent 측 동형(미착수, 본 버전 범위):** `agent-manager` 를 같은 방식으로 격상 — 공유 원칙(description·
> 필드 감사)은 재사용, 고유는 **위임 그래프·역할 중첩·budget 상속**(agent-doc G1·G2·G4). reference 공유 검토.

### 3.2 검증기 공식 표준 정렬 (P0 — H1)
착수: `src/bot/skill-parser.ts` `validateSkill`, `src/bot/agent-validate.ts`.
- `name`: ≤64 + `^[a-z0-9]+(-[a-z0-9]+)*$`(연속/양끝 하이픈 금지) + 예약어(`anthropic`/`claude`) +
  **부모 디렉터리명 일치**.
- `description`: ≤1024 상한 · 1인칭 린트("I can"/"You can"→warn) · vague 휴리스틱("helps with"→warn) ·
  "Use when"/"사용 시점" 트리거절 부재 시 info.
- 본문 500줄 / 참조 1단계 깊이 린트.
- **agency-agents 식 originality 게이트**(8-word shingle, FAIL ≥40%/WARN ≥20%) 도입 검토 — 역할/설명 중복 탐지.

### 3.3 eval 루프를 corpus 에 (P0 — H2.5, 자가개선의 채점기)
착수: `src/analyze/validator-corpus.ts` + `npm run test:corpus`.
- **description 트리거 eval:** 20쿼리(should 8–10 + near-miss 네거 8–10) × 3런, 임계 0.5, train60/val40.
- **output A/B:** `evals/evals.json` 을 with_skill/without_skill 로, pass-rate + token·duration **delta**.
- 이 두 eval = §3.5 행동층 자가개선의 채점기. **검증기 없는 asset 은 정적 게이트로 폴백.**

### 3.4 frontmatter 필드 감사 — 결정 + 일괄 적용 (P1)
`pm_conventions`·`category` 는 skill-parser 의 `known` 집합에 없어 **`extra` 로 떨어진다 = 파싱되나
검증·구동 안 됨**. 단 `pm_conventions` 4필드는 *의미 없는 게 아니라* 실재 PM 규율이다(출처
`workshop-solosquad-description.md` §175~178) — "진짜 규율인데 읽는 코드만 없는" 케이스.

**값 분포(2026-06-23 실측 28 skill)가 결정을 가른다 — 한 필드가 어디서나 같은 값이면 per-skill 정보가
아니라 글로벌 기본값이다:**

| 필드 | 분포 | 해석 |
|---|---|---|
| `anti_sycophancy` | **28/28 true** | per-skill 정보 0 → **글로벌 규율** |
| `post_labeling` | 26 true / 2 false | 거의 균일 → **글로벌 + 드문 예외** |
| `hard_gate` | 17 false / 11 true | **진짜 가변** → per-skill 신호 |
| `minimum_approaches` | 16×1 / 12×2 | **진짜 가변** → per-skill 신호 |
| `category` | 7개 값 분산 | 그룹 정보 有, 표시용으로만 소비 |

→ **결정 대상**(§6.1 에 옵션). 확정 후 **전 skill 통일 적용**(한 skill 만 손대지 않는다 — 드리프트 방지).
결정 전까지 기존 필드 유지. **새 decorative 필드 추가는 지금부터 금지.**

### 3.5 자가개선 2층 골격 (P1 — loop engineering 전 최소 도입)
- **①행동층(SkillOpt 식):** `skill-author`/refine 위에 rollout→reflect(성공/실패 분리)→bounded patch-edit
  (`Lt=4`/floor 2)→held-out gate + rejected-edit buffer + protected slow-update. **§3.3 채점기 있는
  skill 부터.** 목표 본문 ~920토큰. optimizer 별도 모델 불필요(matched 도 동작).
- **②경험층(메모리):** `agent-profile.yaml`/org 메모리에 **고정 taxonomy(MEMORY.md `type` enum 재사용)
  + session-start 주입 + human-approved write + full-block replace + 턴당 1회 + 휘발성 거부**. Vercel
  personal-agent-template 패턴 이식. **검증기 불필요 → 본 버전에서 우선 착수 가능.**

### 3.6 작성 권위의 통칭어 정합 (P2 — `asset`→`primitive`)
5종(skill/agent/workflow/goal/cron) 통칭 `asset` 이 SKILL.md 번들 `assets/` 폴더와 충돌 → **`primitive`**
로 개명(별도 결정 기록). 본 버전에서 *신규* 문구는 `primitive` 사용, 발행/이력(마이그레이션 메시지·기존
PRD 파일명)은 historical record 로 유지, CLI `asset` 은 alias/deprecate 경로. **`assets/` 번들 폴더는 불변.**

## 6. 미해결 / 결정 필요 (사용자 결정 대기)

### 6.1 §3.4 — pm_conventions·category 처리 (wire vs 글로벌화 vs 제거)
분포(§3.4)가 "전부 wire / 전부 제거"의 이분법이 아니라 **필드별 갈림**임을 보여준다.
- **(A) 데이터 기반 하이브리드 — 분석상 권장.**
  - `anti_sycophancy`(28/28)·`post_labeling`(26/2) → **글로벌화**: per-skill frontmatter 에서 제거하고
    **PM 모드 세션 기본값으로 1곳 wire**(Chief/PM base 지침 주입). 예외(post_labeling:false 2개)는 해당
    skill 본문에 사유와 함께. 효과: 28곳 중복·드리프트 제거 + 실제 작동.
  - `hard_gate`(17/11)·`minimum_approaches`(16/12) → **wire(load-bearing 승격)**: 진짜 가변이므로
    제거하면 신호 손실. skill 실행 시 런타임이 읽어 강제(`minimum_approaches:2`→"≥2 접근 비교" 주입,
    `hard_gate:true`→exit_criteria 게이팅). 주의: `hard_gate` 는 *workflow stage* 레벨에선 이미
    `workflow-validate.ts` 가 읽음 — skill 레벨 의미와 정합 필요(충돌 아닌 레이어 구분).
  - `category`(7값) → `solosquad skill list` 그룹화에 쓰면 enum 검증 후 load-bearing, 아니면 폴더서
    유도 가능하니 **드롭**. 남기면 **반드시 enum 강제**.
- **(B) 전부 런타임 wire** — 4 convention 모두 연결 + category enum. 비용: 균일 필드(anti_sycophancy)도
  per-skill 표면으로 영구화(정보량 0인데 유지).
- **(C) 전부 제거** — frontmatter 에서 삭제, 규율은 본문 산문으로. 가장 싸나 hard_gate·minimum_approaches
  의 per-skill 신호를 잃고 본문 수동 기재에 의존.
- 착수점: wire 시 스킬 프롬프트 조립부(`spawn-assembler` 등) + skill-parser `known` 집합 추가. 글로벌화
  시 Chief/PM base 지침 1곳 + 28 skill 에서 2필드 삭제(마이그레이션).

### 6.2 §3.1 — agent-manager 격상 + 작성 노하우 공유 방식

  고유 30%만 각 매니저 `references/`(skill: triggers·번들·loop_mode / agent: 위임그래프·역할중첩·budget).
  장점: **드리프트 0**(우리가 설파하는 single-source 원칙과 일관). 단점: SKILL.md 번들 "references 폴더내
  1단계" 규약과 어긋나 마켓 export 시 inline 필요(=H4, 후순위).
- **(B) 각자 완전 보유(복붙)** — 폴더 자기완결·portable, 단 **DRY 위반 → 공통 규칙 2곳 동기화 부담**(우리가
  경고한 안티패턴).
- **(C) 지금은 skill-manager 만, agent-manager 보류** — 범위 최소, 단 비대칭 방치(어차피 v1.3.6 범위).

  점진공개(필요 시만 로드), 100줄 미만(참조 규칙 자체 준수).
