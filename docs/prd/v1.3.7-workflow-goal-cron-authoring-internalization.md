## 1. 배경 / 문제

v1.3.6 으로 **skill·agent 두 primitive 만** "작성 권위"를 갖췄다. 비교하면 나머지 세
primitive 의 생성 과정은 다음이 비어 있다:

| 차원 | skill·agent (v1.3.6) | workflow·goal·cron (v1.3.5) |
|---|---|---|
| 공유 작성 코어 | ✓ `skill-core/core.md`(~70%) | ❌ 없음 |
| 고유 작성 표준 | ✓ 각 매니저 `references/` | ❌ 없음(노하우가 ideation 문서에만) |
| 생애주기 | ✓ launch-your-agent 4페이즈 | ❌ 단순 CRUD |
| 수용 게이트 rubric | ✓ "리뷰 없이 실행할 자신이 드는가" 3~6 binary | ❌ 없음 |
| 결정적 validator | ✓ `validateSkill` / `agent validate --graph` | △ (cron✓·workflow✓ / **goal✗**) |

**핵심 통찰:** workflow/goal/cron 의 작성 노하우는 **이미 `260618-*` ideation 문서에 정리
되어 있다** — 매니저 skill 로 *내재화되지 않았을 뿐*이다. 매니저가 표준을 들고 있지 않으니,
Chief 가 이 자산을 만들거나 고칠 때 "좋은 것"의 단일 기준 없이 즉흥적으로 합성한다.

또한 결정적 계층에 **단 하나의 갭**이 있다 — goal 만 전용 validator 가 없고(`parseGoalFile`
의 inline throw 만 존재) `solosquad validate` 집계에서도 빠져 있다(`KINDS = skill·agent·
workflow·cron`). workflow·cron 은 이미 정합(`validateWorkflow`·`validateCronDef` + 집계 포함).

**세 번째 공백 — 인터뷰(암묵지 추출)가 agent 에만 있다.** primitive 가 생겨나는 경로는 4가지다:
⑴ 사용자 명시 지시 · ⑵ 등록 리포 마이그레이션(`adopt`/`add repo` 스캔) · ⑶ Chief 대화 중 실시간
감지 · ⑷ **시스템 자동 마이닝 제안**(나이틀리 `chief-compaction` 의 `trajectory-extractor`·
`freq-keyword-miner`, 제안만). 그런데 **사용자의 암묵지를 끌어내 자산화하는 인터뷰**는
`agent-manager/references/lifecycle.md` Phase 1(8-클러스터 enumerable)에만 있고, skill·workflow·
goal·cron 은 자유 텍스트로 즉흥 합성한다. 인터뷰의 목적은 폼-필링이 아니라 **사용자가 암묵적으로
아는 것(절차·판단·예외·품질 기준)을 끌어내 primitive 로 시스템화**하는 것이다 — 이게 §2.3 애자일
기획·§3.2.5 "암묵지→SKILL" 비전의 진입점이다.

## 2. 핵심 개념

- **매니저 = 작성 권위.** `{workflow,goal,cron}-manager` 가 단순 CRUD 가 아니라 **"좋은
  작성이 무엇인가"의 단일 기준**을 보유한다. v1.3.6 의 `{skill,agent}-manager` 와 동형.
- **표준은 점진공개로 담는다 — 단일 `primitive-core.md`.** 사용자 결정(2026-06-24):
  `skill-core/core.md` 를 **`primitive-core.md` 로 흡수·확장**해 5종 primitive 의 작성 표준을
  한 코어 파일에 둔다(§3.2). universal 레이어(철학·인터뷰·4-mode)가 cross-cutting 이고,
  skill/agent(워크스페이스 베이스) vs workflow/goal/cron(org 조립물) 차이는 §0 분류로 표현.
  매니저는 *작성/개선할 때만* 자기 섹션을 점프해 읽는다(lazy-load).
- **차용은 메커니즘, 차별화는 자산성.** v1.3.6 결론 그대로 — 선언적 `SKILL.md` 자산 + 정적
  validator + 번들 불변/org 오버레이가 우리 자리. ideation 의 업계 메커니즘(DAG cycle 검증·
  metric guardrail·IANA tz·idempotency·dead-man's-switch)은 적극 차용해 표준에 박제.
- **순서:** 본 버전은 **작성 표준 박제 + goal validator 정렬 + 번들 워크플로 재구성(§3.6) +
  기획 편향 가드(§3.7)**. 자가개선(eval-게이트 텍스트 학습)은 채점기가 전제이며, workflow/goal/
  cron 의 채점기 설계는 v1.3.6 §3.5① 의 연장으로 **후속(v1.3.8+)** 으로 미룬다 — 본 버전 비범위(§6).
- **Chief ↔ PM 역할 분담 (전제, 작성·실행·루프 소유권을 가른다).**
  - **Chief** = 사용자와 대화하며 요구를 **구체화→과제화**하고, 결과를 이해하기 쉽게 **보고**한다.
    과제를 PM 에 전달. 작업 최종 완료 시 **회고→루프 엔지니어링**(primitive 개선·메모리/세션 관리).
  - **PM** = 과제 달성을 위해 **워크플로를 관장**하고 각 agent 에 **업무 분장**, 실질 수행 스프린트를
    리딩. 단 메신저로 진행을 **가시화**(지시/토론/트러블슈팅/중간보고) — 내부에서만 돌지 않음.
  - **핸드오프:** PM 이 과제 완료 시 **Chief 에 먼저 결과 공유 → 이 지점에서 agent 간 리뷰** →
    OK 면 Chief 가 사용자 보고.
  - **본 PRD 함의:** 작성 인터뷰(암묵지 추출, §3.4)·case 감지·루프 엔지니어링(refine)은 **Chief**
    소유(사용자-대면 구체화 + 회고). 워크플로 *실행* 조립·업무 분장은 **PM**. 매니저 skill 의
    `used_by:["chief","product-manager"]` 가 이 분담을 반영 — Chief=작성/개선, PM=실행.

## 3. 스코프

### 3.1 매니저를 작성 권위로 격상 (workflow·goal·cron)

세 매니저 SKILL.md 를 v1.3.6 형판으로 재작성한다. 각 매니저 본문은 lean — **3대 요지 +
작성 권위 framing + CRUD 흐름에 표준 적용 단계 삽입**, 상세는 `skill-core/primitive-core.md` 로
점진공개. description 도 "…의 대화형 매니저이자 **작성 표준의 권위**" 로 격상.

- **workflow-manager** — 3대 요지: ⑴ stage = DAG 노드, 순환 금지(제출 시 검증) · ⑵
  exit_criteria = measure+operator+threshold(free-text 금지) · ⑶ 단순하게 시작(정적 체인
  < 정적 DAG < 동적). 기존 "템플릿 점수화 합성" 흐름은 유지하되 표준 적용 단계로 감싼다.
- **goal-manager** — 3대 요지: ⑴ 2층(영속 metric/제약 + 휘발 cycle) · ⑵ composite + ALL
  pass(단일 North Star = Goodhart 함정) · ⑶ 종료조건 명시(N연속 keep·budget·discard-streak).
  metric provenance(name/formula/source/threshold/direction) 를 생성 절차의 1급으로.
- **cron-manager** — 3대 요지: ⑴ IANA tz 이름(offset 금지) · ⑵ idempotent 프롬프트(정확히-
  한번 불가, 적어도-한번) · ⑶ translate-then-confirm + next-N 미리보기 + pause≠delete. 기존
  CRUD 단계(미리보기·테스트)는 이미 좋음 — 표준 framing 만 추가.

### 3.2 공유 코어 구조 — `core.md` → 단일 `primitive-core.md` (2026-06-24 확정)

5종 primitive 의 작성 표준을 **한 코어 파일**로 통합한다. 사용자 결정(2026-06-24): 2-파일
(skill-agent.md + workflow.md) 안을 폐기 — universal 레이어(철학·인터뷰·4-mode)가 cross-
cutting 이라 쪼개면 드리프트가 생긴다. skill/agent(워크스페이스 베이스) vs workflow/goal/cron
(org 조립물)의 성격 차이는 **파일 분리가 아니라 §0 분류 + 포맷별 섹션**으로 표현.

```
skills/skill-core/
  primitive-core.md   ← 기존 core.md 흡수·확장 (5종 단일 진실원)
```

구성(ToC, lazy-load — 매니저는 자기 섹션만 점프):

- **§0 primitive 분류 + 단위·조립 관계** *(신규, §2 사용자 통찰 박제)*
  - skill·agent = **워크스페이스 단위 베이스**(SKILL.md 포맷, 번들/글로벌·org 오버레이).
  - workflow·goal·cron = **org 단위 조립물** — stage/pipeline 이 **기존 skill·agent 를 참조**해
    구성된다(validator 가 agent-ref 실존 검사). ⇒ wf/goal/cron 작성 = *조립*이지 *발명*이 아니다.
  - **함의:** wf/goal/cron 을 만들 때 필요한 skill·agent 가 없으면 [[skill-manager]]·[[agent-
    manager]] 로 **베이스부터 만든 뒤 조립**. 마이그레이션 시 stage 의 agent-ref 가 워크스페이스
    베이스에 해소되는지 검증(없으면 함께 adopt).
- **§1 universal 작성 철학 (5종 공유 ~70%)** — 2층 분해(영속 정의 + 휘발 실행) · validation-as-
  gate(parse→semantic→pre-exec) · measurement contract(명시적 성공 정의·composite guardrail·
  bounded resource) · idempotency · event sourcing/audit · 메타데이터 디스커버리 · **simplicity**
  (start simple).
- **§2 인터뷰 — 암묵지 추출 (5종 공유, canonical)** — 목적=암묵지 시스템화 · **초안-앵커 4-mode**
  (§3.4B) · primitive별 추출 클러스터 표(§3.4A). 이 절이 4-mode 의 유일 진실원(포인터 불필요).
- **§3 SKILL.md 포맷 작성 (skill·agent)** — description 공식 · 정량 한도 · frontmatter 필드 감사 ·
  본문 절차규율+명명패턴 · 번들 구조(scripts/references/assets) · eval 골격. (= 구 core.md 본문.)
- **§4 조립 포맷 작성 (workflow·goal·cron)** — *§0 의 조립 원칙 위에서:*
  - **workflow** — stage DAG·`depends_on`/`handoff_to` 엣지·agent 바인딩(`<team>/<agent>`)·
    `_handoff.md` 스냅샷·timeout/retry·sub-workflow depth≤2.
  - **goal** — metric provenance(source 실존·formula 평가가능)·수렴 patience·discard-streak·
    loop-detection·비가역 액션 승인·Goodhart 실패모드 표.
  - **cron** — IANA tz/DST(00:00–03:00)·필드 수(5/6)·overlap(skip 기본)·catch-up(off)·jitter·
    dead-man's-switch.
- **§5 수용 게이트 rubric** — primitive별 3~6 binary(§3.3).

- **리네임 안전:** `core.md` → `primitive-core.md` 코드 경로 파손 0 — `core.md` 는 코드에서 *주석
  으로만* 언급(`src/bot/skill-parser.ts:83`). 텍스트 참조만 갱신(§5).
- **길이 예산:** 5종 통합이라 **<450줄** 목표. ToC + 섹션 점프로 lazy-load. 초과 시 §4 의 wf/goal/
  cron 블록을 각 매니저 `references/` 로 분할(후속). 토큰 부담은 작성 시점 한정.

### 3.3 생애주기 + 수용 게이트 rubric (primitive별)

v1.3.6 `agent-manager/references/lifecycle.md` 의 launch-your-agent 4페이즈(interview→stage→
grade/iterate→schedule) + "리뷰 없이 실행할 자신이 드는가" 패턴을 세 primitive 로 사상.
`primitive-core.md §5` 에 primitive별 binary rubric 으로 박제(ideation Part H 체크리스트가 직접 입력):

- **workflow** — acyclic? · 모든 stage id unique + agent 실존? · handoff/depends 도달가능
  (orphan 0)? · exit_criteria measurable? · stage 실패행동 정의? · 재실행 idempotent?
- **goal** — metric 5필드 완비 + source 실존 + formula 평가가능? · composite ALL-pass? ·
  pipeline agent 실존? · 수렴+종료 조건 명확? · 재측정 결정적? · 비가역 액션 승인 게이트?
- **cron** — tz=IANA? · next-N+readback 확인? · overlap 명시? · idempotent? · 다중 cron jitter?
  · pause≠delete? · min-interval 가드? · 조용한 미스 경보?

### 3.4 암묵지 추출 인터뷰 — 전 primitive + 4-mode (2026-06-24 추가)

agent 에만 있던 인터뷰를 **skill·workflow·goal·cron 으로 확장**한다. 두 축:

**(A) primitive별 추출 클러스터** — agent 8-클러스터(`lifecycle.md` Phase 1)에 대응하는 enumerable
셋을 각 primitive 코어에 박제. **클러스터 = 수용 rubric(§3.3) = validator 검사 필드와 1:1 정렬**
(인터뷰가 끌어내고 → rubric 이 채점하고 → validator 가 강제). 핵심은 **암묵지 클러스터**(굵게):

| primitive | 추출 클러스터 |
|---|---|
| **skill** | trigger(비명시 케이스 포함) · **procedure(습관적 절차)** · **gotchas(피하려 배운 실수·예외)** · I/O · non-goal · **verification(됐는지 판단 기준)** · bundle · shape |
| **workflow** | objective/done · stages(머릿속 DAG) · **handoff(다음 단계가 알아야 할 것)** · exit_criteria(측정가능) · agents · **failure(단계 실패 복구)** · simplicity check |
| **goal** | objective · metric(name/formula/source/threshold/direction) · **guardrail(개선하며 깨면 안 될 것=composite ALL-pass)** · pipeline · termination(수렴·포기) · 비가역 액션 승인 |
| **cron** | task · cadence · timezone · **overlap/catch-up(늦거나 겹치면)** · report · idempotency(2회 안전?) · **silent-miss(돌았어야 했는데 안 돈 기준)** |

- 클러스터는 **enumerable AskUserQuestion**(폼 심문 아님). 사용자/초안이 N개 클러스터를 이미
  채우면 **남은 것만** 묻는다(agent Phase 1 규율 동일).
- 위치: 5종 클러스터 표 모두 `primitive-core.md §2`(§3.2). agent 는 기존 `lifecycle.md` 유지하되
  클러스터 표를 코어로 상호참조.

**(B) 초안-앵커 4-mode (draft-anchored, cross-cutting)** — **인터뷰는 항상 매니저가 만든 초안에
앵커링**된다(백지 심문 금지). 4-mode 는 *초안의 원재료가 어디서 오고 얼마나 완성돼 있나*로 갈린다.
case 는 **Chief 가 감지해 `[creation_case:N]` 마커로** 매니저에 전달:

| case | 초안 원재료 | 인터뷰 동작 |
|---|---|---|
| ⑴ 명시 지시 | 사용자 첫 서술 | 매니저가 서술→초안 제시 → 빈 클러스터 추출 |
| **⑵ 마이그레이션(adopt)** | **아티팩트 역공학** | **매니저가 리포 아티팩트 분석→초안 제시 → 그 위에서 코드에 없는 WHY·판단·예외 추출**(핵심 — §3.4C) |
| ⑶ 대화 감지 | Chief 추론 shape | 추론 shape→초안 제시 → 미정 클러스터만 |
| ⑷ 마이닝 제안 | 마이너 패턴 | 패턴=초안 → "N회 반복, 진짜 절차? 트리거·판단은?" *규칙(암묵지)* 추출 |

- 4-mode 표·초안-앵커 원칙은 `primitive-core.md §2` 단일 진실원(포인터 불필요 — 단일 코어).
  agent 인터뷰도 본 4-mode 를 따른다(기존엔 full 만 암묵).
- **배선(P0):** case⑵ → `src/cli/adopt-offer.ts`/`adopt.ts` 적용 직전 — 매니저가 스캔된 아티팩트로
  *초안 enrich* 후 갭필 인터뷰. case⑷ → `src/cron/trajectory-extractor.ts` `applySuggestion` /
  `freq-keyword-miner.ts` confirm 경로 — 패턴 초안에 규칙 인터뷰. **스캐너/마이너 로직 불변** —
  인터뷰는 *적용 직전 enrich 단계*(adopt byte-identical·miner propose-only 안 깸; enrich=새 필드/
  본문 추가).

**(C) 마이그레이션 = 1급 흐름 (사용자 강조)** — adopt 는 단순 복사가 아니라 **분석→초안→인터뷰→
조립검증** 4스텝:
1. **분석** — 매니저가 리포 아티팩트(SKILL.md/workflow.yaml/goal.md/cron.yaml)를 역공학해 클러스터를
   *추정 채움*(예: workflow stage 시퀀스·agent-ref·exit_criteria 를 읽어 초안 구성).
2. **초안 제시** — 추정값 + **빈 칸(코드에 없는 것 = WHY·판단·예외·품질기준)** 을 명시해 사용자에 제시.
3. **인터뷰** — 빈 칸 위주로 암묵지 추출(case⑵ mode). "이 단계 왜? 언제 건너뛰나? 실패 시 어떻게?"
4. **조립 검증(§0)** — wf/goal/cron 이면 stage/pipeline 의 **skill·agent-ref 가 워크스페이스 베이스에
   해소되는지** 확인 — 없으면 해당 skill/agent 도 함께 adopt 하거나 [[agent-manager]]/[[skill-manager]]
   로 베이스 생성 후 조립(미해소 ref 로 등록 금지).

### 3.6 번들 워크플로 재구성 (2026-06-24 추가)

작성 표준을 박제하는 김에 **번들 워크플로 자체를 표준에 맞게 재구성**한다. 토대 = **워크플로의
본질 원칙**(아래) — `primitive-core.md §4` 의 workflow 절·인터뷰 클러스터(§3.4A)의 1급 기준.

**워크플로 본질 원칙 (핵심):** 워크플로는 *행위*가 아니다. **목표(goal) + 근거(rationale) +
방법(method) → 결론(conclusion) → 다음 단계로의 핸드오프**를 담은 *판단 단위*다. 단순 행위라면
워크플로가 아니라 **skill** 이어야 한다. ⇒ 모든 stage 는 "무엇을 왜 어떤 근거·방법으로 하고, 어떤
결론을 다음에 넘기는가"를 명시해야 하며, 이것이 없으면 그 stage 는 skill 한 줄로 충분하다.

**(A) 레거시 4종 전면 삭제** — `discovery-cycle`·`pmf-validation`·`autoplan-pm`·`weekly-retro`.
v1.1 산물로, 2-메인(new-build·improvement) 모델에 흡수됨(workflow-manager SKILL.md 의 "레거시→
프리셋/서브셋" 정리의 종착). 삭제 시 참조 정리 필수:
- `src/util/scaffold.ts`(add-org 시드 — `problem-definition` 외 레거시 시드 제거).
- `skills/workflow-manager/SKILL.md`(레거시 표 삭제)·`skills/primitive-review/SKILL.md`·
  `agents/specialists/data-analyst/SKILL.md` 의 참조.
- **`weekly-retro` 의 가치 substance 는 보존 위치 이동** — leading-indicators 집계는 이미
  `src/util/leading-indicators.ts` + `crons/leading-indicator.md` 에 있고, **회고→루프 엔지니어링은
  Chief 역할**(§2 Chief/PM)이므로 워크플로가 아니라 Chief 의 retrospective(+`retrospective`·
  `skill-refinement`·`workflow-refinement` skill)로 수행. pmf-validation 의 hard-gate `exit_criteria`
  패턴(유일하게 measurable 게이트 명시)은 **본질 원칙의 모범 예시로 `primitive-core.md §4` 에 인용**.
- test: `test/workflow-validate.test.ts` 등 레거시 id fixture 갱신.

**(B) 문제 정의 체인 분해 — skill↔workflow 재타입** — 현 monolithic `problem-definition`(5-skill
1체인)을 폐기하고, 본질 원칙으로 재배치:
- **scqa·five-whys·tdcc → 워크플로로 승격** — 각자 *목표·근거·방법→결론*을 담는 판단 단위(예:
  five-whys = "어디까지 왜 파고들고, 어떤 근원-결론을 다음에 넘기나"). 단순 변환이 아님.
- **mece·xyz-hypothesis → skill 유지** — 기계적 변환/생성(분해·가설문 생성)이라 행위 단위.
- **메인이 문제 성격에 따라 적합한 문제정의 워크플로를 선택** — Chief/PM 이 입력 성격으로 scqa
  (구조화 필요)·five-whys(근원 추적)·tdcc(지표 매핑) 중 적합한 것을 `_workflow/` 로 합성(전부 강제
  체인 금지 — 맥락 적응).
- **결정(2026-06-24): 흡수(replace).** 승격된 scqa/five-whys/tdcc 는 동명 skill 을 **제거하고
  워크플로가 그 절차를 흡수**한다(같은 이름이 skill·workflow 양쪽이면 타입 혼란 → 본질 원칙 정합).
  영향: `skills/{scqa,five-whys,tdcc}/` 삭제 + agent `skills_used`(product-manager·product-designer)
  에서 3종 제거(mece·xyz-hypothesis 는 skill 로 잔존하므로 유지) + 워크플로 stage 가 직접 절차 수행.

**(C) 잔존 워크플로를 본질 원칙으로 재서술** — 행위 나열을 *목표·근거·방법·결론*으로 승격:
- **idea-refinement** — "아이디어 고도화"는 *목표(stage)*일 뿐. "고도화를 위해 어떤 서브워크플로/
  프레임워크를 돌릴 것인가"로 재서술 — 입력 성격에 맞는 프레임워크 선택.
- **market-research** — 행위가 아니라 *어떤 목표·근거·방법으로 조사하고 어떤 결론을 다음에 넘기는가*.
- **kpi-check** — 단순 행위였다면 skill 이었을 것. 받은 과제를 바로 진행하기 전, **프로덕트 방향성·
  북극성 지표·기대 성과를 확인해 agent 팀 목표를 얼라인**시키고 그 위에서 PM 이 업무 분장하게 하는
  *정렬 게이트*. (목표·근거 = 방향성/북극성, 결론 = 얼라인된 KPI baseline → PM 업무분장.)

### 3.7 기획 에이전틱 워크플로 — 3대 편향 가드 (2026-06-24 추가)

기획 워크플로의 **prompt·context·harness·loop 엔지니어링**은 아래 3대 편향을 *경계·해결*하도록
설계한다. 각 편향은 **양 극단 사이의 균형점**이 핵심(한쪽만 막으면 반대 실패로 빠짐). →
`primitive-core.md §4`(워크플로 작성)에 가드로 박제 + 기획 워크플로 stage 프롬프트 규율.

| 편향 | 실패 모드 | 반대 극단(같이 경계) | 가드(설계 처방) |
|---|---|---|---|
| **① 자기부정** | 어제 긍정한 걸 오늘 부정, 내일 다시 번복 — 사용자가 AI 기획을 불신 | 논리적 전환·반성 없는 **무작정 고집** | **원칙 있는 수정만** — 입장 변경은 *명시적 논리/근거*가 있을 때만(없으면 일관 유지). LLM 구조적 원리(상태 없는 확률 재생성 → 매 턴 belief 재구성) 이해 위에서 **결정·근거를 durable-md(`_log.md`/`_handoff.md`)에 고정**해 턴 간 표류 차단. |
| **② 학습편향** | 사용자가 *가능한 사고 안에서만* 응답 — RAG·입력 사전지식·사용자 기획습관 한계에 갇혀 **단순 자동화**로 전락 | **할루시네이션** · 스프린트 스코프 벗어난 과잉 디테일 | 사용자 천장 위로 **업계 모범·표준 프레임을 근거와 함께 제시**(천장 상향)하되, 출처-근거(evidence_ref)·스코프 게이트로 환각·과잉기획 차단. "더 나은 기획"이 목표 — 자동화 아님. |
| **③ 확증편향** | AI 가 제 논리로 설명하면 사용자가 무의식적으로 *전부 맞다* 착각 → 사용자 기획이 AI 에 종속(주객전도) | (사용자 주체성 상실) | **가상 용어 창조 금지 — 업계 표준 용어·통용 개념만 사용**. AI 가 *자기 기획의 한계*를 명시(불확실·가정·반증조건)해 사용자가 검증 주체로 남게. anti-sycophancy(≥2 접근 비교)와 정합. |

- 이 3종은 **기획(planning) 워크플로 전반의 공통 가드** — `primitive-core.md §4` workflow 절에
  "기획 워크플로 설계 원칙"으로 명문화하고, new-build·improvement·문제정의·hypothesis 등 기획
  워크플로 stage task 프롬프트가 이를 참조.
- v1.3.6 자가개선(eval/refine)·v1.4.0 세션 메모리와 접점: ①은 durable-md(세션 표류 방지, v1.4.0),
  ②③은 eval 채점 기준(표준용어·근거·한계명시)로 후속 연결.
