## 1. 발견 경위

dogfood (2026-06-01) 에서 다음 증상:

1. `solosquad init` Discord 경로 완주 — 토큰 입력, handle 확인까지 정상
2. **앱(Application) ID 를 묻는 단계가 없음** — 사용자는 Developer Portal 에서 Application ID 를 복사해 둔 채 입력할 곳을 못 찾음
3. init 마지막에 **"Discord Invite URL" 블록이 아예 출력 안 됨** — 브라우저도 안 열림
4. 이후 `solosquad discord invite-url` 도 `✗ No bot_application_id found` 로 실패

→ v1.2.6 PRD §3.1 line 76/120-122 가 약속한 *"application_client_id prompt → invite URL 자동 합성 → 브라우저 open → click 1회"* 흐름의 **앞단 두 단계가 통째로 부재**.

## 2. 근본 원인 — 존재하지 않는 API 필드 1개

`src/cli/init.ts` 의 `fetchBotIdentity()` (discord 분기):

```ts
const res = await fetch("https://discord.com/api/v10/users/@me", { ... });
const body = (await res.json()) as { id?; username?; application_id? };
return { handle, botUserId: body.id, appId: body.application_id };  // ← 항상 undefined
```

**Discord 의 `GET /users/@me` 는 봇의 *User* 객체를 반환하며, 이 객체에는 `application_id` 필드가 없다.** (봇 User 객체 = `id` + `username` + avatar 등. application id 는 별도 리소스.) 따라서:

| 단계 | 결과 |
|---|---|
| `body.application_id` | 언제나 `undefined` |
| `BotIdentity.appId` | 언제나 `undefined` |
| init Step 4 invite URL 블록 (`if (... && identityChoice?.bot.appId)`) | **항상 skip** → URL 출력 / 브라우저 open 안 됨 |
| `user.yaml.bot_application_id` | `undefined` 저장 |
| 이후 `solosquad discord invite-url` | `bot_application_id` 없음 → 실패 |

게다가 **prompt fallback 도 없었다** — 자동 감지가 조용히 실패하면 사용자에게 물어볼 단계 자체가 코드에 없어, "앱 ID 를 안 물어본다" 는 증상으로 직결.

같은 결함이 `src/cli/doctor-discord.ts` Hop 2 (`liveAppId = me.application_id ?? null`) 에도 존재 → doctor 의 invite-url 힌트 + `bot_application_id missing` 경고도 항상 죽어있었음.

## 3. 수정

### 핵심 사실 2가지

- **정식 엔드포인트**: `GET /oauth2/applications/@me` (Bot 토큰 auth) 가 Application 객체를 반환하며 `.id` 가 곧 application(client) id.
- **불변식**: Discord 봇은 봇 User id == Application id (동일 snowflake). 따라서 위 호출이 실패해도 봇 user id 로 안전하게 폴백 가능.

### §A.1 — init.ts: 자동 감지 + 명시적 prompt

| # | 작업 |
|---|---|
| A.1.1 | `fetchDiscordApplicationId(token)` 신규 — `GET /oauth2/applications/@me` 로 app id 조회, 실패 시 `null`. |
| A.1.2 | `fetchBotIdentity()` discord 분기 — 존재하지 않는 `body.application_id` 의존 제거. `appId = (await fetchDiscordApplicationId(token)) ?? body.id` (봇 user id 폴백). |
| A.1.3 | `promptHandleSelection()` 에 **Discord Application(Client) ID 확인 prompt** 추가 (PRD §3.1). 자동 감지값을 기본값으로 Enter 수락. 감지 실패 시 *Developer Portal → General Information → Application ID* 붙여넣기 안내. `/^\d{17,20}$/` 검증, 빈 입력 skip 허용. discord-only (slack 은 별도 invite 모델). |
| A.1.4 | **owner User ID 자동 prefill** — `fetchDiscordApplicationId` 를 `fetchDiscordApplication` 로 확장해 `application.owner.id` 도 반환 (team app 은 `owner` 가 합성 team user 라 skip). owner-only 게이트 prompt 가 이 값을 default 로 Enter 수락. 봇 토큰으로 *유일하게* 알 수 없는 값은 *사람 운영자의 user id* 인데, 솔로 파운더 케이스(Developer Portal 계정 == 명령자)를 자동으로 덮음. team app / 타인 명령 케이스는 기존대로 수동 입력 또는 first-message hydration. |

### §A.2 — doctor-discord.ts: Hop 2 app id 채움

## 4. v1.2.6 의 *모든 게이트 통과* 원인 분석

`tsc`, test, docs-check 모두 PASS 한 채 invite URL 흐름이 죽어 출시된 이유:

| 게이트 | 왜 안 잡았나 |
|---|---|
| `tsc --noEmit` | `body.application_id` 는 *타입상* 적법한 optional 필드 접근. 런타임에 Discord 가 그 필드를 안 줄 뿐 — 타입 시스템 밖. |
| `npm test` | `discord-invite-url.test.ts` 는 `buildInviteUrl()` (순수 함수, app id *주어졌을 때*) 만 검증. app id 가 *어디서 오는지* (`fetchBotIdentity`) 는 네트워크 호출이라 unit-test 안 됨. |
| `docs-check` | 문자열 매칭만. API 필드 존재 여부 영역 밖. |
| Manual 재현 | init 을 *끝까지* 돌려 invite URL 블록까지 가야 발견 — 대부분 토큰/handle 단계에서 멈춤. |

→ 교훈: 외부 API 응답 shape 가정은 *실제 응답 1건* 으로 검증해야 한다. `appId` 는 v1.2.6 publish 이래 단 한 번도 채워진 적이 없다.
