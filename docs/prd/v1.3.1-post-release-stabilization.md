## 1. 발견 경위

`solosquad@1.3.0` publish 후 PR #8(`feat/v1.3.0-messenger-ux-overhaul` → `main`)
머지를 시도하자 CI 가 전 플랫폼 red. v1.3.0 코드/의존성 변경이 유발한 게 아니라
**main 도 이미 같은 게이트로 red** 였던 사전존재 조건이, "머지 전 green" 기준 때문에
비로소 정면으로 드러난 케이스.

핵심 진단 함정 2개:
- **fail-fast 가 진짜 원인을 가렸다.** matrix `fail-fast`(기본 true) 때문에 Node 18
  잡이 `npm ci` 에서 죽으면 나머지 Node 20/22 잡이 *컴파일 도중 cancel* → "전부 fail"
  로 보여 어느 조합이 실제 통과하는지 알 수 없었음.
- **audit fail-fast 가 빌드 실패를 가렸다.** 첫 런에서 `npm audit` 게이트가 한 잡에서
  먼저 터지자 나머지가 cancel → Node 18/20 의 native-build 실패가 첫 런엔 안 보였음.

## 2. 근본 원인 (서로 다른 3건)

## 7. skill-manager — `skill-author` 확장 (신규 후속, 기획)

> **위치 주의:** 본 §7 은 v1.3.1 의 원래 범위(CI/deps 하드닝)와 무관한 **신규 기능 기획**이다.
> 2026-06-17 SKILL.md 작성법 전수 조사([`docs/ideation/260617-skill-md-authoring-best-practices.md`])
> 에서 도출된 후속을 박제한다. 코드 0줄(기획만) — 실제 구현은 별도 브랜치/버전에서 진행해도
> 무방하며, 규모가 커지면 독립 PRD(예: v1.4.x)로 승격한다.

### 7.1 배경 / 문제

현재 `src/bot/skill-author.ts` 는 SKILL.md 를 **1-shot 생성**만 한다. 위 ideation 조사 결론:

- **검증 갭:** 우리 `validateSkill`(`skill-parser.ts`)은 공식/표준 규칙 다수를 미검사 —
  `name` ≤64자·`^[a-z0-9]+(-[a-z0-9]+)*$`·예약어(`anthropic`/`claude`)·**부모 디렉터리명 일치**,
  `description` ≤1024자·**3인칭(1인칭 린트)**·vague 휴리스틱·"Use when" 트리거절 부재.
- **개선 루프 부재:** 생성 후 품질을 *측정·개선*하는 경로가 없음. 업계는 이미 **SkillOpt**
  (MS, [arXiv](https://arxiv.org/pdf/2605.23904))식 rollout→reflect→bounded-edit→held-out
  검증 게이팅으로 SKILL.md 를 *학습*시킨다(스텝당 4–8 편집, 효과적 스킬 중앙값 ~920토큰).
- 즉 `skill-author` 라는 이름은 "생성"만 함의해 역할이 좁다.

### 7.2 변경: `skill-author` → `skill-manager`

`src/bot/skill-author.ts` → `src/bot/skill-manager.ts` 로 개명하고, 단일 "저자"에서 SKILL 의
**전 생애주기 관리자**로 확장한다. 4개 역량:

| 역량 | 동사 | 설명 | 토대(기존) |
|---|---|---|---|
| **생성(create)** | author | 표준 description 공식("`&lt;무엇을&gt;` — `&lt;구조&gt;`. `&lt;출처&gt;` 기반. 사용 시점: A,B,C. 제외: X")으로 SKILL.md 초안 생성. 3인칭·트리거 첫 문장·약간 pushy. | `skill-author.ts` |
| **검토(review)** | review | 생성/수정된 SKILL 을 ideation Part I 체크리스트로 LLM 검토 — description 품질, 본문 <500줄, 참조 1단계, non-goal 유무, 절차적 규율 중심성. | `skill-pr-review/`(PR 리뷰 기존) |
| **검사(validate)** | validate | §7.1 갭 포함한 정적 검증 강화(`validateSkill` 확장). 공식 정량/문체 + SoloSquad 확장(`schema_version`·slash 예약어·freq cap·`dev_permissions`). | `skill-parser.ts validateSkill` |
| **개선(refine)** | refine | SkillOpt 식 최적화 루프: corpus rollout → 성공/실패 반성 → **bounded edit(4–8/스텝)** → held-out 검증 시에만 채택 + 거부 buffer. protected-section 으로 내구 지침 보호. | `validator-corpus.ts` + `test:corpus`(eval 골격) |

### 7.3 구현 노트

- **개명 영향 범위:** `skill-author.ts` import 처 + `validate-skills`/author 호출부 grep 후 일괄.
  공개 CLI 표면(`solosquad agent …`)이 `author` 서브커맨드를 노출하면 하위호환 alias 유지.
- **검사(validate) 강화는 단독 선출고 가능**(P0, 가장 싸고 corpus 호환 ↑). 생성·검토·개선은 P1.
- **refine 는 우리 강점 재활용:** `skill-author`(=공식 "Claude A") + `validator-corpus`/`test:corpus`
  (=eval) 골격을 이미 보유 → SkillOpt 루프를 *얹는* 형태라 신규 인프라 최소.
- **검증 게이트 연동:** `npm run validate-skills`(현 `agent validate --all && test:corpus`)에
  강화된 validate + (선택) `skills-ref validate` 를 합류.

### 7.4 범위 / 비범위

- **In:** 개명 + 4역량(create/review/validate/refine)의 인터페이스 설계, validate 강화(P0).
- **Out(후속):** 마켓플레이스 배포 경로(`plugin.json`/kebab 정렬), 어휘 정렬(`dev_permissions`↔MCP
  annotations) — ideation Part H4·H5. stateful 스킬(v0.6 trajectory)과의 protected-section 통합.

## 8. stale `assets/agents/` 제거 (신규 후속, 정리)

  재지정. stale doc-comment(`agents-builder.ts`·`frontmatter-diff.ts`)도 정정.
- **AGENTS.md** — v1.1 flat 에이전트 레이아웃 + `teams/` 를 캐노니컬로 문서화.

### 9.3 작업 노트 / 비범위

- **이중 상태(knowledge·routines) 우선 해소가 가장 싸다(P0).** 리졸버(`getKnowledgeDir`·
  `getRoutinesDir`)를 최상위(`getBundleRoot()/knowledge`·`/schedules`)로 재지정 + 옛
  `assets/` 사본 제거. 단 `getRoutinesDir` 를 참조하는 마이그레이션 스크립트(0.5.0/0.8.2)와
  스케줄러 경로 회귀를 함께 검증해야 함.
- **templates 분산은 가장 무겁다(P1).** 22개를 사용처 skill 의 `assets/` 로 흩고 bootstrap
  3종(AGENTS.md·agent-profile.yaml·hooks.json)만 init 전용 경로로 이관 — `goal.ts` +
  과거 migration 스크립트 다수가 `getAssetsDir()/templates` 를 직접 참조해 영향 범위 큼.
- **Out:** 본 §9 는 추적만. 실제 이전 시 번들 화이트리스트(`package.json.files`, v1.2.3
  교훈) 회귀 + `npm pack --dry-run` seed-경로 grep 검증을 동반해야 함. 규모상 v1.4.x/별도
  PRD 승격 권장.
