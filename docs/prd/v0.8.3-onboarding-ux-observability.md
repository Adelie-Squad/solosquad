# v0.8.3 — Onboarding UX + Observability

> v0.8.x 시리즈의 마지막 패치. 사용자가 *처음 SoloSquad를 만났을 때*의 경험과 *문제가 생겼을 때 디버깅*하는 경험을 잡는다. 다섯 축: (a) **기존 리포 마이그레이션 UX** — `add repo --dry-run` + 폴더 매핑 표 + 위험 시나리오 감지, (b) **master-guide v0.7→v0.8 재정합** — v0.7 출시 시 §6.1/§8.1만 갱신했던 drift 흡수, (c) **logger + `solosquad logs` CLI** — 구조화 로그·파일 출력·rolling, (d) **`solosquad logout` 제거** — v0.7에서 추가된 명령 가치 < 복잡도로 판정, 명령 축소, (e) **업데이트 ↔ 마이그레이션 구분 명확화** — `update`(npm) vs `migrate`(workspace schema) 흐름도. 부수: v0.6에서 보류한 **trajectory 자동 등록 ROI 게이트 결정**도 v0.8.3 출시 직전 측정 후 박제.

**로드맵:** v0.8 Multi-User Messenger → v0.8.1 Security & Lifecycle Pair → v0.8.2 Dev Capability → **v0.8.3 Onboarding UX + Observability** → v0.9 안정화 → v1.0 정식 출시

**의존:** v0.5 (analyze + add repo 기본 흐름), v0.6 (trajectory extractor + freq miner — 자동 등록 게이트 대상), v0.7 (master-guide §6.1/§8.1에서 첫 v0.7 절 추가 완료)

**기준 버전:** v0.8.3. v1.0 진입 직전 마지막 polish 패치.

**분류:** UX + Docs + Observability. 새 기능 추가 적음 (logs CLI 1개), 대부분 갱신.

## 3. 기존 리포 마이그레이션 UX

### 3.1 시나리오

사용자 alice가 이미 product code를 `~/projects/my-saas/`에서 작업하고 있다. SoloSquad를 시작하면서 이 리포를 SoloSquad 워크스페이스 안으로 들여오고 싶다.

### 3.2 권장 5단계

```
[현재 상태]
~/projects/my-saas/                    # alice의 기존 리포
  ├── .git/
  ├── src/
  ├── package.json
  └── ...

[Step 1] 기존 작업 정리 + commit + push
  cd ~/projects/my-saas
  git status                           # untracked·unstaged 0건 확인
  git push origin main                 # 안전망

[Step 2] SoloSquad 초기화
  mkdir ~/solosquad && cd ~/solosquad
  solosquad init                       # 워크스페이스 + 첫 org 생성 (예: "main")

[Step 3] dry-run으로 이동 미리보기 (v0.8.3 신규 플래그)
  solosquad add repo ~/projects/my-saas --dry-run
  → 출력:
     "From: /Users/alice/projects/my-saas
      To:   /Users/alice/solosquad/main/repositories/my-saas
      Files: 1,247 (32.5 MB)  .git: 8.1 MB
      Added: <repo>/.solosquad/repo.yaml (1 file)
      Untouched: 1,246 user files
      IDE workspace files detected: .vscode/settings.json (paths preserved)
      Symlinks pointing INTO this repo from elsewhere: 0 found
      Active processes holding files (lsof): NONE
      Risks: none detected"

[Step 4] 실제 이동
  solosquad add repo ~/projects/my-saas
  → 결과:
     "Moved → /Users/alice/solosquad/main/repositories/my-saas
      Registered in .org.yaml
      .git history preserved (size unchanged)"

[Step 5] 검증 + 정리
  solosquad sync                       # repositories/ ↔ .org.yaml 정합
  cd ~/solosquad/main/repositories/my-saas
  git status                           # working tree clean
  git log -1                           # 직전 commit 그대로
```

### 3.3 폴더 매핑 표

| 사용자 자산 | 이동 전 위치 | 이동 후 위치 | 변경 |
|---|---|---|---|
| 메인 product code 리포 | `~/projects/<name>/` | `~/solosquad/<org>/repositories/<name>/` | 디렉토리 이동 |
| `.git/` 히스토리 | `~/projects/<name>/.git/` | `~/solosquad/<org>/repositories/<name>/.git/` | 그대로 (`mv` 의 일부) |
| IDE workspace 파일 | `~/projects/<name>/.vscode/` 등 | `~/solosquad/<org>/repositories/<name>/.vscode/` | 그대로. 상대경로 설정이면 그대로 작동. **절대경로 설정이면 사용자가 수동 갱신** |
| `.env.local` 등 secret | `~/projects/<name>/.env*` | `~/solosquad/<org>/repositories/<name>/.env*` | 그대로 (user-authored 파일은 절대 SoloSquad가 변형하지 않음) |
| SoloSquad 메타 (신규) | 없음 | `~/solosquad/<org>/repositories/<name>/.solosquad/repo.yaml` | 신규 1개 파일. 다른 SoloSquad 파일은 repo 안에 절대 안 만듦 (v0.7 클래스 A* 룰) |

### 3.4 위험 시나리오 + dry-run 감지

| 위험 | dry-run이 감지 + 출력 |
|---|---|
| 이동 도중 IDE/dev server가 파일을 열고 있어 lock | `lsof <path>` (Unix) / `handle.exe` (Win) 호출 → 활성 프로세스 발견 시 경고 |
| 다른 곳에서 절대경로 심볼릭 링크가 이 리포를 가리킴 | 부모 디렉토리 1단계 + `~/`에서 `readlink -f` 스캔 → 발견 시 경고 |
| repo 내부에서 상위 디렉토리로 가는 절대경로 참조 (config·docs) | `grep -rn "/projects/<name>/"` 같은 패턴 매칭 |
| 외부에서 cron/스크립트가 `~/projects/<name>/`를 가리킴 | SoloSquad가 모르는 영역 — 안내 메시지로만 (cron·alias·CI runner 경로 갱신 권고) |
| 같은 org에 같은 slug 리포가 이미 존재 | `add repo` 거부. 사용자가 `--slug <new>` 또는 다른 org 지정 |

### 3.5 신규 CLI 플래그

| 플래그 | 동작 |
|---|---|
| `--dry-run` | 이동 시뮬레이션. 디스크 변경 0 |
| `--inspect <path>` | dry-run의 별칭. 자체적인 분석 보고서 출력 (`tree`, 사이즈, IDE 설정, 심링크) |
| `--keep-original` | 이동 대신 *복사*. 원본을 그대로 두고 워크스페이스에 사본 등록. `.git/` 도 복사. 디스크 2× 사용 |
| `--from-report <md>` | v0.5의 analyze 보고서 기반 머지 (기존 기능, 명시) |

- §4.5 "기존 리포 가져오기" 신설 — 위 5단계 + 폴더 매핑 표
- §4.6 "위험 시나리오" — §3.4

## 5. logger 확장 + `solosquad logs` CLI

### 5.1 logger 확장

`src/util/logger.ts`:
- 로그 레벨: `error / warn / info / debug` — 환경변수 `SOLOSQUAD_LOG_LEVEL` (기본 `info`)
- 구조화 JSON 출력 옵션 (`SOLOSQUAD_LOG_FORMAT=json` — 기본 `text`)
- 파일 출력: `SOLOSQUAD_LOG_FILE=1` 시 `<workspace>/.solosquad/logs/solosquad-YYYY-MM-DD.log` (rolling)
- 보존: 최근 14일. 그 이전 파일은 자동 삭제 (rotate routine 일일 실행)
- 기존 API (`logger.info/warn/error`) 그대로 — backward-compat 보장

### 5.2 `solosquad logs` CLI

```bash
solosquad logs [--level X] [--tail N] [--follow] [--since "1 hour ago"]
               [--type runtime|costs|spawn|stop-hook|dev-confirm|migration]
```

| 플래그 | 동작 |
|---|---|
| `--level X` | error/warn/info/debug 필터 |
| `--tail N` | 최근 N줄 (default 50) |
| `--follow` | tail -f 모드 |
| `--since "1 hour ago"` | 시간 필터 (chrono natural language) |
| `--type X` | runtime(기본 — solosquad-YYYY-MM-DD.log) / costs(agent-costs.jsonl) / spawn(spawn-decisions.jsonl) / stop-hook(stop-hook-events.jsonl) / dev-confirm(dev-confirmations.jsonl) / migration(migration-costs.jsonl) |

다중 type 동시 조회:
```bash
solosquad logs --type runtime --type costs --tail 100 --follow
```

### 5.3 rotate routine

`assets/routines/log-rotate.md` 신규 — 매일 00:30 실행:
- `<workspace>/.solosquad/logs/`에서 14일 이전 파일 삭제
- 다음 날짜 파일로 자동 전환 (logger.ts 측에서 이미 처리하지만 retention만 routine이 보장)

### 5.4 신규 모듈

| 파일 | 역할 |
|---|---|
| `src/util/logger.ts` 확장 | 로그 레벨 + 파일 출력 + rolling |
| `src/cli/logs.ts` (신규) | `solosquad logs` CLI |
| `assets/routines/log-rotate.md` (신규) | 일일 retention |

## 6. `solosquad logout` 제거 + 명령어 축소

### 6.1 `solosquad logout` 제거 사유

v0.7에서 gh CLI 패턴 차용으로 추가했지만 **실제 가치 < 복잡도**로 판정. v0.8.3에서 제거.

| 기능 | logout이 했던 것 | 더 단순한 대체 |
|---|---|---|
| 봇 일시 정지 | `logout.lock` 드롭 → bot/schedule 진입 거부 | `Ctrl+C`로 봇 프로세스 종료. 다시 시작은 `solosquad bot` |
| .env 시크릿 마스킹 | `*TOKEN/*KEY/*SECRET` 패턴 매칭 후 `***REDACTED-AT-<ts>***` 치환 | 사용자가 .env 편집 또는 토큰 자체를 messenger 콘솔에서 revoke (REVOKE-CHECKLIST.md가 이미 안내) |
| sessions `_archived/`로 이동 | session-id 파일 이동 | 사용자가 `<org>/.solosquad/sessions/`에서 수동 이동 가능. 또는 `solosquad pm reset --user <id>`로 신규 발급 (이미 존재) |
| REVOKE-CHECKLIST.md 생성 | workspace root에 markdown 동봉 | `solosquad uninstall --archive-only`가 이미 동일 markdown 생성 + REVOKE-CHECKLIST는 archive에도 들어감 |

**결정**: `solosquad logout` CLI 제거. 관련 코드(`src/cli/logout.ts`·`logoutLockPath`·`bot/schedule이 logout.lock 확인`)도 함께 제거. 대체 흐름은 master-guide §9/§10에 안내.

### 6.2 v0.7→v0.8.3 사용자 transition

기존 v0.7 사용자(개발자 본인만)는 0건이므로 *backwards compat layer 없음*. master-guide에서 logout이 v0.8.3에서 제거됐다는 1줄 안내만.

### 6.3 다른 명령 축소 후보 (사용자 검토 필요)

본 패치에선 *제거하지 않음*. 별도 v0.9 명령 축소 라운드에서 결정:

| 후보 | 현재 역할 | 축소 검토 사유 |
|---|---|---|
| `solosquad goal show` + `solosquad goal status` | show=spec+cycles / status=runtime info | 사용자에게 차이가 모호. `show --runtime` 같은 단일 명령으로 합치는 게 깨끗 |
| `solosquad pm compact` | PM session 강제 compaction (수동 트리거) | 23:00 daily routine이 이미 처리. 디버깅용으로만 쓰임. `solosquad pm reset`이 더 명확 — 합칠지 검토 |
| `solosquad agent reload` | fs_watch.mode=manual에서 라우터 재빌드 | `fs_watch.mode: auto`가 기본이라 거의 안 쓰임. doctor 하위 명령(`doctor --rebuild-router`)로 흡수 가능 |

위 3건은 v0.9 candidate. v0.8.3에서는 *결정 미수*.

## 7. 업데이트 ↔ 마이그레이션 구분 명확화

### 7.1 현재 혼동

- `solosquad update` — npm `latest` 확인 + 자동 self-update (CLI 자체 바이너리)
- `solosquad migrate` — 워크스페이스 schema/layout을 새 CLI 버전에 맞게 정합

두 명령은 이미 분리돼 있지만 매뉴얼에 흐름도가 없어 *언제 어느 쪽을 써야 하는지* 사용자가 혼동.

### 7.2 명확화 (master-guide §6 신규 절)

```
[새 npm 릴리스 감지 시 흐름]

1. solosquad doctor
   ├─ "CLI: v0.7.0,  workspace: v0.7.0" → OK
   ├─ "CLI: v0.8.0,  workspace: v0.7.0" → "solosquad migrate --apply 권고" 출력
   └─ "CLI: v0.6.0,  workspace: v0.7.0" → "npm install -g solosquad@latest 또는 solosquad update 권고"

2. CLI 업그레이드:
   solosquad update          # npm registry 확인 + 자동 self-install
                              # 또는 npm install -g solosquad@latest 직접

3. (CLI가 새 버전이 되면) 워크스페이스 정합:
   solosquad migrate --dry-run    # 무엇이 바뀔지 미리보기
   solosquad migrate --apply      # 적용 (자동 backup → apply → verify)
   solosquad migrate --rollback   # 직전 backup으로 복원 (필요 시)

차이 요약:
- update    = CLI 바이너리(npm 패키지) 자체를 갱신
- migrate   = 워크스페이스(.solosquad/·org/·workflows/·memory/ 등)의 schema/layout을 새 CLI 버전에 맞게 변환
```

### 7.3 doctor 보강

`solosquad doctor`가 출력에서 CLI version ↔ workspace version mismatch 감지 시 어느 명령이 필요한지 explicit하게 안내:

- CLI > workspace (사용자가 CLI 업그레이드만 함): "migrate를 실행하세요"
- CLI < workspace (사용자가 workspace를 더 새 버전으로 migrate했지만 CLI는 옛 버전): "npm install -g solosquad@latest"
- CLI == workspace: pass

### 7.4 마스터 가이드 trouble shooting 추가

§10 FAQ에 1건:
- "솔로스쿼드 업데이트가 안 됨" → 두 명령의 차이 + 흐름도 다시 안내

## 8. trajectory 자동 등록 게이트 결정

### 6.1 v0.6 §3.X에서 정의된 ROI 4지표

| 지표 | 충족 기준 |
|---|---|
| 30일 내 trajectory 제안 N건 | ≥ 5 |
| 제안 채택률 | ≥ 60% |
| 채택된 SKILL의 다음 30일 사용률 | ≥ 평균 |
| 사용자 reject cooldown 패턴 | < 30% |

### 6.2 v0.8.3 출시 직전 측정

- `solosquad memory stats --event-type author_turn` 등으로 30일 누적 데이터 추출
- CHANGELOG.md `[0.8.3]` entry에 측정값 명시 (예: "제안 7건, 채택 5건 (71%), 다음 30일 사용률 평균 +12%, reject cooldown 패턴 22% — 4지표 모두 통과")

### 6.3 결정 박제

| 결과 | 후속 |
|---|---|
| 4지표 모두 통과 | v0.9에 자동 등록 활성. `workspace.yaml.trajectory.auto_register: true` 기본값 |
| 1개라도 fail | **"제안만 영구"** 잠금. `auto_register`는 cli 플래그로 강제 활성만 가능, default 변경 안 함 |

본 패치는 *측정 + 결정 박제*만 함. 자동 등록 코드 자체는 v0.9에서 추가 (조건 충족 시).

## 10. 비파괴성 + 복구

- add repo dry-run은 디스크 변경 0 (mock fs 단위 테스트 포함)

- logger 확장은 backward-compat — 기존 호출자 코드 0 변경
- trajectory 자동 등록은 본 패치에서 활성화 안 함 — 결정만 박제
