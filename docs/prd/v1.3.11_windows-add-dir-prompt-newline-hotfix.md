> 직후 dogfood 봇에서 *여전히* 등록 repo 를 못 읽는 게 드러난 **2차 결함** 핫픽스. `prd` 스킬
> **핫픽스 양식**(증상→근본원인→영향→해결→복구→회고). 선례 `v1.2.3`·`v1.3.9`.

> **로드맵 위치:** v1.3.10 (게시됨 — Windows 2차 버그 잔존) → **v1.3.11 (본 hotfix)** → 다음.
> 1.3.10 사용자(특히 Windows)는 1.3.11 로 업그레이드.

## 1. 발견 경위

dogfood 사용자가 1.3.10 으로 업데이트(`solosquad update`) + 봇 재시작 후에도 Discord 에서
"bv-po-homepage-nextjs readme 읽어봐" → **여전히** *"Claude requested permissions to read … but
you haven't granted it yet"* 거부. 봇은 확실히 1.3.10(전역 dist 1.3.10, 봇 프로세스 그 이후 시작).

이상한 점: **같은 세션·같은 repo 를 직접 claude 로 resume 하면 읽힘.** 에이전트 로그에 "방금 두 번은
승인되어 읽혔다"(= 디버깅 중 bash 로 돌린 수동 테스트) vs "봇 자신의 Read 만 거부". 봇의 spawn 경로를
그대로 재현(Windows `shell:true` + `quoteWindowsArg` + **multi-line `--append-system-prompt`** + --add-dir)
하니 → **재현됨(거부).** `--append-system-prompt-file` 로 바꾸니 → **읽힘.**

## 2. 근본 원인

봇은 Windows 에서 `spawn(cmd, [], { shell: true })` 로 claude 를 띄우며, args 를 **하나의 명령
문자열**로 조립한다(`spawnClaude`/`quoteWindowsArg`). Chief 시스템 프롬프트(`--append-system-prompt`
값)는 `\n\n[identity]…\n\n[surface]…\n\n[permissions]…` 처럼 **줄바꿈을 포함**한다. cmd.exe 명령문
안의 줄바꿈은 명령을 끊어, **`--append-system-prompt` *뒤*에 오는 플래그가 잘려나간다** —
빌드 순서상 `--add-dir`(과 `--disallowed-tools`)가 그 뒤라 **`--add-dir` 가 claude 에 전달되지 않는다.**

→ 1.3.10 의 `--add-dir` 는 *논리적으로는* 옳게 빌드되지만(`buildArgs` 에 존재), Windows 셸 조립
단계에서 소실. (1.3.10 이 고친 `--input-format stream-json` 무시와는 **별개의 2차 버그** — 둘 다
같은 증상.) bash 재현이 통했던 건 bash 가 cmd.exe 와 줄바꿈 처리가 달라서.

## 3. 영향

- **Windows + 멀티라인 `--append-system-prompt`(=항상)** 인 모든 봇 spawn 에서 `--add-dir` 소실 →
  등록 외부 repo Read/Edit 차단. macOS/Linux 는 shell 없이 args 배열로 spawn 하므로 비영향.
- 1.3.10 의 다른 수정(plain-stdin 입력·권한 모델)은 정상 — 이 버그만 `--add-dir` 를 무력화.

## 4. 해결책

### 4.1 즉시 (코드)

시스템 프롬프트를 **임시 파일로 빼고 `--append-system-prompt-file <path>` 사용** — 멀티라인 값이
명령줄에 아예 안 올라가므로 줄바꿈이 cmd.exe 를 깨지 않고, 뒤의 `--add-dir` 가 보존된다.
- `claude-process.ts`: `writeSystemPromptFile()`(temp 파일 작성) · `buildArgs(inv, promptFile?)`
  (있으면 `--append-system-prompt-file`, 없으면 inline fallback) · `invokeStreaming` 이 spawn 전
  파일 작성, `close`/`error` 에서 삭제(best-effort).
- 크로스플랫폼 적용(파일 방식은 macOS/Linux 도 안전·동등). 회귀 테스트로 "멀티라인 프롬프트면
  `--append-system-prompt-file` 사용 + `--add-dir` 보존" 박제.

### 4.2 사용자 복구 절차

```
solosquad update            # npm 최신(=1.3.11) 설치
solosquad migrate --apply   # 워크스페이스 1.3.10 → 1.3.11 (no-op 범프)
# 봇 재시작 (Ctrl+C → solosquad bot)
```
재시작 후 등록 repo 읽기가 승인 없이 됩니다. (세션 resume 도 fix 적용 — spawn 매 턴 재생성.)

## 6. 회고 — 왜 1.3.10 이 못 잡았나

- 1.3.10 디버깅의 격리 재현이 전부 **bash** 였다 — bash 는 cmd.exe 와 줄바꿈 파싱이 달라 `--add-dir`
  소실이 안 나타남. **봇의 실제 spawn 경로(Windows `shell:true` + 멀티라인 arg)를 그대로 재현하지
  않은 것**이 sleeping bug 를 남겼다.
- 후속 가드: (a) Windows spawn 은 **멀티라인 인자를 명령줄에 올리지 않는다**(파일/stdin 경유)를
  원칙화. (b) 장기적으로 `shell:true` 명령-문자열 조립 대신 args 배열 spawn(claude.cmd 직접 해석)
  검토 — newline 외 다른 셸 메타문자 리스크도 원천 제거.
