# v1.2 — Messenger Connection (Chief on Discord, auto-connect first)

> **핵심 1줄:** 조직 1개당 Discord 봇 1개 (= v1.1 Chief, 사용자가 이름 부여). 사용자는 *자기 Discord ID 가 owner 인 채로* Chief 와 자연어 대화 → Chief 가 TRIAGE 단계에서 *단순 논의* 와 *작업 단위(workflow/schedule/goal)* 분기 → 후자는 `works-<handle>` 채널에 과제 등록 + thread 생성, 에이전트 진행이 그 thread 로 broadcast. 자동 연결 최대화 — Discord 정책상 manual 인 단계만 사용자가 click.
>
> **v1.1 과의 정합:** v1.1.0 의 `<org>/agents/main/chief/SKILL.md` + `src/bot/chief-runner.ts` + `<org>/memory/chief-stage-events.jsonl` 가 **이미 사용자 대면 single agent** 임. v1.2 는 그 single agent 를 Discord 표면에서 *살아 움직이게* 하고, 추가로 *조직 단위 정체성 (Chief 이름)* + *owner-only 게이트* + *works-handle 과제 허브* 를 신설 — multi-bot 분기 없음.

**로드맵 위치:** v1.1.0 ✓ 출시 (multi-agent team) → **v1.2 본 plan (Discord 자동 연결 + 조직 add-org)** → v1.2.x (Slack sub-version) → v1.x (대시보드 · 지식 온톨로지 · LLM backend)

**의존:**
- **v1.1.0** (Chief / 4 main / 20 specialist 구조, chief-runner, chief-stage-events, open_questions[]) — *hard dependency, 출시 완료*
- v0.8.0 (`<org>/.solosquad/users/<handle>.yaml`, `resolveBotIdentity`, `command-<handle>` / `works-<handle>` 채널 페어, `messenger_user_id`)
- v1.0.1 (`@<slug>` mention parser)
- v1.0.2 (author-guard 제거 — *본 v1.2 가 owner-id 게이트 형태로 부분 reversal*, §4.5)
- v1.0.3 (Discord guild-org binding via `ownOrgSlug`, ensureChannels, category lookup)

**기준 버전:** v1.2.0 (Discord) → v1.2.1+ (Slack sub-version). 신규 인프라 5건 (OAuth Invite URL generator + guildCreate onboarding flow + thread-per-workflow + owner-only 게이트 + works-handle 과제 허브), 신규 CLI 2건 (`solosquad discord invite-url`, `solosquad add-org` 보강), schema 변경 5건 (§13.3).

## 0. v1.1.0 → v1.2 narrative 정합

v1.2 작성 초안 (2026-05-24) 은 *5 코어 봇 직원화* 가정으로 작성. 2026-05-27 directive 로 **scope 축소**: 5 봇 → 1 Chief. 이유:

1. **사용자 인지 부담** — 1인 솔로 founder 가 Discord 에 5 봇 분리 인지 / 관리 (5 토큰 발급, 5 invite URL, 5 verification 추적) 는 본 제품 톤(*"24/7 자율 팀"*)과 충돌.
2. **v1.1 의 Chief 가 이미 supervisor** — 4 main + 20 specialist 는 Task tool 로 내부 spawn. Discord 표면에서 *외부 가시 봇* 은 Chief 1개로 충분.
3. **Discord 정책 친화** — verified-app 한도 75 길드 (1 봇 기준), 5 봇이면 봇마다 별도 verification + intent 토글 + 사용자 mention 학습 부담.
4. **회귀 위험 축소** — 5 봇 환경의 mention 라우팅 / echo loop / hop budget 은 multi-bot 일 때만 발생하는 함정. 1 봇 = 0 회귀.

**의도적 폐기 (5-bot 초안의 부분):**
- 별도 봇 × 5 (chief/pm/engineer/designer/marketer) — 보류. 모두 Chief 1개 봇 위에서 Task tool 호출.
- Bot Identity Registry `bots.json` 5-entry 스키마 — 단일 Chief identity 만 user.yaml 에 박제 (구 v0.8.0 single-bot 모델과 동일 인터페이스).
- 9-hop diagnostic — 5-hop 으로 환원 (v1.0.4 P 와 동일).
- Forum Channel topology 강제 — text channel + dynamic thread (gpt-discord-bot 패턴) 로 단순화.

이 결정의 trade-off:
- **잃는 것:** 메신저 표면에서 "팀이 보인다" 효과. → 그러나 v1.1 의 `works-<handle>` 채널 + Chief 가 dispatch 결과를 `[engineer] backend-engineer 가 X 했음` 식으로 narrate 하면 동등 효과.
- **얻는 것:** onboarding 5분 → 1분, 회귀 면적 80% 감소, Discord 100길드 verification skip.

### 0.1 2026-05-28 추가 directive 6건

사용자 directive (2026-05-28) 로 v1.2 scope 에 다음 6 영역 추가/명시:

1. **Chief 이름 부여 (조직 단위)** — Chief 는 조직마다 1개이므로 사용자가 이름 명명 (예: *Hermes*, *Atlas*). 봇 생성 (Discord Developer Portal Bot 생성) 시 *같은 이름을 사용하도록* init/add-org 가 안내. 저장: `user.yaml.chief_name`. 사용처: onboarding embed 제목, narration prefix, doctor 출력. (§4.1, §5.1, §12 #12)
2. **handle 기반 채널명 portability** — 채널 `command-<handle>` / `works-<handle>` 의 `handle` 은 user-level 식별자. 같은 사용자가 Discord 서버 N 개 + 추후 Slack 추가해도 *모든 표면에서 같은 채널 이름* 자동 재사용. v0.8.0 부터 이미 deterministic — 본 plan 은 *명시 + 멀티 메신저 호환성 보증*. (§6.0)
3. **owner-only 게이트 (default ON)** — `message.author.id === user.yaml.messenger_user_id` 인 메시지만 처리. 그 외 사용자가 같은 채널에 메시지 / 멘션 → Chief 는 silently ignore + 첫 1회만 ephemeral 안내 *"이 Chief 는 @owner 의 지시만 받습니다"*. v1.0.2 는 *채널명이 user-id 라 봇이 채널을 못 인식* 한 별개 사유로 author-guard 를 뺐던 결정 — handle 기반 채널명으로 인식 문제가 해소된 이상 v1.2 에서 owner-id 게이트는 *복원 가능 + 바람직*. configurable: `workspace.yaml.messenger.discord.owner_only` (default `true`). (§4.5, §7.1, §12 #13)
4. **TRIAGE kind 분기 — works-handle 과제 허브** — Chief 6+1 의 TRIAGE 가 *kind ∈ {chat, workflow, schedule, goal}* 분류. `chat` 은 `command-<handle>` 채널 내부 평탄 응답. `workflow / schedule / goal` 은 `works-<handle>` 채널에 *task card embed* (제목 / kind / 시작 시각 / 사용자 요청 1줄) 를 post + 그 메시지에 `startThread()` → 이후 모든 sub-agent narration / SYNTHESIZE / DECIDE 가 *그 thread 내부* 에서 진행. command 채널에는 *"📋 작업 등록됨 → <thread link>"* 1줄만 남음. v1.2 초안의 *"command 채널 message 에 thread"* 패턴 폐기. (§6.2, §8.2, §12 #14)
5. **`solosquad add-org` 보강** — 기존 명령 (v1.1.0 신설) 이 *name + provider + remoteUrl + messenger* 만 받고 scaffoldOrg + syncAgents 만 호출. v1.2 가 추가: Chief 이름 prompt, Discord token + application_client_id inline 입력, invite URL 생성 + browser open, 워크플로 기본셋 (problem-definition) 시드, 조직 위계 디렉토리 (`agents/main/chief`, `teams/*`, `memory/open-questions`, `memory/ledger`, `workflows/`) v1.1.0 구조 완전 시드. (§5.5, §12 #15)
6. **워크플로 기본셋 — problem-definition** — 새 조직은 기본으로 *문제 정의 워크플로* 1개 시드. `skills/workflow-maker/assets/workflows/problem-definition/workflow.yaml` 신규 — stages: scqa → 5-whys → mece → tdcc → xyz-hypothesis → one-pager-prd. (§12 #16)

위 6 directive 는 본 plan 의 §1~§15 본문에 inline 반영. 변경 영향 매트릭스 §15.1 changelog.

## 3. 자동 연결 흐름 (3 진입 경로)

### 3.1 Default — OAuth Invite URL (사용자 own bot, 1 click 초대)

**전제:** 사용자가 Developer Portal 에서 Application + Bot 생성 + Token 발급 + MESSAGE_CONTENT intent 활성화 완료 (4 단계 — §3.3 manual 영역, init wizard 가 가이드). *Bot 이름은 Chief 이름 (§4.1) 과 동일하게 권장* — 사용자가 Discord 표면에서 보는 봇 displayName 이 SoloSquad 의 `user.yaml.chief_name` 과 일치하면 *조직 정체성* 이 일관됨 (선택사항이지만 init wizard 가 적극 권장).

**자동화 흐름:**

```
1. solosquad init / add-org 의 "Step 4 — Messenger" 분기에서:
   a) Chief 이름 prompt ("이 조직의 Chief 에게 어떤 이름을 줄까요? 예: Hermes")
      → user.yaml.chief_name 박제
      → "Developer Portal 에서 Bot 생성 시 이 이름을 사용하세요" 안내
   b) 사용자 Discord application client_id 입력 (Developer Portal URL 에서 복사)
   c) 사용자 Bot token 입력 (Developer Portal → Bot → Reset Token → 복사)
2. init 가 client_id + 권장 permissions bitfield (§4) 로 invite URL 합성:
   https://discord.com/oauth2/authorize?client_id=<id>&scope=bot+applications.commands&permissions=<bitfield>
3. URL 출력 + browser 자동 open (실패 시 clipboard 복사 fallback)
4. 사용자가 브라우저에서 "Authorize" click + 자기 길드 선택
5. Discord 가 봇을 길드에 추가 → SoloSquad bot 프로세스 (이미 running) 의 `guildCreate` 이벤트 발화
6. §5 onboarding 자동 시작
```

**원자성:** 사용자 click 1회 (Authorize). 그 외 9 단계는 모두 자동.

### 3.2 Alternative — BYO Bot manual (Developer Portal 자체 가이드)

`solosquad init --byo-bot-manual` (또는 init wizard 분기 명시 선택) 시 invite URL 자동 생성 skip. 사용자가 직접 OAuth URL 생성 → 초대. v1.0.4 까지의 기본 흐름과 동일. 자동화 1단계 줄어듦 — *통제 우선 사용자* 용.

### 3.3 자동화 가능 vs Discord 정책 manual

**Discord 정책상 manual (자동화 불가):**

| 단계 | 이유 |
|---|---|
| Discord Application + Bot 생성 | 사용자 계정 본인 인증 필요. API 우회 0. |
| Token 발급 / Reset | UI 만. Token Manager API 없음. |
| MESSAGE_CONTENT privileged intent 토글 | Developer Portal UI 만. Application API 에 없음. |
| OAuth "Authorize" click + guild 선택 | 사람 액션 필수. *어떤 봇이 어떤 길드에 access 할지* 의 명시적 동의. |

→ init wizard 가 각 단계의 *정확한 URL + 토글 위치 + 복사할 값* 안내 + clipboard helper (예: client_id prompt 직후 invite URL 을 clipboard 에 자동 복사). 사용자 manual 시간 ≤ 3분 목표.

**자동화 가능 (SoloSquad 책임):**

| 영역 | 메커니즘 |
|---|---|
| invite URL 합성 | `solosquad discord invite-url` (또는 init 내부 호출) |
| 봇 client login | discord.js `Client.login(token)` — v0.8.0 부터 동작 |
| guild join 감지 | `guildCreate` listener — v1.0.3 부터 동작 |
| 카테고리 + 채널 auto-create | `ensureChannels` — v0.8.0 부터 동작 (v1.0.4 가 robust 화) |
| guild-org binding 박제 | `syncGuildProductMapping` — v1.0.4 가 load-or-empty 로 완성 |
| Chief 첫 대화 trigger | §5 — `guildCreate` 후 systemChannel 에 onboarding embed + button |
| Slash command bulk register | `Routes.applicationGuildCommands` — guild scope 즉시 반영 |

### 3.4 SoloSquad 공식 봇 운영 mode (v1.x slot, 본 v1.2 미포함)

v1.x 에서 *SoloSquad 가 공식 봇 1개 운영* + 사용자가 자기 길드에 초대만 하는 mode 추가 검토:
- 사용자 manual 0 (토큰 발급 / intent 토글 / OAuth 모두 SoloSquad 가 사전 처리)
- Discord 100길드 도달 시 verification 필요 → SoloSquad 운영 부담
- 사용자 토큰이 SoloSquad 서버에 저장 → 보안/신뢰 부담

v1.2 는 BYO 단일 모드. 공식 봇 mode 는 사용자 base 가 충분히 형성된 후 별도 plan.

## 4. Bot Identity & Token (단일 Chief)

### 4.1 user.yaml 평탄 스키마 (v0.8.0 그대로 + Chief 이름 1 필드)

v0.8.0 의 single-bot user.yaml 평탄 스키마를 *그대로* 사용. v1.2 가 *`chief_name` 1 필드만 신설* — 봇 application client_id 는 기존 `bot_application_id` 필드 (v0.8.0 부터 존재, Discord REST `/users/@me.application_id` 로 자동 채움) 를 재사용 (별도 신설 X):

```yaml
# <org>/.solosquad/users/<handle>.yaml
schema_version: 1
handle: w1n
display_name: "정승원"
messenger: discord
messenger_user_id: "1234567890"          # v0.8.0 사용자 본인 Discord ID
                                          # ← v1.2 §4.5 owner-only 게이트의 single source
chief_name: "Hermes"                      # v1.2 신규 — 조직 단위 Chief 이름.
                                          # init/add-org 에서 prompt. Developer Portal Bot 이름과
                                          # 동일 사용 권장 (Discord 표면에서 일관)
bot_application_id: "9876543210"          # v0.8.0 — Discord application ID. invite URL 합성에도 재사용
bot_user_id: "1111111111"                 # v0.8.0 — 봇이 처음 login 시 client.user.id 로 자동 채움
joined_at: "2026-05-28T14:32:00+09:00"

  command: "command-w1n"                  # v0.8.0 deriveChannelNames(handle) 결과

```

> **v0.8.0 user.yaml 호환:** breaking 0. `chief_name` 만 신규 optional. 향후 multi-bot 도입 시점에만 `bots.chief.*` nested 로 재구조 (v1.x slot).

> **Chief 이름 미설정 시:** migration 으로 1.1.0 → 1.2.0 올라간 사용자는 `chief_name` 필드가 없을 수 있음. doctor 가 missing 감지 → *"`solosquad init` 또는 user.yaml 직접 편집으로 Chief 이름 지정 권장"* hint. fallback: `"Chief"` 문자열 그대로 사용 (동작 무결, 정체성 약함).

### 4.1.1 handle portability (멀티 메신저 / 멀티 서버)

`handle` 은 user-level 식별자 — *어느 messenger / 몇 개의 Discord 서버* 에서도 동일하게 재사용. 예시:
- 사용자 `handle: w1n` 이 Discord 서버 A + B + C 에 봇 초대 → 세 서버 모두 `command-w1n` / `works-w1n` 채널 (각 서버마다 `ensureChannels` 가 같은 이름으로 생성)
- v1.2.x 에서 Slack workspace 추가 → 동일 handle 그대로 → `#command-w1n` / `#works-w1n` Slack 채널 생성
- handle 변경하려면 `user.yaml.handle` 수정 + 봇 재시작 (channel rename 은 별도 migration 필요 — v1.2 scope 외)

이 portability 는 v0.8.0 부터 deterministic 하지만 v1.2 가 *명시 + 보증* — `deriveChannelNames(handle)` 이 모든 adapter 의 single source.

### 4.2 권장 permissions bitfield

`scope=bot applications.commands` + 다음 권한 (overkill 도 underkill 도 회피):

| Permission | bit | 필요 이유 |
|---|---|---|
| View Channels | 1024 | 메시지 수신 가능 채널 인지 |
| Send Messages | 2048 | Chief 응답 송신 |
| Send Messages in Threads | 274877906944 | thread-per-workflow (§6.2) |
| Create Public Threads | 34359738368 | thread 자동 시작 |
| Manage Threads | 17179869184 | auto-archive / rename |
| Read Message History | 65536 | referencedMessage chain (§7.3) |
| Embed Links | 16384 | 결과 보고 embed |
| Attach Files | 32768 | design doc / 이미지 첨부 |
| Use Application Commands | 2147483648 | `/chat` slash (§7.4) |
| Manage Channels | 16 | `#command-<handle>` / `#works-<handle>` 자동 생성 (§5) |

**합계 bitfield**: 합산 값은 publish 직전 calculator 로 재산정 후 invite-url 모듈 상수로 박제 (`DEFAULT_PERMISSIONS_BITFIELD`).

**제외 (Discord 심사 트리거 위험):** `Administrator`, `Manage Guild`, `Manage Roles`, `Kick/Ban`, `Mention Everyone`.

### 4.3 토큰 1개

- 단일 `DISCORD_TOKEN` env var — v1.0.x 와 동일 위치 (`<workspace>/.solosquad/.env`).
- multi-bot env var (`DISCORD_TOKEN_CHIEF` / `_PM` / `_ENG` / ...) 도입 보류 (v1.2 scope 외).
- 토큰 검증: `solosquad doctor --discord` 가 Discord REST `/users/@me` 호출로 token 유효성 1초 내 확인 + bot_user_id mismatch 시 actionable hint.

### 4.4 Single source of truth = Chief 봇 자체

본 v1.2 는 별도 `bots.json` registry 신설 *안 함*. v1.0.4 의 `<org>/discord/config.yaml` (load-or-empty 자동 생성) 이 *Chief 봇 ↔ guild ↔ org* 매핑 single source. v1.1.0 의 chief-runner 는 user.yaml (flat) 만 보면 됨.

### 4.5 Owner-only 게이트 (v1.0.2 부분 reversal)

**기본 동작 (default ON):** Chief 가 메시지를 처리하는 조건 = `message.author.id === user.yaml.messenger_user_id`. 그 외 사용자가 같은 채널에 메시지 / 멘션 → silently ignore + 첫 1회 ephemeral 안내 *"이 Chief 는 @owner 의 지시만 받습니다. owner 가 채널 ACL 을 조정하지 않는 한 응답하지 않습니다."*.

**v1.0.2 결정과의 정합:**
- v1.0.2 (`docs/prd/v1.0.2-discord-author-guard-decoupling.md`) 는 author-guard 를 *제거* 했음. 명목적 사유: *"Discord channel ACL 이 canonical permission boundary"*.
- **실제 사유 (2026-05-28 사용자 directive 로 명확화):** 당시 채널명이 *user-id 기반* 이어서 봇이 채널을 인식하지 못했음. author check 만 단독으로 두면 채널 인식 실패와 권한 체크 실패가 구분 안 됨 → guard 제거 + 채널명 schema 를 *handle 기반* 으로 재설계 했던 흐름.
- handle 기반 채널명 (`command-<handle>` / `works-<handle>`, v0.8.0~v1.0.4 까지 안정화) 은 인식 문제 해소. *인식 = 채널명 매칭* 이 우선 boundary, *권한 = author-id* 가 그 다음 boundary 로 정리되면서 v1.2 가 author-guard 를 *재도입 가능* + *바람직* 으로 결론.

**구현 boundary:**
1. messageCreate listener 가 먼저 `parseChannelName(channelName)` 으로 *이 채널이 내 command/works 인가* 확인 (이미 v1.0.4 동작)
2. 채널 매칭 통과 시 *그 다음에* `message.author.id === user.yaml.messenger_user_id` 체크
3. 불일치 → `interaction.reply({ ephemeral: true })` 1회 (LRU `<guild-id>:<author-id>` 1시간 dedupe) → return

**configurable:**
- `workspace.yaml.messenger.discord.owner_only` (default `true`)
- `false` 로 설정 시 v1.0.2 mode 복귀 — Discord channel ACL 위임. 콜라보레이터에게 명시적으로 채널 share 한 사용자 케이스용. doctor 가 *"owner-only 비활성. 채널 ACL 신뢰 필요"* warning 표시.

**Slack 등가 (v1.2.x):** `event.user === user.yaml.messenger_user_id` 동일 패턴. Slack 의 ephemeral 메시지는 `chat.postEphemeral` API.

### 5.2 Idempotency

### 5.3 Button Interaction Handler

```typescript
// src/messenger/discord-onboarding.ts (신규)
client.on(Events.InteractionCreate, async (interaction) => {
  if (!interaction.isButton()) return;
  if (!interaction.customId.startsWith("chief:onboard:")) return;

  const action = interaction.customId.split(":")[2]; // "auto" | "manual"
  if (action === "auto") {
    await interaction.deferReply({ ephemeral: true });
    await ensureChannels(interaction.guild!);
    await interaction.editReply("✓ 채널 생성 완료. #command-<handle> 로 가세요.");
  } else if (action === "manual") {
    // 사용자가 채널 mention 으로 직접 지정하는 다단계 prompt
    await interaction.reply({ content: "지정할 채널을 mention 으로 알려주세요. 예: <#1234567890>", ephemeral: true });
  }
});
```

- Chief embed 가 송신 X (systemChannel write 권한도 없을 수 있음)
- 봇 DM 으로 사용자에게 *"권한 부족. 봇 재초대하거나 `Manage Channels` 권한 부여"* + 새 invite URL 첨부

## 6. Channel + Thread Topology

### 6.0 handle 기반 채널명 (멀티 메신저 / 멀티 서버 portable)

- `command-<handle>` / `works-<handle>` 의 `handle` 은 `user.yaml.handle` (v0.8.0). user-level 식별자.
- 같은 사용자가 Discord 서버 N + Slack workspace M 에 봇 연결 → 모든 표면이 *같은 이름의 채널 페어* 자동 생성 (각 adapter 가 `deriveChannelNames(handle)` 호출).
- 봇은 *handle 기반 채널명 매칭* 후 *owner-id author check* (§4.5) 2단 boundary. v1.0.2 때의 *user-id 채널명 → 봇 인식 실패* 결함은 해소됨.

### 6.1 v1.0.x 페어 — v1.2 에서 역할 격상

- `#command-<handle>` — 사용자 input 채널. Chief 만 listen + owner-only. (v0.8.0)
  - v1.2: TRIAGE 결과가 *chat* 이면 여기서 평탄 응답. 작업 단위면 *"📋 작업 등록됨 → <thread link>"* 1줄만.
- `#works-<handle>` — v0.8 의 *비-conversational output* 정의에서 **과제 허브** 로 격상.
  - v1.2: 작업 단위 (workflow / schedule / goal) 의 task card embed 가 여기에 post. 각 카드에 thread 시작.
  - 기존 routine 결과 / broadcast 도 동일 채널 — 별도 메시지 (thread 없이) 로 송신해서 카드와 구분.

### 6.2 v1.2 핵심 — TRIAGE kind 분기 (chat / workflow / schedule / goal)

Chief 6+1 stage 의 TRIAGE 결과가 `kind` 필드 ∈ {chat, workflow, schedule, goal} 출력 (chief-stage-events.jsonl 의 stage_completed event 에 박제).

**`chat` (단순 논의, 가벼운 질문, 메모리 lookup 등):**
- `command-<handle>` 채널 내부에서 *평탄 응답*. thread 생성 X.
- 응답 길이 > 2000 자 (Discord limit) 시 chunk 분할.
- referencedMessage chain (§7.3) 으로 context 유지 — 최대 chain 깊이 8.

**`workflow` / `schedule` / `goal` (작업 단위):**
- Chief 가 `works-<handle>` 채널에 *task card embed* 송신:
  ```
  Title: 📋 {kind | uppercase}: <짧은 제목>
  Description:
    요청: <사용자 1줄 요약>
    workflow_id: wf-YYYYMMDD-<slug>
    시작: <KST time>
  Color: blue (workflow) / green (schedule) / orange (goal)
  Footer: {chief.name} · Chief
  ```
- 그 메시지에 `message.startThread({ name: '<kind>-YYYYMMDD-<slug>', autoArchiveDuration: 10080 })`
- thread URL + thread_id 박제 → `<org>/workflows/wf-YYYY-MM-DD-<slug>/discord-thread.txt`
- 이후 모든 dispatch / sub-agent narration / SYNTHESIZE / DECIDE 메시지가 *그 thread 내부*
- DECIDE 최종 응답 시 *원래 command 채널* 에 *"✓ 완료 → <thread link>: <decision 1줄>"* 1줄 announce

**관련 정보 흐름:**

```
[#command-w1n]                          [#works-w1n]
  사용자: "이번 주 PMF 가설 검증 루틴 만들자"
  Chief: 📋 작업 등록됨 → <link>                       ← embed 1줄

                                          ┌─ [task card embed] WORKFLOW: PMF 가설 검증 루틴
                                          │   workflow_id: wf-20260528-pmf-validation
                                          │   요청: 이번 주 PMF 가설 검증 루틴 만들자
                                          │   시작: 2026-05-28 14:32 KST
                                          │
                                          └─ [thread: workflow-20260528-pmf-validation]
                                              [Chief] 작업 분해 중...
                                              [Chief] dispatch → pm, designer (병렬 2)
                                              [pm] discovery-synthesis (archive 8건)
                                              [pm] problem-definition P3 MECE
                                              [designer] ux-designer wireframe 초안
                                              [Chief] 통합 완료. design doc <link>

  Chief: ✓ 완료 → <thread link>: pmf-validation     ← decision announce
         workflow 첫 cycle 설계 박제
```

### 6.3 Thread ↔ Workflow 1:1 매핑

- `<org>/workflows/wf-YYYY-MM-DD-<slug>/discord-thread.txt` — thread URL + thread_id + works 채널 message_id 박제
- chief-runner 가 워크플로우 reconcile 시 (`_status.yaml` 에서 stage 부활) 같은 thread 에 *"thread 재개됨"* 알림
- thread archive (7일 후 자동) 시에도 박제 유지 — 재activate 시 같은 thread 사용

### 6.4 Forum Channel 사용 안 함

v1.2 초안의 Forum Channel topology 폐기. 이유: solo founder 1인 환경에서 *모든 메시지 = thread* 강제는 메타데이터 overhead + 모바일 UX 저하. text channel + 동적 thread 가 더 가벼움.

## 7. Chief 대화 UX

### 7.1 사용자 → Chief → 응답 흐름

```
[사용자 메시지 in #command-<handle>]
  ↓
Discord adapter messageCreate
  ├─ [1] parseChannelName(channelName) — command/works/그 외 (v1.0.4)
  ├─ [2] ownHandle match (v0.8.0)
  ├─ [3] §4.5 owner-only gate: author.id === messenger_user_id (v1.2 신규)
  │     ↳ 불일치 시 ephemeral 1회 안내 + return
  └─ [4] getProductByGuild → product
  ↓
chiefRunner.handleUserMessage({ userText, orgSlug, orgCwd, userId })
  ↓
Chief 6+1 stage (v1.1 §5.2):
  TRIAGE (kind ∈ {chat,workflow,schedule,goal}) → DECOMPOSE → DISPATCH
    → AWAIT → SYNTHESIZE → DECIDE → RETROSPECT
  ↓
chiefRunner returns { text, kind, workflow_id?, costUsd, durationMs, ... }
  ↓
Discord adapter:
  - kind === "chat" → message.channel.send(text)  (= command 채널 평탄)
  - kind ∈ {workflow,schedule,goal} → §6.2 works task card + thread
    + command 채널엔 "📋 작업 등록됨 → <thread link>" 1줄
```

v1.1.0 이 [1]+[2]+[4] + chiefRunner 본체 구현 완료. v1.2 가 [3] owner-only + TRIAGE kind 분기 + thread 라우팅 + onboarding embed + button interaction 추가.

### 7.2 Typing indicator + 응답 streaming

- Chief 가 응답 생성 중: `message.channel.sendTyping()` (이미 v0.8.0 부터 동작)
- 응답 길이 > 2000 자 (Discord limit): chunk 분할 + 첫 chunk 에 thread 시작 권유

### 7.3 Message Context (referencedMessage chain — llmcord 패턴)

- 사용자가 봇 메시지에 reply (Discord 의 message.reference) 하면 Chief 는 referencedMessage + 그 references chain 을 컨텍스트에 inject
- 최대 chain 깊이 = 8 (워크플로우 thread 안에서는 thread 전체)
- chain 중 봇 자신 메시지와 사용자 메시지 모두 chronological order 로 inject
- LRU message cache (1000 entries / 길드) 로 Discord REST `GET /channels/{id}/messages/{id}` 호출 절약

### 7.4 `/chat` slash command (low-intent backup)

MESSAGE_CONTENT privileged intent 가 거부된 경우 (Discord verification 우려, 또는 100길드 초과) 대안:

```
/chat <message>
```

- Discord 가 slash command interaction 으로 메시지 본문 전달 → intent 무관
- Chief 가 slash 응답을 thread 로 자동 분기 (gpt-discord-bot 패턴)
- 본 v1.2.0 에는 register 만, default 라우팅은 여전히 message + mention. 100길드 도달 시 slash-first 로 전환.

## 8. 과제 진행 / 보고 (Chief 6+1 stage → Discord projection)

### 8.1 Stage event → Discord 메시지 변환

`<org>/memory/chief-stage-events.jsonl` (v1.1 §5.2) 의 각 event 가 Discord 메시지로 자동 projection. 송신 target 은 *TRIAGE kind* 에 따라 다름:
- `kind === chat` → 모든 narration 생략. DECIDE 응답만 command 채널.
- `kind ∈ {workflow,schedule,goal}` → §6.2 thread 내부로 narration.

| Chief stage | Discord narration (Korean, thread 내부) |
|---|---|
| TRIAGE | `kind` 가 *작업 단위* 면 task card embed 가 works 채널에 post 되는 시점 = TRIAGE 직후. narration 자체는 thread 미발화. |
| DECOMPOSE | "🗂 작업 분해 중... PM 호출 결정." |
| DISPATCH | "📤 dispatch: pm, engineer (병렬 2)" |
| AWAIT (open_questions[]) | "❓ 추가 정보 필요:\n  • [Q1] 타깃 사용자 어느 segment?\n  • [Q2] 배포 timing 우선?" (thread 내 batch, 사용자 응답 대기) |
| SYNTHESIZE | (생략 — 결과 메시지 자체로 표현) |
| DECIDE | thread 마지막 메시지 + command 채널에 *"✓ 완료 → <thread link>: <1줄 decision>"* 1줄 |
| RETROSPECT | thread 내 *"📝 회고 박제 → <ledger entry>"* 1줄. weekly-retro broadcast 는 별도 (works 채널, thread 외부) |

### 8.2 Sub-agent 활동 narration (works thread 내부)

PM / Engineer / Designer / Marketer 가 spawn 되었을 때 사용자가 *지금 누가 뭘 하고 있는지* 확인 가능. 송신 위치 = §6.2 task card 의 thread 내부:

```
[#works-w1n]
  📋 WORKFLOW: 새 랜딩 페이지
  요청: 새 랜딩 페이지 만들자
  workflow_id: wf-20260528-landing-page
  시작: 2026-05-28 14:50 KST
  ──────────────
  [thread: workflow-20260528-landing-page]
    [Chief] 🗂 작업 분해 중
    [Chief] 📤 dispatch — pm + designer (병렬 2)
      ↳ [pm] discovery-synthesis (archive 8건)
      ↳ [pm] problem-definition P3 MECE
      ↳ [designer] ux-designer wireframe 초안
    [Chief] ✅ 통합 완료. design doc 작성됨 → <link>
```

각 sub-step 메시지는 `chief-stage-events.jsonl` 의 `dispatched` / `skills_used` 필드를 message text 로 format. 동일 thread 내 5초 debounce + 동일 sub-agent 의 연속 skill 메시지는 1개로 batch.

### 8.3 open_questions[] handling (works thread 내부)

PM 이 `<org>/memory/open-questions/<task-id>.json` 에 blocking 질문을 남기면:
- Chief 가 해당 *workflow thread (works 채널)* 에 batch 메시지로 출력 (v1.1 §6.3 — 사용자에게 한 번에)
- 사용자가 같은 thread 에서 reply → Chief 가 referencedMessage chain 으로 answer 매칭 → PM 재spawn

`kind === chat` 인 turn 에서 open_questions 발생 시 (드물 — chat 은 보통 즉답) command 채널 자체에서 batch.

### 8.4 status / progress 자가 보고

사용자가 *"지금 뭐 하고 있어?"* 식 메시지 → Chief 가 `chief-stage-events.jsonl` 최근 30분 읽어서:

```
현재 진행:
  • turn-<id> AWAIT (PM 의 open_questions 2건 사용자 응답 대기)
  • turn-<other> DISPATCH 직후 (engineer 2 spawn 진행 중, 평균 3분 예상)

최근 완료:
  • 12분 전 RETROSPECT — 어제 캠페인 회고 (skill-refinement 1건 제안됨)
```

이 self-report 는 `solosquad status` CLI 의 Chief-version (skill `triage` 의 일부로 구현).

## 9. Safety Net (echo guard, token budget, rate limit)

### 9.1 Echo guard (단일 봇 환경)

5-bot 환경의 봇간 mention loop 가 사라졌으므로 echo guard 도 단순화:
- 봇은 `message.author.id === client.user.id` 메시지 무조건 skip (v0.8.0 부터 동작)
- 사용자가 같은 메시지를 1초 내 ≥3회 반복 송신 시 Chief 가 *"같은 요청을 받았어요. 한 번만 처리할게요."* + 첫 메시지만 처리

### 9.2 Token budget per thread

- workflow thread 의 누적 응답 token > `workspace.yaml.discord.thread_token_budget` (default: 80,000 ≈ 한 Claude 컨텍스트) 초과 시 Chief 가 *"thread 컨텍스트가 가득 찼어요. 새 thread 로 이어갈까요?"* prompt
- 사용자 동의 시 새 thread 시작 + 직전 thread 의 design doc / decision 요약 link

### 9.3 Discord rate limit

- discord.js v15 가 자동 처리 (`rest.setRetryLimit`, 429 백오프)
- 추가 안전망: Chief 의 sub-agent narration (§8.2) 메시지를 5초 debounce + 동일 thread 내 연속 메시지 batch ≤5

### 9.4 작성 권한 회수 시나리오

봇이 길드에서 채널 권한 회수당함 → discord.js 가 `DiscordAPIError: Missing Permissions` throw → Chief 가 캐치해서 DM 으로 사용자에게 알림 + invite URL 재발급

## 10. 5-hop Binding Diagnostic (v1.0.4 P + 단일봇 정합)

`solosquad doctor --discord` 가 다음 5 hop 모두 통과해야 connect ready:

| Hop | 검증 | 실패 시 hint |
|---|---|---|
| 1 | DISCORD_TOKEN env 존재 + format (`<id>.<timestamp>.<hmac>`) | `Set DISCORD_TOKEN in <workspace>/.solosquad/.env.` |
| 2 | Token 으로 Discord REST `/users/@me` 호출 → 봇 user ID 받음 | `Token 무효. Developer Portal → Bot → Reset Token 후 재입력` |
| 3 | bot_user_id 가 user.yaml.bot_user_id 와 일치 | `Bot ID 불일치. Reset Token 후 doctor 재실행하면 자동 갱신됨` |
| 4 | 봇이 ≥1 guild 에 속해 있음 (Gateway READY 후 guilds.cache) | `OAuth Invite URL 사용해 길드에 초대하세요: ${invite_url}` |
| 5 | `<org>/discord/config.yaml.guild_id` 가 길드 중 하나와 일치 + `loadProducts()` 에 해당 org 포함 + `command-<handle>` 채널 존재 | `solosquad init` 또는 `solosquad add-org` 재실행. config.yaml load-or-empty 자동 채움. |

각 hop 실패는 attributable + actionable. silent fail 0.
