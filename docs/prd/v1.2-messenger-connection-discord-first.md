## 3. Channel Topology — `#command` + `#works` (Forum)

### 3.1 `#command-<handle>` — 사용자 ↔ PM only

**구조:**
- 채널 가시성: org guild 의 카테고리 `solosquad` 아래 (v1.0.3 §13.6.16 정합)
- 멤버: 사용자 1명 + PM 봇 1명. *다른 specialist 봇은 명시적으로 제외*
- 메시지 라우팅:
  - 사용자 → PM 봇 (전체 listen, 4-channel router 적용)
  - PM 봇 → 사용자 (요약 + cite + 필요시 forum link)
- DM passthrough: PM 봇은 Discord DM 도 listen → 응답은 *`#command-<handle>` 채널 cross-post* (audit log)

**Application-level author-guard:**
- v1.0.2 single-bot author-guard 정신 → multi-user 일반화
- 메시지 sender 가 channel.allowed_user_ids allowlist 에 있는지 PM 봇이 application-level check
- 채널 ACL (Discord role / Slack workspace) 만 의존 안 함 — 다른 사용자 초대 시 PM 봇이 무시 + audit log

### 3.2 `#works-<handle>` — Discord Forum Channel

**구조:**
- 채널 타입 = **Forum Channel** (Discord 표준)
- 카테고리: `solosquad` (v1.0.3 동일)
- 멤버: 사용자 1명 + 모든 specialist 봇 4명 (eng / design / growth / strategy)
- 각 *과제 / workflow / goal cycle = forum post 1개* (영구 + 24h auto-archive 면역 + 검색 + 태그)

**Forum tags (PM routing 의 1차 hint):**
- `bug` `feat` `refactor` `docs` `design` `review` `urgent` `experiment` `solved`
- PM 봇이 post 생성 시 task 의 1차 tag 자동 부여 (사용자 의도 → SKILL frontmatter `domain_tags` → forum tag 매핑 — v1.1 §5.2 정합)

**v1.1 의 workflow / goal cycle 정합:**
- workflow 1개 = forum post 1개 — `<org>/workflows/<wf-id>/_status.yaml` 의 `forum_post_url` 필드 신설
- goal cycle 1개 = forum post 1개 — `<org>/goals/<id>/_events.jsonl` 의 `forum_post_url` 박제

### 3.3 Forum Channel 의 *영구성* 이유 (Best Practice U)

`docs/ideation/2026-05-23-multi-agent-v1.1-synthesis.html` §10.U 박제:

| 측면 | Discord standard Thread | Discord Forum Channel |
|---|---|---|
| Auto-archive | 기본 24h | 면역 |
| 검색 | 채널 내 message search | 채널 내 post search + 태그 filter |
| 카탈로그화 | 시간순 | 태그·status·activity 정렬 가능 |
| 사용자 visible | thread list 깊게 들어가야 | 채널 상위 layout |

**과제 carry-over 보장** — 1주일 후 사용자가 "지난주 SKILL refactor 토론" 같은 검색 가능.

### 4.3 Bot startup flow

```
1. solosquad bot <name>          # 또는 solosquad bots (5 봇 병렬)
2. process 진입점 (bin/solosquad.ts) — 신규 인자 <bot-name> 인식
3. resolveBotIdentity(workspace, botName):
   3.1 user yaml `bots.<bot-name>` 읽기
   3.2 bot_token_env 의 환경변수 token 로드 (token precedence — Best Practice O)
   3.3 messenger gateway 연결 (Discord login / Slack socket)
   3.4 connected bot_user_id 가 yaml 의 bot_user_id 와 일치 검증 (mismatch 면 hard fail + actionable hint — Best Practice P)
4. loadBotIdentityRegistry() → 본 org 의 다른 봇 4명 ID 메모리 캐시
5. loadSkill(yaml.skill_dir) — *자기 SKILL 만* (Best Practice T — Specialist SKILL isolation)
   - pm: orchestrator/SKILL.md
   - eng: engineering/{architect,fde,...}/SKILL.md 전체
   - design/growth/strategy: 각 team 의 SKILL 셋
6. subscribeChannels(yaml.channels.command, yaml.channels.works):
   - command: PM 만 listen
   - works: 모든 봇 listen, 단 `app_mention` 이벤트만 react (mention-only intent)
7. 헬스 ping 시작 — `<org>/memory/bot-health/<bot-name>.jsonl`
```

## 5. Mention Routing & DM Passthrough (Best Practice S)

### 5.1 Mention-only intent (specialist 봇)

**규칙:**
- 모든 specialist 봇 (eng/design/growth/strategy) 은 *자기 botUserId 가 mention 된 메시지에만 react*
- Discord `app_mention` privileged intent / Slack `app_mention` event 만 subscribe
- 다른 봇/사용자 mention 메시지에는 *이벤트조차 받지 않음* (Discord) 또는 *받아도 무시* (Slack)

**예외:**
- PM 봇은 `#command-<handle>` 의 *모든* 메시지를 listen (사용자가 PM 멘션 없이 자연 입력 가능)
- PM 봇은 DM / `message.im` 도 listen (§5.3 cross-post)

### 5.2 구현 — discord-adapter.ts

```typescript
// src/messenger/discord-adapter.ts (v1.2 신규 라우팅)
client.on('messageCreate', (msg) => {
  const myBotId = this.identity.bot_user_id;
  // 1. self-message ignore (echo loop 가드 1)
  if (msg.author.id === myBotId) return;
  // 2. PM 만 #command 전체 listen
  if (this.identity.role === 'supervisor' && msg.channel.id === this.commandChannelId) {
    return this.routeToCommand(msg);
  }
  // 3. specialist 는 mention-only
  if (this.identity.role === 'specialist' && msg.mentions.users.has(myBotId)) {
    return this.routeToWorks(msg);
  }
  // 4. else: ignore (echo loop 방지)
});
```

### 5.3 DM passthrough (PM 만)

- PM 봇이 Discord DM 받음 → 사용자 식별 → `#command-<handle>` 채널에 cross-post + 동일 thread 에서 응답
- 모바일 UX: 사용자가 *어디서든* PM 호출 가능, audit log 는 항상 채널에 유지
- 근거: Salesforce Slackbot mobile UX 패턴

### 5.4 Routing — PM Triage stage (v1.1 §5.2 정합)

PM 의 Task dispatch 결정:
1. 1차 deterministic — SKILL `domain_tags` + `routing_keywords` 매칭 (v1.1 §5.2)
2. 2차 LLM fallback — 후보 0/2+ 시

**v1.2 의 메신저 측 기여:** mention parser 가 사용자 메시지의 `@<team-bot>` 명시적 mention 처리 (예: 사용자가 `@eng 이거 처리해`) → routing 의 0차 (명시 우선)

### 5.5 PM 단일 voice + cite

- 사용자가 *보는 봇 = PM 만* (`#command` 안)
- PM 이 specialist 회신을 *직접 paste 안 함* → 요약 + cite:
  > "growth 가 가설 3개 제안: (1)... (2)... (3)... — [forum post link]"
- 사용자 디버깅 가능 (link 클릭 → 원본 specialist 메시지)

## 6. Safety Net — Echo Loop / Hop / Token Budget

### 6.1 Echo loop 3중 가드 (Best Practice S 확장)

**위험:** 봇 A 가 봇 B 멘션 → 봇 B 응답에 봇 A 멘션 포함 → 무한 (AnythingLLM/llmcord/Botpress 사례)

**3중 방어:**

| # | 가드 | 작동 |
|---|---|---|
| 1 | Self-author guard | 봇은 자기 botUserId 의 메시지에 절대 react 안 함 (§5.2 step 1) |
| 2 | Bot-author rate-limit | 다른 봇의 멘션에는 *forum post 별 rate limit* (10 mentions / 10분) |
| 3 | Content-hash dedup | 동일 forum post 안에서 `(sender_bot, sha256(content)[:8])` 가 30초 내 재발생 시 무시 (idempotency) |

### 6.2 Hop counter (Best Practice V — Composable Termination Algebra)

- 모든 forum post 시작 시 `hop_count = 0`
- PM → specialist 멘션 = `hop_count++`
- specialist → PM 회신 = `hop_count++`
- **max `hop_count` = 6** (PM → spec → PM → spec → PM → close)
- 초과 시 PM 봇이 *"hop budget exhausted — synthesizing and closing thread"* 메시지 + 사용자 cross-post 후 강제 종료

**근거:** Anthropic engineering "circuit breaker" + LangGraph `recursion_limit` 기본 ~20 의 *축소판* (channel 환경은 latency × LLM cost × token 더 비쌈)

**구현 위치:** `<org>/memory/handoff-trace.jsonl` (§9) 의 hop record append-only.

### 6.3 Thread token budget (Best Practice R)

- 각 forum post 누적 토큰 ceiling:
  - prompt + completion 합 **200,000 tokens**
  - 메시지 수 ceiling **50개**
- 누적은 `handoff-trace.jsonl` 합산
- 초과 시 hop counter 동일하게 PM 강제 종료 + 사용자 alert

**근거:** Anthropic 자체 측정 — multi-agent = single chat × **15배 토큰**. "Cut the Crap" 논문 2-11.8배. → SoloSquad 가 본 사용자 cost 폭증 막는 *유일한 방어선*.

### 6.4 Composable Termination Algebra (Best Practice V)

forum post 종료 조건:
```
TERMINATE ⟺ (hop_count ≥ 6)
         OR (token_budget ≥ 200_000)
         OR (message_count ≥ 50)
         OR (user_says "stop" | "취소" | "중단")
         OR (PM emits "DONE" | "RESOLVED")
         OR (forum tag changes to "solved")
```

→ AutoGen v0.4 의 `MaxMessage AND TokenUsage AND/OR` 합의 정합.

const REQUIRED_TAGS = ['bug', 'feat', 'refactor', 'docs', 'design', 'review', 'urgent', 'experiment', 'solved'];

async function ensureForumTags(forumChannel: ForumChannel) {

## 8. 9-hop Binding Diagnostic (Best Practice P 전체 적용)

### 8.1 v1.0.4 의 5-hop → v1.2 의 9-hop 일반화

v1.0.4 commit `32c0dde` 의 `diagnoseProductByGuildFailure` 가 *single-bot* 환경의 5-hop binding chain 진단. v1.2 의 *multi-bot* 환경은 4 hop 추가:

| Hop | 진단 항목 | 메시지 |
|---|---|---|
| 1 | Bot Token loaded | "bot.<name> token not in env (key: <bot_token_env>)" |
| 2 | Bot Identity verified | "bot.<name> connected bot_user_id <id> ≠ user.yaml.bots.<name>.bot_user_id <expected>" |
| 3 | ownOrgSlug resolved | (v1.0.4 hop 3) "bot has no resolved org. user.yaml matching failed" |
| 4a | config.yaml present | (v1.0.4 hop 4a) "config.yaml missing at <path>" |
| 4b | guild_id stamped | (v1.0.4 hop 4b) "no guild_id field in config.yaml" |
| 4c | guild_id matches | (v1.0.4 hop 4c) "config.yaml.guild_id=<a> but message came from guild <b>" |
| 5 | loadProducts includes org | (v1.0.4 hop 5) "loadProducts() does not include org=<slug>" |
| **6** | Bot Identity Registry loaded | "bots.json registry not loaded for org=<slug>" |
| **7** | Peer bot lookup | "bot.<peer-name> not found in bots.json. mention failed" |
| **8** | Forum channel exists | "#works-<handle> not a Forum channel (Discord) or thread-only not enforced (Slack)" |
| **9** | Mention permission | "bot lacks Message Content Intent (Discord) or `app_mention` scope (Slack)" |

### 8.2 구현 — `diagnoseBindingFailure` helper

v1.0.4 의 `diagnoseProductByGuildFailure` 를 *generic* 화:

```typescript
// src/messenger/discord-adapter.ts
function diagnoseBindingFailure(ctx: BindingContext): { hop: number; message: string; hint: string } {
  // hop 1
  if (!ctx.bot_token) return { hop: 1, message: `bot.${ctx.bot_name} token not in env (key: ${ctx.bot_token_env})`, hint: `Set DISCORD_TOKEN_${ctx.bot_name.toUpperCase()} env var. See solosquad doctor --bots` };
  // hop 2
  if (ctx.connected_bot_user_id !== ctx.expected_bot_user_id) return { hop: 2, message: `bot.${ctx.bot_name} connected bot_user_id ${ctx.connected_bot_user_id} ≠ expected ${ctx.expected_bot_user_id}`, hint: `Update user.yaml or re-create bot. See solosquad doctor --bots` };
  // hop 3~5 (v1.0.4 existing)
  // ...
  // hop 6
  if (!ctx.bots_registry_loaded) return { hop: 6, message: `bots.json registry not loaded for org=${ctx.org_slug}`, hint: `Run solosquad bots to regenerate registry` };
  // hop 7
  if (!ctx.peer_bot_found) return { hop: 7, message: `bot.${ctx.peer_bot_name} not found in bots.json. mention failed`, hint: `Add peer bot to user.yaml.bots and run solosquad bots` };
  // hop 8
  if (!ctx.forum_channel) return { hop: 8, message: `#works-${ctx.handle} not a Forum channel`, hint: `Re-run solosquad init or create Forum manually` };
  // hop 9
  if (!ctx.mention_intent) return { hop: 9, message: `bot lacks Message Content Intent`, hint: `Enable in Discord Developer Portal: https://discord.com/developers/applications` };
  // success
  return { hop: 0, message: 'binding success', hint: '' };
}
```

### 8.3 actionable hint — 모든 hop

Best Practice P 전체 적용 — 모든 hop 에 *어디부터 디버깅* 명시. `solosquad doctor --bots` 가 본 helper 호출.
