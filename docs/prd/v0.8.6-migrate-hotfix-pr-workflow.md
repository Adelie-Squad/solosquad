## 1. 핵심 회귀 버그: migrate.ts hardcoded version

### 1.1 증상

```
$ solosquad --version
0.8.5

$ solosquad doctor
  △ CLI v0.8.5 > workspace v0.4.0 — Run: solosquad migrate --apply

$ solosquad migrate --apply
✓ Workspace is already at 0.4.0. Nothing to migrate.

$ solosquad migrate --dry-run
✓ Workspace is already at 0.4.0. Nothing to migrate.
```

doctor가 *migrate 필요* 안내하는데 migrate는 *no-op*. 사용자는 워크스페이스 갱신 불가능. workaround로 `--to 0.8.6 --apply` 명시해야만 동작 (대다수 사용자가 모름).

### 1.2 원인

`src/cli/migrate.ts:8`:
```typescript
const CLI_VERSION_TARGET = "0.4.0";
```

`migrateCommand`가:
```typescript
const target = opts.to ?? CLI_VERSION_TARGET;   // → "0.4.0"
const result = await runMigration({ workspace, targetVersion: target, ... });
```

runner.ts:64-74:
```typescript
if (source === opts.targetVersion) {
  console.log("Workspace is already at ... Nothing to migrate.");
  return { success: true, chain: [], ... };
}
```

- 워크스페이스 v0.4.0 + target "0.4.0" → source === target → no-op
- 새 워크스페이스 v0.8.5 (v0.8.5 init이 박은 값) + target "0.4.0" → resolveChain이 *역방향* 마이그레이션 시도 → 에러

즉 **v0.4.0 이후 모든 minor/patch에서 migrate가 결정적으로 깨진 상태**였음. doctor는 mismatch 잘 감지(`recommendForVersionMismatch` 로직), 사용자가 마지못해 안내 따라가도 결과 *Nothing to migrate*. 회귀 1년 잔존.

### 1.3 수정

**v0.8.5와 동일 패턴 적용** — `src/util/version.ts`의 `SOLOSQUAD_VERSION` import.

```typescript
import { SOLOSQUAD_VERSION } from "../util/version.js";
const CLI_VERSION_TARGET = SOLOSQUAD_VERSION;
```

`SOLOSQUAD_VERSION`은 `package.json.version` 동적 참조 (v0.8.5에서 dev/prod 경로 모두 검증된 헬퍼). 이후 `npm publish` 시 자동 정합.

### 1.4 backward-compat

- 기존 v0.4.0~v0.8.5 워크스페이스: 이미 깨진 상태였으므로 *수정 후 정상 동작* — 즉시 chain 따라 v0.8.6까지 올라옴
- 새 init (v0.8.6 wizard): `SOLOSQUAD_VERSION = "0.8.6"` 그대로 박힘, migrate 즉시 no-op (의도된 상태)
- 회귀 risk: `SOLOSQUAD_VERSION`을 import하는 곳이 늘었지만 함수 시그니처 변화 없음

### 1.5 검증

- typecheck 통과 ✓
- 566/566 tests 그린 유지
- 실측: v0.4.0 워크스페이스에 v0.8.6 CLI로 `solosquad migrate --dry-run` → chain 9개 표시 (`0.4.0→0.5.0→...→0.8.6`)
- v0.8.5 워크스페이스에서 `solosquad migrate --apply` → version stamp만 0.8.6로 bump (no schema change)

### 1.6 회귀 방지 — *모든 stale 버전 상수* 일제 점검

grep 결과 (`grep -rn '"0\.[0-9]\.[0-9]"' src/cli/`):
- `src/cli/init.ts` ✓ (v0.8.5에서 fix)

향후 같은 함정 방지:

## 2. master-guide §10.4 + §10.5 박제

### 2.1 §10.4 Uninstall · 재설치 · 마이그레이션 회피

**문제 인식**: npm v7+의 `preuninstall` 훅이 글로벌 패키지에서 호출되지 않는 한계(`npm/cli#3042`) 때문에, 사용자가 *순서를 잘못 지키면* farewell archive 안 만들어진 채 `npm uninstall -g solosquad`가 바이너리만 제거 → REVOKE-CHECKLIST 못 받음 + .env 토큰 마스킹 archive 안 생김.

**박제 내용** (KO + EN 양쪽):
- 안전한 uninstall 6단계 (① 봇·스케줄러 정지 → ② dry-run → ③ uninstall 실행 → ④ archive zip 보관 → ⑤ REVOKE-CHECKLIST 외부 자원 정리 → ⑥ npm 제거)
- uninstall + reinstall로 **마이그레이션 chain 우회** 흐름 (큰 chain 건너뛰는 합법 우회로)
- 새 init 후 doctor 경고 7종 분류 표 (PowerShell / gh CLI / Docker / npm uninstall hook / version mismatch / live PIDs / stale uninstall.lock)
- "uninstall 직전 PID 정지 필수" warn callout (archive snapshot SHA 일치 위해)

### 2.2 §10.5 봇 · 스케줄러 · 에이전트 git 작업 (push 범위로 한정)

**문제 인식 1 — 스케줄러가 디폴트 실행 안 됨**: SoloSquad는 데몬·자동 시작이 없는데, 사용자가 그걸 모르고 *"왜 routine이 안 돌아?"* 또는 반대로 *"왜 doctor에서 live PID 경고가 떠?"* 혼동 발생.

**문제 인식 2 — 에이전트 책임 경계 미명시**: v0.8.2 `dev_capability` 기능은 있지만, *에이전트가 어디까지 자동으로 하고 어디부터 사용자 책임인지* master-guide에 명시 없음. 초기 draft에는 *gh CLI 기반 PR 자동 생성*까지 포함했으나, 다음 두 문제 발견:
1. **gh CLI 셋업이 *그 자체로 별도 onboarding 단계*** — 솔로 사용자 marginal cost 큼
2. **push 인증 셋업이 *gh CLI 유무와 무관하게 항상 필요*** — gh CLI는 PR API 호출용일 뿐, push는 git의 credential helper 영역
→ **v0.8.6 범위 결정: 에이전트는 git push까지. PR 생성·리뷰·머지는 사용자가 GitHub 웹 UI**

**박제 내용 (v0.8.6 최종)**:
- 스케줄러 비자동 실행 명시 + Windows `Get-CimInstance` 매칭 로직 공개
- 에이전트 책임 경계 표 — push까지 ✓ / PR 라이프사이클은 사용자 책임
- 전제 3건 (git push 인증 / repo 등록 / workspace.yaml dev_capability) — **gh CLI 제거**
- push 흐름 (사용자 메시지 → PM 분류 → BD spawn → dev-confirm gate → push → "compare URL" 회신)
- 온보딩 추가 5건 — **gh CLI 단계 제거**:
  - Step 1.5 git push 인증 확인 (첫 dev 사용자만)
  - Step 7.5 repo 등록 + push 검증
  - Step 7.7 workspace.yaml dev_capability 명시
  - Step 8.5 메신저 dry test
  - Step 8.7 branch protection (선택)
- **인증 절차 없음 — SoloSquad는 안내만** — §4.2 Step 1 callout과 §10.1 트러블슈팅에 짧은 한 문단씩만. git 표준 흐름 (Windows GCM 자동 / macOS osxkeychain / Linux는 사용자 별도 셋업)에 위임. 별도 절(§10.5.1 같은 절차 가이드) *추가하지 않음*. 자세한 절차는 GitHub 공식 docs 링크로 위임
- v1.x followup callout — gh CLI 트랙 / MCP 트랙 / 다중-에이전트 토론을 *모두 v1.x 슬롯*으로 분리

## 4. 에이전트가 기획 → 브랜치 → 작업 → push하는 방법 (v0.8.6 현재 범위)

### 4.1 v0.8.6 현재 — *push까지가 에이전트 범위, PR은 사용자 책임*

**전제 (사용자 사전 셋업)**:
- git push 인증 (cred helper / SSH key / PAT — §10.5.1) — *gh CLI 불필요*
- repo 등록 (`solosquad add repo`)
- workspace.yaml `dev_capability.enabled: true`
- SKILL에 `dev_capability: true` + 적절한 `dev_permissions`

**흐름**:
```
1. 기획 (사용자 또는 PM 슬래시):
   "/plan Stripe 결제 통합" → PM이 PRD.md + _status.yaml 생성

2. 브랜치 (사용자 명시 또는 첫 specialist가):
   사용자: "feat/stripe-integration 브랜치에서 작업 시작"
   PM → backend-developer spawn (target_repo: my-saas)
   BD: git checkout -b feat/stripe-integration && [코드 작성] && commit

3. push:
   BD: git push → dev-confirm gate (30분 timeout) → 사용자 y → push 실행
   BD 회신: "✓ Branch pushed. PR 생성: https://github.com/.../compare/main...feat/stripe-integration"

4. PR 생성·검토·머지 — *여기부터 사용자*:
   사용자가 URL 클릭 → GitHub 웹 UI에서 PR 생성 → 코드 검토 → 머지
   (필요시 사용자가 메신저로 "BD에게 추가 변경 요청" → 반복)
```

각 PR 라이프사이클 단계가 *사용자 명시 액션*. 반자동 — 코드 작업만 자동, *결정·머지 통제 100% 사용자*.

### 4.2 v1.x 자동 — workflow.yaml `git_workflow` + `reviewers` (코드 없음, 박제만)

§3.2 설계 따라 *사용자 한 마디* → 전체 흐름 자동:
- 브랜치 자동 생성 (workflow.id 기반 명명)
- 단계별 commit 자동 (각 stage가 같은 branch에 push)
- 마지막 stage 종료 시 `auto_pr` 트리거 → dev-confirm → PR 생성 (gh 또는 MCP 트랙)
- `reviewers` 리스트 에이전트 자동 spawn (round 1)
- PM이 리뷰 결과 aggregation → works-handle post
- 사용자가 *결정 지점* (discussion round 2 트리거 / merge 승인)에서만 개입

여전히 **사람 게이트 영구 보존**:
- `git push` (dev-confirm)
- `gh pr merge` 또는 MCP merge tool (dev-confirm + workflow.yaml `auto_merge: false` 영구)
- 자동 round-trip 횟수 cap (`discussion_rounds`, 기본 2)
