## 1. 배경 / 문제

세 가지 운영 결함이 같은 증상("봇이 등록 repo를 못 읽음 / 작업마다 승인 요구로 흐름 끊김")으로

### 1.1 결함 A — `--add-dir` 무력화 (hotfix 양식)

- **증상:** Discord 봇이 등록된 외부 repo(`C:\Dev\bv-po-*`)의 파일 Read 시 *"Claude requested
  permissions to read … but you haven't granted it yet"* 거부. 봇은 1.3.9, repo 등록·`--add-dir`
  배선 전부 정상인데도.
- **근본 원인(격리 검증 완료):** **Claude Code 2.1.191 이 `--input-format stream-json`(stdin JSON
  입력) 모드에서 `--add-dir` 를 무시한다.** 봇은 라이브 토큰 스트리밍(`--include-partial-messages`)을
  위해 사용자 메시지를 stream-json 으로 stdin 에 흘려보내므로(`claude-process.ts:streamInput`),
  `--add-dir <repos>` 가 적용되지 않음. → 등록 repo 가 허용 디렉토리에 안 들어가 Read 거부.
- **증거 매트릭스:**

  | 입력 모드 | --add-dir | 결과 |
  |---|---|---|
  | `-p` (프롬프트 인자) | 동일 경로 | ✅ 읽힘 |
  | `--output-format stream-json` + `-p` | 동일 경로 | ✅ 읽힘 |
  | **`--input-format stream-json`** (봇) | 동일 경로 | ❌ ACCESS_DENIED |
  | settings `permissions.additionalDirectories` + stream-json | — | ❌ ACCESS_DENIED (우회 안 됨) |

  런타임 계측으로 `collectRegisteredRepoPaths`=3 경로 반환, `buildArgs` `hasAddDir=true` 확인 —
  **solosquad 측은 전부 정상.** claude CLI 2.1.x 회귀.
- **영향:** claude-code 2.1.x + 외부 repo(path-reference 모델)를 쓰는 **모든 solosquad 사용자**가
  봇에서 등록 repo Read/Edit 차단. (`-p` 기반 `solosquad chat` 은 정상 — 봇만.)

### 1.2 결함 B — 안전 작업도 매번 승인 요구 → 흐름 단절

- 등록 repo CRUD·WebFetch·git commit/push·일반 Bash 같은 **안전한 작업마다** 권한 확인이
  걸려 작업이 중간에 끊김. 사용자는 "Claude Code 의 shift+tab auto-mode 가 허용하는 범위를
  기본으로 허용"하길 원함 — 일일이 작업 중간 승인 금지.
- (결함 A 가 이 증상의 큰 부분이었음 — Read 거부가 "승인 필요"로 표출. A 해결 후 남는 건

## 2. 핵심 개념

1. **claude 입력 모드 ⟂ `--add-dir`.** `--input-format stream-json` 일 때만 `--add-dir` 무시(검증).
   출력 스트리밍(`--output-format stream-json` + `--include-partial-messages`)은 `--add-dir` 와 무관 —
   입력만 `-p` 로 바꾸면 스트리밍 유지하며 dir 접근 복구.
2. **권한 = 2층.** ⑴ solosquad 정책(allowed/disallowed tools + permission-mode) ⑵ claude 권한 시스템.
   원칙: **안전 작업은 기본 허용**(흐름 무중단), **되돌리기 어려운 것만 게이트**(파괴·protected push·PR merge).
3. **headless 승인 UI 는 solosquad 가 중계해야 한다.** `claude --print` 는 터미널 권한 프롬프트를
   Discord 에 못 띄움. 게이트가 필요하면 **PreToolUse 훅이 Discord 버튼 카드**(Allow/Deny)를
   `command-<handle>` 에 띄우고 결과를 훅에 되먹인다(v1.3.0 dev-confirm 카드 확장). 에이전트는
   승인을 *유발*만 하고 *안내*하지 않는다.
4. **세션 재시작 정책 = 기본 resume + 조건부 reset.** spawn/권한/입력경로를 바꾸는 마이그레이션은
   세션을 아카이브(기존 패턴)해 새 정책을 픽업. 평상시 재시작은 resume(연속성·캐시).

## 3. 스코프

### 3.1 `--add-dir` 살리기 — 봇 입력 경로 수정 (결함 A)

- 봇 spawn 입력을 **`--input-format stream-json`(stdin) → `-p`/prompt 인자**로 전환. **출력은
  `--output-format stream-json` + `--include-partial-messages` 유지**(Discord 라이브 스트리밍 불변).
- `claude-process.ts` `buildArgs`: `--input-format stream-json` 제거, 사용자 메시지를 `-p <text>`(또는
  `--prompt`)로. `streamInput`(stdin write) 경로는 입력 미사용 시 no-op.
- 검증: claude 2.1.191 에서 `--output-format stream-json + -p + --add-dir` = 정상(완료). 회귀
  테스트로 "stream-json 입력이면 --add-dir 무시" 가드.
- **엣지(§7 OQ#1):** 매우 긴 메시지(Windows argv ~32KB 한도)·첨부/이미지 콘텐츠. 한도 초과 시
  fallback(파일 경유 또는 분할). MVP 는 텍스트 메시지 기준.

### 3.2 권한 디폴트 허용 — 안전 작업 무중단 (결함 B·Q2–Q4)

`chief-permissions.ts` 재정의. **기본 허용(승인 0)** vs **게이트(되돌리기 어려움)** 분리:

| 분류 | 항목 | 처리 |
|---|---|---|
| 🟢 기본 허용 | Read·Grep·Glob·Edit·Write(=acceptEdits)·WebFetch·WebSearch·Bash(일반)·git add/commit·**feature branch push**·gh pr create | 승인 없이 즉시 |
| 🔴 게이트(카드) | **protected branch(main/master/develop) push** · gh pr merge/close · 파괴(rm -rf·sudo·mkfs·dd) | Discord 버튼 카드(§3.3) |

- permission-mode: `acceptEdits` 유지(파일 편집 자동). Bash·WebFetch 등은 allow-list 로 기본 허용.
- "auto-mode 동등 이상" — Claude Code shift+tab(acceptEdits)이 자동화하는 편집 + 사용자가 명시한
  안전 작업(bash·webfetch·git)을 기본 허용. **작업 중간 승인 프롬프트 0 이 목표**(게이트 3종 제외).
- dev OFF(읽기 전용 모드)는 유지 옵션 — 단 기본은 dev ON(안전 기본 허용).

### 3.3 Discord 승인 UI — 게이트 남는 것만 버튼 카드 (결함 C·Q5)

- 게이트 3종(protected push·PR merge/close·파괴)만 **Discord 버튼 카드**(discord.js Components:
  Allow / Deny 버튼)를 `command-<handle>` 에 게시. dev-confirm 훅이 카드 게시 → 버튼 클릭 →
  훅 exit(allow/deny). **텍스트 "허용/거부"도 fallback 인식**(버튼 못 쓰는 환경).
- 에이전트 시스템 프롬프트 정정: "권한은 시스템이 카드로 처리 — 너는 '눌러 주세요' 안내 금지".
  환각성 승인 안내 제거.

### 3.4 세션 재시작 정책 (Q6 — §6 D3 반영)

- **기본 resume 유지.** 구현 중 확인된 사실: **spawn 인자(`buildArgs`)는 매 턴 재생성**되므로
  1.3.10 의 입력경로·권한 변경은 **resumed 세션도 다음 턴에 자동 픽업**한다 → **강제 세션
  아카이브 불필요**(마이그레이션은 no-op 버전 범프). 옛 거부 이력으로 인한 *클린 슬레이트*가
  필요하면 봇의 기존 `session-not-found` 회전 또는 수동 리셋으로 충분.
- (후속 후보, §3.5) 세션 나이/크기 임계 reset 권유 + 수동 `/reset`.

### 3.5 이번 버전에서 다루지 않음 (R2 — 보류, §8 영구 비범위와 구분)

| 보류 | 이유 | 후속 |
|---|---|---|
| `/reset` 수동 명령 + 세션 나이/크기 자동 권유 | 1.3.10 핵심 아님(마이그레이션 아카이브로 충분) | 후속 |
| 권한 정책 GUI/설정 화면 | 코드/`pm.git` 설정으로 충분 | 후속 |
| 긴 메시지/첨부 입력 fallback 고도화 | OQ#1, MVP 는 텍스트 기준 | 후속 patch |
| claude-code 다운그레이드 자동화 | -p 전환(§3.1)이 근본 해결 | — |

## 5. 마이그레이션 영향

- `1.3.9-to-1.3.10`: **no-op 버전 범프**. spawn 인자가 매 턴 재생성되므로(§3.4) 입력경로·권한
  변경은 기존 세션 재개 시 자동 적용 — 세션 데이터를 건드릴 필요 없음. 사용자 데이터 무변경.
- claude-code 버전 의존: 2.1.x 에서 검증. 입력 경로 변경은 claude-code 버전 무관하게 안전.

## 6. 설계 결정 (approaches 비교 — `minimum_approaches: 2`)

**D1. `--add-dir` 무력화 수정** — 채택: A
- A) **봇 입력을 `-p` 로 전환**(stream-json 입력 제거) · B) claude-code 다운그레이드(working 버전 pin) ·
  C) 업스트림 대기.
- 권고 A. rationale: 근본 해결(우리 통제 범위) + 출력 스트리밍 유지(검증됨). B 는 claude-code 개선
  상실·핀 관리 부담, C 는 무기한 차단. 반증: `-p` 가 긴 메시지/첨부에서 깨지면 그 경로만 stream-json
  유지(혼합) → 후속(OQ#1). **업스트림 리포트는 병행**(A 와 무관하게).

**D2. 권한 모델** — 채택: A
- A) **큐레이트 allow-list**(안전 기본 허용 + 게이트 3종) · B) `bypassPermissions`(전부 허용) ·
  C) 현행 유지.
- 권고 A. rationale: 사용자 요구(안전 작업 무중단)와 안전(되돌리기 어려운 것 게이트)을 동시 충족.
  B 는 protected push·rm -rf 까지 무중단 → 과위험. C 는 흐름 단절 지속. 반증: 게이트 3종조차
  방해라면 사용자 옵트인으로 B 제공.
- `[결정: git push]` feature branch = 기본 허용, protected(main/master/develop) = 게이트. (pm.git
  `protected_branches` 로 사용자 조정.)

**D3. 세션 재시작 정책** — 채택: 하이브리드(기본 resume + 조건부 reset)
- A) 항상 resume · B) 항상 reset · C) **기본 resume + spawn/설정 변경 시 아카이브→fresh**(채택).
- 비교:

  | 축 | resume | reset |
  |---|---|---|
  | 연속성 | ✅ 과제 맥락 유지 | ❌ 상실(작업 중 치명) |
  | 비용/지연 | ✅ 캐시 재사용 | ❌ 재구성 |
  | 설정 변경 반영 | ❌ 지연 리스크 | ✅ 즉시 |
  | 상태 청결 | ❌ 오염 누적($4.25·5/28생 실측) | ✅ 클린 |

  rationale: 평상 재시작은 연속성·비용이 이기고(resume), 권한·입력경로가 바뀌는 릴리즈는 청결·정확성이
  이긴다(reset). solosquad 가 이미 마이그레이션-기반 아카이브 보유 → C 가 최소 비용 최적. 반증:
  사용자가 매 재시작 클린을 원하면 `/reset` 옵트인(§3.5).
