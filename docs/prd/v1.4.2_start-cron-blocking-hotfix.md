# v1.4.2 — `solosquad start` / `bot --with-cron` 봇 미기동 핫픽스

> **상태:** 출시 (2026-06-27) · 핫픽스 (1.4.1 위). PRD↔release 1:1 (hotfix-format).
> **유형:** [개발] 버그 픽스(단일 프로세스 기동 순서).

## 1. 증상

1.4.1 의 `solosquad start`(=`bot --with-cron`)로 띄우면 **스케줄러만 뜨고 봇이 안 떠** — 디스코드에서 응답 없음. 로그상 `[Scheduler] … Scheduler started` 까지만 찍히고 `[Bot] …`(로그인·Ready) 라인이 **하나도 안 나옴**. 별도 `solosquad bot` 는 정상.

## 2. 근본원인

`startScheduler()` 는 단독 `cron start` 용으로 **함수 끝에 무한 keep-alive** 를 갖는다:
```ts
console.log("[Scheduler] Scheduler started");
await sendStartupNotification(crons);
await new Promise<void>(() => {});  // ← 영영 resolve 안 됨
```
1.4.1 의 `bot --with-cron` 경로가 이 함수를 **`await startScheduler()` 로 봇보다 먼저** 호출 → keep-alive 에서 영구 블록 → 뒤따르는 `startBot()` 이 **호출조차 안 됨** → 봇 미연결.

## 3. 수정

`startScheduler(opts: { keepAlive?: boolean })` 추가. 봇 동반 경로는 **`keepAlive: false`** 로 호출 → 스케줄러가 cron 등록 후 **리턴**, 그 다음 `startBot()` 이 실행되어 프로세스 수명을 봇이 소유(등록된 cron 타이머·fs-watcher 는 공유 이벤트루프에 그대로 상주). 단독 `cron start` 는 기본값(`keepAlive: true`) 유지.

- `src/cron/index.ts` — `startScheduler({ keepAlive })`, 끝 keep-alive 를 `keepAlive !== false` 일 때만.
- `src/cli/index.ts` — `bot --with-cron` 가 `startScheduler({ keepAlive: false })` 후 `startBot()`.
- 부수: 봇 연결과 스케줄러 notifier 연결이 **순차**(스케줄러 setup 완료 → 봇)라 identify 레이스도 줄어듦. scheduler 싱글톤 lock(1.4.1) 그대로 — 별도 `cron start` 이중발화 방지 유지.

## 4. 검증

- 회귀: `solosquad start` → `[Scheduler] Scheduler started` 뒤로 `[Bot] … Ready` 가 이어지고 디스코드 응답.
- 단독 `solosquad bot`·`cron start` 무영향(전체 테스트 그린).
- 연속성 마이그레이션 `1.4.1-to-1.4.2` plain bump(기동 순서 픽스 — 데이터·spawn 무변경, 세션 리셋 없음).

## 5. 회고

- 함수가 "등록"과 "keep-alive"를 한 데 묶어 `await` 의미가 호출 맥락에 따라 달랐던 게 함정. 임베드 시 keep-alive 분리(옵션)로 해소.
- 가드: 단일 프로세스 기동 경로는 **봇 로그인 라인까지 콘솔로 확인**하는 dogfood 를 릴리스 전 체크리스트에 추가(순수 단위 테스트로는 프로세스 상주가 안 잡힘).

## 6. 추가 핫픽스 — rate-limit 알림 도배 제거

**증상:** 디스코드 답변마다 `⚠️ Claude Code reported a rate-limit constraint…` 가 **매번 같이** 떴다.

**근본원인:** Claude Code 가 매 턴 rate-limit 상태(`allowed`/`warning`/`exceeded`)를 보고하는데(`chief-runner.ts:903` `rate_limit_event`), 봇이 **비-`allowed` 면 무조건** echo(`index.ts` `if (reply.rateLimited)`). 사용량이 한도에 **근접(`warning`)** 한 사용자는 매 턴 `warning` 이 와서 답변마다 도배됐다(봇은 정상 동작 — `warning` 은 차단 아님).

**수정:**
- chief-runner 가 상태/`resetsAt` 를 `ChiefReply.rateLimit: { status, resetsAt }` 로 전달(비-allowed 를 `exceeded`|`warning` 두 단계로 정규화).
- `RateLimitNotifier`(`bot/rate-limit-notice.ts`) 가 **유저별·(status,리셋윈도우)별 1회만** 알림: `warning` = 윈도우당 1회(`resetsAt` 를 Discord 상대시각으로 표시), `exceeded` = 즉시(전이 시 재알림). 봇 프로세스 메모리 상태(재시작 시 1회 재알림 — 무해).

**검증:** dedupe 5케이스(첫 1회·동일윈도우 무음·윈도우롤오버 재알림·warning→exceeded 전이·유저별 독립) + chief-runner 가 `rate_limit_event` 를 `reply.rateLimit` 로 전달(`warning`/`allowed`).
