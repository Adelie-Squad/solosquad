## 1. Summary

봇이 과제(works 스레드) 단위로 **별도 claude 프로세스를 스폰**하는 워커 러너를 신설한다. 자식 세션 키 `(userId, orgSlug, taskId)` → `sessions/<user>__<task>.json`, cwd = 과제 scope repo(S-6 리졸버), 수명 = **스폰 → hydrate → flush → kill**(요약만 메인 Chief 세션으로 환류). 리스너를 works-스레드 메시지가 **해당 과제 자식 세션**으로 라우팅하도록 확장(v1.4.1 Approach A 의 공유-세션 → 격리-자식으로 승격). 부수로 §5.6 **goal 중간개입**을 같은 워커 가로채기 인프라로 얹는다(engine goal-runner **미편집** — 참조만).

### 2.2 문제 (공백 — v1.4.0 §11.3 감사)

- **격리 부재:** v1.4.1 은 스레드를 읽지만 **공유 세션**이라 과제 A 맥락이 과제 B 로 샌다. "그 작업에 대해서만" 이 성립 안 함(Approach A 의 한계, 의도된 후속).
- **flush 불가:** 단일 세션은 과제 종료 시 컨텍스트를 버릴 수 없다 → 장수 과제가 메인 Chief 를 블로트.
- **위치 제어 불가:** 과제가 특정 repo 작업이어도 in-process Task 는 부모 cwd 상속 → 봇이 cwd·경계를 강제 못 함.
- **goal 개입 불가:** 자율 goal 중 사용자가 works 스레드에서 한 줄 개입해도 다음 stage 에 반영할 경로가 없다(engine 이 메시지를 안 봄).

## 3. 핵심 개념 — 워커 세션 + 디스패치 가로채기

**세션 토폴로지(하이브리드 M3, v1.4.0 §11.2):**

| 세션 | 표면 | 키 | cwd | 컨텍스트 | 수명 |
|---|---|---|---|---|---|
| **메인(오케스트레이터)** | `command-<handle>` | `sessions/<user>.json`(현행) | org-root | 오케스트레이션·단순 확인 | 장수(chief reset 로 회전) |
| **과제(자식 워커)** | works 스레드 1개=1과제 | **신규** `sessions/<user>__<task>.json` | 과제 scope repo(S-6) | **그 과제만** | 스폰→hydrate→flush→kill, 요약만 잔존 |

- **디스패치 마커 가로채기(v1.4.0 §4):** Chief 가 워커를 "출장 보내는" 마커를 발행 → 봇이 mention-parser/slash 처럼 **Chief 디스패치 *전에* 가로채** 워커를 띄운다. 워커는 봇이 직접 스폰/킬(goal-runner 패턴 일반화, **engine 미편집**).
- **hydrate→flush→kill:** 스폰 시 과제 컨텍스트 주입(hydrate) → 작업 → 종료 시 `_log.md`/`_handoff.md` 요약 flush → 자식 세션 kill(요약만 메인 환류 → 과제 격리 + 메인 위생).
- **후속 라우팅:** 같은 스레드의 이어지는 메시지는 `resolveWorkflowIdByThread` 키로 **같은 자식 세션 resume**(v1.4.1 역방향 조회 재사용).

## 4. 접근 옵션 (≥2)

### A. 봇 스폰 워커 러너 (권고 — v1.4.0 §4 채택 모델)
봇이 `engine/goal-runner` 의 백그라운드 세션 패턴을 **참조**해 새 워커 러너(`src/bot/worker-runner.ts`)를 두고, 자식 세션을 직접 스폰/킬. goal 중간개입도 같은 러너의 가로채기 훅으로.
- **장점:** engine 불변 경계 준수(리스크 #5), 봇이 cwd·hydrate/flush 강제, S-4/S-5·goal개입 단일 인프라.
- **단점:** 새 스폰 경로 신설(in-process Task 우회) → 프로세스 수명·정리 로직 필요. diff 큼.

### B. engine goal-runner 확장 재사용
기존 `engine/goal-runner` 에 works-스레드 세션 관리를 얹어 재사용.
- **장점:** 스폰/수명 로직 재사용, 신규 모듈 최소.
- **단점:** **v1.4.0 이 `src/engine/**` 를 immutable 로 명시**(참조만) — 위반. goal 전용 코드에 대화 세션 책임이 섞여 결합도 폭증. **기각.**

**선택 = A.** engine immutable 경계가 하드 제약이라 B 불가. A 로 하되 goal-runner 는 *패턴만* 참조, 코드 이식 금지.

## 5. 스코프 (S-4)

**In:**
1. **워커 러너 신설**(`src/bot/worker-runner.ts`) — 봇이 자식 claude 프로세스 스폰/킬, hydrate→flush→kill 수명.
2. **자식 세션 키 `(userId, orgSlug, taskId)`** — `session-store.ts` 에 `<user>__<task>.json` 서브세션 CRUD(기존 `(user,org)` 시그니처 보존, 오버로드/신규 함수).
3. **리스너 라우팅 승격** — works-스레드 메시지 → 공유 세션(Approach A) 대신 **해당 과제 자식 세션**(Approach B). `resolveWorkflowIdByThread` 키 재사용.
4. **디스패치 마커 가로채기** — Chief 워커 위임 마커 파싱 → 봇이 워커 스폰(mention-parser 전처리 동형).
5. **메인↔자식 환류** — 과제 flush 시 요약 1줄을 메인 Chief 세션에 주입(§5.1 핸드오프 동형).
6. **§5.6 goal 중간개입** — works 스레드 개입 한 줄을 `_interventions.jsonl` audit + 다음 stage prompt inject(goal whitelist fire-and-forget). **같은 가로채기 인프라**, engine 미편집.
7. **자식 세션 토큰 관리** — §5.1 임계 자동 교대(warn 0.70/rotate 0.85)·§5.2 `_log.md` durable 을 **자식 세션에도 적용**(장수 과제 대비).

**Out(§9):** S-5 메인↔자식 완전 역할분리·학습형 메모리 · S-7 cron 자산 통합 · S-2b 메인 세션 회전 자체 · S-3b GC 삭제.

## 7. 자식 세션 수명 · 토큰 (v1.4.0 §11.4)

- **현행 사실:** 봇 히스토리 절단 없음 — 전적으로 claude `--resume` 압축 의존. per-turn `$5`/`300s` 가드뿐(`chief-runner.ts`). spawn 80k 캡은 자식 세션 미적용.

## 8. 마이그레이션 `1.4.3-to-1.4.4`

- **세션 스키마:** 자식 세션은 신규 파일(`<user>__<task>.json`) — 기존 `<user>.json` 무영향(class A 보존). 강제 시드 없음.
- **spawn 행동 변경 = Yes → §5.7 세션 리셋 적용:** 본 patch 는 **워커 스폰 경로를 신설**(spawn 행동 변경)하므로, v1.4.0 §5.7 원칙에 따라 마이그레이션이 `archiveOrgChiefSessions`(1.4.0 헬퍼)로 **각 org Chief 세션을 아카이브**한다 — 옛 세션이 "단일 세션" 가정으로 오도되지 않게 클린 슬레이트. (1.4.0~1.4.2 는 No 라 no-op 이었음.)
- **immutable 규칙:** 발행분 마이그레이션 미편집, 신규 `1.4.3-to-1.4.4` 만 추가.

## 10. 검증 (Given-When-Then AC — R8)

- `npx tsc --noEmit` · `npm run build` · `npm test` · `solosquad validate` green.
- **AC-1 (격리):** *Given* 과제 스레드 A·B 각각 자식 세션, *When* A 스레드에서 "X 를 기억해" 후 B 스레드에서 "방금 뭐라 했지?", *Then* B 는 X 를 모른다(세션 격리). *Then* command 채널(메인)도 A·B 세부 미보유.
- **AC-2 (후속 라우팅):** *Given* 과제 A 자식 세션 존재, *When* 같은 스레드에 후속 메시지, *Then* 새 스레드/세션 생성 없이 **같은 자식 세션 resume**(`resolveWorkflowIdByThread` 키).
- **AC-3 (환류):** *Given* 과제 A 종료(스레드 archive), *When* flush, *Then* 요약 1줄만 메인 Chief 세션에 주입 + 자식 세션 kill. 메인은 과제 A 원문 미보유.
- **AC-4 (cwd·S-6 소비):** *Given* 과제가 repo `b` 지정, *When* 워커 스폰, *Then* cwd=`b` 실경로(S-6 `resolveExecutionContext`), addDirs 정확.
- **AC-5 (세션 리셋):** *Given* 1.4.3→1.4.4 마이그레이션, *When* apply, *Then* 각 org Chief 세션 아카이브(spawn 변경 릴리즈) + 다음 턴 fresh 세션.
- **AC-6 (goal 개입):** *Given* 자율 goal 진행 중, *When* 사용자가 works 스레드에 한 줄 개입, *Then* `_interventions.jsonl` 기록 + 다음 stage prompt 에 inject(engine 코드 무변경).
- **스모크:** `build-test`. **pre-publish:** docs-freshness 게이트 + docs-check.
