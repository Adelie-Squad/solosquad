# v1.3.1 — Post-release CI / dependency hardening (on v1.3.0)

> **상태:** 정리/박제 (2026-06-16). v1.3.0 을 npm 에 publish + PR #8 머지하는
> 과정에서 **사전 존재하던** CI·의존성 문제가 머지 게이트에 드러나 동일 day 수정.
> 실제 작업은 main 의 commit chain `9efd3b9 → 417537f → 28f996c` (PR #8 에 포함되어
> 머지됨). 본 문서는 *역사적 박제* — v1.3.0 기능과 무관한 **빌드 파이프라인/의존성
> baseline 정합** 작업.
>
> **npm publish 여부:** 선택. 변경 3건 중 tarball 에 실제로 들어가는 것은
> `package.json.engines` 힌트(`>=18`→`>=20`)뿐이고(lockfile·CI 설정은 미배포),
> 그마저 better-sqlite3 가 이미 install 시 강제하던 사실의 명문화라 **별도 1.3.1
> publish 는 불요** — 다음 minor publish 에 자연 흡수해도 무방. 게시된 `solosquad@1.3.0`
> tarball 은 영향 없음(릴리스 커밋 `1880657` 기준, lockfile/CI/engines 동작 무변).

**로드맵 위치:** v1.3.0 (메신저 UX 대개편 — 게이트+인터랙션+P0/P1) → **v1.3.1 (본
post-release CI/deps 하드닝)** → v1.4.0 (세션 오케스트레이션).

---

## 1. 발견 경위

`solosquad@1.3.0` publish 후 PR #8(`feat/v1.3.0-messenger-ux-overhaul` → `main`)
머지를 시도하자 CI 가 전 플랫폼 red. v1.3.0 코드/의존성 변경이 유발한 게 아니라
**main 도 이미 같은 게이트로 red** 였던 사전존재 조건이, "머지 전 green" 기준 때문에
비로소 정면으로 드러난 케이스.

핵심 진단 함정 2개:
- **fail-fast 가 진짜 원인을 가렸다.** matrix `fail-fast`(기본 true) 때문에 Node 18
  잡이 `npm ci` 에서 죽으면 나머지 Node 20/22 잡이 *컴파일 도중 cancel* → "전부 fail"
  로 보여 어느 조합이 실제 통과하는지 알 수 없었음.
- **audit fail-fast 가 빌드 실패를 가렸다.** 첫 런에서 `npm audit` 게이트가 한 잡에서
  먼저 터지자 나머지가 cancel → Node 18/20 의 native-build 실패가 첫 런엔 안 보였음.

## 2. 근본 원인 (서로 다른 3건)

| # | 증상 | 근본 원인 |
|---|---|---|
| A | CI `npm audit --audit-level=high` 게이트 red (전 플랫폼) | 전이 의존성의 신규 고위험 권고 3건 — `esbuild`(Deno 바이너리 무결성 RCE / Windows dev-server 파일읽기), `form-data`(CRLF injection), `ws`(tiny-fragment DoS). v1.3.0 은 **직접 의존성을 추가하지 않음** → 권고는 시간이 지나며 publish 된 것이고 게이트가 뒤늦게 trip. |
| B | `npm ci` 가 Node 18 (ubuntu/windows) 에서 `better-sqlite3` 빌드 실패 | `better-sqlite3` 12.x 의 `engines` = `20.x \|\| 22.x \|\| …` — **Node 18 미지원**. Node 18 prebuilt 부재 → 소스 컴파일 폴백 → 툴체인 없는 러너(특히 Windows = VS 부재)에서 실패. 그런데 `package.json.engines` 는 `>=18` 로 **거짓 주장** 중이었음. |
| C | `npm ci` 가 Windows + Node 20 에서만 추가 실패 | `better-sqlite3` 12.x 가 win32 + Node 20(ABI 115) prebuilt 를 제공하지 않음 → 소스 빌드 → Windows 러너 VS 부재로 실패. (Node 22 win prebuilt 는 존재 → windows-22 통과. ubuntu-20 은 gcc 로 소스 컴파일 성공.) |

## 3. 영향

- v1.3.0 자체 기능/배포물은 정상(audit/CI 는 게시 tarball 에 미포함). 영향은 **레포의
  머지 게이트 + 향후 개발자 install 경험**에 한정.
- B 의 결과로 `engines: ">=18"` 은 사실과 불일치 — Node 18 사용자는 install 단계에서
  better-sqlite3 가 어차피 거부(또는 소스빌드 실패).

## 4. 변경 내역

### A — 고위험 권고 제거 (`9efd3b9`)
- `npm audit fix` (non-breaking, semver 범위 내) → `esbuild`(→0.28.1)·`form-data`·`ws`
  high 3건 해소 + `qs`·`js-yaml` moderate 동반 정리. **`package.json` 직접 의존성
  무변경, `package-lock.json` 만 변경.** 적용 후 `npm audit --audit-level=high` exit 0.
- 잔존: `uuid`(via `node-cron`) moderate 2건 — `--force`(node-cron major) 필요 +
  high 게이트 비대상이라 **후속**으로 분리.

### B — Node baseline 정합 (`417537f`)
- `package.json` `engines.node` **`>=18` → `>=20`** (better-sqlite3 현실 반영).
- CI matrix node **`[18,20,22]` → `[20,22]`**.
- `strategy.fail-fast: false` — 조합별 실제 결과를 노출(취소가 진짜 실패를 가리지 않게).
- CHANGELOG `[1.3.0]` 에 인프라 변경 박제(engines·audit).

### C — Windows+Node20 조합 제외 (`28f996c`)
- matrix `exclude: { os: windows-latest, node: 20 }` — better-sqlite3 win32 Node20
  prebuilt 부재 + 러너 VS 부재의 결정적 조합만 제거. **Windows 는 Node 22 로 커버 유지.**

## 5. 결과 / 검증

- 최종 CI: **5/5 green** — ubuntu(20·22) · macos(20·22) · windows(22).
- 로컬 `npx tsc --noEmit` · `npm run build` · `npm test` (790 pass) green 유지.
- PR #8 `--merge` 로 main 머지 (2026-06-16). 태그 `v1.3.0` 은 release 커밋
  `1880657` 을 그대로 가리킴(push 된 v\* 태그 강제 이동 금지 — 본 하드닝은 tarball
  동작 무변이라 retag 불요).

## 6. 비범위 / 후속

- `uuid`(via `node-cron`) moderate 권고 — `node-cron` major 업그레이드(`--force`)가
  필요하고 high 게이트 비대상. 별도 검토 후 처리.
- Windows + Node 20 네이티브 커버리지 복구(예: 러너에 build-tools 설치 또는
  better-sqlite3 prebuilt 갱신 추적) — 현재는 Node 22 로 Windows 커버.
- moderate 권고를 게이트에 포함할지(`--audit-level=moderate`) 정책 재검토.
