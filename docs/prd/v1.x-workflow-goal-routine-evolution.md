# v1.x — Workflow / Goal / Routine 고도화

> 정식 출시(v1.0) 직후의 v1.x 시리즈는 **작업 흐름 자체의 진화**를 다룬다. v0.x가 *기능을 만든* 시리즈였다면 v1.x는 *기능들 사이의 연결과 자율성*을 깊게 만드는 시리즈. 사용자(개발자 본인) ideation 답변 7건(2026-05-15)을 박제한 통합 plan. 세부 minor/patch 분리는 v1.0 출시 직후 본 plan을 *분할*하는 단계에서 결정 (v0.8 시리즈가 v0.8.0/0.8.1/0.8.2/0.8.3으로 split된 패턴 참조).

**로드맵 위치:** v0.8 시리즈 → v0.9 안정화 + 자체 사용 검증 → **v1.0 정식 출시 (API stability 시작)** → **v1.x 본 plan slot** → v1.1 대시보드 → v1.2 지식 온톨로지 → v1.3 일정 + 메모

**의존:** v0.4 (goal-runner CONFIRMING 상태머신), v0.5 (워크플로우 메이커 5단계), v0.6 (trajectory + freq miner), v0.8.0 (멀티 유저 채널 + broadcast cross-user feed §3.6 v2), v0.8.2 (dev_capability), v0.9 (자체 사용 검증 30+일 데이터)

**기준 버전:** v1.x (구체 minor/patch는 split 단계에서). v1.0 출시 후 1~2 분기 내 진입 예정.

**분류:** Capability 진화. 새 인프라 1건(실험) + 4 기존 기능 고도화(goal·워크플로우 메이커·루틴·인간 승인).

## 2. Leading Indicator — 24/7 자율 팀 측정 (Q1 b)

### 2.1 측정 지표

```
[일일 — morning brief에 포함]
  대화→작업 변환률    = (워크플로우+goal+루틴 생성 횟수) / (대화 메시지 수)
  자동 PR 성공률      = (gh pr create 성공) / (dev_capability spawn 횟수)
  자율 goal cycle 수   = 24h 동안 keep된 cycle 수
  dev_capability 활용도 = (engineering 5 SKILL spawn) / (전체 spawn)

[주간 — evening brief 일요일에 포함]
  자율 팀 hours      = 사용자 활성 시간 대비 봇 자율 작업 시간 비율
  R&R 충돌 횟수       = broadcast feed에서 동일 작업 중복 발견 (사용자 간)
  명령 → ack까지 시간 = 워크플로우 종료 ack의 latency 분포
```

### 2.2 측정 인프라

- `<org>/memory/leading-indicators.jsonl` (신규) — 매일 00:00 routine이 24h 데이터에서 4 지표 계산 + append
- morning brief routine이 이 jsonl 마지막 row를 brief 본문에 inline
- v0.6 trajectory miner와 정합 — leading indicator가 SKILL 추출 ROI의 supporting 지표

### 2.3 lagging 지표 (참고)

- 멀티 프로덕트(a): 워크스페이스당 active org 수·cross-product 작업 횟수
- 실험 기획(c): PMF 검증 goal 수·A/B 결과 통과율
- 측정 시 *상관관계*만 본다 — leading 지표가 올라가면 lagging 따라옴

## 3. Goal-Runner 고도화 (Q4 + Q5)

### 3.1 1-active-per-org 박제 (Q5)

- `<org>/goals/.active-goal` 단일 파일 — active goal id 1개 기록
- `solosquad goal run <id>`가 active 있으면 거부 + 안내: "현재 active goal: <id>. 종료 후 다시 시도하거나 `goal stop <active>` 호출"
- `solosquad goal queue <id>` (신규) — 큐에 추가, active 종료 시 자동 promote
- migration: 기존 (v0.4) 워크스페이스는 active 추적 없음 → 첫 `goal run` 시 자동으로 `active-goal` 파일 생성

### 3.2 중간 통지 + 중간 개입 (Q4)

**기본 흐름**:
```
1. 사용자: "PMF 검증 goal 시작"
2. PM: works-<handle>에 thread 생성, "PMF 검증 시작. cycle 1 진행 중"
3. cycle 1 진행:
   - 단계별로 PM이 thread에 상태 update (예: "stage 2/5 — segment 정의")
   - 사용자가 thread에 답변 가능 ("스타트업 segment부터")
   - PM이 메시지 감지 → 다음 stage prompt에 inject (실시간 반영)
4. cycle 1 종료:
   - PM: "cycle 1 결과: signup_cvr +5%. cycle 2 진행 vs 중단? [continue/stop]"
   - 사용자 ack → cycle 2 또는 종료
5. (옵션) 사용자가 fire-and-forget 화이트리스트: `goal whitelist <id>`
   - 중간 통지 생략 + 자동 ack
   - 종료 시 1회 보고만
```

**구현 보강**:
- `src/engine/goal-runner.ts` cycle loop 안에 *사용자 메시지 폴링* 추가 — 매 stage 사이 50~100ms마다 PM session 의 incoming 메시지 체크 (Claude SDK streaming 활용)
- `<org>/goals/<id>/_interventions.jsonl` — 사용자 중간 개입 audit log
- `goal.md` frontmatter에 `whitelisted: true` 옵션 — `goal whitelist <id>` 시 set

### 3.3 멀티 프로덕트 cross-goal (TBD)

- 1 goal이 여러 org reference하는 모델은 *여전히 TBD*. v1.x 진행 중 사용 데이터 보고 결정
- 잠재 옵션: `goal.md` 의 `target_orgs: [org-a, org-b]` 리스트 — 단 active 룰은 *각 org의 단일 active*로 유지 (cross-goal은 N org에서 동시 active 가능)

## 4. 워크플로우 메이커 — 암묵지 1차 source (Q2 b)

### 4.1 사용자 명시 슬래시 + 자연어 인식

**슬래시 입력**:
```bash
/save-as-skill <name>
  → 직전 PM 대화의 마지막 N 메시지를 컨텍스트로 SKILL.md draft 생성
  → SANDBOX_PROMPT 단계로 진입
```

**자연어 입력**:
```
사용자: "지금 한 거 SKILL로 저장해" / "방금 한 작업을 다음에도 자동으로" / etc
  → PM이 의도 파싱 → `_meta/workflow-maker` SKILL 호출
  → SKILL frontmatter draft 자동 추출 (name 미지정 시 사용자에게 prompt)
```

**source-of-truth 정책**:
- 1차: 사용자 명시 (위 두 경로)
- 2차: v0.6 freq miner — *제안만*, 자동 등록 안 함. 사용자가 명시 승인해야 SKILL.md 작성
- 3차 (v1.2 지식 온톨로지 정합): Notion·Obsidian·README → MCP server → SKILL draft 생성
- 4차 (장기): goal-runner keep/discard 패턴에서 *암묵 규칙* 역추출 — v1.x 후반

### 4.2 추출 흐름

```
1. 사용자 명시 트리거 (슬래시 또는 자연어)
2. PM이 직전 대화 컨텍스트 슬라이스 (마지막 N 메시지 또는 직전 워크플로우)
3. _meta/workflow-maker SKILL spawn:
   - CLARIFY: name·triggers·inputs·outputs 사용자에게 prompt
   - DRAFT: frontmatter + body 작성
   - SANDBOX_PROMPT: 1회 실행 테스트
   - AWAIT_CONFIRM: 사용자 검토
   - APPLIED: assets/agents/<team>/<name>/SKILL.md 또는 <org>/.agents/<name>/SKILL.md 저장
4. agent-router가 hot-reload로 인식 (v0.6 fs-watcher)
```

### 4.3 freq miner backup

- 30일 거절 cooldown 유지 (v0.6 §3.4)
- *자동 등록 안 함* (Q2 b 박제) — 사용자가 명시 승인 클릭해야 APPLIED
- 제안 알림은 morning brief에 inline: "최근 30일 freq miner가 N건 패턴 감지. `solosquad agent freq-suggestions` 로 확인"

## 5. 루틴 개인화 (Q6 사용자별)

### 5.1 사용자별 발송 모델

- 디폴트 3 루틴 (morning/evening brief + pm compaction) 모두 *user-scoped*
- 각 user yaml에 routine 설정 추가:
  ```yaml
  routines:
    morning_brief:
      enabled: true
      time: "08:00"
    evening_brief:
      enabled: true
      time: "18:00"
    pm_compaction:
      enabled: true
      time: "23:00"
  ```
- timezone은 user yaml에 명시 (기본 `Intl.DateTimeFormat().resolvedOptions().timeZone`)
- 발송 채널: 자기 `works-<handle>` 채널 thread

### 5.2 broadcast cross-user feed와 정합 (§3.6 v2)

- broadcast 활성 시: designated 봇이 *워크스페이스 요약* 1회 추가 push (각 사용자 brief 합쳐서)
- alice·bob·charlie의 brief는 각자 채널 + workspace 요약은 broadcast — 사용자 visible signal 깨끗

### 5.3 cron 등록 변경

- 현재 (v0.6): `src/scheduler/routines.ts`가 workspace 1 cron 등록
- v1.x: per-user cron 등록 — `<org>/.solosquad/users/<handle>.yaml`의 routine 설정 + timezone 기반 N개 cron entry 동적 등록
- `solosquad schedule` 시작 시 모든 user yaml 스캔 후 cron 등록

### 5.4 housekeeping은 workspace-scoped

- archive-rotate (v0.6) + log-rotate (v0.8.3)는 *system maintenance* — 사용자별 불필요. 그대로 workspace cron 1건씩 유지

## 6. 실험 인프라 — 신규 (Q7 a + Amplitude 패턴)

### 6.1 디렉토리 구조

```
<org>/experiments/
  <experiment-id>/
    manifest.yaml          (스펙)
    variants/
      control.yaml         (variant 정의)
      treatment.yaml
    metrics/
      conversion_rate.tsv  (시계열 결과)
      revenue.tsv
    _events.jsonl          (실험 라이프사이클: created/started/stopped/concluded)
    _last-fetch.iso        (마지막 데이터 수집 시점)
    decision.md            (최종 권고)
```

### 6.2 `manifest.yaml` 스키마

```yaml
schema_version: 1
id: signup-cvr-2026-q3
name: "Signup CVR 2-field reduction"
status: running  # draft|running|stopped|concluded
created_by: alice
created_at: "2026-08-01T09:00:00+09:00"
goal_ref: pmf-2026-q3  # 어느 goal과 연결됐는가

variants:
  control:
    description: "5-field signup form (existing)"
    allocation_pct: 50
  treatment:
    description: "2-field signup form (name + email only)"
    allocation_pct: 50

metrics:
  primary:
    name: signup_cvr
    direction: maximize
    threshold_delta_pct: 5      # 5% 이상 변화면 significant
  secondary:
    - name: time_to_signup
      direction: minimize
    - name: revenue_per_visitor
      direction: maximize

duration:
  sample_target: 2000           # variant당 sample 수
  max_days: 14
  min_runtime_days: 3

statistical:
  significance_threshold: 0.05  # p-value
  test: "two_proportion_z"      # or "welch_t" / "mann_whitney"
```

### 6.3 Amplitude 패턴 차용

**Ask Amplitude / Compose 데이터 분석 자동화 agent**의 핵심 기술 4종을 SoloSquad 실험 인프라에 차용:

1. **자연어 → query 자동 변환** — 사용자/specialist가 "signup_cvr를 KST 09:00~21:00 segment에서 측정" → `data-collector` SKILL이 metric source(Stripe·GA·Mixpanel 등 MCP) 호출
2. **anomaly/funnel/cohort 자동 분석** — `data-analyst` SKILL이 결과 fetch 후 anomaly detection (3σ 이상 outlier flag) + funnel drop-off 분석 + cohort 비교
3. **statistical significance 자동 check** — manifest.yaml의 `statistical.test`에 따라 p-value 계산 + confidence interval. 통과 시 `decision.md`에 *권고* 작성
4. **결정 → 권고 변환** — significance 통과 + delta ≥ threshold → "treatment 채택 권고". 미달 → "추가 cycle 또는 abandon 권고". `goal.md`의 다음 cycle 입력으로 자동 emit

### 6.4 CLI

```bash
solosquad experiment new <id> [--from-goal <goal-id>]
solosquad experiment list [--org <slug>] [--status <draft|running|...>]
solosquad experiment show <id>
solosquad experiment run <id>          # cron 등록 + 첫 fetch
solosquad experiment stop <id>
solosquad experiment conclude <id>     # 사용자 ack 후 decision.md 박제 → goal cycle 입력
```

### 6.5 goal 5 기본 정합

본 인프라 위에서 *PMF 검증·A/B 테스트·포지셔닝* goal이 자율 사이클 가능:
- PMF 검증 goal: experiment 시리즈 자동 생성 → segment 가설 검증
- A/B 테스트 goal: 1 experiment per cycle
- 포지셔닝 goal: messaging variants 비교 experiment

### 6.6 cron 통합

- `experiment.run` 시 `<org>/experiments/<id>/manifest.yaml`의 `duration` 기반 cron 등록
- 매일 또는 매시간 결과 fetch routine 발동
- min_runtime_days 미충족이면 fetch만, 충족 + sample_target 도달 시 significance check + decision emit
- decision emit 시 goal cycle 에 자동 ack 요청
