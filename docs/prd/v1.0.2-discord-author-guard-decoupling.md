## 1. Incident 박제

### 1.1 환경
- Workspace user yaml: `<workspace>/rosyocean/.solosquad/users/w1n.yaml`, `handle: w1n`
- Discord 사용자명: `seungw1n.` (**trailing dot** — Discord 2023+ unique username 정책 허용 charset)
- 봇 부팅 로그:
  ```
  [Discord Bot] Bound to handle=w1n org=rosyocean (channels: command-w1n / works-w1n)
  ```

### 1.2 트리거 + 봇 응답
사용자가 자신의 `command-w1n` 채널에 메시지 → 봇이 메시지 무시 + DM:
```
이 채널은 w1n의 명령 전용입니다. command-seungw1n. 채널을 사용하세요.
```
(`command-seungw1n.` 자체가 *Discord 가 허용 안 하는 채널명* — 봇의 안내 자체가 *수행 불가능한 우회 제안*)

### 1.3 Root cause — 두 charset 의 영구 불일치

[`src/bot/author-guard.ts:22`](../../src/bot/author-guard.ts#L22):
```ts
return parsed.handle === authorHandle.trim().toLowerCase();
```
- `parsed.handle` ← 채널명 `command-w1n` → `"w1n"` (charset `[a-z0-9_]`)
- `authorHandle` ← `message.author.username.toLowerCase()` → `"seungw1n."` (Discord charset 임의)

| Charset | 허용 문자 |
|---|---|
| Discord username (2023+) | `a-z`, `A-Z`, `0-9`, `_`, `.`, 일부 unicode |
| SoloSquad handle / 채널명 | `a-z`, `0-9`, `_` |

**교집합** = `a-z`, `0-9`, `_`. Discord username 이 `.` 하나라도 포함하면 *정의상 영원히 채널명과 mismatch*. 정규화 (`normalizeHandle`) 가 `.` → `_` 로 바꾸지만 비교 식은 *원본 username* 을 씀 → 어떤 정규화 우회도 불가.

### 1.4 v0.8 §3.4 가 깔고 있던 invariant 의 정직한 회상

v0.8 plan (`docs/plan/v0.8-multiuser-messenger.md`) §3.4 author-guard 의 의도:
- 1차 방어: 메신저 ACL (Discord 채널 권한)
- 2차 방어 (author-guard): 사용자가 *실수로* 다른 사람 채널에 초대받았을 때 봇이 *거기서 응답 안 하게*

암묵 가정 (당시 박제 안 됨): **"Discord 사용자명 = SoloSquad handle"**.

이 가정이 깨지는 시나리오 (보편적):
- Discord 사용자명을 본명/별명/실명으로 *길게* 쓰는 사용자 (대다수)
- SoloSquad handle 은 짧고 기억하기 쉽게 정한 사용자 (대다수)
- Discord 2023 unique username 정책: `.` 허용 → `seungw1n.` 같은 username 가능
- Discord 글로벌 display name 도입 — username 과 표시 이름 별개

→ **솔로 파운더 단일 사용 시나리오에서도 false positive 가 기본값**. v0.8 §3.4 의 정직성 promise (`defense in depth`) 가 *실제로는 false positive 생성기* 였음 박제.

## 2. 채택안 — handle = SoloSquad 의 유일 canonical user identifier

### 2.1 핵심 reframe

| 차원 | v1.0.1 까지 | v1.0.2 부터 |
|---|---|---|
| handle 의 정체 | 채널명 derivation + author-guard 비교 1축 | **SoloSquad 의 유일 canonical user identifier**. 채널명 derivation + workflow 라우팅 + memory partition 키 |
| Discord username 의 정체 | author-guard 비교 1축 | **raw 입력**일 뿐 — 비교 대상 아님 |
| Discord ACL 의 정체 | 1차 방어 (author-guard 가 2차 덧붙임) | **유일 permission boundary** — SoloSquad 가 owner 가 아니므로 위에 더 못 얹음 |
| 메시지 author identity | 비교 시 차단 | **audit log 박제만** — `events.jsonl` 및 console 로그에 `id`, `username` 박제, 게이팅 0 |

### 2.2 메시지 처리 흐름 — Before vs After

**v1.0.1 (now):**
```
Discord message arrived
  → parseChannelName(channelName) → {kind, handle}
  → 채널이 본 봇 소유인지 확인 (handle === ownHandle)
  → isAuthorizedAuthor(channelName, message.author.username)  ← gate
      ├ true: PM 호출
      └ false: bot.send(unauthorizedAuthorMessage) → return
```

**v1.0.2 (after):**
```
Discord message arrived
  → parseChannelName(channelName) → {kind, handle}
  → 채널이 본 봇 소유인지 확인 (handle === ownHandle)
  → console.log("[Discord] author: id=… username=…")  ← audit (no gate)
  → PM 호출
```

→ 결정 분기 1개 사라짐. DM 발사 경로 1개 사라짐. *seungw1n.* 류 false positive 영구 0.

### 2.3 채택하지 않은 안

| 안 | 기각 사유 |
|---|---|
| TOFU (id/username 학습 + strict 비교) — 본 plan 의 v1 draft | 학습 *전* 첫 메시지가 *handle 비교* 에 걸려 학습 못 함. lazy 라도 회귀. *online API fetch* 는 migration offline 원칙 깨짐 |
| `normalizeHandle` 우회 — username 도 같은 정규화 적용해 비교 | `.` → `_` 변환 후 비교? Discord 가 `.` 와 `_` 를 둘 다 허용해 *별개 사용자가 같은 normalized 값* 가질 수 있음 → 새로운 false positive |
| schema 에 `discord_user_id` nested 추가 + id 기반 비교 | 의미 있는 use case 0. 채널 ACL 이 이미 보장. schema bump 비용만 |
| author-guard 호출 자체는 두고 항상 true 반환 | dead code 박제. 더 honestly *제거* + audit log 가 정직 |

### 2.4 채택안 정직한 트레이드오프

| 트레이드오프 | 영향 |
|---|---|
| 봇이 *다른 사용자의* `command-<X>` 에 *잘못 초대됨* + 그 사용자가 메시지 → 봇이 `<X>` 의 PM 세션에 흘려보냄 | 1차 방어 (Discord ACL) 가 *사용자가 의도적으로 잘못 설정* 해야만 발생. audit log 로 사후 추적 가능. v0.8 §3.4 가 *방어한다고 주장* 했지만 실제 시나리오는 *owner 의 설정 실수* 만 잡았고, 그조차 1차 ACL 보강이 더 정직 |
| 사용자가 의도적으로 친구를 자기 channel-<own> 에 초대 → 친구도 메시지 처리됨 | **개선** — v1.0.1 까지는 false positive (owner 의도된 협업이 차단). v1.0.2 는 owner 의도 가정 = 정직 |
| 사용자 가시 *기능 추가* 0 | 의도된 동작 — patch 범주 |

## 3. 온보딩 wizard reorder — Step 3.5 신설

### 3.1 변경 사유

v1.0.1 까지 wizard 흐름:
```
Step 3: Configuration (name/role/messenger choice/token 입력)
saveEnv
Step 3.5: Timezone
Step 4: workspace.yaml
Step 5: 첫 organization
  Step 5.1: 리포지토리 등록
  Step 5.2: User Identification ← 토큰 4단계 후
```

**문제:** *토큰 입력 → handle 확인* 사이에 *timezone / workspace.yaml / org / repos* 4 단계가 끼어듦. 사용자는 "메신저 토큰을 막 입력했는데 왜 timezone 부터 묻지? 내 handle 은 언제?" 라는 narrative 단절.

### 3.2 v1.0.2 흐름

```
Step 3: Configuration (name/role/messenger choice/token 입력)
saveEnv
Step 3.5: Your Handle on {messenger}  ← NEW (was 5.2)
  - fetchBotIdentity(token) — messenger API 호출
  - promptHandleSelection() — guidance + 입력
  - identityChoice 변수에 stash (no yaml save yet — org 미존재)
Step 4: Timezone & Daily Briefs (was 3.5)
Step 5: workspace.yaml (silent — banner 없음, was 4)
Step 6: First Organization (was 5)
  Step 6.1: Register Repositories (was 5.1)
  saveUserYamlForChoice(stashed identityChoice)  ← silent persist
Step 7: Safety & Security (was 6)
Step 7.5: Onboarding Track (was 6.5)
Step 8: Layout (was 7)
```

**Step 5.2 banner 자체 사라짐** — handle 결심은 Step 3.5 에서 끝났고 yaml write 는 Step 6 안에 silent 흡수.

### 3.3 `registerUserIdentity` 모놀리식 → 3-phase 분리

| Phase | 함수 | 호출 시점 | UI 여부 |
|---|---|---|---|
| 1. fetch | `fetchBotIdentity(messenger, token)` | Step 3.5 | no UI — API 호출만 |
| 2. prompt | `promptHandleSelection({...})` | Step 3.5 | inquirer prompt + guidance copy |
| 3. save | `saveUserYamlForChoice({orgSlug, choice, ...})` | Step 6 (org 생성 직후) | console log only |

→ Step 3.5 가 (1)+(2), Step 6 가 (3) 호출. Phase 분리로 *org 미존재 시점에 handle 결정* 가능.

### 3.4 guidance copy — *다른 메신저 멤버와 다르게* 안내

Step 3.5 의 prompt 위 출력:

```
  Bot account on discord: w1n (auto-detected).
  Channels will be created as: command-<your-handle> / works-<your-handle>

  💡 Pick a handle that is unique in your messenger server
  - Different from other discord members' usernames or display names
    → avoids "who said this" confusion in shared channels
  - Charset: only [a-z 0-9 _]; other chars become _ automatically
  - This handle = your SoloSquad identity (workflow routing, memory partition)
```

설계 결정:
- 멤버 *목록 자동 fetch* 안 함 — init 시점에 봇이 아직 *어느 guild 에도 안 들어가 있을* 수 있고, `users:read` (Slack) / `GUILD_MEMBERS` privileged intent (Discord) 추가 필요. patch 스코프 초과
- 텍스트 가이드만 — 정직하게 "충돌 가능성" 알림 + 책임은 사용자에게 위임
- "*usernames or display names*" 둘 다 언급 — Discord global display name 시대 반영
