## 1. Summary

맥 미니 사용자는 SoloSquad 봇을 **로컬**에, 그 외 사용자는 **클라우드**에 띄워 메신저로 봇과 대화하며 원격 24/7 작업을 한다. 기본 클라우드를 (이전 초안의 GCP에서) **Railway**로 정한다 — 서버 관리 없이 GitHub 연동/템플릿으로 가장 빠르게 띄울 수 있기 때문. 본 patch는 ①배포 자산에 남은 죽은 명령(`solosquad schedule`, v1.3.3 폐기) 정리 ②**Railway를 cloud-deployment 1순위로 승격**(원클릭 템플릿 + 요금 투명성 + 비용 절감 가이드) ③**클라우드에서 봇 세션이 죽으면 별도 watchdog가 메신저로 통지**하는 기능을 더한다. **단, 참고 스크립트(OpenClaw)의 "터미널 없이 8분 원클릭"은 SoloSquad에 그대로 적용 불가** — 현재 `solosquad init`이 인터랙티브이고 메신저가 Discord 전용이라(§2.1), 그 간극을 어디까지 1.3.12에서 좁힐지를 §6에서 결정한다. 원격 재시작은 비-우선(후속).

## 2. Background

- **JTBD:** "맥을 끄거나 자리를 비워도 클라우드의 SoloSquad 봇과 메신저로 24/7 일하고 싶다. 띄우는 게 *쉬웠으면*, 요금이 *예측 가능*했으면, 그리고 봇이 *조용히 죽으면 메신저로 바로* 알려줬으면."

### 2.1 참고 스크립트(OpenClaw-on-Railway) ↔ SoloSquad 정합 (R3 — 요구사항·간극 전사)

스크립트가 보여준 매력과 SoloSquad 현재 상태의 차이를 **그대로 전사**해 오해를 차단한다:

| 스크립트의 요소(OpenClaw) | SoloSquad 현재 | 1.3.12 함의 |
|---|---|---|
| **원클릭 "Deploy on Railway" 버튼**(GitHub 템플릿) | `railway.toml`/Procfile **없음** → 템플릿 미존재 | 템플릿 설정 신설 가능(코드 소량). 단 아래 온보딩이 병목 |
| **터미널 없이** 웹 setup UI(username/pw)에서 토큰·모델 입력 → 8분 | `solosquad init`이 **전면 인터랙티브**(inquirer), 비대화형/env-var 모드 **없음** | **핵심 병목.** 진짜 원클릭엔 비대화형 init 또는 `railway ssh` 쉘 경유 필요 → [결정 2] |
| **Telegram + BotFather + 페어링 코드** | 메신저 **Discord 전용**(`src/messenger/discord-*`), Telegram/Slack 어댑터 없음 | Telegram 흐름 **그대로 불가**. Discord invite-URL 온보딩 사용 |
| **Serverless sleep**(무요청 시 잠자고 메시지로 깨움) = 최대 비용 절감 | Discord는 **상시 gateway WebSocket** — 자면 인바운드 수신 불가 → **sleep-wake 성립 안 함** | SoloSquad on Discord는 **항상 켜둬야** 함 → 그 RAM 비용이 바닥값. sleep 절감은 webhook 메신저(미지원) 전제 |
| **GLM-4.7/5**(최저가 모델) | **Claude 전용**(claude-code 래핑) | 모델 선택지 N/A |
| **메모리 cap** `--max-old-space-size=1024` (env) | Node 프로세스 동일 적용 가능 | **재사용 O** — 비용 절감 가이드에 채택 |
| **$5 선불 + 사용량 과금**(bill shock) | 동일(Railway 공통) | **재사용 O** — 요금 투명성 절 신설 |

- **결론:** 스크립트에서 **재사용**할 것 = (원클릭 템플릿 설정 · 요금 투명성 · 메모리 cap · 비용 경고). **SoloSquad에 안 맞는** 것 = (Telegram/BotFather · serverless sleep · GLM · 웹 setup UI). 후자를 모방하려면 별도 大작업(messenger 추상화·비대화형 온보딩)이 필요 → 1.3.12 범위 밖(§스코프).

### 2.2 기존 자산의 결함

- `manual/cloud-deployment.md`는 Railway를 **Option B(얇음)**로만 두고 1순위는 VPS. 그리고 Railway start command가 `npm run build && node dist/... schedule` — **죽은 `schedule`**.
- `docker/docker-compose.yml` scheduler가 `command: ["solosquad","schedule"]` → 부팅 즉시 `unknown command 'schedule'`.
- health: PID 파일·graceful drain·`Restart=always`는 **재기동**만, **"죽음/부활"을 메신저로 통지 안 함**(post 주체가 죽음). `events.ts`의 in-process 이벤트는 봇 생존 중 신호일 뿐.

### 6.2 조각 A-2 — Railway를 기본 클라우드로 승격

cloud-deployment.md 재편: **Railway = Option 1(추천)**, VPS/Fly/ECS는 후순위 유지. 추가 산출:

- **요금 투명성 절(신설, 스크립트 재사용):** Railway = **$5/월 선불(Hobby) 최소 + 사용량(RAM·네트워크·스토리지·CPU) 추가 과금**. CPU는 사용 시에만, 실측상 **메모리가 주 비용**. bill-shock 경고 + 대시보드에서 사용량 확인 안내. SoloSquad on Discord는 **항상-켜둠**이라 always-on RAM이 바닥값(예상 월 ~$8–20, 워크로드 의존).
- **비용 절감 가이드(스크립트 재사용 + SoloSquad 보정):**
  - env `NODE_OPTIONS=--max-old-space-size=1024` 로 힙 상한(메모리 과금 억제).
  - **Serverless/Sleep는 권장하지 않음(Discord)** — 이유 명시(gateway WS는 sleep 시 메시지 수신·웨이크 불가; Railway sleep은 *인바운드 HTTP 부재 ~10분*에 트리거되는데 Discord 봇엔 인바운드가 없어 깨울 통로 자체가 없음). 항상-켜둠 + RAM cap이 1.3.12 권고.
  - **(후속 카드 — 비용이 실제로 아플 때) sleep을 살리는 2경로, 둘 다 1.3.12 밖:**
    - **a안(슬래시-wake):** Discord Interactions Endpoint(HTTP)로 `/sq <내용>` 슬래시 커맨드를 wake 진입점 삼아 인바운드 POST로 컨테이너 기동 → gateway 연결 후 자연어 대화. 제약: 첫 wake는 콜드부팅이 **3초 ack 데드라인** 초과로 실패 가능 · 자는 중 보낸 평범한 메시지는 유실.
    - **b안(도어맨/워커 분리):** 경량 항상-켜둠 프로세스가 gateway WS만 들고(싼 RAM), 무거운 에이전트 실행 워커는 sleep → 메시지 시 깨움. 자연어 UX 무손실. ≈ cloud-deployment Option E(Agent SDK+Sandbox 분리).
- **원클릭 지향 템플릿 [결정 2]:** 두 경로 §6.2.1 중 택1.

#### 6.2.1 원클릭 온보딩 간극 — ≥2 approaches (gstack rule)

스크립트의 "터미널 없이 8분"은 **비대화형 온보딩**이 전제. SoloSquad `init`은 인터랙티브 → 메우는 방식 2안:

| | Approach A — `railway.toml` 템플릿 + 배포 후 `railway ssh` → `solosquad init`(**권고, patch 범위**) | Approach B — 비대화형 env-var `solosquad init --yes`(완전 원클릭) |
|---|---|---|
| 흐름 | "Deploy on Railway" 버튼 → 컨테이너 기동 → 사용자가 **`railway ssh`로 컨테이너 안에 접속**해 `init` 1회(대화형 그대로) → env에 토큰. **`railway ssh`인 이유:** init이 만드는 워크스페이스(`agents/`·`.solosquad/`)가 **컨테이너 영구 볼륨**에 생겨야 재기동에도 유지됨. (`railway run`은 *내 노트북에서* 실행 → 파일이 로컬에 생겨 부적합) | 템플릿 + env-var(DISCORD_TOKEN·OWNER_*·핸들)만으로 `init` 자동 완료, 쉘 불필요 |
| 작업량 | **소** — 템플릿 파일 + 문서. init 코드 무변경 | **대** — `init` 전체를 env-var 주입·검증·idempotent로 재설계(인터랙티브 분기 다수) |
| 스크립트 재현도 | 중(원클릭 후 1스텝 남음) | 고(진짜 원클릭) |
| patch 적합성 | ✅ | ⚠️ 사실상 minor 기능 |
| 리스크 | 사용자에 `railway ssh` 1스텝 요구 | init 회귀면 모든 신규 온보딩 깨짐 |

- **권고: Approach A.** patch 범위에 맞고 init 코드 리스크 0. **B(완전 원클릭 = 비대화형 init)는 별도 minor로 분리** 권고(messenger 추상화·웹 setup UI와 함께 묶는 게 자연스러움). 반증: Approach A의 "`railway ssh` 1스텝"이 비-개발 사용자에게도 충분히 쉬우면 B 불요; 그 1스텝조차 마찰이 크다는 dogfood 근거가 나오면 B로 승격.

### 6.3 조각 B — 봇 세션 health → 메신저 알림 (≥2 approaches)

**제약:** 죽은 봇은 스스로 못 알림 → 알림 주체 = 봇과 **다른 failure domain의 살아있는 프로세스**. compose/Railway는 bot·cron을 별도 서비스로 분리 → 활용. (Discord는 항상-켜둠이라 §2.1 sleep과의 충돌 없음 — heartbeat 정지 = 진짜 사망.)

| | Approach A — Heartbeat + watchdog(cron 데몬 내장) + boot/drain announce **(권고)** | Approach B — Railway/외부 uptime 모니터 | Approach C — process-manager OnFailure 훅 |
|---|---|---|---|
| 메커니즘 | 봇이 `<ws>/.solosquad/bot.heartbeat`에 **턴-독립 타이머**로 timestamp 기록. cron 데몬(`cron start`, 상시·별도)이 staleness>임계 시 메신저로 `🔴 봇 무응답(heartbeat N s stale)` post. 부팅 `🟢 online`, drain `🟡 draining` best-effort | 봇 `/healthz` HTTP → Railway healthcheck/외부 uptime → 알림 채널 | systemd `OnFailure=`(VPS) / Docker healthcheck+사이드카 |
| 하드킬 포착 | ✅ 독립 프로세스 | ✅ 외부 | ⚠️ |
| *대화하는 메신저*에 도달 | ✅ | ❌ 보통 email/webhook | ✅ |
| 신규 ingress/보안면 | ❌ 없음 | ⚠️ HTTP 공개 | ❌ |
| 포터블(Railway/Docker/VPS) | ✅ | ⚠️ 플랫폼별 | ⚠️ systemd 종속 |
| 서비스 전체 다운 포착 | ❌(둘 다 죽음) | ✅ | ❌ |

- **권고: Approach A.** 메신저 직접 도달·포터블·신규 보안면 0·상시 cron 데몬 재사용. **B는 보완 레이어로 문서화만**(서비스 전체 다운은 Railway 자체 알림/외부 uptime 2차 — 구현 비대상).
- **반증 조건:** watchdog가 **정상 장시간 Chief 턴**을 사망 오인 → false 알림. **가드:** heartbeat는 턴 루프와 **독립 setInterval**(턴이 길어도 계속 기록), 임계=주기×3+여유(주기 30s→임계 120s). false-positive가 튜닝으로 0 수렴 안 하면 A 기각 → C로 후퇴.
- **[결정 3]** watchdog 호스트: cron 데몬 내장 vs 전용 `solosquad watchdog`. **bot-only 배포(cron 미기동)는 내장이면 watchdog 부재** → "클라우드는 bot+cron 동반 기동" 가이드 박제 또는 전용 프로세스. (Railway 템플릿이 두 서비스 기본 포함하면 자연 해소.)

| **Usable** | watchdog 오경보 0(장시간 턴 ≠ 사망) · `railway ssh` 후 `init` 1스텝이 충분히 쉬움 | 회귀 케이스 · dogfood |
| **Viable** | cron 데몬 재사용(신규 인프라 0) · 템플릿은 설정 파일 | 코드 리뷰 |
| **Feasible** | heartbeat=기존 `isProcessAlive`/PID 패턴 동형 · 템플릿=`railway.toml` | 스파이크 |

### 6.5 Hard gate

- watchdog **정상 장시간 턴 오경보 0**(회귀 박제) — 미달 시 ship 금지.
- Railway/Docker 기동 후 **bot·cron 두 서비스 모두 healthy**(scheduler 더는 안 죽음).
- 요금 절 수치(선불 $5·사용량·메모리 주도)·비용 절감 env가 실 Railway 1회 검증.

## 8. Metrics & Verification

- **Success threshold:** ①신규 Railway 배포 소요 ≤ **15분**(템플릿+`railway ssh` 후 `init`, dogfood 실측) ②**MTTA**(봇 사망→메신저 알림) ≤ **120s** ③watchdog **false-down = 0**(장시간 턴 포함 4주) ④요금 절 수치가 실 청구서와 일치(검증 1회).
- **Leading:** 강제 `kill -9` 후 알림 실측 · false-down 카운트 · dogfood 배포 타임.
- **Lagging:** "봇 죽은 줄 몰랐다"·"요금 폭탄" 리포트 0 수렴.
- **수용 기준 (R8 — Given-When-Then):**
  ```
  Given  Railway에 bot + cron 두 서비스가 떠 있고 watchdog 활성
  When   bot 프로세스가 SIGKILL(하드킬)로 사망
  Then   ≤120s 내 command/status 채널에 "🔴 봇 무응답(heartbeat N s stale)" 1회 post

  Given  bot이 정상적으로 5분 이상 걸리는 Chief 턴을 처리 중
  When   그 5분 동안
  Then   watchdog는 사망 알림을 보내지 않는다(heartbeat 독립 타이머가 계속 기록)

  Given  사용자가 "Deploy on Railway"로 컨테이너를 띄우고 railway ssh로 접속해 solosquad init 완료
  When   docker/Railway 스택이 기동
  Then   bot·cron 두 서비스가 running이고 cron 서비스가 'unknown command schedule'로 죽지 않는다
  ```

> **R6 분기 — "변경 surface에 LLM/에이전트가 들어가나?" → No.** 본 patch surface(railway 템플릿·compose·Dockerfile·배포/요금 문서·heartbeat·staleness 폴링·announce)는 전부 **결정론적 ops·infra**. 봇이 감싸는 Claude 에이전트의 동작·프롬프트·허용 답변 범위는 미변경 → §8 AI 부록 4칸 **비활성**(R6 No 경로).
