## 1. 배경 / 문제

- 솔로스쿼드가 founder 에게 주는 *가시적* 가치 = 코딩 에이전트 플랜 모드를 고도화한
  **잘 쓴 PRD**. 기획 워크플로의 결과를 담고, 버전으로 관리되며, 아이데이션·리포트와
  구별되어 개발이 그 PRD 에서 출발하는 스펙.
- 그런데 **문서 관리 권위가 없다.** `prd` 스킬은 *한 PRD 를 어떻게 쓰나*(8-section)만
  갖췄고, *어디에·어떤 버전으로·언제 최신화·외부냐 내부냐*의 큐레이션 층이 없다.
- **솔로스쿼드 자기 docs 실태:** `docs/prd/`(52)에 버전 PRD + `architecture.md` +
  `product-roadmap.md` 혼재 · `docs/ideation/`(19)에 리포트 혼재 · `docs/reports/`
  부재(기존 `prd` 스킬은 이미 그 경로를 전제) · 명명 불일치(`v0.1-…` 2자리,
  `V0.3-INTEGRATION-TEST-PLAN` 대문자).
- **외부/내부 경계가 암묵적:** `package.json.files`(=`docs/` 제외, `manual/`·
  `CHANGELOG`·README 포함)로 사실상 강제되지만, 어떤 스킬도 이를 명문화·검사하지 않음.
- **PRD shape 혼동(본 PRD 가 자초한 실증):** 내부 dev PRD 와 founder 제품용 PRD 가
  같은 "PRD" 라는 이름을 공유해 혼선. 관리 체계가 이를 분리해야 한다.

## 2. 핵심 개념

1. **두 직교 축.** ⑴ **외부(사용자 대면) vs 내부(기획·개발 전용)** — 경계는
   `package.json.files` 로 강제(load-bearing). 내부 문서는 npm 으로 안 새고, 외부 문서는
   배포 게이트로 최신화. ⑵ **스코프 = repo 계층 vs org 워크스페이스 계층** — **docs·버전의
   단위는 repository**: 각 repo 가 자기 `package.json`(version)·`docs/`·CHANGELOG·manual 을
   갖고 **독립적으로** x/y/z 를 올린다(또는 안 올린다). release-bound 문서(prd·architecture·
   roadmap·README·CHANGELOG·manual)는 **repo 계층**, 결정 전 발산·근거(ideation·reports)는
   **org 워크스페이스 계층**(cross-repo). **단일 repo 프로젝트(SoloSquad 자신)는 두 계층이
   같은 `docs/` 로 접힌다.** PM 은 각 repo 를 기준으로 문서를 사고한다.
2. **PRD ↔ release 버전 1:1 (per-repo).** PRD 가 "이 *repo* 의 이 버전에 무엇이 들어가나"의
   single source. `<repo>/package.json.version = vN.N.N → git tag → publish`. **작업했다고
   반드시 버전을 올리는 건 아니다** — 한 워크플로우가 여러 repo 를 건드려도 *release 하는 repo
   만* 버전·PRD 가 따른다(작업만 하고 배포 스킵 가능). PRD 자체는 외부로 안 나가고
   manual·CHANGELOG 가 외부 노출을 대행.
3. **ideation ≠ report ≠ prd.** ideation="왜/만약"(발산·결정 전·폐기 안 함) ·
   report="무엇이 사실인가"(근거 스냅샷·PRD 가 `evidence_ref` 인용) · prd="무엇을 왜
   어떻게"(수렴·개발계획+요구사항·버전 결합·ship 후 frozen).
4. **PRD shape 는 맥락별로 다르다.** ⑴ 솔로스쿼드 내부 dev PRD(본 양식) ⑵ founder
   제품 기획 PRD(`prd` 스킬 8-section + new-build/improvement 누적 2양식). `docs`
   스킬이 어느 맥락에 어느 shape 인지 명문화한다.
5. **명명 규칙: `<version>_<name>_<date>`** — 세 토큰을 `_` 로, 토큰 내부 띄어쓰기는
   `-` 로. 존재하는 토큰만 사용. 날짜는 `YYMMDD` 압축형(하이픈 혼동 방지).

### 3.2 문서 분류 체계 — 외부/내부

| 축 | 타입 | 위치 | npm | 청자 | 기획모드 |
|---|---|---|---|---|---|
| 🌐 외부 | readme | `README.md` / `README.kr.md` | ✅(자동) | 잠재 사용자·기여자 | 진입점 |
| | changelog | `CHANGELOG.md` | ✅ files | 사용자 | 릴리즈 이력 |
| | manual † | `manual/master-guide_{ko,en}.html`, `manual/*.md` | ✅ files | **제품 사용자** | 사용 안내 |
| 🔒 내부 | prd | `docs/prd/v<N.N.N>_<name>.md` | ✗ | 기획·개발(본인) | 수렴·스펙 |
| | ideation | `docs/ideation/<name>_<YYMMDD>.md` | ✗ | 기획 | 발산 |
| | reports | `docs/reports/<name>_<YYMMDD>.md` ⚠️신설 | ✗ | 기획(근거) | 근거 |
| | poc | `docs/poc/<name>.{mjs,md}` | ✗ | 개발 | 실증(코드) |
| | policy | `docs/policy/<topic>.md` | ✗ | 개발 | 불변 계약 |
| | architecture | `docs/architecture.md` ⚠️승격 | ✗ | 개발 | 구조 단일진실(living) |
| | roadmap | `docs/roadmap.md` ⚠️승격 | ✗* | 기획(본인) | 비전(living) |

\* roadmap 외부 공개 제품이면 `package.json.files` 옵트인.
† manual 은 **제품 특성 조건부** — 사용자가 코드/문서를 봐야 하는 제품(agent 시스템·OSS·CLI;
SoloSquad 해당)만 작성. 일반 웹/모바일 앱은 생략 가능(게이트도 manual 부재 시 스킵).

**스코프 계층 (핵심개념 §2-⑵):** 위 위치의 `docs/`·루트는 **repo-relative**. 타입을 두 계층으로 가른다 —

| 계층 | 타입 | 단위 | 비고 |
|---|---|---|---|
| **repo** `<repo>/docs/`·루트 | prd · architecture · roadmap · poc · policy · README · CHANGELOG · manual | repository (자기 `package.json`·git) | release-bound, 버전 결합, 게이트 대상 |
| **org** `<org>/docs/` (워크스페이스) | ideation · reports | org (cross-repo) | 결정 전 발산·근거, 워크스페이스라 class A 무관·쓰기 가능 |

단일 repo 프로젝트(SoloSquad 자신)는 두 계층이 같은 `docs/` 로 접힘 — 그래서 본 repo 의
`docs/ideation/`·`docs/reports/` 가 같은 트리에 공존한다(일반 founder 의 multi-repo 에선 org 계층).

> 설계 근거(approaches 비교·가정 분류·open questions)는 §6–§7 에 분리 수록 —
> 본 §3 은 *무엇을* 의 스코프, §6 은 *왜 이 안인가*의 결정 근거.

| ideation | `name_date` | `docs-management_260624.md` |

- 버전 항상 3자리 `vN.N.N`. 발행분 비표준(`v0.1-…`·`V0.3-…`)은 immutable —
  **fix-forward**(신규만 규칙 준수, 과거 리네임 안 함).

### 3.4 PRD ↔ 배포 버전 결합

PRD 1개 = **한 repo 의** release 버전 1개. `docs` 스킬이 PRD 생성 시 버전 토큰을 검증하고,
**그 repo 의** `package.json.version`·git tag 와의 1:1 을 게이트(§3.5)에서 확인. 작업만 하고
release 안 하는 repo 는 버전·PRD 강제 없음(핵심개념 §2-2).

**핫픽스 예외 (OQ#1 resolved, 2026-06-25):** 핫픽스(예: 1.3.8.1)는 **별도 PRD 를 만들지
않고 부모 `vN.N.N` PRD 에 `## Hotfix` 섹션을 누적**한다. 정합 1:1(PRD↔minor/patch) 유지 ·
파일 폭발 없음 · 게이트는 핫픽스 버전을 부모 PRD 의 hotfix 멘션으로 통과. 복수 기능 동시
ship 은 한 버전 = 한 PRD 원칙대로 PRD 본문에서 요구사항을 나눠 담는다(버전 쪼개기 아님).

### 3.5 배포 전 문서 최신화 게이트 (4종 → 6종)

`scripts/check-docs-freshness.ts` 확장:
```
1. 해당 vN.N.N PRD frozen 존재 (repo 계층)
2~. roadmap·architecture·CHANGELOG·README 가 vN.N.N 멘션 (존재하는 것만)
   + manual(ko·en) 은 **존재 시에만** vN.N.N 멘션 검사(부재 = 제품 특성상 생략 = 스킵)
N. 불변식: package.json.files 에 docs/ 미포함(내부 누출 방지)
```
`prepublishOnly` 에 wired 유지. 메모리의 3-docs/4-docs 게이트 규칙과 정렬. 게이트는 "필수 코어
(roadmap·architecture·CHANGELOG·README) + 조건부(manual)" — manual 없는 repo 도 통과.

### 3.7 PRD 작성 shape 보강 — `prd` 스킬 + 기획 워크플로 (Q6·Q7·Q8)

`docs` 가 *어디에·어떤 버전으로* 라면, 한 PRD *를 어떻게 쓰나*의 shape 도 함께 확정한다
(맥락 상실·오버스펙 방지). `prd` 스킬에 **8규칙(R1–R8)** 명문화 + 기획 워크플로 리뷰 게이트 강화.

**(A) 맥락·오버스펙 축 (R1–R5):**

- **R1 버전 계층 인지** — `vX.Y.Z` 자리수별 정의 범위(X=umbrella, Z=상위 인용). PRD 상단 소속목표 블록.
- **R2 §스코프 = 포함 + 이번-버전 비포함** — 경계를 스코프에서 직접(영구 §비범위와 구분).
- **R3 지시 요구사항 전사 + 결정 표시** — 요구사항→충족 추적, `[결정 필요]` 인라인.
- **R4 As-Is/To-Be = 개념 + 형태 이중** — 표/다이어그램, 의미축 + 경로/구조 변화 둘 다.
- **R5 작업 체크리스트 = 워크플로 리뷰 대상** — `requirements-analysis` §리뷰에서 담당
  스페셜리스트가 실행가능성·**오버스펙**·누락 검토해 가감·확정(리뷰 전 non-binding).

**(B) 확률적 산출물·검증 축 (R6–R8, Q8 — ideation `260625-ai-planning-insights.md` 반영):**

21개 외부 'AI 기획' 소스 종합(`evidence_ref`)에서 도출. 외부 담론 대비 우리 PRD 양식의 최대
공백 = **"AI 제품을 위한 PRD 칸이 없다"**. R1–R5 가 *맥락* 축이면 R6–R8 은 *확률성/검증* 축이다.

- **R6 AI 제품 PRD 분기 (결정론/확률론 = 세 번째 축).** PRD 생성 시 `prd` 스킬이 "이 제품에
  LLM/에이전트가 들어가나?"를 묻는다. **Yes → §8(§검증)에 AI 부록 4칸:** ① **허용 답변 범위**
  (range of acceptable answers — "무엇이 일어나야"가 아니라 "어떤 답이 받아들일 만한가")
  ② **Eval Plan**(입력·기대출력·실제출력·합격여부 셋 20~30 케이스, 회귀셋) ③ **실패 정의 +
  가드레일**(Always/Ask/Never 경계) ④ **3단계 평가 피라미드**(규칙기반 → LLM-as-judge → 사람).
  No → 기존 결정론 §검증 그대로. 근거: 외부 4소스(원티드랩 PO·AI PRD·Evals=Moat·AI PM 로드맵)가
  "AI PRD = 허용 범위 + Eval" 로 수렴. ※ 외부/내부(1차)·결정론/확률론(세 번째)은 직교 축.
- **R7 PRD 완성도 단계 척도 (binary → 단계).** HARD GATE 의 이진 충족과 별개로 PRD *성숙도*를
  표기: **30**(가치정의·퍼소나·메인시나리오 = 리뷰 시작 가능) → **50**(+배경·중요 유즈케이스) →
  **100**(ship 직전) → ship 후 **70 리셋**(운영 갱신 여지). `docs` 스킬의 PRD frozen 판정(§3.4)과
  연결 — frozen = 100 도달.
- **R8 수용 기준 = Given-When-Then AC.** §검증의 수용 기준을 **Given-When-Then** 으로 써 테스트로
  직역 가능하게. PRD(무엇을)→**AC(언제 완성인가)**→Policy(어떤 기준 일관성) 문서 위계를 인지.
  (예: `Given 비회원 / When 이메일 가입 / Then 인증 메일 발송`)

**(C) 어휘·프레임 정렬 (저비용 흡수):**
- **Spec Kit 4게이트**(Specify→Plan→Tasks→Implement)로 `requirements-analysis` 단계 어휘 정렬(이미 사실상 동형).
- **가치문장 4요소** — §가치제안 1-line 을 "누가·왜·어떻게·어떤 가치" 구조로.
- **산문 우선** — PRD 는 텍스트 서술형 우선("어설픈 그림 금지"), 다이어그램은 보조.
- **지시의 저주 회피** — R5 체크리스트는 다운스트림에 *1건씩* dispatch(한 번에 한 태스크).

- 손댈: `skills/prd/SKILL.md`(R1–R8 + HARD GATE + AI 분기) · `requirements-analysis/workflow.yaml`(리뷰 가드).

## 6. 설계 결정 (approaches 비교)

> PM 컨벤션 `minimum_approaches: 2` — 핵심 결정마다 ≥2 안 비교 + 권고 + 반증조건.
> 채택안은 §3 스코프가 구현하는 것. 사후 명명(post-labeling): 안을 먼저 처방하지
> 않고 후보를 나열한 뒤 명명.

**D1. 문서 관리 권위를 어디에 두나** — 채택: A
- A) 신규 `docs` 스킬(단일 권위 매니저) · B) 기존 `prd` 스킬 확장 · C) `docs/policy/`
  정책 문서 + 게이트 스크립트만(스킬 無).
- 권고 A. rationale: `prd`=per-PRD writer 와 큐레이션=cross-doc 권위는 관심사가
  다름(SRP) — 한 스킬에 합치면 두 양식(내부 dev/founder 제품) 혼동이 더 악화.
  C 는 "어디 배치·언제 최신화" 의 대화형 판단을 담을 주체가 없어 게이트가 *사후 거부*만
  하고 *사전 안내*를 못 함. 반증조건: 큐레이션이 거의 결정적(분기 없음)으로 밝혀지면
  대화형 스킬은 과설계 → B/C 로 회귀.

**D2. architecture/roadmap 의 위치** — 채택: A
- A) `docs/` 직속 living 문서로 승격 · B) `docs/prd/` 유지(현행).
- 권고 A. rationale: `docs/prd/` 의 의미축은 "버전 PRD = ship 후 frozen" 인데
  architecture/roadmap 은 계속 갱신되는 living 문서 → 생애주기 충돌·discovery 오염.
  반증조건: 참조 6곳 수정·INDEX 분리 비용이 분류 명확성 이득을 초과하면 B 유지.

**D3. 발행분 비표준 명명 처리** — 채택: A
- A) fix-forward(신규만 규칙 준수, 과거 유지) · B) 일괄 리네임.
- 권고 A. rationale: 발행분 파일명은 git 히스토리·외부 링크에서 immutable 취급 —
  리네임은 참조 깨짐 위험 대비 이득 미미. 반증조건: 자동 INDEX 생성이 명명 불일치로
  실패하면 깨지는 파일만 부분 리네임.

**D4. report 의 거처** — 채택: A
- A) `docs/reports/` 신설·집결 · B) `ideation/`·`trend-record/` 산재 유지.
- 권고 A. rationale: PRD 의 `evidence_ref` 인용 대상이 한 곳으로 모여 근거 추적이
  결정적. 반증조건: 리포트 수가 극소수면 신설 폴더가 빈 디렉토리 오버헤드.
