## 0. 동기

SoloSquad 의 Chief 는 점점 **장시간·다중 repo** 작업을 받는다. ideation 문서가
정리한 원칙(상주 위치 · 출장/소거 · 교대/계층 메모리 · 체크포인트)과 실제
레퍼런스(§5: Codex 90% auto-compact + handoff summary + repo 내 durable-md;
Qwen QWEN.md + 70% 압축 + RAG 분해)에 비추어 **현재 우리 구조의 공백**을 메운다.

ideation §5.4 결론: **우리는 (a)상주·(d)복구는 정석에 가깝고, 가장 큰 공백은
⑴임계 기반 자동 교대(compaction)와 ⑵학습형 cross-session 메모리, 그리고
⑶다중 repo 세션 격리다.**

> hydrate + flush)" 모델과 다르다 — 우리는 **출장을 안 가고 모든 짐을 org 통제실로
> 끌어온다.**

## 2. 공백 분석

1. **컨텍스트 블로트.** 다중 repo 작업이 한 세션에 누적 → 8-layer drop 으로 방어하나,
   장기 턴에서 세션 자체가 비대해짐. 토큰-임계 자동 교대 부재(§1).
2. **per-repo 격리·flush 없음.** repo A 작업 로그가 같은 세션에 남아 repo B 턴까지
   오염. ideation §3②③ 의 "출장 후 머릿속 비우기(flush)"가 구조적으로 불가능.
3. **specialist cwd 미제어.** in-process Task 라 봇이 "repo 에 들어가 일하고 나오는"
   경계를 못 만든다. 위치 제어는 프롬프트 텍스트(절대경로)에 의존.
4. **학습형 cross-session 메모리 약함.** `pm-compaction`(완료 워크플로 외부화)은
   있으나 Codex Memories 류 "선호·패턴 자동 누적/병합"은 없음.
5. **사문화 자원.** `resolveOrgCwd` 가 외부경로 repo 를 이해 못 해 repo-cwd 기능이
   죽어 있음 → per-repo 실행의 기초 블록이 깨진 상태.

## 3. 세션 분할 모델 (설계 옵션)

| 모델 | 설명 | 세션 토폴로지 | 장점 | 단점 | 채택 |
|---|---|---|---|---|---|
| **M0 (현행)** | 단일 Chief 세션, org-cwd, --add-dir, in-process Task | (user,org)×1 | 단순, 단일 기억 | 블로트·격리불가·flush불가 | (기준선) |
| **M1 (Codex 패턴 · 단일세션 강화)** | M0 유지 + ⑴토큰-임계 **자동 교대(handover)** ⑵워크플로별 **durable-md 외부화** ⑶메모리 계층 공식화 | (user,org)×1 | 프로세스 모델 무변경, 저위험 | per-repo 격리는 여전히 없음 | ✅ **v1.4.0 (S-1~S-3)** |
| **M2 (per-repo 디태치드 워커)** | repo 중작업을 **봇이 별도 claude 프로세스로 스폰**(cwd=repo, 자체 session-id, hydrate→flush→1줄 요약 반환) — ideation "출장" 모델 | (user,org)×1 + (user,org,repo)×N | 진짜 격리·flush, 위치 제어 | **새 스폰 경로 신설**(in-process Task 우회), 복잡 | ◐ **v1.4.x (S-4)** |
| **M3 (하이브리드 · 권고 종착지)** | Chief 는 org-root 오케스트레이터(짧은 턴)로 남고, 장기·다중 repo 실행만 **M2 워커 세션**에 위임. 워커는 자체 교대+복구, 요약만 Chief 메모리로 환류 | M2 + 명시적 역할 분리 | 오케스트레이션/실행 분리, 각자 컨텍스트 위생 | 가장 큰 변경 | ◐ **v1.4.x (S-5)** |

**권고 경로:** **v1.4.0 = M1 단계(S-1~S-3)** 먼저 깔고 → **v1.4.x = M2/M3 per-repo
워커 세션**. M1 이 M2 의 토대(교대·복구·durable-md·메모리 계층)를 먼저 깔아 두므로
점진 이행이 매끄럽다.

## 4. 실행 위치·제어 매트릭스 (목표 상태 M3)

| 세션 유형 | cwd | session-id 위치 | 누가 제어 | 생명주기 |
|---|---|---|---|---|
| **Chief (오케스트레이터)** | `<workspace>/<org>/` (org root) | `sessions/<user>.json` | 봇(`chief-runner`) | 장수(long-lived), `chief reset` 으로 회전 |
| **Repo 워커 (M2)** | **repo 실제 경로**(`<slug>.yaml.path`, getRepoDir 해석) | `sessions/<user>__<repo>.json` (신규 서브세션) | 봇(신규 워커 러너) | 작업 단위 — 스폰→hydrate→flush→kill, 요약만 잔존 |
| **Goal 백그라운드 (v0.4)** | org root | `bg-<goal-id>-<run-ts>` | `engine/goal-runner`(immutable) | goal 사이클 |
| **specialist (in-process Task)** | 부모 cwd 상속 | (별도 없음 — 부모 세션 내) | Chief(네이티브 Task) | 단발 |

> **제어 핵심:** M2 워커는 **봇이 직접 스폰/킬/교대**(goal-runner 패턴 일반화)하므로
> Chief 의 in-process Task 와 달리 **위치(cwd=repo)와 경계(hydrate/flush)를 봇이
> 강제**할 수 있다. Chief 는 워커를 "출장 보내는" 디스패치 마커만 발행, 봇이
> mention-parser/slash 처럼 **Chief 디스패치 전에 가로채** 워커를 띄운다.

### 5.7 spawn-영향 릴리즈의 Chief 세션 자동 리셋 (v1.3.10/1.3.11 회고에서 승격) — S-3

**문제(실증):** v1.3.10/v1.3.11 이 봇 spawn(입력경로·`--add-dir`)을 고쳤지만, 마이그레이션이
세션을 안 건드려 사용자는 **옛 Chief 세션을 resume** 했다. spawn 인자는 매 턴 재생성돼 *기능상*
fix 는 적용되나, 옛 세션 transcript 에 "접근 불가 / 허용 누르세요 / 차단" 턴이 누적돼 에이전트가
그 맥락에 휘둘린다(클린 슬레이트 부재). 과거에도 `--add-dir` 류 변경마다 세션을 수동/임시로
아카이브해 왔다(세션 저장소의 `post-v1.2.8-add-dir-fix` 흔적).

**원칙:** **봇 spawn(입력경로·권한 모델·tool 가용성·system-prompt 구조)을 바꾸는 릴리즈는 그
마이그레이션에서 Chief 세션을 아카이브(rotate)** 한다 — 옛 세션 history 가 *오도(misleading)* 가
되는 변경이면 클린 세션을 강제. 단순 버전 범프(데이터·spawn 무변경)는 리셋하지 않음(연속성 유지).

- **메커니즘(기존 재사용):** `session-store.rotate(orgSlug, userId, reason)` — 마이그레이션이 각
  org 의 활성 Chief 세션 레코드를 순회하며 `reason: "post-vX.Y.Z-spawn-change"` 로 rotate. 다음 봇
  턴은 fresh 세션 생성(현 `session-not-found→rotate` 경로와 동형). 데이터·워크스페이스 무변경.
- **분류 게이트:** 마이그레이션 작성 시 "이 릴리즈가 spawn 행동을 바꾸나?" 체크 → Yes 면 세션 rotate
  포함, No 면 no-op 범프. (1.3.10/1.3.11 은 Yes 였어야 함 — 회고 부채.)
- **왜 S-3(v1.4.0):** rotate 인프라가 이미 있고(§5.1), 마이그레이션 1줄 헬퍼라 저위험. 리스크
  #2(rotate 전 백업/검증)와 동일 fail-safe 적용.
- **연계:** [[feedback_npm_release_prep]] 배포 준비 체크리스트에 "spawn 변경 릴리즈면 마이그레이션
  세션 rotate 포함" 항목 추가.

## 10. 다중 repo 실행 컨텍스트 일반화 + 자산 통합 (S-6 · S-7)

> 근거: `docs/ideation/260621-multi-repo-execution.md`(멀티 repo 실행 모델 + 업계 레퍼런스
> 전수: monorepo/polyrepo · `--add-dir` · Gerrit topics · GitHub gh-aw 오케스트레이터+워커 ·
> Squad "드롭박스" · cross-repo-orchestration 스킬). 본 절은 §1~§5 의 **Chief 세션 한정** 논의를
> **네 실행 경로(chief·goal·cron·workflow) 전체**로 확장한다.

### 10.0 배경 — repo·자산 컨텍스트가 Chief 에만 있다 (cross-cutting 공백)

§1 은 Chief 세션을 다뤘지만, 실측하면 **goal·cron 은 repo 도 자산도 못 받는다**:

| 경로 | repo 접근 | skill | agent(Task) | workflow |
|---|---|---|---|---|
| **chief 대화** | 전 repo `--add-dir` + `@slug` 마커 | spawn-assembler 8-layer 주입 | in-process Task ✅ | 생성/전진 ✅ |
| **goal** | `--add-dir` **없음**(org-root만) | 없음 | 없음 | 사이클 내 한정 |
| **cron** | **repo-blind**(`resolveOrgCwd` 외부경로 미인식 → org-root 추락) | **없음**(skills 미노출) | **금지**(`pm-compaction.md`: "Do NOT call subagent via Task") | **읽기만** |

→ "SoloSquad 는 멀티 repo·멀티 자산 작업을 지향"하는데, 그 능력이 **Chief 대화 한 경로에만** 있다.
S-1(resolveOrgCwd 수정)은 이 공백의 *일부*(repo cwd)만 메운다. S-6/S-7 이 **나머지 전부**를 메운다.

### 10.1 공통 실행 컨텍스트 리졸버 `resolveExecutionContext` (S-6)

흩어진 셋(`resolveOrgCwd` · chief `collectRegisteredRepoPaths` · goal 의 맨 org-root)을 **하나로 통합**.
입력 `(org, targetSpec?)` → 출력 `{ cwd, addDirs, repoManifest, scope }`. 네 경로가 전부 호출.
`resolveRepoCwd`(외부경로 해석, S-1) 경유 → cron 의 org-root 추락이 **자동 소멸**. goal 의 `--add-dir`
누락도 동시 해소.

### 10.2 기본 = 전 repo, repo 매니페스트 주입 (S-6)

- **기본 멀티:** 지정 없으면 cwd=org-root + `addDirs`=org 전 repo. 지정 시 좁힘(1개면 cwd=그 repo).
- **매니페스트:** `repositories/*.yaml` → "이 org 엔 a·b·c 가 있고 각 경로·role" 한 블록을 **모든
  프롬프트에 주입**(Google `repo` manifest · GitHub Squad·gh-aw 의 "구조 맵 먼저 읽기"와 동형).
  현재 spawn-assembler 의 단일-repo Layer 8 을 "org 전체 로스터"로 일반화.

### 10.3 지정 문법 통일 — "비우면 전체, 적으면 그것들" (S-6)

- chief: `@slug` → `[target_repos:…]`(존재) · cron: `crons/<id>.yaml` 에 `repos: [a,b]`(생략=전체) ·
  workflow stage: `target_repo`→`target_repos` · goal: `target_repo`→`target_repos`(생략=전체).

### 10.4 cron 설정 시 workflow·skill·agent 활용 방법 (S-7)

**AS-IS:** cron 은 고립된 단일 `claude --print` 프롬프트다 — skill 미노출, Task 금지, workflow 읽기만
(§10.0 표). 즉 **현재는 cron 에서 세 자산을 사실상 못 쓴다.**

**TO-BE — 세 가지 활용 패턴**(전부 §10.1 리졸버가 컨텍스트를 깔아준 위에서):

| 자산 | cron 활용 방법 | 예시 | 업계 대응 |
|---|---|---|---|
| **skill** | 리졸버가 skills 디렉토리를 노출(`--add-dir`)하거나 관련 skill 을 프롬프트에 주입 → cron 프롬프트가 skill 절차를 **이름으로 호출**(모델이 SKILL.md description 으로 선택) | "매일 전 repo 보안 점검" cron 이 `security-review` skill 사용 | cross-repo-orchestration 스킬 |
| **agent** | cron 경로에 Task/spawn-assembler 배선 → cron 이 **무거운·멀티 repo 작업을 sub-agent 로 분담**. cron=오케스트레이터, agent=워커 | "전 repo CI 점검" cron → repo 별 agent fan-out | **gh-aw 오케스트레이터+워커** |
| **workflow** | cron 이 워크플로를 **킥오프/전진**(현재 chief 전용) → 시간 트리거가 결정적 다단계를 시작 | "매주 월 회고 워크플로 자동 시작" | gh-aw 스케줄 디스패치 · 웨이브 실행 |

**설계 원칙:**
- cron 정의(`crons/<id>.yaml`)에 선택 필드 — `skills: [...]`, `workflow: <id>`(킥오프 대상),
  `spawn: true`(agent 위임 허용) — 을 두고 리졸버가 해당 컨텍스트를 배선.
- **무인 실행 안전:** agent fan-out·workflow 킥오프는 *쓰기* 를 동반할 수 있다 → cron 기본은
  **읽기·요약**, 쓰기 위임은 dev-confirm·repo 별 커밋 규약(ideation §3.5) 준수. 무인 멀티 repo 쓰기는
  명시 opt-in.
- **컨텍스트 폭증 가드:** 전 repo × 자산 → 토큰 폭증. scope 를 작업에 꼭 필요한 repo 로 좁히는 규율
  (매니페스트는 전체, 코드는 scope).

### 10.5 GitHub 생태계 패턴 ↔ SoloSquad 매핑 (재설계 아닌 배선)

| 업계 패턴 | SoloSquad 대응 | 상태 |
|---|---|---|
| 오케스트레이터+워커 (gh-aw) | chief→sub-agent · **cron→agent(S-7)** · M2/M3 워커 세션 | 부분(대화)·신규(cron) |
| "드롭박스" 상태 (Squad `decisions.md`) | `memory/decisions.jsonl` + `_handoff.md` + `_log.md`(S-3) | **이미 정합** |
| 구조 맵 먼저 (AGENTS.md) | `AGENTS.md` + repo 매니페스트(§10.2) | 신규 |
| 웨이브 실행 / repo 범위 분해 | workflow stage DAG + chief DECOMPOSE | 부분 |

> 결론: 멀티 repo·멀티 자산 능력은 **새 아키텍처가 아니라 배선**이다 — ① 공통 리졸버(S-6) ②
> 매니페스트 주입(S-6) ③ 지정 문법 통일(S-6) ④ cron 자산 배선(S-7). M2/M3(per-repo 워커)는 이
> 위에서 자연스럽게 얹힌다.

## 11. 메신저 표면 — Works 스레드 = 과제 세션 (M2/M3 의 대화 UX) · S-4/S-5

> 근거: 2026-06-27 설계 대화(사용자). §3 M2/M3(per-repo/하이브리드 워커 세션)와 §5.6(goal
> 중간 개입)이 다룬 **세션 분할**을 **메신저 사용자 경험**으로 구체화한다. 이 절은 v1.4.1
> (클라우드 배포+health) PRD에서 다루던 스레드 요구를 **세션 모델에 맞는 본가**로 흡수한 것.
> **세션 모델 결정(사용자): 하이브리드(M3)** — command=메인 오케스트레이터 세션, 각 과제
> 스레드=자식 워커 세션.

### 11.0 목표 (UX)

- **Works-`<handle>` 채널의 각 과제 스레드 안에서**, 사용자가 메시지를 보내면 Chief가 **그 과제에
  대해서만** 읽고 **스레드 안에서** 답한다. (현재는 스레드가 표시·내레이션 전용 — 사용자 입력 미수신)
- command-`<handle>` 채널은 **메인 세션**(짧은 턴·오케스트레이션·단순 확인 답변)으로 남는다.

### 11.1 과제(스레드) 생성 기준 — triage 강화

스레드는 **"코드를 확인하거나 기능을 활용할 때마다"가 아니라 *과제* 단위**로 생긴다.

| 분류 | 정의 | 표면 |
|---|---|---|
| **단순 확인/조회** | Chief가 확인해서 바로 답할 수 있는 것(코드 열람·상태 확인·짧은 질의) | **command 채널 flat 답변**(스레드 X). 현 `chat` triage 유지 |
| **공식 기획 과제** | 기획/디자인/개발 **워크플로**를 돌리거나, **PRD 같은 문서를 생성**할 정도의 과제 | **works 스레드 신설**(= 자식 세션). 현 `workflow`/`goal` triage 강화 |
| **과제 후속작업** | 같은 기획 과제의 이어지는 작업·질문 | **별도 스레드 X — 같은 과제 스레드에서** 진행(라우팅) |

- 기존 TRIAGE(`chat`/`workflow`/`cron`/`goal`, `index.ts:248`)를 재사용하되 **판정 기준을 위 표로
  명문화**. 핵심 변화 = "단순 확인이 과제로 새지 않게"(불필요한 스레드 폭증 방지) + "후속은 새
  스레드가 아니라 기존 과제 스레드로".

### 11.2 세션 토폴로지 (하이브리드 = M3 구체화)

| 세션 | 표면 | session-id 키 | cwd | 컨텍스트 |
|---|---|---|---|---|
| **메인(Chief 오케스트레이터)** | command-`<handle>` | `sessions/<user>.json`(현행) | org-root | 오케스트레이션·단순 확인 |
| **과제(자식 워커)** | works 스레드 1개 = 1과제 | **신규** `sessions/<user>__<task>.json` (§4 의 `<user>__<repo>` 일반화) | 과제 scope(repo 지정 시 그 repo, S-1/S-6 리졸버) | **그 과제만** — '그 작업에 대해서만' 성립 |

- 자식 세션은 §4 실행위치 매트릭스의 **Repo 워커(M2)** 를 *과제 단위*로 일반화한 것. 메인은 과제
  **요약만** 환류받음(§5.1 핸드오프 동형) → 과제 격리 + 메인 컨텍스트 위생.

### 11.3 현재 코드 간극 (감사 결과 — 구현 대상)

| 간극 | 현재 | 필요 |
|---|---|---|
| **스레드 입력 미수신** | 리스너가 `command-<handle>`만 수신, works·스레드는 경계에서 무시 (`discord-adapter.ts:340-345`) | **리스너를 과제 스레드로 확장** + 스레드 메시지 → 해당 과제 자식 세션으로 라우팅 |
| **단일 세션 키** | `(userId, orgSlug)`당 1개 (`session-store.ts`, `chief-runner.ts:447`) | **`(userId, orgSlug, taskId)` 자식 세션** 추가 |
| **과제↔스레드 매핑** | thread id를 `<wf>/discord-thread.txt`에 저장, reconcile이 같은 스레드 resume (`discord-task-card.ts:15-25`) | **재사용** — 후속 메시지 라우팅 키로 |
| **입력=단일 메시지** | Chief는 트리거 메시지 1줄만 받고 연속성은 claude `--resume` (`chief-runner.ts:632`) | 자식 세션이 **자체 resume id**로 과제 맥락 유지(채널 히스토리 fetch 불필요 — 현행 패턴 그대로) |

### 11.4 토큰 관리 (자식 세션)

- **현행 사실:** 봇 차원 히스토리 절단 없음 — 전적으로 Claude Code `--resume` 압축 의존. per-turn
  가드 `$5`/`300s`(`chief-runner.ts:635`)뿐. spawn은 별도 80k 캡(자식 세션엔 미적용).
- **이 설계의 추가 부담:** 과제 스레드 autoArchive **7일**(`discord-task-card.ts:69`) → 장수 과제
  자식 세션이 누적. → **§5.1 임계 자동 교대(warn 0.70 / rotate 0.85) + §5.2 durable-md(`_log.md`)를
  자식 세션에도 적용** 필수. 과제 종료(스레드 archive) 시 자식 세션 flush→요약만 메인 환류 후 kill
  (M2 hydrate→flush→kill 수명, §4).

### 11.5 단계·범위

- **S-4(M2)** 에 합류: 과제 자식 세션 = per-task 워커 세션(리스너 스레드 확장 + `<user>__<task>`
  키 + hydrate/flush). **S-5(M3)** 에서 메인↔자식 역할 분리 완성. → **v1.4.x**(v1.4.0 비범위, §9).
- §5.6(goal 중간 개입, works thread)과 **같은 가로채기 인프라** 사용 — 중복 신설 아님.
