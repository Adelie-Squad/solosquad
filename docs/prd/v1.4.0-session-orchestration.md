## 0. 동기

SoloSquad 의 Chief 는 점점 **장시간·다중 repo** 작업을 받는다. ideation 문서가
정리한 원칙(상주 위치 · 출장/소거 · 교대/계층 메모리 · 체크포인트)과 실제
레퍼런스(§5: Codex 90% auto-compact + handoff summary + repo 내 durable-md;
Qwen QWEN.md + 70% 압축 + RAG 분해)에 비추어 **현재 우리 구조의 공백**을 메운다.

ideation §5.4 결론: **우리는 (a)상주·(d)복구는 정석에 가깝고, 가장 큰 공백은
⑴임계 기반 자동 교대(compaction)와 ⑵학습형 cross-session 메모리, 그리고
⑶다중 repo 세션 격리다.**

## 1. 현재 아키텍처 (코드 기준 사실)

> **요약 모델:** 지금은 **"단일 거대 Chief 세션 · org-root 고정 · 모든 repo 를
> --add-dir · specialist 는 in-process Task"**. ideation 의 "출장(cwd 전환 +
> hydrate + flush)" 모델과 다르다 — 우리는 **출장을 안 가고 모든 짐을 org 통제실로
> 끌어온다.**

## 2. 공백 분석

1. **컨텍스트 블로트.** 다중 repo 작업이 한 세션에 누적 → 8-layer drop 으로 방어하나,
   장기 턴에서 세션 자체가 비대해짐. 토큰-임계 자동 교대 부재(§1).
2. **per-repo 격리·flush 없음.** repo A 작업 로그가 같은 세션에 남아 repo B 턴까지
   오염. ideation §3②③ 의 "출장 후 머릿속 비우기(flush)"가 구조적으로 불가능.
3. **specialist cwd 미제어.** in-process Task 라 봇이 "repo 에 들어가 일하고 나오는"
   경계를 못 만든다. 위치 제어는 프롬프트 텍스트(절대경로)에 의존.
4. **학습형 cross-session 메모리 약함.** `pm-compaction`(완료 워크플로 외부화)은
   있으나 Codex Memories 류 "선호·패턴 자동 누적/병합"은 없음.
5. **사문화 자원.** `resolveOrgCwd` 가 외부경로 repo 를 이해 못 해 repo-cwd 기능이
   죽어 있음 → per-repo 실행의 기초 블록이 깨진 상태.

## 3. 세션 분할 모델 (설계 옵션)

| 모델 | 설명 | 세션 토폴로지 | 장점 | 단점 | 채택 |
|---|---|---|---|---|---|
| **M0 (현행)** | 단일 Chief 세션, org-cwd, --add-dir, in-process Task | (user,org)×1 | 단순, 단일 기억 | 블로트·격리불가·flush불가 | (기준선) |
| **M1 (Codex 패턴 · 단일세션 강화)** | M0 유지 + ⑴토큰-임계 **자동 교대(handover)** ⑵워크플로별 **durable-md 외부화** ⑶메모리 계층 공식화 | (user,org)×1 | 프로세스 모델 무변경, 저위험 | per-repo 격리는 여전히 없음 | ✅ **v1.4.0 (S-1~S-3)** |
| **M2 (per-repo 디태치드 워커)** | repo 중작업을 **봇이 별도 claude 프로세스로 스폰**(cwd=repo, 자체 session-id, hydrate→flush→1줄 요약 반환) — ideation "출장" 모델 | (user,org)×1 + (user,org,repo)×N | 진짜 격리·flush, 위치 제어 | **새 스폰 경로 신설**(in-process Task 우회), 복잡 | ◐ **v1.4.x (S-4)** |
| **M3 (하이브리드 · 권고 종착지)** | Chief 는 org-root 오케스트레이터(짧은 턴)로 남고, 장기·다중 repo 실행만 **M2 워커 세션**에 위임. 워커는 자체 교대+복구, 요약만 Chief 메모리로 환류 | M2 + 명시적 역할 분리 | 오케스트레이션/실행 분리, 각자 컨텍스트 위생 | 가장 큰 변경 | ◐ **v1.4.x (S-5)** |

**권고 경로:** **v1.4.0 = M1 단계(S-1~S-3)** 먼저 깔고 → **v1.4.x = M2/M3 per-repo
워커 세션**. M1 이 M2 의 토대(교대·복구·durable-md·메모리 계층)를 먼저 깔아 두므로
점진 이행이 매끄럽다.

## 4. 실행 위치·제어 매트릭스 (목표 상태 M3)

| 세션 유형 | cwd | session-id 위치 | 누가 제어 | 생명주기 |
|---|---|---|---|---|
| **Chief (오케스트레이터)** | `<workspace>/<org>/` (org root) | `sessions/<user>.json` | 봇(`chief-runner`) | 장수(long-lived), `chief reset` 으로 회전 |
| **Repo 워커 (M2)** | **repo 실제 경로**(`<slug>.yaml.path`, getRepoDir 해석) | `sessions/<user>__<repo>.json` (신규 서브세션) | 봇(신규 워커 러너) | 작업 단위 — 스폰→hydrate→flush→kill, 요약만 잔존 |
| **Goal 백그라운드 (v0.4)** | org root | `bg-<goal-id>-<run-ts>` | `engine/goal-runner`(immutable) | goal 사이클 |
| **specialist (in-process Task)** | 부모 cwd 상속 | (별도 없음 — 부모 세션 내) | Chief(네이티브 Task) | 단발 |

> **제어 핵심:** M2 워커는 **봇이 직접 스폰/킬/교대**(goal-runner 패턴 일반화)하므로
> Chief 의 in-process Task 와 달리 **위치(cwd=repo)와 경계(hydrate/flush)를 봇이
> 강제**할 수 있다. Chief 는 워커를 "출장 보내는" 디스패치 마커만 발행, 봇이
> mention-parser/slash 처럼 **Chief 디스패치 전에 가로채** 워커를 띄운다.

### 5.1 토큰-임계 자동 교대 (handover) — S-2
- **관측:** `chief-runner` 가 stream-json `result`/usage 에서 누적 세션 토큰을 읽는다
  (이미 cost 캡처 중 — usage 필드 추가 캡처).
- **임계(설정화):** `chief.compaction.warn_pct`(기본 **0.70**) / `rotate_pct`(기본
  **0.85**). 레퍼런스 스펙트럼(Codex 0.90 / Qwen 0.70 / ideation 0.64·0.80) 사이에서
  보수적 기본. *주의:* Chief 는 `claude --print --resume` 이라 **Claude Code 자체
  auto-compact 도 내부에서 동작** → 우리 교대는 그 위에 얹는 **세션 회전(rotation)**
  레이어다(중복 압축 아님, 세션 갈아끼우기).
- **흐름:**
  1. `warn_pct` 도달 → Chief 에게 "지금까지 진행/결정/남은 TODO 를
     `memory/chief-skills/<wf-id>.md` 로 핸드오프 요약 작성" 지시(on-demand
     compaction).
  2. `rotate_pct` 도달 → `session-store.rotate(... "auto-compaction")` 로 세션
     회전 + **새 세션 첫 입력에 핸드오프 요약 주입**(`appendSystemPrompt` 또는
     첫 user 메시지). `chief.session_rotated` 이벤트 기록.
- **재사용:** `session-store.rotate`(이미 있음) + `chiefEventsPath` + durable-md.

## 8. 리스크

1. **이중 압축 충돌.** Claude Code 네이티브 auto-compact 와 우리 세션 회전이 겹쳐
   요약이 요약을 부를 수 있음 → 회전은 **명시적 임계 + 사용자 가시 이벤트**로만,
   네이티브 압축은 그대로 둠(우리는 "갈아끼우기"만).
2. **회전 중 유실.** 핸드오프 md 작성 실패 후 회전하면 컨텍스트 증발 → **md 작성
   성공 확인 후에만 rotate**(fail-safe: 실패 시 회전 보류 + 경고).
3. **usage 신뢰도.** stream-json usage 가 모델/버전에 의존 → 없으면 char-heuristic
   (spawn-assembler 의 chars/4) fallback + doctor 점검.
4. **per-repo 워커(M2) hot-path 위험.** in-process Task 우회는 스폰 경로 신설 →
   v1.4.x 로 분리, 단일세션(M1)에서 충분히 검증 후 이행.
5. **immutable 경계.** `src/engine/**`(goal-runner) 의 백그라운드 세션 패턴을
   **참조만** 하고 수정하지 않음(M2 워커는 신규 모듈로).
