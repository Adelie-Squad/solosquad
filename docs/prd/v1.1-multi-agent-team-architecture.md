# v1.1 — Multi-Agent Team Architecture

> **본 문서는 2026-05-27 directive 통합 재작성본.** 이전 §0~§22 amendment 누적 구조 폐기. 본 문서가 v1.1 단일 진실(single source of truth).
>
> **핵심 결정 5건 (2026-05-27 통합 directive):**
> 1. **Chief ≠ PM** — 두 역할 분리 유지. Chief = 사용자 대면 + 과제화 + 오케스트레이션 + 회고. PM = 자율 문제 정의자 (사용자와 직접 대화 안 함).
> 2. **모든 폴더 평탄(flat)** — `specialists/{team}/{name}/` 이중 중첩 폐기. `skills/pm/` 같은 중간 위계 없음. 팀 멤버십은 `teams/{team}/composition.yaml` 데이터로.
> 3. **agentskills.io 표준 채택** — skill 폴더 = `SKILL.md` + `assets/` + `scripts/` + `references/`. 루트 `templates/` 폴더 폐지, 각 skill의 `assets/`로 분산.
> 4. **PM 고도화 = v1.1 최우선** — gstack(garrytan) + RO-PNA(pna-builders) + phuryn(pm-skills) 3개 외부 reference 통합.
> 5. **OKR=Chief 결정 / 마일스톤·WBS=PM 결정** — 의사결정 권한 명확 분리.

**로드맵 위치:** v1.0.4 → **v1.1 본 plan** → v1.2 (메신저 연결 / L1 Gateway) → v1.x (대시보드 / LLM backend 추상화)

## 3. 5-Layer Team-Centric Architecture

### 3.1 위계 (Hermes V2 5-layer 차용)

```
L1. Gateway          ── Discord / Slack            [v1.2로 위임]
L2. Orchestrator     ── Chief (org) + Scheduler    ← v1.1 핵심
L3. Team             ── 4 팀 × {KNOWLEDGE, OKR, composition.yaml}
L4. Main Agent       ── pm, engineer, designer, marketer (workspace bundle)
L5. Specialist       ── 20명 평탄 + N skill 평탄
```

**v1.1.0 구현 범위**: L2~L5 (내부 아키텍처). L1 Gateway는 v1.2 별도 plan.

### 3.2 5 핵심 bot

| Bot | 위치 | 책임 | 사용자 직접 대화 |
|---|---|---|---|
| **Chief** | `<org>/agents/main/chief/SKILL.md` (org-customized) | 사용자 소통 / 결과 보고 / 과제화(triage) / 에이전트 오케스트레이션 / 회고·자가학습 / Skill·Workflow 개선 | ✅ 유일 |
| **PM** (Product Manager) | `agents/main/pm/SKILL.md` (workspace bundle) | 문제 발견·정의 / 가설·실험 설계 / 데이터 기반 판단 / 마일스톤·WBS·일정 분해 | ❌ Chief 경유 |
| **Engineer** | `agents/main/engineer/SKILL.md` | engineering team supervisor | ❌ Chief 경유 |
| **Designer** | `agents/main/designer/SKILL.md` | design team supervisor | ❌ Chief 경유 |
| **Marketer** | `agents/main/marketer/SKILL.md` | marketing team supervisor | ❌ Chief 경유 |

### 3.3 4 팀 구성

Chief는 **어느 팀에도 속하지 않고** 4팀 위에서 dispatch.

| 팀 | Main | Specialist (멤버 수) |
|---|---|---|
| **product** | pm | pmf-planner, feature-planner, idea-scoper, business-strategist, policy-architect, data-analyst (6) |
| **engineering** | engineer | backend-engineer, architect, fde, creative-frontend, data-engineer, cloud-admin, qa-engineer, security-engineer (8) |
| **design** | designer | researcher, ux-designer, ui-designer (3) |
| **marketing** | marketer | gtm-strategist, brand-marketer, performance-marketer (3) |

**총합**: Chief 1 + 4 main + 20 specialist = **25 agent**. + ~20 skill.

> Note: 이전 PRD의 "25 → 20 (5 병합)" 결정 중 **content-marketer 병합은 취소**. content-writing은 skill로 별도 운영, brand-marketer는 sub-agent로 유지. 결과: 20 specialist.

## 5. Chief Sub-System

### 5.1 Chief 책임 4가지

1. **사용자 소통 / 결과 보고** — 유일한 사용자 대면 bot
2. **과제화 (Task triage)** — 입력을 4분류: 단순 논의 / workflow / schedule / goal
3. **에이전트 오케스트레이션** — 4 main bot에게 dispatch + 결과 종합
4. **회고 / 자가학습 / Skill·Workflow 개선** — skill 종료 시 retrospective + refinement 호출

### 5.2 Chief 6+1 Stage State Machine

```
TRIAGE → DECOMPOSE → DISPATCH → AWAIT → SYNTHESIZE → DECIDE → RETROSPECT
  ↑                                                                  │
  └──────────────────── next turn ───────────────────────────────────┘
```

| Stage | 입력 | 출력 | 사용 skill |
|---|---|---|---|
| **TRIAGE** | 사용자 메시지 | 분류 결과(discussion/workflow/schedule/goal) + Educational Nudge (불명확 시 KNOWLEDGE 가이드 선제시) | `triage/SKILL.md` |
| **DECOMPOSE** | triage 결과 | 작업 분해 (PM 호출 / 다른 main 직접 / single skill 실행) | — |
| **DISPATCH** | 분해 결과 | spawn 명령 (PM, engineer, designer, marketer) | — |
| **AWAIT** | dispatch 응답 대기 | open_questions[] 도착 시 사용자에게 batch 질의 | — |
| **SYNTHESIZE** | 모든 sub-agent 응답 | 통합 결과 | — |
| **DECIDE** | 통합 결과 + OKR | 결정 (반영 / 다음 cycle / 폐기) | — |
| **RETROSPECT** | 완료된 cycle | skill/workflow 개선 제안, ledger 기록 | `retrospective/`, `skill-refinement/`, `workflow-refinement/` |

각 stage 전환은 `<org>/memory/chief-stage-events.jsonl` 에 기록 — 추적성 보장.

### 5.3 Chief 의 의사결정 권한

- **OKR** (분기 단위) — `team/{team}/OKR.md` 작성 / 갱신 / 평가
- **Task 분류** (turn 단위) — discussion / workflow / schedule / goal
- **Schedule 등록** — `<org>/schedules/` 또는 schedules/ override
- **사용자 대면 응답 톤** — `user/voice.md` 기반 + 도메인 전문가 페르소나

### 5.4 Triage Stage 0 — Educational Nudge

사용자 입력이 막연할 때 PM/specialist를 spawn 하기 전에 **사용자에게 KNOWLEDGE 가이드 선제시**:

> "마케팅 캠페인 만들고 싶다고 하셨는데, 저희 팀 KNOWLEDGE에 따르면 [growth playbook 3단계: Audit → Acquire → Activate] 가 있습니다. 어느 단계에서 시작할까요? 또는 처음부터 가설 정의가 필요하신가요?"

이는 **사후 라벨링 원칙** (RO-PNA 차용): 프레임워크를 선처방하지 않고, 사용자 의도가 모이면 사후 라벨로 안내.

## 6. PM Sub-System — v1.1 최우선 ★

### 6.1 핵심 원칙

> **PM은 사용자와 직접 대화하지 않는다.** Chief가 유일한 user-facing bot. PM은 dispatch 받아 백그라운드에서 자율 분석.

이유:
- 사용자 컨텍스트 분기점이 분산되면 일관된 도메인 페르소나 유지 불가
- PM의 deep work는 긴 컨텍스트 + 다단계 reasoning — 대화 latency와 양립 불가
- Chief가 모든 사용자 question을 batch 처리해야 효율적

### 6.2 PM 자율 작동 흐름

```
[User → Chief]
   사용자 메시지

[Chief TRIAGE → DECOMPOSE]
   "PM deep work 필요" 판단
   → spawn PM(brief, context)

[PM 자율 실행 — 사용자 부재]
   1. 9-layer JIT context 읽음:
      - workspace knowledge / team KNOWLEDGE / team OKR / SKILL
      - <org>/core, profile, domain, memory, archive
   2. Skill chain (자체 reasoning):
      a) discovery-synthesis     ← archive.sqlite + customer.md에서 JTBD 신호 추출
      b) problem-definition       ← SCQA→5-Whys→MECE→TDCC→XYZ chain (RO-PNA 6-Phase 자체화)
      c) opportunity-tree         ← Teresa Torres OST + gstack Six Forcing Questions 자가검증
      d) hypothesis-design        ← XYZ + If-Then-Because + V/U/V/F assumption 분류
      e) prd-writer               ← 8-section PRD (phuryn template)
      f) wbs-decomposition        ← 마일스톤 → WBS 분해
   3. Output JSON:
      {
        "design_doc": "...",          // PRD 본문
        "milestones": [...],          // WBS
        "open_questions": [...],      // 컨텍스트로 풀 수 없는 항목
        "confidence_score": 0..100,   // RO-PNA Confidence Score
        "evidence_refs": [...]        // archive entries, knowledge sections
      }

[PM → Chief]

[Chief AWAIT]
   open_questions 가 비어있으면 → SYNTHESIZE 진입
   비어있지 않으면 → 사용자에게 batch 질의 (1~N개를 한 번에)

[User → Chief]
   batch 답변

[Chief → re-dispatch PM(resolved_questions)]

[PM 재실행]
   resolved_questions 통합 → 최종 design doc
```

### 6.6 RO-PNA 6-Phase의 PM 내부화 (`skills/problem-definition/assets/reasoning-chain.md`)

```markdown
# Problem Definition — PM 자체 reasoning chain

## P1. SCQA (Context Mining)
- Situation: archive.sqlite의 최근 N건 + customers.md
- Complication: memory ledger + workflow 실패 기록
- Question: PM 자체 구문화
- Answer: 가설 후보 N개
→ Situation/Complication 데이터 부족 시 open_questions append

## P2. 5-Whys (Root Cause)
P1.Q 에 대해 5단계 체인. 근본 원인 1문장.
→ 5단계 못 채우면 → open_questions.append({ stage: "5-whys", question: "Why-N 단계에서 X가 사실인지" })

## P3. MECE (Decomposition)
근본원인을 후행/선행 문제로 분해. 누락 검증.

## P4. TDCC (5필드)
후행지표 / 선행문제(3) / 기회 / 인과 / 미지
→ 미지 필드 = open_questions

## P5. XYZ Hypothesis
[X%] of [세그먼트 Y] will [행동 Z] within [기간], because [근거]
→ metric 미정 = open_questions

## P6. 1-pager PRD synthesis
P1~P5 결과를 PRD 8-section에 매핑. prd-writer로 호출.
```
