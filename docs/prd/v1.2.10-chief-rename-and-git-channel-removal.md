# Part A — PM → Chief 리네임 (구현 완료)

v1.1 이 org-level supervisor 를 PM→Chief 로 리브랜딩했으나 출력 카피·내부 식별자·CLI
동사·에이전트 정체성(SKILL)·이벤트 네임스페이스에 "pm" 잔재가 남아 사용자가
`solosquad pm reset` / "PM session" 을 계속 마주쳤다.

## A.1 "pm" 은 세 가지 의미 — 블라인드 치환 금지

| 층위 | 의미 | 처리 |
|---|---|---|
| **(A) 세션 드라이버 / 사용자 표면** | org-level supervisor = Chief 그 자체 | ✅ rename |
| **(B) 영속 데이터 계약** | 디스크/와이어 식별자(이벤트 kind, `workspace.yaml` `pm:` 키, `pm-compaction` 루틴 id, `system-pm-compaction` 스레드, `memory/pm-skills/`) | ⚠️ read-compat 후 rename 가능한 것만 / 나머지 keep+문서화 |
| **(C) 유령 "PM 에이전트"** | chief-runner 주석의 `agents/main/pm/` — **디스크에 없음** | 🧹 stale 주석 정정 |

## A.2 적용 결과 (구현 완료)

- **CLI**: `solosquad chief status/reset/compact` 정규화. `solosquad pm …` 은 숨김

- **타입**: `PmConfig`→`ChiefConfig`(컴파일 타임만, yaml 키 `pm` 유지).

# Part C — git-<handle> VCS 채널 제거 (롤백)

## C.0 결정 — SoloSquad 는 알림 기능에 관여하지 않는다

v1.2.9 Part B 는 `git-<handle>` 채널 + push 알림 코드(`git-event-notify.ts`)를
깔았다(게이트 dormant 라 실제 발사는 안 됨). 검토(2026-06-08) 결과 **이 자체 알림
파이프라인을 폐기**한다:

- **필요한 것은 "push 승인"뿐.** 안전의 핵심은 dev-confirm **승인 게이트**(→ v1.3.0)
  이고, 승인 UX 는 `command-<handle>` 채널에서 끝난다. 전용 git 채널은 승인과 **분리
  가능한 별도 관측 레이어**라 안전 기능엔 불필요하다.
- **알림은 플랫폼 네이티브가 우월.** "완료된 push 알림"은 사용자가 **GitHub→Discord
  네이티브 webhook**(repo settings 에 Discord webhook URL 등록)으로 제로코드·전건·
  실제 도착 기준으로 받을 수 있다. SoloSquad 가 로컬 `git log` 로 재구현하는 것은
  플랫폼이 공짜로 주는 것의 재발명이다.
- **귀속(attribution)도 위임 가능.** "이 push 가 에이전트 작업인가"는 v1.3.0 이
  **커밋 trailer 에 workflow-id 를 스탬프**하면 네이티브 webhook 의 커밋 메시지에
  자연히 노출된다 — 자체 알림 파이프 불필요.

→ **SoloSquad 역할 수축:** ⑴ push **승인 게이트**(v1.3.0) + ⑵ (선택) 커밋
workflow-id 스탬프. **알림 채널 운영은 사용자/GitHub/Discord 에 위임.**

## C.1 제거 범위 (코드만 — 기존 데이터 방치)

기존 v1.2.9 사용자는 이미 `git-<handle>` 채널과 `channels.git` yaml 필드
(schema_version 2)를 갖고 있다. **마이그레이션으로 정리하지 않는다**(데이터 삭제 위험
회피) — 코드 동작만 제거하고 기존 채널은 고아로 방치, 사용자가 원하면 수동 삭제.
