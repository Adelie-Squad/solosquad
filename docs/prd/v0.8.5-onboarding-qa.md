## 1. 핵심 버그: SOLOSQUAD_VERSION 하드코딩 (회귀)

### 1.1 증상

```
$ solosquad init       # CLI v0.8.4
... (마법사 정상 완료)

$ solosquad bot
  ⚠ Workspace layout is behind the installed CLI.
    Workspace: v0.4.0   CLI: v0.8.4
    Run a migration before continuing:
      solosquad migrate --apply
```

fresh init 직후 즉시 마이그레이션 경고. 신규 사용자는 *왜 갓 만든 워크스페이스가 구버전?* 혼동.

### 1.2 원인

`src/cli/init.ts:29`:
```typescript
const SOLOSQUAD_VERSION = "0.4.0";
```

- 이 상수가 `workspace.yaml.version` + `last_migrated_to`에 박힘 (line 630, 644)
- v0.4 시점에 박힌 후 갱신되지 않음 — package.json의 `"0.8.4"`와 disconnect
- 모든 `migrate --apply`는 정상 동작하지만, fresh init 사용자에게는 *불필요한 추가 단계 1개* 강요

### 1.3 수정

`src/util/version.ts` 신설:

```typescript
import { createRequire } from "module";
const require = createRequire(import.meta.url);
const pkg = require("../../package.json") as { version: string };
export const SOLOSQUAD_VERSION: string = pkg.version;
```

`src/cli/init.ts` line 29 제거 + import로 교체. 이후 release마다 `package.json` bump 한 곳만 갱신해도 init이 따라옴.

### 1.4 backward-compat

- 기존 v0.4.0~v0.8.3 워크스페이스: 이미 `migrate --apply`로 v0.8.x까지 올라가야 하므로 영향 없음
- 신규 init: 즉시 *current CLI 버전*으로 박힘 → migration 경고 0
- 회귀 risk: `SOLOSQUAD_VERSION`을 import하는 다른 모듈이 있다면 동일 효과

### 1.5 검증

- `test/init-version.test.ts` 신설 — fresh init → workspace.yaml.version === package.json.version 동등
- `solosquad doctor` 신규 워크스페이스에서 mismatch 경고 0건 확인

### 3.4 `.solosquad/` 폴더 위계 설명 절 신설 (§3.3)

기존 §3 "구조"에 *왜 위계마다 `.solosquad/`가 있는가* 설명 추가:

```
<workspace>/                       ← 사용자가 mkdir로 만든 디렉터리
├── .solosquad/                    ← (A) 전역 시스템 설정
│   ├── workspace.yaml             ← 워크스페이스 메타 + timezone + brief 시간
│   ├── .env                       ← 토큰·OWNER_NAME·OWNER_ROLE (gitignored)
│   ├── agents/                    ← 25 specialist SKILL (번들 init)
│   ├── routines/                  ← cron routine 정의
│   ├── core/                      ← 시스템 코어 설정
│   ├── templates/                 ← 워크플로우 템플릿
│   ├── orchestrator/              ← PM session 설정
│   └── knowledge/                 ← 사용자 누적 craft + 도메인 노트
├── docker-compose.yml             ← (워크스페이스 root에 노출 — 사용자가 직접 편집)
├── Dockerfile
└── <org>/                         ← 사용자 콘텐츠 (org = 사업/제품)
    ├── .org.yaml                  ← org 메타 (provider, remote_url, products)
    ├── .solosquad/                ← (B) 멀티유저 식별
    │   └── users/<handle>.yaml    ← 메신저 handle → bot_user_id 매핑
    ├── core/                      ← org 단위 PRINCIPLES.md, VOICE.md
    ├── agent-profile.yaml         ← 25 agent의 org별 modifier
    ├── domain/                    ← org 도메인 지식
    ├── memory/                    ← JSONL 메모리 (signals·experiments·decisions)
    ├── workflows/                 ← org 워크플로우 인스턴스
    ├── repositories/<repo>/       ← 사용자 코드 (불가침)
    │   └── .solosquad/            ← (C) repo ↔ org 메타
    │       └── repo.yaml          ← repo가 어느 org에 속하는지 + role
    └── <messenger>/               ← 메신저별 어댑터 자산
```

**왜 위계마다 `.solosquad/`?**
- 모두 *SoloSquad가 관리하는 시스템 메타데이터*. 사용자 콘텐츠(`memory/`·`workflows/`·`repositories/`)와 *디스크에서 시각적으로 분리*. dotfile 컨벤션으로 ls 기본 출력에 안 보임
- `.gitignore`에 `.solosquad/` 한 줄로 모든 위계 시스템 자산 제외 가능
- 위계마다 *담는 내용이 다른 이유*: 시스템 정보의 **scope**가 다르기 때문
  - 워크스페이스 레벨 = 전역 (모든 org가 공유하는 agent SKILL·envs·timezone)
  - org 레벨 = org당 1회 박혀야 하는 식별 (멀티유저 handle 매핑)
  - repo 레벨 = repo가 어느 org 소속인지·역할은 무엇인지 (한 repo가 여러 org에 나타날 일 없음)

### 3.5 routine 4건 영구 제거 + 표 갱신

product-roadmap §3.2.8(2026-05-15 박제)에서 "디폴트 3건만"로 결정됐던 4건을 v0.8.5에서 *비-디폴트 유지조차 그만두고 영구 제거*. 사유: 비-디폴트로 둬도 결국 사용자 도메인 prompt(product brief·experiment ledger)가 있어야 의미 있는 출력을 내므로 cron 슬롯·UI 표 자리를 차지할 가치 없음. 도메인 분석은 사용자가 작성하는 워크플로우·goal로 표현하는 게 맞음.

**제거 대상 4건**:
- `signal-scan` (12:00) — 외부 시그널 탐지
- `experiment-check` (16:00) — 실험 status·signal_strength 점검
- `weekly-review` (Sun 20:00) — 주간 회고
- `v06-retrospective-stats` (Sun 22:00) — v0.5 데이터원 ETL 보고서

**인프라 routine 2건 → 1건 통합**:
- 구 `archive-rotate` (00:00) + 구 `log-rotate` (00:30)를 단일 `system-housekeeping` (00:00)으로 통합
- 사유: 둘 다 자정 idle 시간의 결정적 housekeeping. silent (사용자 알림 X) · 멱등. 둘을 분리 cron으로 둘 이유가 없음. 통합으로 cron 1슬롯·UI 1행 절약
- 격리: `rotateArchive()` + `rotateLogs()` 각각 try/catch — 한쪽 실패가 다른 쪽 진행 막지 않음
- 결정적 함수(`src/memory/archive-rotate.ts:rotateArchive` + `src/util/logger.ts:rotateLogs`)는 *변경 없이* 그대로 호출. 통합은 dispatch 레벨에서만

**최종 routine 표 (v0.8.5+)**:

| 루틴 | 시간 | 분류 | 목적 |
|---|---|---|---|
| Morning Brief | 08:00 (workspace tz) | 사용자 brief | 야간 일과 요약 |
| Evening Brief | 18:00 | 사용자 brief | 오늘 결정·완료 + 내일 우선순위 |
| PM Compaction | 23:00 | 컨텍스트 관리 | 완료 워크플로우 외부화 (`memory/pm-skills/`) |
| System Housekeeping | 00:00 | 인프라 (silent) | ① FTS5 archive + retention · ② logs/ 14일 이상 삭제 (try/catch 격리) |

**backward-compat**:
- `workspace.yaml.background_routines` 키는 forward-compat 위해 *읽기 무시*로 유지 (에러 X)
- 기존 워크스페이스 cron 등록은 *별도 등록이 아닌 ROUTINES 배열 기반*이라 ROUTINES에서 빠지면 자동으로 schedule 해제
- `solosquad run-routine signal-scan`은 routine 부재 에러 (의도된 동작 — 제거됐으므로)

## 4. wizard 문구 정합 (init.ts)

### 4.1 redundant CWD prompt 제거

현재 흐름 (`src/cli/init.ts:387-440` `resolveInitWorkspace`):
- 부모에 `.solosquad/` 없음 + CWD에 없음 → "Initialize workspace at: <cwd>" prompt 1개
- 부모에 `.solosquad/` 있음 → 3-way 선택 (cwd / existing / custom)

사용자 인지: `mkdir foo && cd foo && solosquad init` 후 *"여기에 만들까요?"* 또 묻는 게 이상함. mkdir로 이미 명시한 결정을 또 확인.

**변경**:
- 부모에 `.solosquad/` 없을 때: prompt 제거, CWD 그대로 사용 + 안내 메시지만 1줄 (`Workspace will be created at: <cwd>`)
- 부모에 `.solosquad/` 있을 때: 3-way 분기 유지 (충돌 가능성 명시 필요)
- 사용자가 *다른 경로* 원하면? → 마법사 시작 *전*에 `cd` 해야 함. master-guide Step 5에 명시.

### 4.2 wizard 각 prompt 헬프 텍스트

inquirer prompt 위에 `console.log(chalk.dim("..."))` 한 줄씩 추가해서 *왜 묻는지* 안내:

- name: `"PM·brief가 당신을 부를 때 사용합니다."`
- role: `"founder / developer / designer 등. agent들이 응답 톤·기술 깊이를 조정합니다."`
- messenger: `"1 워크스페이스 = 1 메신저. n잡 시나리오는 워크스페이스를 여러 개 만드세요."`
- timezone: `"daily brief 시각 + 모든 routine cron의 기준입니다."`
- handle (confirmation prompt): `"메신저 채널 페어 \`command-<handle>\` / \`works-<handle>\`에 사용됩니다. 영소문자·숫자·_만 허용 — 그 외 문자는 _로 자동 변환."`
- org name: `"사업/제품 단위. 한 워크스페이스에 org 여러 개 가능 (예: tesla + spacex)."`
- provider: `"org의 코드 호스팅 위치. add repo 시 git URL host 추정에 사용. 없으면 local."`
- remoteUrl: `"org 홈페이지(예: github.com/tesla)가 있으면 입력. 없으면 Enter."`

scheduler 정리는 본 patch에서 진행. wizard `saveWorkspaceYaml` 호출에서 `background_routines` 블록 *제거* (init.ts:667-670). `ROUTINES`는 9→5. 자세한 코드 변경은 §3.5 + product-roadmap §3.2.8 "구현 결과" 참조.
