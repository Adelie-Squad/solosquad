# v1.2.2 — GitHub-aligned 계층 구조 재편

> Workspace / Product / Project 용어의 혼동을 해소하기 위해 디렉토리 계층을 **GitHub(GitLab) 관례와 일치하는 용어·구조**로 전면 재편한다. 단일 트리 아래에 organization → repository, orchestrator는 workflow 인스턴스를 organization 레벨에서 관리.

**기준 버전:** v1.1.5
**분류:** 구조 재편 — Breaking change
**심각도:** High — 초기 사용자가 구조를 오해해 잘못된 위치에 작업하거나, project 용어를 git 리포지토리와 동일시하는 혼동을 반복 유발. 조기에 재편하지 않으면 이후 확장(멀티 리포, Cross-repo workflow)의 기반이 없어짐.

> **본 문서는 초기 버전의 "용어 정의 + 프롬프트 보강" 스코프에서 확장됨.** 사용자 피드백(2026-04-21)으로 직렬적 개선이 아닌 전면 재편이 필요하다고 판단. 마이그레이션 절차는 별도 문서(`v1.2.3-migration-process.md`)에서 다룸.

---

## 1. 배경 — 사용자 피드백과 설계 결정

### 1.1 초기 사용자 QA에서 반복 제기된 혼동

**Q1.** 프로젝트는 리포지토리 단위인가? `bv-ai-native-po/projects/` 하위에 리포를 clone하면 되는가? 그럼 `bv-ai-native-po`는 깃허브로 치면 organization인가?

**Q2.** `bv-ai-native-po`는 `my-workspace`의 하위인가? 별도 경로에 있으면 어떤 문제가 생기는가? Workspace와 Product를 분리할 필요가 있는가?

### 1.2 설계 결정

- **GitHub 용어와 일치시킨다** — Workspace(루트), Organization, Repository. 개발자가 이미 아는 멘탈 모델로 학습비용 0.
- **"Product"는 디렉토리 계층에서 제거**, 개념 태그로만 `.org.yaml`에 유지. 한 조직이 여러 제품을 가질 수 있고, 여러 조직이 한 제품에 기여하는 유연성은 yaml 선언으로 충분.
- **"Project" → "Workflow"** 이름 변경. Orchestrator가 만드는 단계별 협업 인스턴스임을 용어로 명확히.
- **단일 트리 루트** — `REPOS_BASE_PATH` 분리 구조 폐기. 모든 것이 Workspace 루트 아래. Documents 하위에 두는 기본값도 폐기.
- **Workspace 루트 이름 자유화** — `.solosquad/` 하위 폴더 존재로 감지. 기본 이름은 `solosquad`, 사용자가 자유롭게 바꿀 수 있음. 한 머신에 여러 루트 가능(예: 직장인 모드 / 1인 창업가 모드 / 부업 부캐).
- **Organization 이름은 사용자 직접 입력** — URL 파싱·API 호출 없음. 대신 init 프롬프트에서 **구체적 예시로 안내**해 사용자가 자신의 브랜드/팀 단위를 고를 수 있게 함.
- **자동 clone 미지원** — repository는 사용자가 직접 `git clone`/`git init`. 향후 별도 스코프에서 논의.

---

## 2. 목표

1. 사용자가 setup 직후 **한 번에** "내 workspace/organization/repository가 어디에 뭐가 있다"를 이해.
2. 멀티 리포지토리 조직(프론트/백/데이터 분리)을 **네이티브로 수용**.
3. Orchestrator 워크플로우가 repo 경계를 넘나들며 에이전트를 조율할 수 있는 기반 마련.
4. 용어가 IDE/Git 관례와 어긋나지 않아 학습·오해 비용 최소화.

## 3. 새 디렉토리 구조

### 3.1 개요 다이어그램

```
<workspace-root>/                  ← 사용자가 고른 경로·이름 (기본: ~/solosquad, 예: ~/solosquad-work, ~/solosquad-side)
  .solosquad/                      ← 시스템 설정 — 이 폴더 존재로 workspace 감지
    workspace.yaml                 ← workspace 메타 (display name, version, created_at)
    .env                           ← 토큰, MESSENGER 설정 (기존 루트 .env에서 이동)
    agents/                        ← 에이전트 정의 (기존 루트에서 이동)
    routines/                      ← 루틴 프롬프트 (기존 루트에서 이동)
    core/                          ← 오너 프로필, 원칙 (기존 루트에서 이동)
    templates/                     ← PRD/handoff/status 템플릿
    orchestrator/                  ← 오케스트레이터 SKILL
  <organization-slug>/             ← 조직 = GitHub/GitLab org 대응 또는 로컬 그룹
    .org.yaml                      ← 조직 메타 (name, provider, remote_url, products)
    memory/                        ← 조직 레벨 메모리 (JSONL)
      hypotheses.jsonl
      experiments.jsonl
      decisions.jsonl
      signals.jsonl
      routine-logs/
    workflows/                     ← orchestrator가 만드는 워크플로우 인스턴스
      <workflow-id>/
        _status.yaml
        prd.md
        stage-N-<name>/
    slack/  or  discord/  or  telegram/   ← 플랫폼 채널 매핑 (조직 레벨)
    <repository-slug>/             ← 실제 git repo
      .git/
      .solosquad/
        repo.yaml                  ← role, language, linked_org, notes
      (코드)
    <repository-slug>/             ← 한 조직이 여러 repo 보유 가능
      ...
  <organization-slug>/             ← 한 workspace에 여러 org 가능
    ...
```

### 3.2 Workspace 루트

- **기본 이름** — `solosquad`. 사용자가 입력을 생략하면 init은 `~/solosquad/`를 제안.
- **변경 자유** — 한 머신에 여러 루트를 둘 때 **페르소나 단위로 분리**하는 것이 설계 의도. 예:
  - `~/solosquad/` — 기본(개인 창작·실험)
  - `~/solosquad-work/` — 직장인으로서 업무 관련 부캐
  - `~/solosquad-founder/` — 1인 창업가 부캐
  - `~/solosquad-side/` — 부업·사이드 프로젝트 부캐
  각 루트는 자기만의 `.solosquad/.env`(토큰·메신저), 오너 프로필, 오가니제이션 셋을 가짐 → 직장 이메일·사내 Slack이 개인 프로젝트와 섞이지 않음.
- **감지 로직** — CLI 실행 시 CWD에서 **위로 올라가며 `.solosquad/` 폴더가 있는 첫 디렉토리**를 workspace 루트로 인식 (npm의 `package.json` 탐색과 동일).
- **다중 루트 전환** — 다른 루트로 이동하려면 `cd`만 하면 됨. 해당 루트의 `.solosquad/.env`가 자동 적용.

**`.solosquad/workspace.yaml` 스키마:**
```yaml
version: "1.2.2"                   # 이 workspace가 처음 생성된 solosquad 버전
display_name: "Solosquad"          # status/logs에 표시되는 이름 (기본: 루트 폴더 이름)
persona: "personal"                # 사용자가 자유롭게 태깅 (personal/work/founder/side 등 아무 문자열)
created_at: "2026-04-22T10:00:00Z"
last_migrated_to: "1.2.2"          # 마지막으로 마이그레이션이 수행된 버전
```

### 3.3 Organization

- **정의** — 여러 repository를 묶는 **브랜드/팀 단위**. GitHub org가 있다면 그 이름과 1:1 매칭, 없으면 사용자가 자유롭게 네이밍. 디렉토리 이름이 그대로 slug 역할.
- **입력 방식** — 사용자가 직접 타이핑. URL 파싱·API 호출 없음. init 프롬프트에서 사용자 경험을 돕기 위해 **구체적 예시를 함께 제시**.

**init 프롬프트 안내 문구 (참고 사례는 실제 1인 창업가의 부캐 분할):**

```
Organization name:
  Your brand, team, or GitHub-org-level grouping.
  One organization holds multiple repositories that share the same
  identity, ownership, or product lineup.

  Examples from real users:
    toronchul       — personal brand for freelance work
    pna             — 2-person startup, several products
    sapunsapoon     — side-project collective
    adelie-squad    — open-source team (https://github.com/Adelie-Squad)

  If you already have a GitHub/GitLab organization, reuse that exact
  slug. Otherwise, pick a short name (lowercase, hyphen-separated).

> _
```

- **remote_url은 선택 메타데이터** — `.org.yaml`에 사용자가 직접 기입(없어도 됨). 파싱·자동 fetch 없음.
- **provider 필드** — `github` / `gitlab` / `gitea` / `local` 중 하나. 사용자 선택. 향후 이 값에 따라 repo sync 시 remote 관례가 달라질 수 있음 (예: HTTPS vs SSH clone 명령 제안).

**`.org.yaml` 스키마:**
```yaml
name: "toronchul"                  # 표시 이름 (사용자 입력)
slug: "toronchul"                  # 디렉토리명과 일치 (소문자·하이픈)
provider: "github"                 # github / gitlab / gitea / local (사용자 선택)
remote_url: "https://github.com/toronchul"   # 선택. 없으면 null.
homepage: null                     # 선택. 사용자가 나중에 수동 기입.
products:                          # 개념 분류 — 여러 repo를 "어떤 제품을 위해 존재하는가"로 묶음
  - name: "Blog Platform"
    slug: "blog"
    description: "개인 블로그 SaaS"
    repos: ["blog-frontend", "blog-api"]
  - name: "Newsletter Tool"
    slug: "newsletter"
    description: "이메일 뉴스레터 자동화"
    repos: ["newsletter-core"]
description: |
  토론철 개인 브랜드의 제품·리포 묶음.
created_at: "2026-04-22T10:00:00Z"
```

### 3.4 Repository

- **생성 방식** — **사용자가 직접 `git clone` 또는 `git init`**. 자동 clone 없음.
- **등록** — 조직 디렉토리 안에 repo 폴더가 추가된 후 `solosquad sync`(신규 명령)가 `.solosquad/repo.yaml`을 자동 생성 (기존에 없을 때).
- **역할 선언** — `repo.yaml`의 `role` 필드로 frontend/backend/data/infra/docs 등 분류.

**`<org>/<repo>/.solosquad/repo.yaml` 스키마:**
```yaml
slug: "solosquad"
name: "SoloSquad CLI"
role: "main"                       # main / frontend / backend / data / infra / docs / unknown
language: "typescript"             # 자동 감지 (package.json, Cargo.toml 등으로)
linked_org: "adelie-squad"         # 상위 org slug와 일치해야 함
remote_url: "https://github.com/Adelie-Squad/solosquad"   # git remote에서 추출
products: ["solosquad"]            # 이 repo가 구현하는 제품 slug (0개 이상)
notes: |
  Free-text notes visible to agents.
registered_at: "2026-04-22T10:00:00Z"
```

### 3.5 Workflow (기존 Project)

- **위치** — `<org>/workflows/<workflow-id>/`
- **내부 구조**는 기존 `projects/{id}/`와 동일 (`_status.yaml`, `prd.md`, `sessions/`, `stage-N-*/`, `_handoff.md` 등)
- **Cross-repo 지원** — `_status.yaml`의 각 stage에 `target_repo` 필드 추가 (4장 참조)

## 4. Cross-Repository Workflow 모델

### 4.1 원칙

- **Workflow 상태는 조직 레벨에서 중앙 집중** (`<org>/workflows/<id>/_status.yaml`)
- **Claude Code 세션은 stage별 `target_repo`의 cwd로 실행** — 즉 단일 repo 안에서만 편집/commit
- **Handoff에 repo 간 연계 힌트 기록** — 다음 stage가 참조해야 할 변경 지점을 `_handoff.md`에 명시

### 4.2 `_status.yaml` 확장 예시

```yaml
workflow_id: "pmf-signup-flow"
project: "solosquad"                # 어떤 product의 워크플로우인지
created_at: "2026-04-22T11:00:00Z"
stages:
  - id: stage-1-api
    team: engineering
    agents: ["api-developer", "backend-developer"]
    target_repo: "solosquad-backend"
    status: in_progress
    depends_on: []

  - id: stage-2-ui
    team: engineering
    agents: ["creative-frontend"]
    target_repo: "solosquad-frontend"
    status: pending
    depends_on: ["stage-1-api"]
    upstream_handoff: "workflows/pmf-signup-flow/stage-1-api/_handoff.md"

  - id: stage-3-schema
    team: engineering
    agents: ["data-engineer"]
    target_repo: "solosquad-data"
    status: pending
    depends_on: ["stage-1-api"]    # 병렬 진행 가능
```

### 4.3 Claude 세션 spawn 규칙

`src/bot/index.ts` 변경:

```ts
// 기존
const productDir = path.join(getReposBase(), product.slug);

// 새 구조
const orgDir = path.join(getWorkspaceRoot(), org.slug);
const workflowStage = await resolveActiveStage(orgDir, workflowId);
const repoDir = path.join(orgDir, workflowStage.target_repo);
const result = await runClaude(prompt, repoDir);
```

- 세션 cwd는 **repo 단위**로 제한 → safety preamble의 `.env`·`credentials` 차단 그대로 작동
- 다른 repo의 코드를 "읽기"만 해야 하는 경우, **stage 진입 시 필요한 파일을 workflow 폴더로 스냅샷 복사** → Claude에게 컨텍스트 주입. 쓰기는 절대 target_repo에만.

### 4.4 미해결 / 차기 스코프

- **의존 repo 간 버전 정합성** — 백엔드 API 변경 + 프론트 대응이 동시 머지되어야 할 때 PR 타이밍 조율. v1.2.2에서는 **사람이 조율**, 자동화는 차기.
- **Monorepo 감지** — 하나의 repo 안에 `apps/frontend`, `apps/backend`가 있는 경우. v1.2.2는 **repo 단위만 지원**.
- **부분 실패 복구** — stage-2 실패 시 stage-1 commit 롤백 여부. v1.2.2는 **forward-only**, 롤백은 사용자가 수동.

## 5. 용어 정의 (최종)

| 용어 | 정의 | 예시 |
|---|---|---|
| **Workspace** | `.solosquad/` 하위 폴더를 가진 사용자 지정 루트 디렉토리. 기본 이름 `solosquad`. 페르소나 단위 분리를 위해 여러 개 가능. | `~/solosquad/`(기본), `~/solosquad-work/`(직장), `~/solosquad-founder/`(창업), `~/solosquad-side/`(부업) |
| **Organization** | 여러 repository를 묶는 브랜드/팀 단위. GitHub/GitLab org와 1:1. 없으면 사용자가 자유롭게 네이밍. | `toronchul`, `pna`, `sapunsapoon`, `adelie-squad` |
| **Repository** | 실제 git 리포지토리(clone/init). 코드 편집·commit의 단위. | `solosquad`, `blog-frontend`, `blog-api` |
| **Workflow** | Orchestrator가 만드는 단계별 협업 인스턴스. PRD + handoff + 산출물. | `pmf-signup-flow`, `rebranding-q2` |
| **Product** | 한 개 이상의 repository가 구현하는 제품/서비스 **개념**. 디렉토리가 아닌 `.org.yaml`의 `products:` 항목. | `blog`, `newsletter`, `solosquad` |

| 파일 | 위치 | 목적 |
|---|---|---|
| `workspace.yaml` | `.solosquad/` | Workspace 메타, 버전 추적 |
| `.env` | `.solosquad/` | 토큰, MESSENGER 등 (기존 루트 `.env` 이동) |
| `.org.yaml` | `<org>/` | Organization 메타 + products 선언 |
| `repo.yaml` | `<org>/<repo>/.solosquad/` | Repository 역할·링크 |
| `_status.yaml` | `<org>/workflows/<id>/` | Workflow 진행 상태 (`target_repo` 필드 포함) |

## 8. Phase 분할

| Phase | 내용 | 예상 시간 |
|---|---|---|
| **1** | 경로 해석 재작성 (`src/util/paths.ts`: `.solosquad/` 탐색, workspace 루트 감지) + `.solosquad/workspace.yaml` 읽기/쓰기 | 4시간 |
| **2** | `solosquad init` 재작성 (workspace → org 온보딩 → messenger → 레이아웃 트리) | 6시간 |
| **3** | `.org.yaml` / `repo.yaml` 스키마 + loader, `solosquad add org`, `solosquad add repo`, `solosquad sync` 구현 | 8시간 |
| **4** | `solosquad bot` / `scheduler` / `status` / `doctor` 경로 로직 일괄 교체 | 6시간 |
| **5** | Cross-repo workflow — `_status.yaml`에 `target_repo` 필드 수용, claude-runner cwd 결정 로직 | 4시간 |
| **6** | 문서 갱신 (setup-guide.md, README.md, 용어 정의 섹션, 다이어그램) | 3시간 |
| **7** | 마이그레이션 (`solosquad migrate`, v1.1.x → v1.2.2) | v1.2.3 참조 |

**총 구현 시간 (마이그레이션 제외):** ~31시간. 실 작업 + QA 포함 1.5~2주 예상.

## 10. 릴리스 전략

- **breaking change** — v1.1.x 사용자는 필수 마이그레이션
- **버전:** `v1.2.2` (사용자 규칙: 3자리, 구조 재편이라도 patch 자리로 유지)
- **배포 순서:**
  1. v1.2.3 마이그레이션 프레임워크를 먼저 구현 (v1.2.2 없이 배포해도 동작; 기존 구조 유지)
  2. v1.2.2 구조 재편 구현 + `solosquad migrate` v1.1.x → v1.2.2 스크립트 포함
  3. 동시 배포: `solosquad@1.2.2`
- **릴리스 노트에 마이그레이션 안내 강제 노출** — 기존 사용자가 `solosquad update` 후 `solosquad migrate`를 반드시 실행하도록 update 명령이 migrate 필요 여부 탐지 후 안내
- **하위 호환:** v1.1.x 구조 감지 시 읽기 전용으로 동작하되 `bot`/`schedule`은 거부하고 migrate 유도

## 11. 릴리스 노트 초안

```
v1.2.2 — 2026-0x-xx   (BREAKING)

Restructure directory hierarchy to match GitHub conventions:
  <workspace-root>/
    .solosquad/                    (system config moved here)
    <organization>/
      .org.yaml
      memory/
      workflows/                   (renamed from `projects/`)
      <repository>/
        .solosquad/repo.yaml

- "Product" → "Organization" (dir). Products are now a concept inside .org.yaml.
- "Project" → "Workflow". Orchestrator tracks workflows per-org.
- REPOS_BASE_PATH env var removed. Everything lives under the workspace root.
- Workspace root name is user-chosen; detected by presence of `.solosquad/`.
- New commands: `add org`, `add repo`, `sync`, `migrate`.
- Cross-repo workflows: workflow `_status.yaml` now carries `target_repo`
  per stage; Claude sessions spawn with repo-level cwd.

MIGRATION: existing v1.1.x users must run `solosquad migrate`. See
docs/v1.2.3-migration-process.md for details.
```
