# v1.2.2 — GitHub-aligned 계층 구조 재편

> Workspace / Product / Project 용어의 혼동을 해소하기 위해 디렉토리 계층을 **GitHub(GitLab) 관례와 일치하는 용어·구조**로 전면 재편한다. 단일 트리 아래에 organization → repository, orchestrator는 workflow 인스턴스를 organization 레벨에서 관리.

**기준 버전:** v1.1.5
**분류:** 구조 재편 — Breaking change
**심각도:** High — 초기 사용자가 구조를 오해해 잘못된 위치에 작업하거나, project 용어를 git 리포지토리와 동일시하는 혼동을 반복 유발. 조기에 재편하지 않으면 이후 확장(멀티 리포, Cross-repo workflow)의 기반이 없어짐.

> **본 문서는 초기 버전의 "용어 정의 + 프롬프트 보강" 스코프에서 확장됨.** 사용자 피드백(2026-04-21)으로 직렬적 개선이 아닌 전면 재편이 필요하다고 판단. 마이그레이션 절차는 별도 문서(`v1.2.3-migration-process.md`)에서 다룸.

---

## 1. 배경 — 사용자 피드백과 설계 결정

### 1.1 초기 사용자 QA에서 반복 제기된 혼동

**Q1.** 프로젝트는 리포지토리 단위인가? `bv-ai-native-po/projects/` 하위에 리포를 clone하면 되는가? 그럼 `bv-ai-native-po`는 깃허브로 치면 organization인가?

**Q2.** `bv-ai-native-po`는 `my-workspace`의 하위인가? 별도 경로에 있으면 어떤 문제가 생기는가? Workspace와 Product를 분리할 필요가 있는가?

### 1.2 설계 결정

- **GitHub 용어와 일치시킨다** — Workspace(루트), Organization, Repository. 개발자가 이미 아는 멘탈 모델로 학습비용 0.
- **"Product"는 디렉토리 계층에서 제거**, `.org.yaml`의 `products:` 필드에 메타데이터로만 유지. 한 조직이 여러 제품을 가질 수 있고, 여러 조직이 한 제품에 기여하는 유연성은 yaml 선언으로 충분.
- **"Project" → "Workflow"** 이름 변경. Orchestrator가 만드는 단계별 협업 인스턴스임을 용어로 명확히.
- **단일 트리 루트** — `REPOS_BASE_PATH` 분리 구조 폐기. 모든 것이 Workspace 루트 아래. Documents 하위에 두는 기본값도 폐기.
- **Workspace 루트 이름 자유화** — `.solosquad/` 하위 폴더 존재로 감지. 기본 이름은 `solosquad`, 사용자가 자유롭게 바꿀 수 있음. 한 머신에 여러 루트 가능(예: 직장인 모드 / 1인 창업가 모드 / 부업 부캐).
- **Organization 이름은 사용자 직접 입력** — URL 파싱·API 호출 없음. 대신 init 프롬프트에서 **구체적 예시로 안내**해 사용자가 자신의 브랜드/팀 단위를 고를 수 있게 함.
- **자동 clone 미지원** — repository는 사용자가 직접 `git clone`/`git init`. 향후 별도 스코프에서 논의.

---

## 2. 목표

1. 사용자가 setup 직후 **한 번에** "내 workspace/organization/repository가 어디에 뭐가 있다"를 이해.
2. 멀티 리포지토리 조직(프론트/백/데이터 분리)을 **네이티브로 수용**.
3. Orchestrator 워크플로우가 repo 경계를 넘나들며 에이전트를 조율할 수 있는 기반 마련.
4. 용어가 IDE/Git 관례와 어긋나지 않아 학습·오해 비용 최소화.

## 3. 새 디렉토리 구조

### 3.1 개요 다이어그램

```
<workspace-root>/                  ← 사용자가 고른 경로·이름 (기본: ~/solosquad)
  .solosquad/                      ← 시스템 설정 — 이 폴더 존재로 workspace 감지
    workspace.yaml                 ← workspace 메타 (display name, version, created_at)
    .env                           ← 토큰, MESSENGER 설정
    agents/ routines/ core/ templates/ orchestrator/
  <organization-slug>/             ← 조직 = GitHub/GitLab org 대응 또는 로컬 그룹
    .org.yaml                      ← 조직 메타 (name, provider, remote_url, products)
    memory/                        ← 조직 레벨 메모리 (JSONL + routine-logs/)
    workflows/                     ← orchestrator가 만드는 워크플로우 인스턴스
      <workflow-id>/
        _status.yaml
        prd.md
        stage-N-<name>/
    slack/  or  discord/  or  telegram/   ← 플랫폼 채널 매핑 (조직 레벨)
    repositories/                  ← ★ 코드 저장소 컨테이너 (v1.2.0)
      <repository-slug>/           ← 실제 git repo
        .git/
        .solosquad/
          repo.yaml                ← role, language, linked_org, notes
        (코드)
      <repository-slug>/           ← 한 조직이 여러 repo 보유 가능
        ...
  <organization-slug>/             ← 한 workspace에 여러 org 가능
    ...
```

> **왜 `repositories/` 중간 계층을 두는가** — org 폴더 안에는 시스템 폴더(`memory/`, `workflows/`, `slack/`)와 코드 저장소가 공존합니다. GitHub/GitLab 의 org/repo 관례(평평 구조)를 그대로 따르지 않은 이유:
>
> 1. GitHub org 페이지에는 **시스템 폴더가 없음** — 순수하게 repo 네임스페이스. 우리 org 폴더는 운영 인프라까지 포함하므로 성격이 다름.
> 2. 피어 프로젝트(OpenClaw, Ralph, Hermes) 조사 결과, 아무도 "시스템 폴더 + 코드 저장소" 를 한 층에 섞어두지 않음. 우리 설계가 독특.
> 3. 시각적 분리 이득 — `ls <org>/` 는 운영 폴더만, `ls <org>/repositories/` 는 코드만 보임.
> 4. 이름 충돌 방지 — 저장소 슬러그가 `memory`, `workflows` 같은 예약어와 부딪힐 위험 제거.
>
> 트레이드오프: `cd <org>/repositories/<repo>` 로 한 단계 길어짐. IDE 자동완성·셸 alias 로 상쇄 가능.

### 3.2 Workspace 루트

- **기본 이름** — `solosquad`. 사용자가 입력을 생략하면 init은 `~/solosquad/`를 제안.
- **변경 자유** — 한 머신에 여러 루트를 둘 때 **역할·도메인 단위로 분리**하는 것이 설계 의도. 예:
  - `~/solosquad/` — 기본 (미설정 시 기본값)
  - `~/solopreneur/` — 1인 창업가 페르소나 (개인 사이드 프로덕트들)
  - `~/elon-24-7/` — "엘론처럼 풀가동" 모드 (메인 운영 중인 조직들 전부)
  - `~/musk-land/` — Musk 산하 전 영역 통합 (Tesla + SpaceX + X + xAI)
  - `~/moonshot/` — 장기 R&D·실험적 문샷 (Neuralink / The Boring Company)

  각 루트는 자기만의 `.solosquad/.env`(토큰·메신저), 오너 프로필, 오가니제이션 셋을 가짐 → Tesla에서 쓰는 Slack 계정이 X 운영 계정과 섞이지 않음.
- **감지 로직** — CLI 실행 시 CWD에서 **위로 올라가며 `.solosquad/` 폴더가 있는 첫 디렉토리**를 workspace 루트로 인식 (npm의 `package.json` 탐색과 동일).
- **다중 루트 전환** — 다른 루트로 이동하려면 `cd`만 하면 됨. 해당 루트의 `.solosquad/.env`가 자동 적용.

**`.solosquad/workspace.yaml` 스키마:**
```yaml
version: "1.2.2"                   # 이 workspace가 처음 생성된 solosquad 버전
display_name: "Solosquad"          # status/logs에 표시되는 이름 (기본: 루트 폴더 이름)
persona: "personal"                # 사용자가 자유롭게 태깅 (personal/work/founder/side 등 아무 문자열)
created_at: "2026-04-22T10:00:00Z"
last_migrated_to: "1.2.2"          # 마지막으로 마이그레이션이 수행된 버전
```

### 3.3 Organization

- **정의** — 여러 repository를 묶는 **브랜드/팀 단위**. GitHub org가 있다면 그 이름과 1:1 매칭, 없으면 사용자가 자유롭게 네이밍. 디렉토리 이름이 그대로 slug 역할.
- **입력 방식** — 사용자가 직접 타이핑. URL 파싱·API 호출 없음. init 프롬프트에서 사용자 경험을 돕기 위해 **구체적 예시를 함께 제시**.

**init 프롬프트 안내 문구 (참고 사례는 실제 1인 창업가의 부캐 분할):**

```
Organization name:
  Your brand, team, or GitHub-org-level grouping.
  One organization holds multiple repositories that share the same
  identity, ownership, or product lineup.

  Examples (Elon Musk's portfolio, for illustration):
    tesla           — EVs, energy, autopilot (https://github.com/teslamotors)
    spacex          — rockets, Starlink
    neuralink       — brain-computer interface
    xai             — Grok, LLM research (https://github.com/xai-org)
    x-platform      — the X social network

  If you already have a GitHub/GitLab organization, reuse that exact
  slug. Otherwise, pick a short name (lowercase, hyphen-separated).

> _
```

- **remote_url은 선택 메타데이터** — `.org.yaml`에 사용자가 직접 기입(없어도 됨). 파싱·자동 fetch 없음.
- **provider 필드** — `github` / `gitlab` / `gitea` / `local` 중 하나. 사용자 선택. 향후 이 값에 따라 repo sync 시 remote 관례가 달라질 수 있음 (예: HTTPS vs SSH clone 명령 제안).

**`.org.yaml` 스키마 (Tesla 예시):**
```yaml
name: "Tesla"                      # 표시 이름 (사용자 입력)
slug: "tesla"                      # 디렉토리명과 일치 (소문자·하이픈)
provider: "github"                 # github / gitlab / gitea / local (사용자 선택)
remote_url: "https://github.com/teslamotors"   # 선택. 없으면 null.
homepage: "https://www.tesla.com"  # 선택. 사용자가 나중에 수동 기입.
products:                          # `.org.yaml`의 products 필드 — 여러 repo를 "어떤 제품을 위해 존재하는가"로 묶음
  - name: "Autopilot"
    slug: "autopilot"
    description: "Full Self-Driving 소프트웨어 스택"
    repos: ["autopilot-vision", "autopilot-planner"]
  - name: "Supercharger Network"
    slug: "supercharger"
    description: "슈퍼차저 충전 인프라"
    repos: ["supercharger-api", "supercharger-mobile"]
description: |
  Tesla — EV, 배터리, 에너지, 자율주행.
created_at: "2026-04-22T10:00:00Z"
```

### 3.4 Repository

- **생성 방식** — **사용자가 직접 `git clone` 또는 `git init`**. 자동 clone 없음.
- **등록** — 조직 디렉토리 안에 repo 폴더가 추가된 후 `solosquad sync`(신규 명령)가 `.solosquad/repo.yaml`을 자동 생성 (기존에 없을 때).
- **역할 선언** — `repo.yaml`의 `role` 필드로 frontend/backend/data/infra/docs 등 분류.

**`<org>/<repo>/.solosquad/repo.yaml` 스키마 (Tesla Autopilot 예시):**
```yaml
slug: "autopilot-vision"
name: "Autopilot Vision Stack"
role: "data"                       # main / frontend / backend / data / infra / docs / unknown
language: "python"                 # 자동 감지 (package.json, pyproject.toml 등으로)
linked_org: "tesla"                # 상위 org slug와 일치해야 함
remote_url: "https://github.com/teslamotors/autopilot-vision"   # git remote에서 추출
products: ["autopilot"]            # 이 repo가 구현하는 제품 slug (0개 이상)
notes: |
  Free-text notes visible to agents.
registered_at: "2026-04-22T10:00:00Z"
```

### 3.5 Workflow (기존 Project)

- **위치** — `<org>/workflows/<workflow-id>/`
- **내부 구조**는 기존 `projects/{id}/`와 동일 (`_status.yaml`, `prd.md`, `sessions/`, `stage-N-*/`, `_handoff.md` 등)
- **Cross-repo 지원** — `_status.yaml`의 각 stage에 `target_repo` 필드 추가 (4장 참조)

### 3.6 Messenger의 계층 위치

**원칙: 한 workspace = 한 메신저 플랫폼 = 한 봇 계정.** 단순한 정신 모델을 우선시.

메신저 관련 정보는 Workspace와 Organization 두 계층에 분산:

| 정보 | 위치 | 계층 | 역할 |
|---|---|---|---|
| 봇 토큰 (`DISCORD_TOKEN` / `SLACK_BOT_TOKEN`+`SLACK_APP_TOKEN` / `TELEGRAM_BOT_TOKEN`) | `.solosquad/.env` | **Workspace** | 이 workspace가 쓰는 봇 계정 한 세트 |
| 메신저 선택 (`MESSENGER=slack`) | `.solosquad/.env` | **Workspace** | **단일 값만 허용** — `discord,slack` 같은 복수 지정 불허 |
| 명령 채널 이름 (`SLACK_COMMAND_CHANNEL`) | `.solosquad/.env` | **Workspace** | 봇이 리스닝할 채널명 (workspace 전역 1개) |
| 채널 ID ↔ org 매핑 | `<org>/slack/config.yaml` (또는 `<org>/discord/`, `<org>/telegram/`) | **Organization** | 루틴 결과를 각 org의 어느 채널로 내보낼지 |

**Repository / Workflow 레벨에는 메신저 설정이 없다.** 메시지 수신 시 활성 org가 먼저 결정되고, 그 안에서 workflow·target_repo가 후속 판정됨.

#### 설계 의도

- **단일 메신저·단일 봇**이면 설정·운영·디버깅이 모두 단순. 루틴 결과 전송, 에러 추적, 토큰 교체 모두 1개 플랫폼만 관리하면 됨.
- 여러 플랫폼(Slack + Discord)을 동시에 쓰려는 사용자는 **별도 workspace**로 분리. 예: `~/solopreneur` 은 Slack 봇, `~/elon-24-7` 은 Discord 봇. 이것이 워크스페이스 페르소나 분리의 실질적 이유 중 하나.
- 한 workspace 내 여러 org가 공존하는 경우, 각 org는 **명령 채널을 공유하거나** 서로 다른 채널을 쓸 수 있음(v1.2.2 이후 확장 여지). 초기 v1.2.2는 **워크스페이스당 1 org + 1 명령 채널**을 기본 권장 경로로 삼음.
- 멀티 메신저 지원(v1.1.x의 `MESSENGER=discord,slack`)은 **v1.2.2에서 제거**. 필요한 사용자는 마이그레이션 시 2개 workspace로 자동 분기하거나 수동 분할을 안내받음.

## 4. Cross-Repository Workflow 모델

### 4.1 원칙

- **Workflow 상태는 조직 레벨에서 중앙 집중** (`<org>/workflows/<id>/_status.yaml`)
- **Claude Code 세션은 stage별 `target_repo`의 cwd로 실행** — 즉 단일 repo 안에서만 편집/commit
- **Handoff에 repo 간 연계 힌트 기록** — 다음 stage가 참조해야 할 변경 지점을 `_handoff.md`에 명시

### 4.2 `_status.yaml` 확장 예시

```yaml
# Tesla 조직의 Model Y 리프레시 워크플로우
workflow_id: "model-y-refresh"
project: "autopilot"                # 어떤 product의 워크플로우인지 (.org.yaml의 products slug)
created_at: "2026-04-22T11:00:00Z"
stages:
  - id: stage-1-vision
    team: engineering
    agents: ["data-engineer", "api-developer"]
    target_repo: "autopilot-vision"
    status: in_progress
    depends_on: []

  - id: stage-2-planner
    team: engineering
    agents: ["backend-developer"]
    target_repo: "autopilot-planner"
    status: pending
    depends_on: ["stage-1-vision"]
    upstream_handoff: "workflows/model-y-refresh/stage-1-vision/_handoff.md"

  - id: stage-3-hmi
    team: engineering
    agents: ["creative-frontend"]
    target_repo: "model-y-hmi"
    status: pending
    depends_on: ["stage-1-vision"]  # 병렬 진행 가능
```

### 4.3 Claude 세션 spawn 규칙

`src/bot/index.ts` 변경:

```ts
// 기존
const productDir = path.join(getReposBase(), product.slug);

// 새 구조
const orgDir = path.join(getWorkspaceRoot(), org.slug);
const workflowStage = await resolveActiveStage(orgDir, workflowId);
const repoDir = path.join(orgDir, workflowStage.target_repo);
const result = await runClaude(prompt, repoDir);
```

- 세션 cwd는 **repo 단위**로 제한 → safety preamble의 `.env`·`credentials` 차단 그대로 작동
- 다른 repo의 코드를 "읽기"만 해야 하는 경우, **stage 진입 시 필요한 파일을 workflow 폴더로 스냅샷 복사** → Claude에게 컨텍스트 주입. 쓰기는 절대 target_repo에만.

### 4.4 미해결 / 차기 스코프

- **의존 repo 간 버전 정합성** — 백엔드 API 변경 + 프론트 대응이 동시 머지되어야 할 때 PR 타이밍 조율. v1.2.2에서는 **사람이 조율**, 자동화는 차기.
- **Monorepo 감지** — 하나의 repo 안에 `apps/frontend`, `apps/backend`가 있는 경우. v1.2.2는 **repo 단위만 지원**.
- **부분 실패 복구** — stage-2 실패 시 stage-1 commit 롤백 여부. v1.2.2는 **forward-only**, 롤백은 사용자가 수동.

## 5. 용어 정의 (최종)

| 용어 | 정의 | 예시 |
|---|---|---|
| **Workspace** | `.solosquad/` 하위 폴더를 가진 사용자 지정 루트 디렉토리. 기본 이름 `solosquad`. 역할·도메인 단위 분리를 위해 여러 개 가능. | `~/solosquad/`(기본), `~/solopreneur/`(1인 창업), `~/elon-24-7/`(풀가동 모드), `~/musk-land/`(Musk 산하 통합), `~/moonshot/`(장기 R&D) |
| **Organization** | 여러 repository를 묶는 브랜드/팀 단위. GitHub/GitLab org와 1:1. 없으면 사용자가 자유롭게 네이밍. | `tesla`, `spacex`, `neuralink`, `xai`, `x-platform` |
| **Repository** | 실제 git 리포지토리(clone/init). 코드 편집·commit의 단위. | `autopilot-vision`, `starship-firmware`, `grok-training` |
| **Workflow** | Orchestrator가 만드는 단계별 협업 인스턴스. PRD + handoff + 산출물. | `model-y-refresh`, `starship-v3-controller`, `grok-fine-tune` |
| **Product** | 한 개 이상의 repository가 구현하는 제품/서비스 **개념**. 디렉토리가 아닌 `.org.yaml`의 `products:` 항목. | `autopilot`, `supercharger`, `starlink`, `grok` |

## 10. 릴리스 전략

- **breaking change** — v1.1.x 사용자는 필수 마이그레이션
- **npm 버전:** `v1.2.0` — 1.1.5 다음 첫 minor 점프. 본 문서의 파일명(`v1.2.2-...`)은 작업 블록 라벨일 뿐, 실제 배포는 `v1.2.0`.
- **배포 구성:** 구조 재편(본 문서) + 마이그레이션 프레임워크(`v1.2.3-migration-process.md`) + v1.1.x→v1.2.0 마이그레이션 스크립트를 한 릴리스에 통합.
- **릴리스 노트에 마이그레이션 안내 강제 노출** — 기존 사용자가 `solosquad update` 후 `solosquad migrate`를 반드시 실행하도록 update 명령이 migrate 필요 여부 탐지 후 안내.
- **하위 호환:** v1.1.x 구조 감지 시 `doctor`가 migrate 유도.
