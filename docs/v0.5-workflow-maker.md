# v0.5 — 워크플로우 메이커 & 스킬 시스템 통합

> 솔로 파운더가 메신저에서 PM과 대화하며 자신의 지식·암묵지를 SKILL.md + 워크플로우 체인으로 자동 변환한다. 기존 `.claude/skills/`를 가진 코드베이스도 동일 파이프라인으로 흡수. v0.3 PM 모드를 "사용자 의도 → 실행 인프라" 컴파일러로 확장.

**로드맵:** v0.1 크로스 플랫폼 → v0.2 안전/보안 → v0.3 PM 모드 → v0.4 자율 실행 → **v0.5 워크플로우 메이커** → v0.6 디폴트 워크플로우 → v1.1 대시보드 상호작용 → v1.2 지식 온톨로지 → v2.0 커뮤니티 → v3.0 코워킹

**의존:** v0.3 (PM 모드 — 사용자 대화 진행자 + Task tool 흉내), v0.4 (자율 실행 — 새로 만든 워크플로우를 program.md 파이프라인에 배치)

**기준 버전:** v0.5.0

**통합 출처:** 본 스펙은 이전 별도 트랙이었던 "스킬 분석기 (구 v0.3/v0.4)" + "스킬 자유도 (구 v0.6)"를 단일 v0.5로 합친 결과물. 두 트랙이 결국 같은 SKILL.md 프론트매터 스키마와 동일 라우팅 레이어를 공유하기 때문에 통합 시 표면적이 절반으로 줄어듦.

## 핵심 질문 & 성공 기준

| 핵심 질문 | 성공 기준 |
|---|---|
| 사용자가 코드를 안 만지고 메신저 대화만으로 새 skill·워크플로우를 등록할 수 있는가? | "이런 작업 자동화하고 싶다" → PM 명확화 1-2턴 → SKILL.md/workflow yaml draft → 승인 → 다음 호출부터 즉시 사용 가능 |
| 기존 `.claude/skills/` 자산을 잃지 않고 흡수할 수 있는가? | `solosquad analyze repo <path>` → 4-라벨 분류 보고서 → `add repo --from-report` 적용 → workspace agent와 충돌 없이 머지 |
| 자동 생성된 skill이 손으로 쓴 것과 동일 품질·동일 포맷인가? | Anthropic Agent Skills 공식 포맷 100% 호환, 라우터·PM이 양쪽을 구분 못 함, validator 통과 |
| 결정성이 깨지지 않는가? | 대화형 author 루프와 분석은 LLM 사용, `sync`·`migrate`·라우팅 적용은 LLM-free 유지 |

## 1. 현재 문제

| 문제 | 원인 |
|---|---|
| 에이전트 25개 고정 | 새 skill 추가 시 코드 수정 + AGENT_ROUTES 키워드 수동 등록 필요 |

## 2. 원칙

| 원칙 | 구현 |
|---|---|
| **메신저-네이티브 author 루프** | PM이 LLMREI(요구공학 LLM 면담) 스타일로 1-2턴 명확화 → draft 제시 → 사용자 확정. GUI 없이 메신저 thread만으로 완결 |
| **Anthropic Agent Skills 공식 포맷** | YAML frontmatter (`name`, `description`, optional `triggers`/`inputs`/`outputs`/`handoff_to`) + Markdown body. Progressive disclosure (본문은 짧게, 상세는 references 폴더) |
| **Skill that authors skills** | Superpowers/Plugin Developer Toolkit 패턴 차용. 메타-skill(`assets/agents/_meta/workflow-maker/SKILL.md`)이 새 skill을 생성하는 책임 단일 보유 |
| **3-tier 검색 경로** | 프로젝트 로컬 `./.agents/` > 사용자 글로벌 `~/.solosquad/agents/` > 번들 `assets/agents/`. 상위 계층이 override |
| **Deterministic vs LLM 분리** | `sync`·`migrate`·라우팅 등록은 LLM 금지. `analyze`·메신저 author 루프만 LLM. 분류 결과 → `applier`가 결정적 머지 |
| **사람 승인 게이트** | author 루프 끝에 "확인하시겠습니까?"·`analyze` 끝에 Markdown 보고서. 자동 적용 없음 |
| **Migration by hash** | repo 분석 ledger에 `path + SHA256[:12]` 기록. 두 번째 실행은 신규/변경분만 LLM 호출 |

## 3. 두 흐름 = 같은 컴파일 타깃

```
                ┌─────────────────────────────────┐
사용자 대화 ──→ │  Workflow Maker (메타-skill)    │
                │  - 명확화 질문                  │
                │  - SKILL.md 생성                │
                │  - workflow yaml 생성           │
                └────────┬────────────────────────┘
                         │
                         ▼
                ┌─────────────────────────────────┐
                │  SKILL.md (Anthropic 포맷)      │
                │  + workflow.yaml                │  ← 단일 컴파일 타깃
                └────────▲────────────────────────┘
                         │
                ┌────────┴────────────────────────┐
기존 repo  ──→ │  Repo Analyzer                  │
.claude/      │  - 4-label 분류 (LLM)           │
skills/       │  - workflow 매칭 (결정적)       │
              │  - applier (결정적 머지)        │
              └─────────────────────────────────┘
```

**핵심**: author 루프와 analyzer는 *입력 형태*만 다르고 *출력 형태와 저장 경로*는 동일하다. 따라서 검증·라우팅·렌더링 로직은 단일.

## 4. SKILL.md 스키마 (Anthropic Agent Skills 호환)

```markdown
---
name: "Custom Analyst"
description: "도메인 특화 데이터 분석 — 부동산 등기부 신호 추출"
team: strategy                          # SoloSquad 확장 필드
triggers:                               # SoloSquad 확장 — 라우팅 키워드
  - "커스텀 분석"
  - "custom analysis"
inputs:
  required: [data_source, analysis_goal]
  optional: [time_range]
outputs:
  - custom_report.md
handoff_to:                             # 다음 단계 후보
  - feature-planner
  - pmf-planner
scope: agent                            # workspace | org | repo
confidence: 1.0                         # author 루프 산출 시 1.0, analyzer 시 0.x
source: messenger-author-2026-05-12     # author 출처 추적
---

# Custom Analyst Agent

> 1-2 단락 정체성 정의.

## Process
...
```

**필수 필드:** `name`, `description` (Anthropic 사양). 나머지는 SoloSquad 확장이며 누락 시 라우터가 fallback 처리.

**Progressive disclosure:** 본문은 짧게 유지, 상세 절차는 같은 폴더의 `references/*.md`로 분리. Anthropic Skill Authoring Best Practices 준수.

## 5. 메신저 Author 루프 (LLMREI 스타일)

### 5.1 트리거

사용자: "owner-command에 '이런 작업 자주 하니까 자동화하고 싶다 — 매주 등기부 알림 보고서 만들어줘'"

PM이 의도를 "신규 skill 또는 workflow 생성"으로 분류 → meta-skill 로드.

### 5.2 명확화 (1-2턴)

```
PM: 이 작업을 자동화하려고 합니다. 몇 가지만 확인할게요.
    1. 입력은 무엇인가요? (예: 등기부 API 응답 / 수동 입력 데이터 / 다른 에이전트 출력)
    2. 출력 형태는? (예: 메신저 알림 / Markdown 리포트 / JSONL)
    3. 매주 자동 실행되어야 하나요, 아니면 사용자가 트리거해야 하나요?

사용자: 1) 등기부 OpenAPI 응답, 2) 메신저 알림 + Markdown, 3) 매주 월요일 자동
```

명확화는 최대 2턴. 사용자가 답 못 하면 default 채택 + 추후 수정 가능 명시.

### 5.3 Draft 제시 → 확정

PM이 SKILL.md + (선택적으로) workflow.yaml + (선택적으로) 라우틴 cron entry draft를 thread에 표시.

```
PM: 아래로 생성하려고 합니다. 확정할까요?

    [SKILL.md preview]
    ---
    name: Real Estate Watcher
    triggers: [등기부 알림, 부동산 모니터링]
    ...
    
    [routine entry]
    cron: "0 9 * * MON"  # 매주 월요일 09:00
    
    저장 위치: org/<org>/agents/data/real-estate-watcher/SKILL.md
    
    승인 (y/N):
```

사용자 `y` → applier가 결정적으로 파일 저장 + 라우터 reload + 메모리 시드.

### 5.4 검증

저장 직후 `solosquad agent validate <path>` 자동 실행 — frontmatter 스키마, 필수 섹션, 충돌 검사. 실패 시 PM이 사용자에게 보고 + rollback 옵션 제시.

### 5.5 Dry-run sandbox (선택)

Claude Agent SDK로 새 skill을 격리 세션에서 1회 호출 → 출력 미리보기 → 사용자가 본 출력 확인 후 production 등록. 대화 thread 추가 1턴.

## 6. 기존 Repo 마이그레이션 (analyze)

### 6.1 CLI 흐름

```bash
$ solosquad analyze repo /path/to/legacy-repo
Scanning .claude/skills/ ...                       12 files found
Loading analysis ledger ...                        (none — first run)
Analyzing (LLM, batched) ...
  [1/12] skills/deploy-staging.md      → codebase-fact (0.92)
  [2/12] skills/pm-review.md           → role          (0.81)
  [3/12] skills/prd-to-impl.md         → workflow      (0.88)
  ...
Computing workflow match ...                       Feature Expansion (0.87)

✓ Report: .solosquad/analysis/2026-05-12-initial.md
✓ Ledger: .solosquad/analysis-ledger.yaml

$ solosquad add repo /path/to/legacy-repo --from-report 2026-05-12-initial.md
```

### 6.2 4-라벨 분류

| 라벨 | 시그널 | 이주 대상 |
|---|---|---|
| **role** | "Act as X", 페르소나·추론 스타일 | Workspace agent (25개 중 머지 또는 신규) |
| **workflow** | 스테이지 핸드오프 체인 (A→B→C) | Org `<org>/workflows/` |
| **codebase-fact** | 특정 파일·경로·배포 타깃 | Repo `.claude/skills/` 유지 |
| **domain** | 제품·비즈니스 룰·도메인 사전 | Repo 유지 (cross-repo면 org) |

다중 후보 시 우선순위: `codebase-fact > domain > workflow > role` (구체 → 추상). Confidence < 0.7이면 `ambiguous` 마킹 → 사람 판정 필수.

### 6.3 Workflow 매칭 (결정적, LLM 없음)

분류 결과의 집계 함수. 4종 SoloSquad 템플릿 (PMF / Feature / Rebranding / Rapid Prototype)에 매핑 → 최고 cover_rate 추천. 모든 cover_rate < 0.5면 "no-match" → 커스텀 workflow 권유.

### 6.4 Ledger (증분 처리)

`<repo>/.solosquad/analysis-ledger.yaml`:

```yaml
version: 1
analyzed:
  - path: .claude/skills/deploy-staging.md
    hash: a3f1c8e2b9d4
    classification: codebase-fact
    confidence: 0.92
    destination: repo (stay)
    applied: false
workflow_match:
  template: feature-expansion
  cover_rate: 0.87
model:
  fingerprint: "opus-4-7-2026-04"
```

두 번째 실행은 hash 변경된 파일만 재분석. workflow 매칭만 다시 계산하는 경로는 LLM-free.

### 6.5 Applier — 결정적 적용

`add repo --from-report` 흐름:
1. `~/.solosquad-backups/<ISO8601>-repo-onboard/`에 백업 (v0.2.3 프레임워크 재사용)
2. role 라벨 → workspace agent 머지 (기본 `append section` 전략, `--merge-policy override|replace`로 변경 가능)
3. workflow 라벨 → org workflows로 이동
4. codebase-fact / domain → repo 유지
5. ledger `applied: true` 마킹
6. Verify (라우터 reload + 충돌 재검사)

실패 시 `solosquad migrate --rollback`으로 백업 복원.

## 7. 3-tier 검색 경로 + 자동 라우팅

```
1. ./.agents/{team}/{agent}/SKILL.md          → 프로젝트 로컬 (최우선)
2. ~/.solosquad/agents/{team}/{agent}/SKILL.md → 사용자 글로벌
3. assets/agents/{team}/{agent}/SKILL.md       → 번들 기본
```

같은 이름이면 상위 계층 override. `AGENT_ROUTES` 하드코딩 제거 → `buildRoutes()`가 시작 시 3-tier 스캔 + frontmatter `triggers` 파싱.

```typescript
// src/bot/agent-router.ts (재작성)
export async function buildRoutes(): Promise<Record<string, AgentRef>> {
  const routes: Record<string, AgentRef> = {};
  for (const tier of [bundleRoot, userGlobal, projectLocal]) {
    for (const skill of await scanSkills(tier)) {
      const fm = parseFrontmatter(skill);
      for (const trigger of fm.triggers ?? []) {
        routes[trigger] = { team: fm.team, agent: fm.name, source: tier };
      }
    }
  }
  return routes;
}
```

**Hot-reload**: meta-skill이 새 SKILL.md를 저장하면 `bot/index.ts`가 라우터를 즉시 재빌드. 봇 재시작 불필요.

## 8. 신규 CLI

```bash
# 대화형 author는 메신저 기본, CLI는 보조
solosquad agent add --name custom-analyst --team strategy   # 스캐폴딩
solosquad agent list                                          # 번들+글로벌+로컬 모두 표시
solosquad agent info <name>                                   # frontmatter + 소스 경로
solosquad agent validate <path>                               # 스키마 + 필수 섹션 검증

# 기존 repo 흡수
solosquad analyze repo <path>                                 # LLM 분류 + Markdown 보고서
solosquad analyze repo <path> --force                         # 모델 업그레이드 시 전체 재분석
solosquad analyze repo <path> --prune-orphans                 # 파일 사라진 ledger 항목 정리
solosquad add repo <path> --from-report <report.md>           # 적용 (backup → verify → apply)
solosquad add repo <path> --merge-policy append|override|replace
```

## 11. 테스트 전략

### 11.1 Fixture 기반 분석 회귀

```
test/analyze/fixtures/
  empty-skills/                     # nothing to do
  pure-codebase-facts/              # 이주 0건
  mixed-typical/                    # 4 라벨 골고루
  ambiguous-heavy/                  # 사람 개입 강조
  matches-feature-expansion/        # cover_rate ≥ 0.85
  no-workflow-match/                # 커스텀 권유
```

분류 안정성: 같은 fixture 100회 분류 시 라벨 일치율 ≥ 95%. 모델 업그레이드 시 fixture 재실행 + diff 리포트.

### 11.2 Author 루프 E2E

- PM이 단일 메신저 메시지 1건 → 명확화 1턴 → draft → 확정 → SKILL.md 저장 → 라우터에서 발견됨
- Validator 실패 시 rollback
- Dry-run sandbox 실행 + 출력이 사용자 thread에 표시
- Hot-reload — 봇 재시작 없이 새 trigger가 라우터에 잡힘

### 11.3 회귀

- `analyze` 두 번째 실행 시 LLM 호출 0회 (ledger 정확성)
- `add repo --from-report` 적용 후 `applied: true`
- `migrate`·`sync`에는 LLM 호출 없음 (deterministic 보장)

## 12. Out of scope (v0.5에서 다루지 않음)

- **자동 trigger 생성** — `triggers` 키워드를 LLM이 추론해서 자동 채워주는 건 v0.6에서 디폴트 워크플로우 튜닝과 함께 결정
- **메타-skill 자체 자동 진화** — author 루프 자체를 사용자 피드백으로 개선하는 건 v2.0
- **타 워크스페이스 skill 공유 / 마켓플레이스** — Claude Code Plugins 형태로 패키징·공유는 v2.0
- **온톨로지 연계** — author 루프 산출물의 의미 태깅·교차 참조는 v1.2 (지식 온톨로지)
- **GUI** — author 루프는 메신저-네이티브. GUI는 v1.1 대시보드 상호작용의 read-only 뷰만
