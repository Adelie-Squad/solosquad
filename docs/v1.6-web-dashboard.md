# v1.6 — 웹 대시보드

> 솔로 파운더의 AI 팀 운영 현황을 한눈에 파악하는 웹 대시보드. CLI 모니터링의 시각적 확장.

**로드맵:** v1.1 크로스 플랫폼 → v1.2 안전/보안 → v1.3 자율 실행 → v1.4 스킬 자유도 → v1.5 지식 온톨로지 → **v1.6 웹 대시보드** → v2.0 커뮤니티 → v3.0 코워킹

**의존:** v1.3 (자율 실행 — results.tsv 시각화), v1.5 (지식 온톨로지 — Knowledge Explorer)

---

## 1. 왜 대시보드인가

### 현재 모니터링 수단의 한계

| 수단 | 한계 |
|------|------|
| `solosquad status` CLI | 터미널 전용. 시각화 없음. 로컬에서만 확인 |
| 메신저 채널 보고 | 텍스트 기반. 전체 현황 조망 어려움. 히스토리 검색 불편 |
| JSONL 메모리 파일 | 직접 열어봐야 함. 트렌드/패턴 파악 불가 |
| results.tsv (v1.1) | 텍스트 테이블. 메트릭 추이 시각화 불가 |

### 대시보드가 해결하는 것

| 니즈 | 해결 |
|------|------|
| "밤새 돌린 결과가 어떻게 됐지?" | 실험 타임라인 + keep/discard 요약 |
| "이번 주 어떤 시그널이 잡혔지?" | 시그널 피드 + 필터링 + 트렌드 |
| "프로젝트 전체 진행률은?" | 프로젝트 파이프라인 뷰 + 스테이지 상태 |
| "에이전트들이 뭘 했지?" | 에이전트 활동 로그 + 지식 축적 현황 |
| "모바일에서 확인하고 싶다" | 반응형 웹 → 어디서든 접근 |

## 2. 리포지토리 구조

### 분리 근거

| 이유 | 설명 |
|------|------|
| 배포 단위 상이 | CLI = `npm install -g`, 웹 = Vercel 호스팅 |
| 의존성 오염 방지 | React/Next.js + DB가 CLI npm 패키지에 포함되면 안됨 |
| 릴리즈 주기 독립 | CLI 기능 추가와 웹 UI 수정은 비동기적 |

### 리포 맵

```
Organization: Adelie-Squad/
├── solopreneur-team-agents   ← CLI npm 패키지 (현재)
├── solopreneur-dashboard     ← 웹 대시보드 (v1.2)
└── solopreneur-api           ← API 서버 (v1.2 Phase 2)
```

### 독립 사용 보장

| 조합 | 가능 여부 | 동기화 |
|------|----------|--------|
| CLI만 (웹 없이) | O | 현재와 동일 |
| 웹만 (CLI 없이) | O | 웹에서 직접 데이터 입력/조회 |
| CLI + 웹 | O | API로 자동 동기화 |

## 4. 데이터 아키텍처

### 4.1 데이터 흐름

```
Phase 1 (Git Pull):
  CLI → Git push → Dashboard가 Git에서 읽기

Phase 2 (API Push):
  CLI → API → DB → Dashboard
       ↑ 루틴/실험 완료 시 자동 push
```

**Phase 1은 API 없이 빠르게 시작.** CLI가 Git push한 파일을 대시보드가 직접 파싱.

### 4.2 데이터 소스 매핑 (Phase 1)

| 대시보드 페이지 | 데이터 소스 | 파싱 대상 |
|---------------|-----------|----------|
| Command Center | `_status.yaml` + routine-logs/ | YAML + JSONL |
| Product Overview | `_status.yaml` + `_handoff.md` | YAML + Markdown |
| Experiments | `results.tsv` | TSV |
| Signals | `memory/signals.jsonl` | JSONL |
| Memory Explorer | `memory/*.jsonl` | JSONL |
| Agent Activity | `results.tsv` (agent 컬럼 집계) | TSV |
| Knowledge Explorer | `knowledge/**/*.md` | Markdown + YAML frontmatter |

### 4.3 데이터 모델 (Phase 2 DB)

```
User
├── id, name, email, avatar
└── products: Product[]

Product
├── id, slug, name, brief
├── repoUrl                        ← Git 리포 연결
├── projects: Project[]
├── routineLogs: RoutineLog[]
└── memory: MemoryEntry[]

Project
├── id, name, workflowType         ← PMF Discovery, Feature Exp, etc.
├── status: _status.yaml 구조
├── programMd: string               ← 자율 실행 프로그램
├── experiments: Experiment[]
└── stages: Stage[]

Stage
├── id, name, order
├── status: pending | in_progress | completed | needs_revision
├── assignedAgent: string
├── handoff: Handoff | null
└── artifacts: string[]

Experiment
├── id, cycle, timestamp
├── agent, metric, value
├── status: keep | discard | crash
├── commit, description
└── projectId

RoutineLog
├── id, routineId, executedAt
├── result: string (markdown)
└── extractedData: json

MemoryEntry
├── id, type: signal | experiment | decision
├── data: json
├── createdAt
└── productId

Knowledge
├── id, type: pattern | anti-pattern | heuristic | decision | learning
├── scope: team | agent | project
├── team, agent (nullable)
├── tags: string[]
├── confidence: float
├── content: string (markdown)
├── source: string
└── createdAt, updatedAt
```

### 4.4 API 엔드포인트 (Phase 2)

```
── 동기화 (CLI → API) ──
POST   /api/sync/routine-log         루틴 결과 push
POST   /api/sync/experiment          실험 결과 push (results.tsv 행)
POST   /api/sync/memory              메모리 항목 push
POST   /api/sync/knowledge           지식 항목 push
POST   /api/sync/status              프로젝트 상태 push (_status.yaml)

── 조회 (Dashboard → API) ──
GET    /api/products                 제품 목록
GET    /api/products/:slug           제품 상세
GET    /api/products/:slug/projects  프로젝트 목록
GET    /api/products/:slug/experiments?metric=&since=  실험 이력
GET    /api/products/:slug/signals?tags=&since=        시그널 피드
GET    /api/products/:slug/memory?type=                메모리 항목
GET    /api/products/:slug/knowledge?scope=&agent=     지식 검색
GET    /api/products/:slug/agents/activity             에이전트 활동 집계

── 관리 ──
POST   /api/products                 제품 등록
PATCH  /api/products/:slug           제품 수정
POST   /api/knowledge/:id/promote    지식 승격 승인
```


```
DB:          Supabase (PostgreSQL + Auth + Realtime)

실시간:      Supabase Realtime (WebSocket)
배포:        Vercel Edge Functions 또는 Cloudflare Workers
```

## 6. 구현 계획

### Phase 1: 정적 대시보드 (Git Pull 방식)

API 없이, 샘플 데이터 또는 Git 리포 파일을 직접 파싱하여 대시보드 구현.

| # | 작업 | 산출물 |
|---|------|--------|
