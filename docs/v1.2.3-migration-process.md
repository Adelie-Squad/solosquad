# v1.2.3 — 마이그레이션 프레임워크

> 구조·용어·스펙 업그레이드가 이후에도 반복될 것을 전제로, **버전 간 workspace 이관**을 안전·예측가능·되돌릴 수 있게 만드는 범용 프레임워크. v1.1.x → v1.2.2 이관을 첫 구체 케이스로 포함하되, 앞으로 모든 업그레이드에 재사용.

**기준 버전:** v1.2.2 (이 문서의 결과물은 v1.2.2와 동시 배포)
**분류:** 기반 인프라 — 사용자 경험 + 운영 안정성
**심각도:** High — 마이그레이션 프로세스가 취약하면 사용자 데이터 유실·롤백 불가로 신뢰도 급락

---

## 1. 목적

1. **사용자 신뢰 보존** — 업그레이드가 두렵지 않아야 한다. 기존 데이터는 반드시 보존되고, 문제 발생 시 즉시 되돌릴 수 있어야 함.
2. **재사용 가능한 구조** — v1.1.x→v1.2.2뿐 아니라 v1.2.2→v1.3.0, v1.3.0→v1.4.0 등 모든 전환이 동일한 틀로 처리.
3. **기여자에게 명확한 규약** — 새 버전에서 구조를 바꿀 때 "마이그레이션 스크립트를 어디에 어떻게 추가하는지" 모호함 제거.
4. **CI/QA 회귀 방지** — 실제 사용자 데이터와 유사한 픽스처로 전환 테스트 자동화.

## 2. 원칙

| 원칙 | 구현 방식 |
|---|---|
| **Detect-first** | 실행 시 현재 workspace의 버전·구조를 자동 감지. 사용자가 버전을 직접 지정할 필요 없음. |
| **Dry-run by default** | `solosquad migrate`의 기본 동작은 "무엇이 바뀔지 보여주기". 실제 변경은 `--apply` 플래그 필요. |
| **Backup before mutation** | Apply 모드는 실제 이동/변경 전에 workspace 전체 스냅샷을 자동 저장. |
| **Idempotent** | 이미 마이그레이션된 workspace에 다시 실행해도 no-op 혹은 안전하게 재확인만. |
| **Reversible when feasible** | Backup에서 복원하는 `solosquad migrate --rollback`. 불가능한 단방향 변경은 사전에 명시 경고. |
| **Clear communication** | 진행 상황을 단계별로 출력, 완료 후 요약, 실패 시 구체 에러 + 다음 조치. |
| **Versioned** | 각 전환은 독립된 스크립트 모듈로 존재. 체이닝(`1.1→1.2→1.3`)은 프레임워크가 자동 수행. |

## 4. 프레임워크 구조 (코드 레벨)

### 4.1 디렉토리 배치

```
src/
  migrations/
    index.ts                      # registry: Map<version-pair, MigrationScript>
    types.ts                      # Migration, MigrationPlan, MigrationStep 타입
    runner.ts                     # detect / plan / backup / apply / verify / rollback
    detect.ts                     # workspace 구조 → 버전 식별
    backup.ts                     # tar/zip로 workspace 스냅샷
    scripts/
      1.1.x-to-1.2.2.ts           # 첫 구체 전환 스크립트
      1.2.2-to-1.3.0.ts           # (나중에 추가)
      ...
  cli/
    migrate.ts                    # CLI 엔트리포인트 (commander 서브커맨드)
```

### 4.2 Migration 인터페이스

```typescript
// src/migrations/types.ts
export interface Migration {
  from: string;                   // "1.1.x" | "1.1.5" — 정규식 지원
  to: string;                     // "1.2.2"
  description: string;            // 사람이 읽는 요약
  detect(workspace: string): Promise<boolean>;   // 이 소스 버전에 해당하는지
  plan(workspace: string): Promise<MigrationPlan>;   // dry-run 결과
  apply(workspace: string, plan: MigrationPlan): Promise<void>;
  verify(workspace: string): Promise<VerifyResult>;  // apply 후 상태 확인
  // rollback은 backup.ts가 범용으로 처리; 스크립트는 forward-only면 OK
}

export interface MigrationPlan {
  steps: MigrationStep[];
  warnings: string[];             // 사용자에게 명시 노출
  irreversible_changes: string[]; // 롤백 불가한 변경 명시
  estimated_disk_delta_mb: number;
}

export interface MigrationStep {
  kind: "move" | "rename" | "generate" | "remove" | "update";
  from?: string;
  to?: string;
  description: string;
  payload?: unknown;              // kind에 따라 다른 데이터
}
```

### 4.3 Runner 동작

```typescript
// src/migrations/runner.ts (요약)
export async function migrate(opts: {
  workspace: string;
  targetVersion: string;
  dryRun: boolean;
  applyConfirmed: boolean;
  backupDir: string;
}) {
  const sourceVersion = await detectWorkspaceVersion(opts.workspace);
  const chain = resolveMigrationChain(sourceVersion, opts.targetVersion);

  // 1. Plan every step
  const plans = [];
  for (const m of chain) {
    plans.push({ migration: m, plan: await m.plan(opts.workspace) });
  }
  printPlans(plans);

  if (opts.dryRun) return;

  // 2. Backup
  const backupPath = await createBackup(opts.workspace, opts.backupDir);
  console.log(`Backup: ${backupPath}`);

  // 3. Apply sequentially with verify
  for (const { migration, plan } of plans) {
    await migration.apply(opts.workspace, plan);
    const result = await migration.verify(opts.workspace);
    if (!result.ok) {
      console.error(`Verify failed at ${migration.to}: ${result.error}`);
      console.log(`Rollback available: solosquad migrate --rollback`);
      process.exit(1);
    }
  }

  // 4. Write new workspace.yaml version
  await writeWorkspaceVersion(opts.workspace, opts.targetVersion);
  console.log("✓ Migration complete.");
}
```

### 4.4 Detection

`src/migrations/detect.ts`:

- v1.1.x — 루트에 `agents/`, `routines/`, `core/`, `.env`가 있고 `.solosquad/` 없음
- v1.2.2 — `.solosquad/workspace.yaml` 존재. `workspace.yaml.version`을 읽어 더 구체화
- 불명확 — 사용자에게 명시 요청 (`--source-version`)

## 5. Backup / Rollback

### 5.1 Backup 저장소

- 위치: `~/.solosquad-backups/<ISO8601>-v<source>/`
- 형식: 원본 파일트리 그대로 복사 (tar/zip 미사용 — 용량보다 복원 속도·디버깅 용이성 우선)
- 제외: `.git/` (이미 git에 버전관리되므로 불필요; 용량 절감)
- 메타: `<backup-dir>/.meta.yaml`에 원본 workspace 경로, 소스 버전, 타임스탬프, 실행된 마이그레이션 체인

### 5.2 Retention 정책

- 최근 5개 보존. 6번째 생성 시 가장 오래된 것 자동 삭제.
- `solosquad migrate --list-backups`로 조회.
- `solosquad migrate --delete-backup <id>`로 수동 삭제.

### 5.3 Rollback 제약

- Rollback 후 **현재 설치된 CLI 버전과 복원된 workspace 버전이 불일치**할 수 있음. 그 경우 `solosquad doctor`가 경고.
- 사용자에게 안내: 마이그레이션 전 버전으로 CLI를 다운그레이드하거나(`npm install -g solosquad@<version>`), 다시 migrate 실행.

## 6. 테스트 전략

### 6.1 Fixture 기반 회귀 테스트

`test/migrations/` 하위에 각 소스 버전의 "대표 workspace" 픽스처를 저장:

```
test/migrations/
  fixtures/
    v1.1.x-minimal/              # 최소 구성 (1 product, 0 repo)
    v1.1.x-multi-product/        # 제품 여러 개
    v1.1.x-with-routine-logs/    # 메모리/로그 존재
  1.1.x-to-1.2.2.test.ts         # 각 픽스처 대상 migrate → 구조 검증
```

### 6.2 각 마이그레이션 스크립트가 갖춰야 할 테스트

- **Dry-run plan 정확성** — 계획 단계가 실제 변경과 일치
- **Apply 결과** — 예상 구조가 생성되고, 메모리/설정 데이터 보존
- **Idempotency** — 이미 마이그레이션된 workspace에 재실행 시 변경 없음
- **Corrupt input** — 일부 파일 누락·깨짐 상태에서 우아하게 실패

### 6.3 CI 연동

- GitHub Actions에서 각 fixture × 각 migration 스크립트 매트릭스 실행
- v1.2.x 내 패치 릴리스 시에도 fixture는 계속 통과해야 함

## 7. 구체 케이스: v1.1.x → v1.2.2

### 7.1 입력 조건

```
WORKSPACE_ROOT  (예: /Users/you/my-workspace)
  agents/
  routines/
  core/
  templates/
  orchestrator/
  .env
  (docker-compose.yml, Dockerfile)

REPOS_BASE_PATH  (예: ~/Documents/solosquad-repos)
  bv-ai-native-po/
    product/
    memory/ (hypotheses.jsonl, experiments.jsonl, decisions.jsonl, signals.jsonl, routine-logs/)
    projects/
    slack/
  (추가 제품들...)
```

### 7.2 매핑 규칙

| 기존 위치 | 새 위치 | 비고 |
|---|---|---|
| `<ws>/agents/` | `<ws>/.solosquad/agents/` | 이동 |
| `<ws>/routines/` | `<ws>/.solosquad/routines/` | 이동 |
| `<ws>/core/` | `<ws>/.solosquad/core/` | 이동 |
| `<ws>/templates/` | `<ws>/.solosquad/templates/` | 이동 |
| `<ws>/orchestrator/` | `<ws>/.solosquad/orchestrator/` | 이동 |
| `<ws>/.env` | `<ws>/.solosquad/.env` | 이동 + `REPOS_BASE_PATH` 제거 |
| `<ws>/docker-compose.yml` | `<ws>/docker-compose.yml` | 유지, 내부 mount 경로 업데이트 |
| `<ws>/Dockerfile` | `<ws>/Dockerfile` | 유지 |
| `<REPOS_BASE_PATH>/<slug>/product/` | `<ws>/<slug>/` 하위로 이동 | `product/` 자체 폴더는 제거하고 brief.md만 org 루트로 |
| `<REPOS_BASE_PATH>/<slug>/memory/` | `<ws>/<slug>/memory/` | 이동 (JSONL 보존) |
| `<REPOS_BASE_PATH>/<slug>/projects/` | `<ws>/<slug>/workflows/` | **이름 변경** |
| `<REPOS_BASE_PATH>/<slug>/slack/` or `discord/` or `telegram/` | `<ws>/<slug>/slack/` 등 | 이동 |
| (없음) | `<ws>/<slug>/.org.yaml` | **신규 생성** — slug, provider(기본 local), products 기초 선언 |
| (없음) | `<ws>/.solosquad/workspace.yaml` | **신규 생성** |

### 7.3 `.org.yaml` 자동 생성 템플릿

기존 `core/products.json`의 `github_org` 필드가 있었다면:
```yaml
name: "<product.name>"
slug: "<product.slug>"
provider: "github"
remote_url: "https://github.com/<product.github_org>"
products:
  - name: "<product.name>"
    slug: "<product.slug>"
    repos: []
created_at: "<migration_timestamp>"
```

없었다면:
```yaml
name: "<product.name>"
slug: "<product.slug>"
provider: "local"
remote_url: null
products:
  - name: "<product.name>"
    slug: "<product.slug>"
    repos: []
created_at: "<migration_timestamp>"
```

### 7.4 엣지 케이스

| 상황 | 처리 |
|---|---|
| `REPOS_BASE_PATH` 디렉토리 자체가 없음 | 에러 없이 org만 생성 생략. 경고 출력. |
| 제품 폴더에 `projects/` 하위 항목이 있음 | `workflows/`로 그대로 이동. 내부 경로 참조(`_status.yaml` 등)는 스캔하여 `projects/` → `workflows/` 치환. |
| 하나의 workspace가 여러 REPOS_BASE_PATH를 사용 | 감지 불가 (`.env`에 하나만 저장됨). 사용자에게 추가 경로가 있는지 질문. |
| 제품 slug에 대문자/공백 포함 | slug는 디렉토리명이었으므로 이미 소문자+하이픈. 그대로 사용. |
| 같은 slug의 폴더가 이미 `<ws>/<slug>/`에 존재 | 충돌. 마이그레이션 중단하고 사용자 수동 해결 요청. |
| `REPOS_BASE_PATH`가 workspace 루트의 하위(`./repos`)로 설정됨 | 정상 케이스. 각 제품을 상위로 옮김 후 빈 `./repos/` 삭제. |

### 7.5 Forward-only 변경

- `REPOS_BASE_PATH` 환경변수 제거 — 자동 복원 불가 (사용자 필요 시 backup에서 수동 복사)
- `.org.yaml`의 provider=local 기본값 — 사용자가 추후 `solosquad add org --import-url`로 업그레이드

## 8. 향후 버전 마이그레이션 작성 가이드

새 버전에서 구조·스펙을 바꿀 때 기여자가 따라야 할 절차:

1. **변경 분류** — data-compatible (migrate 불필요) / structural (스크립트 필요) / config-breaking (env 변경 포함)
2. **스크립트 작성** — `src/migrations/scripts/<from>-to-<to>.ts`에 Migration 인터페이스 구현
3. **레지스트리 등록** — `src/migrations/index.ts`에 추가
4. **픽스처 생성** — `test/migrations/fixtures/v<from>-...`에 샘플 workspace 추가
5. **테스트 작성** — `test/migrations/<from>-to-<to>.test.ts`
6. **문서화** — 해당 버전의 릴리스 노트에 migration 요구 사항 명시, `docs/v<new>-<topic>.md`에 상세
7. **다중 버전 체이닝 검증** — 가장 오래된 지원 버전부터 새 버전까지 실행이 성공하는지
