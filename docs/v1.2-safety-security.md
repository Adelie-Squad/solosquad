# v1.2 — 안전/보안 레이어 (Safety & Security)

> AI 에이전트의 입출력 검증, 민감 파일 보호, 감사 로깅을 통해 무인 실행 시에도 안전한 시스템 구축.

**로드맵:** v1.1 크로스 플랫폼 → **v1.2 안전/보안** → v1.3 자율 실행 → v1.4 스킬 자유도 → v1.5 지식 온톨로지 → v1.6 웹 대시보드 → v2.0 커뮤니티 → v3.0 코워킹

**의존성:** v1.1 크로스 플랫폼 (플랫폼별 경로/권한 처리 필요)

## 2. AI 에이전트 보안 사고 사례

실제 발생했거나 재현된 AI 에이전트 보안 사고를 분석하여 방어 기준을 도출한다.

| 사례 | 시스템 | 사고 내용 | 교훈 |
|------|--------|----------|------|
| 재귀 스포닝 | AutoGPT | 에이전트가 자기 자신을 재귀적으로 생성, 리소스 무한 소비 | 실행 횟수/깊이 제한 필요 |
| 악성 패키지 설치 | AutoGPT | `pip install`로 악성 패키지를 설치하여 시스템 침해 | 패키지 설치 명령 화이트리스트 필요 |
| 비인가 API 호출 | AutoGPT | 계획에 없는 외부 API를 자율적으로 호출, 비용 발생 | API 호출 범위 제한 필요 |
| 시크릿 유출 | GitHub Copilot | 학습 데이터에 포함된 API 키, 토큰이 코드 제안에 노출 | 출력 시크릿 패턴 탐지 필요 |
| 파일 무단 덮어쓰기 | Cursor | 사용자 확인 없이 기존 파일을 덮어쓰기, 코드 유실 | 파일 변경 전 확인 메커니즘 필요 |
| 클라우드 리소스 스핀업 | Devin | 불필요한 클라우드 리소스를 생성, 예산 초과 | 인프라 명령 실행 제한 필요 |
| CI/CD 파이프라인 파괴 | Devin | CI/CD 설정을 잘못 변경하여 배포 파이프라인 중단 | CI/CD 파일 변경 보호 필요 |
| 민감 정보 공개 채널 게시 | Slack/Discord 봇 | 내부 데이터를 공개 채널에 게시하여 정보 유출 | 채널별 출력 필터링 필요 |
| Git force push | AI 코딩 도구 | `git push --force`로 팀 히스토리 파괴 | 위험 명령 차단 필요 |
| .env 파일 커밋 | AI 코딩 도구 | .env 파일을 git에 커밋하여 시크릿이 리포지토리에 노출 | .gitignore 검증 + 커밋 전 검사 필요 |
| API 비용 폭주 | 자율 에이전트 | 루프 실행으로 수시간 만에 $500-1000 API 비용 발생 | 비용 상한선 + 실행 횟수 제한 필요 |

## 3. 위협 모델

이 시스템에서 발생 가능한 4가지 위험 실행 경로를 분석한다.

### 3.1 사용자 메시지 → 에이전트 실행 경로

```
사용자 메시지 (Discord/Slack/Telegram)
  → 메신저 어댑터 (길이/내용 검증 없음)
  → agent-router.ts (키워드 매칭)
  → SKILL.md 로드 (인젝션 가능)
  → claude-runner.ts (execFile + stdin)
  → Claude 서브프로세스 (출력 검증 없음)
  → 파일 저장 / 채널 응답
```

**위험:** 사용자가 메시지에 프롬프트 인젝션을 삽입하면, SKILL.md의 지시를 무시하고 임의 명령을 실행할 수 있다. 출력에 민감 정보가 포함되어도 그대로 채널에 게시된다.

### 3.2 스케줄러 무인 실행 경로

```
cron 스케줄 트리거
  → routine 프롬프트 로드
  → claude-runner.ts 실행
  → Claude 응답
  → JSONL 메모리에 저장 (signals.jsonl, decisions.jsonl 등)
  → 다음 루틴이 저장된 데이터를 컨텍스트로 로드
```

**위험:** 무인 실행이므로 인간 감독이 없다. 이전 루틴이 저장한 JSONL 데이터에 프롬프트 인젝션이 포함되면, 다음 루틴이 이를 컨텍스트로 로드하여 오염된 지시를 실행한다 (간접 프롬프트 인젝션).

### 3.3 claude-runner.ts 실행 환경

```typescript
// 현재 코드: 안전 컨텍스트 없이 실행
execFile("claude", args, { ... }, (err, stdout) => { ... });
// stdin으로 프롬프트 전달, 안전 프리앰블 없음
```

**위험:** Claude 서브프로세스에 안전 지시가 없어, 요청받은 대로 파일 삭제, 시크릿 읽기, 위험 명령 실행이 가능하다.

### 3.4 메신저 어댑터 입력 경로

```
Discord: message.content → 길이 제한 없음 (Discord 자체 2000자 제한은 있으나 봇 API는 다름)
Slack: event.text → 길이 제한 없음
Telegram: msg.text → 길이 제한 없음
```

**위험:** 극단적으로 긴 메시지로 토큰 비용 폭주, 또는 특수 포맷의 메시지로 파서 오류를 유발할 수 있다.

## 4. 코드 레벨 보안 (구현 대상)

### 4.1 입력 검증

사용자 메시지와 루틴 프롬프트에 대한 기본 검증을 추가한다.

```typescript
// src/util/safety.ts (신규)

const MAX_INPUT_LENGTH = 4000;

export function validateInput(input: string): { valid: boolean; reason?: string } {
  if (!input || input.trim().length === 0) {
    return { valid: false, reason: "빈 입력은 허용되지 않습니다." };
  }
  if (input.length > MAX_INPUT_LENGTH) {
    return { valid: false, reason: `입력이 ${MAX_INPUT_LENGTH}자를 초과합니다. (${input.length}자)` };
  }
  return { valid: true };
}
```

**적용 위치:**
- `src/messenger/discord.ts` — 메시지 수신 시
- `src/messenger/slack.ts` — 이벤트 수신 시
- `src/messenger/telegram.ts` — 메시지 수신 시
- `src/scheduler/runner.ts` — 루틴 프롬프트 로드 후

### 4.3 민감 파일 보호

안전 프리앰블에 더해, 코드 레벨에서도 파일 접근을 검증한다.

```typescript
// src/util/safety.ts 추가

const SENSITIVE_PATTERNS = [
  /\.env($|\.)/,
  /\.ssh\//,
  /credentials\.json/,
  /\.pem$/,
  /\.key$/,
  /id_rsa/,
  /\.aws\/credentials/,
  /\.gcloud\//,
  /secret[s]?\.(json|yaml|yml)$/,
];

export function isSensitivePath(filePath: string): boolean {
  return SENSITIVE_PATTERNS.some((pattern) => pattern.test(filePath));
}
```

### 4.4 출력 새니타이징

Claude 응답에서 민감 정보를 탐지하고 마스킹한 후 JSONL에 저장한다.

```typescript
// src/util/safety.ts 추가

const SECRET_PATTERNS = [
  { name: "AWS Key", pattern: /AKIA[0-9A-Z]{16}/g },
  { name: "Generic Secret", pattern: /(?:secret|token|password|api[_-]?key)\s*[:=]\s*['"][^'"]{8,}['"]/gi },
  { name: "Bearer Token", pattern: /Bearer\s+[A-Za-z0-9\-._~+/]+=*/g },
  { name: "Private Key", pattern: /-----BEGIN (?:RSA |EC )?PRIVATE KEY-----/g },
  { name: "Slack Token", pattern: /xox[bpors]-[0-9a-zA-Z-]+/g },
  { name: "Discord Token", pattern: /[MN][A-Za-z\d]{23,}\.[\w-]{6}\.[\w-]{27,}/g },
  { name: "GitHub Token", pattern: /gh[ps]_[A-Za-z0-9_]{36,}/g },
];

export function sanitizeOutput(output: string): { sanitized: string; redacted: string[] } {
  const redacted: string[] = [];
  let sanitized = output;

  for (const { name, pattern } of SECRET_PATTERNS) {
    sanitized = sanitized.replace(pattern, (match) => {
      redacted.push(`${name}: ${match.substring(0, 8)}...`);
      return `[REDACTED:${name}]`;
    });
  }

  return { sanitized, redacted };
}
```

**적용 위치:**
- `src/scheduler/memory.ts` — JSONL 저장 전
- `src/bot/claude-runner.ts` — 응답을 채널에 전달하기 전

## 5. 프로세스 레벨 보안 (가이드라인)

코드 구현 외에, 운영 과정에서 준수해야 할 보안 가이드라인을 정의한다.

### 5.1 안전 원칙

| 원칙 | 설명 | 적용 |
|------|------|------|
| 최소 권한 (Least Privilege) | 에이전트에게 필요한 최소한의 권한만 부여 | 봇 토큰 스코프 최소화, 파일 접근 범위 제한 |
| 인간 개입 (Human-in-the-Loop) | 위험한 작업 전 인간의 확인을 받음 | 파일 삭제, 배포, 비용 발생 작업 전 승인 요청 |
| 심층 방어 (Defense in Depth) | 단일 방어에 의존하지 않고 다층 방어 구축 | 입력 검증 + 안전 프리앰블 + 출력 새니타이징 |
| 감사 추적 (Audit Trail) | 모든 에이전트 실행을 기록하여 사후 분석 가능 | 실행 로그 + 입출력 기록 + 타임스탬프 |

### 5.2 시크릿 관리

- **.env 파일은 절대 커밋하지 않는다.** `.gitignore`에 반드시 포함.
- **토큰 로테이션:** 봇 토큰, API 키는 90일마다 교체를 권장한다.
- **최소 봇 스코프:** 봇에 필요한 최소한의 권한만 부여한다.
  - Discord: `Send Messages`, `Read Message History` (관리자 권한 불필요)
  - Slack: `chat:write`, `channels:history` (파일 접근 불필요)
  - Telegram: 그룹 관리 권한 불필요
- **환경변수 관리:** `ANTHROPIC_API_KEY`, `DISCORD_TOKEN` 등은 `.env` 파일 또는 시스템 환경변수로만 관리하고, 코드에 하드코딩하지 않는다.

### 5.3 파일 권한

```bash
# 프로덕트 디렉토리: 소유자만 접근
chmod 700 ~/repos/{product-slug}

# 민감 파일: 소유자만 읽기/쓰기
chmod 600 .env
chmod 600 *.pem
chmod 600 *.key

# 메모리 디렉토리: 소유자만 접근
chmod 700 ~/repos/{product-slug}/memory
```

### 5.4 메신저 플랫폼 보안

| 플랫폼 | 최소 토큰 스코프 | 주의 사항 |
|--------|----------------|----------|
| Discord | `Send Messages`, `Read Message History`, `Message Content` (Privileged Intent) | 관리자 권한 부여 금지, 서버 관리 스코프 불필요 |
| Slack | `chat:write`, `channels:history`, `app_mentions:read` | `files:read`, `admin.*` 스코프 불필요 |
| Telegram | 그룹 메시지 읽기/쓰기 | 그룹 관리자 권한 불필요, `/setprivacy` 설정 확인 |

**공통 주의 사항:**
- 봇이 응답하는 채널을 제한한다 (허용된 채널 ID 리스트).
- 공개 채널에는 민감 정보가 포함된 응답을 게시하지 않는다.
- 봇 토큰이 노출되면 즉시 재발급한다.

### 5.5 루틴 실행 안전

스케줄러로 무인 실행되는 루틴은 추가 안전 조치가 필요하다.

**리뷰 모드 (Review Mode) 개념:**

```
[일반 모드]  루틴 실행 → 결과 저장 → 채널 게시 (현재)

[리뷰 모드]  루틴 실행 → 결과 임시 저장 → 오너에게 DM 알림
             → 오너 승인 → 결과 저장 + 채널 게시
             → 오너 거부 → 결과 폐기 + 로그 기록
```

- 최초 도입 시 모든 루틴을 리뷰 모드로 실행하여 안전성 확인.
- 충분히 검증된 루틴은 자동 모드로 전환.
- 루틴 프롬프트가 변경되면 다시 리뷰 모드로 전환.
