# 멀티 리포지토리 AI 에이전트와 초장기 세션/메모리 관리 설계 가이드

AI 에이전트가 하나의 폴더를 넘어 여러 개의 리포지토리(Multi-Repository)를 오가며 **30시간 이상 장기 작업**을 수행할 때, 시스템을 어떻게 설계해야 안전하고 똑똑하게 움직이는지 알기 쉽게 정리한 문서입니다.

---

## 1. 오케스트레이터는 어디서 상주하며 일하나요?

여러 폴더(리포지토리)를 관리하는 대장 AI를 '오케스트레이터(Orchestrator)'라고 합니다. 이 대장 AI는 자식 폴더에 들어가서 일하지 않고, 언제나 모든 폴더를 내려다볼 수 있는 최상위 부모 디렉토리(Root)에 머무릅니다.

### 📂 표준 디렉토리 구조 예시

```text
/workspace (★ 대장 AI가 상주하는 글로벌 세션 위치)
├── workspace.json (또는 MONO_MAP.md: 리포지토리들의 역할과 지도)
├── SYSTEM.md (대장 AI의 전체 행동 강령)
│
├── [repo-a-frontend]/ (자식 폴더 1: 프론트엔드)
│   ├── .git/
│   └── skill.md (프론트엔드 작업용 지침)
│
└── [repo-b-backend]/ (자식 폴더 2: 백엔드)
    ├── .git/
    └── skill.md (백엔드 작업용 지침)

```

대장 AI는 이 `/workspace`라는 통제실에서 전체적인 계획을 세우고 상황을 기록합니다.

---

## 2. 여러 폴더를 오가며 작업하는 3단계 흐름

대장 AI가 "백엔드를 고친 뒤, 프론트엔드를 연동해줘"라는 명령을 받으면 다음과 같이 **출장**을 다니듯 작업합니다.

### 1단계: 지도 확인 (탐색)

* 최상위에 있는 `workspace.json`이나 지도를 읽고, 어떤 폴더가 무슨 일을 하는지 파악합니다. 지도 파일이 없다면 폴더들을 하나씩 열어보며 `.git`이 있는지 스스로 스캔합니다.

### 2단계: 출장 가기 (로컬 세션 켜기)

* **주소지 변경:** 백엔드를 고쳐야 한다면, 하네스(인프라 시스템)가 AI의 작업 위치를 `/workspace/repo-b-backend`로 튕겨줍니다.
* **기억 채워 넣기 (Hydration):** AI의 머릿속에 백엔드 폴더 안의 코드 구조와 그 폴더 전용 지침(`skill.md`)을 쏙 넣어줍니다. 이제 AI는 자기가 백엔드 개발자가 된 것처럼 집중해서 일합니다.

### 3단계: 짐 정리하고 복귀 (메모리 비우기)

* 백엔드 작업이 끝나면, 그동안 쌓인 수많은 줄의 코드와 에러 로그를 AI의 머릿속에서 **전부 지워버립니다(Flush).** 토큰(비용)을 아끼고 머리를 맑게 유지하기 위해서입니다.
* 대신 "백엔드 코드 어떤 걸 고쳤음"이라는 딱 한 줄짜리 요약본만 챙겨서 다시 최상위 통제실(`/workspace`)로 돌아옵니다. 그 후 다음 출장지인 프론트엔드 폴더로 이동합니다.

## 3. 30시간 넘게 일할 때의 특별한 기억력(세션/메모리) 관리법

AI에게 30시간 연속으로 일을 시키면 머릿속(컨텍스트 윈도우)이 수많은 대화와 에러 로그로 가득 차서 **결국 터지거나 헛소리(환각)를 시작합니다.** 전문가들은 이를 막기 위해 3가지 전략을 씁니다.

### ① AI 교대 근무 패턴 (Handover)

하나의 대화 창을 30시간 동안 쓰지 않고, 중간에 AI를 새 뇌로 갈아끼우는 방식입니다.

* **64% 경고 지점:** AI의 기억 용량이 64% 정도 차면, 하네스가 "정신이 온전할 때 지금까지 한 일, 알아낸 팁을 `.md` 파일에 미리 저장해라"라고 지시합니다.
* **80% 교대 지점:** 80%에 도달하면 지금 쓰던 대화 창을 강제로 폭파(Kill)합니다. 그리고 아무 기억도 없는 깨끗한 새 대화 창을 연 뒤, 방금 백업해 둔 핵심 요약본 파일만 쏙 넣어주고 일을 이어서 하게 만듭니다.

> **현실 보정:** 이 64%/80% 같은 수치는 절대값이 아니라 **시스템마다 다릅니다.** 실제 상용 에이전트의 자동 압축(compaction) 트리거는 — OpenAI **Codex는 컨텍스트 윈도우의 ~90%**(`min(사용자설정, window×0.90)`)에서, **Qwen Code는 70%**(`contextPercentageThreshold: 0.7`)에서 발동합니다. 자세한 구현은 아래 **§5** 참고.

### ② 기억의 3단계 계층 분리

컴퓨터에 RAM과 하드디스크가 있듯, AI 시스템도 기억을 쪼갭니다.

* **단기 기억 (Working):** 최근 몇 분 동안 나눈 대화 원본과 방금 실행한 명령어 로그 (가장 빠르고 가벼운 저장소에 보관)
* **활동 기억 (Episodic):** 30시간 동안 AI가 생각하고 행동했던 모든 순간의 기록 (문제가 생겼을 때 되감기하기 위한 용도)
* **장기 기억 (Long-term):** 자가학습을 통해 얻은 노하우, 성공 패턴, 고정된 규칙 (따로 보관하여 평생 기억)

### ③ 현명하게 잊어버리기 (가비지 컬렉션)

* AI가 10시간 전에 코드 빌드에 실패해서 5번 넘게 삽질했던 지저분한 에러 로그는 과감히 삭제합니다.
* 대신 "A 라이브러리는 버전 충돌이 나니 앞으로 쓰지 말 것"이라는 교훈 한 줄만 `LEARNINGS.md`에 남기고 원본은 지워서 AI의 시야를 항상 깨끗하게 유지합니다.

## 4. 시스템이 갑자기 뻗어도 안심하는 법 (복구 시스템)

30시간 동안 가동되면 컴퓨터가 꺼지거나 인터넷이 끊기는 배달 사고가 무조건 발생합니다.

* **슈퍼스텝 체크포인트:** AI가 `계획 수립 → 코드 수정 → 테스트 → 커밋`이라는 큰 단계들을 통과할 때마다 하네스가 저장단추(스냅샷)를 누릅니다.
* **무중단 재개:** 28시간째에 서버가 다운되더라도, 새로 켜진 시스템이 마지막 저장단추 위치를 읽어와서 **27시간 59분에 AI가 하려던 행동부터 완벽하게 이어서 시작**합니다.

## 5. 실제 레퍼런스: Codex와 Qwen은 어떻게 하나

위 1~4의 원칙(상주 위치 · 출장/소거 · 교대/계층 메모리 · 체크포인트)이 **실제 상용/오픈소스 에이전트에서 어떻게 구현되어 있는지**를 대조해, SoloSquad 설계에 가져올 만한 패턴을 추린다.

> **검증 수준 표기:** 〔공식〕 = 공식 문서/소스 레포 / 〔2차〕 = 소스코드 분석·기술 블로그 / 〔미검증〕 = 1차 출처로 확인 못 함. 수치는 모델/버전에 따라 변함.

### 5.1 OpenAI Codex (CLI + 클라우드 에이전트)

* **세션이 사는 곳 (영속):** 모든 세션이 **rollout JSONL** 파일로 디스크에 남는다 — `~/.codex/sessions/YYYY/MM/DD/rollout-*.jsonl`, `ThreadId`로 키잉 〔2차, 소스 분석〕. 재개는 `codex resume` / `--last`(현재 cwd 기준 최근) / `codex resume <SESSION_ID>` 〔공식〕. → **§4의 "무중단 재개"를 디스크 트랜스크립트로 구현한 형태.**
* **압축 (자동 교대의 실제 모습):** 컨텍스트가 차면 자동 compaction. 발동점은 `min(사용자설정, window×0.90)` — **윈도우 90% 초과 설정은 조용히 무시** 〔2차, `codex-rs/core/src/compact.rs`〕. 수동 트리거 `/compact`(포커스 지시 가능) 〔공식 동작〕. **방식 = "handoff summary"**: 진행상황·핵심 결정·제약/사용자 선호·남은 TODO를 요약하고, **user 메시지는 원문 보존 + 최근 ~20k 토큰**만 남긴 뒤 **assistant·tool 메시지는 전부 삭제** 〔2차〕. → **§3①의 "요약본만 챙겨 새 창으로"의 정밀판.** "무엇을 버리고 무엇을 남기나"가 명시적(=user 의도는 원문, 중간 삽질은 폐기).
* **정적 지침 (계층 메모리의 상단):** `AGENTS.md` — 글로벌(`~/.codex/`) → 프로젝트 루트 → cwd 순으로 **root→cwd 병합**(가까운 게 나중에 와서 override), 디렉토리당 1개, **기본 32 KiB 상한**, `AGENTS.override.md`는 확장이 아니라 **교체** 〔공식〕. → SoloSquad의 `AGENTS.md`/3-layer context와 직접 대응.
* **학습형 메모리 (cross-session):** **Codex Memories** — `~/.codex/memories/`에 선호·워크플로·관례를 **세션을 넘겨** 보존 〔공식〕. 내부는 SQLite + 유휴 **~6시간** 후 비동기 생성 + 무엇을 기억할지/병합할지 **2-모델 파이프라인** 〔2차〕. 읽기/쓰기 스위치 분리, 비밀정보 redaction 〔공식〕. → **§3③ "교훈 한 줄만 LEARNINGS.md에"의 제품화.**
* **초장기 실행:** 클라우드는 격리 컨테이너(상태 **최대 12h 캐시**) 〔공식〕. 레퍼런스 실행은 **~25시간 무중단 · ~1,300만 토큰 · ~3만 LOC** 〔공식 블로그〕. **핵심 통찰:** 자동 압축에만 기대지 않고 **상태를 repo 안의 영속 마크다운으로 외부화** — `Prompt.md`(고정 스펙=드리프트 방지) · `Plan.md`(마일스톤+검증 커맨드) · `Implement.md`(작업 규칙) · `Documentation.md`(실행 감사 로그) + **단계마다 stop-and-fix 검증 루프**(test/lint/build) 〔공식〕. → **§3·§4 전체를 "컨텍스트 윈도우가 아니라 파일시스템이 진짜 메모리"로 푼 모범 사례.**

### 5.2 Qwen (Alibaba)

* **먼저 — "Qwen3.7"은 공식 릴리스가 아니다** 〔미검증〕. 실재하는 것은 ▸ **Qwen Code**(CLI 에이전트, Gemini-CLI/Codex 류 포크) ▸ **Qwen-Agent**(파이썬 에이전트 프레임워크) ▸ **Qwen3 / Qwen3-Coder**(모델, 네이티브 **256K** → **~1M** 확장) ▸ 연구물 **QwenLong-L1 / -CPRS**. "3.7"은 Qwen3.5/3.6/Coder-Next의 혼동으로 추정.
* **Qwen Code (CLI):** 계층형 컨텍스트 파일 **`QWEN.md`**(`@경로` import, `/memory refresh|show`) = AGENTS.md 등가물 〔공식〕. 압축은 **토큰 한도의 70%**(`chatCompression.contextPercentageThreshold: 0.7`)에서 발동 〔공식〕 — 단 **1차 압축 이후 잘 안 줄고, 압축 후 재개 시 토큰 카운트가 어긋나는 버그**가 이슈로 보고됨 〔2차, GitHub issues〕. → 교대/복구를 어설프게 구현하면 생기는 함정의 실사례.
* **Qwen-Agent (8k→1M, RAG로 우회):** 윈도우를 넓히는 게 아니라 **분해**로 푼다 — ① BM25로 512토큰 청크 검색 ② 청크별 관련성 평가 ③ 서브질문으로 쪼개 ②를 반복하며 **Lv3 에이전트 메모리에 답을 누적** 〔공식 블로그〕. 여기서 "메모리"는 별도 모듈이 아니라 **누적 추론 루프** 자체. → **§2의 "필요한 것만 켜고 나머지는 안 본다"의 검색-기반 버전.**
* **연구 계층:** **QwenLong-CPRS**(컨텍스트 압축, **72.6×~290.5×**) 〔공식 arXiv〕, **QwenLong-L1/L1.5**(장기추론 RL + 256K→유효 **1M~4M** "Memory Agent") 〔공식 arXiv, 최신·실전 검증 적음〕. → 우리 §3의 "압축/요약"을 모델 레벨에서 밀어붙인 방향.

### 5.3 4축 대조표

| 축 | OpenAI Codex | Qwen |
|---|---|---|
| **(a) 세션 상태 위치** | CLI: 로컬 rollout JSONL(`~/.codex/sessions/…`, ThreadId). 클라우드: 컨테이너(≤12h 캐시)+repo | Qwen Code: 로컬 세션 + 계층 `QWEN.md`. Qwen-Agent: **프로세스 내 누적 메모리**(디스크 영속 포맷 문서화 안 됨) |
| **(b) 압축 트리거·방식** | `min(설정, window×0.90)`; handoff summary(user 원문+최근 ~20k 유지, assistant·tool 폐기); `/compact` 수동 | Qwen Code: **70%** 임계 요약(버그 보고). Qwen-Agent: 임계 없이 **BM25 분해**. CPRS: 연구용 압축기 |
| **(c) 계층·영속 메모리** | `AGENTS.md`(정적, 32KiB, override) + **Codex Memories**(학습형, SQLite, 비동기) | `QWEN.md`(정적) — **학습형 cross-session 메모리는 미문서화**(`/memory`는 정적 파일 갱신만) |
| **(d) 복구·재개** | `codex resume`로 rollout 재수화; 클라우드 컨테이너 캐시(재개 버그 존재) | 세션 재개 가능하나 압축-후-재개 토큰 회계 버그; Qwen-Agent는 영속 복구 포맷 없음 |
