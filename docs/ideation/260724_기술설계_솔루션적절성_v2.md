## PART A. 기술 설계·아키텍처

*(PART A는 v1과 동일 — 아래는 요지 유지 + 사업계획서 교차참조 보강)*

### 1. 3계층 워크스페이스 + 8단계 JIT 조립
**[강점]** 3계층 격리(워크스페이스/조직/리포)는 개념·구현 모두 우수. 조립기 자체도 키워드 스코어링·토큰 상한·drop 감사기록까지 정교.
**[약점]** **8단계 조립기가 실제 spawn 경로에 미연결**(preflightSpawn 런타임 호출 0건, 코드 주석이 "host가 Task를 가로챌 수 없다"고 자인). 실제 주입은 Claude Code 네이티브 subagent가 담당. 한국어 토큰 추정(`문자수/4`) 과소평가로 cap이 안전 보장이 아님.

### 2. 에이전트 팀 구조·라우팅
**[강점]** 라우터 결정적(규칙기반 `@slug` + LLM은 분류만), 세션 mutex, hot-reload 원자적 swap.
**[약점]** 팀 taxonomy 문서-구현 정면 불일치(문서 product/eng/design/marketing 4팀 vs 실제 brand/business/core/engineering/product 5팀), 에이전트 수 과장(문서 25 vs 실제 19).

### 3. 메모리(FTS5)
**[강점]** 벡터DB 없이 SQLite FTS5는 규모에 맞는 절제. hot/cold 2단 + git 스냅샷 수명모델 명확.
**[약점]** 한국어 형태소 미지원(조사 매칭 실패), lexical-only(개념 회상 불가). 수용 가능한 trade-off.

### 4. 하네스/루프(가드·stop-hook·HITL)
**[강점]** 세션 복구·예산 가드·oscillation 감지·HITL 승인 게이트는 실제 작동.
**[약점]** **Output/Path 가드·stop-hook이 goal-runner에 미연결**(테스트에서만 호출). "3계층 가드레일"의 절반이 dead code. 엔진 e2e 통합테스트 부재.

### 5. 세션·cron·봇 통합
**[강점]** 스케줄러 싱글톤 락, 6+1 stage durable 기록·narration.
**[약점]** 메신저 실연동 통합테스트 부재, 사실상 Discord 단일(Slack/Telegram 비활성).

### 6. 마이그레이션·스키마·배포
**[강점]** **이 저장소 최고 수준** — 43개 전환 스크립트, dry-run/백업/rollback/idempotent/chainable, 6표면 스키마 안정성.
**[약점]** Docker 경로참조 모델 정합성 위험, docker-compose legacy 잔존.

### 7. 코드 품질
**[강점]** 164 테스트, ESM 순수성 체커, 보수적 의존성.
**[약점]** **테스트 수가 커버리지 착시** — 순수함수·fake 프로세스 집중, 실 런타임/통합 경로 blind spot(§1·§4 미연결 모듈이 그 증상).

### 8. 기술 종합 — 최우선 리스크
**① 자율 goal 엔진의 "측정"이 유사난수 placeholder(코드 검증).** `goal run`/`verify`가 wiring하는 measurer는 `makePlaceholderMeasurer()`(goal.ts:362,497)이고, 지표를 측정하지 않고 `source|formula|commitSha` 해시를 [0,1)로 매핑(goal.ts:629-642). 주석도 "실제 측정은 v0.4.x 패치"라 자인.

> **🔴 v2 신규 — 사업계획서 교차참조(중대):** 최신 사업계획서의 **주력 유료 제품이 "가상 시장 수요 검증 시뮬레이션"**이다. 이 제품은 정확히 이 자율/측정/시뮬레이션 스택 위에 선다. 따라서 "측정이 유사난수"라는 결함은 더 이상 학술적 흠이 아니라 **유료 제품의 신뢰성 리스크**다. 게다가 Report ②에서 밝혔듯 자사 백테스트가 "raw 시뮬은 反예측·절대 전환율 주장 금지"라 결론냈으므로, **판단 근거가 되는 측정 계층이 스텁인 채로 "데이터 기반 검증"을 유료화하는 것**은 제품의 핵심 철학(反아첨·데이터 기반)과 정면 충돌한다. 유료 출시 전 measurer를 실측으로 교체하거나, 산출물을 "예측 아님·상대 랭킹 보조"로 명확히 한정해야 한다.

**③ 확장성 제약** — 실제 주입이 Claude Code 네이티브 subagent 의존이라 조립기의 토큰 예산 강제 불가.

## PART B. 솔루션(기획 프레임워크) 적절성

*(PART B는 v1과 동일 — 요지 유지)*

- **§9 문제정의 프레임워크:** 검증기는 워크플로 **구조**(순환·참조 무결성·depth≤2)를 진짜로 강제하나, 기획 **내용**(HARD GATE·≥2접근·confidence)은 LLM용 마크다운 체크리스트일 뿐. "6단계 강제 체인"은 실제로 **폐기**됐고 "선택 메뉴"로 재배치됨.
- **§10 3대 함정:** 자기부정(durable-md 고정)·확증편향(자기 기능 죽인 백테스트)은 진짜 구조·문화 앵커. 그러나 스킬 레이어의 anti_sycophancy·min_approaches는 검증기 없는 프롬프트 플래그.
- **§11 output≠goal:** 일부 조작화(판단근거 동반 반환). 그러나 **앙상블/복수세션 의사결정은 미구현**(ideation 인용뿐).
- **§12 암묵지→워크플로우:** 초안-앵커 방식은 잘 설계됨. 그러나 실제 고객 인터뷰 transcript 수동 업로드(N≥5)를 솔로 빌더가 수행할지는 미검증.
- **§13 데이터 검증:** 방법론 정직하나 검증된 건 주로 "부정". 0-to-1 앵커 부재로 타깃에게 적용 곤란(chicken-and-egg).
- **§14 사용성/과설계:** 5분 온보딩은 실재하나 **매뉴얼-프레임워크 버전 격차**(매뉴얼은 v1.2.x 기준)로 핵심 가치가 안 보임. PM 자율실행(no user Q&A)이 "대신 기획→승인"에 가까워 확증편향 유발 위험. → **Q4·Q1 답변(PART C)과 직접 연결.**
- **§15 총평:** 개념적 정합 강하나 "강제·앙상블·데이터검증" 3주장이 각각 과대·미구현·타깃 부적용. 정직한 재기술: "함정을 구조적으로 막는 시스템"이 아니라 "표준 프레임을 자율 실행하며 durable-md로 표류를 줄이는 성실한 어시스턴트".

## PART C. 설계 심층 질문 — 워크샵 참여자 Q&A (신규)

> 각 답변은 저장소 실물 코드·내부 ideation 문서를 근거로 하며, **[설계 의도] / [객관적 평가] / [근거]** 로 구분한다. 요지: 대부분의 질문이 **정당한 지적**이고, 여러 항목은 **저장소 자신의 내부 문서가 이미 같은 문제를 인정**하고 있다.

### Q1. 왜 사용자는 Chief와 대화하고 Chief가 PM에게 지시하는가? PM이 실질 오케스트레이터인데 — 맥락 손실·불필요한 단계는 없나?

**[설계 의도]** Chief는 "유일한 user-facing bot"으로 소통·과제화(TRIAGE)·종합(SYNTHESIZE)·회고를 맡고, PM은 "사용자와 대화하지 않는" 자율 에이전트로 product 팀 specialist를 오케스트레이션하며 PMF 가설을 소유한다(chief SKILL.md, product-manager SKILL.md). 즉 **오케스트레이션이 2단**이다: Chief(조직 전역·크로스팀) → PM(product 팀 내부). PM이 독립 컨텍스트 윈도우에서 돌아 메인 오염을 막고, `open_questions[]`는 Chief가 batch로 모아 물어 기획 중 사용자 개입을 최소화한다.

**[객관적 평가]** 지적이 타당하다. 실질 리스크가 둘 있다:
- **맥락 손실 표면 증가:** 사용자→Chief→PM→specialist는 **최대 3단 spawn**이고 각 단계가 별도 컨텍스트 윈도우다. 원래 이 손실을 줄여야 할 8단계 조립기·핸드오프 슬라이서가 **런타임 미연결**(PART A §1)이라, 핸드오프 충실도는 Claude Code 네이티브 subagent + `_handoff.md` 마크다운에 의존한다. 즉 "맥락 손실 없음"을 보장하는 장치가 홍보만큼 작동하지 않는다.
- **불필요한 단계 위험:** 단순 작업에도 2중 오케스트레이터를 태우면 토큰·지연이 늘어난다. **완화 장치는 있다** — Chief의 DECOMPOSE는 "PM 호출 / 다른 main 직접 / single skill"을 선택할 수 있어(chief SKILL.md) 간단한 건 PM을 건너뛴다. 그러나 product 기획은 반드시 Chief→PM을 타므로, PM이 Chief 대비 **실측 가능한 기획 깊이를 더한다는 증거**가 없으면 이 2단은 관료제로 보인다.
- **추가 모호성:** architecture.md는 engineer/designer/marketer의 `used_by=[chief, pm]`이라 명시해 **Chief와 PM 양쪽이 engineer를 dispatch**할 수 있는 DAG가 된다 → 동일 작업의 이중 위임·책임 경계 모호 위험.

**[근거]** `agents/main/chief/SKILL.md`(6+1 stage, DECOMPOSE 분기), `agents/main/product-manager/SKILL.md`(자율 실행·no user Q&A·open_questions batch), `docs/architecture.md`(pm.used_by / DAG), PART A §1(조립기 미연결).

> **권고:** (a) PM이 더하는 기획 깊이를 A/B로 실측(Chief 직접 vs Chief→PM), (b) 단순 product 작업은 Chief→specialist 직결 fast-path 허용, (c) engineer 이중 위임 경로 중 하나로 정리.

### Q2. Chief와 PM의 차이는 무엇이고 왜 구분하는가?

**[설계 의도]** 명시적으로 다르다:

| | Chief | PM |
|---|---|---|
| 사용자 대면 | ✅ 유일 | ❌ (Chief 경유) |
| 역할 | 소통·TRIAGE(chat/workflow/cron/goal)·크로스팀 종합·회고/자가학습 | 문제정의·PMF/가설·PRD·WBS·product 팀 오케스트레이션 |
| 팀 | core | product |
| 엄격도 | hard_gate **false**, min_approaches **1** | hard_gate **true**, min_approaches **2** |
| 커스터마이즈 | **org 단위 도메인 전문가화**(CUSTOMIZATION_GUIDE) | workspace 번들 표준 |

**[객관적 평가]** 구분은 **개념적으로 방어 가능**하다 — "비서실장/프론트데스크(Chief)" vs "제품총괄(PM)". 특히 엄격도 차이(PM만 hard_gate·≥2접근)와 커스터마이즈 스코프(Chief만 org별)가 실질적 근거다. 그러나 솔로+LLM 맥락에서 경계가 흐려진다: 둘 다 leader tier, 둘 다 오케스트레이션하며, Chief의 DECOMPOSE도 이미 분해(=기획적) 행위를 한다. **솔로 사용자가 주로 product 기획만 한다면** 두 에이전트는 관료적으로 느껴질 수 있다. 이 구분은 (i) 기획이 전담 rigor 에이전트를 정당화할 만큼 깊고, (ii) Chief가 비-product 도메인(eng/business/brand)까지 저글링할 때 값을 한다.

**[근거]** 두 SKILL.md frontmatter(hard_gate·min_approaches·used_by·team), `agents/main/chief/CUSTOMIZATION_GUIDE.md`.

> **판정:** 조직 확장을 위한 추상으로는 정당, 중앙값 솔로 사용자에겐 과구조 가능성. "Chief=대화+회고+크로스팀, PM=깊은 기획"이라는 값을 산출물 품질로 증명 필요.

### Q3. 메인/서브 에이전트, 메인/서브 워크플로우 같은 형식적 구조가 실효성이 있는가?

**[설계 의도]** 인간 조직도·학술 멀티에이전트(MetaGPT)를 본떠 5팀·main+members·서브워크플로(depth≤2)로 구조화. specialist마다 bounded 컨텍스트(격리), 라우팅 결정성, 서브워크플로 무결성 검증이 실효 신호.

**[객관적 평가]** 실효성이 **부분적으로만** 있다.
- **실효 있는 부분:** ① 인간 가독성·멘탈모델(사용자가 "PM/엔지니어에게 시킨다"를 직관), ② 스킬·지식의 조직화(team shared_skills), ③ 라우팅 결정성, ④ 컨텍스트 격리.
- **형식이 실질을 앞서는 위험:** 실제 컨텍스트 주입이 조립기가 아니라 Claude Code 네이티브 subagent라(PART A §1), 팀 KNOWLEDGE·OKR 레이어가 서브에이전트에 **신뢰성 있게 도달하는지 불확실**. 게다가 팀 taxonomy가 문서(4팀)와 코드(5팀)가 어긋나고(PART A §2) v2.0에서 조직 개편이 있었던 걸 보면 **구조 자체가 자주 흔들린다**(형식의 유지비용). MetaGPT식 역할분업은 "1회성 소프트웨어 생성"엔 학술 검증됐으나, SoloSquad의 **상시·다도메인 조직**은 훨씬 무겁다.

**[근거]** `teams/*/composition.yaml`, PART A §1·§2, `docs/ideation/260623-squad-org-restructure.md`(개편 흔적).

> **판정:** 구조는 "조직화·가독성·라우팅"에서 실효 있으나, "협업 충실도"는 미연결 조립기+마크다운 핸드오프에 의존해 형식만큼 작동하지 않을 위험. **main/sub 구분이 flat한 skill-chain 대비 산출 품질을 올리는지 실측**해야 정당화된다.

### Q4. 디스코드보다 터미널이 더 편하다.

**[설계 의도 / 자기인식]** **사업계획서가 이미 같은 결론을 담고 있다.** 창업자는 "초기엔 기획자가 CLI를 어려워해 Discord가 편할 거라 생각했으나, 실제로 개발자·바이브코딩 경험자는 터미널을 훨씬 편해했다"며 **유저 저니를 2트랙**(이동 중 러프한 아이디어·리서치=Discord, 본격 업무·코드=터미널)으로 분리했다고 명시한다(`SoloSquad_plan_260723.md`).

**[객관적 평가]** 이 피드백은 **창업자의 관찰과 일치하는 정당한 지적**이며, Figma/UX 배경과 결합해 잠재적 차별점("사용성이 사고를 방해하면 만족도가 즉시 붕괴")이다. 그러나 **아키텍처가 이 통찰을 아직 못 따라간다**:
- 제품의 간판이 "메신저로 24시간"이고, 봇+cron+메신저 어댑터 전체가 **Discord 중심**이다. Chief는 `#owner-command` 채널로만 도달 가능(chief SKILL.md).
- 반면 터미널 경로는 사실상 "repo에서 `claude`를 직접 실행"하는 것이라, **오케스트레이션·HITL 승인·org 메모리 기록이 메신저를 경유하도록 배선**돼 있어 터미널 사용자는 이 게이트를 우회하거나 못 누린다.

**[근거]** `SoloSquad_plan_260723.md`(2트랙 저니), `agents/main/chief/SKILL.md`(#owner-command 단일 창구), PART A §5(메신저 중심 배선).

> **권고:** 터미널을 "claude가 도는 장소"가 아니라 **1급 제어 표면**으로 승격 — 터미널에서도 Chief 오케스트레이션·승인·메모리를 Discord 왕복 없이 쓰게 하라. 이게 핵심 파워유저(개발자)의 이탈을 막는다.

### Q5. 하나의 팀이 복수 조직·복수 리포에서 일하는데, 조직/리포마다 특화된 에이전트·skill이 있을 수 있지 않나? 워크플로우만으로 해결되나?

**[설계 의도]** 라우터가 **3-tier 스캔**을 한다: `<org>/.agents/`(org 로컬, **최우선**) > `~/.solosquad/agents/`(user 전역) > `<workspace>/.solosquad/agents/`(번들). 즉 **org 단위 에이전트 오버라이드가 이미 존재**하고, Chief는 명시적으로 org별 도메인 전문가화된다(CUSTOMIZATION_GUIDE).

**[객관적 평가]** 지적이 **절반 맞다.**
- **조직 특화 = 지원됨:** org 로컬 tier로 어떤 에이전트든 org별 SKILL.md 오버라이드 가능 + Chief org 커스터마이즈. 다만 이 메커니즘이 **저평가·저문서화**돼 "하나의 팀"이라는 마케팅이 균일성을 오해하게 만든다.
- **리포 특화 = 빈약:** **per-repo 에이전트/스킬 스코핑이 없다.** repo는 `repo.yaml`의 role·language 태그 + 워크플로 `target_repo`만 갖고, 자기 스택에 튜닝된 전용 architect/skill은 없다. 이질적 repo(예: Rust 서비스 + Next.js 앱)를 하나의 engineering 팀이 다루면 repo-특화 에이전트 대비 열위일 수 있다.
- **스킬 스코핑:** 3-tier 오버라이드는 **에이전트(SKILL.md 라우팅) 대상**이고, 팀 shared_skills가 참조하는 스킬 자체는 워크스페이스 번들이라 org/repo 3-tier로 갈리지 않는다.

**[근거]** `src/bot/agent-router.ts:22-25`(3-tier, org 최우선), `agents/main/chief/CUSTOMIZATION_GUIDE.md`, `teams/*/composition.yaml`(shared_skills=전역), repo 메타는 role/language뿐.

> **판정:** 워크플로우만으로는 특화가 안 되고, **org는 에이전트 오버라이드로 해결(문서화 필요), repo는 미해결**. 권고: 3-tier 스코핑을 스킬·repo 레벨로 확장하거나, "org-override가 특화 경로"임을 명문화.

### Q6. 왜 repo 단위가 아니라 조직 단위로 워크플로우를 설계하는가? 모노레포·폴리레포도 가능한데, 멀티 리포 작업이 에이전틱 코딩 시대에 실용적인가?

**[설계 의도]** 의사결정·시그널·가설·OKR이 **repo 경계를 넘어 축적**되므로 메모리·워크플로의 단위를 org로 둔다. 내부 문서도 "org 단위 공유 브레인은 올바른 선택"이라 본다(멀티리포 ideation §1.3).

**[객관적 평가]** **기획/의사결정 레이어의 org 단위는 정당**하다(비즈니스 컨텍스트의 자연 단위가 repo가 아니라 org). 그러나 질문의 실용성 우려는 **저장소 자신의 내부 문서가 이미 인정**한다:
- 멀티리포 실행은 **chief 대화에서만 우연히 작동**하고, **goal·cron은 `--add-dir`가 없어 외부 repo에 아예 못 닿으며**, 지정(targeting) 문법이 4개 실행 경로에서 제각각(`@slug`/`target_repo`/frontmatter/없음)이다 — "멀티 repo는 chief에만 되고 나머지는 단일 repo조차 깨졌다"(멀티리포 ideation §1.2·§1.4, 코드 file:line 근거).
- 업계 휴리스틱(Spacelift): PR의 **>30%가 repo 경계를 넘으면 모노레포**, **<10%면 폴리레포**가 유리. AI는 이 trade-off를 없애지 않고 "무엇이 더 중요한지"만 바꾼다. SoloSquad는 본질적으로 **폴리레포 + 조정 계층**인데, 폴리레포의 약점(원자적 cross-repo 변경)을 메울 조정 계층(--add-dir 매니페스트 주입 + 쓰기 권한)이 **아직 미완**이다(VS Code multi-root 실패가 반면교사).

**[근거]** `docs/ideation/260621-multi-repo-execution.md`(§1.2 세 경로 진단표, §1.3 org 메모리 타당, §2.1 Spacelift 휴리스틱), `src/bot/workflow-resolver.ts`, `src/engine/goal-runner.ts`(add-dir 부재).

> **판정:** org 단위 워크플로는 **기획 레이어에선 옳고, 실행 레이어에선 현재 부채**다. 에이전틱 코딩 시대에 --add-dir는 옳은 방향이나 매니페스트+권한 배선이 선행돼야 실용적. 많은 솔로 빌더에겐 모노레포가 단순하며, SoloSquad의 폴리레포 베팅은 **조정 계층이 실제 작동한 뒤에야** 정당화된다.

### Q7. 에이전트 팀이 복수 조직을 관할할 실효성이 있는가? 조직당 팀이 낫지 않나?

**[설계 의도]** 실제로는 **하이브리드**다: Layer 0(워크스페이스)에 팀·에이전트 **정의를 공유**하되, org마다 (a) 메모리 격리, (b) Chief 도메인 커스터마이즈(org당 1개), (c) org 로컬 에이전트 오버라이드(Q5)를 제공한다.

**[객관적 평가]** 이 하이브리드는 **1인 제약에 합리적**이다. 공유 base(19 에이전트를 org마다 재정의 안 함)로 유지비용을 아끼고, org 격리·Chief 커스터마이즈로 "조직당 팀"의 실질(맥락 분리·도메인 특화)을 상당 부분 이미 달성한다. "완전한 조직당 팀"은 유지보수를 N배로 늘린다. 남는 우려는 **오버라이드가 실제로 쓰이느냐(발견성·수작업 부담)** — 매우 다른 사업(핀테크 vs 컨슈머)을 겸하는 창업자에게 공유 specialist 정의가 너무 일반적일 수 있고, 이를 메우는 org 오버라이드는 사용자가 안 할 가능성이 있다.

**[근거]** `src/bot/agent-router.ts`(3-tier), `agents/main/chief/CUSTOMIZATION_GUIDE.md`, `<org>/memory` 격리(architecture.md §3).

> **판정:** 하이브리드가 효율적이며 "조직당 팀"보다 1인에 적합. 관건은 org 오버라이드의 **실사용률**. 온보딩이 org별 Chief 커스터마이즈를 강제/유도하면 실효성이 오른다.

### Q8. 오케스트레이션을 TypeScript와 Python 중 왜 TS를 택했는가?

**[결정적 발견]** **저장소 내부 최종 리뷰가 이미 "TS 근거는 약했고 Python이 맞다"고 자평했다.** `docs/ideation/260721-solosquad-final-review.md`의 최상단 결론: *"최종 결론: Python 오케스트레이션 중심 + TypeScript 프론트엔드."* 이 문서는 기존 TS 근거 3개를 스스로 해부한다:

| 기존 TS 근거 | 내부 리뷰 판정 |
|---|---|
| "웹 대시보드와 같은 언어여야" | ❌ 약함 (HTTP/JSON으로 통신하면 그만) |
| "CLI가 npm 배포라 TS" | 🔺 부분적 (CLI는 얇은 런처, Python도 가능) |
| "Claude Agent SDK 세션 훅이 TS 우선" | ✅ **유일하게 유효** (단 Python도 shell 훅 우회 가능) |

**[객관적 평가]** 정직한 답은 이렇다:
- **원래 TS 근거는 약했다(자인).** 진짜로 TS가 유리한 유일한 축은 Claude Agent SDK의 네이티브 세션 훅(SessionStart/End/TaskCompleted)이다. 그런데 **현 구현은 SDK가 아니라 Claude Code를 child process(`claude --print`)로 띄운다**(멀티리포 ideation §2.3, claude-runner.ts) — 즉 **SDK 네이티브 훅을 애초에 쓰지 않으므로** TS의 그 유일한 이점조차 실현하지 않고 있다.
- **핵심 루프가 Python 친화적.** 내부 리뷰는 SoloSquad의 무게중심이 "가설→시뮬레이션 검증→분석→재귀 개선"(특히 **유료 VMS 시뮬레이션 + ICML 연구**)이라 분석·시뮬레이션 밀착이 본질이고, 이는 압도적으로 Python 유리라고 결론냈다. 실제로 ICML 해커톤 수요 시뮬 연구도 Python으로 수행됐다.
- **그러나 전면 재작성은 비권장.** TS→Python 재작성은 PART A에서 A− 평가한 **성숙한 인프라(43개 마이그레이션·세션관리·스케줄러·ESM 게이트)를 통째로 버리는** 거대·고위험 작업이다. 내부 리뷰의 제안(Python core + TS frontend 모노레포, schema 단일진실, pipx/curl 배포)은 이상적이나 v1.4.3의 sunk maturity를 무시한다.

**[근거]** `docs/ideation/260721-solosquad-final-review.md`(전체), `docs/ideation/260621-multi-repo-execution.md §2.3`(child process 방식), `src/bot/claude-runner.ts`.

> **판정 및 권고:** "왜 TS?"의 정직한 답은 **"초기엔 웹·npm 근거로 택했으나 그 근거가 약했음을 내부적으로 인정했고, 분석·시뮬레이션 중심 무게중심상 Python이 더 맞다"**이다. 다만 전면 재작성 대신 **저위험 분할**을 권한다: 유료 VMS 시뮬레이션·분석 코어만 Python 모듈로 분리(subprocess/IPC로 호출)하고, 성숙한 TS 인프라·오케스트레이션은 유지. 이는 유료 제품(Python 친화)과 기존 자산(TS 성숙)을 모두 살린다. **핵심: SDK 세션 훅을 안 쓰는 현 구조에선 TS의 유일한 우위조차 실현되지 않고 있음을 인지할 것.**
