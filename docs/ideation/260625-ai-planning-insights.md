## 1. 종합 테제 — 4개 큰 흐름

수집된 18개 글을 가로지르면 **"무엇을 만들지를 결정론적으로 명세하던 기획"에서
"판단 환경·허용 범위·평가 기준을 설계하고, 에이전트가 세부를 확장하도록 오케스트레이션하는
기획"으로의 이동**이 거의 모든 소스의 공통 골격이다. 이를 4흐름으로 쪼갠다.

### A. PRD 의 무게중심 이동 — "단일 행동" → "허용 답변 범위 + Eval"
가장 강하고 반복적인 테제. (소스 7·12·16·19)

- **결정론 → 확률적.** 전통 SW는 "입력 A → 출력 B" 단일 행동을 명세하고 버그 고치면
  100% 정상화. AI 기능은 같은 입력에 출력이 달라지므로 **"받아들일 만한 답변의 범위
  (range of acceptable answers)"** 를 설계 단계에서 규정해야 한다. (yozm 3809)
  > "기존 PRD가 '단일 행동'을 정의했다면, AI PRD는 '허용 가능한 답변의 범위'를 정의."
  > "'무엇이 일어나야 하는가'가 아니라 '어떤 답이 받아들여질 만한가'를 정의."
- **Eval Plan 이 AI PRD 의 심장.** 입력–기대출력–실제출력–합격여부를 기록하는
  스프레드시트 20~30 케이스로 시작 → 운영 6개월이면 200줄. (yozm 3809)
- **"프롬프트 수렁(prompt swamp)"** — A를 고치니 B가 깨지는 무한 루프. 근본 원인은
  **회귀 테스트 부재**. 프롬프트 수정마다 누적 Eval 셋 전체 재실행이 유일한 탈출구. (yozm 3809)
- **3단계 평가 피라미드:** ① 규칙 기반(자동·저렴) → ② LLM-as-a-Judge(빠르나 편향) →
  ③ 사람 평가(느리고 비싸나 최종 검증). (yozm 3809)
- **평가 기준·가드레일·데이터셋 = 모델 교체 후에도 남는 고유 자산(Moat).** 모델·프롬프트는
  계속 변하므로 PO는 평가 체계에 우선순위를 둔다. (aitimes 210603, eopla 36939)
  > "기존 제품은 '시스템 동작 정의'가 주업무라면, AI 에이전트는 '판단 환경 설계'가 주업무." (aitimes)
  > "집요한 평가(Evaluation)가 곧 제품의 해자(Moat)가 되는 시대." (eopla 36939)
- **AI PRD 체크리스트 8항목** (yozm 3809): 기능 개요 / 입출력 명세 / 시스템 프롬프트 초안 /
  품질 기준 / 실패 정의 / Eval 계획 / 모니터링 계획 / 리스크 정의.

### B. agentic planning — 고수준 비전 먼저, 코드 전 계획 강제, 경계 체계
"어떻게 AI에게 일을 시키나"의 실무 레시피. (소스 5·6·8·14)

- **고수준 비전 먼저, 세부는 에이전트가 확장.** 대규모 스펙을 한꺼번에 주지 말고 목표+핵심
  요구사항만 명확히 → 에이전트가 상세 계획 작성 → Plan Mode 읽기전용 검토 후 실행. (hada 25949)
- **"지시의 저주(curse of instructions)"** — 한 프롬프트에 요구사항이 많을수록 각각을 정확히
  따르는 성능이 급락. 한 번에 하나의 태스크. (hada 25949) ↔ yozm 3221 의 "전체 PRD 일괄 입력
  금지, 시나리오→유즈케이스→예외 순 반복" 과 같은 말.
- **코드 전 계획 강제**가 잘못된 구현을 사전 차단. 구현 전 15~20분 연구: 베스트프랙티스 검색 →
  기존 코드베이스 패턴 검색 → 라이브러리 확인. (hada 24266)
- **병렬 연구 에이전트** — 버그 재현·BP 검색·코드베이스 분석·git 히스토리 조사를 에이전트가
  병렬 수행 후 단일 계획으로 트레이드오프 포함 통합. 병렬은 초반 2~3개 독립 태스크만. (hada 24266, 25949)
- **3단계 경계 체계 — Always do / Ask first / Never do.** 에이전트 자율성과 인간 통제의 균형. (hada 25949)
- **누적 학습의 성문화** — "이거 싫어함 / 잘 잡았어" 피드백을 CLAUDE.md 류 문서에
  "X 작업 시 Y 체크" 규칙으로 축적. 쌓이면 "Security Checker", "Event Tracking Expert" 등
  **전문 리뷰어 에이전트**로 승격. (hada 24266)
- **PRD = AI 지시의 전제이자 컨텍스트 앵커.** 텍스트 서술형 필수("어설프게 그림 그리지 말 것"),
  `@PRD.md` 한 파일을 읽혀두면 프로젝트 내내 맥락 유지. (yozm 3221, wonhwa 78)
  > "말로 하는 게 제일 쉽다. 글로 써 보면 머릿속 에덴동산도 유니콘도 사라진다." (yozm 3221)
  > 기획 "모던한 2층 집" → PRD "1층 바닥 대리석, 2층 방 3개 화장실 2개." (wonhwa 78)
- **프레임워크:** GitHub **Spec Kit 4단계 게이트 — Specify → Plan → Tasks → Implement.** (hada 25949)

### C. 기획자 역할 이동 — 정의자 → 가이드 설계자 / 오케스트레이터 / 검토·판단자
(소스 1·3·4·7·20)

- **IC → 오케스트레이터.** 창업자 역할이 개인 기여자에서 에이전트 오케스트레이터로. 생명주기
  Idea→Scale 4단계 각각에 "목표 / 종료 기준 / 실패 모드 / AI 연습 / 프레임워크 / 프롬프트"
  6요소 표준화. (hada 30598)
- **스펙 정의 → 가이드 설계.** "변수 대응형 가이드라인"을 짠다. (hada 27343)
  > "스펙을 정의하는 기획에서 가이드를 설계하는 기획으로"
  > "최악의 데이터부터 버티는 디자인" / "유저에게 조종석을 주는 인터페이스"
- **Skills = 업무 매뉴얼화.** "이 상황에선 이렇게" 규칙을 정의해두면 일관 실행. 일회성 질문과
  달리 작업 흐름을 반복 재사용. **Subagent 로 복잡 업무 분해**가 더 안정적. 기획자는
  데이터 조회→1차 분석→초안까지 자동화 후 "검토와 판단에 집중". (brunch @c5738)
- **CLI/슬래시 커맨드로 일상 전체 자동화.** IDE를 4년 피한 기획자(Teresa Torres)가 매일 아침
  `/today` 한 줄로 할 일 생성, 밤사이 논문 요약 자동 전달. 실무자가 자기 시스템을 "소유". (eopla 38874)
- **PO 는 정답 대신 시험지(평가 기준)를 설계한다.** (aitimes 210603)

### D. 문서 체계·위계 — 살아있는 단일 진실 + PRD/AC/Policy
(소스 8·18)

- **PRD → AC → Policy 3단 위계.** PRD(무엇을) → AC(언제 완성인가) → Policy(어떤 기준으로
  일관성 유지). 우수한 PM 문서는 템플릿이 아니라 **의사결정 도구**. (onemorethink)
  - AC 는 **Given-When-Then** 으로 작성 → 테스트 자동화/QA 연계.
    > "Given 비회원 / When 이메일 가입 시도 / Then 인증 메일 발송."
- **스펙 = 살아있는 단일 진실.** SPEC.md 를 Git 에 커밋, '생성 후 방치'가 아니라 프로젝트와
  함께 진화. 효과적 스펙 6영역: Commands / Testing / Project Structure / Code Style /
  Git Workflow / Boundaries. (hada 25949)

### (전이) 인접 원리 — AI 기획 직접 주제는 아니나 방법론에 흡수 가능
- **전략 = 트레이드오프 선택.** "둘 다"가 아니라 포기를 명시 → PRD 의 **비범위/Non-goal** 근거. (hada 27186)
- **청자 기준 역설계.** 평가 기준(배점)을 역분석해 콘텐츠 비중 결정. 청자가 실제 필요로 하는
  정보 우선. (eopla 32173) ↔ 우리 메모리 'trend-tracker 청자 = 본인', '문서 위계' 원칙과 일치.
- **도메인 전문성 = 문제정의 역량.** 표면 불편을 데이터로 진짜 문제로 재정의. (brunch @try-harder)
- **소수 고신호 지표.** 대시보드 관리보다 "문제보다 빠르게 학습". (hada 27185)

## 3. 가장 큰 발견 — "founder 가 만드는 건 대개 AI 제품인데, 우리 PRD 양식엔 Eval 층이 없다"

소스 7·12·16·19 가 한목소리로 말하는 것: **AI 제품의 PRD 는 결정론적 SW 의 PRD 와
다른 종(種)이다.** "사용자는 X를 할 수 있다" 식 행동 명세만으로는 부족하고, **"이 출력이
받아들일 만한가"의 판단 환경 + 회귀 가능한 Eval 셋 + 가드레일**이 핵심이 된다.

그런데 SoloSquad 의 두 PRD shape([[v1.3.8_docs-management]] §2-4):
- **내부 dev PRD**(본 양식): 결정론적 기능 명세 → 현 양식으로 충분. Eval 층 불필요.
- **founder 제품 PRD**(prd 스킬 8-section): founder 가 SoloSquad 로 만드는 제품은 **십중팔구
  LLM/에이전트 제품**. 그렇다면 8-section 에 **Eval Plan / 허용 답변 범위 / 가드레일 / 실패
  정의**가 들어가야 한다. 지금은 없다.

> 이건 [[v1.3.8_docs-management]] 가 "PRD shape 를 맥락별로 명문화"한다면서도 *AI 제품 PRD vs
> 일반 제품 PRD* 라는 **세 번째 축**을 놓치고 있다는 뜻. v1.3.8 은 외부/내부 축은 잡았지만
> 결정론/확률론 축은 안 잡았다.

이게 맞다면 후속 버전 후보: **"AI 제품 PRD 양식" = founder 8-section + Eval 부록**.
prd 스킬이 "이 제품에 LLM/에이전트가 들어가나?"를 묻고, 그렇다면 yozm 3809 의 8항목 체크리스트
(입출력 명세·시스템 프롬프트 초안·품질 기준·실패 정의·Eval 계획·모니터링·리스크)를 분기 추가.

## 4. 곧장 흡수 가능한 실무 패턴 (낮은 비용)

소스에서 바로 우리 primitive 에 박을 수 있는 것들:

1. **Spec Kit 4단계 게이트(Specify→Plan→Tasks→Implement)** (hada 25949) — requirements-analysis
   워크플로의 단계 명명을 이 어휘로 정렬하면 외부 담론과 호환. 이미 사실상 이 구조.
2. **Always/Ask/Never 3분류를 에이전트 boundaries 에 명시화** (hada 25949) — 현재 dev confirm
   gate 가 'Ask first' 만 강하게 잡음. 'Never do' 를 에이전트 SKILL.md 에 명문 블록으로.
3. **Given-When-Then AC 층** (onemorethink) — PRD 의 §검증(수용 기준)을 Given-When-Then 으로
   쓰면 테스트 자동 생성과 연계. 우리 내부 dev PRD §9 검증을 이 포맷으로 강화 검토.
4. **PRD 완성도 척도(30→50→100점, 배포 후 70 리셋)** (yozm 3221) — PRD 가 "언제 리뷰 시작 /
   언제 ship 가능"인지 단계 표시. docs 스킬의 PRD frozen 판정 기준에 흡수 가능.
5. **누적 Eval 셋 = 회귀 테스트** (yozm 3809) — 우리가 만드는 *스킬/에이전트*도 LLM 산출물이므로,
   primitive-review 에 "프롬프트 수정 시 누적 케이스 재실행" 개념이 들어갈 여지. (지금은 1회성 검증)
6. **트레이드오프 통합 단일 계획** (hada 24266) — PM convention `minimum_approaches: 2` 와 정확히
   일치. 이미 [[v1.3.8_docs-management]] §6 에서 dogfooding 중.
