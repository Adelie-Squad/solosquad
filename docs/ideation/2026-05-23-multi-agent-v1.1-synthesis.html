  2. AI 에이전트 하네스 보고서 (docs/ideation/AI_Agent_Harness_Report.md) — 이론·청사진·SoloSquad 비교가 이미 §7 에 박제됨. 4 채택 권고 (Educational Nudge / Dependency Injection / Experiment Proposal / Board of Agents) 가 결론.

  4. v1.1 plan (docs/prd/v1.1-multi-agent-messenger-collaboration.md, untracked) — 5 코어 봇 직원화 + Forum Channel + 9-hop diagnostic + 10 BP (Q~Z).

3.Hermes Agent 설계안 V2 핵심 (Team-Centric)

docs/ideation/multi-agent-directory.md 의 *외부 시각 멀티 에이전트 설계안 V2*. 2026-05-23 개정. V1 대비 5건 변화 박제 — Team layer 신설 / Specialist Pool 분리 / Webhook 약화 / Context Injection 4단계 / Python+Pydantic+FastAPI+Redis 명시.

3.1 위계 — 4-layer → 5-layer (Team 신설)

계층구성역할SoloSquad 대응
L1. GatewayDiscord/Slack + Webhook사용자 접점 + 가상 페르소나 출력src/messenger/
L2. OrchestratorWorkflow + Scheduler[지휘] 전체 공정 + 에이전트 호출 순서PM session (src/bot/pm-runner.ts) + src/scheduler/ + WorkflowReconciler
L3. Team V2 NEWTeam Goal + Knowledge Base[지식 공유] 팀 공동 목표 + 벡터 DB 공통 지식assets/agents/{team}/KNOWLEDGE.md (v0.6 §2.1) 정합
L4. AgentMain Agents (Role-based)[판단] 팀 목표 인지 + sub-agent/skill 활용 결정PM 봇 + (v1.1 plan) 4 specialist 봇 = 5 코어
L5. SpecialistShared Sub-Agents + Skills[실행] 재사용 전문 지능 + 물리 도구25 specialist via Task tool (Claude Code subagent — isolated context) + Bash/MCP tools

핵심 정합: V2 의 L3 Team layer 가 SoloSquad v0.6 의 assets/agents/{team}/KNOWLEDGE.md 와 *정확히 일치*. SoloSquad 가 이미 *4 team × team-shared KNOWLEDGE* 구조 (strategy/growth/experience/engineering) 채택 — V2 의 새 L3 가 SoloSquad 의 기존 구조를 *외부에서 재확인*.

3.5 Specialist Pool — 재사용 sub-agent V2 신설

§4.3 of Hermes V2 — "Specialist Pool"

  • 개념: specialists/ 폴더의 sub-agent = *독립된 인격체 + 고성능 지능 도구*
  • 공유: "검색어 최적화 전문가" specialist 를 *리서처* 도 호출, *작가* 도 호출
  • 연결: 메인 에이전트가 specialist 의 DESCRIPTION.md 를 읽고 *마치 skill 처럼* 사고(reasoning) 업무 요청

SoloSquad 정합: v1.1 plan §3.1 의 5 코어 봇 (PM + 4 specialist 봇) 모델 — Hermes V2 의 *main + specialists 분리* 와 정확히 일치. PM 봇 = main, 4 봇 (eng/design/growth/strategy) = specialist pool. 본 25 SKILL.md 는 *4 specialist 봇 안의 sub-skill* 로 매핑 (v1.1 plan §8.2 표).

차이: Hermes 는 specialists 의 *재사용성* 강조 (한 specialist 여러 main 호출). SoloSquad 는 *team-bound* (eng SKILL 은 eng 봇만, design SKILL 은 design 봇만 — 권한 격리). V2 의 cross-team 재사용 패턴은 SoloSquad 의 *meta SKILL* (workflow-maker) 정도만 채택 중. → §8 갭 후속 검토 후보 신설.

3.6 Context Injection — 4단계 명시 V2 신설

V2 §4.2 — 에이전트 실행 시 *프롬프트 조립* 순서:

단계Hermes V2SoloSquad v0.6 §2.2 8-layer JIT 매핑
1. 팀 목표"우리는 OO 서비스를 홍보하는 팀이다"Layer 4 (<org>/core/PRINCIPLES.md) + Layer 5 (<org>/agent-profile.yaml 의 team 절)
2. 팀 지식 (RAG)"우리 서비스의 주요 특징은 A·B·C"Layer 2 (agents/{team}/KNOWLEDGE.md — same team only) + Layer 6 (<org>/domain/)
3. 개인 역할"너는 이 팀의 리서처로서 시장 조사를"Layer 3 (agents/{team}/{agent}/SKILL.md — agent identity)
4. 현재 상황"기획자가 보낸 초안을 검토하라"Layer 7 (<org>/workflows/<id>/_handoff.md slice + memory recall) + Layer 8 (target repo context)

→ SoloSquad 의 8-layer JIT 가 V2 의 4단계를 *완전 포함*. SoloSquad 는 V2 의 4단계 외에도 *Layer 1 workspace knowledge* + *Layer 5 agent-profile budget* + *Layer 8 target repo* 추가로 주입 — 더 정교함.

8.1 Educational Nudge 우선 높음

패턴: 사용자가 막연한 입력 → PM 이 *도메인 가이드 우선 제시* → 사용자 ack 후 dispatch.

예시 (Harness §6.3):

사용자: "가격 책정 어떻게 하지?"
PM (Educational Nudge): "현재 시장의 LTV/CAC 비율은 3:1이 적합합니다. 이에 따른 가격 구간은… [strategy/policy-architect KNOWLEDGE.md slice]"
PM: "이 가이드 기준으로 계속할까요?"

v1.1 plan 적용 위치: §5.2 supervisor 패턴 + §10 (PM SKILL.md) 룰 추가.

구현: PM 의 triage step 0 = "막연도 점수 측정 → 기준치 초과 시 관련 team KNOWLEDGE.md slice 인용 + ack 대기".

크기: orchestrator SKILL.md ~20줄 + agent KNOWLEDGE.md ↔ domain 매핑 표.

8.2 Dependency Injection (지표) 우선 높음

패턴: Stage 도메인 키워드 감지 시 *관련 지표 데이터* 자동 첨부 (예: BM/Pricing stage → LTV/CAC 자동 inject).

v1.1 plan 적용 위치: §3.4 (skill preload) + 8-layer JIT 의 layer 6.5 신설 — "stage 키워드 매칭 → <org>/memory/signals.jsonl 필터링 inject".

구현:

  1. stage ↔ 도메인 키워드 매핑 (예: BM = ["가격", "LTV", "CAC", "revenue", "pricing"])
  2. spawn-assembler 에 *지표 layer* 추가 — 매핑 매칭되면 signals.jsonl 의 최근 N rows 의 키 값 추출
  3. subagent context 에 <injected-metrics> 블록으로 prepend

크기: spawn-assembler ~50줄 + 매핑 yaml.

8.3 Experiment Proposal 자율 노드 우선 중

패턴: PM 이 *다음 실험 가설을 자율 제안* → 사용자 ack → 사이클 진입.

v1.1 plan 현재: §11.4 (Amplitude 패턴) 가 *결과 분석 → 결정 권고* 까지만 명시. *다음 cycle 가설 자율 제안* 단계 부재.

적용 위치: §11 goal cycle 의 새 stage — PROPOSE_NEXT 박제.

구현: v0.4 goal-runner CONFIRMING 상태머신에 PROPOSE → AWAIT_USER_ACK → CYCLE 추가. PM 봇이 직전 cycle 결과 + signals.jsonl + KNOWLEDGE.md 의 가설 패턴 라이브러리 참조하여 *제안* 발화.

8.4 Board of Agents 메타포 우선 낮음
