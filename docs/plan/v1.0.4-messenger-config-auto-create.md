### 1.2 Root cause 추적 — *2-단계 분기 중 첫 번째만 정정*

v1.0.3 의 `syncGuildProductMapping` 본 두 줄:
```ts
if (!this.client || !this.ownOrgSlug) return;        // ① ownOrgSlug 결정자 (v1.0.3 정정 ✓)
const orgSlug = this.ownOrgSlug;
const configFile = path.join(getReposBase(), orgSlug, "discord", "config.yaml");
if (!fs.existsSync(configFile)) return;              // ② file existence 분기 (v1.0.3 미정정 ✗)
```

v1.0.3 가 ①의 서버명 휴리스틱 (`guild.name.includes(slug)`) 은 제거했지만 ②의 silent early-return 은 그대로 둠. **두 번째 분기는 같은 *silent-bail* 패턴** — *file 이 없으면 조용히 종료*. v0.2.x scaffoldOrg 가 `<org>/discord/` *빈 디렉터리만* 만들고 `config.yaml` 은 never 작성 → 모든 fresh `solosquad init` 워크스페이스가 ② 에서 차단.

### 1.3 정직한 박제 — v1.0.3 plan §6 패턴에 *3번째 변형 (silent file-bail)* 추가

v1.0.3 plan §6 박제:
> *외부 자유 입력과 내부 슬러그를 비교하는 코드 + v0.1.x 잔재 string literal 모두 trip-wire 대상*.

본 v1.0.4 가 추가하는 **3번째 변형:**
> *권위 결정자가 있는데도 옛 기록 파일 유무로 silently bail 하는 코드도 trip-wire 대상*. `if (!fs.existsSync(x)) return;` 류 패턴은 *기록이 없으면 만들어 둬야* 진행 가능한 결정자 정보가 이미 손에 있는 경우 (본 케이스: ownOrgSlug + guild 객체) silent bail 대신 *auto-write* 가 정직.

향후 회귀 catcher 가이드라인 보강 (v1.0.4 catcher 자체가 이 가이드라인 따름).

## 2. Bug G — `syncGuildProductMapping` load-or-empty + auto-write

### 2.1 채택안 — load-or-empty 패턴

```ts
private syncGuildProductMapping(): void {
  if (!this.client || !this.ownOrgSlug) return;
  const orgSlug = this.ownOrgSlug;
  const configDir = path.join(getReposBase(), orgSlug, "discord");
  const configFile = path.join(configDir, "config.yaml");
  fs.mkdirSync(configDir, { recursive: true });        // 디렉터리 보장

  // v1.0.4 — load-or-empty. Pre-v1.0.4 returned silently when missing.
  const config = fs.existsSync(configFile)
    ? ((yaml.load(...) ?? {}) as Record<string, unknown>)
    : ({} as Record<string, unknown>);

  // (이하 guild.id + channel.id 박제 + dirty 플래그로 writeFile 한 번만)
}
```

핵심 변경:
- **file 없으면 빈 객체로 시작** — silent bail 제거
- **dirty 플래그** 도입 — guild_id 또는 channel id 가 실제로 바뀐 경우에만 writeFile (idempotent 보강)
- mkdir -p — 디렉터리 자체도 누락된 케이스 방어 (scaffoldOrg 가 만들었어야 하지만 사용자가 지웠을 수 있음)

봇 첫 시작 시 흐름:
1. `ownOrgSlug = "rosyocean"` (channel-bootstrap 가 결정)
2. `configFile = <reposBase>/rosyocean/discord/config.yaml` — 없음
3. load-or-empty → `config = {}`
4. guild 0 또는 1개 — 1개라면 그 guild 의 ID 박제 → dirty=true
5. writeFile + `[Discord] Bound guild <name> (<id>) → org=rosyocean` 로그
6. 다음 메시지부터 `getProductByGuild` 정상 동작 (configFile 존재 + guild_id 일치)

### 2.2 회귀 catcher (`test/v1.0.4-config-auto-create.test.ts`, 4 cases)
- discord-adapter source 가 더 이상 `if (!fs.existsSync(configFile)) return;` 패턴 안 가짐 (silent-bail trip-wire)
- load-or-empty 삼항식 존재 (auto-create 의미)
- `mkdirSync(configDir, recursive: true)` 호출 존재 (디렉터리 보장)
- `Bound guild ... → org=` 로그 보존 (v1.0.3 의 명시-결정 로그 유지)

### 7.2 Discord ↔ AI Agent/LLM 연결 *9-reference 조사* (v1.0.4 진행 중 추가 박제)

OpenClaw / Claude Code Channels / LangChain / AutoGen (Composio) / Composio MCP / llmcord / openai/gpt-discord-bot / LibreChat-DiscordBot / AnythingLLM 등 9개 레퍼런스 비교 (`docs/trend-record/2026-05-23-discord-agent-references.md` 신규 박제 후보).

**모든 레퍼런스 공통 *절대 필수* 값 3개:** ① Bot Token, ② Message Content Intent, ③ OAuth `bot` scope. 나머지는 자동 캡처 가능.

**바인딩 시점 패턴 4가지:**
- (a) Env-only (openai/gpt-discord-bot, llmcord, LibreChat) — 결정적이지만 ID Copy-paste 실수 시 silent 차단
- (b) **Runtime pairing (OpenClaw, Claude Code Channels)** — 자동 캡처 + 명시 approve. 주류 표준
- (c) Auto-discovery + OAuth (Composio) — MCP 위임, 외부 의존
- (d) Per-call (LangChain) — 정책 외부화

**SoloSquad 진단:** (b) 패턴의 *절반만 채택* — 자동 캡처 (`syncGuildProductMapping` auto-write, v1.0.4) 는 도입했지만 *명시 approve 단계 부재*. 따라서 자동 캡처 chain 의 어느 마디가 깨져도 silent fail. v1.0.2 → v1.0.4 의 4번 누적 회귀가 모두 이 패턴.

**5 Best Practice (9-reference 합의):**
- **L:** 명시 approve 게이트 의무화 — 자동 캡처된 snowflake 도 사용자 *한 번* approve. *"Don't auto-pick even when there's only one"* (Claude Code Channels SKILL.md 명시 원칙)
- **M:** pending → allowlist 2-state 머신 — 페어링 임시/잠금 상태 분리 + TTL 만료
- **N:** snowflake 종류별 명명 강제 — `SenderId` / `ChatId` / `GuildId` / `ChannelId` 별 TypeScript branded type. 현재 모두 `string` — type-level 혼동 가능
- **O:** token 해석 우선순위 명문화 — env < `.solosquad/.env` < CLI flag, strict precedence + 경계 로그
- **P:** **누락값 hard fail + actionable hint** — 모든 silent `return null;` / `return;` 패턴 제거 → `throw` 또는 *진단 메시지*. **본 v1.0.4 가 Best Practice P 의 일부 즉시 적용** (`diagnoseProductByGuildFailure` helper + 5-hop 진단 메시지)

### 7.3 v1.x 슬롯 — *L+M+N+O+P remainder*

위 5 best practice 중 v1.0.4 에 *Best Practice P 일부만 흡수*. 나머지는 *v1.1 redesign* 후보:

| 항목 | 표면 | 권고 슬롯 |
|---|---|---|
| L (페어링 코드 + approve CLI) | 신규 CLI 명령 `solosquad pair` — *minor* (api-stability §3) | **v1.1** |
| M (snowflake branded types) | TS type 추가 + 전수 적용 — 8+ 파일 surface | v1.1 또는 v1.2 |
| N (silent early-return 전수 제거) | 전 코드베이스 audit + hard-fail 변환 — 표면 크지만 동작 변화 명확 | v1.1 |
| O (token precedence 명문화) | `src/util/config.ts:loadEnv` 등 명시 + 로깅 | v1.0.5 patch (작음) |
| **P 잔여** (Discord 외 다른 silent bail) | scaffold/migration/scheduler 등에서 동일 패턴 audit | v1.0.5 ~ v1.1 점진 |

본 v1.0.4 는 *Discord 메신저 어댑터의 5-hop 진단* 만 도입 → 차후 redesign 의 *디버깅 인프라* 역할.
