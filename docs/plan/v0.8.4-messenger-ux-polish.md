# v0.8.4 — Messenger UX Polish

> v0.8.0에서 멀티 유저 채널 모델은 잡았지만, 사용자 진입 시점의 *셋업 경험*과 *메신저 내부 사용자 매뉴얼*은 아직 정책 수준. 사용자가 코드를 직접 보지 않고 메신저를 통해 모든 작업을 한다는 product goal #2를 지키려면, 메신저 셋업 자체가 *오류 없이 끝나고* 봇을 즉시 활용할 수 있어야 한다. v0.8.x 시리즈 polish 마지막 패치.

**로드맵:** v0.8.0 Multi-User Messenger → v0.8.1 Security & Lifecycle Pair → v0.8.2 Dev Capability → v0.8.3 Onboarding UX + Observability → **v0.8.4 Messenger UX Polish** → v0.9 안정화 + 자체 사용 검증 → v1.0 정식 출시

**의존:** v0.8.0 (`command-<handle>` / `works-<handle>` 모델), v0.5 (`_meta/workflow-maker` SKILL 패턴), v0.8.3 (`solosquad doctor` mismatch 감지)

**기준 버전:** v0.8.4. patch. 새 모델 도입 없이 사용자 진입 경로(셋업·learn·복구) 다듬기만.

**분류:** UX polish + 메신저 내부 매뉴얼. 새 기능 추가 적음 (`@bot help`·`solosquad init --verify`·sticky setup-guide), 대부분 기존 흐름 개선.

## 핵심 질문 & 성공 기준

| 핵심 질문 | 성공 기준 |
|---|---|
| 사용자가 README를 보지 않아도 메신저에서 setup·명령어를 학습할 수 있는가? | `@bot help` 또는 `/help` 슬래시로 1줄 → 봇이 채널에 step-by-step setup·CLI 표·운영 가이드 출력 |
| 셋업 시나리오가 *오류 없이* 끝나는가? | `solosquad init --verify` 1회 실행으로 (a) 봇 token API 호출 성공, (b) 채널 페어 생성 성공, (c) 첫 메시지 echo 통과까지 e2e 검증. 실패 시 정확한 진단 메시지 |
| 사용자 직접 설정이 *최소화*되는가? | init wizard 6단계 → 4단계로 축소 (handle 자동 추출·timezone 자동 감지·broadcast 기본 off라 입력 0). 사용자 입력 = 봇 token 1건 + org 이름 1건 + 첫 messenger 채널 초대 |
| 채널 생성 실패 시 *복구 가능*한가? | doctor가 부재한 채널 감지 + `solosquad messenger ensure-channels` 명령으로 미생성 채널 idempotent 재생성 |
| 봇이 새 사용자에게 자동으로 *환영 + 사용법* 안내하는가? | 첫 install 시 `command-<handle>`에 1회 sticky welcome message — setup 완료 확인 + 다음 단계 4건 + `@bot help` 안내 |

## 1. 현재 문제 (v0.8.3 시점)

| 문제 | 원인 |
|---|---|
| 메신저 내부 사용자 매뉴얼 부재 — 사용자가 명령어·셋업·운영 가이드를 알려면 README나 master-guide.html을 별도로 열어야 함. *코드를 안 보는 사용자* 가정과 모순 | 봇 응답에 도움말이 노출되지 않음 (`@bot help` 자동 응답 없음) |
| 셋업 e2e 검증 부재 — `solosquad init` 마법사가 끝나도 봇이 실제로 채널 생성·메시지 수발신 가능한지 확인하는 단계 없음. 사용자가 `solosquad bot` 시작했을 때 처음 발견 → 그 시점에 문제가 있으면 복구 절차 모호 | init과 bot 사이 검증 gap |
| 채널 부재 복구 명령 없음 — 봇 startup 시 채널 페어가 부재하면 자동 생성 시도하지만, API 권한 부족 등으로 실패하면 봇이 침묵. 재시도 방법 명시 없음 | 진단 → 명령으로 가는 흐름 부재 |
| init wizard 6단계 — 사용자 입력 항목이 많아 손이 많이 감 (handle·timezone·메신저 선택·token·org 이름·broadcast 등). 일부는 자동 추출 가능 | 마법사가 모든 항목을 사용자에게 물음 |
| 봇이 새 사용자에게 *환영 메시지 + 사용법*을 자동으로 안내하지 않음 — 첫 install 후 `command-<handle>`에 입장만 하고 끝 | 자동 환영 흐름 부재 |
| 봇 사용자 매뉴얼 (`@bot help` 또는 sticky message) 의 콘텐츠 source 부재 — README와 master-guide는 *코드 작업자/관리자* 시점, 메신저 사용자 시점 매뉴얼 별도 필요 | 콘텐츠 자체 + 메신저 노출 양쪽 부재 |

## 2. 원칙

| 원칙 | 구현 |
|---|---|
| **메신저가 1차 사용자 인터페이스** | README·CLI는 *코드 관리자* 시점. 일반 사용자는 메신저 안에서 모든 학습·운영 가능해야 함. 봇이 "이 채널에서 무엇을 할 수 있는가"를 자동 안내 |
| **셋업 오류는 init에서 빠른 fail + 명확한 진단** | `--verify` 단계가 통과해야 init이 끝나는 것. 실패 시 사용자가 *바로 다음에 무엇을 해야 하는지* 명시 |
| **사용자 직접 설정 최소화** | 자동 추출 가능한 값(handle·timezone·display_name)은 prompt 없이 자동. 사용자가 입력하는 항목은 *대체 불가*한 값만 (봇 token, org 이름) |
| **채널 부재는 idempotent 복구** | `solosquad messenger ensure-channels` 명령으로 채널 페어 생성 재시도. 이미 있으면 no-op. 실패 시 정확한 권한 진단 |
| **봇 매뉴얼의 source-of-truth는 한 곳** | `assets/messenger-manual/` 디렉토리 1개. README·master-guide·`@bot help` 모두 이 source에서 빌드 |

## 4. 셋업 e2e 검증 — `solosquad init --verify`

### 4.1 흐름

```
solosquad init
  ↓ (기존 wizard, 단계 단순화 — §5)
  ↓
solosquad init --verify (자동 호출 + 명시 호출도 지원)
  ↓ 5 단계 e2e:
  1. .env 봇 token 검증 (API ping)
  2. 채널 페어 자동 생성 (이미 있으면 no-op)
  3. 봇이 #command-<handle>에 첫 sticky welcome 메시지 push
  4. 같은 채널에서 echo 테스트 (자동 echo: "verify-ping" → 봇 응답 확인)
  5. broadcast 채널 (opt-in일 때) 생성 확인

각 단계 실패 시:
  ✗ "API ping failed: invalid_token"
    → 토큰 재확인. .env의 DISCORD_TOKEN 또는 SLACK_BOT_TOKEN 검토
  ✗ "channel create failed: missing_scope (Slack)"
    → 봇에 channels:manage 스코프 추가 + reinstall
  ✗ "echo timeout: 봇이 응답 안 함"
    → solosquad bot 별도 터미널에서 실행 확인
```

### 4.2 자동 실행 vs 명시 호출

- `solosquad init`이 종료 시 자동으로 `--verify` 실행 (사용자 입력 후)
- 별도 호출도 가능: `solosquad init --verify` (init 후 다시 검증)
- doctor에 same 검증 메뉴 추가 (`solosquad doctor --messenger-verify`)

## 5. init wizard 단계 축소

### 5.1 현재 (v0.8.3) vs 신규 (v0.8.4)

| 항목 | v0.8.3 (6단계) | v0.8.4 (4단계) |
|---|---|---|
| Owner name | 사용자 입력 | 메신저 handle에서 자동 추출 (`<handle>` 그대로) — 변경 원하면 그때만 prompt |
| Owner role | 사용자 입력 | 생략 (기본 "founder"). 변경은 `.env` 또는 `agent-profile.yaml`에서 |
| Timezone | 사용자 입력 | `Intl.DateTimeFormat().resolvedOptions().timeZone` 자동 (사용자가 명시 변경 prompt만) |
| Messenger 선택 | 사용자 선택 (discord/slack) | 사용자 선택 유지 |
| 봇 token | 사용자 입력 | 사용자 입력 유지 (대체 불가) |
| Org 이름 | 사용자 입력 | 사용자 입력 유지 (대체 불가) |
| broadcast | 사용자 선택 | 자동 off (기본). 변경은 별도 명령으로 |
| handle | 사용자 입력 + 확인 | 봇 token API로 자동 추출 + 명시 확인만 |

**핵심**: 사용자 입력 = 봇 token + org 이름 + (선택) messenger + 첫 messenger 채널 초대. 다른 항목은 자동.

### 5.2 wizard 흐름

```
1. 메신저 선택 (discord / slack)
2. 봇 token 입력 → API ping → handle·display_name·bot_user_id 자동 추출
   "감지됨: alice (Alice — Discord). 채널 'command-alice' + 'works-alice'
    생성합니다. 변경하려면 입력 [기본: alice]:"
3. 첫 org 이름
   (timezone·owner_name·role·broadcast 등은 자동/기본값 — 사용자에게 묻지 않음)
4. --verify e2e 자동 실행 + 결과 보고
```

## 6. 봇 입장 환영 메시지 — Sticky

### 6.1 트리거

채널 페어 첫 생성 시점 (`channel-bootstrap.ts`의 채널 생성 직후) 1회.

### 6.2 콘텐츠

```
👋 SoloSquad 봇이 여기서 알리스(alice) 워크스페이스를 운영합니다.

빠른 시작:
  • 명령은 이 채널 (#command-alice)에서. 다른 사용자가 입력하면 봇이 무시합니다
  • 작업 결과·진행 보고는 #works-alice 에 스레드로 나타납니다
  • `@bot help` — 사용 가능한 명령·specialist 목록
  • 작업 3분류: 워크플로우 / goal / 루틴 (자세히 `@bot help`)

다음 단계 (권장):
  1. "오늘 morning brief 받아볼게" — 자동 routine 활성화 확인
  2. "specialist 25명 보여줘" — 사용 가능 agent 목록
  3. "[제품 이름]의 PMF 검증 goal 시작" — 첫 goal 실행
```

Discord: pin to channel. Slack: pin to channel (`pins.add`).

봇은 이 메시지를 idempotent하게 다시 push하지 않음 — channel pinned messages에 자기 발신 sticky가 이미 있으면 skip.

## 7. 채널 부재 복구 — `solosquad messenger ensure-channels`

### 7.1 명령

```bash
solosquad messenger ensure-channels [--org <slug>] [--dry-run]
```

### 7.2 동작

1. `<org>/.solosquad/users/*.yaml` 로드
2. 자기 봇이 listen하는 user yaml의 `channels.command` + `channels.works` 메신저에 존재 확인
3. 부재 시 생성 시도 (private + permission overwrites)
4. 실패 시 정확한 진단 (권한 부족·메신저 API rate limit 등)
5. broadcast (opt-in 시) 페어도 확인

`--dry-run`: 무엇이 생성될지 출력만, API 호출 없이 종료.

### 7.3 사용 시나리오

- 사용자가 봇 종료한 상태에서 채널 *수동 삭제*하고 다시 시작 → 부재 감지 → 재생성
- 권한 변경 후 (예: 봇에 channels:manage 새로 추가) → 재시도

## 9. 비파괴성 + 복구

- `init --verify`는 verify 자체로 인한 채널 생성은 *idempotent* (이미 있으면 no-op)
- `ensure-channels`는 항상 idempotent
- welcome sticky는 1회 push + pinned message dedup으로 재실행 안전
- 기존 v0.8.3 사용자(있다면)는 마이그레이션이 자동 — 추가 user yaml 작성 0, sticky만 다음 봇 startup 시 시도
