# v1.0 — 정식 출시 (Official Launch)

> v0.x 전체는 *솔로 파운더 자기 사용*을 위한 빠른 반복 구간. v1.0은 **공개 사용자 약속이 시작되는 마일스톤** — surface freeze + api-stability promise 발효. 본 plan 초기 draft는 *코드 변경 0*을 약속했으나 v0.9.2 사용자 검증 회고 결과 v1.0 진입 직전 두 가지 *작은 코드 변경*을 흡수하기로 결정 (§1.3): ① `solosquad init` Step 1.5에 Claude Code 인증 흡수, ② repo 등록을 *path-reference로 단일화*(URL clone + Move-into-workspace 제거). 본 plan은 그 결정의 박제 + publish 절차 + post-launch 가이드.

**로드맵 위치:** v0.9.2 precheck hotfix → **v1.0 formal launch** → v1.1 대시보드 → v1.2 지식 온톨로지 → v1.3 일정·메모 → v1.x workflow/goal/routine 고도화 → v1.x LLM backend abstraction

**의존:**
- v0.8.1 `docs/api-stability.md` (정책 문서, *v1.0 발효 대기 상태*)
- v0.8.4 `docs/plan/v0.8.4-cli-surface-reduction.md` §11 (CLI surface freeze 체크리스트)
- v0.9.1 master-guide §4.2 Step 1 prerequisites + Model B (path-reference) 인프라
- v0.9.2 uninstall precheck hotfix (정직성 차단 false-positive 0)
- v0.x 시리즈 누적 dogfood 검증 (사용자 본인 — 30+일 자기 사용)

**기준 버전:** v1.0.0. 새 기능 0, 인프라 0. 출시 표면 freeze + 약속 발효 + 진입 흐름 정합 2 항목.

**분류:** Release activation — surface freeze + api-stability promise + onboarding 정합 코드 2건.

## 1. v1.0이 무엇인지 / 무엇이 아닌지

### 1.1 본 마일스톤
- **`api-stability.md`의 약속 발효** — 6개 `schema_version` 표면의 deprecation 정책이 v1.x.x bullet로 활성화
- **CLI surface freeze 발효** — `v0.8.4-cli-surface-reduction.md §11`의 42개 사용자 호출 가능 명령이 *SemVer 약속의 대상*
- **`workspace.yaml.version` 의 의미 변경** — v0.x.x = free-form bump, v1.x.x = breaking은 major 필수
- **release 라이프사이클의 시작** — v1.0 이후 새 기능은 *minor*, breaking은 *major*. v0.x식 자유 bump 종료

### 1.2 본 마일스톤이 아님
- 새 기능 0건. 사용자 노출 *명령 surface* (CLI 명령 42개 / `.solosquad/` 디렉터리 / 메신저 채널 / SKILL frontmatter / workspace.yaml) 는 v0.9.2와 동일
- 마이그레이션 schema 변경 0건. migration 0.9.2 → 1.0.0은 `workspace.yaml.version` bump only
- breaking change *데이터* 0건. v0.9.x 워크스페이스가 *그대로* v1.0.0 CLI에서 동작 (자동 마이그레이션 + 기존 데이터 손실 0). 단 *진입 흐름 UX 2건* 정리됨 (§1.3)
- **Codex 백엔드 도입 X** — `docs/plan/v1.x-llm-backend-abstraction.md`에 박제된 10가지 차단점은 *v1.x 슬롯*. v1.0은 Claude Code (OAuth) 단일 백엔드 유지
- v1.x 신규 plan(workflow·goal·routine 고도화 / 대시보드 / 지식 온톨로지 / 일정·메모)은 *별도 minor/patch 슬롯*. v1.0에서 진입하지 않음

### 1.3 v1.0이 흡수한 코드 변경 2건 (당초 draft의 "code change 0" 조정)

v0.9.2 사용자 dogfood에서 발견된 *진입 흐름 마찰* 2가지를 v0.9.3 patch 대신 **v1.0에 직접 흡수**:

| # | 변경 | 사유 | 영향 |
|---|---|---|---|
| 1 | `solosquad init` Step 1.5 — Claude Code 인증 흡수 | v0.9.x까지 "`solosquad init` + 별도 `claude login`" 2단계가 사용자 자가설치 시 마찰. wizard가 `claude auth status` 점검 → 미로그인 시 `claude login` spawn (inherit stdio) → OAuth 완료 대기. 이미 로그인된 사용자는 1초 스킵. | `src/cli/init.ts` 신규 함수 3개 (`ensureClaudeAuth` / `getClaudeAuthStatus` / `runClaudeLogin`) + Step 1 Environment Check 직후 호출 |
| 2 | repo 등록을 *path-reference 단일화* — URL clone + Move/Copy 제거 | v0.9.1 Model B (path-reference) 도입 후에도 URL clone + Move 분기가 *3-way UX*로 남아 *"내 repo가 어디 있지"* 인지 부담 지속. v1.0은 SoloSquad가 git clone semantics(auth·branch·depth·submodules·LFS)를 *책임지지 않음*. 사용자 자기 git toolchain으로 clone → 경로 등록. | `src/cli/init.ts:registerRepoInline` URL 거부 + non-git 거부 + path-ref-only path. `src/cli/add-repo.ts` 동일 — URL 거부 + 모든 입력이 `registerPathReference`로 funnel. `--keep-original` 은 no-op deprecation warning. |

**backward compat**:
- v0.9.x 워크스페이스 안에 이미 Model A로 들어와 있는 `<workspace>/<org>/repositories/<slug>/` 트리는 `resolveRepoCwd` legacy 분기로 *영구 동작* (코드 제거 X)
- v0.9.x에서 발급된 `<slug>.yaml` path-reference 등록은 그대로 동작
- 신규 등록만 path-ref-only — 기존 사용자 회귀 0건

**측정값**: 572 + 1 regression catcher (`test/v1.0-path-ref-only.test.ts`). 분기 제거로 `add-repo.ts` 39 lines 감소, `init.ts:registerRepoInline` 단순화 (123 → 96 lines).

## 4. Surface Freeze 박제

### 4.1 schema_version 6 면 (api-stability.md §1)

| Schema | 경로 | v1.0 freeze 값 |
|---|---|---|
| Workspace version | `<workspace>/.solosquad/workspace.yaml` → `version` | `1.0.0` |
| Org metadata | `<org>/.org.yaml` → `schema_version` | `1` |
| Org agent profile | `<org>/agent-profile.yaml` → `schema_version` | `1` |
| SKILL frontmatter | `assets/agents/<team>/<name>/SKILL.md` → `schema_version` | `1` (v0.8.1 backfilled) |
| Archive metadata | `archive.zip/archive.yaml` → `schema_version` | `1` |
| Archive manifest | `archive.zip/manifest.tsv` 첫 주석 줄 | `1` |

→ 각 surface 다음 bump 규칙: *minor*에서만 schema bump, 이전 schema는 **다음 minor까지 read-only**로 유지. v1.4에서 v1.2의 schema=1 drop 가능 (2-minor read window).

### 4.2 CLI surface — 42 명령 (v0.8.4 plan §11)

| 그룹 | 명령 | 수 |
|---|---|---|
| Top-level | `init`, `bot`, `schedule`, `status`, `update`, `doctor`, `run-routine`, `sync`, `migrate`, `uninstall`, `logs`, `import` | 12 |
| `readiness` / `add` / `analyze` / `pm` / `workflow` / `goal` / `memory` / `agent` / `messenger` / `archive` / `backup` | 11개 그룹 / 30개 subcommand | 30 |
| **합계** | | **42** |

→ 명령 추가 = minor / 명령 제거 또는 rename = major (v2.0+) / 플래그 추가 = minor / 플래그 제거 = major / default 변경 = major (security fix는 예외 + CHANGELOG 명시).

### 4.3 의도된 컨벤션 예외 (v1.0 freeze에 포함)

- `solosquad migrate`는 **dry-run이 default** (다른 destructive 명령은 `--dry-run`이 opt-in). v0.8.4 §9 의도된 비대칭 — *되돌리기 어려움이 default를 정당화*. v1.0 freeze에 포함 (정상화 X).

### 4.4 메신저 백엔드 — Discord 단일

v1.0 SemVer 약속의 대상이 되는 메신저는 **Discord 1종**입니다. `src/messenger/discord-adapter.ts` (어댑터 인터페이스 + 채널 페어 컨벤션 + author-guard) 가 v1.0 freeze에 포함. `src/messenger/slack-adapter.ts` 도 코드에 동봉되어 v0.9.x 사용자의 기존 운영을 계속 지원하지만 **v1.0 freeze에는 포함되지 않습니다** (§5.3 post-v1.0 슬롯 박제).

- README / README.kr.md tagline: "Discord bot" (단수)
- master-guide §5: §5.2 Discord 8-step 권장 + §5.1 Slack은 *post-v1.0* 배지
- workspace.yaml.messenger: discord 값만 v1.0 SemVer 약속. slack 값도 동작하나 회귀·약속 외

### 4.5 freeze에 포함되지 않는 영역

- **메신저 채널 layout** (`<org>/.org.yaml.messenger`) — schema_version 외 영역
- **routine 프롬프트 바디** (`assets/routines/*.md`) — content이지 schema 아님. 사용자 편집은 마이그레이션이 보존
- **SKILL.md 바디** — 프론트매터만 schema 대상

## 5. 박제 — v1.0이 약속하지 않는 것

### 5.1 LLM 백엔드 — Claude Code (OAuth) 단일 유지

`docs/plan/v1.x-llm-backend-abstraction.md`에 박제된 10가지 차단점은 *v1.x 슬롯*. v1.0은:
- README / README.kr.md — "Claude Code (OAuth) is the single backend"
- master-guide §4.2 — Claude Code 단일 백엔드 명시
- `workspace.yaml`에 `llm_backend` 필드 *도입 안 함*

→ Codex / 그 외 OAuth CLI 지원 약속 0건. v1.x 슬롯 진입 트리거 발생 시 별도 minor.

### 5.3 Slack 메신저 — post-v1.0 슬롯

v0.x 전체 기간 동안 SoloSquad는 Discord와 Slack 두 메신저를 사용자에게 제공했고, `src/messenger/slack-adapter.ts` 어댑터가 동작. 그러나 v1.0 SemVer 약속을 발효시키는 시점에 다음을 박제:

- **v1.0의 공식 지원 메신저는 Discord 1종**. README / README.kr / master-guide 모두 *Discord-first* 로 재정렬
- **Slack 어댑터 코드는 codebase에 그대로 유지** — v0.9.x에서 Slack을 운영 중이던 사용자는 *계속 동작*하지만 v1.0 회귀 약속 / SemVer 약속 대상이 아님
- master-guide §5.1 (Slack 9-step) 은 *post-v1.0 슬롯* 배지로 강등 + §5.2 Discord 권장
- FAQ "Slack 동시 사용 가능?" 질문은 *"v1.0 공식 지원은 Discord"* 로 재작성

**박제 사유** (v1.0 직전 사용자 검증 회고):
1. v0.9.1+에서 Slack `conversations.create`가 private 채널을 만들지만 사용자 invite를 자동 수행하지 않음 — 사용자가 admin 권한 또는 수동 API call로 self-invite 필요. UX 친화 X
2. Slack OAuth scope 요구가 6+ (channels:read·channels:manage·groups:read·groups:write·chat:write·app_mentions:read·channels:history) — Discord는 1개 토큰 + 기본 권한만으로 동작
3. Slack 워크스페이스 admin 권한이 없는 사용자의 진입 마찰 큼 — 개인 Slack 워크스페이스가 없으면 봇 install 자체 불가
4. v0.x 전체 dogfood가 Discord 중심으로 누적 — Slack 경로는 통합 테스트 커버리지 낮음
5. v0.9.2 직전 발견된 *Slack 채널 invite 누락* 같은 UX 갭은 v1.0 SemVer 약속 발효 시점에 *해소되지 않은 상태*

**post-v1.0 슬롯 재진입 트리거**:
- 사용자(본인) 또는 외부 사용자가 Slack 경로를 *주 메신저*로 운영하기 시작 → 그 시점 v1.x minor (예: `v1.x-slack-parity.md` 신설) 에서 invite 자동화·scope onboarding·channel UX 갭 보강 후 v1.x SemVer 약속 발효

→ Slack 어댑터 제거 / deprecation 시도 X. *코드 보존 + 약속 외 박제*가 v1.0의 선택.

### 5.2 v1.x 신규 기능 — 별도 minor/patch

`docs/plan/v1.x-workflow-goal-routine-evolution.md` (Q1~Q7 ideation 통합), `v1.1-dashboard-interaction.md`, `v1.2-knowledge-ontology.md`, `v1.3-schedule-memo.md` (예정) — *모두 v1.0 진입 후 별도 minor*. v1.0 자체는 어느 하나도 진입하지 않음.

### 5.3 PR API 자동화 (v0.8.6 §3 슬롯)

dev_capability가 *git push까지*가 v1.0 범위. PR 생성·리뷰·머지는 *사용자 GitHub 웹 UI*. v1.x autonomous engine 확장 슬롯 — `v0.8.6-migrate-hotfix-pr-workflow.md` §3 + `v1.x-workflow-goal-routine-evolution.md`.

## 9. 실패 모드 + rollback

| 시나리오 | 대응 |
|---|---|
| publish 후 즉시 회귀 발견 | npm unpublish 72h 윈도우 활용 → v1.0.1 hotfix bump (v0.9.0 burn 패턴 재사용) |
| 사용자 워크스페이스 마이그레이션 실패 | `~/.solosquad-backups/<workspace>-<ts>/` 자동 백업 → `solosquad rollback` 또는 직접 yaml 복원 |
| api-stability 약속 위반 발견 | 즉시 v1.0.1+에서 정합 패치 + CHANGELOG에 위반·복구 시점 박제 |
| dogfood gate §2 사후 미충족 발견 | v1.0 후속 v1.0.1 patch + 부족분 보강. v1.0 자체 unpublish는 *마지막 수단* |

→ v1.0 자체는 *코드 변경 0건*이라 회귀 가능성 매우 낮음. 위 시나리오는 *후속 minor/patch* 에서 발생하는 정상적인 라이프사이클.

## 10. 본 plan이 박제하는 약속

v1.0은 *솔로 파운더 자기 사용 검증 완료* + *공개 사용자 약속 시작* 의 분기점. 코드는 v0.9.2와 동일하지만 다음 약속이 **이 시점부터 SemVer로 보장**:

- 6개 schema_version surface의 deprecation 정책 (api-stability.md §2 v1.x.x bullet)
- 42개 CLI 명령의 SemVer 약속 (v0.8.4 §11)
- `workspace.yaml.version`이 SoloSquad CLI SemVer를 1:1 추적
- 의도된 컨벤션 예외 (migrate dry-run default 등)는 v1.0 freeze에 *포함*

→ v1.0 publish 시점 이후 새 약속 추가·기존 약속 변경은 *minor/major bump + CHANGELOG 박제* 형식으로만 가능. 본 plan은 *그 라이프사이클의 진입 게이트*.
