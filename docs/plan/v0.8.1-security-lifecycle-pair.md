# v0.8.1 — Security & Lifecycle Pair

> v0.7로 라이프사이클 *생성* 측은 닫혔지만, **복원·검증·보안 SLA**가 부족한 상태로 v1.0이 다가오고 있다. 이 패치는 (a) **npm audit 7건 해소** (urgent — security), (b) **`solosquad import <zip>`** (v0.7 archive와 짝을 이루는 복원 명령), (c) **`solosquad archive verify/info/list`** (archive 신뢰성 검증), (d) **API stability 문서** (v1.0 약속의 표면적 정의)를 한 릴리스에 모은다. 모두 v1.0 정식 출시 *전제* 항목.

**로드맵:** v0.8 Multi-User Messenger → **v0.8.1 Security & Lifecycle Pair** → v0.8.2 Dev Capability → v0.8.3 Onboarding UX + Observability → v0.9 안정화 → v1.0 정식 출시

**의존:** v0.7 (archive 포맷·manifest·schema_version 1 — import의 prerequisite), v0.8 (없음 — 메신저 모델과 독립)

**기준 버전:** v0.8.1. patch이지만 4건 모두 *완결성*에 관한 작업 — feature 추가 없이 기존 partial work를 닫는다.

**분류:** Security + Lifecycle pair + API contract.

## 3. `npm audit` 7건 해소

### 3.1 현재 상태
- undici 경유 `@discordjs/rest` 0.5.0~2.6.0 + `discord.js` 14.0~14.25
- 7 advisories: unbounded decompression chain, WebSocket parser crash, HTTP request smuggling, memory exhaustion, etc.
- `npm audit fix` 가능 여부 확인 + minor/major 영향도 PoC

### 3.2 해소 전략 (우선순위순)

| 우선순위 | 전략 | 적용 조건 |
|---|---|---|
| 1 | **discord.js 14.16 → 14.21+** (undici 패치 포함된 patch/minor) | 14.21 changelog 점검 후 breaking 0 확인되면 채택 — 가장 안전 |
| 2 | **`npm overrides`** — `package.json`에 `"overrides": { "undici": "^6.21.0" }` | 1번이 안 될 때 fallback. discord.js가 undici 6.21+를 받아들이는지 매트릭스 검증 필요 |
| 3 | **discord.js 14 → 15 메이저 업그레이드** | 1·2 모두 막힐 때만. v14 deprecation 일부 제거 — breaking change 검토 필수 |

### 3.3 CI 가드 추가
- `.github/workflows/ci.yml`에 `npm audit --audit-level=high` 단계 추가
- main 푸시 시 high CVE 발견되면 빌드 실패
- moderate 이상은 warn (issue 자동 생성 권고)

## 4. `solosquad import <zip>`

### 4.1 CLI 인터페이스

```bash
solosquad import <archive.zip> [--workspace <path>] [--into <org-slug>]
                               [--dry-run] [--merge | --replace]
```

| 플래그 | 동작 |
|---|---|
| `<archive.zip>` | 필수. v0.7 형식 archive (`archive_format: "zip-v1"`) |
| `--workspace <path>` | 신규 워크스페이스 위치. 미지정 시 archive `workspace_slug` 기반 새 폴더 |
| `--into <org-slug>` | 기존 워크스페이스의 특정 org에 머지. 미지정 시 archive의 모든 org를 워크스페이스 root에 풀음 |
| `--dry-run` | 디스크 변경 0. 무엇이 어디로 풀릴지 트리 + 충돌 분석 출력 |
| `--merge` (기본 박제) | 같은 slug org 존재 시 memory/workflows/goals 합침 (§4.3) |
| `--replace` | 같은 slug org 존재 시 덮어쓰기. 사용자 명시 확인 prompt 필수 |

### 4.2 흐름

```
1. archive.zip 무결성 검증
   ├─ manifest.tsv 추출 후 schema_version=1 확인
   ├─ 각 entry SHA256 재계산 → manifest와 대조 (mismatch 시 거부)
   └─ archive.yaml schema 호환성:
        min_solosquad_version <= 현재 CLI version
        archive_format == "zip-v1"
        max_schema_version_supported >= 1

2. (--dry-run면) 트리 + 충돌 분석 출력 후 종료

3. 적용 단계 (journal에 기록 — v0.7 idempotent 패턴 그대로):
   ├─ workspace 폴더 준비 (--workspace) 또는 기존 워크스페이스 확인 (--into)
   ├─ archive 풀기:
   │   workspace/         → <workspace>/
   │   orgs/<slug>/       → <workspace>/<slug>/
   │   credentials/env.template → <workspace>/.solosquad/.env.template (사용자가
   │                              값 채우도록 안내. 실제 .env로 자동 변환 안 함)
   │   manual-revoke-required/  → 안내 출력 후 폐기 (이미 사용된 archive)
   └─ verify:
        manifest SHA × 풀린 파일 SHA 1:1 일치
        workspace.yaml 로드 성공
        org yaml 로드 성공
        memory/archive.sqlite (있다면) integrity_check 통과

4. 사용자 코드 안내:
   ├─ `<repo>/.solosquad/repo.yaml`은 archive에서 복원되지만
   ├─ repo source code 자체는 archive에 미포함 (v0.7 클래스 A 정책)
   └─ 안내: "각 repo는 별도 git clone으로 복원 후 같은 자리에 두세요"
```

### 4.3 `--merge` 충돌 처리 (박제)

같은 slug org 존재 시:

| 자산 | 처리 |
|---|---|
| memory JSONL (signals/experiments/decisions/...) | content hash + ts 기준 dedup 후 append. 정렬 보존 |
| `memory/archive.sqlite` (v0.6 FTS5) | archive 측 rows를 *추가* id로 import. 같은 row id 충돌 시 id 재할당 + 매핑 테이블 기록 |
| `workflows/<wf-id>/` | id 충돌 시 거부 → 사용자가 archive 측 rename 또는 `--replace` |
| `goals/<goal-id>/` | id 충돌 시 거부 → 동일 |
| `<workspace>/AGENTS.md` | 본문 다르면 archive 측을 `AGENTS.imported.md`로 별도 저장. 사용자 수동 머지 안내 |
| `<workspace>/.solosquad/{core, knowledge}/` | 파일 단위 merge — 같은 파일 충돌 시 `.imported.md` 별도 저장 |
| `<org>/agent-profile.yaml` | yaml deep-merge. narrowing-only invariant 위반 시 거부 |

### 4.4 journal 통합

v0.7의 `<workspace>/.solosquad/uninstall.journal.jsonl`과 동일 패턴으로 import journal:
- 파일: `<workspace>/.solosquad/import.journal.jsonl`
- stage: `import.verify`, `import.unpack`, `import.merge-org`, `import.verify-post`
- 중단 시 재실행 idempotent (이미 풀린 파일은 SHA 확인 후 skip)

### 4.5 신규 모듈

| 파일 | 역할 |
|---|---|
| `src/cli/import.ts` | CLI 오케스트레이션 |
| `src/lifecycle/import.ts` | dry-run + 풀기 + merge/replace 흐름 |
| `src/lifecycle/archive-reader.ts` | yauzl 기반 zip 읽기 (archiver는 writer-only) |
| `src/lifecycle/merge-strategy.ts` | --merge 충돌 처리 정책 |

## 5. `solosquad archive verify/info/list`

### 5.1 CLI 인터페이스

```bash
solosquad archive verify <archive.zip>       # 무결성 + schema 호환
solosquad archive info <archive.zip>         # 메타 + entry 요약
solosquad archive list <archive.zip> [--class <X>]  # manifest 출력
```

### 5.2 verify

- manifest.tsv 추출 후 schema_version=1 확인
- 각 entry의 zip 내 실제 SHA256 재계산 → manifest SHA와 대조
- archive.yaml schema 호환성 확인
- 보고서:
  ```
  ✓ schema_version=1 호환
  ✓ 439 entries, 모두 SHA 일치
  ✓ archive_format=zip-v1
  ✓ min_solosquad_version=0.7.0 ≤ current CLI v0.8.1
  ```
- 실패 시:
  ```
  ✗ entry orgs/myorg/memory/signals.jsonl: SHA mismatch
      manifest: abc123...
      actual:   def456...
  ```

### 5.3 info

- archive.yaml 메타 + manifest 클래스별 요약
- 출력:
  ```
  export_ts:        2026-05-13T22:00:00+09:00
  solosquad_version: 0.7.0
  workspace_slug:    myworkspace
  archive_format:    zip-v1
  schema_version:    1
  included_orgs:     [myorg, secondary-org]
  
  Manifest summary:
  Class A*   3 entries, 1.2 KB
  Class B    412 entries, 320 MB
  Class C    20 entries, 8 KB
  Class D    1 entry (env.template, redacted)
  Total      436 entries, 320.0 MB
  ```

### 5.4 list

- manifest.tsv 전체 출력 (`--class B`로 필터)
- 대량 archive에서 특정 자산만 확인할 때

### 5.5 신규 모듈

| 파일 | 역할 |
|---|---|
| `src/cli/archive.ts` | CLI 명령 `archive verify/info/list` |
| `src/lifecycle/archive-reader.ts` | yauzl 기반 reader (§4와 공유) |

## 6. API Stability 정책 문서 (`docs/api-stability.md`)

### 6.1 6 schema_version 표

| Schema | 파일/필드 | 현재 | bump 룰 |
|---|---|---|---|
| `workspace.yaml.version` | SoloSquad 자체 version | 0.8.x | SemVer. 0.x는 free-form, 1.0부터 breaking은 major bump |
| `<org>/.org.yaml.schema_version` | org 메타 | 1 | 필드 *추가*는 minor·동일 schema. 필드 *제거/타입 변경*은 schema_version bump |
| `<org>/agent-profile.yaml.schema_version` | org 색채 modifier | 1 | 동일 룰. narrowing-only invariant는 schema와 무관 |
| `assets/agents/<>/SKILL.md` frontmatter `schema_version` | SKILL 정의 | (v0.8.1에서 신설 추가) | 동일 룰 |
| `archive.zip/archive.yaml.schema_version` | farewell archive | 1 | 동일 룰. `max_schema_version_supported` 사용해 import 호환 거부 |
| `archive.zip/manifest.tsv` 첫 줄 `schema_version=1` | manifest 포맷 | 1 | 동일 룰 |

### 6.2 Deprecation 기간

- **v1.0 이전 (현재 v0.8.x)**: schema_version bump 자유. CHANGELOG에 기재
- **v1.0 ~ v1.x minor**: schema_version bump 시 *이전 schema도 1 minor 더 지원*
  - 예: v1.2에서 schema=2 도입 → v1.3까지 schema=1도 읽음
  - v1.4부터 schema=1 read 코드 제거 가능
- **v2.0 major bump**: 옛 schema 영구 거부 가능

### 6.3 SKILL.md frontmatter `schema_version: 1` 신설

v0.8.1에서 25 번들 SKILL.md에 `schema_version: 1` frontmatter 추가:

```yaml
---
name: backend-developer
team: engineering
schema_version: 1                  # v0.8.1 신설
triggers:
  keyword: [...]
collab_pattern: hierarchical
---
```

`scripts/inject-skill-schema-version.ts` (신규) — idempotent 백필. CI 게이트 추가 (`solosquad agent validate --all --corpus` 가 schema_version 누락 시 fail).

### 6.4 v1.0 약속 발효

v1.0 정식 출시 시점에 `docs/api-stability.md`는 *공개 약속*이 된다. 이후 SemVer 룰 강제:
- minor bump (v1.1 → v1.2)에서 schema_version bump 허용, 단 deprecation 기간 1 minor
- major bump (v1.x → v2.0)에서 옛 schema 영구 거부 허용

## 8. 비파괴성 + 복구

- import는 항상 dry-run 권장. 실제 적용 시 journal 기록 → 중단 시 재실행 idempotent
- `--merge` 충돌은 거부 또는 별도 파일(`.imported.md`) — 사용자 자산 덮어쓰지 않음
- archive verify 실패 시 import 진입 거부 (안전 우선)
- npm audit 해소 PoC는 별도 브랜치에서 회귀 그린 확인 후 main 머지
