## 1. 핵심 회귀 버그: migrate.ts hardcoded version

### 1.1 증상

```
$ solosquad --version
0.8.5

$ solosquad doctor
  △ CLI v0.8.5 > workspace v0.4.0 — Run: solosquad migrate --apply

$ solosquad migrate --apply
✓ Workspace is already at 0.4.0. Nothing to migrate.

$ solosquad migrate --dry-run
✓ Workspace is already at 0.4.0. Nothing to migrate.
```

doctor가 *migrate 필요* 안내하는데 migrate는 *no-op*. 사용자는 워크스페이스 갱신 불가능. workaround로 `--to 0.8.6 --apply` 명시해야만 동작 (대다수 사용자가 모름).

### 1.2 원인

`src/cli/migrate.ts:8`:
```typescript
const CLI_VERSION_TARGET = "0.4.0";
```

`migrateCommand`가:
```typescript
const target = opts.to ?? CLI_VERSION_TARGET;   // → "0.4.0"
const result = await runMigration({ workspace, targetVersion: target, ... });
```

runner.ts:64-74:
```typescript
if (source === opts.targetVersion) {
  console.log("Workspace is already at ... Nothing to migrate.");
  return { success: true, chain: [], ... };
}
```

- 워크스페이스 v0.4.0 + target "0.4.0" → source === target → no-op
- 새 워크스페이스 v0.8.5 (v0.8.5 init이 박은 값) + target "0.4.0" → resolveChain이 *역방향* 마이그레이션 시도 → 에러

즉 **v0.4.0 이후 모든 minor/patch에서 migrate가 결정적으로 깨진 상태**였음. doctor는 mismatch 잘 감지(`recommendForVersionMismatch` 로직), 사용자가 마지못해 안내 따라가도 결과 *Nothing to migrate*. 회귀 1년 잔존.

### 1.3 수정

**v0.8.5와 동일 패턴 적용** — `src/util/version.ts`의 `SOLOSQUAD_VERSION` import.

```typescript
import { SOLOSQUAD_VERSION } from "../util/version.js";
const CLI_VERSION_TARGET = SOLOSQUAD_VERSION;
```

`SOLOSQUAD_VERSION`은 `package.json.version` 동적 참조 (v0.8.5에서 dev/prod 경로 모두 검증된 헬퍼). 이후 `npm publish` 시 자동 정합.

### 1.4 backward-compat

- 기존 v0.4.0~v0.8.5 워크스페이스: 이미 깨진 상태였으므로 *수정 후 정상 동작* — 즉시 chain 따라 v0.8.6까지 올라옴
- 새 init (v0.8.6 wizard): `SOLOSQUAD_VERSION = "0.8.6"` 그대로 박힘, migrate 즉시 no-op (의도된 상태)
- 회귀 risk: `SOLOSQUAD_VERSION`을 import하는 곳이 늘었지만 함수 시그니처 변화 없음

### 1.5 검증

- typecheck 통과 ✓
- 566/566 tests 그린 유지
- 실측: v0.4.0 워크스페이스에 v0.8.6 CLI로 `solosquad migrate --dry-run` → chain 9개 표시 (`0.4.0→0.5.0→...→0.8.6`)
- v0.8.5 워크스페이스에서 `solosquad migrate --apply` → version stamp만 0.8.6로 bump (no schema change)

### 1.6 회귀 방지 — *모든 stale 버전 상수* 일제 점검

grep 결과 (`grep -rn '"0\.[0-9]\.[0-9]"' src/cli/`):
- `src/cli/init.ts` ✓ (v0.8.5에서 fix)
- `src/cli/migrate.ts` ← **이번 패치**
- 그 외 src/cli 디렉터리에서 stale 버전 상수 0건 확인

향후 같은 함정 방지:
- 새 CLI 모듈에서 버전 default 필요하면 *반드시* `import { SOLOSQUAD_VERSION } from "../util/version.js"`
- 정적 분석 룰 추가 가능 슬롯 (v1.x) — `eslint-plugin-no-stale-version` 같은 자체 규칙

## 2. master-guide §10.4 + §10.5 박제

### 2.1 §10.4 Uninstall · 재설치 · 마이그레이션 회피

**문제 인식**: npm v7+의 `preuninstall` 훅이 글로벌 패키지에서 호출되지 않는 한계(`npm/cli#3042`) 때문에, 사용자가 *순서를 잘못 지키면* farewell archive 안 만들어진 채 `npm uninstall -g solosquad`가 바이너리만 제거 → REVOKE-CHECKLIST 못 받음 + .env 토큰 마스킹 archive 안 생김.

**박제 내용** (KO + EN 양쪽):
- 안전한 uninstall 6단계 (① 봇·스케줄러 정지 → ② dry-run → ③ uninstall 실행 → ④ archive zip 보관 → ⑤ REVOKE-CHECKLIST 외부 자원 정리 → ⑥ npm 제거)
- uninstall + reinstall로 **마이그레이션 chain 우회** 흐름 (큰 chain 건너뛰는 합법 우회로)
- 새 init 후 doctor 경고 7종 분류 표 (PowerShell / gh CLI / Docker / npm uninstall hook / version mismatch / live PIDs / stale uninstall.lock)
- "uninstall 직전 PID 정지 필수" warn callout (archive snapshot SHA 일치 위해)

### 2.2 §10.5 봇 · 스케줄러 · dev_capability 운영

**문제 인식 1 — 스케줄러가 디폴트 실행 안 됨**: SoloSquad는 데몬·자동 시작이 없는데, 사용자가 그걸 모르고 *"왜 routine이 안 돌아?"* 또는 반대로 *"왜 doctor에서 live PID 경고가 떠?"* 혼동 발생.

**문제 인식 2 — PR 워크플로 setup 미문서화**: v0.8.2 `dev_capability` 기능은 있지만, *에이전트가 실제로 PR 생성하려면 사용자가 사전에 뭘 셋업해야 하는지* master-guide에 명시 없음. gh CLI install / auth / repo permission / workspace.yaml 활성 4가지가 누락되면 spawn은 되는데 PR 단계에서 silent fail.

**박제 내용**:
- 스케줄러 비자동 실행 명시 + Windows `Get-CimInstance` 매칭 로직 공개
- PR 워크플로 전제 조건 3건 (gh CLI + auth / repo permission / workspace.yaml dev_capability)
- 단일 에이전트 PR 생성 흐름 (사용자 메시지 → PM 분류 → BD spawn → dev-confirm gate → push → gh pr create)
- **다중 에이전트 PR 리뷰·토론 현재 상태 표** (v0.8.5 부분 지원 / v1.x 슬롯 명시)
- 온보딩 추가 항목 5건 (Step 3.5 gh CLI / Step 7.5 repo 등록·권한 / Step 7.7 dev_capability 활성 / Step 8.5 메신저 dry test / Step 8.7 branch protection)
- v1.x followup callout — 완전 자동 다중-에이전트 PR 토론은 autonomous engine 확장 슬롯

## 3. v1.x 슬롯: 자동 다중-에이전트 PR 토론 → 머지 (설계 박제)

### 3.1 현재 상태 (v0.8.5/v0.8.6)

| 흐름 단계 | 자동화 수준 | 한계 |
|---|---|---|
| 에이전트 1명이 PR 생성 | ✓ 완전 자동 | dev_capability:pr + 30분 dev-confirm gate |
| 에이전트가 자기 *브랜치* 명명 | △ ad-hoc | SKILL이 자유롭게 `git checkout -b ...` 결정. 워크플로/goal과 무관한 명명 |
| 다른 에이전트가 PR 리뷰 | △ 사용자 명시 트리거 | "QA로 #42 리뷰해줘" 필요 |
| PR 토론 round-trip | ✗ 미지원 | PM이 한 번에 다 spawn 못함, 사람이 spawn 매개 |
| 에이전트가 머지 결정 | ✗ 영구 거부 | 자동 머지 거부 룰 (v0.8.2 박제) |

### 3.2 v1.x 설계 — branch_per_workflow

#### workflow.yaml 스키마 확장 (제안)

```yaml
# <org>/workflows/wf-pmf-stripe/workflow.yaml
schema_version: 2
id: wf-pmf-stripe
title: "Stripe payment PMF discovery"

git_workflow:                                      # v1.x 신규 섹션
  enabled: true
  base_branch: main
  branch_pattern: "feat/wf-{id}"                   # → feat/wf-pmf-stripe
  auto_pr: true                                    # 마지막 stage 종료 시 자동 PR 생성
  pr_title_pattern: "{wf.title} ({wf.id})"
  pr_body_template: "templates/pr-body-wf.md"

reviewers:                                         # v1.x 신규 — 자동 리뷰 spawn
  - agent: engineering/qa-engineer
    focus: "regression risk, test coverage"
    timing: after-stage-build
  - agent: engineering/security-engineer
    focus: "credential handling, OWASP top 10"
    timing: after-stage-build
  - agent: strategy/business-strategist
    focus: "scope creep, alignment with PRD"
    timing: after-stage-research

discussion_rounds: 2                               # 최대 round-trip 횟수 (사람 매개 X)
auto_merge: false                                  # 영구 false (박제)
```

#### SKILL 프론트매터 확장 (제안)

```yaml
# 25 specialist 중 일부에 추가
---
name: "QA Engineer"
dev_capability: pr
can_review_pr: true                                # v1.x 신규 — gh pr review 권한
review_focus: ["regression", "test coverage"]      # 자동 리뷰 spawn 시 inject
review_comment_template: "templates/review-qa.md"  # 표준 코멘트 형식
---
```

#### 자동 흐름 (v1.x)

```
사용자: "Stripe 통합 PMF discovery 시작"
   ↓
PM: spawn workflow wf-pmf-stripe → 자동 git checkout -b feat/wf-pmf-stripe
   ↓
[Stage 1 — research]  desk-researcher spawn → commit "research: market scan"
   ↓
[Stage 2 — planning]  pmf-planner spawn → commit "plan: validation hypothesis"
   ↓
[Stage 3 — design]    ui-designer spawn → commit "design: payment flow UI"
   ↓
[Stage 4 — build]     backend-developer spawn → commit "build: Stripe integration"
                      → workflow.yaml.git_workflow.auto_pr=true →
                      → dev-confirm gate (사용자 y) → push + gh pr create
   ↓
PR #42 생성됨 (works-handle에 URL 회신)
   ↓
[Auto-review round 1] (workflow.yaml.reviewers 자동 spawn, 사람 트리거 불필요)
   ├─ qa-engineer:        gh pr review 42 --comment --body "..."  (review_focus 기반)
   ├─ security-engineer:  gh pr review 42 --comment --body "..."
   └─ business-strategist: gh pr review 42 --comment --body "..."
   ↓
[Discussion aggregation] PM이 3개 리뷰 코멘트 → works-handle에 요약 post
   ├─ "QA: signup_cvr 측정 누락 — Stage 4 보강 권고"
   ├─ "Security: STRIPE_SECRET_KEY .env에 있는지 확인 (✓ 확인됨)"
   └─ "Strategy: PRD §2 PMF 가설과 정합 — pass"
   ↓
[Auto-review round 2] (사용자 명시 또는 reviewer 응답 필요)
   사용자: "QA 의견 반영해서 backend-developer로 commit 추가"
   ↓
backend-developer spawn → commit "test: cvr smoke harness" → push
   ↓
[Discussion round 2] QA Engineer spawn (gh pr view로 새 commit 확인) → approve
   ↓
PM이 "리뷰 통과. merge할까요?" → 사용자 y → dev-confirm gate
   ↓
gh pr merge 42 --squash → <org>/memory/pr-discussions.jsonl에 round-trip 박제
```

## 4. 에이전트가 기획 → 브랜치 → 작업 → PR → 토론 → 머지하는 방법 (v0.8.5/v0.8.6 현재 가능 vs v1.x 자동)

### 4.1 현재 가능 — 사용자가 spawn 매개 (반자동)

**전제**:
- gh CLI installed + `gh auth login`
- repo 등록 + Write 권한
- workspace.yaml `dev_capability.enabled: true`
- SKILL에 `dev_capability: pr`

**흐름**:
```
1. 기획 (사용자 또는 PM 슬래시):
   "/plan Stripe 결제 통합" → PM이 PRD.md + _status.yaml 생성
   (이 시점에 워크스페이스 git 브랜치 자동 안 만듦 — v0.8.6에선 사용자가 결정)

2. 브랜치 (사용자 명시 또는 첫 specialist가):
   사용자: "feat/stripe-integration 브랜치에서 작업 시작"
   PM → backend-developer spawn (target_repo: my-saas, working_branch: feat/stripe-integration)
   BD: git checkout -b feat/stripe-integration && [코드 작성] && commit

3. 작업 완료 → PR:
   BD: push  → dev-confirm gate → 사용자 y → gh pr create
   "PR #42 https://..." → works-handle 회신

4. 다른 에이전트 리뷰 (사용자가 명시 spawn):
   사용자: "QA Engineer로 PR #42 리뷰해줘"
   PM → qa-engineer spawn → gh pr view 42 → 분석 → gh pr review 42 --comment

5. 토론 round-trip (사용자가 명시):
   사용자: "BD가 QA 의견 반영해줘"
   PM → backend-developer spawn → 새 commit → push → PR 업데이트
   사용자: "QA 다시 리뷰" → qa-engineer 재spawn → approve

6. 머지:
   사용자: "PR #42 머지" → engineering agent spawn → gh pr merge 42 → dev-confirm gate → 사용자 y
   _status.yaml 완료 마킹 + workflow archive
```

각 단계 *사용자가 spawn 명시 트리거*. 반자동.

### 4.2 v1.x 자동 — workflow.yaml `git_workflow` + `reviewers` 정합

§3.2 설계 따라 *사용자 한 마디*("Stripe 통합 PMF discovery 시작") → 전체 흐름 자동:
- 브랜치 자동 생성 (workflow.id 기반 명명)
- 단계별 commit 자동 (각 stage가 같은 branch에 push)
- 마지막 stage 종료 시 `auto_pr` 트리거 → dev-confirm → PR
- `reviewers` 리스트의 에이전트 자동 spawn (round 1)
- PM이 리뷰 결과 aggregation → works-handle post
- 사용자가 *결정 지점* (discussion round 2 트리거 / merge 승인)에서만 개입

여전히 **사람 게이트 영구 보존**:
- `git push` (dev-confirm)
- `gh pr merge` (dev-confirm + workflow.yaml `auto_merge: false` 영구)
- 자동 round-trip 횟수 cap (`discussion_rounds`, 기본 2)

### 4.3 본 plan에서 무엇이 *지금 들어가나*

**문서만** — `master-guide.html` §10.5에 위 흐름 두 종류(v0.8.6 현재 vs v1.x 자동) 박제 + 온보딩 단계 5건 명시.

**코드는 v1.x 슬롯**.
