# v0.8 — Team Scale & Dev Capability

> v0.7로 라이프사이클이 닫힌 후, v1.0 정식 출시 직전 마지막 큰 슬롯. 세 가지 축으로 SoloSquad를 "공개 사용자 약속 가능한 도구"로 끌어올린다: (a) **멀티 유저 메신저 채널** — 같은 Discord 서버·Slack 워크스페이스에 N명 팀원이 각자 설치, (b) **개발 능력 완성** — 에이전트가 코드 수정 + commit/push/PR/merge 까지 end-to-end, (c) **기존 리포 마이그레이션 UX** — 사용자가 자기 코드를 SoloSquad 안으로 들여올 때의 명확한 단계 + 폴더 이동 안내. 동시에 v0.7 검토에서 P0/P1으로 식별된 carryover(import 페어, npm audit, archive verify, master-guide 재정합, logger, trajectory 게이트 결정)도 흡수.

**로드맵:** v0.1 크로스 플랫폼 → v0.2 안전/보안 → v0.3 PM 모드 → v0.4 자율 실행 → v0.5 워크플로우 메이커 → v0.6 디폴트 워크플로우 + 메모리 아카이브 → v0.7 Uninstall & Lifecycle → **v0.8 Team Scale & Dev Capability** → v0.9 (안정화 슬롯) → **v1.0 정식 출시** → v1.1 대시보드 → v1.2 지식 온톨로지

**의존:** v0.3 (PM 세션 — 사용자 식별의 진입점), v0.4 (`goal-runner` modifiable_paths 화이트리스트 — dev 능력의 가드), v0.5 (4채널 트리거 — 채널별 라우팅 재사용), v0.6 (Org Layer · 8-layer JIT — 사용자별 컨텍스트 격리), v0.7 (archive 포맷·lockfile·journal — import 페어의 prerequisite)

**기준 버전:** v0.8.0. v0.9는 안정화/i18n/자체 사용 검증 슬롯으로 따로 잡고, v0.8은 **새 기능 추가 마지막 릴리스**.

**분류:** Capability + UX 릴리스. v0.7이 라이프사이클 *닫기*였다면 v0.8은 라이프사이클 *확장 + 다듬기*. 새 기능 3개 + carryover 6개.

| 원칙 | 구현 |
|---|---|
| **메신저-네이티브 격리** | 사용자별 채널 페어는 메신저 ACL(Discord 채널 private + 초대, Slack private channel)을 *주된* 권한 메커니즘으로. SoloSquad 코드는 *defense in depth*로 author 검증만 추가. 권한 시스템을 재발명하지 않음 |

## 3. 멀티 유저 메신저 채널 모델

### 3.1 문제 정의

현재 v0.7까지의 가정:

> "1 워크스페이스 = 1 owner = 1 메신저 봇 = 2 채널 (`owner-command` + `workflow`)"

v0.8 가정:

> "1 메신저 서버/워크스페이스에 N명의 owner가 각자 자기 머신에서 설치. 각 owner는 자기 채널 페어를 가진다. 각자의 메모리·워크플로·goal은 자기 머신의 워크스페이스에 격리되어 있다 (물리적 분리)."

### 3.2 채널 명명 규칙

| 채널 | 이름 | 가시성 | 누가 명령 가능 |
|---|---|---|---|
| 개인 명령 채널 | `command-<handle>` | private (초대만) | 채널 owner(`<handle>`) 만. 다른 메시지는 봇이 ignore + ephemeral DM "이 채널은 <handle>의 명령 채널입니다" |
| 개인 작업 채널 | `works-<handle>` | private (초대만) | 봇이 그 owner의 PM 세션 결과 + goal-runner 출력 + 워크플로 핸드오프를 푸시. 사용자도 메모 가능 |
| (선택) 공유 브로드캐스트 | `solosquad-broadcast` | public 또는 팀 visible | 봇이 morning/evening brief, weekly review, signal-scan 결과를 푸시. 1 워크스페이스당 1개. 사용자 명령은 받지 않음 |

`<handle>` 정의:
- **Discord**: 글로벌 unique handle (2023 이후 `username` 필드 — display name 아님)
- **Slack**: 워크스페이스 내 unique handle (`users.info → name`)
- 영문 소문자·숫자·`_`만 허용. 다른 문자는 `_`로 치환 후 사용자 확인 prompt

명령 1예:
```
Alice의 채널 (alice = Discord handle "alice"):
  #command-alice    ← alice만 명령 가능
  #works-alice      ← alice의 PM 세션 결과 + 작업 트리

Bob의 채널 (bob = Discord handle "bob"):
  #command-bob      ← bob만 명령 가능
  #works-bob        ← bob의 PM 세션 결과 + 작업 트리

(선택) 공유:
  #solosquad-broadcast   ← 8:00 morning brief, 18:00 evening brief 등 자동 푸시
```

### 3.3 봇 multiplicity 정책

**원칙**: 사용자 1명 = 봇 application 1개 = 봇 프로세스 1개.

같은 Discord 서버에 2명 설치:
- alice의 머신: 자기 .env에 alice의 봇 token. `solosquad bot`은 alice의 봇 process로 시작
- bob의 머신: 자기 .env에 bob의 봇 token. `solosquad bot`은 bob의 봇 process로 시작
- 두 봇은 *같은* 서버에 멤버로 들어있지만 *다른 user ID*. 각자 자기 채널 페어만 listen
- 채널 ↔ 봇 매핑: `<org>/.solosquad/users/<handle>.yaml` 의 `bot_user_id` 필드로 박제

**거부되는 시나리오**:
- 한 봇 token을 두 머신이 동시에 사용 → Discord/Slack에서 disconnect 또는 rate-limit. 명시적으로 README에 "1 token = 1 머신" 안내

### 3.4 사용자 식별 흐름

`solosquad init` 단계 (v0.8 변경):

```
6. 메신저 토큰 입력
   ↓
6a. 봇 토큰으로 messenger API 호출:
      Discord: GET /users/@me  → username
      Slack:   auth.test → users.info → name
   ↓
6b. 추출된 handle을 사용자에게 확인:
      "이 봇 token의 messenger handle은 'alice'로 감지됐어요. 채널은
       'command-alice' + 'works-alice'로 생성됩니다. 변경하려면 입력하세요
       [기본: alice]:"
   ↓
6c. 사용자가 입력한 handle을 `<org>/.solosquad/users/<handle>.yaml`에 저장:
      handle: alice
      display_name: "Alice (Founder)"
      messenger: discord
      bot_application_id: 1234567890
      bot_user_id: 9876543210
      joined_at: "2026-05-15T10:00:00Z"
      workspace_path: "/Users/alice/solosquad"
      session_id: "<pm-session-id>"
      channels:
        command: "command-alice"
        works: "works-alice"
   ↓
7. 봇 startup 시 사용자 yaml 로드 → 자기 채널 페어 자동 생성 (없으면)
```

`<org>/.solosquad/users/` 디렉토리는 v0.6 §2.2 8-layer JIT inject의 Layer 5 (org-level metadata)로 spawn-time에 자동 주입.

### 3.5 author 검증 (defense in depth)

메신저 ACL이 1차 방어선이지만, 잘못 초대된 사용자가 다른 사람 채널에 글을 쓰는 사고를 막기 위해:

```typescript
// src/bot/author-guard.ts (신규)
function isAuthorizedAuthor(channelName: string, authorHandle: string): boolean {
  // command-<handle> 또는 works-<handle> 채널이면 owner만 명령 가능
  const match = channelName.match(/^(command|works)-(.+)$/);
  if (!match) return true;  // 다른 채널은 통과 (broadcast 등)
  const channelOwner = match[2];
  return channelOwner === authorHandle;
}

// adapter에서 메시지 수신 시:
if (!isAuthorizedAuthor(message.channelName, message.author.username)) {
  // 1) 채널엔 응답 안 함, 2) 메시지 author에게 ephemeral DM 1회
  await sendEphemeral(message.author, `이 채널은 ${channelOwner}의 명령 전용입니다. command-${authorHandle} 채널을 사용하세요.`);
  return;
}
```

### 3.6 채널 생성 / 발견 / 충돌 (결정 박제)

봇 startup 시 (`src/bot/channel-bootstrap.ts` 신규):

1. `<org>/.solosquad/users/*.yaml` 전부 로드
2. 자기 `bot_user_id`와 매칭되는 yaml 1건 식별 (없으면 init 안내 후 종료)
3. 해당 yaml의 `channels.command` + `channels.works` 가 messenger에 존재하는지 확인
4. 부재 시 생성 (private + 자기 봇 + owner 사용자만 초대)
5. broadcast 채널 처리 — §3.7
6. 다른 user의 채널 페어(`command-bob` 등)는 *listen 안 함, 무시*

**Handle 충돌 정책 (박제 — 거부)**:
- `<org>/.solosquad/users/<handle>.yaml` 이 이미 존재할 때 같은 handle로 init 시도 → **명시적 거부**
- 에러 메시지: `"<handle>은 이미 이 워크스페이스에 등록되어 있습니다. 다른 messenger handle 또는 별도 워크스페이스를 사용하세요."`
- 자동 `-2` suffix 같은 silent disambiguation 안 함 — 사용자 인지 부담은 explicit 거부로 옮김

**Slack 무료 워크스페이스 채널 cap**:
- Slack free tier: 10 channels per workspace
- N명 팀에서 2N+1 채널이면 N=5에서 cap 도달
- v0.8 README에서 "Slack 무료는 N≤4 권장, 그 이상은 Slack Pro 또는 Discord" 명시
- 대안: Slack thread-only mode (1개 채널 + 사용자별 thread). v1.0 슬롯

### 3.7 broadcast 채널 정책 (결정 박제 — 옵션 B)

`#solosquad-broadcast` 처리:

| 항목 | 결정 |
|---|---|
| 기본값 | **off** (`workspace.yaml.messenger.broadcast_enabled: false`) |
| Opt-in 시 채널 생성 | 첫 번째 install 시 1회 생성. workspace.yaml에 `broadcast_channel: "solosquad-broadcast"` stamp |
| 발송 권한 | **designated 봇 1개만** — first-installed 봇(workspace.yaml `broadcast_owner_handle: <first-handle>`로 박제). 다른 봇은 자기 `works-<handle>`로만 brief 푸시 |
| 발송 내용 | morning/evening brief, weekly review, signal-scan 결과. designated 사용자 워크스페이스 데이터 기반 |
| 명시 안내 | broadcast 활성화 시 README/master-guide에 "공유 brief는 <handle> 워크스페이스 데이터 시점" 박스로 표시 |

설계 정합:
- 봇 multiplicity(§3.3)와 정합 — 모든 봇이 broadcast 푸시하면 N건 중복

- 팀 visibility — opt-in으로만 활성. solo founder가 알아서 켬

## 4. 개발 능력 확장 — Engineering 에이전트 end-to-end

### 4.1 목표

사용자가 메신저에 1줄 입력:
```
@bot product-x에 다크모드 토글 추가해줘. PR까지 만들어줘.
```

→ engineering 팀 specialist 1명(Backend Developer / FDE 등)이 spawn → repo cwd로 이동 → 코드 분석 → 수정 → 테스트 → commit → push → `gh pr create` → PR URL을 사용자에게 회신. **머지는 사용자 confirmation 후만 실행**.

### 4.2 권한 정책 (결정 박제)

SKILL.md frontmatter에 신규 필드:

```yaml
---
name: backend-developer
team: engineering
dev_capability: true              # v0.8 신규
dev_permissions:
  bash:
    allowed: ["git", "gh", "npm", "node", "tsc", "pytest"]
    denied: ["rm -rf /", "sudo", "chmod 777"]
  network: false                  # MCP 서버 외 outbound 차단
  push_targets:
    requires_confirmation: true   # PR push 직전 사용자 확인 강제
  merge:
    auto: false                   # 자동 머지 영구 거부
---
```

**SKILL 기본값 박제** — 25 번들 SKILL.md 중 다음 5건만 `dev_capability: true`:

| 팀 | 에이전트 | dev_capability | 사유 |
|---|---|---|---|
| engineering | Backend Developer | `true` | 핵심 coding actor |
| engineering | FDE (Forward Deployed Engineer) | `true` | end-to-end 구현 |
| engineering | API Developer | `true` | endpoint 추가/수정 + PR |
| engineering | Creative Frontend | `true` | UI 컴포넌트 + PR |
| engineering | QA Engineer | `true` | 테스트 작성 + commit |
| engineering | Architect | `false` | advice + review, 코드 안 만짐 |
| engineering | Cloud Admin | `false` | infra advice, 코드 안 만짐 |
| engineering | Data Engineer | `false` | 데이터 design advice |
| engineering | Data Collector | `false` | research/spec, 코드 안 만짐 |
| engineering | Security Engineer | `false` | review-only |
| 그 외 전 SKILL (strategy/growth/experience 15건 + meta) | `false` | non-coding |

**Workspace 마스터 토글** — `workspace.yaml.dev_capability.enabled` (기본 `true`):
- `false` 설정 시 위 5개 SKILL의 `dev_capability: true`도 *비활성화* (워크스페이스 전체에서 dev 액션 0건)
- 솔로 사용자나 dev 자동화 거부 환경(예: client confidential repo)에서 일괄 끌 때 사용
- 활성 토글이 `true`라도 개별 SKILL의 `dev_capability: false`는 그대로 false (narrowing-only 원칙, v0.6 agent-profile.yaml과 정합)

spawn-assembler에서 인식:

```typescript
// src/bot/spawn-assembler.ts 확장
function applyDevPermissions(spawnConfig: SpawnConfig, skill: ParsedSkill): SpawnConfig {
  if (!skill.frontmatter.dev_capability) {
    spawnConfig.allowedTools = ["Read", "Grep", "Glob"];  // read-only
    return spawnConfig;
  }
  const perms = skill.frontmatter.dev_permissions;
  spawnConfig.allowedTools = ["Read", "Grep", "Glob", "Edit", "Write", "Bash"];
  spawnConfig.bashAllowlist = perms.bash.allowed;
  spawnConfig.bashDenylist = perms.bash.denied;
  spawnConfig.requirePushConfirmation = perms.push_targets.requires_confirmation;
  return spawnConfig;
}
```

### 4.3 PR 흐름

PM 세션이 dev 작업을 위임할 때 spawn 프롬프트 템플릿 (`assets/orchestrator/SKILL.md` 확장):

```markdown
## Engineering Spawn Template (v0.8)

작업 단계를 다음 순서로 수행:

1. **Branch 생성**: `git checkout -b feat/<short-slug>` (main에서 분기)
2. **분석 + 수정**: 사용자 의도를 코드로 변환. Edit/Write 도구 사용
3. **테스트**: 영향받은 영역의 테스트 실행. 실패 시 수정·재시도
4. **커밋**: `git add <specific files>` → `git commit -m "<imperative summary>"`. **`-A` 금지, hooks skip 금지**
5. **푸시**: `git push origin feat/<short-slug>` — 이 단계 *직전*에 PM에게 "push 진행?" 보고
6. **PR 생성**: `gh pr create --title <60자 이내> --body <HEREDOC>` — base는 main
7. **회신**: PR URL을 사용자에게 1줄로 회신. *머지 시도 금지*

머지는 사용자가 별도 명령 `@bot PR-<id> 머지해`로 요청해야 PM이 다시 위임한다.
이 단계도 사용자 confirmation prompt 통과 후만 실행.
```

### 4.4 사용자 confirmation gate

`src/bot/dev-confirm.ts` 신규:

- spawn 중 `Bash` 도구로 `git push`·`gh pr merge`·`gh pr close` 명령이 시도되면:
  1. 명령 실행 *전에* PM session으로 confirmation request 이벤트 발행
  2. PM이 사용자에게 메신저로 "다음 명령을 실행할까요? `git push origin feat/dark-mode` [y/N]"
  3. 사용자 응답 받기 전까지 Bash 도구 blocking
  4. 응답 `y` → 실행, 그 외 → "취소됨" 반환
- 30분 timeout: 자동 취소
- 모든 confirmation 이벤트는 `<org>/memory/dev-confirmations.jsonl`에 append

### 4.5 gh CLI 인증

전제: 사용자가 머신에 `gh auth login` 1회 완료. SoloSquad는 자기 인증 안 함.

`solosquad doctor` v0.8 점검 추가:
- `gh --version` 존재 확인
- `gh auth status` 통과 확인
- 통과 시 "engineering 에이전트가 PR 생성 가능" 표시
- 실패 시 "gh auth login 권고" + dev_capability 에이전트는 PR 단계 건너뜀 (코드 변경 + commit만)

### 4.6 회귀

- engineering 5 SKILL.md에 `dev_capability: true` + `dev_permissions` 추가
- 단위 테스트: spawn-assembler가 `dev_capability: false`인 SKILL에 Bash 도구 거부
- E2E 테스트 (모의 repo): "타이포 수정 + commit + push + PR 생성"이 confirmation gate 통과 후 PR URL 반환

## 5. 기존 리포 마이그레이션 가이드 (사용자 시점 상세)

### 5.1 시나리오

사용자 alice가 이미 product code를 `~/projects/my-saas/`에서 작업하고 있다. SoloSquad를 시작하면서 이 리포를 SoloSquad 워크스페이스 안으로 들여오고 싶다.

### 5.2 5단계 흐름 (권장)

```
[현재 상태]
~/projects/my-saas/                    # alice의 기존 리포
  ├── .git/
  ├── src/
  ├── package.json
  └── ...

[Step 1] 기존 작업 정리 + commit + push
  cd ~/projects/my-saas
  git status                           # untracked·unstaged 0건 확인
  git push origin main                 # 안전망

[Step 2] SoloSquad 초기화
  mkdir ~/solosquad && cd ~/solosquad
  solosquad init                       # 워크스페이스 + 첫 org 생성 (예: org slug = "main")

[Step 3] dry-run으로 이동 미리보기 (v0.8 신규 플래그)
  solosquad add repo ~/projects/my-saas --dry-run
  → 출력:
     "From: /Users/alice/projects/my-saas
      To:   /Users/alice/solosquad/main/repositories/my-saas
      Files: 1,247 (32.5 MB)  .git: 8.1 MB
      Added: <repo>/.solosquad/repo.yaml (1 file)
      Untouched: 1,246 user files
      IDE workspace files detected: .vscode/settings.json (paths preserved)
      Symlinks pointing INTO this repo from elsewhere: 0 found"

[Step 4] 실제 이동
  solosquad add repo ~/projects/my-saas
  → 결과:
     "Moved → /Users/alice/solosquad/main/repositories/my-saas
      Registered in .org.yaml
      .git history preserved (size unchanged)"

[Step 5] 검증 + 정리
  solosquad sync                       # repositories/ ↔ .org.yaml 정합 확인
  cd ~/solosquad/main/repositories/my-saas
  git status                           # working tree clean
  git log -1                           # 직전 commit 그대로
```

[이동 후 상태]
```
~/solosquad/                            # SoloSquad 워크스페이스
  ├── .solosquad/                       # config·workspace.yaml·.env
  ├── AGENTS.md                         # cross-tool 영속 가이드
  └── main/                             # org (= "main" slug)
      ├── .org.yaml
      ├── memory/
      ├── workflows/
      ├── goals/
      └── repositories/
          └── my-saas/                  # 이동된 리포
              ├── .git/                 # 그대로
              ├── src/                  # 그대로
              ├── package.json          # 그대로
              ├── .vscode/              # 그대로
              └── .solosquad/           # 신규 — 단일 메타 파일
                  └── repo.yaml         # slug, name, role, registered_at
```

### 5.3 폴더 매핑 표

| 사용자 자산 | 이동 전 위치 | 이동 후 위치 | 변경 |
|---|---|---|---|
| 메인 product code 리포 | `~/projects/<name>/` | `~/solosquad/<org>/repositories/<name>/` | 디렉토리 이동 |
| `.git/` 히스토리 | `~/projects/<name>/.git/` | `~/solosquad/<org>/repositories/<name>/.git/` | 그대로 (`mv` 의 일부) |
| IDE workspace 파일 | `~/projects/<name>/.vscode/` 등 | `~/solosquad/<org>/repositories/<name>/.vscode/` | 그대로. 상대경로 설정이면 그대로 작동. **절대경로 설정이면 사용자가 수동 갱신** |
| `.env.local` 등 secret | `~/projects/<name>/.env*` | `~/solosquad/<org>/repositories/<name>/.env*` | 그대로 (user-authored 파일은 절대 SoloSquad가 변형하지 않음) |
| SoloSquad 메타 (신규) | 없음 | `~/solosquad/<org>/repositories/<name>/.solosquad/repo.yaml` | 신규 1개 파일. 다른 SoloSquad 파일은 repo 안에 절대 안 만듦 (v0.7 클래스 A* 룰) |

### 5.4 위험 시나리오 + 회피

| 위험 | 회피 |
|---|---|
| 이동 도중 IDE/dev server가 파일을 열고 있어 lock | `solosquad add repo` 전에 IDE 닫기·dev server 정지. dry-run 출력에 활성 프로세스 감지 시 경고 추가 (v0.8 §10 #6) |

## 7. v0.7 carryovers (P0/P1 흡수)

### 7.1 `solosquad import <zip>` (P0 #1)

v0.7 §15 v1.0 슬롯이었지만 v0.8로 당김. 이유: import 없는 archive는 절반의 라이프사이클이고, v1.0 *직전*보다 v0.9 안정화 슬롯에 충분히 시험해야 안전.

**CLI 인터페이스** (결정 박제 — `--merge` 기본):
```bash
solosquad import <archive.zip> [--workspace <path>] [--into <org-slug>] [--dry-run]
                               [--merge | --replace]
```

**흐름**:
1. archive.zip 무결성 검증 (manifest SHA × 실제 SHA)
2. archive.yaml schema 호환성 (`min_solosquad_version <= 현재 CLI version <= max_schema_version_supported`)
3. dry-run: 어떤 파일이 어디로 풀릴지 트리 출력. 디스크 변경 0
4. 적용:
   - `--workspace <path>` 지정 시 그 경로에 새 워크스페이스 구성. 없으면 archive의 workspace_slug로 새 폴더
   - `--into <org-slug>` 지정 시 기존 워크스페이스의 해당 org에 추가 (충돌 시 거부)
   - **`--merge` (기본 박제)**: 같은 slug org 존재 시 memory/workflows/goals를 *합침*
     - JSONL 파일은 `ts` + content hash 기준으로 중복 행 제거 후 append (정렬 보존)
     - `archive.sqlite` (v0.6 FTS5)는 source 우선 — 같은 row id 충돌 시 archive의 것을 *추가* row로 (id 재할당)
     - `workflows/<wf-id>/`·`goals/<goal-id>/` 폴더는 동일 id 존재 시 거부 (사용자가 archive 측 id rename 또는 `--replace` 사용)
     - `<workspace>/AGENTS.md` 충돌 시 archive 측을 `AGENTS.imported.md`로 별도 저장 + 사용자에게 수동 머지 안내
   - `--replace`: 같은 slug org 존재 시 *덮어쓰기*. 사용자 명시 확인 prompt 통과 후만 진입
5. `repositories/<slug>/` 안의 `repo.yaml` 복원 — 하지만 사용자 코드는 archive에 없으므로 별도 안내 ("이 archive에는 사용자 코드가 포함되지 않습니다. <repo>는 빈 SoloSquad 메타만 복원되며, 실제 코드는 git clone 후 같은 자리에 두면 됩니다.")
6. journal에 `import` 단계 기록 (v0.7 idempotent 재개와 동일 패턴)

7 CVE 해소:
- discord.js 14.16 → 14.21+ (undici 패치 버전 포함된 minor) 또는 15.x 메이저 업그레이드
- 메이저 업그레이드 시 breaking change 검토 — discord.js 15에는 v14 deprecation 일부 제거
- npm overrides 방식도 가능: `package.json`에 `"overrides": { "undici": "^6.21.0" }`

v0.8 진입 전 PoC 권고: feat 브랜치에서 discord.js 14.21 +0% breaking 확인 → 적용 우선. 안 되면 overrides fallback.

### 7.3 `solosquad archive verify <zip>` (P0 #3)

`src/cli/archive.ts` (신규) — `archive` 서브그룹:
- `verify <zip>` — manifest SHA × 실제 SHA + archive.yaml schema check. 무결성 보고서 출력
- `info <zip>` — archive.yaml 메타 + manifest 요약 (entry 수·클래스별·총 사이즈) 출력
- `list <zip>` — manifest.tsv 전체 또는 `--class <X>` 필터 출력

`src/lifecycle/archive-reader.ts` (신규) — yauzl 같은 가벼운 pure-JS zip reader (또는 archiver의 sister lib) 추가. archive 작성에 쓴 archiver는 writer-only.

### 7.4 API stability 정책 문서 (P0 #4)

`docs/api-stability.md` 신설. 다음 6개 schema_version의 bump 룰 명시:

| schema | 현재 | bump 룰 |
|---|---|---|
| `workspace.yaml.version` (SoloSquad 자체 버전) | 0.8.0 | SemVer. 0.x는 free-form, 1.0부터 breaking은 major bump |
| `<org>/.org.yaml.schema_version` | 1 | 필드 *추가*는 minor·동일 schema. 필드 *제거/타입 변경*은 schema_version bump |
| `<org>/agent-profile.yaml.schema_version` | 1 | 동일 룰. narrowing-only invariant는 schema와 무관 |
| `assets/agents/<>/SKILL.md` frontmatter | (스키마 버전 키 없음 — v0.8에서 신설 권고) | v0.8 §7.4 — `schema_version: 1` 추가하고 동일 룰 적용 |
| `archive.zip/archive.yaml.schema_version` | 1 | 동일 룰. `max_schema_version_supported` 사용해 import 호환 거부 |
| `archive.zip/manifest.tsv` 첫 줄 `schema_version=1` | 1 | 동일 룰 |

deprecation 기간:
- v1.0 이후 minor bump에서 schema_version 변경 시 *이전 schema도 1 minor 더 지원* (예: v1.2에서 schema=2 도입, v1.3까지 schema=1도 읽음)
- v1.x major bump (v2.0)에서만 옛 schema 영구 거부 가능

## 10. 마이그레이션 0.7.x → 0.8.0

1. `package.json.version` 0.7.x → 0.8.0
2. `workspace.yaml` 버전 stamp + 신규 키 보강:

   ```yaml
     broadcast_enabled: false       # 기본 보수 (§3.7 박제)

   ```
3. 현재 사용자를 첫 user로 등록: `<org>/.solosquad/users/<owner-handle>.yaml`
   - handle: `.env`의 봇 token에서 messenger API 호출로 추출 (Discord `/users/@me`, Slack `auth.test`)
   - bot_user_id: 동일 경로로 획득
   - channels: **새 명명 규칙으로** (`command-<handle>` / `works-<handle>`). 첫 봇 startup 시 자동 생성
4. 25 SKILL.md에 `schema_version: 1` 추가 (`scripts/inject-skill-schema-version.ts` 호출)

6. `~/.solosquad-backups/`에 0.7 → 0.8 전 스냅샷 생성 (마이그레이션 표준)
7. 마이그레이션 verify 단계에서 *legacy 채널 안내* 1회 출력:
   `"기존 #owner-command / #workflow 채널은 봇이 더 이상 listen 하지 않습니다. 새 채널 command-<handle> / works-<handle>가 첫 봇 startup 시 생성됩니다. legacy 채널은 메신저에서 수동 archive 권장."`
