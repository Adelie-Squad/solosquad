# v0.4 — 밤새 자율 작업 완료 엔진

> "밤에 지시하고 아침에 결과를 받는다." OpenAI Codex의 **`/goal` + `AGENTS.md` 2계층 구조**를 차용한 자율 실행 루프.

**로드맵:** v0.1 크로스 플랫폼 → v0.2 안전/보안 → v0.3 PM 모드 (npm 1.2.5 출시) → **v0.4 자율 실행** → v0.5 워크플로우 메이커 → v0.6 디폴트 워크플로우 → v1.1 대시보드 상호작용 → v1.2 지식 온톨로지 → v2.0 커뮤니티 → v3.0 코워킹

**의존:** v0.1 (크로스 플랫폼), v0.2 (안전/보안 — 밤새 자율 실행 전 보안 기반 필수), v0.3 (PM 모드 — PM이 자율 루프의 진행자, npm 1.2.5)

## 2. 채택 구조 — Codex `/goal` + `AGENTS.md` 2계층

2026년 4월 OpenAI Codex CLI 0.128.0이 도입한 [`/goal`](https://developers.openai.com/codex/use-cases/follow-goals) 메커니즘이 본 기능과 가장 정렬된 모던 참조다. 핵심 분리:

| 계층 | Codex 명칭 | 역할 | 편집 주체 |
|---|---|---|---|
| 휘발성 의도 | **goal** (한 문장 + 정지 조건) | "지금 무엇을 달성하나" — 실행마다 갈아끼움 | 사람 only (LLM은 `create_goal` / `update_goal(complete)`만) |
| 영속 가이드 | **`AGENTS.md`** | 프로젝트 컨벤션·도구·금지 경로·자원 위치 — 실행 간 공유 | 사람 only |

이 2계층은 다음 두 가지를 분리해서 누리는 게 핵심:
1. **자율 루프 트리거 가벼움** — `solosquad goal new` 시 사용자가 30초 만에 의도를 적고 실행. 매번 거대한 `program.md` 다시 안 적음
2. **컨벤션·가드레일은 한 번 박제** — `AGENTS.md`에 immutable_paths·도구 사용 규칙·output 가드를 박아두면 모든 goal이 자동 상속

추가로 `AGENTS.md`는 **Aider·Cursor·Codex 진영의 cross-tool 표준**이라 SoloSquad 사용자가 다른 에이전트 도구를 병용해도 한 파일이 함께 인식된다 ([Aider docs](https://aider.chat/docs/usage/conventions.html), [Codex AGENTS.md spec](https://developers.openai.com/codex/agents-md/)). 솔로 파운더 워크플로 측면에서 상당히 중요한 호환 효과.

원조 영감인 Karpathy의 [autoresearch](https://github.com/karpathy/autoresearch)는 메트릭 게이팅·git revert·"NEVER STOP" 루프의 **운영 패턴**으로만 차용한다. 어휘는 Codex 진영으로 정렬.

| 차용 항목 | 출처 | 우리 적용 |
|---|---|---|
| 2계층 분리 (volatile goal / persistent AGENTS.md) | Codex `/goal` | 본 v0.4의 골자 |
| `AGENTS.md` 파일 명칭·역할 | Codex + Aider 공통 표준 | 워크스페이스 루트에 1개. 기존 `CLAUDE.md`와 공존 |
| `goal` 1줄 문장 + 정지 조건 | Codex `/goal` | `<org>/goals/<goal-id>/goal.md` frontmatter |
| 메트릭 게이팅·keep/discard·git revert | Karpathy autoresearch (운영 패턴) | §4 cycle loop |
| 단일 메트릭 판단 | autoresearch | `goal.md`의 `metrics:` |
| Git = 실험 메모리 | autoresearch | v0.3 `git-snapshot.ts` 재사용 |

### 2.1 Data Reconciliation 원칙 (보고서 §4 차용)

자율 루프만 차용하면 **Goodhart 법칙**에 노출된다 — 에이전트가 메트릭 산출 로직을 왜곡하면 keep/discard가 무의미해진다. 보고서의 "지표 산출 로직과 데이터 마트 원천값의 일치 여부 확인" 원칙을 v0.4 기본 룰로 채택:

1. **`metric_provenance` 필드 의무화** — 모든 메트릭 값은 산출 공식·원천 파일 경로·라인 범위를 함께 기록한다. results.tsv·_events.jsonl 양쪽에 포함.
2. **재계산 가능성** — 같은 commit에서 evaluator를 재실행하면 동일 값이 나와야 한다. `solosquad goal verify <id> --cycle <n>`로 검증.
3. **공식 변경은 새 사이클** — metric 산출 공식 자체를 바꾸면 이전 사이클과 비교 불가. 새 results.tsv 섹션으로 분리.
4. **메트릭/엔진 자기 수정 금지** — 에이전트 spawn의 *수정 가능 경로* 화이트리스트는 다음을 **포함할 수 없다** (`AGENTS.md` 디폴트로 박제, override 시 경고):
   - `src/engine/**` (parser·evaluator·tracker·reconciliation·guards)
   - `assets/templates/results.tsv`, `assets/templates/goal.md`
   - 현재 실행 중인 `goal.md` 자체
   - `AGENTS.md` 자체
   - `<org>/goals/<goal-id>/results.tsv` (해당 goal의 누적 결과)

   Codex `/goal`의 격리(사람만 goal·AGENTS.md 수정, LLM은 propose만)와 동일 정신. evaluator를 LLM이 만지면 keep/discard 자체가 신뢰 불가.

이 룰이 없으면 자율 루프는 "메트릭 자체를 조작하면 cheap win"의 함정에 빠진다.

## 3. 자율 실행 아키텍처

v0.3 PM mode 위에 *얇은 컴파일러*로 얹는다. **에이전트 실행·핸드오프·spawn은 v0.3 PM session + native Task tool이 이미 제공**한다. v0.4는 그 위에서 (a) `goal.md`를 PM이 이해할 system prompt로 컴파일, (b) `AGENTS.md`를 영속 가이드로 합성, (c) cycle 단위 메트릭 평가·git rollback, (d) 누적 비용·시간·discard 카운터 — 이 네 가지만 더한다.

```
사용자 의도 (goal.md, 사람만 수정)
    +
워크스페이스 컨벤션 (AGENTS.md, 사람만 수정)
    │
    ▼  solosquad goal run <goal-id> --hours 8
┌──────────────────────────────────────────────────────────┐
│  Goal Runner (src/engine/)                                │
│                                                          │
│  1. goal-parser     : goal.md → GoalSpec                  │
│     - 1줄 의도 + 메트릭(provenance) + 파이프라인 + 종료조건   │
│     - target_repo·org 필수 (8-layer assembler 입력)         │
│                                                          │
│  2. agents-md-loader: AGENTS.md → PersistentGuide          │
│     - immutable_paths·도메인 화이트리스트·Output 가드 정책    │
│     - 워크스페이스 루트 1개, 모든 goal이 공유                │
│                                                          │
│  3. guards (Input)  : 비용·시간·경로 화이트리스트 + immutable │
│     - goal.md의 modifiable_paths(있으면) ∩ AGENTS.md의       │
│       immutable_paths 차집합으로 최종 수정 가능 경로 결정     │
│                                                          │
│  4. spawn dedicated PM session                            │
│     session-id = bg-<goal-id>-<run-ts>  (사용자 PM과 분리)   │
│     cwd        = org root  (v0.3 §3.2.1)                  │
│     system prompt = orchestrator/SKILL.md +                │
│                     goal-md-spec.md +                      │
│                     GoalSpec (rendered) +                  │
│                     AGENTS.md 본문                          │
│                                                          │
│  5. per-cycle loop                                         │
│     ┌────────────────────────────────────────────────┐    │
│     │ a. git-snapshot.commit (v0.3 재사용)            │    │
│     │    msg: chore(spawn): goal-<id> cycle-<n>       │    │
│     │ b. PM이 Task로 pipeline stage chain (native)    │    │
│     │    → _events.jsonl에 task_id·cost·duration       │    │
│     │ c. evaluator.measure(metric, provenance)        │    │
│     │ d. status = keep | discard                       │    │
│     │    keep    → commit message 보강 (metric=val)    │    │
│     │    discard → git-snapshot.revert(pre-cycle)      │    │
│     │ e. tracker.append(results.tsv, task_id 외래키)   │    │
│     │ f. reconciliation.verify (동일 commit 재계산)    │    │
│     │ g. guards (Runtime): timeout·discard streak·예산 │    │
│     └────────────────────────────────────────────────┘    │
│                                                          │
│  6. 종료 트리거 발생 → ship 결정 (§ 종료 정책)              │
│  7. summary → morning-brief routine 큐                    │
└──────────────────────────────────────────────────────────┘
    │
    ▼
아침 08:00 morning-brief가 #owner-command thread에 결과 요약 post
(자율 run은 메신저 직접 전송 금지 — Output 가드)
```

**PM 세션 격리 결정** (v0.3 PoC #1: 같은 session-id 동시 호출 시 jsonl interleaved):

- 자율 run은 **사용자 PM session을 점유하지 않는다**. `bg-<goal-id>-<run-ts>` 형식의 dedicated session-id를 발급.
- 22:00 자율 run 중 23:30 사용자 메시지가 와도 사용자 PM session은 별도 — 두 session 모두 정상 동작.
- background session도 `session-store.ts`(v0.3)에 등록 — `(user-id, org-slug, kind=background)` 튜플. `solosquad pm status`에 `[bg] goal-<id> running 4h12m / 8h00m`으로 surface.

**v0.3 자산 재사용 (신규 모듈 아님)**:

| v0.3 모듈 | v0.4에서의 역할 |
|---|---|
| `pm-runner.ts` | goal+AGENTS system prompt를 background session에 inject. mutex·stderr regex 그대로 |
| `agents-builder.ts` (8-layer assembler) | spawn 컨텍스트 조립. autonomous spawn도 동일 경로 |
| `git-snapshot.ts` | spawn-pre auto-commit + `evaluator.discard` 시 revert 호출. v0.4 evaluator가 keep 시 commit 메시지 보강만 |
| `events.ts` (`_events.jsonl`) | task_id·cost·duration 그대로 흘림. v0.4 `results.tsv`가 task_id를 외래키로 참조 |
| `session-store.ts` | background session 등록 + `total_cost_usd` 누적기 (discard cycle도 비용 적재) |

v0.4 신규는 *goal+AGENTS 파서 + 메트릭 평가 + 누적 트래커*뿐.

## 4. `goal.md` + `AGENTS.md` 설계

### 4.1 `<org>/goals/<goal-id>/goal.md` — 1회 실행 의도

```markdown
---
schema_version: 1                  # results.tsv 호환성 — v0.6 trajectory 추출 의존
goal_id: {kebab-case-slug}          # 디렉토리·session-id 접미사
org: {org-slug}                     # PM session cwd 결정
target_repo: {repo-slug or null}    # 단일 repo 작업이면 명시. 8-layer assembler [8] inject
cycle_unit: pipeline_pass           # "한 cycle = pipeline 1회 통과" 고정. stage 단위 cycle은 v0.5+
---

# Goal: {1줄 의도}

{1~3줄 부연 — Codex /goal의 "Acceptance" / "Stop rule" 정신}

## 메트릭
- {metric_name}:
    formula: {산출 공식, 1줄}              # provenance — 재계산 가능해야 함
    source: {원천 파일·경로 또는 URL}      # provenance — 측정 기준점
    threshold: {keep 판단 값}
    direction: maximize | minimize         # 종료·oscillation 판단

## 에이전트 파이프라인 (정적 ordered list)
1. {agent_name}: {이 단계에서 할 일}
2. {agent_name}: {이 단계에서 할 일}
3. ...
# cycle 안에서 stage 순서는 정적. 각 stage의 *프롬프트 변형*만 PM이 propose 단계에서 동적 생성.

## 시간·비용 예산
- 시간: {N}시간 또는 {N}사이클
- API 비용: $N per cycle, $M total  # discard cycle도 적재

## 종료 조건 (먼저 도달한 것 우선)
- 모든 메트릭 ≥ threshold 도달 + 연속 2 cycle 유지 (oscillation 방어, § 종료 정책 참조)
- 시간 예산 소진
- N회 연속 discard (교착 탈출, 기본값 5)
- API 비용 한도 도달

## 수정 가능 경로 (이 goal 한정 override, optional)
- {디렉토리} : 기본은 AGENTS.md의 modifiable_paths 상속. 이 goal만 좁히려면 명시.

## Signal trigger (선택 — §5 signal-scan 자동 발화)
auto: prompt    # auto: true | prompt | false
                # prompt: urgency=high 발견 시 PM이 사용자에게 "이 goal 실행할까?" 1턴 묻고 yes만 실행
                # true: 자동 실행 (사용자 사전 승인 필요 — 강한 자율)
                # false: 알림만, 실행 안 함 (디폴트)
```

**Codex `/goal`과의 차이**: Codex는 휘발성 SQLite row + 자연어 1문장. SoloSquad는 메트릭·파이프라인·시간 예산이 결합된 **구조화 파일**이 필요하므로 markdown + frontmatter로 풀어 적는다. 다만 *본문 첫 줄 H1*은 Codex의 goal 문장과 동일 역할을 한다 — 사용자가 30초에 "무엇을 달성하려는가" 한 문장 적는다.

### 4.2 `<workspace>/AGENTS.md` — 영속 가이드 (cross-tool 표준)

워크스페이스 루트에 단 1개. SoloSquad·Codex·Aider·Cursor가 모두 동일 파일을 인식해 컨벤션을 공유한다. 기존 `CLAUDE.md`와 공존 — 전자는 Anthropic 진영 narrative, 후자는 cross-tool 표준 (`solosquad init`이 둘 다 생성).

```markdown
# AGENTS.md

> Persistent project guide. All AI tools (SoloSquad, Codex, Aider, Cursor)
> should read this. Human-edited only.

## Project
{워크스페이스 명·도메인·핵심 stakeholder}

## Tone & Style
{글쓰기·코드 스타일·금지 표현}

## Build / Test
{빌드 명령·테스트 명령·필수 lint 단계}

## Code conventions
{언어별 규칙·금지 패턴·선호 라이브러리}

## SoloSquad v0.4 — Autonomous Goal Conventions

### Immutable paths (절대 수정 금지)
- src/engine/**                  # parser·evaluator·tracker·reconciliation·guards
- assets/templates/results.tsv
- assets/templates/goal.md
- AGENTS.md                       # 이 파일 자체
- <org>/goals/<goal-id>/goal.md   # 실행 중 goal 자신
- <org>/goals/<goal-id>/results.tsv

### Modifiable paths (자율 실행에서 수정 허용)
- <org>/workflows/<wf-id>/
- <org>/memory/
- (각 goal.md가 추가로 좁힐 수 있음)

### External side-effects (자율 실행 동안 금지 — Output 가드)
- 메신저 직접 전송: 금지. 결과는 morning-brief routine 경유
- 이메일·결제·외부 API mutating call: 금지
- 외부 HTTP 호출 화이트리스트: {도메인 목록} (비어 있으면 외부 차단)

### Guardrail thresholds
- 1 stage 응답 타임아웃: 600s
- 연속 discard 강제 종료: 5 cycles
- 비용 캡 90% 도달 시: 마지막 cycle만 허용 후 종료
```

**3단계 가드레일 구분의 의도**: 보고서의 "사전 가드레일"은 솔로 환경에서 인프라 비용이 큼. v0.4는 *외부 비가역 부수효과만* Input/Output 가드로 사전 차단하고, 나머지(코드 변경·메모리 변경)는 `git reset`으로 사후 롤백한다. v0.3 `git-snapshot`을 그대로 재사용:

```
v0.3 git-snapshot.commit  →  PM이 Task로 pipeline 실행  →  evaluator.measure
       (pre-cycle 자동)                                          │
                                                       ┌─────────┴─────────┐
                                                       keep                discard
                                                       │                   │
                                                  commit msg 보강       v0.3 git-snapshot.revert
                                                  (metric=val)         (pre-cycle 커밋 reset)
                                                  results.tsv:keep     results.tsv:discard
                                                  (task_id 외래키)      (task_id 외래키)
```

evaluator는 keep/discard 판단·메시지 보강·tracker append만 담당. git 트랜잭션 자체는 `git-snapshot.ts`(v0.3) 공개 API 호출.

## 5. 실험 추적

**저장 경로**: `<org>/goals/<goal-id>/results.tsv` — **goal 단위**(cross-workflow). 한 goal이 여러 workflow를 생성해도 모든 cycle이 한 results.tsv에 누적.

**스키마** (10필드, append-only, 첫 줄 헤더 다음 `# schema_version=1` 주석):

```tsv
# schema_version=1
cycle	timestamp	agent	metric	value	status	commit	provenance	task_id	description
1	2026-03-19T22:05Z	desk-researcher	signal_relevance	0.45	discard	-	formula=count(matching_keywords)/total;source=memory/signals.jsonl	tsk_01HX...	초기 시장 스캔
```

**`task_id` 외래키 (v0.3 `_events.jsonl` 연결)**: v0.3 `events.ts`가 spawn마다 발급한 `task_id`를 그대로 사용. cycle row → spawn 이벤트(`<org>/workflows/<wf-id>/_events.jsonl`의 동일 task_id) → cost·duration·subagent transcript 추적 가능. `solosquad workflow show <id>`(v0.3)는 양쪽을 JOIN해 surface.

**provenance 컬럼 (보고서 §4 Data Reconciliation 적용)**: 같은 commit에서 동일 provenance로 재계산 시 동일 value가 나와야 한다. `solosquad goal verify <id> --cycle <n>`로 자동 검증. 공식 변경 시 새 메트릭 이름 사용.

**`schema_version` 의의 (v0.6 의존)**: v0.6 §3.1 trajectory→skill 추출 트리거가 results.tsv 스키마를 입력으로 받는다. 스키마 breaking change 시 `schema_version` bump → 구 버전 results.tsv는 v0.6 추출기가 silently skip. 호환 깨짐 추적 가능.

**Git 커밋 컨벤션:**
```
[cycle-{N}] {status} {agent}: {description} ({metric}={value})
```
v0.3 `git-snapshot.ts`의 pre-cycle 커밋 메시지(`chore(spawn): goal-<id> cycle-<n>`)를 keep 시점에 위 형식으로 amend가 아닌 **새 commit 추가**(amend 금지 — published commit 보호). discard 시 pre-cycle 커밋으로 `reset --hard`, results.tsv에는 status=discard 행만 남음.

### 5.1 signal-scan trigger 연동 — 알림 vs 자율 실행 분리

v0.2.4 `signal-scan` routine은 read-only — JSONL 적재만. v0.4부터 `urgency: high/med/low` 필드를 결과 JSON에 추가. **자동 실행 여부는 `goal.md`의 `signal_trigger.auto` 필드 (§4.1)에 의해 사용자 사전 결정**.

```
signal-scan (12:00 daily) → urgency=high 발견
  │
  ├─ matching goal.md 검색: <org>/goals/*/goal.md 중 signal_trigger 설정된 것
  │
  ├─ auto=false (디폴트)  → PM thread에 "🚨 high urgency 신호 N건. 관련 goal 없음" 알림. 끝.
  ├─ auto=prompt          → PM이 사용자에게 "신호 X 발견. goal <goal-id> 실행할까요? (8h, 예상 비용 $5)" 1턴. yes만 실행.
  └─ auto=true            → 즉시 background session으로 `solosquad goal run <goal-id> --hours <budget>` 발화.
                             결과는 다음 morning-brief에 포함.
```

**자율 정도의 명확화**: 디폴트는 *alert only*. *prompt* 모드는 1-click execute. *true* 모드는 강한 자율 — 사용자가 `goal.md`에 명시적으로 켜야 함 (예: PMF Discovery는 prompt, 사후 분석 routine은 false 권장). v0.4 출시 시 4종 디폴트 goal은 모두 `auto: false` 또는 `prompt`로 시작.

## 6. CLI

```bash
# goal 신규 작성 (템플릿 복사)
solosquad goal new <goal-id>

# 워크스페이스 goal 일람
solosquad goal list

# 특정 goal 상세 (goal.md + 누적 results.tsv 슬라이스 + 최근 cycle)
solosquad goal show <goal-id>

# 밤새 자율 실행 (시간 예산 우선)
solosquad goal run <goal-id> --hours 8

# 단일 사이클만 실행 (디버깅)
solosquad goal run <goal-id> --cycles 1

# 실행 중인 goal 일람 (background session 포함)
solosquad goal status [<goal-id>]

# 실행 중 goal 중단
solosquad goal stop <goal-id>

# 특정 cycle의 provenance 재계산 검증
solosquad goal verify <goal-id> --cycle <cycle-num>
```

**v0.3 명령과의 통합**:

- `solosquad workflow list` (v0.3)는 자율 run으로 생성된 workflow도 동일하게 표시. `meta.source = "goal:<goal-id>"` 필드로 출처 구분.
- `solosquad workflow show <wf-id>`는 `_events.jsonl` + `results.tsv`(task_id JOIN)를 합쳐 cycle별 cost·metric·status 표 출력.
- `solosquad pm status`는 `[user] sid-... idle 12m`와 `[bg] goal-<id> running 4h12m / 8h00m cost=$2.31/$5` 두 라인을 함께 노출.
- `solosquad rollback --workflow <wf-id>` (v0.3)는 자율 run의 cycle 커밋도 동일 메커니즘으로 revert — v0.4 별도 rollback CLI 불필요.

`<goal-id>` 인자는 `<org>/goals/<goal-id>/goal.md`로 해석. cwd 추론은 v0.3 `resolveOrgCwd` 재사용. 외부 절대경로의 goal.md는 `--file <path>` 로 우회 가능.

## 8. 종료 정책 + Oscillation 방어

§4.1 goal.md `## 종료 조건` 항목 중 "모든 메트릭 > threshold + 연속 2 cycle 유지"의 운영 세부:

```
state machine (per goal run)
  ────────────────────────────────
  RUNNING ──[모든 메트릭 ≥ threshold 첫 도달]──→ CONFIRMING(1)
                                                   │
                          [다음 cycle도 ≥ threshold]│  [다음 cycle 어느 메트릭이라도 < threshold]
                                                   ▼
                                              CONFIRMING(2) ←─────── RUNNING (best-keep 갱신)
                                                   │
                          [3번째 cycle도 유지]      │
                                                   ▼
                                                CONVERGED → ship
```

**`best-keep` 추적**: tracker가 keep cycle마다 `<org>/goals/<goal-id>/_best.json`을 갱신 — 모든 메트릭이 threshold 이상이면서 가장 높은 합산 점수를 가진 cycle의 commit·timestamp 기록. CONVERGED 도달 시 그 commit을 ship 후보로 surface. 시간 예산 소진 등 다른 종료 트리거 시에도 `_best.json`이 있으면 그 commit을, 없으면 마지막 keep commit을 ship 후보로 안내.

**oscillation 감지**: 직전 5 cycle 중 keep ↔ discard ≥ 3회 전환 시 morning-brief에 "메트릭 진동 — threshold 재조정 권장" 1줄 추가. 자동 종료는 하지 않음 (사용자 판단 영역).
