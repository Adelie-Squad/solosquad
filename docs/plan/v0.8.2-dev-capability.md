# v0.8.2 — Dev Capability

> Engineering 에이전트가 **코드 수정 + commit + push + PR 생성**까지 end-to-end로 수행하는 능력을 1급 시민화한다. 현재 v0.8까지는 PM 세션이 Task 도구로 specialist를 위임할 수 있지만, Bash·git·gh 사용은 *암묵적* — SKILL이 자신의 dev 권한을 선언하지 못한다. v0.8.2는 frontmatter에 명시적 권한 + push/merge 시 사용자 confirmation gate + gh CLI 통합을 도입한다. **자동 머지는 영구 거부**.

**로드맵:** v0.8 Multi-User Messenger → v0.8.1 Security & Lifecycle Pair → **v0.8.2 Dev Capability** → v0.8.3 Onboarding UX + Observability → v0.9 안정화 → v1.0 정식 출시

**의존:** v0.4 (`goal-runner` modifiable_paths 화이트리스트 — dev_permissions의 영감), v0.5 (SKILL frontmatter 스키마), v0.6 (8-layer JIT spawn-assembler — dev 권한 주입 지점), v0.8.1 (SKILL.md `schema_version: 1` 신설)

**기준 버전:** v0.8.2. patch이지만 새 기능 도입 — frontmatter 필드 + spawn-assembler 동작 + confirmation gate.

**분류:** Capability 확장 + 권한 정책.

## 핵심 질문 & 성공 기준

| 핵심 질문 | 성공 기준 |
|---|---|
| 사용자가 "X 기능 추가해줘. PR까지 만들어줘" 1줄에 PR URL 회신을 받는가? | engineering 에이전트가 코드 수정 + 테스트 + commit + push + `gh pr create`까지 자율. 사용자 confirmation gate를 push 단계 직전에 통과 |
| 자동 머지는 영구히 거부되는가? | `gh pr merge` 명령은 사용자 명시 별도 요청(`@bot PR-<id> 머지해`) + confirmation 통과 후만 실행. SKILL frontmatter의 `merge.auto: false` 영구 고정 |
| 비-dev SKILL이 실수로 Bash를 호출할 수 없는가? | `dev_capability: false` (기본) SKILL은 Bash 도구 자체가 spawn config에서 제외 |
| 위험 명령(rm -rf /·sudo)이 차단되는가? | workspace.yaml `dev_capability.bash_denylist` 가 모든 SKILL에 강제 적용 (allowlist override 불가) |

## 1. 현재 문제 (v0.8 시점)

| 문제 | 원인 |
|---|---|
| Bash 권한 wide-open — `claude-process.ts`가 `--allowed-tools` 미전달이라 default 권한이지만, dev 액션 정책이 SKILL.md에 미반영. PR 생성/머지 흐름이 1차 시민이 아님 | 권한 정책 + 워크플로 미설계 |
| `git push`/`gh pr merge` 같은 sensitive 명령이 사용자 confirmation 없이 실행 가능 | confirmation gate 미설계 |
| Architect/Cloud Admin/Security Engineer 같은 advice-only 에이전트가 실수로 코드 수정해도 막을 장치 없음 | 역할 ↔ 권한 매핑 부재 |
| gh CLI 인증 상태가 doctor에서 점검 안 됨 | v0.8까지 dev 워크플로가 1급 시민 아니어서 |

## 2. 원칙

| 원칙 | 구현 |
|---|---|
| **dev 능력은 명시 선언** | SKILL.md frontmatter에 `dev_capability: true` 박힌 에이전트만 Bash·Edit·Write 도구 활성. 기본 `false` (read-only) |
| **자동 머지 영구 거부** | `gh pr merge` 자동 실행 0건. 사용자가 명시 요청 → PM이 별도 위임 → confirmation 통과 후만 |
| **민감 명령 confirmation gate 의무** | `git push`, `gh pr merge`, `gh pr close` 등은 PM 세션으로 confirmation 요청. 사용자 응답 전까지 Bash blocking |
| **위험 명령은 workspace 레벨 denylist** | `rm -rf /`, `sudo`, `chmod 777` 등은 SKILL allowlist에 포함돼도 거부 |
| **engineering 5 SKILL만 박제 활성화** | Backend Developer / FDE / API Developer / Creative Frontend / QA Engineer만 `dev_capability: true`. Architect / Cloud Admin / Data Engineer / Data Collector / Security Engineer는 `false` (advice + review만). 그 외 15 SKILL(strategy/growth/experience)도 모두 `false` |
| **워크스페이스 마스터 토글** | `workspace.yaml.dev_capability.enabled` (기본 `true`) — `false` 시 모든 SKILL의 dev_capability 무력화. client confidential repo·sandbox 환경용 |

## 3. SKILL.md frontmatter 스키마

### 3.1 신규 필드

```yaml
---
name: backend-developer
team: engineering
schema_version: 1                  # v0.8.1에서 신설
dev_capability: true               # v0.8.2 신규
dev_permissions:
  bash:
    allowed: ["git", "gh", "npm", "node", "tsc", "pytest", "go", "cargo"]
    denied: ["rm -rf /", "sudo", "chmod 777"]
  network: false                  # MCP 서버 외 outbound 차단 (curl/wget 거부)
  push_targets:
    requires_confirmation: true   # PR push 직전 사용자 확인 강제
  merge:
    auto: false                   # 자동 머지 영구 거부
triggers:
  keyword: [...]
collab_pattern: hierarchical
---
```

| 팀 | 에이전트 | dev_capability | 사유 |

| engineering | Architect | `false` | advice + review, 코드 안 만짐 |
| engineering | Cloud Admin | `false` | infra advice, 코드 안 만짐 |
| engineering | Data Engineer | `false` | 데이터 design advice |
| engineering | Data Collector | `false` | research/spec, 코드 안 만짐 |

| strategy 7 + growth 4 + experience 4 + meta | (전부) | `false` | non-coding |

### 3.3 Workspace 마스터 토글 (박제)

`workspace.yaml`:
```yaml
dev_capability:
  enabled: true                    # 마스터 (기본 true)
  require_push_confirmation: true  # 모든 push에 confirmation (overrideable per SKILL 아님)
  bash_denylist:                   # 모든 SKILL에 강제 적용
    - "rm -rf /"
    - "sudo"
    - "chmod 777"
    - "curl http*"                 # network: false인 경우 추가
```

`enabled: false` 효과:
- 25 SKILL 모두 dev 액션 0건 (개별 `dev_capability: true`도 무력화)
- spawn-assembler가 Bash 도구 자체 제외
- 솔로 사용자·sandbox·client confidential repo 환경용

`require_push_confirmation: false` 거부 — 항상 `true`만 허용 (보수적 정책 박제).

## 4. spawn-assembler 권한 주입

### 4.1 흐름

```typescript
// src/bot/spawn-assembler.ts 확장
function applyDevPermissions(spawnConfig: SpawnConfig, skill: ParsedSkill, workspaceYaml: WorkspaceYaml): SpawnConfig {
  const masterEnabled = workspaceYaml.dev_capability?.enabled ?? true;
  const skillEnabled = skill.frontmatter.dev_capability === true;

  if (!masterEnabled || !skillEnabled) {
    // read-only 모드
    spawnConfig.allowedTools = ["Read", "Grep", "Glob"];
    spawnConfig.disallowedTools = ["Bash", "Edit", "Write"];
    return spawnConfig;
  }

  const perms = skill.frontmatter.dev_permissions ?? {};
  spawnConfig.allowedTools = ["Read", "Grep", "Glob", "Edit", "Write", "Bash"];
  spawnConfig.bashAllowlist = perms.bash?.allowed ?? [];
  spawnConfig.bashDenylist = [
    ...(workspaceYaml.dev_capability?.bash_denylist ?? []),  // workspace 강제
    ...(perms.bash?.denied ?? []),                            // SKILL 자체 추가
  ];
  spawnConfig.requirePushConfirmation = perms.push_targets?.requires_confirmation
    ?? workspaceYaml.dev_capability?.require_push_confirmation
    ?? true;
  spawnConfig.networkAllowed = perms.network ?? false;
  return spawnConfig;
}
```

### 4.2 Claude Code `--allowed-tools` 매핑

`src/bot/claude-process.ts`가 spawn 시 `--allowed-tools`·`--disallowed-tools` 명시 전달:
```bash
claude code \
  --allowed-tools "Read,Grep,Glob,Edit,Write,Bash" \
  --bash-allowlist "git,gh,npm,node,tsc,pytest" \
  --bash-denylist "rm -rf /,sudo,chmod 777"
```

(Claude Code CLI가 native `--bash-allowlist`를 지원하지 않으면 SoloSquad 측 wrapping으로 Bash 명령 pre-check)

## 5. PR 흐름

### 5.1 PM 세션의 dev 위임 템플릿

`assets/orchestrator/SKILL.md` 확장 — engineering 위임 시 다음 system prompt 주입:

```markdown
## Engineering Spawn Template (v0.8.2)

작업 단계를 다음 순서로 수행:

1. **Branch 생성**: `git checkout -b feat/<short-slug>` (main에서 분기)
2. **분석 + 수정**: 사용자 의도를 코드로 변환. Edit/Write 도구 사용
3. **테스트**: 영향받은 영역의 테스트 실행. 실패 시 수정·재시도
4. **커밋**: `git add <specific files>` → `git commit -m "<imperative summary>"`.
   **`-A` 금지, hooks skip 금지**
5. **푸시**: `git push origin feat/<short-slug>` — 이 단계 *직전*에 PM에게
   "push 진행?" 보고 (confirmation gate)
6. **PR 생성**: `gh pr create --title <60자 이내> --body <HEREDOC>` — base는 main
7. **회신**: PR URL을 사용자에게 1줄로 회신. *머지 시도 금지*

머지는 사용자가 별도 명령 `@bot PR-<id> 머지해`로 요청해야 PM이 다시 위임.
이 단계도 사용자 confirmation prompt 통과 후만 실행.
```

### 5.2 사용자 confirmation gate

`src/bot/dev-confirm.ts` 신규:

- spawn 도중 `Bash` 도구로 `git push`·`gh pr merge`·`gh pr close` 명령이 시도되면:
  1. 명령 실행 *전에* PM session으로 confirmation request 이벤트 발행
  2. PM이 사용자 채널(`command-<handle>`)에 "다음 명령을 실행할까요? `git push origin feat/dark-mode` [y/N]"
  3. 사용자 응답 받기 전까지 Bash 도구 blocking
  4. 응답 `y` → 실행, 그 외 → "취소됨" 반환
- 30분 timeout: 자동 취소
- 모든 confirmation 이벤트는 `<org>/memory/dev-confirmations.jsonl`에 append (감사 로그)

`dev-confirmations.jsonl` 포맷:
```jsonl
{"ts":"2026-05-15T12:00:00Z","user":"alice","skill":"backend-developer","cmd":"git push origin feat/dark-mode","decision":"y","duration_ms":4200}
{"ts":"2026-05-15T12:01:00Z","user":"bob","skill":"api-developer","cmd":"gh pr merge 42","decision":"timeout","duration_ms":1800000}
```

### 5.3 gh CLI 인증

전제: 사용자가 머신에 `gh auth login` 1회 완료. SoloSquad는 자기 인증 안 함.

`solosquad doctor` v0.8.2 점검 추가:
- `gh --version` 존재 확인
- `gh auth status` 통과 확인
- 통과 시 "engineering 에이전트가 PR 생성 가능" 표시
- 실패 시 "gh auth login 권고" + dev_capability 에이전트는 PR 단계 건너뜀 (코드 변경 + commit만)
