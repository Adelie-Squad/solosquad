# v0.5 — 워크플로우 메이커 & 스킬 시스템 통합

> 솔로 파운더가 메신저에서 PM과 대화하며 자신의 지식·암묵지를 SKILL.md + 워크플로우 체인으로 자동 변환한다. 기존 `.claude/skills/`를 가진 코드베이스도 동일 파이프라인으로 흡수. v0.3 PM 모드를 "사용자 의도 → 실행 인프라" 컴파일러로 확장.

**로드맵:** v0.1 크로스 플랫폼 → v0.2 안전/보안 → v0.3 PM 모드 → v0.4 자율 실행 → **v0.5 워크플로우 메이커** → v0.6 디폴트 워크플로우 → v1.1 대시보드 상호작용 → v1.2 지식 온톨로지 → v2.0 커뮤니티 → v3.0 코워킹

**의존:** v0.3 (PM 모드 — 사용자 대화 진행자 + 내장 Task tool), v0.4 (자율 실행 — 새로 만든 워크플로우를 `goal.md` 파이프라인에 배치)

**기준 버전:** v0.5.0

**통합 출처:** 본 스펙은 이전 별도 트랙이었던 "스킬 분석기 (구 v0.3/v0.4)" + "스킬 자유도 (구 v0.6)"를 단일 v0.5로 합친 결과물. 두 트랙이 결국 같은 SKILL.md 프론트매터 스키마와 동일 라우팅 레이어를 공유하기 때문에 통합 시 표면적이 절반으로 줄어듦.

## 핵심 질문 & 성공 기준

| 핵심 질문 | 성공 기준 |
|---|---|
| 사용자가 코드를 안 만지고 메신저 대화만으로 새 skill·워크플로우를 등록할 수 있는가? | "이런 작업 자동화하고 싶다" → PM 명확화 1-2턴 → SKILL.md/workflow yaml draft → 승인 → 다음 호출부터 즉시 사용 가능 |
| 기존 `.claude/skills/` 자산을 잃지 않고 흡수할 수 있는가? | `solosquad analyze repo <path>` → 4-라벨 분류 보고서 → `add repo --from-report` 적용 → workspace agent와 충돌 없이 머지 |
| 자동 생성된 skill이 손으로 쓴 것과 동일 품질·동일 포맷인가? | Anthropic Agent Skills 공식 포맷 100% 호환, 라우터·PM이 양쪽을 구분 못 함, validator 통과 |
| 결정성이 깨지지 않는가? | 대화형 author 루프와 분석은 LLM 사용, `sync`·`migrate`·라우팅 적용은 LLM-free 유지 |

## 1. 현재 문제

| 문제 | 원인 |
|---|---|
| 에이전트 25개 고정 | 새 skill 추가 시 코드 수정 + AGENT_ROUTES 키워드 수동 등록 필요 |
| SKILL.md 포맷 암묵 | 필수/선택 필드 불명확, 검증 도구 없음 |

| Workflow 매칭 휴리스틱 없음 | 기존 skill 체인이 4종 템플릿 중 어디에 가까운지 자동 판정 부재 |
| Scanner가 `_` prefix 폴더 일괄 skip | `agents-builder.ts:76`이 `_teams/` 회피용으로 모든 `_*/` 폴더를 무시 — 메타-skill 위치(`_meta/workflow-maker/`)와 충돌 |
| 키워드 라우팅 정책 이중 정의 | AGENTS.md는 "scheduler·fallback용 유지", v0.5 §10은 "AGENT_ROUTES 코드 상수 삭제" — 두 문서 충돌 해소 필요 |
| Author 루프 비용 통제 부재 | 명확화 + draft + validate + dry-run sandbox = LLM N회 호출. 일/주 cap 정책 없음 — paperclip budget envelope 패턴 미적용 |
| Spec-gate 트리거 카테고리 누락 | 2026-05-13 트렌드 레코드가 Ralphathon spec-gate를 5번째 트리거 유형으로 명시(Anthropic 2025-12 stop-hook). triggers 4채널과의 직교 관계 명세 부재 |

## 2. 원칙

| 원칙 | 구현 |
|---|---|
| **메신저-네이티브 author 루프** | PM이 LLMREI(요구공학 LLM 면담) 스타일로 1-2턴 명확화 → draft 제시 → 사용자 확정. GUI 없이 메신저 thread만으로 완결 |
| **Anthropic Agent Skills 공식 포맷** | YAML frontmatter (`name`, `description`, optional `triggers`/`inputs`/`outputs`/`handoff_to`) + Markdown body. Progressive disclosure (본문은 짧게, 상세는 references 폴더) |
| **Stateless Skill / Stateful Agent** (보고서 §1.1 차용) | SKILL.md 자체는 stateless 매뉴얼 — frontmatter에 `stateful: false`가 기본값. state를 들고 있는 actor(PM session·routine run)는 `stateful: true`로 명시. author 루프가 새 SKILL을 만들 때 어느 쪽인지 결정 가이드 제공 |
| **Skill that authors skills** | Superpowers/Plugin Developer Toolkit 패턴 차용. 메타-skill(`assets/agents/_meta/workflow-maker/SKILL.md`)이 새 skill을 생성하는 책임 단일 보유 |
| **3-tier 검색 경로 + 4종 trigger 채널** | 프로젝트 로컬 `./.agents/` > 사용자 글로벌 `~/.solosquad/agents/` > 번들 `assets/agents/`. 4종 trigger 채널: 슬래시(`/`)·키워드 일치·대화 누적 빈도(Tier 2.D)·PM 명시 호출 |
| **Deterministic vs LLM 분리** | `sync`·`migrate`·라우팅 등록은 LLM 금지. `analyze`·메신저 author 루프만 LLM. 분류 결과 → `applier`가 결정적 머지 |
| **사람 승인 게이트** | author 루프 끝에 "확인하시겠습니까?"·`analyze` 끝에 Markdown 보고서. 자동 적용 없음 |
| **Migration by hash** | repo 분석 ledger에 `path + SHA256[:12]` 기록. 두 번째 실행은 신규/변경분만 LLM 호출 |
| **Just-in-Time Injection** (보고서 §1.2 원칙) | 라우터는 *매치되는* SKILL만 컨텍스트에 inject. 25개 + 사용자 추가 N개 전부 로드 금지. 빈도 자동 로드도 임계치 이상만 |

## 3. 두 흐름 = 같은 컴파일 타깃

```
                ┌─────────────────────────────────┐
사용자 대화 ──→ │  Workflow Maker (메타-skill)    │
                │  - 명확화 질문                  │
                │  - SKILL.md 생성                │
                │  - workflow yaml 생성           │
                └────────┬────────────────────────┘
                         │
                         ▼
                ┌─────────────────────────────────┐
                │  SKILL.md (Anthropic 포맷)      │
                │  + workflow.yaml                │  ← 단일 컴파일 타깃
                │  + goal.md (optional, v0.4)     │
                └────────▲───────┬────────────────┘
                         │       │
                         │       └──→ v0.4 goal-runner (자율 cycle)
                         │            ─ stop_condition 기반
                         │       ┌──→ v0.3 PM 모드 (인터랙티브)
                         │       │    ─ Task tool로 단계 호출
                ┌────────┴────────────────────────┐
기존 repo  ──→ │  Repo Analyzer                  │
.claude/      │  - 4-label 분류 (LLM)           │
skills/       │  - workflow 매칭 (결정적)       │
              │  - applier (결정적 머지)        │
              └─────────────────────────────────┘
```

**핵심**: author 루프와 analyzer는 *입력 형태*만 다르고 *출력 형태와 저장 경로*는 동일하다. 따라서 검증·라우팅·렌더링 로직은 단일.

**3개 컴파일 타깃의 분기 (v0.3·v0.4와의 정합):**
- `SKILL.md` — 모든 산출의 기본. 라우터에 등록되어 PM이 호출.
- `workflow.yaml` — 다단계 체인. v0.3 PM 모드가 인터랙티브로 진행 (`Task` tool 순차 호출).
- `goal.md` — *자율 cycle* (v0.4 goal-runner). author 루프 마지막에 "이 워크플로우를 자율 실행으로도 등록하시겠습니까?" 명시 게이트. yes 시 `goals/<id>/goal.md` + `AGENTS.md` modifiable_paths_override 자동 생성. v0.4 §3 파이프라인을 그대로 재사용 — v0.5는 *생성 게이트*만 추가.

## 4. SKILL.md 스키마 (Anthropic Agent Skills 호환)

```markdown
---
name: "Custom Analyst"
description: "도메인 특화 데이터 분석 — 부동산 등기부 신호 추출"
team: strategy                          # SoloSquad 확장 필드
stateful: false                         # 보고서 §1.1 — SKILL=stateless, actor=stateful
triggers:                               # 4종 채널 (slash/keyword/freq/explicit)
  slash: ["/analyze-realestate"]        # 명시적 의도 — 즉시 로드
  keyword: ["커스텀 분석", "custom analysis"]  # 메시지 1턴 매칭
  freq:                                 # 누적 빈도 자동 로드 (Tier 2.D)
    keywords: ["등기부", "신호", "부동산"]
    window_turns: 10                    # 최근 N턴 윈도우
    threshold: 3                        # 합산 출현 임계
    cooldown_turns: 6                   # 자동 로드 후 N턴 재발화 금지 (hysteresis)
  explicit: true                        # PM이 직접 호출 가능
loop_mode:                              # 선택. spec-gate(Ralph) 모드 명시 — triggers와 직교
  kind: spec-gate                       # null | spec-gate (v0.5에선 둘만 지원)
  spec_path: "<repo>/spec/realestate.md"
  stop_when: "all tests pass"           # v0.4 goal.md termination과 호환
budget:                                 # 선택. paperclip envelope 차용. author 루프 비용 cap
  per_call_usd: 1.5                     # 한 번 호출당 상한 — 초과 시 PM 거부
  daily_usd: 10                         # 일일 누적 상한
inputs:
  required: [data_source, analysis_goal]
  optional: [time_range]
outputs:
  - custom_report.md
handoff_to:                             # 다음 단계 후보
  - feature-planner
  - pmf-planner
scope: agent                            # workspace | org | repo
confidence: 1.0                         # author 루프 산출 시 1.0, analyzer 시 0.x
source: messenger-author-2026-05-12     # author 출처 추적
---

# Custom Analyst Agent

> 1-2 단락 정체성 정의.

## Process
...
```

**필수 필드:** `name`, `description` (Anthropic 사양). 나머지는 SoloSquad 확장이며 누락 시 라우터가 fallback 처리.

**`stateful` 필드 의미:** false면 매 호출이 fresh session(예: 25개 specialist SKILL). true면 actor가 state를 들고 다님(예: PM session, long-running routine). author 루프가 새 SKILL 생성 시 "이건 매 호출이 독립적인가?"를 물어 자동 추론.

**v0.5에선 `stateful: true` *생성*은 out of scope (§12)**: 현재 stateful actor는 (a) PM(`orchestrator/SKILL.md` — 단일 인스턴스, 워크스페이스 부팅 시 결정), (b) v0.4 goal-runner(background session) 두 가지뿐. 사용자가 새 stateful SKILL을 author 루프로 만들면 어느 actor가 들고 운영할지 명세가 없음 — 이는 v0.6 trajectory 추출과 함께 결정. v0.5의 새 SKILL은 모두 `stateful: false`로 강제 (validator가 거부).

**`loop_mode` 필드 의미:** spec-gate 트리거는 *컨텍스트 inject* 신호가 아니라 *루프 종료 조건*이라 `triggers`와 직교. `loop_mode.kind: spec-gate`이면 author 루프가 자동으로 `goal.md` 컴파일 타깃을 추가 생성하고 v0.4 goal-runner에 등록 (§3 다이어그램 분기 참조). v0.5 본문 §7 라우터는 spec-gate SKILL을 keyword/explicit 채널로 평소처럼 발견하고, 실행 단계에서 goal-runner로 넘김.

**`budget` 필드 의미:** paperclip OSS의 budget envelope 패턴 차용 (2026-05-13 트렌드 레코드 권고). author 루프 자체(`workspace.yaml` 글로벌)와 개별 SKILL(`SKILL.md` frontmatter) 두 계층. 개별 SKILL은 dry-run sandbox·explicit 호출에서 cap을 forward. 글로벌 cap은 author 루프의 명확화·draft·validate 합산. 도달 시 PM이 "Daily budget 도달 — 내일 다시 시도해주세요" 응답.

**조직별 색채는 frontmatter에 넣지 않는다 (v0.6 §2.2와 정합)**: 톤·강조점·금지사항 등 *조직 색채*는 SKILL.md frontmatter나 본문에 박지 않는다. 같은 specialist를 여러 org에서 쓸 때 SKILL이 한 org에 락인되면 안 됨. 색채는 v0.6 §2.2에서 도입할 **`<org>/agent-profile.yaml`** 의 modifier 섹션으로 흡수한다. author 루프가 새 SKILL을 만들 때 사용자가 "이 옵션은 조직마다 다른가?" 라고 물으면 → agent-profile로 분기, "보편 craft인가?" → SKILL.md.

**triggers 4종 채널 의미:**
- `slash` — gstack 패턴. 사용자가 명시 의도 표현 시 즉시 로드 (모호성 없음)
- `keyword` — 기존 v0.2.4 키워드 매칭. 단일 메시지 1턴
- `freq` — 보고서 §1.2 JIT Injection 원칙 + Baseline phuryn auto-load 융합. 최근 N턴 누적 빈도가 임계 초과 시 자동 로드 (시맨틱 임베딩 없이 카운팅으로 동등 효과)
- `explicit` — PM이 추론으로 직접 호출

라우터는 우선순위 `slash > explicit > keyword > freq` 순으로 매칭. 동일 trigger 채널 내에서는 3-tier 검색 경로 적용.

**Progressive disclosure:** 본문은 짧게 유지, 상세 절차는 같은 폴더의 `references/*.md`로 분리. Anthropic Skill Authoring Best Practices 준수.

## 5. 메신저 Author 루프 (LLMREI 스타일)

### 5.1 트리거

사용자: "owner-command에 '이런 작업 자주 하니까 자동화하고 싶다 — 매주 등기부 알림 보고서 만들어줘'"

PM이 의도를 "신규 skill 또는 workflow 생성"으로 분류 → meta-skill 로드.

### 5.2 명확화 (1-2턴)

```
PM: 이 작업을 자동화하려고 합니다. 몇 가지만 확인할게요.
    1. 입력은 무엇인가요? (예: 등기부 API 응답 / 수동 입력 데이터 / 다른 에이전트 출력)
    2. 출력 형태는? (예: 메신저 알림 / Markdown 리포트 / JSONL)
    3. 매주 자동 실행되어야 하나요, 아니면 사용자가 트리거해야 하나요?

사용자: 1) 등기부 OpenAPI 응답, 2) 메신저 알림 + Markdown, 3) 매주 월요일 자동
```

명확화는 최대 2턴. 사용자가 답 못 하면 default 채택 + 추후 수정 가능 명시.

### 5.3 Draft 제시 → 확정

PM이 SKILL.md + (선택적으로) workflow.yaml + (선택적으로) 라우틴 cron entry draft를 thread에 표시.

```
PM: 아래로 생성하려고 합니다. 확정할까요?

    [SKILL.md preview]
    ---
    name: Real Estate Watcher
    triggers: [등기부 알림, 부동산 모니터링]
    ...
    
    [routine entry]
    cron: "0 9 * * MON"  # 매주 월요일 09:00
    
    저장 위치: org/<org>/agents/data/real-estate-watcher/SKILL.md
    
    승인 (y/N):
```

사용자 `y` → applier가 결정적으로 파일 저장 + 라우터 reload + 메모리 시드.

### 5.4 검증

저장 직후 `solosquad agent validate <path>` 자동 실행 — frontmatter 스키마, 필수 섹션, 충돌 검사. 실패 시 PM이 사용자에게 보고 + rollback 옵션 제시.

### 5.5 Dry-run sandbox (선택)

Claude Agent SDK로 새 skill을 격리 세션에서 1회 호출 → 출력 미리보기 → 사용자가 본 출력 확인 후 production 등록. 대화 thread 추가 1턴.

**Sandbox 입력 프로토콜:**
1. PM이 사용자에게 "샘플 입력 1건을 보내주세요 (실제 운영에서 받을 메시지와 비슷한 형태)" 요청 — 추가 1턴.
2. 입력이 SKILL frontmatter `inputs.required`를 만족하는지 결정적 검증. 부족 시 PM이 누락 필드 안내.
3. 격리 세션 호출 — `budget.per_call_usd` cap 적용, timeout 60초.
4. 출력 첫 500자(또는 토큰 1k 중 작은 쪽) 사용자 thread에 표시. 출력이 더 길면 `<org>/workflows/_drafts/<skill>-dryrun-<ts>.md`에 전체 저장 후 경로 안내.

## 7. 3-tier 검색 경로 + 4채널 라우팅

```
1. <workspace>/<org>/.agents/{team}/{agent}/SKILL.md   → org 로컬 (최우선)
2. ~/.solosquad/agents/{team}/{agent}/SKILL.md          → 사용자 글로벌
3. <workspace>/.solosquad/agents/{team}/{agent}/SKILL.md → 워크스페이스 번들 (init 시 복사된 기본 25개)
```

같은 이름이면 상위 계층 override. `AGENT_ROUTES` 하드코딩 제거 → `buildRoutes()`가 시작 시 3-tier 스캔 + frontmatter `triggers` 파싱 (slash/keyword/freq/explicit 4채널).

**1순위 tier 위치 명확화:** "프로젝트 로컬"은 *SoloSquad 워크스페이스 안의 org 폴더*(`<workspace>/<org>/.agents/`)를 의미. *사용자 product repo*(`<workspace>/<org>/repositories/<repo>/`)에는 v0.2 이래 `.claude/skills/`가 codebase-fact용으로 따로 존재 — 이건 *라우터 검색 대상이 아님* (`codebase-fact` 라벨대로 repo에 머묾, Task tool로 spawn 시 자동 cwd 컨텍스트로 로드되는 별개 경로).

**`AGENT_ROUTES` 정책 결정:** *완전 삭제* 채택. scheduler 라우틴(`background-routines.*`)은 v0.2.4 이래 *프롬프트 문자열에 agent명 명시* 방식이라 라우터 키워드 매칭이 사실상 비활성. v0.5에선 (a) `src/bot/agent-router.ts`의 `AGENT_ROUTES` 상수 완전 제거, (b) `findAgent()` 시그니처 유지하고 내부를 `buildRoutes()` 결과 조회로 교체, (c) **AGENTS.md L131 "Legacy keyword routing is retained..." 문구 동시 업데이트** — 본 v0.5 PR에 묶어서 인간 검토(human-edited only 규약). 검토 완료 전까지는 두 문서 일시 불일치 허용, PR 머지 시 동기 머지.

**메타-skill scanner 분리:** `_meta/` 폴더는 일반 scanner(`listSourceAgents()`, `agents-builder.ts:74`)에서 계속 skip. 별도 `listMetaSkills()` 함수 신설 — `assets/agents/_meta/*/SKILL.md`만 스캔, frontmatter `triggers.explicit: true`만 인정, slash/keyword/freq 채널 등록 거부. `workflow-maker` 메타-skill은 PM이 `[META workflow-maker]` explicit 호출로만 진입. v0.6 §2.1에서 `_teams/` 폴더가 폐지되면 `_` prefix 회피 규약 자체가 단순해짐 — 그때 다시 정리.

```typescript
// src/bot/agent-router.ts (재작성)
type TriggerChannel = "slash" | "keyword" | "freq" | "explicit";

export async function buildRoutes(): Promise<RouteIndex> {
  const idx: RouteIndex = { slash: {}, keyword: {}, freq: [], explicit: {} };
  for (const tier of [bundleRoot, userGlobal, projectLocal]) {
    for (const skill of await scanSkills(tier)) {
      const fm = parseFrontmatter(skill);
      const ref = { team: fm.team, agent: fm.name, source: tier, stateful: fm.stateful ?? false };
      for (const t of fm.triggers?.slash ?? [])    idx.slash[t]   = ref;
      for (const t of fm.triggers?.keyword ?? []) idx.keyword[t] = ref;
      if (fm.triggers?.freq) idx.freq.push({ ref, ...fm.triggers.freq });
      if (fm.triggers?.explicit) idx.explicit[fm.name] = ref;
    }
  }
  return idx;
}

// 매 메시지 처리 — 우선순위 slash > explicit > keyword > freq
export function resolve(message: string, history: Turn[], idx: RouteIndex): AgentRef | null {
  // 1. slash 우선
  const slashHit = parseSlash(message);
  if (slashHit && idx.slash[slashHit]) return idx.slash[slashHit];
  // 2. 키워드 1턴 매칭
  for (const [kw, ref] of Object.entries(idx.keyword)) {
    if (message.includes(kw)) return ref;
  }
  // 3. 빈도 자동 로드 (Tier 2.D) — 시맨틱 임베딩 없이 카운팅
  const recent = history.slice(-Math.max(...idx.freq.map(f => f.window_turns)));
  for (const f of idx.freq) {
    const window = recent.slice(-f.window_turns).map(t => t.text).join(" ");
    const score = f.keywords.reduce((n, kw) => n + (window.match(new RegExp(kw, "g"))?.length ?? 0), 0);
    if (score >= f.threshold) return f.ref;
  }
  return null;  // PM이 explicit으로 직접 호출하거나 fallback
}
```

**자동 로드 통지 (Just-in-Time 원칙 + UX 명확성)**: `freq` 채널로 로드된 SKILL은 사용자에게 1회 알림 — `🧠 {agent} 컨텍스트 자동 로드됨 (최근 키워드: ...)`. 사용자가 의도하지 않은 라우팅을 즉시 거부할 수 있음.

**Freq 채널 hysteresis (cooldown):** 한 번 자동 로드된 SKILL은 frontmatter `triggers.freq.cooldown_turns` (기본 6) 동안 같은 SKILL의 freq 매칭 score 계산을 건너뜀. 사용자가 같은 주제로 길게 대화할 때 같은 SKILL이 매 턴 재발화하는 잡음 회피. 라우터 메모리 구조: `<org>/.solosquad/sessions/<user>.json`에 `freq_cooldowns: { skill_name: until_turn }` 필드 추가. session-store 1줄 변경.

**Freq 채널 성능 한도:** freq 등록 SKILL이 증가하면 매 메시지 정규식 매칭 비용 증가 — workspace당 freq 등록 SKILL 상한 **20개** (§13 측정 항목). validator가 20번째 등록 시도부터 거부. 키워드 토큰화 후 hashmap 카운터 구현으로 매칭을 O(|tokens|)에서 끝내고 정규식은 fallback.

**Hot-reload (atomic swap):** meta-skill이 새 SKILL.md를 저장하면 `bot/index.ts`가 라우터를 재빌드. 재빌드 중 도착한 메시지는 *이전 인덱스*로 처리. 빌드 완료 후 ref 교체(`routeIndexRef = newIndex`) — 빌드 중 도달한 메시지가 부분 인덱스에 닿지 않음. 봇 재시작 불필요. 빌드 실패 시 이전 인덱스 유지 + PM에 경고 메시지.

**Spec-gate 트리거의 직교성:** §4 `loop_mode.kind: spec-gate` SKILL은 *triggers 4채널*과 무관 — 라우터는 keyword/explicit 채널로 *spawn 진입*만 결정, 실행 단계에서 `loop_mode`를 보고 v0.4 goal-runner로 위임(spec-gate)할지 일반 Task call(null)을 할지 분기. Anthropic 2025-12 stop-hook 플러그인은 v0.4 goal-runner의 `stop_condition` 위에 *추가 종료 신호*로 적층 — v0.5에선 hook 인터페이스 안정화를 기다리지 않고 `loop_mode.stop_when` 텍스트만 명세, 실제 stop-hook 연동은 v0.6 후속.

**v0.3 슬래시 5종과의 관계**: v0.3에서 도입한 PM 단계 슬래시(`/think /plan /build /review /ship`)는 PM SKILL 본문에 인식 규칙이 있고, v0.5의 `triggers.slash`는 *개별 SKILL 자체로 라우팅*하는 별개 진입점이다. 두 슬래시 공간은 prefix로 구분(PM 단계는 5개 고정, 사용자 정의는 자유 — 충돌 시 검증 오류로 거부).

분류 안정성: 같은 fixture 100회 분류 시 라벨 일치율 ≥ 95%. 모델 업그레이드 시 fixture 재실행 + diff 리포트.

### 11.2 Author 루프 E2E

- PM이 단일 메신저 메시지 1건 → 명확화 1턴 → draft → 확정 → SKILL.md 저장 → 라우터에서 발견됨
- Validator 실패 시 rollback
- Dry-run sandbox 실행 + 출력이 사용자 thread에 표시
- Hot-reload — 봇 재시작 없이 새 trigger가 라우터에 잡힘

### 11.3 회귀

- `analyze` 두 번째 실행 시 LLM 호출 0회 (ledger 정확성)
- `add repo --from-report` 적용 후 `applied: true`
- `migrate`·`sync`에는 LLM 호출 없음 (deterministic 보장)

### 11.4 외부 코퍼스 cross-validation

- **Fixture**: `anthropics/skills` 공개 레포 N개 SKILL.md (npm test 첫 실행 시 fetch & 캐시 → `test/.cache/anthropic-skills/`, 이후 재사용)
- 케이스 1 — 파싱: 각 SKILL.md를 v0.5 `skill-parser.ts`로 frontmatter 파싱 → 에러 0건
- 케이스 2 — Round-trip: 파싱 → 재출력(`writeFrontmatter()`) → 원본과 비교 → bytes 일치
- 케이스 3 — Anthropic 필수 필드 (`name`, `description`)만 있고 SoloSquad 확장 필드는 부재일 때 → validator가 통과 + fallback 동작

### 11.5 v0.5 신규 회귀

- **Slash 충돌**: `triggers.slash: ["/think"]` (PM 단계 슬래시 5종 중) 등록 시도 → validator 거부 + 명확한 에러 메시지
- **Slash prefix 충돌**: `triggers.slash: ["/thinker"]` (PM 슬래시의 prefix) 도 거부 (정책 확정 — 5종 슬래시 *완전 일치* 외에는 등록 가능)
- **Hot-reload 동시성**: 라우터 재빌드 중 N건 메시지 도달 → 전부 이전 인덱스로 처리 → 빌드 완료 후 차후 메시지부터 새 인덱스
- **Freq cooldown**: 동일 freq SKILL이 cooldown_turns 동안 재발화 안 됨
- **Freq SKILL 상한**: 21번째 freq 등록 시도 → validator 거부 (워크스페이스 20개 상한)
- **Author budget**: 일일 cap 도달 → 다음 author 호출 거부 + 로그 기록. cap reset(자정) 후 정상 동작
- **Meta-skill scanner 분리**: `_meta/` 폴더의 SKILL은 일반 라우터에 등록되지 않음 + explicit 호출만 허용
