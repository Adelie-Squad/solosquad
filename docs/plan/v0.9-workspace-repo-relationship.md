# v0.9 — Workspace ↔ Repository Relationship Redesign

> 솔로 사용자가 **여러 product, 여러 org, 여러 repository**를 갖는다는 SoloSquad의 1차 시나리오를 고려할 때, 현재 *repos-inside-workspace-tree* 모델이 *과도하게 제약*한다는 것을 v0.8.5~v0.8.6 사용자 테스트에서 발견. v0.9는 **repository 위치를 워크스페이스 밖으로 해방** + Hermes/Codex 등 peer agent 모델 비교를 통한 *direct vs sandbox* 결정 박제. 본 patch는 **plan only — 코드 변경 0건**. 분석 + 후보 4종 비교 + 추천 박제. 구현은 v0.9.x patches에서.

**로드맵:** v0.8.7 Tiny Stabilization → **v0.9 Workspace ↔ Repo 재설계** → v1.0 정식 출시

**의존:** v0.7 class A 불가침 룰 (repositories/ 사용자 코드 어떤 명령으로도 변형 X), v0.8.3 `add repo --dry-run` (외부 절대경로·심링크·IDE config drift 감지)

**기준 버전:** v0.9. **plan 박제 patch** — 코드 0건, 문서만. 구현은 후속 v0.9.x.

**분류:** Architecture redesign. 사용자 노출 표면적 변화 *없음* (본 patch). 구현 patch들이 표면적 추가.

## 1. 현재 모델 (v0.8.7 기준) — *repos-inside-workspace-tree* 강제

### 1.1 디스크 레이아웃 가정

```
<workspace>/                    ← 사용자가 mkdir로 만든 디렉터리 (위치 자유, v0.8.4)
├── .solosquad/                 ← 시스템 메타
└── <org>/                      ← 사업 단위
    └── repositories/           ← *repo는 여기 안에 있어야 함*
        └── <repo>/             ← 사용자 코드 (.git 포함)
            ├── .git/
            ├── .solosquad/repo.yaml
            └── (소스코드)
```

### 1.2 코드에서 *강제되는 위치*

`src/util/paths.ts:132-164`:
```typescript
export function getRepositoriesDir(orgSlug, workspace?) {
  return path.join(getOrgDir(orgSlug, workspace), "repositories");
}

export function getRepoDir(orgSlug, repoSlug, workspace?) {
  return path.join(getRepositoriesDir(orgSlug, workspace), repoSlug);
}

export function resolveRepoCwd(orgSlug, repoSlug, workspace?) {
  const canonical = path.join(root, orgSlug, "repositories", repoSlug);
  if (fs.existsSync(canonical)) return canonical;
  // ... legacy fallback to org root
}
```

→ **repo가 `<workspace>/<org>/repositories/<repo>/`에 *물리적으로 존재*해야** spawn 시 cwd 해석 성공. 다른 곳에 있으면 *resolveRepoCwd가 못 찾음*.

### 1.3 `solosquad add repo`의 동작 (v0.8.3)

사용자가 외부 경로의 repo를 등록하면:
- **이동 (default)**: `<external>/foo` → `<workspace>/<org>/repositories/foo/` 물리적 이동
- **복사 (`--keep-original`)**: 원본 유지하고 워크스페이스 트리로 복사
- **dry-run (`--dry-run`)**: 위 위험 5종 미리 알림 (IDE lock, 심링크, 외부 절대경로, slug 충돌, 활성 프로세스)

→ *어떤 옵션이든* repo의 *카피*가 `<workspace>/<org>/repositories/<repo>/`에 있어야 함. *외부 경로의 repo를 그 자리에서 다루는 옵션이 없음*.

## 2. 현재 모델의 한계 — 사용자 시나리오에서 드러난 마찰

### 2.1 시나리오 A — 솔로 founder, *이미 dev 환경 있음*

```
~/Dev/                              ← 사용자가 이미 dev tree로 사용 중
├── my-saas/                        ← VSCode workspace, GitHub repo
├── side-project/                   ← 별도 GitHub repo
├── consulting-client-X/            ← 외부 드라이브 마운트, 클라이언트 코드
└── ...
```

여기에 SoloSquad 도입하려면:
- **현재 모델**: `~/solosquad-workspace/<org>/repositories/`로 *위 repo들을 모두 이동*. VSCode workspace 절대경로 깨짐, CI runner 경로 깨짐, 외부 cron 깨짐, npm link 깨짐, 익숙한 디렉터리 구조 변경
- **사용자 마찰**: *기존 dev 환경 전체 reorganize*. 진입 장벽 매우 높음

### 2.2 시나리오 B — *드라이브 경계 넘는* repo

```
C:\Dev\my-saas\                     ← 메인 드라이브
D:\Backup\old-product\              ← 백업 드라이브 (저장 공간 분리)
\\server\client-X\                  ← 네트워크 드라이브 (회사 컴퓨터)
```

- **현재 모델**: 단일 `<workspace>` 트리에 모두 들어가야 함 → *드라이브 경계 무시 못 함*. Windows의 다른 letter drive, mounted volume, network share 등은 *물리적 이동 불가능*
- **결과**: SoloSquad가 *이런 repo는 다룰 수 없음*

### 2.3 시나리오 C — *n잡 founder*, 워크스페이스 여러 개

```
~/solopreneur/                      ← 본업 워크스페이스 (Slack 봇)
├── .solosquad/
└── personal-saas/
    └── repositories/
        └── my-saas/                ← 여기에 *복사본*
~/personal-lab/                     ← 사이드 워크스페이스 (Discord 봇)
└── (다른 repo들)

~/Dev/my-saas/                      ← *원본* (여기서 개발 계속)
```

- 현재 모델로 *워크스페이스에 repo 추가* 시 원본과 복사본이 *둘 다 생김*. *어느 게 진실*인지 혼란
- 사용자가 ~/Dev/my-saas에서 commit → ~/solopreneur/.../my-saas에는 안 반영
- 또는 SoloSquad가 ~/solopreneur/.../my-saas에서 commit → ~/Dev/my-saas와 divergence

### 2.4 시나리오 D — *SoloSquad 자체 코드 작업* (재귀)

본 SoloSquad repo (`C:\Dev\solo-founder-agents`)에 SoloSquad를 적용하면:
- `solosquad init`을 어디서 실행할지? — SoloSquad 코드 트리 *안*에서? 그러면 `solosquad-workspace/<org>/repositories/solo-founder-agents/` 가 *자기 자신의 위치*
- 또는 *밖*에서? 그러면 SoloSquad가 자기 코드를 다른 트리로 *옮겨야* 함
- *재귀 / 자기참조* 문제. 현재 모델로 해결 불가

## 3. Peer agent 모델 비교 — Hermes, OpenAI Codex 등

### 3.1 Hermes Agent (사용자 참조 자료)

**Hermes는 *완전 격리 sandbox에 repo clone***:
1. Docker container 또는 WSL 격리 workspace 생성
2. `git clone <remote-url>` → sandbox 내부에 clone
3. `npm install` / `pip install` 등 의존성 설치
4. 코드 작업 (ls/tree/grep + edit + npm test 자가 디버깅 루프)
5. `git commit` + `git push` → 원격에 반영
6. MCP 또는 gh API로 PR 생성

→ **원본 사용자 repo는 *전혀 안 건드림***. 에이전트는 *원격에서 clone*해서 *자기 카피*로 작업하고 *원격에 push*해서 반영. 사용자는 *PR 받아서* 자기 working tree에 pull.

### 3.2 OpenAI Codex `/goal`

- 워크스페이스 안에서 *직접* 작업 (Codex CLI의 cwd 안)
- AGENTS.md (cross-tool 표준) + goal.md
- 자동 git commit, 사용자 명시 승인 후 머지
- Hermes보다 *덜 격리* — 사용자 working tree 직접 수정 (단, dev-confirm gate)

### 3.3 GitHub Copilot Workspace

- *클라우드*에서 sandbox 생성
- repo는 GitHub 원격에서 자동 fetch
- 사용자 로컬 working tree는 안 건드림
- 결과를 PR로만 사용자에게 노출

### 3.4 SoloSquad (현재)

- 사용자 워크스페이스 트리 *내부*에 repo가 있어야 함
- 그 repo의 working tree를 *직접* 수정 (sandbox X, isolation X)
- dev-confirm gate로 sensitive op만 차단

→ SoloSquad는 *Codex와 비슷한 direct 모델*인데, *repo 위치 제약*만 더 엄격. *최악의 조합*: direct working-tree 작업의 위험성 + 위치 제약의 마찰.

### 3.5 모델 비교 표

| | Hermes | Codex `/goal` | Copilot Workspace | SoloSquad (현재) |
|---|---|---|---|---|
| repo 위치 | sandbox 내부 (clone) | 사용자 cwd | 클라우드 (자동 fetch) | 워크스페이스 트리 내부 |
| 사용자 working tree | 안 건드림 | 직접 수정 (dev-confirm 게이트) | 안 건드림 | 직접 수정 (dev-confirm) |
| 격리 | 강 (Docker/WSL) | 약 (사용자 OS 직접) | 강 (cloud sandbox) | 약 (사용자 OS) |
| 다중 product 지원 | ✓ (sandbox 별) | ✓ (cwd 별) | ✓ (PR 별) | ✗ (단일 트리 강제) |
| 드라이브 경계 넘기 | ✓ (remote clone) | ✓ (cwd 자유) | ✓ (cloud) | ✗ (단일 트리) |
| 오프라인 작업 | △ (clone 필요) | ✓ | ✗ | ✓ (트리에 있으면) |
| 원본 commit과 sync | △ (push 충돌 가능) | ✓ (실시간) | ✗ (PR로만) | ✓ (실시간) |

## 4. 후보 모델 4종

### 모델 A — *현재* (move-into-workspace)

- 사용자가 `solosquad add repo <path>` 시 *물리적 이동* 또는 *복사*
- repo가 `<workspace>/<org>/repositories/<repo>/`에 *존재*해야 동작
- spawn cwd = 이 경로

**유지의 이유**: 단일 트리 = 단일 진실. cwd 해석 간단. `.gitignore` 깔끔 (`.solosquad/` 한 줄로 모든 시스템 자산 제외).

**버려야 할 이유**: §2 시나리오 A·B·C·D 모두 미해결.

### 모델 B — *Path reference* (외부 경로 참조)

- `repo.yaml.path: /absolute/path/to/repo` 필드 신설
- `<workspace>/<org>/repositories/<repo>/` 디렉터리는 *존재 안 함* (또는 metadata-only)
- spawn 시 `resolveRepoCwd`가 `repo.yaml.path` 읽어 외부 경로로 cwd 결정
- `add repo --path <external>`로 등록 — 이동·복사 X, 단순 reference

**장점**:
- 사용자 dev tree 무변형 (§2 A 해결)
- 드라이브 경계 자유 (§2 B 해결)
- 워크스페이스 여러 개여도 같은 repo path 공유 가능 (§2 C 해결)
- 재귀 시나리오 OK (§2 D 해결)

**단점**:
- 사용자가 외부에서 repo *이동·삭제*하면 SoloSquad는 못 따라감 → repo.yaml의 path가 stale. `solosquad doctor` 또는 spawn 시점 검증 필요
- Windows의 드라이브 letter / UNC path edge case
- `.gitignore` 같은 SoloSquad 메타가 *외부 repo*에는 안 놓이므로 `.solosquad/repo.yaml`을 *어디 둘지* 결정 필요 (옵션: 외부 repo 안에 `.solosquad/repo.yaml` 두기, 또는 워크스페이스 안에 `<workspace>/<org>/repositories/<repo>.yaml`만 두고 path field로 link)

### 모델 C — *Sandbox clone* (Hermes 방식)

- `solosquad add repo <git-url>` 또는 사용자 working tree에서 `add repo --remote`
- SoloSquad가 *별도 sandbox*에 clone (`<workspace>/.solosquad/sandboxes/<org>/<repo>/`)
- 에이전트는 sandbox 안에서 작업 → push → PR
- 사용자 working tree와 *완전 격리*

**장점**:
- 사용자 working tree 완전 안전
- 에이전트가 망쳐도 격리됨
- v0.8.2 dev_capability의 *진짜 isolation* — push 게이트가 *마지막 방어선*이 아니라 *추가 안전망*

**단점**:
- 디스크 × 2 (사용자 + sandbox)
- 사용자가 working tree에서 commit 한 게 sandbox에 *수동 sync* 필요 (`git pull`)
- 또는 SoloSquad가 자동 sync — *git fetch 빈도·충돌 해결 정책* 필요
- 오프라인 / 네트워크 끊김 시 첫 clone 불가
- *솔로 사용자 1인 시나리오*에 과한 isolation일 수 있음

### 모델 D — *Hybrid* (모드 선택)

- `repo.yaml.mode: reference | clone | move` (default = `reference`)
- 사용자가 등록 시 선택:
  - `add repo --path <ext>` → reference (모델 B)
  - `add repo --remote <git-url>` → clone (모델 C)
  - `add repo --move <ext>` → move (모델 A, 현재 default)

**장점**: 사용자 선택권. 모드별 trade-off 명시.

**단점**: surface area ↑. 코드 분기 복잡도 ↑. 어느 게 default인지 문서 필요.

## 6. 워크스페이스 위치 문제 (이미 v0.8.4에서 해결됨)

사용자 질문: *"솔로스쿼드 워크스페이스는 ~/ 경로가 아니라 사용자가 원하는 디렉토리를 지정할 수 있는데 뭐가 나을지"*

v0.8.4 §8에서 이미 결정 박제됨:
- `solosquad init` default = cwd
- 부모에 `.solosquad/` 있을 때만 3-way prompt (cwd / existing / custom)
- 사용자가 원하는 디렉터리 위치 = 본인 책임

→ **워크스페이스 위치 자체는 자유**. *어디에 만들지 SoloSquad가 정해주지 않음*. 본 v0.9의 진짜 이슈는 *repo 위치*.

→ 두 결정의 분리:
- 워크스페이스 위치 = 사용자 자유 (v0.8.4 결정)
- repo 위치 = 워크스페이스와 *분리 가능해야 함* (본 v0.9 추천)

## 7. 추천 — *모델 B (path reference) as default, 모델 A 유지 (--move opt-in)*

### 7.1 추천 사유

- **솔로 사용자 가치**: §2의 A·B·C·D 시나리오 모두 해결. 진입 장벽 ↓
- **코드 복잡도**: 중간. `resolveRepoCwd`에 외부 경로 분기 1건 + `repo.yaml.path` 필드 추가 + `add repo --path` flow 1건
- **backward-compat**: 현재 `<workspace>/<org>/repositories/<repo>/` 트리 가진 사용자는 *그대로 동작* (`resolveRepoCwd` legacy path 분기 유지)
- **Hermes 방식 (모델 C) skip 사유**: 솔로 사용자 1인 시나리오는 *direct working-tree 작업이 자연스러움*. 사용자가 IDE에서 같은 파일 보고 있고, 에이전트 commit이 즉시 git log에 보임. *teammate metaphor*에 맞음. Sandbox isolation은 *multi-user enterprise* 시나리오에 더 맞고, 우리 target = 솔로
- **모델 D (hybrid) skip 사유**: 3 모드 학습 곡선 ↑. 솔로 사용자가 default 하나로 잘 살면 됨. 필요해지면 v1.x에서 mode 추가

### 7.2 결정 후 디스크 레이아웃

**v0.9.x 신규 (모델 B)**:
```
~/Dev/                                  ← 사용자 dev tree, 그대로
├── my-saas/                            ← 그대로 (자기 .git, 자기 IDE config)
│   └── .solosquad/repo.yaml            ← *추가됨* (1개 파일, class A* 룰)
└── side-project/
    └── .solosquad/repo.yaml

C:\Dev\bv-ai-native-po\                 ← workspace (위치 자유)
├── .solosquad/                         ← 시스템 메타
└── personal/                           ← org
    ├── .org.yaml
    ├── .solosquad/users/<handle>.yaml
    ├── memory/
    ├── workflows/
    └── repositories.yaml               ← *외부 repo 매핑* (신규 metadata)
                                         # 또는 repositories/<repo>.yaml (per-repo metadata)

```

또는 더 간결: `<workspace>/<org>/repositories/<repo>.yaml` (디렉터리 아닌 파일) ← path field로 외부 가리킴.

### 7.3 기존 사용자 backward-compat

- 현재 `<workspace>/<org>/repositories/<repo>/` 트리 있는 사용자 → *그대로 동작*. resolveRepoCwd legacy 분기 유지
- 마이그레이션은 *opt-in* (`solosquad migrate --externalize-repos`) — 명시적 명령으로만 외부화 (자동 이동 안 함)
- v1.0에서도 두 레이아웃 *영구 공존*. user code 불가침 룰이 이를 강제 (`<workspace>/<org>/repositories/<repo>/` 트리도 *class A 사용자 코드*라 SoloSquad가 자동 이동 안 함)

### 7.4 안전망 — `solosquad doctor` 확장 (v0.9.1)

- repo.yaml.path가 가리키는 외부 경로 *존재 검증*
- 외부 경로에 `.git/` 있는지 검증
- 외부 경로 working tree dirty인 경우 *경고만* (작업 시 충돌 가능)
- v0.7 class A 룰 영구 보존 — 어떤 명령도 외부 경로 코드 변형 X (현재와 동일)

## 12. Hermes 비교 — *왜 우리는 모델 C 안 가는가* 정리 박제

사용자가 참조한 Hermes Agent 흐름은 *완전 격리 sandbox + clone-based 작업*. 우리가 *모델 C를 v0.9 default에서 skip한 이유*를 영구 박제:

| Hermes 가정 | SoloSquad 가정 |
|---|---|
| 에이전트가 *원격 환경*에서 동작 (Docker/WSL/cloud) | 에이전트가 *사용자 같은 머신*에서 동작 (Claude Code 호출) |

| PR이 사용자가 받는 첫 산출물 | 사용자가 *작업 진행을 실시간 봄*. PR은 *push 후* 형식 |

→ 모델 C는 *SoloSquad가 multi-user / cloud 운영으로 진화할 때* (v2.x?) 다시 검토할 슬롯. v0.9에서는 *모델 B (path reference)*가 솔로 founder에 충분.

## 13. 자동화 UX 흐름 (v0.9.1 구현 시 default 박제)

모델 B에서 *path-reference 등록*을 사용자가 어떻게 트리거할지 — 4가지 진입점 박제. **default = cwd 인식**.

### 13.1 cwd 인식 — *가장 자연스러운 default*

사용자가 자기 repo로 `cd` 한 다음 인자 없이:
```bash
cd ~/Dev/my-saas
solosquad add repo
```

→ SoloSquad 자동 감지:
```
✓ Detected git repo at cwd: ~/Dev/my-saas
  remote: origin → https://github.com/me/my-saas.git
  language: TypeScript (package.json)
  size: 142 MB
  uncommitted changes: 0

Register as path-reference to org "personal"? [Y/n] y
Role [main/frontend/backend/data/...]: main

✓ Created: <workspace>/personal/repositories/my-saas.yaml
  path: /Users/me/Dev/my-saas (absolute)
  slug: my-saas
  remote_url: https://github.com/me/my-saas.git
  role: main
```

자동 추출:
- 경로 = cwd 절대 경로
- slug = 폴더명
- remote_url = `git remote get-url origin`
- language = `package.json`/`pyproject.toml`/`go.mod`/`Cargo.toml` 감지
- role default = `.git` 있으면 `main`

**사용자 인지 부담 0** — 평소 dev 디렉터리에서 명령만 치면 됨.

### 13.2 명시적 path flag

```bash
solosquad add repo --path ~/Dev/my-saas --org personal --role main
```

스크립트 자동화·CI에 좋음. `--dry-run`과 조합 가능 (실제 등록 전 검증만).

### 13.3 `solosquad init` Step 5.1 확장

마법사가 *처음부터* 외부 repo 등록 받음:
```
-- Step 5.1: Register Repositories (optional) --
You can paste a git URL to clone, or a local path to register.
Leave empty to skip. Repeat to add more.

Add repo to personal (git URL or local path, blank to finish):
> ~/Dev/my-saas

✓ Detected at ~/Dev/my-saas (git repo, origin: github.com/me/my-saas)
  Register mode:
  > Path reference (recommended — no move, no copy)
    Move into workspace (legacy)
    Copy into workspace
  
Role: main
✓ Registered.

Add another repo (blank to finish):
> ~/Dev/side-project
✓ Detected. Register as path-reference? [Y/n]
```

→ 신규 사용자가 init 단계에서 *모든 외부 repo 한 번에* 등록 가능. *별도 add repo 명령 X*.

### 13.4 디렉터리 스캔 — bulk

```bash
solosquad add repo --discover ~/Dev/
```

→ `~/Dev/` 하위 `.git/` 있는 폴더 자동 찾아서:
```
Found 5 git repositories in ~/Dev/:
  [1] my-saas              github.com/me/my-saas       (TypeScript, 142 MB)
  [2] side-project         github.com/me/side          (Python, 23 MB)
  [3] consulting-client-X  gitlab.com/.../X            (Go, 88 MB)
  [4] bv-ga-dashboard      (no remote)                 (TypeScript, 51 MB)
  [5] cursor-talk-to-figma-mcp-main (no remote)        (other, 12 MB)

Select to register (1-5, comma-separated, or 'all'): 1,2,3
For each, choose org and role (interactive).
```

⚠️ *사용자 명시 호출만*. default 동작 X — 사용자 home 자동 스캔은 privacy 이슈 + nested .git noise 많음.

### 13.5 (v1.x slot) GitHub API 연동

```bash
solosquad add repo --discover-github   # gh CLI auth 필요
```

→ `gh repo list me` → 사용자 GitHub 본인 repo 목록 + 로컬 clone 위치 추정 → 매칭되는 것 자동 path-reference 등록.

v0.8.6 결정에 따라 *gh CLI는 optional*. v1.x slot.

### 13.6 정리 — default 진입점

| 사용자 의도 | 명령 |
|---|---|
| 한 repo 등록 (가장 흔함) | `cd <repo> && solosquad add repo` (cwd 인식) |
| 스크립트로 batch | `solosquad add repo --path <ext> --org <slug>` (×N) |
| 새 워크스페이스 처음 | `solosquad init`의 Step 5.1에서 path 입력 |
| 여러 repo 한 번에 검색 | `solosquad add repo --discover <dir>` |
| URL clone 받기 | `solosquad add repo <url>` (default = `~/Dev/<slug>`로 clone 후 path-reference 등록) |

## 14. 워크스페이스 위치 — *1개 per user* 가정 + 권장 사례

### 14.1 v0.9 *기본 멘탈 모델* — 1 사용자 = 1 워크스페이스

모델 B 도입으로 워크스페이스가 *작은 config 폴더* (~ 50 MB)로 축소되면서, **솔로 founder 1인 = 1 워크스페이스에서 모든 SoloSquad 활동 관리** 가 default가 됨:

```
사용자 머신
├── ~/Dev/                              ← 사용자 dev tree, 그대로
│   ├── my-saas/        (1.4 GB)
│   ├── side-project/   (200 MB)
│   ├── consulting-X/   (88 MB)
│   └── ...
│
└── ~/solosquad/                        ← *유일한 워크스페이스* (~ 50 MB)
    ├── .solosquad/                      (시스템 메타 + 25 agents + templates)
    ├── personal/                        ← org 1 (personal product)
    │   ├── .org.yaml
    │   ├── memory/  workflows/  goals/
    │   ├── .solosquad/users/<handle>.yaml
    │   └── repositories/
    │       ├── my-saas.yaml             ← path → ~/Dev/my-saas
    │       └── side-project.yaml        ← path → ~/Dev/side-project
    ├── work/                            ← org 2 (day-job)
    │   ├── .org.yaml
    │   ├── memory/  workflows/
    │   └── repositories/
    │       └── dayjob.yaml              ← path → ~/work/dayjob
    └── consulting/                      ← org 3 (consulting client)
        └── repositories/
            └── client-X.yaml            ← path → /Volumes/External/client-X
```

→ **1 워크스페이스가 모든 사용자 활동의 *허브***. 모든 org·repo·memory·workflow·goal이 여기서 관리.

### 14.2 권장 위치

| 위치 | 권장 여부 | 사유 |
|---|---|---|
| **`~/solosquad/`** (또는 `~/agents/`, `~/sf/`) | ✓ **권장** | 사용자 home 안 짧고 의미 있는 이름. 눈에 보임. 백업·이동 쉬움 |
| `~/Documents/solosquad/` | △ 가능 | OS별 Documents 경로 변동 (한국어 Windows 등 edge case) |
| `~/.solosquad/` (hidden) | △ 가능 | dotfile 컨벤션. Windows hidden 옵션 켜야 보임 — 사용자 자기 워크스페이스 *잊기 쉬움* |
| `/opt/solosquad/` 등 system path | ✗ 비권장 | 권한 이슈, 사용자 home에서 너무 멀어짐 |

**default 동작 변화 없음**: `solosquad init`이 자동 `~/solosquad/`로 강제하지 않음 (v0.8.4 cwd default 그대로). 단, master-guide Step 5 예시 코드를 *짧은 이름* 권장으로:

```bash
# 권장 (v0.9)
mkdir solosquad && cd solosquad && solosquad init

# 또는 본인 취향
mkdir agents && cd agents && solosquad init
mkdir sf && cd sf && solosquad init     # short
```

### 14.3 *예외* — n잡 멀티 메신저 시나리오

`MESSENGER=discord,slack` 같은 *단일 워크스페이스에서 멀티 메신저*는 v0.2.0에서 영구 제거. 따라서 한 사용자가 *서로 다른 메신저를 동시에* 사용해야 할 때만 워크스페이스를 여러 개 운영:

```
~/solopreneur/     ← Slack 봇 (본업 페르소나)
~/personal-lab/    ← Discord 봇 (사이드 페르소나)
```

이는 **default가 아니라 예외 시나리오**. master-guide §9.2가 *멀티 워크스페이스 페르소나 분리*로 다룸 — v0.9에서도 *그 절은 유지*하되, *core 안내는 1 워크스페이스 가정*.

→ **요약**: SoloSquad core mental model = *1 user → 1 workspace → N orgs → N repos (path-referenced)*. 멀티 워크스페이스는 *멀티 메신저 페르소나가 필요할 때만 advanced option*.

### 14.4 워크스페이스가 *작아져서 생기는* 추가 이점 (모델 B)

- **dotfiles로 백업 가능** — git repo로 워크스페이스 트래킹 (~ 50 MB) → GitHub private repo에 push → 다른 머신에서 clone하면 *같은 SoloSquad 설정* 복원 (path만 머신별로 다르므로 path field만 매핑 갱신)
- **외장 디스크 휴대성** — 워크스페이스를 USB·외장 SSD에 두고 머신 옮겨가며 사용 (repo는 각 머신 로컬 ~/Dev/에 있다 가정. path만 머신별로 갱신)
- **클라우드 동기화** — Dropbox·iCloud·OneDrive에 워크스페이스 폴더 두기 (`.solosquad/.env`는 gitignored이므로 별도 처리). 50 MB는 무료 티어로도 충분
- 모델 A에서는 *불가능* (repo가 트리 안에 있어서 GB대 동기화 부담 + git tree 안 git tree 충돌)
