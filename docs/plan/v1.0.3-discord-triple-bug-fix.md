## 1. 개념 정합 — Product / Org / Workspace

본 plan doc 안의 변수명·필드명에 *Product* 가 곳곳 나옴 (예: `loadProducts()`, `getProductByGuild()`, `Product` 타입). 정직히 박제:

| 계층 | 정체 | 코드 안 위치 |
|---|---|---|
| **Workspace** | *최상위 컨테이너*. 한 사용자/팀의 SoloSquad 인스턴스 전체 | `<workspace>/.solosquad/workspace.yaml` |
| **Product** | *개념상 워크스페이스 레벨*. 여러 org 가 협업해 만들 수 있는 사용자 가치 단위 | 코드 안엔 명시적 모델 없음 — v0.1.x 잔재 타입 `Product` 가 *org 데이터를 그대로 합성한 가짜 product* 로 존재 |
| **Org** | *비즈니스 단위*. 한 product 를 만드는 *팀 / 회사 / 부서 / 콘트랙터 중 하나*. 메모리·워크플로우·repo 격리 단위 | `<workspace>/<org-slug>/.org.yaml` |
| **Repo** | *코드 단위*. org 가 보유한 git repo (path-reference). 한 org 가 여러 repo, 한 product 가 여러 repo (잠재적으로 여러 org 의 repo 포함) | `<org>/.solosquad/repositories/<slug>.yaml` |

**현재 코드 단순화:** v0.2.2+ 에서 *1 workspace = 1 org = 1 implicit product* 라고 가정해서 `loadProducts()` 가 org 를 *그대로 product 로* 합성. 솔로 파운더 단일 사용 시나리오에선 무해하지만 *진짜 product 모델 (여러 org 가 협업)* 은 아직 코드에 없음.

**본 v1.0.3 의 입장:** *Product/Org 타입 리네이밍은 cosmetic surface 가 넓어 v2.0 슬롯으로 분리*. v1.0.3 은 *행동 버그* 만 fix. 다만 변경할 코드 안의 변수명은 *"product" 그대로 유지* 하되 새 함수·주석엔 *org* 라는 정확한 어휘 사용.

## 2. Bug A — `versionMatches` slice 산수가 patch-level migration 영구 차단

### 2.1 Incident
사용자 워크스페이스 `workspace.yaml.version: 1.0.0` 에서 v1.0.2 CLI 로 `solosquad migrate --apply` 실행:
```
✗ Migration failed: No migration found for source version 1.0.0
```

`doctor` 는 `CLI v1.0.2 > workspace v1.0.0 — Run: solosquad migrate --apply` 안내했지만 그 명령이 실행 안 됨.

### 2.2 Root cause — slice arithmetic
[`src/migrations/detect.ts:56-64`](../../src/migrations/detect.ts#L56):
```ts
export function versionMatches(spec: string, detected: string): boolean {
  if (spec === detected) return true;
  if (spec.endsWith(".x")) {
    const prefix = spec.slice(0, -1); // "1.0.0.x" → "1.0.0."
    return detected.startsWith(prefix);
  }
  return false;
}
```

`versionMatches("1.0.0.x", "1.0.0")`:
- `spec === detected` → false
- `prefix = "1.0.0."` (`.slice(0, -1)` 가 마지막 `x` 만 자르고 `.` 가 남음)
- `"1.0.0".startsWith("1.0.0.")` → **false** (detected 끝에 `.` 없음)

→ **`1.0.0` 워크스페이스가 어떤 migration 도 매치 못 함**. 같은 패턴으로 *모든 `X.Y.Z.x` from-spec 이 exact `X.Y.Z` 와 mismatch* — v092ToV100 (`from: "0.9.2.x"`), v100ToV101 (`from: "1.0.0.x"`), v101ToV102 (`from: "1.0.1.x"`), v081ToV082 외 8건 동일.

`from: "0.1.x"` 같은 *minor-loose* 패턴은 우연히 작동 — detected (`"0.1.0"`) 에 `.` 가 따라와서 `startsWith("0.1.")` 매치 성공. *patch-exact* 패턴만 함정.

### 2.3 채택안 — `versionMatches` 한 줄 fix

```ts
export function versionMatches(spec: string, detected: string): boolean {
  if (spec === detected) return true;
  if (spec.endsWith(".x")) {
    const baseExact = spec.slice(0, -2);          // "1.0.0.x" → "1.0.0"
    const prefixWithDot = spec.slice(0, -1);      // "1.0.0.x" → "1.0.0."
    return detected === baseExact || detected.startsWith(prefixWithDot);
  }
  return false;
}
```

- `"0.1.x"` 케이스 (기존 작동) — `baseExact = "0.1"`, `prefixWithDot = "0.1."`. `detected = "0.1.0"` → `startsWith` 매치 ✅ (변화 0)
- `"1.0.0.x"` 케이스 (본 버그) — `baseExact = "1.0.0"`, `prefixWithDot = "1.0.0."`. `detected = "1.0.0"` → `=== baseExact` 매치 ✅ (수정됨)
- `"1.0.0"` 케이스 (exact, .x 없음) — IF 안 들어감, 첫 줄 `spec === detected` 로 처리 (변화 0)

### 2.4 published migration 의 `from` 필드는 *건드리지 않음*
[`AGENTS.md` Immutable paths](../../AGENTS.md): *"`src/migrations/scripts/**` once a script is published in a tagged release ... fix forward in a new migration; do not rewrite history"*. `versionMatches` 는 `src/migrations/detect.ts` (registry helper, not scripts/) 라 영향권 외.

`from` 패턴 자체는 정직히 *옛 잔재* (v0.5.x 까지 적어도 8건의 patch-exact 패턴이 누적). 한 줄 매처 fix 가 모든 누적 함정 동시 해소 + 미래 patch migration 도 같은 함정 안 빠짐.

### 2.5 회귀 catcher (`test/v1.0.3-version-matches.test.ts`, 5 cases)
- `versionMatches("1.0.0.x", "1.0.0")` === true (본 버그 trip-wire)
- `versionMatches("1.0.0.x", "1.0.0.5")` === true (가상 patch level — semver 4-segment 부재이지만 패턴 자체는 보존)
- `versionMatches("0.1.x", "0.1.0")` === true (기존 minor-loose 회귀 0 확인)
- `versionMatches("0.1.x", "0.1")` === true (exact base 매치 신규)
- `versionMatches("0.2.0", "0.2.0")` === true (exact-only 패턴 회귀 0)

## 3. Bug B — `npmGlobalInstallCmd` 가 nvm/brew 사용자에게 *잘못된 sudo 권유*

### 3.1 Incident
사용자 `solosquad update`:
```
Run to update:
  sudo npm install -g solosquad@latest
```

사용자 환경은 nvm-managed Node: prefix `/Users/seungw1n/.nvm/versions/node/v25.2.1` — *본인 소유* (root 아님). `sudo` 절대 불필요. 본인 비밀번호 입력 후 *그 명령이 동작* 했지만 *애초에 권유가 부정확*.

### 3.2 Root cause — root 체크가 prefix 권한 체크로 대체되어야 함
[`src/util/platform.ts:28-33`](../../src/util/platform.ts#L28):
```ts
export function npmGlobalInstallCmd(pkg: string): string {
  if (IS_WINDOWS) return `npm install -g ${pkg}`;
  const isRoot = process.getuid?.() === 0;
  return isRoot ? `npm install -g ${pkg}` : `sudo npm install -g ${pkg}`;
}
```

**가정:** "현재 프로세스가 root 가 아니면 무조건 sudo 필요". 현실:
- nvm / fnm / asdf 사용자 — prefix 가 본인 home 안. sudo 0
- Homebrew (Apple Silicon, `/opt/homebrew`) — 본인 소유. sudo 0
- Homebrew (Intel Mac, `/usr/local`, brew 가 chown) — 본인 소유. sudo 0
- 시스템 패키지 (`apt install nodejs`, default prefix `/usr/local`) — root 소유. sudo 필요 ✓

→ *현대 macOS 사용자 대다수* 가 false sudo 권유. *시스템 패키지 설치자* 만 진짜 sudo 필요.

### 3.3 채택안 — prefix 쓰기 권한 체크

```ts
import fs from "fs";
import { execSync } from "child_process";

export function npmGlobalInstallCmd(pkg: string): string {
  if (IS_WINDOWS) return `npm install -g ${pkg}`;
  try {
    const prefix = execSync("npm config get prefix", {
      encoding: "utf-8",
      stdio: ["ignore", "pipe", "ignore"],
    }).trim();
    fs.accessSync(prefix, fs.constants.W_OK);
    return `npm install -g ${pkg}`;          // 본인 prefix 쓰기 가능 → sudo 0
  } catch {
    return `sudo npm install -g ${pkg}`;     // root-owned prefix → sudo 필요
  }
}
```

설계 결정:
- `execSync("npm config get prefix")` 가 throw 하거나 `accessSync` 가 throw 하면 *fallback = sudo*. 정직 conservative.
- root 자체 체크 (`process.getuid`) 는 제거 — root 로 실행 시 prefix 도 root 소유라 accessSync 통과 → sudo 안 붙음 자동.
- Windows 분기 그대로 (Windows 는 npm prefix 권한 모델 달라 별 분석 필요 — v2.0 슬롯).

### 3.4 회귀 catcher (`test/v1.0.3-npm-install-cmd.test.ts`, 3 cases)
- prefix 가 user-writable mock → `npm install -g X` (sudo 0)
- prefix 가 readonly mock → `sudo npm install -g X`
- `npm config get prefix` 가 throw → `sudo npm install -g X` (fallback safety)

### 3.5 사용자 영향
- nvm/brew 사용자: 다음 `solosquad update` 부터 sudo 권유 사라짐 + 비번 입력 단계 사라짐
- 시스템 패키지 사용자: 변화 0 (여전히 정확한 sudo 권유)

## 4. Bug D — `syncGuildProductMapping` 의 v0.1.x 잔재 휴리스틱이 *서버 이름 ↔ org slug 문자열 비교* 에 묶임

### 4.1 Incident
사용자가 `command-w1n` 채널 (정상 자기 채널) 에서 "하이" 입력 → 봇 응답:
```

```

부팅 로그엔 *`[Discord] Mapped: ... ↔ ...`* 라인 없음 → `syncGuildProductMapping` 의 매핑 시도 자체 실패.

### 4.2 Root cause — 결정자 무시한 약한 휴리스틱

**Discord 서버 이름 안에 org name 또는 slug 가 *문자열로 포함*** 되어야 매핑 발사. 본 사용자의 서버 이름이 `rosyocean` (org slug) 을 포함하지 않으면 매핑 자체 안 일어남.

### 4.3 채택안 — `ownOrgSlug` 직접 사용 (휴리스틱 제거)

```ts
private syncGuildProductMapping(): void {
  if (!this.client || !this.ownOrgSlug) {
    // v0.8 이상: ownOrgSlug 가 있어야 매핑 가능. 없으면 user yaml 미등록.
    return;
  }

  const orgSlug = this.ownOrgSlug;
  const configFile = path.join(getReposBase(), orgSlug, "discord", "config.yaml");
  if (!fs.existsSync(configFile)) return;

  const config = (yaml.load(normalizeLine(fs.readFileSync(configFile, "utf-8"))) ?? {}) as Record<string, unknown>;
  const guilds = Array.from(this.client.guilds.cache.values());

  if (guilds.length === 0) return;
  if (guilds.length > 1) {
    // 다중 guild — v0.8 단일 사용자 가정과 충돌. 명시적 안내 + 첫 guild 로 binding.
    console.log(
      `[Discord Bot] Bot in ${guilds.length} guilds; binding the first (${guilds[0].name}) to org=${orgSlug}. ` +
        `Multi-guild orchestration is not supported in v1.0.x.`,
    );
  }

  const guild = guilds[0];
  if (config.guild_id !== guild.id) {
    config.guild_id = guild.id;
    const channels = (config.channels || {}) as Record<string, string>;
    for (const ch of guild.channels.cache.values()) {
      if (this.channelNames.includes(ch.name)) {
        channels[ch.name.replace(/-/g, "_")] = ch.id;
      }
    }
    config.channels = channels;
    fs.writeFileSync(configFile, yaml.dump(config));
    console.log(`[Discord] Bound guild ${guild.name} (${guild.id}) → org=${orgSlug}`);
  }
}
```

설계 결정:
- *서버 이름 비교 휴리스틱 완전 제거*. v0.1.x 시절 봇이 *어느 org/product 소속인지 모르는 상황* 의 fallback — v0.8 이후 의미 없음.
- *단일 guild 기본 가정* (`works-w1n` / `command-w1n` 한 쌍 = 한 사용자 = 한 봇 = 한 guild). 다중 guild 는 *명시적 안내 후 첫 guild 로 binding* + 향후 정책 결정 슬롯.
- `loadProducts()` 호출 제거 — 코드 안의 잔재 vocab 만 그대로 두되 *실제 결정 경로에서 빠짐*.

### 4.4 인접 변경 — `getProductByGuild` 도 같은 정신

[`src/messenger/discord-adapter.ts:352`](../../src/messenger/discord-adapter.ts#L352):
```ts

```

이 함수는 `messageCreate` 핸들러가 *매 메시지마다* 호출. 그러나 봇은 이미 자기 org 를 *세션 lifetime 내내* 안다 (`this.ownOrgSlug`). config.yaml 의 `guild_id` 일치 여부도 *시작 시 한 번* 검증되면 충분.

본 v1.0.3 변경:

### 4.5 회귀 catcher (`test/v1.0.3-guild-org-binding.test.ts`, 4 cases)

사용자가 `solosquad doctor` 실행 → *post-fact* 경고:
```
△ CLI v1.0.2 > workspace v1.0.0 — Run: solosquad migrate --apply
```

→ **update 가 *successful 박제* 와 *다음 액션 안내* 사이의 끊김** 을 만듦. doctor 에서 같은 안내가 *결국 표시되지만* 사용자 입장에선 *update 흐름 안에서 명확하지 않음* — 두 번 명령 (update + doctor) 으로 분산.

### 4b.2 Root cause — update.ts 가 *post-install workspace 검사 없음*
[`src/cli/update.ts:102-104`](../../src/cli/update.ts#L102):
```ts
execSync(updateCmd, { stdio: "inherit" });
console.log(chalk.green("\n✓ Updated successfully!"));
console.log(chalk.dim("Run `solosquad doctor` to verify."));
```

설치 직후 *workspace.yaml.version* 와 *방금 설치된 CLI version* 비교 없음. 격차 있을 때 명시적 다음 단계 안내 없음.

### 4b.3 채택안 — 설치 직후 workspace lag 검사 + 명시적 다음 단계 안내

```ts

console.log(chalk.green("\n✓ Updated successfully!"));

// v1.0.3 — patch-level update 직후 workspace 버전과의 lag 확인.
```

- 다음 액션이 *정확히 `solosquad migrate --apply`* — doctor 가 *후속* 으로 같은 안내하던 우회 단축

## 4c. Bug F — Discord 채널 카테고리 이름 *"AI Team Reports"* → *"solosquad"*

### 4c.1 사용자 요구
*"디코에서 채널 카테고리 생성할 때 이름 solosquad 로 생성되게"*. 현재 v0.1.x 시절 박힌 *"AI Team Reports"* 이름이 *SoloSquad 브랜드와 정합 안 됨*. 신규 사용자가 봇이 만드는 카테고리 이름을 보고 *"이게 SoloSquad 가 만든 거 맞나"* 라는 sense-making 부담.

### 4c.2 Backward compat 요구
기존 사용자 (= 본 사용자) 는 이미 *"AI Team Reports"* 카테고리 안에 `command-<handle>` / `works-<handle>` 채널이 들어 있음. 새 코드가 카테고리 이름을 *"solosquad"* 로만 찾으면 → 못 찾음 → 새 *"solosquad"* 카테고리 생성 → 빈 카테고리만 추가, 채널은 *"AI Team Reports"* 안에 그대로 남음 → 유저 UX 망가짐.

**채택안 (사용자 확정):** *"둘 다 매칭"* — 코드가 `ch.name === "solosquad" || ch.name === "AI Team Reports"` 둘 다 찾음. 기존 카테고리 *그대로 동작*, 신규 카테고리 *생성 시만 "solosquad"*. 사용자가 원하면 Discord UI 에서 *수동 rename* 가능 (봇이 안 건드림).

### 4c.4 채택안 — 둘 다 매칭 + 신규는 "solosquad"

```ts
// v1.0.3 — category name elevated to brand-consistent "solosquad".
// Existing installs may have the legacy "AI Team Reports" name; we match
// both so the existing tree stays attached to its channels. New installs
// (or users who manually deleted their category) get "solosquad".
const LEGACY_CATEGORY_NAMES = ["solosquad", "AI Team Reports"] as const;

let category = guild.channels.cache.find(
  (ch) =>
    ch.type === ChannelType.GuildCategory &&
    (LEGACY_CATEGORY_NAMES as readonly string[]).includes(ch.name),
);
if (!category) {
  category = await guild.channels.create({
    name: "solosquad",
    type: ChannelType.GuildCategory,

}
```

설계 결정:
- *legacy → canonical* 둘 다 lookup. 봇이 *기존 사용자 카테고리 rename 안 함* — Discord ManageChannels 권한 가정 안 함 + 사용자가 *의도적으로 다른 이름* 으로 바꿔 둔 경우 존중

- *2-name 배열* 로 박제 → 향후 또 이름 바뀌어도 `LEGACY_CATEGORY_NAMES` 만 확장
- 사용자가 본인 카테고리를 수동 rename 하고 싶으면 Discord UI 에서 *"AI Team Reports"* 우클릭 → Edit Category → "solosquad" 입력 (봇 무관, 클라이언트 측 작업)

본 plan 채택안 6건 모두 이 정신:
- v1.0.2 author-guard: handle = canonical (자유 입력 비교 제거)

- v1.0.3 B: prefix 권한 = 진짜 결정자 (UID 추측 제거)
- v1.0.3 D: ownOrgSlug = 봇이 이미 아는 결정자 (서버명 비교 제거)

이 패턴은 *향후 회귀 catcher 설계 가이드라인* — *외부 자유 입력과 내부 슬러그를 비교하는 코드 + v0.1.x 잔재 string literal 모두 trip-wire 대상*.

## 10. v2.0 candidates (본 v1.0.3 가 *드러낸* 잔재)

본 patch 가 fix 하지 않고 *박제만 한* 잔재:

- **`Product` 타입·변수명 → `Org` 로 리네이밍.** v0.1.x 시절 *product = top-level* 가정의 화석. 현재 v0.2.2+ 코드는 *product = org 합성* 으로 단순화하지만 *진짜 product 모델 (여러 org 가 협업)* 은 코드에 없음. 사용자 표면 변경 0 이지만 *내부 vocab* 정합 필요. 변경 표면이 8+ 파일이라 v2.0 슬롯.
- **`loadProducts()` 함수 → `loadOrgs()` 또는 `synthesizeProductsFromOrgs()`.** 위와 같은 줄.
- **Windows npm prefix 권한 모델 분석.** v1.0.3 의 fix B 는 Unix 만. Windows 는 npm prefix 동작 다름 — `npm install -g` 가 `%APPDATA%/npm` 에 쓰는 게 default 라 일반적으로 admin 불필요하나 코드 path 가 그대로 둠.
- **Conceptual `Product` 모델 도입.** *여러 org 가 한 product 를 만든다* 는 사용자 모델을 *실제 코드 구조* 로 끌어올림. v0.1.x 시절 product 가 *그저 dir 명* 이었던 데서 *workspace 레벨 객체* 로 격상 — schema 변경 필요 (workspace.yaml 안에 `products[]` 배열, 각 product 가 `member_orgs[]` 보유 등). 본격 v2.0 작업.
