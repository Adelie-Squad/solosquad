# v1.2 — 사용자 지식·암묵지 온톨로지 + 외부 지식 연결

> 솔로 파운더의 메모·URL·인사이트·고민(명시지 + 암묵지)을 온톨로지로 구조화해 25명 에이전트가 참조하도록 한다. 메신저 채널에서 직접 입력하거나, MCP를 통해 Notion·Obsidian·임의 API·타 에이전트 등 외부 지식 소스를 연결한다.

**로드맵:** v0.1 크로스 플랫폼 → v0.2 안전/보안 → v0.3 PM 모드 → v0.4 자율 실행 → v0.5 워크플로우 메이커 → v0.6 디폴트 워크플로우 → v1.1 대시보드 상호작용 → **v1.2 지식 온톨로지** → v2.0 커뮤니티 → v3.0 코워킹

**의존:** v0.3 (PM 모드 — 메신저 입력의 진입점), v0.4 (자율 실행 — keep/discard 결과에서 지식 자동 추출), v0.5 (워크플로우 메이커 — 에이전트 메타데이터 + frontmatter 스키마 공유)

**기준 버전:** v1.2.0

## 핵심 질문 & 성공 기준

| 핵심 질문 | 성공 기준 |
|---|---|
| 솔로 파운더의 *암묵지*가 시스템에 흘러 들어오는가? | 메신저 채널에 메모/URL/인사이트/고민을 평문으로 던지면 → 자동 분류·태깅 후 온톨로지 진입 |
| 25 에이전트가 같은 지식을 *일관되게* 참조하는가? | 에이전트 실행 시 관련 노드를 자동 주입, 동일 질의에 대한 응답 분기가 < N% |
| Notion/Obsidian/타 API의 지식이 *시스템 안에서* 살아 있는가? | MCP 통한 양방향 동기화, 외부 변경이 5분 내 온톨로지에 반영 |
| 외부 컨텐츠가 prompt injection 벡터가 되지 않는가? | MCP 응답에 대한 sanitization + 명시적 사용자 confirmation gate (Microsoft 가이드 준수) |
| 자가 호스팅·파일 기반 원칙이 유지되는가? | 별도 DB·외부 클라우드 의존 없이 동작 (Letta filesystem 벤치마크 차용) |

## 1. 현재 문제

| 문제 | 원인 |
|---|---|
| TEAM_KNOWLEDGE.md가 수동 작성 | 프로젝트 실행 경험이 축적되지 않음 |
| 사용자 머릿속의 *암묵지*(이 산업의 관행, 자기 고민, 과거 실패 등)가 시스템에 없음 | 입력 채널 부재. 코드에 쓸 수밖에 없는 형태로만 표현됨 |
| Notion·Obsidian에 이미 쌓인 자료가 에이전트와 단절 | 외부 지식 소스 통합 레이어 없음 |
| 에이전트 간 지식 단절 | A가 학습한 것을 B가 모름. 같은 PM 세션 안에서만 공유됨 |
| 프로젝트 간 지식 이전 없음 | 프로젝트 A의 교훈이 프로젝트 B에 전달 안됨 |
| 구조화 안됨 | 자연어 덩어리라 검색/참조 불가, vanity 메모와 가치 있는 결정의 구분 없음 |

## 2. 원칙

| 원칙 | 구현 |
|---|---|
| **File-first** | Markdown + YAML frontmatter가 진실의 원천. 그래프·벡터는 캐시 (Letta filesystem 벤치마크 결과 차용) |
| **Markdown-LD** | frontmatter를 JSON-LD-호환 triple 형태로. 외부 도구가 같은 의미론으로 읽을 수 있도록 (Markdown-LD spec) |
| **Anthropic Memory Tool 호환** | 컨텍스트 메모리 인터페이스는 Anthropic context-management-2025-06-27 호환. 39% task quality / 84% 토큰 감소 보고치 활용 |
| **Temporal-aware** | 같은 사실이라도 *언제* 입력됐는지가 의미를 좌우. Graphiti처럼 시간 축 가진 노드 |
| **Tacit elicitation as a skill** | PM이 LLMREI 패턴으로 사용자의 *암묵*을 묻는 별도 메타-skill. v0.5 author 루프 형제 |
| **MCP-first 외부 연결** | Notion·Obsidian·API·타 에이전트 모두 MCP 서버로. SoloSquad 안에 어댑터 코드 직접 작성하지 않음 |
| **명시적 보안 게이트** | 외부 컨텐츠는 신뢰 등급(L0~L3). L3(외부 웹/MCP) 입력은 PM이 사용자 confirmation 받고 sanitize 후 주입 (Unit42 + Microsoft 가이드) |
| **A2A for other agents** | "타 에이전트 연결"은 MCP가 아닌 A2A 프로토콜(Linux Foundation). MCP = tool, A2A = agent |

## 3. 온톨로지 구조

4계층 (기존 3계층 + Founder Layer 추가):

```
Knowledge Ontology
├── Founder Knowledge (사용자 단위, v1.2 신규)         ← 메신저·MCP 입력
│   ├── notes/            → 일상 메모, URL 스크랩
│   ├── insights/         → "이거 중요" 표시한 통찰
│   ├── concerns/         → 결정 못 한 고민, 미해결 질문
│   ├── tacit/            → LLMREI 면담으로 추출한 암묵 룰
│   └── external/         → MCP 동기화된 외부 지식 (캐시 + 포인터)
│
├── Team Knowledge (팀 단위)
│   ├── frameworks/
│   ├── conventions/
│   └── glossary/
│
├── Agent Knowledge (에이전트 단위)
│   ├── patterns/
│   ├── anti-patterns/
│   └── heuristics/
│
└── Project Knowledge (프로젝트 단위, 자동 축적)
    ├── decisions/
    ├── experiments/
    └── learnings/
```

승격 흐름: Founder `tacit` → 반복 확인 N회 → Team `convention` 자동 승격. Project `learning` → cross-project N회 → Team `framework`.

## 5. 메신저 채널 입력 (Founder Layer 진입점)

### 5.1 채널 동작

`#owner-command` 채널에서 일상 메시지를 그대로 보내거나, 특정 trigger 키워드를 붙임:

```
[user] 메모: 등기부 알림 사용자 인터뷰에서 다들 "가격이 비싸도 정확하면 쓴다"고 함
[user] URL https://example.com/blog/funnel-pattern  ← 자동으로 fetch + 요약
[user] 인사이트: pivot 후보 — B2B 부동산 중개사 타겟
[user] 고민: 가격 책정, 3,900원 vs 9,900원
```

PM이 메시지를 받음 → 의도 분류 (note / insight / concern / tacit-followup) → 적절한 Founder Layer 디렉토리에 저장.

### 5.2 암묵지 추출 (LLMREI 패턴, 별도 메타-skill)

매주 자동 또는 사용자가 `/founder review` 호출 시:

```
PM: 지난 주 메모/고민을 봤습니다. 몇 가지만 확실히 짚을게요.

    1. "가격이 비싸도 정확하면 쓴다" 인터뷰 — 표본 몇 명이었나요?
    2. "B2B 부동산 중개사" 타겟 가설 — 어떤 데이터를 더 보면 확신이 서나요?
    3. 가격 고민은 결정 시한이 있나요?

(답변 후)
    이렇게 정리합니다:
    - Tacit: "정확성이 가격 민감도를 압도한다 (인터뷰 N=4)"  (confidence: 0.55)
    - Concern: "B2B 부동산 중개사 타겟 — 결정 시한 6/1까지" (open)
    
    저장 (y/N): y
```

추출된 Tacit이 N회 반복 확인되면 Team Knowledge로 자동 승격.

### 5.3 PM 컨텍스트 주입

에이전트 실행 시 PM이 관련 Founder Knowledge를 자동 슬라이스해서 _handoff.md에 포함:

```yaml
# _handoff.md 추가 섹션
founder_context:
  active_concerns:
    - "가격 책정 미결 (6/1 시한)"
  applied_tacit:
    - "정확성이 가격 민감도를 압도한다 (confidence 0.55)"
  contradicts_warning: []
```

각 specialist subagent는 fresh session이지만 _handoff에 적힌 *암묵*을 알고 일을 시작.

## 6. MCP 통합 (외부 지식 연결)

### 6.1 채택 프로토콜

**MCP 2025-11-25** 스펙 (SEP-1024 클라이언트 보안 + SEP-835 default-scopes) 위에서 동작. OAuth 기반 인증, server-initiated elicitation, structured tool output 사용.

### 6.2 1차 통합 대상

| 외부 소스 | MCP 서버 | 동기화 모드 | 비고 |
|---|---|---|---|
| **Notion** | `makenotion/notion-mcp-server` (공식, v2.0.0+) | 양방향 | 2025-09-03 Notion API의 `data_sources` 추상 사용 |
| **Obsidian** | `MarkusPfundstein/mcp-obsidian` (커뮤니티) | 양방향 | Local REST API 플러그인 필요 |
| **임의 REST API** | FastMCP OpenAPI integration | 단방향 read | OpenAPI 스펙 → MCP 서버 자동 생성 |
| **임의 REST API (대안)** | AWS `openapi-mcp-server` | 단방향 read | 프로덕션 지향, Python 런타임 회피 |
| **타 에이전트** | A2A 프로토콜 (gRPC + signed agent cards) | request/response | MCP 아님 — agent-to-agent 표준 |
| **MCP 서버 운영** | MCPJungle 또는 Docker MCP Gateway | 게이트웨이 | 자가 호스팅 단일 사용자에 적합 |

### 6.3 등록 흐름

```bash
solosquad mcp add notion          # 대화형 OAuth + workspace 선택
solosquad mcp add obsidian --vault ~/Obsidian/Founder
solosquad mcp add api --openapi https://api.example.com/openapi.json
solosquad mcp list
solosquad mcp test <name>          # tools/prompts/resources 한 번 호출
solosquad mcp allowlist --add <tool-name>
```

GitHub Internal MCP Registry 모델 차용 — 기본은 *registry-only allowlist*. 사용자가 명시적으로 추가한 서버만 활성.

### 6.4 동기화 정책

- **Pull (외부 → 시스템)**: 5분 폴링 + webhook 우선. 외부 변경이 `Founder/external/` 아래 캐시로 들어옴 + 포인터 노드로 표현. 원본은 외부에 남음.
- **Push (시스템 → 외부)**: 사용자가 `/founder export` 호출 시에만. 자동 push 없음 (외부 데이터 손상 방지).
- **충돌**: 외부 변경 우선. 시스템 캐시는 read-only로 표현, edit은 외부 도구에서.

### 6.5 보안 (필수)

Palo Alto Unit 42 + Microsoft Prompt Shields 가이드 준수:

1. **Tool poisoning 방어** — MCP 서버 등록 시 디스크립션을 사용자에게 표시 + 명시 승인
2. **Rug-pull 방어** — 디스크립션 hash를 etcd에 저장. 변경되면 재승인 요구
3. **Indirect prompt injection 방어** — 모든 MCP 응답에 `trust_level: L3` 마킹. PM이 L3 컨텐츠를 specialist에 전달하기 전 sanitize (URL/명령어/system prompt 패턴 차단)
4. **Human-in-the-loop** — 외부 컨텐츠가 새 워크플로우를 *제안*하는 경우 사용자 confirmation 필수
5. **Least privilege** — MCP 토큰은 read-only 권장. write 권한은 사용자가 명시적으로 ack

### 6.6 A2A — 타 에이전트 연결

다른 에이전트(예: 외부 협업자의 SoloSquad 또는 호환 시스템) 호출은 MCP가 아니라 A2A:

- A2A v0.3 (JSON-RPC 2.0 + gRPC + signed agent cards)
- PM이 외부 에이전트를 *specialist 후보*로 인지 (Anthropic Managed Agents 패턴, depth=1)
- 응답은 trust_level L2로 마킹 (서명된 카드 확인 시)

## 7. 자동 축적 (v0.4 자율 실행 연계)

```
v0.4 사이클 완료
    ↓
keep   → Agent patterns/      또는 기존 패턴 confidence 상승
discard → Agent anti-patterns/ 또는 기존 anti-pattern confidence 상승
    ↓
keep/discard와 직접 연관된 Founder concern이 있으면
    → Founder concerns/ 해당 노드에 'resolved_by' 관계 추가
    ↓
패턴이 N회 cross-project 확인되면 → Team frameworks/ 자동 승격
```

Founder concerns는 자동 close 안 함. 사용자가 메신저에서 confirm 해야 closed 상태.

## 8. 에이전트 간 지식 참조 (검색·주입)

에이전트 실행 시 컨텍스트 빌더가 4-소스 검색:

1. **SKILL.md** — 역량 정의 (v0.5)
2. **TEAM_KNOWLEDGE.md + frameworks/conventions** — 팀 공유
3. **Agent patterns/heuristics** — 이 에이전트의 history
4. **Founder context slice** — 활성 concerns/tacit + 현재 프로젝트의 decisions/learnings

**검색 백엔드 (선택):**
- *Default*: 파일 시스템 grep + tags 매칭 (Letta 벤치마크 "filesystem is all you need")
- *Optional*: Mem0 / Zep-Graphiti / Cognee로 그래프 인덱스 빌드 — 사용자가 `solosquad knowledge index --backend mem0` 등으로 활성

기본은 파일 시스템. 그래프 백엔드는 100+ 노드 누적 시점에 사용자 선택.

## 13. 테스트 전략

### 13.1 Founder Layer

- 메신저 메시지 100건 → 의도 분류 정확도 ≥ 90% (note/insight/concern/tacit)
- URL 자동 fetch → 요약 → external/ 캐시 저장 회귀
- Tacit elicitation 면담 2턴 완결률 ≥ 80%

### 13.2 MCP

- Notion MCP — 페이지 1개 등록 → 5분 내 캐시 갱신 확인
- Obsidian MCP — vault read-only 동기화
- OpenAPI → FastMCP — 가짜 API 등록 → tool 1개 호출
- Allowlist 위반 시 호출 차단
- 디스크립션 변경 시 (rug-pull) 재승인 prompt
- L3 sanitize — 알려진 prompt injection 패턴 50건 차단

### 13.3 통합

- 에이전트 실행 시 founder concern 1개가 _handoff에 주입
- 동일 질의에 대한 5회 실행 응답 분기 < 20%
- 같은 지식 노드를 5 agent가 읽어도 결과 일관

## 14. Out of scope (v1.2에서 다루지 않음)

- **자체 그래프 DB 구축** — Mem0/Zep/Cognee 등 어댑터로만 통합. 자체 backend 구현 안 함
- **벡터 임베딩 자체 인덱스 보관** — 그래프 백엔드 어댑터의 책임
- **온톨로지 시각화 GUI** — v1.1 대시보드 상호작용의 Knowledge Explorer 탭으로 일부 시각화. 풀 그래프 편집은 v2.0+
- **MCP 서버 자체 제작** — 우리는 클라이언트만. SoloSquad-as-MCP-server (외부에 우리를 노출)는 v2.0
- **암묵지의 자동 추출 후 자동 적용** — 면담은 자동, 적용은 항상 사용자 confirmation
- **타 사용자 지식 공유** — single-tenant 가정. cross-user 공유는 v2.0 커뮤니티
