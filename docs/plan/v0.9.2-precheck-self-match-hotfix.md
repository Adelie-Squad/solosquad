# v0.9.2 — Uninstall Precheck Self-Match Hotfix (Windows)

> v0.9.1 publish 직후 사용자 머신(`C:\Dev\bv-ai-native-po`)에서 `solosquad uninstall`이 봇·스케줄러가 실제로 안 도는데도 `bot/schedule appears to be running (pid X, Y)`로 차단. PID가 매 호출마다 바뀌는 점이 단서가 되어, `src/lifecycle/precheck.ts:detectLivePids`가 **Windows WMI 쿼리에서 자기 자신(powershell.exe)을 매칭**하던 버그를 발견. 한 줄 fix + 회귀 catcher 1건. *v0.8.x 시리즈에서 자리잡은 "문제 발견 → patch" 패턴 유지*.

**로드맵:** v0.9.1 Model B → **v0.9.2 precheck hotfix** → (v0.9.3 슬롯 유보) → v1.0 정식 출시

**의존:** v0.7 `src/lifecycle/precheck.ts:detectLivePids` (cross-platform PID 검출), v0.8.5 `prepublishOnly` docs-check (4 docs gate)

**기준 버전:** v0.9.2. 새 기능 0, 인프라 0, *발견된 회귀 직접 수정* 1건 + 같은 회귀 catcher 1건.

**분류:** Bug fix (Windows-only). 사용자 노출 표면적 변화 0 — 정직성 차단이 *false positive 없이* 작동.

## 1. Incident — v0.9.1 release 직후 발견

### 1.1 사용자 경험

`C:\Dev\bv-ai-native-po` 워크스페이스(v0.4 layout, archived 사용)에서 `solosquad uninstall` 실행:

```
✗ solosquad bot/schedule appears to be running (pid 50152, 33108).
  Stop these processes first, or rerun with --force.
```

`Stop-Process -Id 50152, 33108 -Force` 시도 → "프로세스 식별자를 찾을 수 없습니다". 즉 **차단 시점에 이미 존재하지 않는 PID**.

재시도하면 **다른 PID**로 같은 차단:
```
(pid 57064, 55628)
(pid 8972, 65692)
```

봇 정지 후 (`solosquad bot` Ctrl+C로 graceful shutdown 확인) 재실행해도 같은 양상. PID가 매번 바뀌는 점이 결정적 단서.

### 1.2 진단 절차

```powershell
Get-CimInstance Win32_Process | Where-Object { $_.CommandLine -match 'solosquad' } |
  Select-Object ProcessId, ParentProcessId, CommandLine | Format-List
```

→ **반환 0건**. 즉 실제 solosquad 패턴 매칭 프로세스 없음에도 precheck는 PID를 반환.

검출 함수가 사용하는 쿼리 본문을 그대로 단독 실행:

```powershell
& powershell -NoProfile -Command "Get-CimInstance Win32_Process | Where-Object { `$_.CommandLine -match 'solosquad' -and `$_.CommandLine -match '(bot|schedule|run-routine)' } | Select-Object ProcessId, Name, CommandLine | Format-List"
```

→ 결과:
```
ProcessId   : 30420
Name        : powershell.exe
CommandLine : "C:\WINDOWS\System32\WindowsPowerShell\v1.0\powershell.exe" -NoProfile -Command
              "Get-CimInstance Win32_Process | Where-Object { $_.CommandLine -match 'solosquad'
              -and $_.CommandLine -match '(bot|schedule|run-routine)' } | ..."
```

**자기 자신만 매칭**. 원인 확정.

## 2. Root cause

### 2.1 메커니즘

`src/lifecycle/precheck.ts:180-207` `detectLivePids()` 의 Windows 분기:

```typescript
const out = execSync(
  `powershell -NoProfile -Command "Get-CimInstance Win32_Process |
    Where-Object { $_.CommandLine -match 'solosquad' -and
                   $_.CommandLine -match '(bot|schedule|run-routine)' } |
    Select-Object -ExpandProperty ProcessId"`,
  { encoding: "utf-8", stdio: ["ignore", "pipe", "ignore"] },
);
```

execSync는 새 powershell.exe를 spawn. 그 프로세스의 `Win32_Process.CommandLine` 필드는 위 -Command 인자 *문자열 전체* 를 포함. 따라서:

| 술어 | 결과 |
|---|---|
| `$_.CommandLine -match 'solosquad'` | **참** — 쿼리 본문에 'solosquad' 리터럴 존재 |
| `$_.CommandLine -match '(bot|schedule|run-routine)'` | **참** — 쿼리 본문에 alternation 리터럴 존재 |

→ 쿼리 자신이 결과에 포함. WMI는 enumeration 도중 스냅샷한 모든 프로세스를 검사하므로 자기 자신도 후보.

### 2.2 기존 process.pid 필터의 한계

`precheck.ts:193`:
```typescript
.filter((n) => Number.isInteger(n) && n > 0 && n !== process.pid);
```

`process.pid`는 **Node 프로세스 PID**. spawn된 powershell.exe의 PID는 별개라 필터를 통과. execSync가 매번 새 powershell을 띄우므로 PID도 매번 다름 — 사용자가 본 "PID가 바뀐다" 증상의 원인.

### 2.3 POSIX 경로(`pgrep -f`) 무영향 확인

```typescript
const out = execSync(
  `pgrep -f "solosquad (bot|schedule|run-routine)" || true`, ...
);
```

`pgrep -f`는 패턴을 정규식으로 컴파일해 각 프로세스 command line에 매칭. pgrep 자신의 command line은 `pgrep -f "solosquad (bot|schedule|run-routine)"`. 이 문자열을 정규식 `solosquad (bot|schedule|run-routine)`으로 매칭 시도하면 — `solosquad` 직후가 *공백+`(`* 로 시작, 그 다음이 `b/s/r` 중 하나가 아니라 `(` 이므로 **alternation 매칭 실패**. 즉 pgrep는 self-match 안 함.

→ 본 hotfix는 **Windows-only**.

## 3. Fix — `Name` 가드 1줄

### 3.1 패치

```diff
- Get-CimInstance Win32_Process |
-   Where-Object { $_.CommandLine -match 'solosquad' -and
+ Get-CimInstance Win32_Process |
+   Where-Object { $_.Name -eq 'node.exe' -and
+                  $_.CommandLine -match 'solosquad' -and
                  $_.CommandLine -match '(bot|schedule|run-routine)' }
```

`$_.Name -eq 'node.exe'`를 **첫 술어**로 배치. PowerShell `-and`는 short-circuit이라 powershell.exe는 첫 조건에서 즉시 false로 탈락, regex match 단계에 도달하지 않음. 실제 `solosquad bot` (Node 프로세스, Name='node.exe')는 세 술어 모두 통과.

### 3.2 대안 비교

| 대안 | 채택 여부 | 사유 |
|---|---|---|
| **`$_.Name -eq 'node.exe'` 가드** | ✓ 채택 | 1줄, 의도가 명확, 실제 패턴 매칭(`solosquad bot` 등은 항상 node.exe)과 정합 |
| 패턴을 변수로 분리해 command line literal 회피 | ✗ | 변수 정의 자체가 command line에 literal로 들어감. 무의미 |
| base64 encoded -EncodedCommand 사용 | ✗ | 디코드된 본문은 다시 literal. 메모리상에서만 회피되고 Win32_Process.CommandLine은 인코드된 base64라 매칭 안 되긴 함. 단, 가독성·디버깅 비용 ↑ |
| 임시 .ps1 파일 작성 후 실행 | ✗ | I/O 추가, 정리 책임, 가독성 ↓. 같은 효과는 Name 가드로 달성 |
| `tasklist /v` + 텍스트 grep | ✗ | tasklist의 CommandLine 컬럼이 항상 채워지지 않음. 신뢰성 ↓ |
| process.ppid 같은 부모 관계 활용 | ✗ | execSync된 powershell의 부모가 Node. 자기 자신 매칭과 무관한 다른 차원의 정보. 본 버그 해결에 부적합 |

### 3.3 fix가 부수 효과를 일으키지 않는 이유

`solosquad bot|schedule|run-routine`은 모두 node CLI 진입점. `Get-Process -Name node` 시 항상 `node.exe`로 나타남. Windows에서 npm 글로벌로 설치된 solosquad의 shim(`solosquad.cmd`, `solosquad.ps1`)도 결국 `node solosquad.js bot` 형태로 node를 invoke하므로 Win32_Process.Name 컬럼은 `node.exe`. → 가드가 실제 검출 대상을 놓치지 않음.

## 4. Regression catcher

### 4.1 검출 설계 — *3회 호출 동일성*

`test/lifecycle-precheck.test.ts`:
```typescript
const r1 = _precheckInternals.detectLivePids().slice().sort();
const r2 = _precheckInternals.detectLivePids().slice().sort();
const r3 = _precheckInternals.detectLivePids().slice().sort();
assert.deepEqual(r1, r2);
assert.deepEqual(r2, r3);
```

### 4.2 왜 *empty array* 가 아니라 *동일성* 인가

- "결과가 비어 있어야 한다"는 단언은 *개발자가 dogfood로 봇 띄우고 있을 때* false fail. CI에서는 통과해도 로컬에서 노이즈.
- "결과가 일정하다"는 단언은 *진짜 봇이 돌 때*는 stable PID로 통과, *봇이 안 돌 때*는 empty array로 통과. self-match 버그가 살아나야만 fail.

### 4.3 catcher가 못 잡는 시나리오

- WMI가 race condition으로 한 번은 powershell 잡고 다음 번엔 못 잡는 *flaky* 경우: r1=[X], r2=[], r3=[Y]처럼 차이가 나면 잡힘. 다만 셋 다 정확히 같은 phantom PID로 나오는 (불가능에 가까운) 경우는 통과. 실측상 0%.
- detect 함수 자체가 throw하는 경우: precheck.ts:204 `try { ... } catch { return []; }` 가 삼키므로 catcher는 [] 3회로 통과. 단 실제 사용 경로(`solosquad uninstall`)도 동일하게 [] 처리되어 false negative 없음.

## 8. 본 plan이 박제하는 교훈

**execSync로 shell 명령을 invoke할 때 그 명령 인자 문자열 자체가 검출 대상의 일부가 될 수 있음**. 특히:

- `ps -ef | grep foo` 같은 grep 명령은 자기 자신을 잡으므로 `grep -v grep` 또는 `pgrep`을 쓴다 — Unix 진영의 well-known idiom.
- Windows에서 `Get-CimInstance Win32_Process | Where-Object { ... }` 패턴은 같은 함정. 본 hotfix의 핵심.

→ *프로세스 검출 함수를 추가/수정할 때는 항상 "이 함수의 invocation 자체가 검출 패턴에 매칭될 수 있는가" 자문*. 동일성·empty 양쪽 케이스로 회귀 catcher 동반.

후속 patch에서 비슷한 invocation을 추가하게 되면 본 plan 참조.
