# v1.0.1 — Discord `ready` deprecation + repo `role` removal + multi-repo `@<slug>` routing

> v1.0.0 정식 출시 직후 첫 patch. 발견된 dependency-level deprecation 1건 + 사용자 가치에 어긋나는 onboarding friction 1건을 한 릴리스로 흡수. 같이 해소되는 의미적 빚: *"솔로스쿼드는 한 agent가 여러 repo를 다룬다"* 는 포지셔닝과 `role=main` 단일 default repo 가정 사이의 모순. 본 plan은 그 결심의 박제 + publish 절차.

**로드맵 위치:** v1.0.0 정식 출시 → **v1.0.1 첫 patch** → v1.1 대시보드 → v1.2 지식 온톨로지

**의존:**
- v1.0.0 `docs/api-stability.md` (Effective as of 2026-05-21) — 본 patch의 schema 변경 0건 제약 + CLI surface freeze 준수
- v0.3 `assets/orchestrator/SKILL.md` (PM mode) — 본 patch가 §"Multi-Repo Intent" 추가
- discord.js 14.26.4 (deprecated `ready` 이벤트 알림 소스)

**기준 버전:** v1.0.1. 새 schema 0, 새 CLI 명령 0. *발견된 deprecation 1건 수정* + *onboarding friction 1건 제거* + *그 자리 메우는 라우팅 메커니즘 1건 추가*.

**분류:** Patch — backward-compat 보장 + breaking 0.

## 1. Incident — v1.0.0 publish 직후 발견

### 1.1 표면 증상

`solosquad bot` 시작 시 콘솔 출력 마지막 줄:

```
(node:1606) DeprecationWarning: The ready event has been renamed to clientReady
to distinguish it from the gateway READY event and will only emit under that name
in v15. Please use clientReady instead.
(Use `node --trace-deprecation ...` to show where the warning was created)
```

봇은 정상 동작. 단일 1회 출력 (Node `process.emitWarning` 기본 동작). v15 업그레이드 전까지는 silent failure 아님.

### 1.2 근본 원인 — 사실 확인

`node_modules/discord.js/src/client/websocket/WebSocketManager.js:375-401` 의 `triggerClientReady()`:

```js
triggerClientReady() {
  this.status = Status.Ready;
  this.client.readyTimestamp = Date.now();

  /** @deprecated Use {@link Client#event:clientReady} instead. */
  if (this.client.emit('ready', this.client) && !deprecationEmitted) {
    deprecationEmitted = true;
    process.emitWarning(
      'The ready event has been renamed to clientReady to distinguish it ' +
        'from the gateway READY event and will only emit under that name ' +
        'in v15. Please use clientReady instead.',
      'DeprecationWarning',
    );
  }
  this.client.emit(Events.ClientReady, this.client);
  this.handlePacket();
}
```

- 라이브러리는 *두 이벤트를 모두* emit (`'ready'` + `Events.ClientReady`).
- `'ready'` 리스너가 *하나라도* 붙어 있으면 첫 1회 경고.
- `deprecationEmitted` 모듈-전역 플래그로 인해 봇 lifetime 내 1회 한정.
- **v15에서 `'ready'` alias 완전 제거.** 그 시점에 우리 코드 `client.on('ready', ...)` 는 *호출되지 않음* — 봇이 로그인되지만 ready 핸들러가 안 깨어나는 silent failure 가 됨.

### 1.3 우리 쪽 트리거

`src/messenger/discord-adapter.ts:91` 단일 라인:

```ts
client.on("ready", async () => { ... })
```

프로젝트 내 다른 `'ready'` 리스너 없음. fix 표면적은 1줄 + 타입 안전성을 위한 `Events` 임포트 1줄.

## 2. v1.0.1이 함께 흡수하는 2번째 문제 — repo `role` 친화도 부채

### 2.1 발견 경위

discord deprecation fix 작업 중 사용자가 onboarding flow 회고 결과 *"`solosquad init` 의 `Role:` 프롬프트가 무엇을 위한 것인지 모르겠다"* 제기. `role` 필드의 실제 *load-bearing* 소비처를 정직히 확인:

| 위치 | 용도 | 사용자 라우팅 관여? |
|---|---|---|
| `src/bot/workflow-resolver.ts:79` (제거 전) `find(c => c.role === "main")` | 스케줄러 routine 의 default cwd 결정 (08:00 morning brief 등) | **❌ 안 함** — 사용자 메시지 routing 과 무관 |
| `src/cli/init.ts:893` (제거 전) `console.log("✓ ${slug} (${role}) registered")` | onboarding 성공 메시지 라벨 | ❌ 표시만 |
| `src/cli/add-repo.ts` `confirmRole()` | 등록 시 prompt | ❌ prompt 자체가 자동 결정 없음 |

→ **사용자가 응답해야 하는 질문이었지만 사용자 메시지 라우팅에는 일절 관여하지 않았다.** PM (LLM) 이 `<org>/repositories/` 의 repo 목록을 보고 매번 target 을 추론하는 구조 (`src/bot/index.ts:56` 주석: *"PM session cwd is fixed at the org root … target_repo cwd branching happens inside subagent prompts"*).

### 2.2 SoloSquad 포지셔닝과의 모순

가치 명제: *"한 agent 가 여러 repo 를 다룬다"*. 그런데 `role=main` 필드는 *"하나를 default 로 미리 지정"* 이라는 가정 — **다중-repo 가 일급 시민이라는 포지셔닝과 정면 충돌**. role 필드를 두면:

- 사용자가 어떤 repo 가 "main" 인지 매번 자문 (현실에선 다 main 임)
- "frontend / backend / data / infra / docs" 값은 *어떤 라우팅에도 안 쓰임* — pure cargo cult
- 다중-repo 사용자에게 "main 만 자동 인식" 이라는 잘못된 신호

### 2.3 그러면 다중-repo 의도는 어떻게 풀까 — 외부 사례 조사

| 패턴 | 대표 도구 | chat-bot 적합도 |
|---|---|---|
| CWD/위치 | gh CLI, git, Aider, Claude Code | ❌ chat 엔 cwd 없음 |
| 세션 시작 시 선택 (lock per-conversation) | Devin, OpenHands, Copilot Workspace | ✅ thread/session 단위 |
| 채널 ↔ repo 바인딩 | **GitHub Slack app** (`/github subscribe owner/repo`), Linear↔GH | ✅✅ chat-native, sticky |
| **명시 mention tag** | **Nx** (`nx run my-app:build`), Bazel (`//path:target`), GitHub Slack `@<repo>` | ✅ 비용 0, 결정적 |
| recency heuristic | (주류 도구 없음 — 너무 깨지기 쉬워서) | ❌ |
| LLM 추론 routing | (주류 도구 없음 — 너무 비싸고 lossy) | ⚠ fallback 한정 |
| sticky "current repo" | VS Code multi-root, Cursor | ⚠ AI 가 바꾸려 들면 깨짐 |

핵심 인사이트 (GitHub Slack 패턴 + Nx 라벨 패턴 공통점): *"메시지/명령 안에 명시 태그"* 가 **routing 시점에 LLM 0 호출**. 비용 0, 결정적, 사용자가 손가락 하나.

### 2.4 v1.0.1 채택안

- **(a) `@<slug>` mention syntax** — `src/bot/mention-parser.ts` 신규 모듈. 사용자가 `@landing-site 히어로 수정` 처럼 명시하면 bot pre-processor 가 `[target_repo:landing-site]` 마커를 메시지 앞에 주입해 PM 에게 전달. **regex 매칭 + 등록 slug 셋과 교집합** — Discord 사용자 핑 (`<@123456789>`) / 타이핑 오류는 silently drop 되어 false positive 0.
- **(b) PM SKILL.md §Multi-Repo Intent 추가** — 마커 honor 우선순위 1, 단일 repo 자동 (사용자 본 케이스), 모호하면 PM 이 짧은 clarifying question 1번. **silent guessing 금지**.
- **(c) workflow-resolver `role=main` 분기 제거** — 첫 등록 repo 로 자연 fallback (스케줄러 default cwd 결정에만 사용). user-driven routing 은 이제 PM 레벨에서만.
- **(d) `RepoYaml.role` `@deprecated` 주석** — schema 필드 자체는 backward-compat 위해 *유지*. v1.0 api-stability §schema "2-minor read window" 정책에 따라 hard 제거는 v2.0 슬롯.
- **(e) `--role` CLI flag** — power-user override 로 *유지하되 deprecation warning* (warnDeprecated). flag surface 제거 = api-stability CLI freeze 위반이므로 v2.0 슬롯.

채택하지 않은 안:
- **채널 ↔ repo 바인딩** (`works-w1n-landing` 컨벤션 또는 `/repo use <slug>`) — 사용자 가치 크지만 user-registry / 채널 자동 생성 로직 확장 필요, v1.0.1 patch 범위로 과함. v1.x minor 슬롯 후보.
- **세션 sticky context** — 활성 workflow 가 있을 때는 이미 `target_repo` 가 stage 단에서 sticky. workflow 없는 ad-hoc 대화의 sticky 는 v1.x 슬롯.
- **LLM 추론 only** — *모호할 때 PM 이 묻기* 패턴으로 충분. silent inference 는 GitHub Slack / OpenHands 사례 모두 "사용자 confusion 의 주범" 으로 박제.

## 4. Backward compatibility — v1.0.0 사용자

### 4.1 데이터

- 기존 `repo.yaml` 파일들의 `role:` 필드 그대로 read. workflow-resolver 는 이제 그 값을 안 보지만 파일에 남아 있어도 무해.
- 신규 등록 `repo.yaml` 도 `role: main` 으로 채워짐 (default). schema 호환 유지.
- 데이터 손실 0, breaking 0.

### 4.2 runtime

- `solosquad bot` — deprecation 경고 사라짐. 사용자가 보던 마지막 출력 1줄 감소.
- `solosquad init` / `solosquad add repo` — `Role:` 프롬프트 사라짐 (1단계 단축). 그 외 동일.
- 스케줄러 routine — `role=main` 인 repo 가 *있던* 사용자라면, 그 repo 가 *우연히* `readdirSync` 순서상 첫번째이면 동일 cwd. 아니면 다른 repo 가 cwd 로 잡힘. 스케줄러 routine 은 org-level (morning brief / weekly review) 이므로 영향 적음. 다중 repo 사용자가 *특정 repo 가 default 여야 한다* 라고 의존하던 경우 v1.0.1 plan §4.3 가이드로 안내.
- PM 메시지 routing — 종전대로 PM (LLM) 이 결정. 추가로 `@<slug>` 명시 채널 신설. 사용자 본 케이스에서 PM 이 모호하다 판단하면 *1번* 의 clarifying question 발사.

### 4.3 다중 repo 사용자 가이드

기존 `role=main` repo 가 스케줄러 routine 의 default cwd 로 *돌아오기를* 의존하던 경우:
- 가장 안전한 명시: `@<slug>` 를 사용자의 routine 응답 안에 적기. PM 이 그 시점부터 해당 repo 로 routing.
- 영구 명시 (v1.x): 채널 ↔ repo 바인딩 슬롯 (v1.x minor) 에서 `/repo use <slug>` 슬래시 도입 예정.
