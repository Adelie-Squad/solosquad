> v0.3~v0.5에서 누적된 실전 데이터(_events.jsonl, results.tsv, 사용자 author 루프 패턴)를 바탕으로 4종 디폴트 워크플로우(PMF Discovery / Feature Expansion / Rebranding / Rapid Prototype)를 손본다. 추가로 **누적 메모리의 검색 가능성**과 **반복 패턴의 SKILL 자동 추출** 두 인프라를 도입 — v1.0 정식 출시 전 마지막 안정화 슬롯.

**로드맵:** v0.1 크로스 플랫폼 → v0.2 안전/보안 → v0.3 PM 모드 → v0.4 자율 실행 → v0.5 워크플로우 메이커 → **v0.6 디폴트 워크플로우 + 메모리 아카이브** → v1.0 정식 출시 → v1.1 대시보드 → v1.2 지식 온톨로지

**확정 시점:** v0.5 출시 약 4~6주 후. 최소 사용자 1명 × 4종 모두 1회 이상 + author 루프 산출물 10건 이상 누적 시점. (단 §4 FTS5 아카이브, §5 trajectory 추출 인프라는 데이터량과 무관하게 진행 가능)

### 2.1 Team = Domain 통합 (워크스페이스 폴더 재편)

별도 검토에서 "직교 도메인 태그(`domains: [pricing, ...]` frontmatter)" 안이 제기됐으나, *team 분류*와 *도메인 태그* 두 레이어를 병행 관리하는 부담이 ROI에 부합하지 않는다. v0.6은 다른 방향 채택: **team 폴더 자체가 도메인이라는 원칙을 명시**하고, 도메인 지식은 같은 폴더 내부에서 공유한다.

#### 현행 구조의 문제

```
agents/{team}/{agent}/SKILL.md           ← 25 agent
agents/_teams/{team}/TEAM_KNOWLEDGE.md   ← 같은 team의 공유 지식, 별도 hierarchy
```

같은 team 자산이 두 평행 hierarchy에 분리. scanner가 `agents/*/*/SKILL.md` 패턴으로 iterate할 때 team-level 파일을 빼기 위한 기술적 회피였지만:

- team 업데이트 시 두 폴더를 모두 만져야 함
- 새 agent 추가 시 team의 공유 컨텍스트 위치가 즉시 보이지 않음
- v0.5 analyzer 'role' 라벨 산출물 destination이 모호 (`agents/{team}/` 인가 `_teams/{team}/` 인가)

#### Before / After

```
BEFORE (v0.2.4)
assets/agents/
├── strategy/
│   ├── pmf-planner/SKILL.md
│   └── feature-planner/SKILL.md
├── growth/
│   └── gtm-strategist/SKILL.md
└── _teams/                                ← 평행 hierarchy
    ├── strategy/TEAM_KNOWLEDGE.md
    └── growth/TEAM_KNOWLEDGE.md

AFTER (v0.6)
assets/agents/
├── strategy/                              ← team = domain
│   ├── KNOWLEDGE.md                       ← 같은 폴더 공유 지식
│   ├── pmf-planner/SKILL.md
│   └── feature-planner/SKILL.md
└── growth/
    ├── KNOWLEDGE.md
    └── gtm-strategist/SKILL.md
```

#### Scanner 룰 단순화

- `agents/{team}/{agent}/SKILL.md` → agent (기존)
- `agents/{team}/KNOWLEDGE.md` → team(=domain) 공유 지식. **같은 team의 모든 agent가 spawn 시점에 자동 컨텍스트로 로드** (보고서 §1.2 JIT Injection — KNOWLEDGE는 같은 도메인 agent 호출 시에만 inject)
- `_teams/` 디렉토리 완전 제거

#### Spawn 시점 컨텍스트 조립 변화

```
spawn(business-strategist, ...)
  └─ assemble:
       [1] agents/strategy/KNOWLEDGE.md          ← 신규 — team(domain) 공유 지식
       [2] agents/strategy/business-strategist/SKILL.md  ← agent 정체성
       [3] org/memory/, org/workflows/<id>/, handoff
       [4] (target_repo 있을 때) repo 컨텍스트
```

같은 team 내 agent끼리 공유 craft·정의·anti-pattern을 KNOWLEDGE.md 1곳에서 관리하면 *agent SKILL.md가 짧아지고*, 새 agent 추가 시 team 일관성이 자동 유지.

#### 마이그레이션 (1.5.0 → 1.6.0)

`1.5.0-to-1.6.0.ts`에 추가:
- 4개 `agents/_teams/{team}/TEAM_KNOWLEDGE.md` → `agents/{team}/KNOWLEDGE.md` 이동
- `agents/_teams/` 디렉토리 제거
- 파일 내용 변경 없음 (비파괴)
- Scanner 코드 1줄 추가 (`KNOWLEDGE.md` 파일명 인식)

#### 팀 분할은 별도 결정

현재 4 team(Strategy / Growth / Experience / Engineering)이 이미 도메인 단위 역할을 한다. 더 세분화(예: Strategy → PMF / GTM / Pricing)는 사용자 사용 데이터를 본 §2 회고 결과에 따라 결정. **본 결정은 *폴더 재편*만 다루고 *팀 분할*은 별도.**

#### 효과 요약

- 직교 태그(`domains: [...]`) 추가 없이 도메인 navigation 가능 — 폴더 구조 자체가 도메인
- 같은 team의 agent들이 같은 KNOWLEDGE.md 자동 공유
- v0.5 analyzer 'role' 라벨 destination 명확화 (해당 team 폴더 안 — KNOWLEDGE.md 보강 또는 새 agent)

### 2.2 Organization Layer Specialization (agent-profile + domain)

같은 역할의 specialist가 조직마다 톤·강조점·금지사항이 다른 문제를 **SKILL override 대신 가벼운 modifier**로 해결한다. 25 specialist SKILL은 워크스페이스에 1세트로 유지, 조직별 차이는 org 위계의 3개 신설 자산이 흡수.

#### 신설 자산

```
<org>/
├── .org.yaml                  (기존)
├── core/                      ★ 신설 — 조직 철학/톤 (workspace core override)
│   ├── PRINCIPLES.md
│   └── VOICE.md
├── agent-profile.yaml         ★ 신설 — 25 agent 조직별 modifier
├── domain/                    ★ 신설 — 조직 도메인 지식 폴더
│   ├── market.md
│   ├── customers.md
│   └── product.md
├── memory/                    (기존)
├── workflows/                 (기존)
├── repositories/              (기존)
└── slack/ | discord/          (기존)
```

#### agent-profile.yaml 형식

25개 SKILL override가 아닌 **modifier 1파일** (~50줄):

```yaml
defaults:
  tone: conservative
  priorities: [profitability, risk-mitigation]
  excluded_recommendations: ["VC fundraising"]
  # v0.6 P0 — v0.5 paperclip envelope을 org × agent 단위로 일반화 (v0.5는 author 루프에만 적용)
  budget:
    daily_usd: 5            # 이 agent의 일일 합산 호출 비용 cap
    weekly_usd: 25
    on_cap_action: pause    # pause | warn

business-strategist:
  emphasis: "한국 SMB 시장 컨텍스트 우선"
  decision_frame: "현금흐름 > 성장률"

content-writer:
  voice: "professional, no hype"
  ban_phrases: ["혁신적인", "획기적인"]

paid-marketer:
  budget_ceiling_krw: 500000   # 별도 광고 예산 — LLM budget과 무관 (domain semantic)
  budget:                       # v0.6 P0 — LLM 호출 cap override (defaults보다 좁힘)
    daily_usd: 2

# v0.6 P0 — author 루프와 동일 budget 시스템(`src/util/cost.ts` + `src/bot/author-budget.ts` from v0.5 S3)을 agent-spawn 단계에서 재사용.
# 호출 직전 cap 도달 시 spawn 거부 + PM에 "Daily budget 도달 — 내일 다시" 응답.
# 누적 JSONL: <org>/memory/agent-costs.jsonl (author-costs.jsonl과 동형).
```

명시되지 않은 agent는 `defaults`만 적용. 명시된 agent는 `defaults + agent별 섹션` 머지. budget은 agent별 섹션이 *defaults보다 좁은* 값만 허용 (보안 — 사용자 글로벌 cap을 agent가 우회 금지).

#### Spawn 시점 컨텍스트 조립 (보고서 §1.2 JIT Injection 확장)

```
spawn(business-strategist, org=Acme, repo?)
  └─ assemble:
       [1] assets/knowledge/                  ← workspace 공용 지식 (§2.3)
       [2] agents/strategy/KNOWLEDGE.md       ← team(=domain) 공유 지식 (§2.1)
       [3] agents/strategy/business-strategist/SKILL.md  ← agent 정체성 (불변)
       [4] <org>/core/                        ← 조직 철학/톤 (NEW)
       [5] <org>/agent-profile.yaml의 business-strategist + defaults 섹션 (NEW)
       [6] <org>/domain/                      ← 조직 도메인 지식 (NEW)
       [7] <org>/memory/, <org>/workflows/<id>/_handoff.md
       [8] (target_repo 있을 때) repo 컨텍스트
```

workspace SKILL은 한 줄도 변경 없음. 조직 색채는 [4][5][6]에서만 들어옴.

#### v0.5 analyzer 'role' destination 정정

| v0.5 (이전) | v0.6 (정정) |
|---|---|
| Workspace agent (25개 중 머지 또는 신규) | **`<org>/agent-profile.yaml`** 의 해당 agent 섹션 (대부분) — 진짜 보편 craft면 workspace KNOWLEDGE.md 또는 신규 agent (드묾) |

리포 스킬이 워크스페이스까지 2단계 위로 올라가는 어색함이 해소됨 — 조직 위계에서 머묾.

#### 마이그레이션

`1.5.0-to-1.6.0.ts`에 추가:
- 각 org에 `core/PRINCIPLES.md`, `core/VOICE.md` 빈 스텁 생성 (사용자 사후 편집)
- 각 org에 `agent-profile.yaml` 빈 스텁 생성 (defaults만 있는 minimal)
- 각 org에 `domain/` 디렉토리 + README.md 생성
- 기존 `<org>/memory/`, `<org>/workflows/` 손대지 않음

비파괴. 사용자가 새 폴더에 무엇도 안 적어도 spawn은 기존 워크스페이스 SKILL만으로 작동.

**v0.6 마이그레이션 dry-run + 검수 절차 (v0.5 S5 패턴 차용, v0.6 P2):**

`0.5.0-to-0.6.0.ts`는 2-pass 구조:

- Pass 1 (자동, dry-run 모드 가능): 휴리스틱 기반 재배치 시뮬레이션 + 보고서 `<org>/memory/migration-2026-XX-dryrun.md` 생성. 이동 N건 / 사람 검토 M건 / 변경 0건 분류. CLI는 `solosquad migrate --dry-run` 으로 호출 — 파일 시스템 변경 없음, 보고서만 생성
- Pass 2 (사람 검수): 사용자가 보고서 검토 + `solosquad migrate --apply --confirm` 실행. `--confirm` 플래그 없으면 거부 (실수 방지)
- 휴리스틱 fail-soft가 발생한 항목(SKILL 본문에서 "조직 색채" 부분 추출 실패 — §2.2 위 `pending_v0.6_redestination` 절차의 매칭 0건 케이스)은 *반드시* `human_review_required: true` 마킹. 자동 적용 금지

이유: v0.5 §6.2의 role/domain 라벨이 v0.6에서 `agent-profile.yaml`·`domain/`으로 재배치되는데, 본문에서 "조직 색채" 부분 추출은 휴리스틱이라 100% 정확 불가. v0.5 S5에서 검증된 dry-run + 사람 보강 패턴(`anthropics/skills` corpus round-trip + 사람 `description` 보강)을 그대로 적용 — 첫 dry-run에서 사용자가 N건을 검토하고 나면 Pass 2가 안전하게 자동.

#### v0.5 ledger 수신측 자동 재분류 (v0.5 §6.2 게이트의 v0.6 측 절차)

v0.5에서 임시 적재된 role/domain 자산을 v0.6 폴더 토폴로지로 자동 이동. v0.5 plan §6.2 약속의 수신측.

1. `<workspace>/<org>/.solosquad/analysis-ledger.yaml`을 스캔 — v0.5에서 분석한 모든 repo의 ledger 통합 로드
2. `pending_v0.6_redestination: true` 항목만 대상으로 분기:
   - `classification: role` → 임시 destination(`~/.solosquad/agents/{team}/{agent}/SKILL.md`) 본문에서 "조직 색채" 부분 추출 → `<org>/agent-profile.yaml`의 해당 agent 섹션 머지. 추출 휴리스틱: 본문 H2/H3 중 톤·금지·우선순위·강조 키워드(`tone`, `voice`, `priority`, `ban`, `excluded`, `emphasis`) 매칭 섹션. 매칭 0건이면 사람 검토 필요로 마크.
   - `classification: domain` → `<org>/memory/domain/*.md` → `<org>/domain/*.md` 이동(`fs.renameSync`)
3. ledger 갱신: `pending_v0.6_redestination: false` + `redestinated_at: <iso>` + `redestination_method: auto | human-review-required`
4. `<org>/memory/migration-2026-XX-redestination.md` 보고서 생성 — 이동 N건 / 사람 검토 필요 M건 / 변경 0건(이미 final이었음) K건
5. 검증: spawn 1회 mocked 실행 → `agent-profile.yaml`의 modifier가 결과에 반영되는지(§7 E2E 회귀와 동일)

비파괴 보장: 원본 파일은 *이동*만, 본문 변경 없음. v0.5 ledger가 없는 워크스페이스(v0.5를 건너뛰고 v0.6 직접 설치)는 본 단계 skip — `applied_at` 표시.

마이그레이션 스크립트 파일: `src/migrations/scripts/0.5.0-to-0.6.0.ts` (이 절차 + §2.1·§2.2·§2.3·§2.4 + §4 archive.sqlite 초기화 통합).

### 2.3 Workspace Knowledge Layer

사용자가 누적하는 *역할 불변 craft + 의사결정 프레임워크 + 도메인 용어집*을 별도 자산화. agent SKILL.md를 짧게 유지하고, 도메인 지식과 craft를 분리.

#### 신설 자산

```
assets/knowledge/                ★ 신설 (npm 번들 — 빈 또는 가이드만)
└── ...

.solosquad/knowledge/            ★ 사용자 워크스페이스
├── decision-frameworks/
│   ├── lean-canvas.md
│   ├── porter-five-forces.md
│   └── ltv-cac.md
├── glossary.md                  ← 도메인 용어집
└── references/
    └── ...
```

#### Path 해결 (기존 패턴 확장)

paths.ts에 `getKnowledgeDir()` 추가 — `.solosquad/knowledge/ → assets/knowledge/` 순. 기존 `getAgentsDir()`·`getCoreDir()` 와 동일 패턴.

#### Spawn 시점 활용

모든 specialist spawn 시 [1] 위치에서 자동 로드 (보고서 §1.2 JIT). 단 *키워드 매치 기반 selective load* — 전체 knowledge를 모든 spawn에 inject하면 컨텍스트 폭증. agent-router가 사용자 메시지와 task description의 키워드로 관련 knowledge 파일만 선택.

#### 외부 안에서 차용한 "Single Source of Truth" 컨셉

직전 검토의 외부 안 §3 `repo-data-context` 발상 — 데이터·지표·시맨틱 레이어를 별도 자산으로 보는 시각. SoloSquad에서는 *물리 리포 분리 거부*, 대신 **`assets/knowledge/` (workspace 공용) + `<org>/domain/` (조직별)** 두 폴더로 같은 목적 달성. 외부 안의 Vector+Graph DB 인프라 없이 markdown + FTS5(§4)로 동일 효과.

#### 마이그레이션

`1.5.0-to-1.6.0.ts`에 추가:
- `.solosquad/knowledge/` 디렉토리 + README.md 생성 (사용자 사후 편집)
- bundled `assets/knowledge/`에 시작 가이드 1장 (How to author your knowledge layer)

### 2.4 핸드오프 표준화에 3패턴 어휘 적용 (보고서 §3 차용)

`_handoff.md`가 비표준인 이유는 stage마다 **협업 패턴이 다른데도 같은 템플릿을 강요**하기 때문. 보고서의 멀티에이전트 협업 3패턴 어휘로 stage 유형을 분류한 뒤 패턴별 슬라이스 스키마를 정의:

| 협업 패턴 | 해당 stage 예시 | _handoff.md 슬라이스 스키마 |
|---|---|---|
| **계층적** (PM → specialist) | 모든 v0.3 spawn | Summary / Artifacts / Decisions / Open Questions (기존 4섹션 유지) |
| **그래프** (조건부 분기 with 공유 상태) | research → planner ↔ data-analyst 양방향, gate 통과까지 루프 | 위 4섹션 + `state_object_diff` (이전 핸드오프 대비 변경 필드만) + `loop_count` |
| **동적 라우팅** (메시지 내용에 따른 라우팅) | content-writer가 brand-marketer 또는 paid-marketer로 분기 | 위 4섹션 + `routing_signal` (어떤 단어/지표가 다음 agent 결정에 영향) |

`templates/handoff.md`는 3 변형을 모두 제공, agent SKILL.md가 자신의 협업 패턴을 frontmatter에 명시(`collab_pattern: hierarchical | graph | dynamic`).

## 3. 패턴 자동 추출 — Hermes Trajectory → Skill (Tier 3.G 승격)

v0.5에서는 *입력 경로만* 제공했던 trajectory → skill 자동 추출을 v0.6에서 **자동 트리거**로 승격한다. 단 ROI 검증을 위해 처음에는 *제안만* — 자동 등록은 v1.0 출시 후 결정.

### 3.1 트리거 조건

```
조건 ALL (보수적 — false positive 회피):
  1. 같은 (agent sequence + workflow template) 조합이 N회(기본 3) 발생
  2. 최근 30일 내
  3. PM이 동일한 명확화 질문을 사용자에게 3회 이상 한 케이스
  4. 모든 결과가 keep (v0.4 results.tsv에서 discard 없음)
```

### 3.2 동작

1. `pm-compaction` routine(v0.3에서 신설된) 야간 실행 중 trajectory 분석 단계 추가
2. 조건 충족 시 `<org>/memory/pm-skills/_suggestions/` 폴더에 draft SKILL.md 생성 — v0.5 frontmatter 포맷 준수, `confidence: 0.7`, `source: auto-extracted-from-trajectory-<timestamp>` 메타데이터
3. 다음 아침 브리핑에 "지난 30일 동안 X 패턴을 3회 실행하셨습니다. SKILL로 등록할까요?" 1줄 알림
4. 사용자 확정 시 **v0.5 S3 `src/bot/skill-author.ts`의 `applyDraft()` 를 그대로 호출** — 자동 등록 없음 (사람 게이트 유지). 별도 applier 작성 금지 (코드 중복 회피)

**v0.6 P0 — v0.5 backend 공유 명세:**

`trajectory-extractor` 모듈(`src/scheduler/trajectory-extractor.ts`, 신규)은 분석 결과를 `AuthorDraft` 타입(v0.5 S3 정의)으로 변환해 `applyDraft({ workspace, orgSlug, draft, destination })`에 위임. v0.5의 atomic write + `rebuildRoutes()` + 검증 흐름이 그대로 재사용된다. trajectory 추출이 author 루프와 *입력 형태*만 다르고 *출력 파이프라인*은 동일 — v0.5 §3 다이어그램의 "Workflow Maker (메타-skill)" 박스가 v0.6에서 *두 입력*(author / trajectory)을 받는 단일 컴파일러로 확장된다.

### 3.3 ROI 측정 룰

자동 제안된 SKILL이 등록 후 30일 내 **실제 호출되지 않으면 false-positive**로 카운트. 임계치 초과 시 trajectory 분석 비활성. 측정 지표:

- 제안 → 등록 전환율 (target ≥ 50%)
- 등록 후 30일 호출률 (target ≥ 30%)
- false-positive 비율 (target < 30%)

세 지표가 모두 만족하면 v1.0 출시 시 자동 등록 게이트 검토. 아니면 영구히 *제안만*.

### 3.4 Freq 키워드 자동 추천 (v0.6 P1)

§3.1~§3.3은 agent sequence + workflow template 조합을 추출하지만, *키워드 자체*가 v0.5 4채널 라우터에서 한 SKILL의 trigger로 빠져 있을 때는 잡지 못한다. 본 절은 라우터 미스 + author 루프 명확화 질문의 N-gram 분석으로 **기존 SKILL frontmatter의 `triggers.keyword`에 추가할 후보를 제안**한다 — 새 SKILL 생성이 아닌 *부분 적용*.

#### 트리거

- 메모리 N-gram 분석 + 라우터 미스 클러스터링 (§4.6 FTS5 인덱싱의 `route_miss` 이벤트 활용)
- 야간 pm-compaction routine에 freq-keyword-miner 단계 추가 (§3.2 trajectory 추출과 동일 hook)

#### 동작

1. 최근 30일 라우터 미스(`event_type: route_miss` from §4.6) + author-draft 명확화 질문(`<org>/.solosquad/sessions/<user>.author-draft.json` 의 `history[].role=pm` 질문 텍스트)에서 빈출 N-gram 추출 (N=1~3, stop-word 제거)
2. 같은 키워드 미스가 3회 이상 + 의미적으로 유사한 기존 SKILL.md가 있으면(기존 frontmatter `triggers.keyword`·`description`과 토큰 overlap ≥ 2) → 해당 SKILL의 `triggers.keyword`에 추가 **제안**
3. 다음 아침 브리핑에 "키워드 X를 N회 입력하셨지만 매칭 SKILL 없음. 기존 Y SKILL에 등록하시겠습니까? (y/n)" 1줄
4. 사용자 확정 시 v0.5 author 루프의 *부분 적용* — 본문은 한 글자도 안 건드리고 frontmatter `triggers.keyword` 배열만 패치. `applyDraft({ ..., destination: existingSkillPath, mode: "frontmatter-only" })` 신호로 v0.5 S3 코드 재사용

#### 안전장치

- **자동 등록 없음** — 사람 게이트. §3.2 trajectory 추출과 같은 보수성
- 같은 제안을 사용자가 거절하면 그 (keyword, skill) 쌍은 N=30일 동안 재제안 금지(`<org>/memory/freq-suggestions-rejected.jsonl`)
- 의미 유사도 임계 미달이면 *어느 SKILL에도 매칭 안 됨* — 빈 알림 회피
- N-gram 분석은 결정적 (빈도 카운팅). LLM 클러스터링은 v1.2 임베딩 도입과 함께 — §6 OOS

#### ROI

라우터 미스율 < 10% 유지(§7 성공 기준에 반영). 본 §의 자동 추천이 라우터 미스의 ≥ 30%를 회복하면 효과 입증, 그렇지 않으면 N=30일 후 비활성. §3.3 trajectory 추출과 동일한 measure-then-decide 패턴.

## 4. FTS5 Cold Archive — 누적 메모리 인덱싱 (Tier 1.B)

### 4.1 문제

v0.2.4 이후 `memory/routine-logs/*.jsonl`은 append-only로만 자라난다. 30일 시점에 grep 비용이 비현실적이 되고, **agent-router가 과거 결정을 인지 못 하는 게 가장 큰 손실** — 같은 신호를 두 번 분석해도 모름.

### 4.2 도입안 (Hermes 패턴 차용 — 보고서 P/E/T 어휘로 정렬)

| 계층 | 저장소 | 보존 정책 |
|---|---|---|
| **Transient** | PM thread context (Claude session prompt cache) | 5분 TTL — Anthropic prompt cache와 정렬 |
| **Episodic (hot)** | `memory/routine-logs/*.jsonl` (최근 7일) | append-only, grep 가능 |
| **Persistent (cold)** | `memory/archive.sqlite` (FTS5 virtual table) | 무제한, 검색 기반 회상 |

### 4.3 동작

```
매일 00:00 cron — archive-rotate routine
  1. memory/routine-logs/*.jsonl 중 8일 이상 된 항목 추출
  2. SQLite FTS5 가상 테이블에 삽입 (snippet, agent, timestamp, source_routine, json_blob)
  3. 원본 JSONL에서 해당 항목 삭제 (디스크 절약)

agent-router 매 메시지
  1. 키워드 매칭 (v0.5 4채널)
  2. 매칭 실패 시 → FTS5 검색 fallback
     SELECT snippet, agent FROM archive WHERE archive MATCH ?
     ORDER BY rank LIMIT 3
  3. 회상된 과거 결정·신호를 PM에 컨텍스트로 inject (Just-in-Time)
  4. 사용자에게 "🧠 과거 N건 회상 (날짜: ...)" 1회 통지 — 투명성
```

### 4.4 캐시 인식 시스템 프롬프트 (Hermes 패턴)

FTS5 회상 결과는 *매 호출마다 다름*. Anthropic prompt cache(5분 TTL)와 충돌하면 cache hit 0이 됨. 회피:

- 시스템 프롬프트는 **회상 결과 미포함** (cacheable)
- 회상 결과는 *사용자 메시지 직전*에 인라인 (cache miss는 회상 부분만)
- 회상 결과는 토큰 500자 이하로 압축 — full transcript 금지

### 4.5 구현 체크리스트

| # | 작업 | 파일 |
|---|---|---|
| 1 | FTS5 virtual table 스키마 | `src/memory/archive-schema.sql` |
| 2 | JSONL → SQLite 마이그레이션 함수 | `src/memory/archive-rotate.ts` |
| 3 | `archive-rotate` routine (매일 00:00) | `assets/routines/archive-rotate.md` + `src/scheduler/routines.ts` |
| 4 | FTS5 검색 fallback을 agent-router에 통합 | `src/bot/agent-router.ts` (v0.5에서 재작성된 버전) |
| 5 | 회상 결과 인라인 + 통지 | `src/bot/pm-runner.ts` |
| 6 | `solosquad memory search <query>` CLI | `src/cli/memory.ts` (신규) |
| 7 | 회귀 테스트 (회상 정확도·캐시 hit rate) | `test/memory/archive.test.ts` |

추정 ~600 LOC.

### 4.6 인덱싱 소스 확장 (v0.6 P1)

§4.2 표의 Persistent(cold) 계층은 기본 `routine-logs/*.jsonl`만 명시했으나, v0.5에서 누적되는 *다른 형태의 이벤트*도 같은 archive로 인덱싱하면 §3 trajectory 추출·§3.4 freq 자동 추천의 입력원이 크게 풍부해진다. 본 절은 인덱싱 소스를 4종 추가한다.

#### 추가 소스

| 이벤트 타입 (`event_type` 컬럼) | 소스 | 용도 |
|---|---|---|
| `route_hit` | `agent-router.ts` `resolve()` 결과가 non-null인 매 message | 어떤 키워드가 어떤 SKILL로 매칭됐는지 빈도 분석 |
| `route_miss` | `agent-router.ts` `resolve()` returns null | §3.4 freq 자동 추천의 핵심 입력. 누락 SKILL 추론 데이터원 |
| `author_turn` | Author 루프 turn 로그 (`<org>/memory/author-events.jsonl`, v0.5 S3 산출물 확장) | 명확화 질문 평균 turn 수, 사용자 거절 비율 |
| `spawn_decision` | PM Task tool 호출 — 어느 SKILL을 왜 골랐는지 (PM 자유 텍스트 + chosen agent ID) | PM의 자유 라우팅과 결정적 라우터 결과의 *불일치*를 표면화 |

#### 스키마 변경

§4.5 표의 archive virtual table에 `event_type` 컬럼 추가 (TEXT, NOT NULL). 기존 `routine-logs/*.jsonl` 항목은 `event_type='routine_log'`로 분류. 4종 신규 이벤트는 각각 위 표의 type 문자열로 적재.

#### Cross-ref

§3 trajectory 추출의 **negative signal**로 `route_miss` 이벤트가 사용된다 — agent sequence 추출은 *성공한* trajectory만, 라우터 미스는 *누락된 SKILL 후보*. 두 신호를 같은 archive에서 같은 SQL로 조회.

#### 측정

신규 4종 이벤트 적재 후 archive 행 수가 routine-logs 단독 대비 ≥ 3배 (사용자 활동량에 따라 가변). §3.4 freq 자동 추천이 본 §의 `route_miss` 이벤트 없이는 작동 불가 — 본 §은 §3.4의 *전제 조건*.

## 5b. Stop-hook 어댑터 (v0.6 P0 — v0.5 spec-gate 실 작동)

v0.5 §4 `loop_mode.kind: spec-gate`는 *명세*까지만 정의됐다. 실제 Anthropic 2025-12 stop-hook 플러그인을 어떻게 호출하고 신호를 받는지는 v0.6에서 결정. 2026-05-13 트렌드 레코드(§차분1)가 "5번째 트리거 유형 spec-gate"로 분류한 패턴을 1급 시민으로 흡수.

### 5b.1 인터페이스 가정

Anthropic stop-hook 플러그인은 (a) `claude --hook stop=<path>` CLI 플래그 또는 (b) `.claude/hooks.json` 설정 파일로 활성화된다고 가정. 두 인터페이스 모두 *세션 종료 직전 외부 명령 실행 + exit code로 계속/중단 신호 전달* 형태. SoloSquad는 후자(설정 파일)를 우선 선택 — 명시적·재현 가능·git-trackable.

### 5b.2 동작

```
v0.4 goal-runner가 spec-gate SKILL의 goal.md를 실행
  └─ 매 cycle 종료 직전 hook 호출:
     └─ src/engine/stop-hook-adapter.ts:
        1. SKILL frontmatter의 stop_when 텍스트 + spec_path 파일 로드
        2. 현재 cycle 결과(results.tsv 마지막 행) 평가
        3. exit 0 = 계속, exit 1 = 충족 (loop 종료)
        4. 결과 <org>/memory/stop-hook-events.jsonl에 기록
```

### 5b.3 안전장치

- Hook이 항상 빠르게 종료되도록 timeout 5초 (SoloSquad가 외부 SIGTERM)
- Hook 실패 시 goal-runner는 보수적으로 *계속* (false positive보다 false negative가 안전)
- 무한 루프 회피: v0.4 goal.md의 `time_budget` / `cycle_budget`가 여전히 hard cap

### 5b.4 호환성

v0.5 spec-gate SKILL은 `loop_mode.stop_when` 텍스트만 가짐. v0.6 어댑터는 그 텍스트를 LLM 평가가 아닌 *결정적 매칭* (예: "all tests pass" → `npm test` 종료 코드 검사)으로 1차 시도, 매칭 실패 시 LLM fallback (`budget.per_call_usd` cap 적용). v0.5의 `loop_mode` 스키마는 변경 없음 — 어댑터가 텍스트를 해석하는 책임.

### 5b.5 측정

- spec-gate cycle의 stop-hook trigger rate (의도한 종료 / 시간 cap 종료 비율)
- Hook timeout 발생률
- LLM fallback 호출률 (높으면 stop_when 텍스트가 비결정적 — 사용자에게 더 명확한 표현 권고)

## 10. fs.watch 기반 자동 hot-reload (v0.6 P2)

### 10.1 문제

v0.5 S2 라우터는 *명시적* `rebuildRoutes()` 호출만 인식한다. Author 루프(`skill-author.ts`)가 SKILL.md를 저장한 직후에는 라우터가 즉시 갱신되지만, **외부 편집**(VSCode 직접 수정 / `git pull` / 별도 CLI로 SKILL 폴더 조작 / 사용자가 수동 frontmatter 패치)은 봇 재시작 전까지 라우터에 반영되지 않는다. 솔로 파운더 시나리오에서는 SKILL 편집이 코드 편집만큼 자주 발생 — 재시작 비용이 사용자 흐름을 끊는다.

### 10.2 도입

- 의존성: `chokidar` 추가 (Node `fs.watch` 위에 cross-platform 추상화 — Windows polling fallback, macOS FSEvents, Linux inotify 자동 처리)
- 3-tier watch 모두 활성:
  - `<workspace>/.solosquad/agents/**/SKILL.md` (사용자 워크스페이스)
  - `<workspace>/<org>/.agents/**/SKILL.md` (조직별 specialist override)
  - `~/.solosquad/agents/**/SKILL.md` (user-global)
- Debounce 300ms — 같은 파일의 연쇄 저장(편집기 atomic write가 2~3 이벤트를 발생) 회피
- Watch 이벤트 발생 → debounce window 종료 시점에 `rebuildRoutes()` 1회 호출 → v0.5 S2의 atomic swap 흐름(`routeIndexRef = newIndex`)을 *그대로 재사용* → PM 메신저에 사용자 통지 1회 송출:

  ```
  🔄 SKILL routes reloaded — N triggers
  ```

### 10.3 안전장치

- Watch 이벤트가 폭주(예: 대규모 `git pull`로 25개 SKILL 동시 변경)해도 debounce가 1회 `rebuildRoutes()`로 수렴 — atomic swap이 race-free
- Validate 실패 SKILL이 폴더에 들어오면 v0.5 S1 validator가 거부 — 라우터에 partial index 노출 0
- Watcher 자체 실패(예: inotify watch limit) 시 즉시 fallback to "재시작 필요" 안내 — 조용한 실패 금지

### 10.4 측정

- 외부 편집 → 라우터 반영까지 < 1초 (debounce 300ms 포함)
- 100개 SKILL 일괄 수정 시 `rebuildRoutes()` 호출 횟수 = 1 (debounce 검증)
- chokidar 메모리 footprint < 10MB

구현 추정 ~120 LOC (`src/bot/agent-router.ts` 내부 + `src/bot/index.ts`에 watcher lifecycle).

## 11. CI 자동 PR 리뷰 봇 (v0.6 P2)

### 11.1 문제

v0.5 S5에서 `solosquad agent validate --all`을 CI 게이트로 추가했지만, *PR 단위*에서 SKILL 변경 사항을 reviewer가 쉽게 검토하기엔 frontmatter diff가 yaml 한복판에 묻혀 가독성이 낮다. SKILL.md PR이 늘어날수록 사람 reviewer 시간이 병목.

### 11.2 도입

PR diff에 `assets/agents/**/SKILL.md` 또는 `<org>/.agents/**/SKILL.md` 변경이 있으면:

1. `solosquad agent validate <changed files>` 실행 — 통과 못 하면 PR fail (v0.5 S5 게이트의 PR-scoped 변형)
2. `npm run validate-skills` — 외부 코퍼스(`anthropics/skills`) round-trip 회귀 재실행
3. **Frontmatter diff을 PR 댓글로 표시** — 변경된 SKILL 파일별 *전/후 frontmatter*를 표 형식으로 추출해 댓글 게시 (본문 diff는 GitHub UI에 이미 가독성 충분)
4. 새 `triggers.keyword` 항목이 다른 SKILL의 기존 keyword와 충돌하면 (v0.5 S2 라우터의 4채널 우선순위 위반) → 댓글에 ⚠ 경고

### 11.3 구현

- `.github/workflows/skill-review.yml` — pull_request 트리거, paths filter on `**/SKILL.md`
- `scripts/skill-pr-review.ts` — `gh pr diff` 출력 파싱 + frontmatter 추출 + 댓글 본문 조립 + `gh pr comment` 호출
- GitHub Actions 토큰 권한: `pull-requests: write` (댓글 게시), `contents: read`

본 §은 v0.5 S5 검증 게이트의 *확장*이지 대체가 아니다 — 기존 `validate-skills` 게이트는 main 머지 차단 책임을 유지, 본 §은 PR-specific *코멘트 봇* 책임을 담당.

### 11.4 측정

- SKILL 변경 PR의 자동 검증 통과율 ≥ 95% (5% 미만은 사람 검토 필요한 본질적 케이스)
- 사람 reviewer 평균 리뷰 시간 절감 — 본 §은 frontmatter diff을 사전 추출해 *읽기 1단계*만 줄임 (정성 측정)
- 키워드 충돌 경고 false positive < 5% (v0.5 S2 prefix 충돌 규칙과 동일 룰 사용)
