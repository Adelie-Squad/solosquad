# v0.8 — Multi-User Messenger

> v0.7로 라이프사이클이 닫힌 후, **첫 conceptual shift** — "1 워크스페이스 = 1 owner = 1 봇 = 2 채널" 가정을 깬다. 같은 Discord 서버·Slack 워크스페이스에 N명의 팀원이 각자 자기 머신에서 SoloSquad를 설치할 수 있으며, 각 사용자는 자기 명령 채널과 작업 채널 페어를 가진다. 정식 출시 전 마지막 *큰 모델 변경* — 이후 v0.8.1/v0.8.2/v0.8.3은 보완·완결 패치 시리즈, v1.0이 정식 출시.

**로드맵:** v0.1 크로스 플랫폼 → v0.2 안전/보안 → v0.3 PM 모드 → v0.4 자율 실행 → v0.5 워크플로우 메이커 → v0.6 디폴트 워크플로우 + 메모리 아카이브 → v0.7 Uninstall & Lifecycle → **v0.8 Multi-User Messenger** → v0.8.1 Security & Lifecycle Pair → v0.8.2 Dev Capability → v0.8.3 Onboarding UX + Observability → v0.9 안정화 + 자체 사용 검증 → **v1.0 정식 출시**

**의존:** v0.3 (PM 세션 — 사용자별 session-id 격리의 기반), v0.6 (Org Layer + 8-layer JIT — 사용자 yaml을 Layer 5에 주입), v0.7 (lockfile·journal 패턴 — 봇 multiplicity 가드)

**기준 버전:** v0.8.0. 단일 테마 minor — 메신저 모델만 다룬다. 다른 pre-launch 보완은 v0.8.1+로 분리.

**분류:** Model change. 채널 명명 규칙·봇 multiplicity·author-guard 도입. 사용자 코드에 손대지 않음.

| 문제 | 원인 |

## 2. 원칙

| 원칙 | 구현 |
|---|---|
| **메신저-네이티브 격리** | 사용자별 채널 페어는 메신저 ACL(Discord private channel + permission overwrites, Slack `is_private=true` conversation)을 1차 권한 메커니즘으로. SoloSquad 코드는 *defense in depth*로 author 검증만 추가. 권한 시스템 재발명 안 함 |
| **각 사용자 = 각자의 봇 application** | 같은 Discord/Slack 서버에 N명 설치 = N개의 봇 application + N개의 봇 프로세스. **공유 봇 토큰은 거부** (rate-limit 충돌·duplicate reply 위험). 사용자 1명당 봇 1개가 자기 채널만 owner |
| **handle 충돌은 명시적 거부** | 자동 `-2` suffix 같은 silent disambiguation 안 함. 같은 messenger handle을 두 번 등록 시도 → init 거부 + 다른 handle 권고 |
| **broadcast는 opt-in** | 공유 brief 채널은 기본 off. 활성화 시 designated 봇 1개만 발송 (모든 봇이 푸시하면 N건 중복) |
| **솔로 사용자도 1급 시민** | 1명 install 시점부터 `command-<handle>` / `works-<handle>` 패턴 그대로. legacy `owner-command`/`workflow` 명명은 v0.8부터 사용 안 함 |

## 3. 채널 모델

### 3.1 명명 규칙

| 채널 | 이름 | 가시성 | 누가 명령 가능 |
|---|---|---|---|
| 개인 명령 채널 | `command-<handle>` | private (초대만) | 채널 owner(`<handle>`) 만. 다른 메시지는 봇이 ignore + ephemeral DM "이 채널은 <handle>의 명령 채널입니다" |
| 개인 작업 채널 | `works-<handle>` | private (초대만) | 봇이 그 owner의 PM 세션 결과 + goal-runner 출력 + 워크플로 핸드오프를 푸시. 사용자도 메모 가능 |
| (선택) 공유 브로드캐스트 | `solosquad-broadcast` | public 또는 팀 visible | 봇이 morning/evening brief, weekly review, signal-scan 결과를 푸시. 1 워크스페이스당 1개. 사용자 명령은 받지 않음 |

`<handle>` 정의:
- **Discord**: 글로벌 unique handle (2023 이후 `username` 필드 — display name 아님)
- **Slack**: 워크스페이스 내 unique handle (`users.info → name`)
- 영문 소문자·숫자·`_`만 허용. 다른 문자는 `_`로 치환 후 사용자 확인 prompt

예시:
```
Alice (Discord handle "alice"):
  #command-alice    ← alice만 명령 가능
  #works-alice      ← alice의 PM 세션 결과 + 작업 트리

Bob (Discord handle "bob"):
  #command-bob      ← bob만 명령 가능
  #works-bob        ← bob의 PM 세션 결과 + 작업 트리

(선택) 공유:
  #solosquad-broadcast   ← 8:00 morning brief 등 자동 푸시
```

### 3.2 봇 multiplicity

**원칙**: 사용자 1명 = 봇 application 1개 = 봇 프로세스 1개.

같은 Discord 서버에 2명 설치:
- alice의 머신: 자기 .env에 alice의 봇 token. `solosquad bot` = alice의 봇 process
- bob의 머신: 자기 .env에 bob의 봇 token. `solosquad bot` = bob의 봇 process
- 두 봇은 *같은* 서버에 멤버지만 *다른 user ID*. 각자 자기 채널 페어만 listen
- 채널 ↔ 봇 매핑: `<org>/.solosquad/users/<handle>.yaml`의 `bot_user_id` 필드

**거부되는 시나리오**:
- 한 봇 token을 두 머신이 동시에 사용 → Discord/Slack에서 disconnect 또는 rate-limit. README에 "1 token = 1 머신" 명시

### 3.3 사용자 식별 흐름

`solosquad init` 단계 변경 (v0.8):

```
6. 메신저 토큰 입력
   ↓
6a. 봇 토큰으로 messenger API 호출:
      Discord: GET /users/@me  → username
      Slack:   auth.test → users.info → name
   ↓
6b. 추출된 handle을 사용자에게 확인:
      "이 봇 token의 messenger handle은 'alice'로 감지됐어요.
       채널은 'command-alice' + 'works-alice'로 생성됩니다.
       변경하려면 입력하세요 [기본: alice]:"
   ↓
6c. 사용자가 입력한 handle을 `<org>/.solosquad/users/<handle>.yaml`에 저장
   ↓
7. 봇 startup 시 사용자 yaml 로드 → 자기 채널 페어 자동 생성
```

`<org>/.solosquad/users/<handle>.yaml` 스키마:
```yaml
schema_version: 1
handle: alice
display_name: "Alice (Founder)"
messenger: discord
bot_application_id: "1234567890"
bot_user_id: "9876543210"
joined_at: "2026-05-15T10:00:00Z"
workspace_path: "/Users/alice/solosquad"
session_id: "<pm-session-id>"
channels:
  command: "command-alice"
  works: "works-alice"
```

이 yaml은 v0.6 §2.2 8-layer JIT inject의 Layer 5 (org-level metadata)로 spawn-time에 자동 주입 — specialist가 사용자 컨텍스트를 인식.

### 3.4 author-guard (defense in depth)

메신저 ACL이 1차 방어선이지만, 잘못 초대된 사용자가 다른 사람 채널에 글을 쓰는 사고를 막기 위해 코드 레벨 가드 추가:

```typescript
// src/bot/author-guard.ts (신규)
export function isAuthorizedAuthor(channelName: string, authorHandle: string): boolean {
  const match = channelName.match(/^(command|works)-(.+)$/);
  if (!match) return true;  // 다른 채널은 통과 (broadcast 등)
  const channelOwner = match[2];
  return channelOwner === authorHandle;
}

// adapter에서 메시지 수신 시:
if (!isAuthorizedAuthor(ctx.channelName, ctx.authorHandle)) {
  await ctx.sendEphemeral(
    `이 채널은 ${channelOwner}의 명령 전용입니다. command-${ctx.authorHandle} 채널을 사용하세요.`
  );
  return;
}
```

### 3.5 채널 생성 / 발견 / 충돌

봇 startup 시 (`src/bot/channel-bootstrap.ts` 신규):

1. `<org>/.solosquad/users/*.yaml` 전부 로드
2. 자기 `bot_user_id`와 매칭되는 yaml 1건 식별 (없으면 init 안내 후 종료)
3. 해당 yaml의 `channels.command` + `channels.works` 가 messenger에 존재하는지 확인
4. 부재 시 생성 (private + 자기 봇 + owner 사용자만 초대)
5. broadcast 채널 처리 — §3.6
6. 다른 user의 채널 페어(`command-bob` 등)는 *listen 안 함, 무시*

**Handle 충돌 정책 (박제 — 거부)**:
- `<org>/.solosquad/users/<handle>.yaml`이 이미 존재할 때 같은 handle로 init 시도 → **명시적 거부**
- 에러 메시지: `"<handle>은 이미 이 워크스페이스에 등록되어 있습니다. 다른 messenger handle 또는 별도 워크스페이스를 사용하세요."`
- 자동 `-2` suffix 같은 silent disambiguation 안 함

**Slack 무료 워크스페이스 채널 cap**:
- Slack free tier: 10 channels per workspace
- N명 팀에서 2N+1 채널이면 N=5에서 cap 도달
- README에 "Slack 무료는 N≤4 권장. 그 이상은 Slack Pro 또는 Discord" 명시
- 대안 (Slack thread-only mode): v1.0 슬롯

### 3.6 broadcast 채널 정책 (박제 — 옵션 B)

`#solosquad-broadcast` 처리:

| 항목 | 결정 |
|---|---|
| 기본값 | **off** (`workspace.yaml.messenger.broadcast_enabled: false`) |
| Opt-in 시 채널 생성 | 첫 install 시 1회 생성. workspace.yaml에 `broadcast_channel: "solosquad-broadcast"` stamp |
| 발송 권한 | **designated 봇 1개만** — first-installed 봇(`broadcast_owner_handle: <first-handle>` 박제). 다른 봇은 자기 `works-<handle>`로만 brief 푸시 |
| 발송 내용 | morning/evening brief, weekly review, signal-scan 결과. designated 사용자 워크스페이스 데이터 기반 |
| 명시 안내 | broadcast 활성화 시 README/master-guide에 "공유 brief는 <handle> 워크스페이스 데이터 시점" 박스로 표시 |
| Designated 사용자 이양 | `solosquad messenger broadcast-handover --to <handle>` — 첫 사용자가 떠날 때 다른 사용자에게 이양 |

설계 정합:
- 봇 multiplicity(§3.2)와 정합 — 모든 봇이 broadcast 푸시하면 N건 중복
- 격리 정합 — broadcast off가 기본이라 공유 데이터 노출 zero-by-default
- 팀 visibility — opt-in으로만 활성. solo founder가 알아서 켬

### 3.7 v0.7 → v0.8 채널 모델 전환 (마이그레이션 무관)

본 릴리스 시점에 기존 v0.7 사용자가 없는 것으로 확인됨(개발자 본인만). 따라서:

- legacy `owner-command` / `workflow` 채널 alias 매핑 **불필요**
- `workspace.yaml.messenger.legacy_channels` 키 **추가 안 함**
- `solosquad messenger migrate-channels` 별도 명령 **추가 안 함**
- 봇은 처음부터 `command-<handle>` / `works-<handle>` 단일 패턴만 인식
- 마이그레이션 스크립트(§6)는 *version bump + 신규 키 보강 + 첫 user yaml 생성* 만 수행. legacy 채널 인식 코드 0건

향후 외부 사용자가 v0.7→v0.8 업그레이드를 시도할 가능성에 대비해 마이그레이션 스크립트 verify 단계에서 1줄 안내 출력:
`"기존 #owner-command / #workflow 채널은 봇이 더 이상 listen 하지 않습니다. 새 채널 command-<handle> / works-<handle>가 첫 봇 startup 시 생성됩니다. legacy 채널은 메신저에서 수동 archive 권장."`

## 5. 비파괴성 + 복구 + 테스트

### 5.1 솔로 사용자 1급 시민 보장
- 1명 install 시점에도 `command-<handle>` + `works-<handle>` 자동 생성. solo 모드 별도 코드 없음
- broadcast 채널은 명시적 opt-in이라 기본 install에 변화 0

### 5.2 봇 multiplicity 안전망
- bot startup 시 자기 `bot_user_id` ↔ yaml 매칭 실패 → init 안내 후 종료 (다른 사용자 채널 침범 0)
- 같은 token을 두 머신이 동시 사용 → Discord/Slack이 disconnect. SoloSquad가 막지 못함. README에 명시

### 5.3 author-guard 정확성
- 회귀: alice 채널에 bob이 메시지 → 봇은 응답 안 함, bob에게 ephemeral DM
- 회귀: alice 채널에 alice가 메시지 → 정상 처리
- 회귀: broadcast 채널에 누구든 메시지 → 봇은 listen 안 함 (broadcast는 봇이 push 전용)

### 5.4 회귀 그린 기준

5. verify 단계에서 legacy 채널 안내 1회 출력
