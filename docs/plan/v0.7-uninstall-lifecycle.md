# v0.7 — Uninstall & Lifecycle (Farewell Archive)

> SoloSquad는 "초기화(reset)" 기능을 두지 않는다. 도구의 라이프사이클은 `solosquad init` (있음) → `solosquad uninstall` (v0.7 신설) 두 진입점으로만 닫는다. uninstall은 (a) 사용자 repositories를 **절대 건드리지 않고**, (b) 누적 지식·워크플로우·메모리를 **단일 archive로 export**한 뒤, (c) SoloSquad 자체 파일만 정리하고, (d) 외부 자원(메신저 봇·OAuth·API 키) revoke 체크리스트를 출력한다.

**로드맵:** v0.1 크로스 플랫폼 → v0.2 안전/보안 → v0.3 PM 모드 → v0.4 자율 실행 → v0.5 워크플로우 메이커 → v0.6 디폴트 워크플로우 + 메모리 아카이브 → **v0.7 Uninstall & Lifecycle** → v1.0 정식 출시 → v1.1 대시보드 → v1.2 지식 온톨로지

**의존:** v0.3 (`.solosquad/sessions/` 매핑 정리 대상), v0.4 (`<org>/goals/<goal-id>/results.tsv` + 워크스페이스 루트 `AGENTS.md` 보존), v0.6 (`memory/archive.sqlite` FTS5 — WAL-safe backup 필요)

**기준 버전:** v0.6 직후. v1.0 정식 출시 전 라이프사이클 완성 슬롯.

**분류:** Lifecycle 인프라 — 기능 신설보다 *기존 자산을 어떻게 닫는가*에 대한 정책.

## 핵심 질문 & 성공 기준

| 핵심 질문 | 성공 기준 |
|---|---|
| 사용자가 SoloSquad를 제거할 때 자신의 product 코드를 잃을 위험이 0인가? | `<org>/repositories/<repo>/` 트리는 uninstall 전후 byte-identical. `--purge`로도 보호. 옵션 자체가 존재하지 않음 |
| 누적된 의사결정·메모리·지식을 영구히 잃지 않고 회수할 수 있는가? | `solosquad uninstall`이 단일 `.zip`을 생성. `solosquad import <zip>` (v1.0 슬롯)으로 새 머신에서 워크스페이스 복원 가능 |
| 봇 토큰·API 키 같은 시크릿이 archive에 평문으로 유출되지 않는가? | archive에는 변수명 + 마스킹된 placeholder만. 외부 revoke 링크 + 체크리스트 동봉 |
| `solosquad reset` 같은 "워크스페이스 초기화" 명령이 없는데도 라이프사이클이 닫히는가? | install ↔ uninstall 2단으로 충분. "초기화"는 *uninstall + 새 워크스페이스에서 install*로 흡수 |

## 1. 현재 문제 (v0.6 시점)

| 문제 | 원인 |
|---|---|
| 사용자가 SoloSquad를 제거하면 `<org>/repositories/<repo>/`의 product 코드가 통째로 위험에 노출 | 워크스페이스 디렉토리 1곳에 SoloSquad 자체 자산 + 사용자 코드가 섞여 있음 |
| 누적된 `_events.jsonl`·`memory/*.jsonl`·`workflows/<id>/`·`results.tsv`·KNOWLEDGE.md를 일괄 회수할 수단 없음 | export/archive 인프라 부재 |
| `.solosquad/.env`의 봇 토큰·API 키 처리 정책이 문서화돼 있지 않음 | uninstall 안내가 "수동 rm"으로만 끝남 — 외부 revoke 누락 빈번 |
| 메신저 채널·OAuth 앱·`~/.claude/projects/` 잔존물의 정리 가이드 부재 | 외부 자원과 로컬 자원의 경계를 코드 어디서도 다루지 않음 |
| `solosquad init`이 두 번째 실행되면 add-org 흐름만 타고 config copy를 skip — "깨끗한 재설치"가 불가능 | init이 idempotent하지만 destructive가 아님 → 사용자가 "초기화"를 원할 때 답이 없음 |
| 마이그레이션 백업(`~/.solosquad-backups/`)이 uninstall과의 관계 미정 | 백업 누적 → 디스크 점유, 또는 의도와 달리 일괄 삭제 위험 |

## 2. 원칙

| 원칙 | 구현 |
|---|---|
| **사용자 코드 절대 불가침** | `<org>/repositories/<repo>/` 트리는 `solosquad uninstall`의 어떤 플래그·서브명령으로도 변경·삭제 대상이 아님. 옵션 자체를 두지 않는다 (OpenClaw 안티패턴 회피 — Issue #6289 closed as not planned) |
| **누적 지식은 archive해서 남긴다** | uninstall은 항상 *farewell archive*를 먼저 생성. 비대화형 모드에서도 archive 단계는 skip 불가 (`--no-archive` 같은 플래그 없음) |
| **"초기화" 기능을 만들지 않는다** | `solosquad reset`·`solosquad clean` 같은 명령은 영구히 추가하지 않음. 의미 있는 라이프사이클은 install ↔ uninstall 2단. "재설치"는 *uninstall + farewell + 새 workspace에서 init* 시퀀스로 자연 표현 |
| **시크릿은 마스킹, revoke는 외부 위임** | `.env`·credential·OAuth 토큰의 실제 값은 archive에 절대 포함 안 함. archive에는 변수 이름과 마스킹된 자리표시자만. 사용자에게 외부 콘솔(Discord/Slack/Anthropic) 링크 출력 |
| **logout과 데이터 정리 분리** (gh CLI 패턴 차용) | `solosquad logout` (가벼움, 토큰 revoke 안내 + .env 마스킹)과 `solosquad uninstall` (무거움, archive + 파일 정리)을 독립 명령으로. logout이 uninstall의 사전 단계는 아님 |
| **npm 훅 의존 금지** | npm v7+는 글로벌 패키지 `preuninstall`/`postuninstall`을 실행하지 않음 (`npm/cli#3042`). user-invoked `solosquad uninstall` 서브명령이 유일한 신뢰 가능한 cleanup 진입점 |
| **마이그레이션 백업과 archive는 별도 자산** | `~/.solosquad-backups/` (migrate가 만든 옛 워크스페이스 스냅샷)는 uninstall이 손대지 않음. archive는 *새로 생성된 작별 산출물*, 백업은 *과거 마이그레이션 안전망* — 두 트랙 분리 |
| **WAL-safe DB 백업** (Hermes 패턴 차용) | v0.6 `memory/archive.sqlite` FTS5는 SQLite `backup()` API로 안전 복사. 실행 중에도 일관성 유지 |

### 3.2 안티패턴 (회피 대상)

| 레퍼런스 | 문제 | 회피 방법 |
|---|---|---|
| **OpenClaw** | `openclaw uninstall`이 `${OPENCLAW_STATE_DIR:-$HOME/.openclaw}` 전체를 무조건 삭제. workspace 보존 opt-in 요구한 Issue #6289 "closed as not planned" — 비복구 데이터 손실 신고 다수 | SoloSquad는 archive를 **사전 강제** (skip 불가). repositories는 **옵션 없이 절대 불가침** |
| **Claude Code CLI** | `/logout`은 인증만 해제, 데이터·바이너리는 사용자가 수동 정리. 공식 가이드가 `rm -rf ~/.claude` + `~/.claude.json` 권장하지만 cleanup 명령 부재 | SoloSquad는 명시적 `uninstall` 서브명령 + archive 자동화로 자동화 격차 메움 |
| **autoresearch / pm-skills / MiroFish** | uninstall 명령 자체 없음 — `results.tsv`·skills 위치·메모리 경로가 README에 미문서화 | SoloSquad는 archive manifest(`manifest.tsv` SHA256)로 모든 보존 파일을 결정적으로 추적 |

### 3.3 공통 공백 = SoloSquad 차별점

7개 레퍼런스 + Claude Code 어디에도 **외부 자원 revoke 가이드**가 없다. 메신저 봇이 만든 채널, OAuth 앱, 클라우드 토큰의 정리는 모두 "사용자 알아서". SoloSquad는 v0.2의 메신저 통합과 v0.3의 멀티 토큰 사용 때문에 이 갭이 더 크다 → `REVOKE-CHECKLIST.md` 자동 생성으로 메움 (§8).

### 3.4 npm 라이프사이클 결정 근거

- [docs.npmjs.com/cli/v10/using-npm/scripts](https://docs.npmjs.com/cli/v10/using-npm/scripts): "npm v7 does not [have uninstall lifecycle scripts]. Removal can happen for a wide variety of reasons, and there's no clear way to currently give the script enough context to be useful."
- [npm/cli#3042](https://github.com/npm/cli/issues/3042): 글로벌 패키지에서도 동일 — `npm uninstall -g <pkg>`는 `preuninstall` 미실행 확인.
- **결론**: SoloSquad는 npm 훅으로 cleanup을 자동화할 수 없다. `solosquad uninstall` 서브명령이 *uninstall 라이프사이클의 유일한 신뢰 진입점*이며, 사용자가 `npm uninstall -g solosquad`보다 *먼저* 실행해야 한다. 이를 안내하는 README + doctor 경고가 필수.

## 4. 데이터 분류

uninstall이 어떻게 다룰지는 자산을 5개 클래스로 나눠 결정.

| 클래스 | 예시 위치 | uninstall 처리 |
|---|---|---|
| **A. 사용자 코드 (불가침)** | `<workspace>/<org>/repositories/<repo>/` (단 — `<repo>/.solosquad/repo.yaml` 1개 예외, A* 참조) | **읽지도 않는다.** archive 미포함, 삭제 절대 안 함. `solosquad uninstall`은 이 디렉토리의 존재만 확인하고 사용자에게 "이 경로는 손대지 않음"을 통지 |
| **A\*. SoloSquad가 repo 내부에 만든 단일 메타 파일** | `<repo>/.solosquad/repo.yaml` (+ 컨테이너 `<repo>/.solosquad/`) — **이게 SoloSquad가 repo 내부에 자동 생성하는 유일한 산출물**. 코드 검증: `src/util/scaffold.ts:73` `scaffoldRepoYaml()`에서 1회만 작성, 그 외 어떤 함수도 `<repo>/` 내부에 쓰지 않음 | **surgical 추출**: archive zip의 `orgs/<org-slug>/repos/<repo-slug>/repo.yaml`로 복사한 후 repo 내부에서 `.solosquad/` 디렉토리만 삭제. **whitelist 길이 1** — repo의 다른 어떤 파일도 (`.git/`·source·`.claude/skills/`·user-authored 모든 것) 건드리지 않음. 결과: uninstall 후 repo는 SoloSquad 흔적 0이 되며, 사용자가 통째로 옮겨도 추가 정리 불필요 |
| **B. 누적 지식 (archive 보존)** | `<org>/memory/*.jsonl`, `<org>/memory/archive.sqlite` (v0.6 FTS5), `<org>/workflows/<id>/`, `<org>/goals/<goal-id>/results.tsv`+`goal.md`+`_best.json`+`_last-run.md` (v0.4), `<workspace>/AGENTS.md` (v0.4, 사람이 편집한 영속 가이드), `<org>/domain/`, `.solosquad/knowledge/`, `<org>/core/`, `assets/agents/{team}/KNOWLEDGE.md` 사용자 편집분, `<org>/memory/pm-skills/*.md` (v0.3 compaction 결과) | archive zip에 통째 포함. 그 후 워크스페이스에서 삭제 |
| **C. 운영 데이터 (archive에 마스킹된 메타만)** | `.solosquad/sessions/<user>.json`, `<workspace>/.solosquad/workspace.yaml`, `<org>/.org.yaml`, `<org>/.solosquad/sessions/` | session-id·타임스탬프·org 메타만 archive에 포함. session jsonl 본문은 archive 제외 (Claude Code 영역) |
| **D. 시크릿 (마스킹 + revoke 안내)** | `.solosquad/.env` (`DISCORD_TOKEN`, `SLACK_BOT_TOKEN`, `ANTHROPIC_API_KEY`), MCP 서버 credential | archive에는 변수명 + `***REDACTED***` 자리표시자만. 실제 값은 archive에 절대 미포함. `REVOKE-CHECKLIST.md`에 외부 콘솔 URL + 봇 ID 동봉 |
| **E. 외부 자원 (안내만)** | Discord/Slack 봇이 만든 채널·앱, Anthropic API 키, `~/.claude/projects/<>/*.jsonl` (Claude Code 관리), 마이그레이션 백업 `~/.solosquad-backups/`, cron 등록분 | SoloSquad는 *삭제하지 않는다*. `REVOKE-CHECKLIST.md`에 어떻게 정리할지 명시. `~/.solosquad-backups/`만 명시적 `--also-purge-backups`로 정리 가능 |

**경계 사례**:
- `<org>/.claude/agents/<agent>.md` (v0.3 agents-builder가 동기화한 SKILL.md 사본): 클래스 C로 분류 — `assets/agents/`에서 결정적으로 재생성 가능하므로 archive 미포함, uninstall 시 삭제.
- `<workspace>/docker-compose.yml`, `Dockerfile`: 클래스 C — `assets/`에서 복사된 것. archive 미포함, 삭제.
- `<repo>/.claude/skills/` (CLAUDE.md 스펙상 "repo-specific skills"로 명시되어 있으나 **현재 코드베이스에 자동 생성 로직 없음** — 사용자가 직접 작성한 경우만 존재): 사용자 작성물로 간주 → 클래스 A (불가침). uninstall이 보지도 않음. v0.5 워크플로우 메이커에서 자동 생성을 추가하게 되면 그 시점에 분류 재검토.
- `~/.claude/projects/<cwd-encoded>/*.jsonl` (Claude Code가 PM session·subagent transcript 저장): 클래스 E. **repo 내부가 아니라 사용자 홈 디렉토리** — `target_repo`로 PM이 repo cwd에서 작업해도 transcript는 ~/.claude/ 아래로 가지 repo 안에 안 들어감. 따라서 사용자가 repo만 옮겨도 transcript는 따라가지 않으며, 반대로 repo를 옮긴 뒤 워크스페이스를 지워도 transcript는 별도 보존. `REVOKE-CHECKLIST.md`에 정리 방법 안내.
- `<repo>/.git/`: 클래스 A — 사용자 git 히스토리. SoloSquad는 `cloneRepo()`로 표준 `git clone`만 호출(`src/util/git.ts`), `.git/` 내부를 직접 쓰는 코드 0건.

## 5. CLI 인터페이스

```bash
# 메인 entry — 항상 archive 먼저, 그 다음 정리
solosquad uninstall [--dry-run] [--archive-only] [--keep-workspace]
                    [--also-purge-backups] [--archive-path <path>] [-y/--yes]

# 가벼운 logout (토큰 revoke 안내 + .env 마스킹만)
solosquad logout [--org <slug>] [--all]

# (v1.0 슬롯) farewell archive 복원 — 별도 머신에서
solosquad import <archive.zip> [--workspace <path>]
```

### 5.1 `solosquad uninstall` 동작 흐름

```
0. Pre-check
   ├─ 워크스페이스 위치 + 클래스 A(repositories/) 존재 여부 통지
   │   "다음 경로는 손대지 않습니다: <repo paths>"
   ├─ 실행 중 프로세스 감지 (solosquad bot/schedule PID 추정)
   ├─ git 상태: <org>/repositories/<repo>/ 트리의 미커밋·미푸시 변경 알림
   │   (사용자에게 push/commit 권장 1회 — uninstall은 강행 가능, 단 명시 확인)
   └─ 비-yes 모드: 확인 prompt

1. Archive 단계 (--no-archive 플래그 없음 — skip 불가)
   ├─ archive 경로 결정: --archive-path 또는 기본 ~/solosquad-archive-<workspace-slug>-<YYYY-MM-DD-HHmm>.zip
   ├─ 클래스 B 전체 복사
   ├─ 클래스 C 메타데이터 추출 (session-id, workspace.yaml, .org.yaml)
   ├─ 클래스 D 마스킹 처리 (.env → .env.template + REVOKE-CHECKLIST.md)
   ├─ v0.6 memory/archive.sqlite는 SQLite backup() API로 일관 복사
   ├─ manifest.tsv 생성 (모든 archive 파일 path + SHA256 + size)
   └─ archive zip 작성

2. --archive-only 면 여기서 종료. 사용자에게 다음 단계 안내 출력.

3. Cleanup 단계
   ├─ <org>/repositories/ 이외의 모든 SoloSquad 자산 삭제:
   │   - .solosquad/ (config·sessions·agents·routines·templates·orchestrator·.env)
   │   - <org>/.org.yaml, <org>/.solosquad/, <org>/.claude/
   │   - <org>/memory/, <org>/workflows/, <org>/goals/, <org>/domain/, <org>/core/
   │   - <workspace>/AGENTS.md (사용자 영속 가이드 — archive에 포함 후 삭제)
   │   - <org>/slack/ | discord/ (채널 config 캐시)
   │   - docker-compose.yml, Dockerfile (assets/ 출처 확인 시에만)
   ├─ <org>/repositories/<repo>/.solosquad/ surgical 제거 (클래스 A* — repo당 1회):
   │   - 각 repo의 .solosquad/repo.yaml을 archive로 옮긴 *후*에만 제거
   │   - 화이트리스트 길이 1 (".solosquad" 디렉토리 이름만 매칭, repo 다른 경로 0건 touch)
   │   - 검증: repo의 .solosquad/ 외 모든 파일은 cleanup 전후 SHA1 동일해야 함 (assertion)
   ├─ --keep-workspace: 위 중 .env·sessions·.claude/agents·routine-logs만 삭제,
   │                    workflows/memory/knowledge·repo.yaml은 디스크에 남김 (재install 사이트로 재사용)
   ├─ --also-purge-backups: ~/.solosquad-backups/ 도 함께 삭제
   └─ 최상위 <workspace> 디렉토리는 *비어 있으면* 삭제, 클래스 A 잔존 시 보존

4. 마무리
   ├─ archive 경로 + 사이즈 출력
   ├─ REVOKE-CHECKLIST.md 경로 출력 + "이 작업을 외부 콘솔에서 직접 수행하세요"
   ├─ npm uninstall -g solosquad 안내 (1줄)
   └─ 종료
```

### 5.2 `--dry-run`

위 0~4 단계를 모두 실행하되 *읽기*만 수행. archive 생성 안 함, 삭제 안 함, 무엇이 archive로 가고 무엇이 삭제될지 트리로 출력. CI/대규모 워크스페이스에서 사전 점검용.

### 5.3 `--archive-only`

archive zip만 만들고 cleanup은 건너뜀. 머신 이전·정기 백업·v0.6 디버깅 용도. `solosquad import` (v1.0)와 페어.

### 5.4 `--keep-workspace`

config·시크릿·session 매핑·routine 로그만 정리. workflows/memory/knowledge는 디스크에 남김. 같은 머신에서 재설치 시 `solosquad init`이 이 데이터를 재사용. 단 `<org>/.org.yaml` 같은 메타는 새로 발급 (충돌 방지).

### 5.5 `--also-purge-backups`

`~/.solosquad-backups/` (마이그레이션 백업)도 같이 삭제. 디폴트는 보존 — 이 백업은 v0.7 archive와 별도 안전망이므로 명시적 opt-in.

### 5.6 `solosquad logout`

archive 없음. 다음만 수행:
- 시크릿 마스킹: `.env`의 토큰 값 → `***REDACTED-AT-<ts>***`로 치환 (변수 키는 보존 — 재설정 가이드용)
- `REVOKE-CHECKLIST.md`만 생성 (워크스페이스 root에)
- session 매핑 archive: `.solosquad/sessions/*.json`을 `.solosquad/sessions/_archived/`로 이동

uninstall과 별도 명령으로 둔 이유 = gh CLI 패턴. 사용자가 "잠시 봇만 정지"하고 싶을 때 데이터 삭제 위험 없이 사용.

### 6.1 `archive.yaml` 스키마

## 7. 시크릿·환경변수 처리

### 7.1 `.env` 마스킹 규칙

```
원본 .env                              archive credentials/env.template
─────────────────────────────────      ───────────────────────────────────
DISCORD_TOKEN=MTI3OD...                DISCORD_TOKEN=***REDACTED-AT-2026-05-13T22:00:00+09:00***
SLACK_BOT_TOKEN=xoxb-12345...          SLACK_BOT_TOKEN=***REDACTED-AT-2026-05-13T22:00:00+09:00***
ANTHROPIC_API_KEY=sk-ant-...           ANTHROPIC_API_KEY=***REDACTED-AT-2026-05-13T22:00:00+09:00***
WORKSPACE_NAME=myworkspace             WORKSPACE_NAME=myworkspace
TIMEZONE=Asia/Seoul                    TIMEZONE=Asia/Seoul
```

**규칙**:
- 키 이름이 known secret 리스트(`*TOKEN`, `*KEY`, `*SECRET`, `*PASSWORD`, `*CREDENTIAL`)에 매치되면 값 마스킹
- 매치되지 않는 변수(`WORKSPACE_NAME`, `TIMEZONE`)는 원본 값 보존 — 재설치 시 동일 설정 복원에 사용
- 매치 패턴은 `src/util/secrets.ts` (신규)에 상수로 정의, 사용자 추가 가능 (`.solosquad/secret-keys.txt` opt-in)
- 마스킹 후 archive에 포함된 값이라도 *언제 마스킹됐는지* 타임스탬프 명시 → uninstall 시점 식별 가능

### 7.2 OS 레벨 환경변수

`.solosquad/.env`가 아닌 `setx`·`export`로 사용자가 직접 설정한 OS 환경변수는 SoloSquad가 인지 못 함. `REVOKE-CHECKLIST.md`에서 일반 안내만 (확인 명령 + 셸별 unset 방법).

### 7.3 MCP 서버 credential

`.solosquad/.env` 외부의 MCP 서버 credential (예: `~/.config/mcp/<server>/credentials.json`)은 사용자 도구 영역. archive 미포함, `manual-revoke-required/mcp.md`에 일반 가이드.

### 7.4 토큰 server-side revoke

gh CLI 패턴 차용 — local 마스킹은 단순한 무력화이며 **server-side revoke가 별도 필요**함을 명시. `REVOKE-CHECKLIST.md`가 외부 콘솔 URL + 봇 application ID를 출력해 1-click 진입 가능하게.

## 8. 외부 자원 Revoke 가이드 (`REVOKE-CHECKLIST.md`)

archive 안 + 워크스페이스 root에 동시 생성. 사용자가 외부 콘솔에서 직접 처리해야 할 항목을 단계별로:

```markdown
# Revoke Checklist — generated 2026-05-13T22:00:00+09:00

> SoloSquad를 제거하기 전/후, 다음 외부 자원을 직접 정리하세요.
> 로컬 파일 정리만으로는 보안이 완결되지 않습니다.

## 1. Anthropic API Key (필수)

- 콘솔: https://console.anthropic.com/settings/keys
- 이 워크스페이스에서 사용된 키 식별 힌트:
  - key prefix: sk-ant-api03-XXXX...  (마지막 호출 시각: 2026-05-13)
- 작업: 해당 키를 "Revoke" 또는 "Disable"
- 확인: 콘솔의 Usage에서 새 사용량이 0인지 24h 후 재확인

## 2. Discord Bot Application (있는 경우)

- 콘솔: https://discord.com/developers/applications
- 봇 application ID: 1234567890123456789
  (위 ID는 .env DISCORD_APPLICATION_ID에서 가져온 것 — 시크릿 아님)
- 작업:
  1. 봇 application 열기 → "Delete App" (영구) 또는
  2. Bot 탭에서 "Reset Token" (토큰만 무효화, app은 유지)
- 봇이 가입한 서버에서: 서버 설정 → Integrations → 봇 제거

## 3. Slack App (있는 경우)

- 콘솔: https://api.slack.com/apps
- 봇이 만든 채널(자동):
  - #owner-command
  - #workflow
- 작업:
  1. 봇 app을 워크스페이스에서 Uninstall
  2. 위 채널은 자동 삭제되지 않음 — 수동으로 Archive 권장
  3. (선택) app 자체 Delete

## 4. ~/.claude/projects/ (Claude Code 영역)

- SoloSquad PM session transcript 위치 추정:
  - ~/.claude/projects/C--Users-me-myworkspace-myorg/01HX...jsonl
  - ~/.claude/projects/C--Users-me-myworkspace-myorg/01HY...jsonl
- Claude Code를 계속 사용한다면 이 파일은 그대로 둬도 됨
- 완전 삭제: rm -rf ~/.claude/projects/C--Users-me-myworkspace-* (위 경로만)

## 5. 백그라운드 프로세스 (있는 경우)

- 다음이 등록돼 있는지 확인:
  - pm2 list → solosquad-bot, solosquad-scheduler
  - systemctl --user list-units → solosquad-*
  - Windows 작업 스케줄러 → SoloSquad 관련 항목
- 모두 stop + disable

## 6. cron / 스케줄러

- crontab -l 에 `solosquad`가 나타나는지 확인
- 있다면 해당 줄 제거: crontab -e
```

**구현 메모**: 위 파일은 부분적으로 사용자 워크스페이스 상태에 따라 동적 생성 (Discord application ID 추출, 봇 채널 이름, 추정 transcript 경로). archive zip 안의 `manual-revoke-required/`에 각 항목을 separate markdown으로도 분리해 archive 단독으로도 가이드 완결.

## 9. 마이그레이션 백업과의 관계

`~/.solosquad-backups/` (또는 Windows `%USERPROFILE%\.solosquad-backups\`):

| 측면 | 정책 |
|---|---|
| 무엇인가 | `solosquad migrate --apply` 실행 시 자동 생성되는 *과거 워크스페이스 상태 스냅샷*. 마이그레이션 실패 시 `migrate --rollback`이 복원 대상으로 사용 |
| v0.7 archive와의 차이 | 마이그레이션 백업 = "이전 버전으로 되돌아가는 안전망". v0.7 archive = "도구를 제거하면서 데이터를 들고 떠나는 farewell". 의미가 다름 |
| uninstall 기본 동작 | 손대지 않음. 마이그레이션 백업은 SoloSquad 라이프사이클 *외부* 자산 (이미 만들어진 안전망)으로 취급 |
| `--also-purge-backups` opt-in | 명시적으로 함께 정리 요청 시만 삭제. 디폴트 보존 |
| 새 머신에서의 의미 | archive는 새 머신에서 import 가능. 마이그레이션 백업은 그 머신의 옛 워크스페이스를 위한 것이라 이전 무의미 — `--also-purge-backups`로 정리 권장 (단, 사용자 결정) |

## 11. 비파괴성 + 복구

### 11.1 dry-run 우선 권장

`solosquad uninstall --dry-run`이 무엇이 archive로 가고 무엇이 삭제될지 트리·사이즈로 보여줌. README + manual + doctor가 "uninstall 실행 전 반드시 dry-run 한 번"을 권장.

### 11.2 archive 후 자체 무결성 검증

archive 작성 직후 zip을 다시 열어 manifest TSV의 SHA256과 zip 내 파일의 실제 SHA256을 대조. 불일치 시 archive를 폐기하고 cleanup 단계 진입 안 함 (사용자 데이터 보호 우선).

### 11.3 cleanup 단계 트랜잭션

cleanup은 (a) archive가 디스크에 무결하게 존재하는 것을 확인한 *후*에만 시작. (b) 삭제는 `<org>/repositories/` 화이트리스트로 *제외*하고 그 외 SoloSquad 자산만 (블랙리스트 아님 — 디폴트 보호). (c) cleanup 도중 실패 시 archive로 사용자 데이터를 다시 만들 수 있음을 안내.

### 11.4 archive 보존 기간

archive 위치는 기본 `~/solosquad-archive-<workspace-slug>-<ts>.zip`. SoloSquad는 이 파일을 자동 삭제하지 않음. 사용자 책임. README에 "이 archive를 별도 스토리지(클라우드·외장)에 백업하세요" 권장.

### 11.5 잘못된 archive 식별

archive.yaml의 `solosquad_version` + `archive_format`이 미래 import에서 호환 가능 여부 결정. `min_solosquad_version`보다 낮은 버전으로 import 시도하면 거부.

## 13. Q&A — 사용자 결정 사항 정리

| 질문 | v0.7 결정 |
|---|---|
| 초기화 기능을 둘 것인가? | **아니오**. `solosquad reset`·`solosquad clean` 같은 명령을 추가하지 않는다. 라이프사이클은 install ↔ uninstall 2단. "재설치"는 *uninstall + farewell + 새 워크스페이스 init*으로 자연 표현 |
| install/uninstall 2단 진입점이 충분한가? | **충분**. npm v7+ 글로벌 훅이 동작 안 하므로 npm 자체 라이프사이클은 부족. SoloSquad의 `init`·`uninstall`이 install 전/후 boundary를 명시적으로 정의 |
| 환경변수 처리 방향 | 알려진 secret 키 패턴은 archive에 **이름만 + 마스킹된 자리표시자**. 실제 값 미포함. 외부 server-side revoke는 `REVOKE-CHECKLIST.md`로 위임 — gh CLI 패턴 |
| 백업 처리 방향 | (a) **v0.7 archive** = 새로 생성되는 *작별 산출물*. uninstall은 항상 생성, skip 불가. (b) **마이그레이션 백업** (`~/.solosquad-backups/`) = 기존 안전망. uninstall은 디폴트로 손대지 않음. `--also-purge-backups`로 명시적 opt-in만 함께 정리. 두 트랙 분리 — 의미가 다른 자산을 같은 룰로 묶지 않음 |

## 15. Out of scope (v0.7에서 다루지 않음)

- **`solosquad import <archive.zip>`** — archive 복원 기능은 v1.0 슬롯. v0.7은 archive *생성*만 수렴. 단 archive 포맷은 v0.7에서 freeze해 v1.0 import가 호환되도록.
- **클라우드 자동 백업** — archive를 자동으로 S3/Drive에 업로드하는 기능. v1.x post-release 선택지.
- **개별 워크플로우 export** — 워크플로우 단위로만 export하고 싶다는 요청은 별도 `solosquad workflow export <wf-id>` 명령으로 (v0.5 워크플로우 메이커 확장 슬롯).
- **자동 토큰 revoke** — Discord/Slack/Anthropic API를 통해 SoloSquad가 직접 토큰을 revoke하는 기능. server-side OAuth 권한 + 사용자 confirmation flow 필요 → v1.x 검토.
- **`solosquad doctor --pre-uninstall`** — uninstall 전 dedicated 점검 모드. 일반 `--dry-run`으로 흡수 가능하므로 별도 추가 안 함.
- **CLI 그래픽 progress bar** — archive walking이 길어질 때 UX 개선. zip 라이브러리 선택 후 결정 (post-출시).
