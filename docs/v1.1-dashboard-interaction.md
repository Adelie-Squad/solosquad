# v1.1 — 대시보드 상호작용 (외부 대시보드 연동)

> 웹 대시보드 자체는 별도 리포지토리(`solopreneur-dashboard` / `solopreneur-api`)에서 개발되며, SoloSquad 본 리포는 그 대시보드와 **상호작용하는 인터페이스만** 제공한다. 본 스펙은 봇·스케줄러가 외부 대시보드로 상태를 푸시하고, 사용자가 대시보드에서 내린 명령을 받아 처리하는 양방향 채널의 계약을 정의한다.

**로드맵:** v0.1 크로스 플랫폼 → v0.2 안전/보안 → v0.3 PM 모드 → v0.4 자율 실행 → v0.5 워크플로우 메이커 → v0.6 디폴트 워크플로우 → **v1.0 정식 출시** → **v1.1 대시보드 상호작용** → v1.2 지식 온톨로지 → v2.0 커뮤니티 → v3.0 코워킹

**의존:** v1.0 (정식 출시 — 메신저-네이티브 코어가 안정된 후), v0.4 (자율 실행 — results.tsv 푸시), v0.5 (워크플로우 메이커 — 워크플로우 메타 푸시). v1.2(지식 온톨로지) 출시 이후 Knowledge Explorer 패널이 대시보드 쪽에서 추가됨.

**기준 버전:** v1.1.0

## 핵심 질문 & 성공 기준

| 핵심 질문 | 성공 기준 |
|---|---|
| 별도 리포에서 개발되는 대시보드와 SoloSquad가 안정적으로 상호작용하는가? | 대시보드 ↔ SoloSquad API 계약 v1, 5초 이내 상태 sync, 인증·권한 검증 통과 |
| 대시보드 없이도 SoloSquad CLI/메신저로 완결되는가? | 대시보드는 항상 *선택* 채널. 미연결 시 모든 핵심 기능이 그대로 동작 |
| 외부 대시보드의 명령이 시스템에 안전하게 들어오는가? | 명령 수신 시 PM 세션을 통한 의도 검증 + 사용자 confirmation gate |

## 1. 왜 별도 리포인가

| 이유 | 설명 |
|---|---|
| **릴리스 주기 독립** | 메신저-네이티브 코어(CLI, 봇, 스케줄러)는 보수적 릴리스. 대시보드는 디자인 반복 빠름. 한 리포에서 묶으면 양쪽이 발목 잡힘 |
| **기술 스택 격리** | 대시보드는 Next.js / TanStack 등 프론트엔드 스택. 본 리포는 Node CLI. 빌드·테스트·배포 분리 |
| **선택적 사용 보장** | 대시보드는 *항상* optional. 미설치 사용자도 모든 기능을 메신저·CLI로 접근. 같은 리포에 두면 "필수 컴포넌트로 오해"되기 쉬움 |
| **별도 보안 영역** | 대시보드는 OAuth·세션 쿠키·웹 공격 표면. CLI/봇과 다른 위협 모델. 격리해서 운영 |

## 2. 리포 맵

```
Adelie-Squad/
├── solosquad                  ← 본 리포: CLI + 봇 + 스케줄러 (현재)
├── solopreneur-dashboard      ← 별도 리포: Next.js 대시보드 (v1.1 짝)
└── solopreneur-api            ← 별도 리포: 대시보드 백엔드 API (v1.1 짝)
```

본 v1.1 스펙은 `solosquad` 리포 안에 머무는 변경만 다룬다 — 외부 두 리포의 내부 설계는 그 리포들의 별도 스펙.

## 3. 양방향 상호작용 모델

```
┌──────────────────────────────────────┐
│   solosquad (이 리포)                │
│                                      │

│                           │ push     │
│                           ▼          │

│                                      │
│   src/dashboard/inbox.ts             │
│     - 폴링 또는 webhook 수신          │
│     ▲ pull/push                      │
│     │                                │
└─────┼────────────────────────────────┘
      │
      ▼
┌──────────────────────────────────────┐
│  solopreneur-api (별도 리포)          │
│  - REST + 웹훅 디스패처               │
│  - SQLite/Postgres 영속               │
└──────────────────────────────────────┘
      ▲
      │ HTTP/SSE
      ▼
┌──────────────────────────────────────┐
│  solopreneur-dashboard (별도 리포)    │
│  - Next.js 프론트엔드                 │
│  - 시각화 / 명령 입력                  │

```

**핵심 원칙:**
- 본 리포는 *대시보드 미연결*에도 100% 동작
- `DASHBOARD_API_URL` 환경 변수 설정 시에만 client 활성
- 명령 수신은 *항상* PM 세션(v0.3)을 통과 — 외부 명령 ≡ 메신저 명령과 동일한 검증

## 5. API 계약 (외부 리포와 합의)

본 스펙은 본 리포에서 부르는 *클라이언트* 형태만 명시. 서버 측 자세한 내용은 `solopreneur-api` 리포 스펙 참조.

### 5.1 Push (SoloSquad → API)

| 엔드포인트 | 시점 | 페이로드 |
|---|---|---|
| `POST /events/routine` | 루틴 완료 직후 | routine_id, status, summary, artifacts, ts |
| `POST /events/pm` | PM session 메시지 in/out | session_id, kind, text, ts |
| `POST /events/workflow` | _status.yaml 변경 시 | workflow_id, stage_id, new_status, ts |
| `POST /metrics/cycle` | v0.4 사이클 결과 | cycle_n, metric, value, kept, commit |

페이로드 검증: 본 리포가 *발송 전* `types.ts` 스키마로 validate. 검증 실패는 로그 후 drop (대시보드 부재 시 안전).

### 5.2 Pull / Inbox (API → SoloSquad)

| 모드 | 트리거 | 동작 |
|---|---|---|
| **폴링 (기본)** | 5초 간격, exponential backoff on idle | `GET /inbox?since=<ts>` 호출 → 새 명령 list |
| **Webhook (선택)** | API가 dashboard에 webhook URL 등록 | localhost에 노출된 endpoint로 수신, 토큰 검증 |

수신된 명령은 PM 세션의 신규 메시지로 inject → v0.3 PM 모드가 명확화·spawn·report 처리.

### 5.3 인증

- **API 토큰**: 등록 시 1회 발급, `.solosquad/.env` 또는 OS 키체인 저장
- **요청 헤더**: `Authorization: Bearer <token>`, `X-SoloSquad-Workspace-Id: <uuid>`
- **재발급**: `solosquad dashboard link --rotate-token`
- **만료**: 토큰 30일 만료, 1주 전부터 doctor가 경고

## 6. 신뢰 모델

- 본 리포 시각에서 대시보드의 명령은 **신뢰 등급 L2** (인증된 자기 서버). v1.2에서 정의될 L0~L3 모델과 호환.
- 단, *대시보드를 통해* 메신저 PM처럼 새 워크플로 트리거하는 경우 → **사용자 confirmation gate 유지** (메신저에서 confirmation 받는 것과 동등)
- 대시보드는 절대 PM 컨텍스트나 specialist transcript에 직접 쓰기 불가. 모든 상태 변경은 PM 세션을 경유.

## 7. Out of scope (v1.1에서 다루지 않음)

- **대시보드 UI 자체** — `solopreneur-dashboard` 리포 스펙으로
- **API 서버 내부 구현** — `solopreneur-api` 리포 스펙으로
- **데이터베이스 스키마** — API 측 스펙으로
- **다중 워크스페이스 통합 뷰** — workspace 1개 1 사용자가 기본. 멀티-tenant는 v2.0
- **모바일 앱** — v2.0+

## 8. 마이그레이션 (v1.0 → v1.1)

스크립트 `1.0.0-to-1.1.0.ts`:
- `.solosquad/.env`에 `DASHBOARD_API_URL=`, `DASHBOARD_API_TOKEN=` 빈 항목 주입 (옵션 표기)
- `workspace.yaml`에 `dashboard: { enabled: false, push: true, poll_interval_seconds: 5 }` 필드 추가
- 버전 1.0.0 → 1.1.0 bump
- 별도 리포 안내 메시지 출력

비파괴. 대시보드 미사용자는 변경 없음.

## 9. 테스트 전략

- 대시보드 미설정 시 모든 CLI/봇 동작 그대로 (`enabled: false`)
- API 응답이 timeout이어도 라우틴/봇은 진행 (push 실패는 로그만)
- 잘못된 토큰으로 push 시 401 응답을 정상 처리 후 사용자 안내
- Inbox 수신 명령이 PM 세션에 정상 inject 되는지 E2E
- 토큰 만료 1주 전 doctor 경고 출력

## 10. 성공 기준 (측정 가능)

| 기준 | 측정 |
|---|---|
| `DASHBOARD_API_URL` 미설정 시 회귀 0건 | 회귀 테스트 |
| Push 평균 지연 < 1초 (localhost 가정) | 통합 |
| Inbox 명령 수신 → PM 처리 시작 < 3초 | E2E |
| 대시보드 다운 상황에서 본 리포 정상 동작 시간 100% | 신뢰성 |
| 토큰 자동 회전 (만료 1주 전) 성공률 ≥ 95% | 운영 |
