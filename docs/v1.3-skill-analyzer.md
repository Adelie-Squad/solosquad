> 기존 `.claude/skills/`를 가진 코드베이스를 SoloSquad 워크스페이스에 합칠 때, LLM이 skill을 분류·이주 제안하고 workflow 템플릿과 매칭하는 적응적 마이그레이션 레이어. v1.2.3 마이그레이션 프레임워크 위에서 동작.

**로드맵:** v1.1 크로스 플랫폼 → v1.2 안전/보안 → **v1.3 스킬 분석기 & Repo 온보딩** → v1.4 스킬 자유도 → v1.5 지식 온톨로지 → v1.6 웹 대시보드

**의존:** v1.2.3 (마이그레이션 프레임워크 — 동일한 dry-run / apply / rollback 계약 위에서 동작)

**기준 버전:** v1.3.0

## 핵심 질문 & 성공 기준

| 핵심 질문 | 성공 기준 |
|---|---|
| 기존 `.claude/skills/`가 있는 repo를 SoloSquad에 안전하게 합칠 수 있는가? | `solosquad analyze repo <path>` 1회 실행 → Markdown 보고서 생성 → 사람 승인 → `add repo --from-report`로 실제 이동 |
| 재실행이 누적 작업 없이 증분 처리되는가? | 두 번째 실행은 신규/수정된 skill에만 LLM 호출, 나머지는 ledger에서 스킵 |
| `sync`·`migrate`의 결정성이 깨지지 않는가? | LLM 호출은 `analyze` 전용. `sync`·`migrate`는 LLM-free, idempotent 유지 |

## 1. 현재 문제

| 문제 | 원인 |
|---|---|
| 기존 repo의 `.claude/skills/`가 워크스페이스 25 agent와 충돌 가능 | 같은 역할 skill이 양쪽에 있을 때 우선순위·머지 정책 부재 |
| 어떤 skill을 어느 레이어로 올릴지 판단 기준 없음 | 역할/워크플로우/코드베이스/도메인의 분류 휴리스틱 없음 |
| Workflow를 사람이 처음부터 짜야 함 | 기존 skill 체인이 SoloSquad 템플릿 4종 중 어디에 가까운지 자동 매칭 부재 |
| `sync`/`migrate`에 LLM을 섞으면 결정성이 깨짐 | sync는 `.org.yaml` ↔ 디스크 상태 정렬, idempotent 보장이 계약 |

## 2. 원칙

| 원칙 | 구현 |
|---|---|
| **Deterministic vs LLM 분리** | `sync`·`migrate`는 LLM 금지. `analyze`는 LLM 전용 신규 명령. |
| **Classify, don't lift** | 기존 skill을 4-라벨로 분류 후 맞는 레이어로 이주. 일괄 흡수 금지 (cross-repo 오염 방지). |
| **사람 승인 게이트** | `analyze`는 Markdown 보고서만 생성. 실제 이동은 `add repo --from-report` 명시 실행. autoresearch §C(a) propose-verify 패턴. |
| **Increment by hash** | Ledger에 `path + SHA256[:12]` 기록. 두 번째 실행은 신규/변경분만 LLM 호출. |
| **Workflow 매칭은 결정적** | 분류 결과의 집계 함수. LLM 추가 호출 없음. |
| **Markdown 출력** | 보고서·ledger 모두 사람이 읽고 `git diff` 가능. 베이스라인 §A 컨센서스. |

## 3. 3-Layer 위계와 역할 분담

| 레이어 | 소유 책임 | 비유 | 정합 레퍼런스 |
|---|---|---|---|
| **Workspace `.solosquad/`** | *WHO* — 에이전트 정의, 루틴, 코어 원칙·라이팅 스타일 | 정체성 (canonical agent library) | gstack 23 슬래시 패키지 |
| **Org `<workspace>/<org>/`** | *HOW* — 워크플로우, 핸드오프 규약, 채널·메신저 설정 | 프로세스 | gstack Think→Plan→Build 체인 |
| **Repo `<org>/repositories/<repo>/.claude/`** | *WHERE* — 이 repo만의 코드 컨벤션, 도메인 사전, 로컬 오버레이 | 컨텍스트 | phuryn auto-load |

### 3.1 관계 계약 (3줄)

1. **Workspace agent**: "나는 X다 — 추론 스타일·품질 기준은 이렇다"
2. **Org workflow**: "스테이지 간 핸드오프 규약은 이렇다"
3. **Repo skill**: "*이 repo*에서는 X를 이렇게 한다"

**핵심 원칙**: 에이전트는 repo skill을 "정체성"이 아니라 "데이터"로 로드한다. 같은 이름의 역할 skill이 repo에 있으면 → 워크스페이스 흡수 후보 시그널.

### 3.2 로딩 순서 (phuryn dual-trigger 정렬)

```
1. Workspace agent SKILL.md          (canonical 정의)
2. Org TEAM_KNOWLEDGE.md + workflow  (프로세스 컨텍스트)
3. Repo .claude/skills/*.md          (로컬 오버레이 — 마지막 머지)
```

## 4. 분류 라벨 (4종)

| 라벨 | 시그널 | 이주 대상 | 예시 |
|---|---|---|---|
| **role** | "Act as X", 코드 무관, 페르소나·추론 스타일 정의 | Workspace agent (기존 25개와 머지 또는 신규) | `pm-review.md`, `code-reviewer.md` |
| **workflow** | 스테이지 핸드오프 체인 (A→B→C), 다중 단계 산출물 | Org `<org>/workflows/` | `prd-to-impl.md`, `release-pipeline.md` |
| **codebase-fact** | 특정 파일·경로·배포 타깃·토큰 참조 | Repo `.claude/skills/` 유지 | `deploy-staging.md`, `tailwind-tokens.md` |
| **domain** | 이 제품만의 용어·비즈니스 룰·도메인 사전 | Repo 유지 (제품이 cross-repo면 org) | `pricing-rules.md`, `compliance-checklist.md` |

### 4.1 Disambiguation 규칙

- 다중 라벨 후보 시 우선순위: `codebase-fact` > `domain` > `workflow` > `role` (구체 → 추상)
- LLM이 confidence ≥ 0.7로 단일 라벨 확정 못 하면 `ambiguous`로 마킹 → 사람 판정 필수, 자동 이동에서 제외
- `role` 라벨일 때 LLM은 25 workspace agent 중 머지 후보를 함께 출력 (`merge_target`)

## 5. Workflow 매칭

분류 결과의 **집계 함수** — LLM 추가 호출 없음, deterministic.

### 5.1 알고리즘

```
1. ledger.analyzed[].classification == "workflow"인 skill만 추출
2. 각 workflow skill을 SoloSquad 템플릿 4종에 매핑:
   - PMF Discovery     (research → planning → design → build → launch)
   - Feature Expansion (analysis → planning → design → build)
   - Rebranding        (research → branding → design → marketing)
   - Rapid Prototype   (refine → build → launch)
3. 템플릿별 cover_rate = (매핑된 스테이지 수) / (템플릿 전체 스테이지 수)
4. 최고 cover_rate가 추천 템플릿. 누락 스테이지 + 미매핑 skill을 보고서에 명시.
5. 모든 cover_rate < 0.5면 "no-match" → 커스텀 workflow 정의 권유
```

### 5.2 출력 예

```markdown
## Workflow 추천
- 가장 가까운 템플릿: Feature Expansion (cover_rate: 0.87)
- 매핑된 스테이지:
  - analysis ← `skills/competitor-scan.md`
  - planning ← `skills/feature-spec.md`
  - build    ← `skills/impl-checklist.md`
- 누락 스테이지: design
- 미매핑 skill: `skills/ab-test-gate.md` → 커스텀 스테이지 후보
```

## 6. Analyzed Ledger

### 6.1 위치

`<workspace>/<org>/repositories/<repo>/.solosquad/analysis-ledger.yaml`

- repo와 함께 이동
- 기본 git 포함 (`solosquad init` 시 옵션으로 변경 가능)
- 사람이 audit 가능

### 6.2 Shape

```yaml
version: 1
analyzed:
  - path: .claude/skills/deploy-staging.md
    hash: a3f1c8e2b9d4               # SHA256[:12]
    analyzed_at: 2026-05-11T14:32:00Z
    classification: codebase-fact
    confidence: 0.92
    destination: repo (stay)
    report: analysis/2026-05-11-initial.md
    applied: false                   # add repo --from-report 실행 여부
    applied_at: null
  - path: .claude/skills/pm-review.md
    hash: 7c2e9d1a4f88
    analyzed_at: 2026-05-11T14:32:00Z
    classification: role
    confidence: 0.81
    merge_target: workspace/strategy/pmf-planner
    destination: workspace agent (merge: append section)
    report: analysis/2026-05-11-initial.md
    applied: false
workflow_match:
  template: feature-expansion
  cover_rate: 0.87
  basis_hash: 8f3a2c1b                # 정렬된 classification entries의 SHA256[:8]
  computed_at: 2026-05-11T14:32:00Z
model:
  name: claude-opus-4-7
  fingerprint: "opus-4-7-2026-04"     # --force 트리거용 모델 ID
```

### 6.3 다음 실행 diff 규칙

| 상황 | 동작 |
|---|---|
| ledger에 있고 hash 동일 | 스킵 (LLM 호출 없음) |
| ledger에 있고 hash 변경 | 재분석 → 기존 entry 교체 |
| ledger에 없는 새 파일 | 분석 |
| 파일은 없는데 ledger에 있음 | orphan 마킹, 보고서에 명시 |
| ledger의 `basis_hash` ≠ 현재 entries 해시 | workflow 매칭만 재계산 (LLM 호출 없음) |
| ledger.model.fingerprint ≠ 현재 모델 | 경고 출력, `--force` 권유 |

## 8. 아키텍처 (코드 레벨)

### 8.1 디렉토리 배치

```
src/
  analyze/
    index.ts                     # 진입점
    types.ts                     # Classification, Ledger, WorkflowMatch
    scanner.ts                   # .claude/skills/ 발견 + SHA256 hash
    ledger.ts                    # ledger 읽기/쓰기/diff
    classifier.ts                # LLM 호출 (4-label 분류, 배치)
    workflow-matcher.ts          # deterministic 집계
    report-writer.ts             # markdown 보고서 생성
    applier.ts                   # --from-report 적용 (deterministic)
    prompts/
      classify-system.md         # 시스템 프롬프트 (prompt cache 대상)
      classify-user.md           # skill 배치 입력 템플릿
  cli/
    analyze.ts                   # solosquad analyze repo
    add-repo.ts                  # 기존 + --from-report 옵션
```

### 8.2 Classifier 인터페이스

```typescript
export type ClassificationLabel =
  | "role"
  | "workflow"
  | "codebase-fact"
  | "domain"
  | "ambiguous";

export interface ClassificationResult {
  label: ClassificationLabel;
  confidence: number;
  destination: string;            // 사람이 읽는 이주 대상 (한 줄)
  merge_target?: string;          // role일 때 머지 후보 워크스페이스 agent 경로
  reasoning: string;              // LLM이 남긴 짧은 설명 (1-2 문장)
}

export interface Classifier {
  classify(
    skills: SkillFile[],
    context: WorkspaceContext,
  ): Promise<ClassificationResult[]>;
}
```

### 8.3 LLM 호출 전략

- **배치**: 한 번에 최대 10 skill을 단일 프롬프트로 (토큰 비용 최적화)
- **시스템 프롬프트**: 4-label 정의 + workspace agent 25개 목록 + workflow 템플릿 4종 — Hermes §3 cache-aware snapshot 패턴 차용, prompt cache로 고정
- **출력**: 구조화된 JSON (Anthropic SDK structured output) → JSONL로 ledger에 결합
- **모델**: `claude-opus-4-7` 기본, `--model` 플래그로 변경 가능. fingerprint를 ledger에 기록해 모델 드리프트 감지

### 8.4 v1.2.3 마이그레이션 프레임워크 위에 얹기

- `analyze`는 Migration 인터페이스를 사용하지 않음 (스키마 변환이 아니라 컨텐츠 분류이므로)
- 하지만 `add repo --from-report`의 *적용 단계*는 동일한 backup → apply → verify → rollback 흐름을 재사용
- `applier.ts`는 `src/migrations/backup.ts`·`src/migrations/runner.ts`를 직접 import

## 9. 테스트 전략

### 9.1 Fixture 기반

```
test/analyze/fixtures/
  empty-skills/                     # .claude/skills/ 없음 → 보고서 "nothing to do"
  pure-codebase-facts/              # 전부 codebase-fact (이주 0건, repo 유지)
  mixed-typical/                    # 4 라벨이 골고루
  ambiguous-heavy/                  # ambiguous 비율 높음 (사람 개입 강조)
  matches-feature-expansion/        # workflow 매칭 cover_rate ≥ 0.85
  no-workflow-match/                # 모든 cover_rate < 0.5 → 커스텀 권유
```

### 9.2 분류 안정성 테스트

- 같은 fixture를 100회 분류 시 라벨 일치율 ≥ 95% (LLM 결정성 검증)
- 일치율 < 95%면 시스템 프롬프트 회귀 신호 → CI 알람
- 모델 업그레이드 시 fixture 전체를 재실행, 기존 분류와 diff 리포트

### 9.3 회귀 시나리오

- 두 번째 실행 시 LLM 호출 0회 (ledger 정확성)
- skill 수정 후 hash 변경 시 그 skill만 재분석
- skill 삭제 후 orphan 마킹
- `--force` 시 전체 재분석 + ledger 덮어쓰기
- `--from-report` 적용 후 두 번째 실행 시 `applied: true` 항목 스킵

- [ ] Ambiguous (confidence < 0.7) skill은 자동 이동에서 제외, 보고서에 manual 표시
- [ ] Orphan skill (파일 없는데 ledger에 있음)이 보고서에 명시되고 `--prune-orphans`로 정리됨
- [ ] Fixture 6종 모두 통과
- [ ] 100회 분류 시 라벨 일치율 ≥ 95%
- [ ] `solosquad migrate`·`sync`에는 LLM 호출이 일절 없음 (deterministic 보장)
- [ ] Backup이 `~/.solosquad-backups/<ISO8601>-repo-onboard/`에 생성, `migrate --rollback`으로 복원 가능

## 12. 남은 결정 (Open Questions)

1. **머지 정책 기본값**
   같은 역할 skill이 workspace agent와 repo 양쪽에 있을 때:
   - **append section** (둘 다 보존, repo 내용을 워크스페이스 SKILL.md의 `## Repo-Specific Adaptations` 섹션으로 추가) — *추천 기본값*
   - override (repo가 이김)
   - replace (workspace가 이김, repo 삭제)
   - 사용자 옵션으로 `--merge-policy` 플래그 노출.

2. **Orphan retention**
   ledger에 있는데 파일이 없는 경우:
   - 즉시 prune
   - **1년 보존 + 보고서에 명시 + `--prune-orphans` 수동 트리거** — *추천 기본값* (audit 흔적)

3. **Ledger git 포함 여부**
   - **기본 포함** (투명성, 팀 협업) — *추천 기본값*
   - gitignore (메타 노출 회피)
   - `solosquad init` 시 사용자에게 질문 → `.gitignore` 자동 조정
