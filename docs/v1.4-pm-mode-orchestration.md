# v1.4 — PM 모드 + 멀티 에이전트 오케스트레이션

> 사용자는 `#owner-command`에서 **PM 에이전트 1명과만 대화**한다. PM이 요청을 PRD화하고, 적절한 조직·리포지토리를 결정하고, 25명 specialist 중 필요한 에이전트들을 ephemeral subagent로 spawn해서 워크플로를 실행한다. 사용자에게는 통합 보고만 도달. 멀티턴, 스레드 기반 진행, 파일 기반 영속.

**기준 버전:** v1.2.4 (직전 릴리스)
**분류:** 패러다임 전환 — 단발 단일 에이전트 → 영구 PM + ephemeral specialist roster
**심각도:** High — 사용자가 시스템을 인식하는 방식 자체가 바뀜. 1차 사용자 가치(PRD→실행→보고)의 핵심.
**선행 작업:** v1.2.4 (`#owner-command` + `#workflow` 2채널 구조, workspace.yaml briefings 인프라)

---

## 1. 배경

### 1.1 v1.2.x까지의 한계

v1.2.4 기준 `solosquad bot`의 실제 동작:

```
사용자 메시지 → agent-router 키워드 매칭 → SKILL.md inject → claude --print (1회) → 응답
```

- `claude --print`는 **single-shot 무상태** 호출. 매 메시지가 분리된 process
- **사용자 대화 history 없음** — 직전 메시지 기억 불가, 멀티턴 명확화 불가
- **PM 개념 없음** — 키워드가 직접 단일 에이전트 SKILL을 선택
- **워크플로 진행자 없음** — `_status.yaml`은 존재하지만 stage를 자동으로 advancing하는 코드 없음
- **`assets/orchestrator/SKILL.md` 존재하지만 봇이 호출하지 않음**

결과: 1인 founder가 "랜딩페이지 리디자인 진행"이라고 입력해도 봇은 그 작업을 PRD화하지 않고, 단일 에이전트 한 번 답하고 끝남.

### 1.2 사용자가 원하는 동작 (요청 그대로)

> "owner-command는 기본적으로 core와 orchestrator 단의 pm 역할의 에이전트가 응답을 해야지. 소통은 하나의 에이전트가 전담해. 요청에 따라 해당 에이전트가 관련 조직, 리포지토리를 찾아서 워크플로우를 실행시키는 거고."

핵심 요건:
1. **PM 에이전트 1명이 사용자와의 모든 소통 전담**
2. **PM이 조직·repo를 동적으로 결정**해서 워크플로 실행
3. 멀티 에이전트 협업의 산출물을 PM이 통합해서 사용자에게 보고

### 1.3 2026년 컨센서스 (조사 결과 요약)

- **Anthropic Effective Harnesses paper (2025-11)**: 컴팩션은 부족, **context resets + durable artifacts**가 load-bearing
- **Claude Code Task tool**: 각 subagent는 **fresh conversation**, 자체 prompt + tool allowlist, **중간 tool calls는 안에 머물고 final message만 부모에게 return**
- **Claude Managed Agents (2026-04)**: Coordinator + roster of specialists, depth=1, **max 25 threads** (SoloSquad의 25 agents와 정확히 일치), session = Claude context 밖의 event log
- **OpenAI**도 같은 방향(Assistants API → Responses + developer-owned state, 2026 H1 sunset)
- **LangGraph/CrewAI**: typed structured context, worker에는 slice만 전달

→ SoloSquad의 PM + 25 specialists + filesystem state는 **컨센서스 토폴로지**.

## 2. 목표

1. **PM 영구 세션** — 사용자(founder)당 1개의 Claude session이 #owner-command thread를 통해 영속
2. **Subagent ephemeral 실행** — Claude Code의 Task tool 패턴을 흉내 (fresh session, fresh context, scoped tools)
3. **워크플로 상태 파일 영속화** — `_status.yaml`, `_handoff.md`, `_events.jsonl`이 진실의 원천
4. **PM compaction** — 완료 워크플로를 PM thread에서 축출, skill markdown으로 외부화
5. **사용자 UX 변화 없음** — `#owner-command`에 자연어로 말하면 PM이 알아서 처리. PRD·실행·보고 모두 thread 안에서

## 3. 세션 모델 결정

### 3.1 채택: 옵션 C (하이브리드)

```
┌─ #owner-command (Slack/Discord thread per workflow OR channel root) ──┐
│  PM session — long-lived, user-bound                                  │
│  - Claude Code session id (--resume <id>)                             │
│  - system prompt = orchestrator/SKILL.md (rewritten for PM mode)      │
│  - state: routing decisions, active workflows, user intent            │
│  - does NOT hold: full subagent transcripts                           │
└───────────────────┬───────────────────────────────────────────────────┘
                    │ Task("desk-researcher", "competitor analysis for X",
                    │      handoff="<wf-id>/_handoff.md")
                    ▼
        ┌───────────────────────────────────┐
        │  Ephemeral subagent (Task tool)   │
        │  - fresh Claude session           │
        │  - prompt = agent SKILL.md +      │
        │            _handoff.md (slice)    │
        │  - returns: final message only    │
        │  - writes: _handoff.md (next),    │
        │            workflow artifacts     │
        └───────────────────────────────────┘
                    │
                    ▼
        Filesystem (durable artifacts — load-bearing)
        ├─ <org>/workflows/<id>/_status.yaml    state machine
        ├─ <org>/workflows/<id>/_events.jsonl   event log (NEW)
        ├─ <org>/workflows/<id>/_handoff.md     typed slice
        ├─ <org>/workflows/<id>/stage-N-*/      stage artifacts
        └─ <org>/memory/pm-skills/*.md          PM compaction (Hermes-style)
```

### 3.2 PM 세션 영속화 — `--resume <id>` 채택

| 방법 | 채택 여부 | 이유 |
|---|---|---|
| `claude --resume <session-id>` | ✅ | Claude Code native. prompt cache 자동. 외부 history 관리 불필요 |
| stdin으로 jsonl history 주입 | ❌ | 매 호출마다 토큰 비용, cache hit 떨어짐 |
| 자체 conversation 저장소 | ❌ | 중복 노력. Claude Code가 이미 처리 |

session-id는 사용자별로 1개 — `<org>/.solosquad/sessions/{user-id}.json` 파일에 저장. 첫 메시지에서 새 session 생성, 이후 메시지는 모두 `--resume`.

### 3.3 Sub-agent 실행 — Task tool 흉내

Anthropic의 Managed Agents API는 안 씀 (self-hosted 철학 + cloud 의존성 회피). 대신 **Claude Code subagent 패턴을 봇 코드에서 직접 구현**:

```typescript
spawn(agentSpec: { team, agent, task, handoffPath, repoSlug })
  → loadSkill(agentSpec.team, agentSpec.agent)
  → loadHandoff(agentSpec.handoffPath)
  → buildPrompt(skill, handoff, task)
  → runFreshClaude(prompt, repoCwd)  // NO --resume, fresh context
  → result = stdout (final message)
  → appendEvent(workflow.eventsLog, { agent, task, result, timestamp })
  → writeNextHandoff(workflow, result)
  → return summary to PM
```

PM은 결과 **요약**만 받음. 전체 transcript는 `_events.jsonl`에 저장됨. PM thread 컨텍스트는 작게 유지.

### 3.4 Depth 1 제한

Claude Managed Agents 사양과 동일하게 **PM → subagent (depth 1)** 로 제한. subagent가 다시 subagent를 spawn하지 않음. 만약 stage가 추가 작업을 요구하면 PM에게 보고하고 PM이 다음 subagent를 spawn. 추적 가능성·격리·디버깅을 위한 결정.

### 4.2 PM 호출 흐름 (사용자 메시지 1건)

```
1. 사용자가 #owner-command에 "랜딩페이지 리디자인 진행" 입력
2. bot/index.ts → handleCommand
3. session-store에서 user-id로 session 조회
   ├─ 없으면 새 session-id 생성, 첫 메시지로 PM 부트스트랩
   └─ 있으면 기존 session-id 사용
4. workspace-meta로 메타정보 빌드 (조직 목록, agents 목록, 활성 워크플로)
5. pm-runner.runPm({
     sessionId,
     userMessage,
     meta,
     spawnHandler: subagent-runner.spawn,
   })
6. Claude Code session에서 PM 응답 생성
   ├─ "명확화 질문" → 텍스트로 reply
   ├─ "spawn(agent, task, ...)" → spawnHandler 호출 → 결과를 PM thread에 inject
   ├─ "최종 보고" → 텍스트로 reply
7. session-store 갱신 (last interaction, active workflow)
```

### 4.3 PM SKILL 재작성 골자

```markdown
# Orchestrator (PM) — SoloSquad

## Identity
당신은 SoloSquad의 PM입니다. 사용자(1인 founder)와 직접 소통하는 유일한 에이전트.

## Core rules
1. **YOU NEVER DO THE WORK**. 모든 실행은 specialist subagent에 위임.
2. 사용자 의도가 모호하면 **간결한 명확화 질문**(1-2개)을 한 번에 던지고 답을 기다림.
3. 의도 확정 후 **PRD를 1페이지로 압축**하고 `<org>/workflows/wf-YYYY-MM-DD-<slug>/PRD.md`에 저장.
4. 워크플로 단계를 분해해서 `_status.yaml`에 stage 목록 작성. 각 stage = (agent, target_repo, depends_on).
5. ready stage를 발견하면 **spawn(agent, task, handoff_path)**을 호출. 한 번에 1 stage만 (parallel은 v1.5).
6. spawn 결과 요약을 받으면 다음 stage 결정 또는 사용자에게 보고.

## Available actions
- `ask(question)` — 사용자에게 질문 (메신저로 reply됨)
- `spawn(agent, task, handoff_path, repo_slug?)` — specialist subagent 호출
- `report(message)` — 사용자에게 최종/중간 보고
- `update_status(workflow_id, stage_id, new_status)` — _status.yaml 갱신

## Available agents (25)
... [strategy/growth/experience/engineering 팀별 명단 + 1줄 설명]

## Available organizations
[workspace-meta가 동적으로 inject]

## Workflow naming
wf-YYYY-MM-DD-<slug> (slug은 lowercase-hyphen, 30자 이내)

## When stuck
사용자에게 ask. 추측으로 진행 금지.
```

## 5. CLI / 메신저 변경

### 5.1 메신저 동작 변화

| v1.2.4 | v1.4 |
|---|---|
| 키워드로 단일 에이전트 매칭 후 1회 답변 | PM이 멀티턴 명확화 → spawn → 최종 보고 |

| `#workflow` 채널은 라우틴·시스템 스레드 | `#workflow`(또는 v1.2.5의 `#work-sessions`)에 워크플로별 thread 생성, 진행 alert |

### 5.2 신규 CLI

```bash
solosquad pm status       # 활성 PM session, 진행 중인 워크플로 목록
solosquad pm reset        # PM session 새로 시작 (기존 history 보관, archive)
solosquad pm compact      # PM compaction 수동 실행 (완료 워크플로 외부화)
solosquad workflow list   # 모든 워크플로 + 상태
solosquad workflow show <id>  # 특정 워크플로의 _status / _handoff / _events 요약
```

### 5.3 기존 키워드 라우터 fallback

PM session 시작 전(첫 메시지 도착 시)에는 기존 `agent-router.findAgent` 결과를 PM의 **힌트**로 전달. PM이 그걸 받아들이거나 다른 에이전트를 골라도 됨. 점진적 마이그레이션 — v1.4 출시 직후에도 기존 사용자가 단발 명령을 보내면 PM이 quick-q로 인식해서 1회 spawn으로 처리.

## 6. PM Compaction 라우틴 (Hermes 패턴)

`assets/routines/pm-compaction.md` — 매일 23:00 (workspace.yaml에 추가 가능):

1. `<org>/workflows/*/`에서 status=completed 워크플로 스캔
2. 각 워크플로의 `_events.jsonl` + `_handoff.md`를 읽어 **2-3 단락 markdown 요약** 생성
3. 결과를 `<org>/memory/pm-skills/<wf-id>.md` 로 저장
4. PM session에 "completed workflow 요약 → 새 skill" 통보 (단일 system message)
5. PM이 자기 thread에서 해당 워크플로 transcript를 잊어도 안전한 상태가 됨

이로써 PM thread는 **최근 active 워크플로만** 컨텍스트로 들고, 오래된 건 디스크에서 query 가능한 형태로 살아 있음.

## 7. 마이그레이션 (v1.2.4 → v1.4)

### 7.1 자동

```bash
solosquad migrate --apply
```

스크립트 `1.2.4-to-1.4.0.ts`가 수행:
- `<org>/.solosquad/sessions/` 디렉토리 생성 (사용자별 session-id 저장소)
- `<org>/memory/pm-skills/` 디렉토리 생성
- 기존 `<org>/workflows/*/`에 `_events.jsonl` 빈 파일 추가 (PM이 이후 append)
- workspace.yaml에 `pm: { compaction_time: "23:00", enabled: true }` 필드 주입
- 버전 1.2.4 → 1.4.0 bump

### 7.2 비파괴

- 기존 `_status.yaml`, `_handoff.md`, JSONL 메모리 그대로 보존
- 기존 키워드 라우팅 로직 fallback으로 남김
- 첫 메시지에 PM session이 새로 생성되므로 사용자 측 작업 없음

### 7.3 사용자 안내

```
v1.4부터 #owner-command에서 PM 에이전트가 응답합니다.
- "기능 추가해줘" 같은 모호한 요청도 PM이 명확화 질문 → PRD → 워크플로 실행으로 자동 진행합니다.
- 단발 질문("이 함수 뭐 하는 거야")은 그대로 PM이 quick-q로 처리합니다.
- 진행 중 워크플로 상태는 `solosquad workflow list`로 확인.
```

## 9. 비파괴성·롤백

- **데이터**: JSONL 메모리·workflows 디렉터리 손대지 않음
- **PM session 데이터**: `.solosquad/sessions/`는 v1.4부터 새로 생성. 롤백 시 무시 (재사용도, 삭제도 안전)

- **롤백**: `solosquad migrate --rollback`은 v1.2.4 workspace.yaml 복원. PM session 파일은 그대로 남지만 사용 안 됨

## 10. Out of scope (v1.4에서 다루지 않음)

- **Parallel stage 실행** — 동일 워크플로의 독립 stage를 동시에 spawn하는 건 v1.5
- **Cross-org workflow** — PM이 여러 조직에 걸친 워크플로를 1개로 묶는 건 v1.5+
- **Subagent depth > 1** — subagent가 다시 subagent spawning은 영구히 금지 (Anthropic Managed Agents와 동일)
- **사용자 1명 ≠ 1 PM session** — 다중 사용자 공유 PM은 v1.5+에서 워크스페이스 레벨 결정
- **PM의 자율 trigger** — 메신저 메시지 없이 PM이 스스로 워크플로 시작하는 건 v1.5의 자율 실행 엔진
- **대시보드 시각화** — v1.8
