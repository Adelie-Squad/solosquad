# v0.4 — 밤새 자율 작업 완료 엔진

> "밤에 지시하고 아침에 결과를 받는다." Autoresearch 패턴을 차용한 자율 실행 루프.

**로드맵:** v0.1 크로스 플랫폼 → v0.2 안전/보안 → v0.3 PM 모드 → **v0.4 자율 실행** → v0.5 워크플로우 메이커 → v0.6 디폴트 워크플로우 → v1.1 대시보드 상호작용 → v1.2 지식 온톨로지 → v2.0 커뮤니티 → v3.0 코워킹

**의존:** v0.1 (크로스 플랫폼), v0.2 (안전/보안 — 밤새 자율 실행 전 보안 기반 필수), v0.3 (PM 모드 — PM이 자율 루프의 진행자)

## 1. 현재 문제

| 문제 | 원인 |
|------|------|
| 에이전트가 1회성 | 매번 사람이 다음 단계를 지시해야 함 |
| 루틴이 보고만 함 | 인사이트 → 행동 연결 없음 |
| 실험 추적 없음 | 뭘 시도했고 뭐가 실패했는지 모름 |
| 밤새 돌릴 수 없음 | 에이전트 체이닝, 자동 판단, 시간 관리 부재 |

## 2. Autoresearch 패턴 차용

Karpathy의 [autoresearch](https://github.com/karpathy/autoresearch) 핵심 구조:

```
program.md (사람: 방향 설정) → train.py (AI: 실행) → 메트릭 측정 → keep/discard → 반복
```

| 원칙 | 적용 |
|------|------|
| `program.md` = 의도 전달 | 프로젝트별 `program.md`로 실행 방향 정의 |
| 단일 메트릭 판단 | keep/discard 자동 판단 기준 명시 |
| Git = 실험 메모리 | 성공만 커밋, 실패는 revert + 로그 |
| 자율 반복 루프 | 시간 예산 내 무한 반복 |

### 2.1 Data Reconciliation 원칙 (보고서 §4 차용)

Autoresearch만 차용하면 **Goodhart 법칙**에 노출됨 — 에이전트가 메트릭 산출 로직 자체를 왜곡하면 keep/discard가 무의미해진다. 보고서의 "지표 산출 로직과 데이터 마트 원천값의 일치 여부 확인" 원칙을 v0.4 기본 룰로 채택:

1. **`metric_provenance` 필드 의무화** — 모든 메트릭 값은 산출 공식·원천 파일 경로·라인 범위를 함께 기록한다. results.tsv·_events.jsonl 양쪽에 포함.
2. **재계산 가능성** — 같은 commit에서 evaluator를 재실행하면 동일 값이 나와야 한다. `solosquad run --verify <cycle>`로 검증.
3. **공식 변경은 새 사이클** — metric 산출 공식 자체를 바꾸면 이전 사이클과 비교 불가. 새 results.tsv 섹션으로 분리.

이 룰이 없으면 자율 루프는 "메트릭 자체를 조작하면 cheap win"의 함정에 빠진다.

## 3. 자율 실행 아키텍처

```
사용자 지시 (program.md)
    ↓
┌─────────────────────────────────┐
│  Execution Engine                │
│                                  │
│  1. program.md 파싱              │
│     → 목표, 메트릭, 범위, 에이전트 체인  │
│                                  │
│  2. 에이전트 체이닝 실행            │
│     → agent A → handoff → agent B → ...  │
│                                  │
│  3. 사이클당 평가                  │
│     → 메트릭 측정 → keep/discard   │
│     → Git 커밋 (keep) / reset (discard)  │
│     → results.tsv 기록            │
│                                  │
│  4. 시간 예산 소진까지 반복          │
│     → 다음 사이클 자동 설계          │
│                                  │
│  5. 완료 → 보고서 생성 + 메신저 전송   │
└─────────────────────────────────┘
    ↓
아침에 결과 확인
```

## 4. `program.md` 설계

```markdown
# Program: {프로젝트명}

## 목표
{무엇을 달성할 것인가}

## 메트릭
- {metric_name}:
    formula: {산출 공식, 1줄}              # provenance — 재계산 가능해야 함
    source: {원천 파일·경로 또는 URL}      # provenance — 측정 기준점
    threshold: {keep 판단 값}

## 에이전트 파이프라인
1. {agent_name}: {이 단계에서 할 일}
2. {agent_name}: {이 단계에서 할 일}
3. ...

## Guardrails (3단계 — 보고서 §2 감사 렌즈)
### Input
- 수정 가능 경로: {파일/디렉토리 목록}
- 수정 불가 경로: {파일/디렉토리 목록}
- API 비용 한도: ${N} per cycle, ${M} total
- 외부 호출 화이트리스트: {도메인 목록} (없으면 외부 차단)

### Runtime
- 사이클당 시간 예산: {N}분
- 1 stage 응답 타임아웃: {N}초
- N회 연속 discard 시 강제 종료 (교착 탈출, 기본값 5)

### Output
- 산출물 스키마 검증: results.tsv 행 형식 / commit 메시지 패턴
- 외부 부수효과 금지: 메신저 전송·이메일·결제 — 자율 루프 동안 비활성

## 시간 예산
{N}시간 또는 {N}사이클

## 종료 조건
- 모든 메트릭 > threshold 달성 시
- 시간 예산 소진 시
- N회 연속 discard 시 (교착 탈출)
- API 비용 한도 초과 시 (Input 가드)
```

**3단계 구분의 의도**: 보고서의 "사전 가드레일"은 솔로 환경에서 인프라 비용이 큼. v0.4는 *외부 비가역 부수효과만* Input/Output 가드로 사전 차단하고, 나머지(코드 변경·메모리 변경)는 `git reset`으로 사후 롤백한다. v0.3 `git-snapshot`과 결합:

```
spawn-pre commit  →  agent execute  →  metric eval
                                            │
                                  ┌─────────┴─────────┐
                                  keep                discard
                                  │                   │
                                  commit              git reset --hard pre
                                  results.tsv         results.tsv (status=discard)
```

## 5. 실험 추적

**results.tsv** (append-only):
```tsv
cycle	timestamp	agent	metric	value	status	commit	provenance	description
1	2026-03-19T22:05	desk-researcher	signal_relevance	0.45	discard	-	formula=count(matching_keywords)/total;source=memory/signals.jsonl	초기 시장 스캔
2	2026-03-19T22:32	desk-researcher	signal_relevance	0.72	keep	a1b2c3d	formula=count(matching_keywords)/total;source=memory/signals.jsonl	타겟 커뮤니티 집중 스캔
3	2026-03-19T23:01	data-analyst	insight_actionability	0.68	keep	d4e5f6g	formula=actionable_count/insight_count;source=workflows/<id>/stage-2/insights.md	패턴 분석
```

**provenance 컬럼 (보고서 §4 Data Reconciliation 적용)**: 같은 commit에서 동일 provenance로 재계산 시 동일 value가 나와야 한다. `solosquad run --verify <cycle>`로 자동 검증. 공식 변경 시 새 메트릭 이름 사용.

**Git 커밋 컨벤션:**
```
[cycle-{N}] {status} {agent}: {description} ({metric}={value})
```

**signal-scan active trigger 연동 (Tier 2.F)**:

기존 v0.2.4 `signal-scan` routine은 read-only — 발견한 신호를 JSONL에 적재만 했다. v0.4부터는 `urgency: high/med/low` 필드를 결과 JSON에 추가하고, `urgency=high` 신호 발견 시 evaluator가 즉시 **새 cycle을 자동 큐잉**한다.

```
signal-scan (12:00 daily)
  → urgency=high 신호 1건 이상
    → results.tsv에 cycle 시작 기록
    → program.md.signal-followup 자동 로드 (사용자 사전 정의)
    → desk-researcher 또는 business-strategist spawn
  → 결과를 #owner-command PM thread로 알림 (사용자가 keep/discard 최종 승인)
```

신호 → 행동 사이의 인간 검토 step은 *유지*. autonomous decision은 아니되, 자동 트리거 큐는 작동.

## 6. CLI

```bash
# 밤새 자율 실행
solosquad run --program program.md --hours 8

# 단일 사이클만 실행 (디버깅)
solosquad run --program program.md --cycles 1

# 현재 진행 상황 확인
solosquad run --status
```
