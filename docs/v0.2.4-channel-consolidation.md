# v0.2.4 — 채널 2개 통합 + 브리프 기반 일과 운영

> Discord/Slack 채널을 **`owner-command` + `workflow` 2개**로 통합하고, 5개 routine을 **하루 2회 브리프(아침/저녁)** 모델로 단순화한다. 모든 과제 진행은 `#workflow` 채널 내부 **스레드**에서 이루어지며, 에러·중간 보고·라우틴 알림이 동일 스레드에서 관리된다. 사용자는 setup 단계에서 **타임존과 브리프 시간**을 선택하고, 기본값은 Asia/Seoul (UTC+09) · 08:00 / 18:00.

**기준 버전:** v0.2.1 (직전 릴리스)
**분류:** UX 단순화 — 비파괴 변경 (메신저 측 기존 채널은 그대로 유지, 사용자가 수동 아카이브)
**심각도:** Medium — 신규 사용자의 셋업 비용을 크게 낮추고, 1.4 PM 모드의 기반을 만든다. 기존 사용자의 데이터(JSONL/workflows)는 영향 없음.

---

## 1. 배경

### 1.1 현재 구조의 문제

| 항목 | 현재 (v0.2.x) | 문제 |
|---|---|---|
| 채널 수 | 6개 (`owner-command`, `daily-brief`, `experiments`, `weekly-review`, `signals`, `errors`) | `errors` 미사용 / 4개 일방향 채널이 시간대만 다른 routine output 분산 — 시각적 부하 |
| 채널 의미 | 시간/주제 혼합 (`daily-brief`는 시간대, `experiments`는 주제) | 사용자 멘탈 모델 일관성 부재 |
| Routine 수 | 5개 (Morning/Signal-Scan/Experiment-Check/Daily-Log/Weekly-Review) | 솔로 파운더 1명이 하루 5번 알림을 받음 — noise → 무시 → 가치 소실 |
| 타임존 | 코드 하드코딩 `Asia/Seoul` (`scheduler/index.ts:105`) | 해외/원격 사용자는 cron 시각을 조정 불가 |
| 과제 실행 공간 | 메신저에 별도 채널 없음. 명령 답변은 `owner-command`에서 단발 | 멀티턴 PM 흐름, 중간 보고, 에러 관리할 공간 부재 |

### 1.2 결정의 근거 (앞선 대화)

- 사용자 입력: 채널 2개로 단축, 한 채널은 입력 전용, 다른 한 채널은 workflow 진행. 과제별 채널은 만들지 않고 **스레드로 분기**.
- 라우틴은 사라지지 않음 — 데이터 수집(signal/experiment 등)은 백그라운드에서 계속 돌고, **2회 브리프가 그 결과를 통합 보고**한다는 분리.
- 타임존은 setup에서 선택, 기본 Asia/Seoul.

## 2. 목표

1. **채널 수 6 → 2** 로 축소. 사용자 멘탈 모델 단순화.
2. **알림 5회 → 2회** 로 통합. 노이즈 ↓, 컨텍스트 ↑.
3. **스레드 = 단위 과제**. PM 모드(v0.4)가 들어설 자리를 미리 확보.
4. **타임존·브리프 시간**을 1급 시민으로. 코드 하드코딩 제거.
5. v0.2.x 사용자의 기존 JSONL 메모리·workflows 데이터 **완전 보존**.

## 3. 새 채널 구조

### 3.1 채널 2개

| 채널 | 방향 | 용도 |
|---|---|---|
| `owner-command` | 양방향 | 사용자 명령 입력, PM과의 PRD 구체화 대화 (멀티턴). v0.4 PM 모드의 진입점 |
| `workflow` | 양방향 (스레드) | 모든 과제 실행. 채널 root에는 **브리프**만, 모든 과제 진행은 스레드 안에서 |

### 3.2 `#workflow` 채널 안의 구조

```
#workflow
├── (root) 08:00 Morning Brief         ← 채널 root에 직접 post
├── (root) 18:00 Evening Brief
│
├── 🧵 [system] daily-signals          ← signal-scan 라우틴 결과 누적
├── 🧵 [system] experiments            ← experiment-check 라우틴 결과 누적
├── 🧵 [system] errors                 ← 시스템 에러 알림 (옛 #errors 대체)
│
├── 🧵 wf-2026-05-12-landing-redesign  ← 사용자 워크플로 1 (PM 모드에서 자동 생성)
├── 🧵 wf-2026-05-15-pricing-test      ← 사용자 워크플로 2
└── 🧵 wf-...                          ← 워크플로마다 1 스레드
```

**규칙**:
- 채널 root에 post하는 것은 **브리프 2종과 워크플로 시작 알림뿐**.
- 모든 라우틴 결과는 **system 스레드**로.
- 사용자 시작 워크플로는 **`wf-<YYYY-MM-DD>-<slug>` 형식의 신규 스레드**에 격리.
- 사용자가 스레드 내부에서 메시지 보내면 → 해당 워크플로의 PM이 받아서 다음 단계에 반영.

### 3.3 메신저별 스레드 구현

| 플랫폼 | 스레드 매커니즘 | 비고 |
|---|---|---|
| Discord | `channel.threads.create({ name, autoArchiveDuration })` | 1주 비활성 시 자동 아카이브. 영구 보관 옵션 추가 |
| Slack | `chat.postMessage` + `thread_ts` 반환 → 후속은 같은 `thread_ts` | 스레드 명시적 객체 없음, 첫 메시지가 root |
| Telegram | **네이티브 스레드 없음** (forum topics는 그룹별 제한) | v0.2.4에서는 단일 chat에 메시지 prefix(`[wf-xxx]`)로 분리 — 현재 동작과 호환. 향후 v0.5에서 forum topic 옵션 검토 |

## 4. 새 라우틴 스케줄

### 4.1 사용자 노출 라우틴 = 브리프 2개

| ID | 기본 시각 | 내용 |
|---|---|---|
| `morning-brief` | 08:00 | (a) 지난 18:00 ~ 오늘 08:00 일과 요약 — system 스레드 누적 + 사용자 워크플로 진행 차분. (b) 오늘 08:00~18:00 예정된 라우틴/워크플로 단계 브리프 |
| `evening-brief` | 18:00 | (a) 오늘 08:00 ~ 18:00 일과 요약. (b) 18:00 ~ 내일 08:00 예정 브리프 |

→ `routines.ts`의 5개 entry가 2개로 축소. 기존 routine 프롬프트 파일(`assets/routines/*.md`)은 일부 재사용·일부 삭제.

### 4.2 백그라운드 라우틴 (사용자에게 직접 알림 안 보냄)

| ID | 기본 시각 | 출력 위치 | 메모리 |
|---|---|---|---|
| `signal-scan` | 12:00 | `[system] daily-signals` 스레드 1줄 요약 | `signals.jsonl` |
| `experiment-check` | 16:00 | `[system] experiments` 스레드 1줄 요약 | `experiments.jsonl` |
| `weekly-review` | 일요일 20:00 | 18:00 브리프에 흡수 + `decisions.jsonl` | `decisions.jsonl` |

→ 사용자에겐 **2개만 보이지만**, 백그라운드는 기존 활동 유지. 브리프가 이들의 JSONL을 시간 차분으로 읽어 요약.

### 4.3 라우틴 단위 시간 변경

`workspace.yaml`에서 사용자가 변경 가능. cron 직접 입력 대신 **HH:MM 시각**으로 단순화. 시간대는 workspace.yaml의 `timezone`을 따름.

## 5. workspace.yaml 신규 필드

```yaml
version: "1.2.4"
display_name: "solosquad"
persona: "personal"
timezone: "Asia/Seoul"             # ← 신규. IANA tz string
briefings:                          # ← 신규
  morning:
    time: "08:00"
    enabled: true
  evening:
    time: "18:00"
    enabled: true
background_routines:                # ← 신규. 사용자가 끌 수 있음
  signal_scan:
    time: "12:00"
    enabled: true
  experiment_check:
    time: "16:00"
    enabled: true
  weekly_review:
    day: "sunday"
    time: "20:00"
    enabled: true
created_at: "..."
last_migrated_to: "1.2.4"
```

**스키마 원칙**: cron 문자열 노출하지 않음. 시각 + 요일(주간만)로 충분.

## 6. init 위저드 변경

### 6.1 신규 prompt

```
-- Step 3: Configuration --
? Workspace display name: solosquad
? Your name: ...
? Your role: ...
? Messenger platform: discord
? Timezone: (example: Asia/Seoul = UTC+09)
    > Asia/Seoul (recommended, UTC+09)
      America/Los_Angeles
      America/New_York
      Europe/London
      UTC
      (Other — type IANA string)

? Morning brief time (HH:MM, local timezone): 08:00
? Evening brief time (HH:MM, local timezone): 18:00

  (Background routines run automatically and feed into the briefs.
   You can disable them later in workspace.yaml.)
```

### 6.2 검증

- Timezone: `Intl.DateTimeFormat().resolvedOptions().timeZone` 또는 `Intl.supportedValuesOf('timeZone')`로 화이트리스트 검증.
- HH:MM: 정규식 + 두 브리프 간 최소 4시간 간격 권장 (경고만, block은 아님).

## 8. 마이그레이션 (v0.2.3 → v0.2.4)

### 8.1 자동

```bash
$ solosquad migrate --apply
Source: v0.2.3
Target: v0.2.4

  ✓ workspace.yaml: timezone="Asia/Seoul" 주입
  ✓ workspace.yaml: briefings.morning.time="08:00", evening.time="18:00" 주입
  ✓ workspace.yaml: background_routines 기본 활성화 주입
  ✓ JSONL 메모리: 변경 없음 (보존)
  ✓ workflows/: 변경 없음 (보존)
```

### 8.2 수동 (사용자 안내)

메신저 측 채널은 봇이 임의로 삭제하지 않음:

```
[i] 봇이 새로 생성한 채널:
       #owner-command  (이미 존재 — 유지)
       #workflow       (신규 생성)

[i] 더 이상 사용하지 않는 기존 채널 (수동 archive 권장):
       #daily-brief
       #experiments
       #weekly-review
       #signals
       #errors

[i] 기존 JSONL 메모리는 그대로 보존됨:
       <org>/memory/signals.jsonl
       <org>/memory/experiments.jsonl
       <org>/memory/decisions.jsonl
       <org>/memory/routine-logs/...
```

→ 사용자가 5개 채널을 archive해도 봇 동작에 영향 없음. 데이터는 파일 시스템이 정본.

## 9. 운영 시나리오 (Walkthrough)

### 9.1 신규 사용자

1. `solosquad init` → timezone Asia/Seoul, 브리프 08:00/18:00 선택.
2. `solosquad bot` 첫 부팅 → Discord 서버에 `#owner-command`, `#workflow` 2개 생성. `[system] daily-signals`, `[system] experiments`, `[system] errors` 스레드 자동 생성.
3. `solosquad schedule` 백그라운드 시작 → 한국 시간 08:00·12:00·16:00·18:00·일20:00 cron 등록.

### 9.2 사용자 워크플로 시작 (v0.4 PM 모드 가정)

1. 사용자가 `#owner-command`에 "랜딩 페이지 리디자인 진행" 입력.
2. PM이 멀티턴으로 PRD 구체화 (같은 채널, 사용자에게 직접 reply).
3. 컨펌 후 PM이 `#workflow` 채널에 `wf-2026-05-12-landing-redesign` 스레드 생성, 첫 메시지로 PRD 요약 + 단계 목록 post.
4. 백그라운드 워커가 단계별로 에이전트 실행 → 단계 완료 시 동일 스레드에 1줄 요약 + 산출물 경로.
5. 에러 발생 시 동일 스레드에 `❌ <stage>: <error>` post. 사용자가 스레드에서 답변하면 PM이 받아 다음 행동 결정.

### 9.3 일과 흐름

```
07:55  signal-scan은 12:00에 돌고, weekly-review는 일20:00에 돔
08:00  📋 Morning Brief (채널 root)
         (a) 어제 18:00 ~ 오늘 08:00 일과 요약:
             • signal-scan 12:00: 3건 신호 추가
             • wf-pricing-test: stage-3 QA 완료
             • errors: 없음
         (b) 오늘 08:00 ~ 18:00 예정:
             • 12:00 signal-scan
             • 16:00 experiment-check
             • wf-landing-redesign stage-4 시작 예정
12:00  signal-scan → [system] daily-signals 스레드에 결과 post
16:00  experiment-check → [system] experiments 스레드에 결과 post
18:00  📋 Evening Brief
         (a) 오늘 08:00 ~ 18:00 일과 요약: ...
         (b) 18:00 ~ 내일 08:00 예정: ...
```

## 10. 비파괴성·롤백

- **데이터**: JSONL과 `workflows/` 디렉터리는 손대지 않음.
- **workspace.yaml 백업**: migrate 시 자동으로 `.solosquad/backups/workspace-1.2.3-<ts>.yaml`.
- **롤백**: `solosquad migrate --rollback`은 v0.2.3 시점 workspace.yaml로 복원. 메신저 채널은 그대로 — 사용자가 archive를 되돌리거나 새 채널을 무시.
- **호환성**: v0.2.3 봇 바이너리는 새 workspace.yaml의 신규 필드를 무시하고 정상 부팅 (forward-compatible). 단, 채널이 2개로 줄어 있으므로 routine output이 사라진 채널로 전송 시도 → `sendToChannel`이 조용히 fail. → 다운그레이드는 **권장하지 않음**.

---

## 11. Out of scope (v0.2.4에서 다루지 않음)

- **PM 모드 / Intent 분류**: v0.4 마일스톤. 본 문서는 채널·스레드·브리프 인프라만 깔아둠.
- **프로젝트 워커 루프 자동화**: v0.5. v0.2.4의 `#workflow` 스레드는 PM 모드 도입 전까지는 라우틴 출력 전용으로 동작.
- **메신저별 thread 동등성**: Telegram 스레드 부재는 prefix workaround. forum topic 도입은 v0.5+ 별도 검토.
- **사용자 정의 백그라운드 라우틴 추가**: workspace.yaml에서 활성/비활성만 가능. 새 라우틴 등록은 v0.3+ 별도 스코프.
